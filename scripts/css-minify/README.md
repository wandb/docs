# CSS minify

Bundles the docs site's CSS partials into a single minified file that
Mintlify serves to the browser.

## Why this exists

The docs site overrides the Mintlify Aspen theme with a stack of custom
CSS partials (`colors.css`, `typography.css`, `topbar.css`, etc.).
Mintlify is told about a single stylesheet via the `stylesheet` key in
`docs.json`, which points at `css/styles.css`. That file used to be a
hand-maintained `@import` chain pointing at the seven partials in
cascade order.

That `@import` chain was the slowest possible way to load CSS:

1. Browser fetches `css/styles.css`.
2. Parser sees seven `@import` statements.
3. Browser fetches each partial **serially**, one round trip at a time.
4. Until the chain finishes, none of the override rules are applied —
   so Mintlify's defaults paint first and then visibly flicker into our
   overrides as each `@import` resolves.

This directory replaces the import chain with a build step:

- The seven partials live here in `scripts/css-minify/` and are the
  source of truth — the only files you edit.
- `main.css` is the bundler's entry point — it `@import`s the partials
  in the correct cascade order.
- [`clean-css-cli`](https://github.com/clean-css/clean-css-cli) bundles
  the imports inline, minifies the result, and overwrites
  `css/styles.css` with the bundled output.
- `docs.json` still references `"stylesheet": "/css/styles.css"`, so
  Mintlify receives one small file instead of eight.

The flicker goes away because there is one round trip and no parser
stall waiting on imports.

## File layout

```
scripts/css-minify/
├── README.md          this file
├── package.json       declares clean-css-cli + chokidar-cli, npm scripts
├── package-lock.json  locked dependency versions (committed)
├── .gitignore         ignores node_modules/
├── main.css           bundler entry — the @import order lives here
├── colors.css         } source partials — the only files you should edit
├── typography.css     }
├── layout.css         }
├── topbar.css         }
├── sidebar.css        }
├── buttons.css        }
├── cards.css          }
└── node_modules/      installed deps (gitignored)

css/
├── styles.css         GENERATED by `npm run build`. Do not edit by hand.
│                      Committed because Mintlify hosting deploys the repo
│                      as-is and serves whatever is on disk.
└── assets/            image assets referenced from the partials
                       (e.g. light-dots.png used by --page-texture, loaded
                       via the absolute URL `/css/assets/light-dots.png`)
```

The `css/` directory at the repo root is no longer the source of CSS —
the only file there you would ever touch is the generated `styles.css`,
and even that you don't edit by hand. Everything else in `css/` is
asset files referenced from the partials.

## First-time setup

```bash
cd scripts/css-minify
npm install
```

This installs `clean-css-cli` (the bundler/minifier) and `chokidar-cli`
(file watcher used by `npm run watch`) into
`scripts/css-minify/node_modules/`. Nothing is installed globally and
nothing lands in the repo root. Both packages and their entire
dependency trees are MIT/ISC/BSD-3-Clause licensed and run no
install-time scripts, which keeps the org's Socket policy happy.

## Daily workflow

### While editing CSS locally

In one terminal, start a watcher that rebuilds `css/styles.css`
(unminified, for easier debugging) every time any partial changes:

```bash
cd scripts/css-minify
npm run watch
```

In another terminal, run the Mintlify dev server as usual
(`mint dev` from the repo root). Mintlify watches `css/styles.css` and
live-reloads the browser whenever the watcher rewrites it, so editing
a partial gives you a near-instant refresh.

### Before committing

Produce the minified production build that will actually ship to users:

```bash
cd scripts/css-minify
npm run build
```

This rewrites `css/styles.css` with the minified output. Stage and
commit `css/styles.css` along with whatever partial(s) you edited.

If you forget this step on a PR it is not fatal — the
[`Build CSS`](../../.github/workflows/build-css.yml) workflow rebuilds
`css/styles.css` and pushes the result back to the PR branch whenever
any file under `scripts/css-minify/` changes. Running the build locally
still has value: your local Mintlify preview reflects the change
immediately, and you avoid the round-trip wait for CI.

## npm scripts

| Script              | What it does                                                                |
| ------------------- | --------------------------------------------------------------------------- |
| `npm run build`     | Bundle + **minify**. Use this before committing. Writes `css/styles.css`.   |
| `npm run build:dev` | Bundle without minifying. Easier to inspect when debugging.                 |
| `npm run watch`     | One-shot dev build, then rebuild on every change to a partial. Uses `build:dev` (unminified) so output stays inspectable. |

All three write to the same path: `../../css/styles.css`.

## Editing CSS

**Always edit the partials in this directory.**

- The partials are heavily commented and organized by concern
  (tokens, layout, topbar, sidebar, buttons, cards). The comments
  document *why* each rule exists, which Mintlify class it overrides,
  and any specificity tricks involved. None of those comments survive
  the bundle step — `clean-css` strips comments to keep the production
  payload small.
- The cascade order in `main.css` matters: tokens first
  (`colors.css`, `typography.css`), then page chrome (`layout.css`),
  then navigation, then components. Re-ordering imports changes which
  rule wins at equal specificity. If you add a new partial, decide
  where it slots in and add a one-line comment in `main.css`
  explaining why.
- **Do not edit `css/styles.css` directly.** Anything you write there
  is wiped the next time anyone runs `npm run build`. Edit the source
  partial in this directory and rebuild.

## Browser targets and modern CSS

`clean-css` is a minifier — it does not transform syntax or add vendor
prefixes. The partials use modern CSS (`:has()`, `color-mix()`, custom
properties, CSS nesting) and clean-css passes those through verbatim.
That works because the docs site only targets evergreen browsers
(Chrome, Safari, Firefox, Edge), all of which support these features
natively.

If the site ever has to support a browser that needs prefixes, fold an
`autoprefixer` step into the build pipeline rather than swapping
minifiers. We picked clean-css specifically because it has no native
binaries and no install-time scripts, which keeps the org's Socket
license/security policy clean.

## Updating dependencies

To bump `clean-css-cli` or `chokidar-cli`:

```bash
cd scripts/css-minify
npm update           # respects semver ranges in package.json
npm run build        # confirm the bundle still produces a valid styles.css
```

Then commit the updated `package-lock.json` along with `css/styles.css`
(in case the new bundler version produces a slightly different
minified output).

## Optional: pre-commit hook

If you want to guarantee `css/styles.css` is always in sync with the
partials, add a Git pre-commit hook that runs the build whenever a
partial under `scripts/css-minify/` is staged. Example, with
[`husky`](https://typicode.github.io/husky/):

```bash
# .husky/pre-commit
if git diff --cached --name-only | grep -qE '^scripts/css-minify/.*\.css$'; then
  (cd scripts/css-minify && npm run build) || exit 1
  git add css/styles.css
fi
```

Not configured by default — a forgotten build only costs a few KB of
bandwidth, which isn't worth the friction of every contributor needing
Husky installed.

## Troubleshooting

**"`cleancss: command not found`"** — you probably ran the script from
the wrong directory. The `cleancss` binary lives in
`scripts/css-minify/node_modules/.bin/`, which npm only puts on `PATH`
when you run scripts via `npm run` from inside `scripts/css-minify/`.

**"Module not found: ./foo.css"** — `main.css` lists a partial that
does not exist in this directory. Check the filename, then check
whether you meant to add or remove a partial.

**"Output looks right locally but the live site still flickers"** —
confirm `css/styles.css` was committed and pushed; Mintlify hosting
serves the file straight from `main`. Also verify it does not still
contain `@import` statements (those would mean the build never ran).

**`npm install` fails with `EPERM` against `~/.npm/_cacache/`** — your
local npm cache contains root-owned files (a historical npm bug). Fix
once with `sudo chown -R $(id -u):$(id -g) ~/.npm`, or work around it
per-install with `npm install --cache "$TMPDIR/npm-cache-cssmin"`.
