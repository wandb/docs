---
title: 감사 로그로 사용자 활동 추적
---

W&amp;B 감사 로그를 사용하여 조직 내 사용자 활동을 추적하고 엔터프라이즈 거버넌스 요구 사항을 충족하십시오. 감사 로그는 JSON 형식으로 제공됩니다. [감사 로그 스키마](#audit-log-schema)를 참조하십시오.

감사 로그에 액세스하는 방법은 W&amp;B 플랫폼 배포 유형에 따라 다릅니다:

| W&amp;B 플랫폼 배포 유형 | 감사 로그 액세스 방법 |
|----------------------------|--------------------------------|
| [Dedicated Cloud](/ko/platform/hosting/hosting-options/dedicated_cloud) | <ul><li>[Instance-level BYOB](/ko/platform/hosting/data-security/secure-storage-connector): 10분마다 인스턴스 수준 버킷(BYOB)으로 동기화됩니다. [API](#fetch-audit-logs-using-api)를 사용해서도 확인할 수 있습니다.</li><li>기본 인스턴스 수준 스토리지: [API](#fetch-audit-logs-using-api)를 사용하는 경우에만 확인할 수 있습니다.</li></ul> |
| [Multi-tenant Cloud](/ko/platform/hosting/hosting-options/multi_tenant_cloud) | 엔터프라이즈 플랜에서만 사용할 수 있습니다. [API](#fetch-audit-logs-using-api)를 사용하는 경우에만 확인할 수 있습니다. |
| [Self-Managed](/ko/platform/hosting/hosting-options/self-managed) | 10분마다 인스턴스 수준 버킷으로 동기화됩니다. [API](#fetch-audit-logs-using-api)를 사용해서도 확인할 수 있습니다. |

감사 로그를 가져온 후 [Pandas](https://pandas.pydata.org/docs/index.html), [Amazon Redshift](https://aws.amazon.com/redshift/), [Google BigQuery](https://cloud.google.com/bigquery), [Microsoft Fabric](https://www.microsoft.com/microsoft-fabric) 같은 도구를 사용하여 분석할 수 있습니다. 일부 감사 로그 분석 도구는 JSON 형식을 지원하지 않습니다. 분석 전에 JSON 형식 감사 로그를 어떻게 변환해야 하는지에 대한 가이드라인과 요구 사항은 사용하는 분석 도구의 문서를 참조하십시오.

로그 형식에 대한 자세한 내용은 [감사 로그 스키마](#audit-log-schema)와 [동작](#actions)을 참조하십시오.

<div id="audit-log-retention">
  ## 감사 로그 보존
</div>

* 특정 기간 동안 감사 로그를 보존해야 하는 경우, W&amp;B에서는 스토리지 버킷 또는 Audit Logging API를 사용해 로그를 정기적으로 장기 보존 스토리지로 이전할 것을 권장합니다.
* [Health Insurance Portability and Accountability Act of 1996 (HIPAA)](https://hhs.gov/hipaa/for-professionals/index.html)의 적용을 받는 경우, 감사 로그는 필수 보존 기간이 끝나기 전까지 어떤 내부 또는 외부 주체도 삭제하거나 수정할 수 없는 환경에서 최소 6년 동안 보존되어야 합니다. [BYOB](/ko/platform/hosting/data-security/secure-storage-connector)가 설정된 HIPAA 준수 [Dedicated Cloud](/ko/platform/hosting/hosting-options/dedicated_cloud) 인스턴스의 경우, 장기 보존 스토리지를 포함한 관리형 스토리지 전반에 대해 가드레일을 구성해야 합니다.

<div id="audit-log-schema">
  ## 감사 로그 스키마
</div>

다음 표는 감사 로그 항목에 나타날 수 있는 모든 키를 알파벳 순으로 보여줍니다. 수행된 작업과 상황에 따라 특정 로그 항목에는 가능한 필드 중 일부만 포함될 수 있습니다.

| Key | Definition |
|---------| -------|
|`action`                  | 이벤트의 [action](#actions).
|`actor_email`             | 해당하는 경우, 작업을 시작한 사용자의 이메일 주소.
|`actor_ip`                | 작업을 시작한 사용자의 IP 주소.
|`actor_user_id`           | 해당하는 경우, 작업을 수행한 로그인된 사용자의 ID.
|`artifact_asset`          | 해당하는 경우, 작업과 연관된 아티팩트 ID.
|`artifact_digest`         | 해당하는 경우, 작업과 연관된 아티팩트 다이제스트.
|`artifact_qualified_name` | 해당하는 경우, 작업과 연관된 아티팩트의 전체 이름.
|`artifact_sequence_asset` | 해당하는 경우, 작업과 연관된 아티팩트 시퀀스 ID.
|`cli_version`             | 해당하는 경우, 작업을 시작한 Python SDK 버전.
|`entity_asset`            | 해당하는 경우, 작업과 연관된 엔터티 또는 팀 ID.
|`entity_name`             | 해당하는 경우, 작업과 연관된 엔터티 또는 팀 이름.
|`project_asset`           | 해당하는 경우, 작업과 연관된 프로젝트.
|`project_name`            | 해당하는 경우, 작업과 연관된 프로젝트 이름.
|`report_asset`            | 해당하는 경우, 작업과 연관된 리포트 ID.
|`report_name`             | 해당하는 경우, 작업과 연관된 리포트 이름.
|`response_code`           | 해당하는 경우, 작업에 대한 HTTP 응답 코드.
|`timestamp`               | [RFC3339 형식](https://www.rfc-editor.org/rfc/rfc3339)의 이벤트 발생 시각. 예를 들어 `2023-01-23T12:34:56Z`는 2023년 1월 23일 12:34:56 UTC를 의미합니다.
|`user_asset`              | 해당하는 경우, 작업을 수행한 사용자가 아니라 작업의 영향을 받는 사용자 자산.
|`user_email`              | 해당하는 경우, 작업을 수행한 사용자의 이메일 주소가 아니라 작업의 영향을 받는 사용자의 이메일 주소.

<div id="personally-identifiable-information-pii">
  ### 개인식별정보(PII)
</div>

이메일 주소, 프로젝트, 팀, 리포트 이름과 같은 개인식별정보(PII)는 API 엔드포인트 옵션을 사용하는 경우에만 확인할 수 있습니다.

* [Self-Managed](/ko/platform/hosting/hosting-options/self-managed) 및
  [Dedicated Cloud](/ko/platform/hosting/hosting-options/dedicated_cloud)의 경우, 조직 관리자(organization admin)는 감사 로그를 가져올 때 [PII를 제외](#exclude-pii)할 수 있습니다.
* [Multi-tenant Cloud](/ko/platform/hosting/hosting-options/multi_tenant_cloud)의 경우, API 엔드포인트는 PII를 포함하여 감사 로그와 관련된 필드를 항상 반환합니다. 이는 설정을 변경할 수 없습니다.

<div id="before-you-begin">
  ## 시작하기 전에
</div>

1. 감사 로그는 조직 수준 관리자만 가져올 수 있습니다. `403` 오류가 발생하면, 본인 또는 서비스 계정에 충분한 권한이 있는지 확인하세요.
2. **Multi-tenant Cloud**: 여러 개의 Multi-tenant Cloud 조직에 속해 있는 경우, 감사 로깅 API 호출이 어떤 조직으로 라우팅될지 결정하는 **Default API organization**을 *반드시* 설정해야 합니다. 그렇지 않으면 다음 오류가 발생합니다:

   ```text
   user is associated with multiple organizations but no valid org ID found in user info
   ```

   기본 API 조직을 지정하려면:

   1. 프로필 이미지를 클릭한 다음 **User Settings**를 클릭합니다.
   2. **Default API organization**에서 조직을 선택합니다.

   이는 서비스 계정에는 적용되지 않습니다. 서비스 계정은 하나의 Multi-tenant Cloud 조직에만 속할 수 있습니다.

<div id="fetch-audit-logs">
  ## 감사 로그 가져오기
</div>

감사 로그를 가져오려면:

1. 인스턴스에 맞는 적절한 API 엔드포인트를 확인합니다:

* [Self-Managed](/ko/platform/hosting/hosting-options/self-managed): `<wandb-platform-url>/admin/audit_logs`
* [Dedicated Cloud](/ko/platform/hosting/hosting-options/dedicated_cloud): `<instance-name>.wandb.io/admin/audit_logs`
* [Multi-tenant Cloud (Enterprise required)](/ko/platform/hosting/hosting-options/multi_tenant_cloud): `https://api.wandb.ai/audit_logs`

  다음 단계에서 `<API-endpoint>`를 해당 API 엔드포인트로 바꿉니다.

1. (선택 사항) 엔드포인트에 추가할 쿼리 매개변수를 구성합니다. 다음 단계에서 `<parameters>`를 결과 문자열로 바꿉니다.
   * `anonymize`: URL에 `anonymize=true` 매개변수가 포함되어 있으면 모든 PII를 제거합니다. 그렇지 않으면 PII가 포함됩니다. [감사 로그를 가져올 때 PII 제외](#exclude-pii-when-fetching-audit-logs)를 참조하세요. Multi-tenant Cloud에서는 지원되지 않으며, 이 경우 PII를 포함한 모든 필드가 포함됩니다.
   * `numdays`와 `startDate` 조합을 사용하여 가져올 로그의 날짜 범위를 구성합니다. 각 매개변수는 선택 사항이며 서로 연동되어 동작합니다.
     * 두 매개변수가 모두 포함되지 않은 경우 오늘 로그만 가져옵니다.
     * `numDays`: `startDate`로부터 과거로 며칠 치의 로그를 가져올지 나타내는 정수입니다. 생략되었거나 `0`으로 설정된 경우 `startDate` 하루에 대한 로그만 가져옵니다. **Multi-tenant Cloud** 조직은 최대 7일 치 감사 로그만 가져올 수 있습니다. 다시 말해 `numDays=9`로 설정하면 실제로는 `numDays=7`이 적용됩니다.
     * `startDate`: 가져올 최신 로그의 기준 날짜를 제어하며, 형식은 `startDate=YYYY-MM-DD`입니다. 생략되었거나 오늘 날짜로 명시적으로 설정된 경우, 오늘부터 `numDays` (Multi-tenant Cloud의 경우 최대 `7`)까지의 로그를 가져옵니다.
2. `<API-endpoint>?<parameters>` 형식으로 완전한 엔드포인트 URL을 구성합니다.
3. 웹 브라우저나 [Postman](https://www.postman.com/downloads/), [HTTPie](https://httpie.io/), 또는 cURL 같은 도구를 사용하여 완전한 API 엔드포인트에 HTTP `GET` 요청을 보냅니다.

API 응답에는 줄바꿈으로 구분된 JSON 객체가 포함됩니다. 객체에는 감사 로그가 인스턴스 수준 버킷으로 동기화될 때와 마찬가지로 [스키마](#audit-log-schema)에 설명된 필드가 포함됩니다. 이런 경우 감사 로그는 버킷의 `/wandb-audit-logs` 디렉터리에 위치합니다.

<div id="use-basic-authentication">
  ### 기본 인증 사용
</div>

API 키를 사용해 감사 로그 API에 기본 인증으로 접근하려면, HTTP 요청의 `Authorization` 헤더를 `Basic` 문자열과 공백 한 칸, 그리고 `username:API-KEY` 형식의 문자열을 Base64로 인코딩한 값을 이어 붙인 문자열로 설정합니다. 다시 말해, 사용자 이름과 API 키를 `:` 문자로 구분해 하나의 문자열로 만든 뒤, 그 결과를 Base64로 인코딩합니다. 예를 들어 `demo:p@55w0rd`로 인증하려면 헤더는 `Authorization: Basic ZGVtbzpwQDU1dzByZA==`가 되어야 합니다.

### 감사 로그를 가져올 때 PII 제외하기 <a id="exclude-pii" />

[Self-Managed](/ko/platform/hosting/hosting-options/self-managed) 및 [Dedicated Cloud](/ko/platform/hosting/hosting-options/dedicated_cloud) 환경에서는 W&amp;B 조직 또는 인스턴스 관리자가 감사 로그를 가져올 때 PII를 제외할 수 있습니다. [Multi-tenant Cloud](/ko/platform/hosting/hosting-options/multi_tenant_cloud) 환경에서는 API 엔드포인트가 PII를 포함한 감사 로그 관련 필드를 항상 반환합니다. 이는 구성 가능한 항목이 아닙니다.

PII를 제외하려면 `anonymize=true` URL 매개변수를 전달하십시오. 예를 들어 W&amp;B 인스턴스 URL이 `https://mycompany.wandb.io`이고, 지난 일주일 동안의 사용자 활동에 대한 감사 로그를 가져오면서 PII를 제외하고 싶다면, 다음과 같은 API 엔드포인트를 사용하십시오:

```text
https://mycompany.wandb.io/admin/audit_logs?anonymize=true&<additional-parameters>.
```

<div id="actions">
  ## Actions
</div>

이 표는 W&amp;B에서 기록될 수 있는 작업을 알파벳순으로 정리한 것입니다.

|Action | Definition |
|-----|-----|
| `artifact:create`             | Artifact가 생성됩니다.
| `artifact:delete   `          | Artifact가 삭제됩니다.
| `artifact:read`               | Artifact가 조회됩니다.
| `project:delete`              | 프로젝트가 삭제됩니다.
| `project:read`                | 프로젝트가 조회됩니다.
| `report:read`                 | 리포트가 조회됩니다. <sup><a href="#1">1</a></sup>
| `run:delete_many`             | 여러 실행이 일괄 삭제됩니다.
| `run:delete`                  | 실행이 삭제됩니다.
| `run:stop`                    | 실행이 중지됩니다.
| `run:undelete_many`           | 휴지통에서 여러 실행이 일괄 복원됩니다.
| `run:update_many`             | 여러 실행이 일괄 업데이트됩니다.
| `run:update`                  | 실행이 업데이트됩니다.
| `sweep:create_agent`          | 스위프 에이전트가 생성됩니다.
| `team:create_service_account` | 팀용 서비스 계정이 생성됩니다.
| `team:create`                 | 팀이 생성됩니다.
| `team:delete`                 | 팀이 삭제됩니다.
| `team:invite_user`            | 사용자가 팀에 초대됩니다.
| `team:uninvite`               | 사용자 또는 서비스 계정에 대한 팀 초대가 취소됩니다.
| `user:create_api_key`         | 사용자 또는 서비스 계정용 API 키가 생성됩니다. <sup><a href="#1">1</a></sup>
| `user:create`                 | 사용자가 생성됩니다. <sup><a href="#1">1</a></sup>
| `user:deactivate`             | 사용자가 비활성화됩니다. <sup><a href="#1">1</a></sup>
| `user:delete_api_key`         | 사용자 또는 서비스 계정용 API 키가 삭제됩니다. <sup><a href="#1">1</a></sup>
| `user:initiate_login`         | 사용자가 로그인을 시작합니다. <sup><a href="#1">1</a></sup>
| `user:login`                  | 사용자가 로그인합니다. <sup><a href="#1">1</a></sup>
| `user:logout`                 | 사용자가 로그아웃합니다. <sup><a href="#1">1</a></sup>
| `user:permanently_delete`     | 사용자가 영구적으로 삭제됩니다. <sup><a href="#1">1</a></sup>
| `user:reactivate`             | 사용자가 다시 활성화됩니다. <sup><a href="#1">1</a></sup>
| `user:read`                   | 사용자 프로필이 조회됩니다. <sup><a href="#1">1</a></sup>
| `user:update`                 | 사용자가 업데이트됩니다. <sup><a href="#1">1</a></sup>

<a id="1">1</a>: [Multi-tenant Cloud](/ko/platform/hosting/hosting-options/multi_tenant_cloud)에서는 다음의 경우 감사 로그가 수집되지 않습니다.

* 공개(Open 또는 Public) 프로젝트
* `report:read` 작업
* 특정 조직에 연결되지 않은 `User` 작업
