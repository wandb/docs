<div id="wb-kubernetes-operator">
  ## W&amp;B Kubernetes Operator
</div>

Kubernetes에서 W&amp;B Server를 배포, 관리, 트러블슈팅하고 확장하는 작업을 단순화하려면 W&amp;B Kubernetes Operator를 사용하세요. Operator를 W&amp;B 인스턴스를 위한 지능형 도우미라고 생각하면 됩니다.

W&amp;B Server 아키텍처와 설계는 AI 개발자 도구 기능을 확장하고, 고성능, 더 나은 확장성, 쉬운 관리를 위한 적절한 기본 구성 요소를 제공하기 위해 지속적으로 발전하고 있습니다. 이러한 발전은 컴퓨트 서비스, 관련 스토리지, 그리고 이들 간의 연결성 모두에 적용됩니다. 다양한 배포 유형 전반에 걸쳐 지속적인 업데이트와 개선을 쉽게 하기 위해, W&amp;B는 Kubernetes operator를 사용합니다.

<Note>
  W&amp;B는 AWS, Google Cloud 및 Azure 퍼블릭 클라우드에서 Dedicated Cloud 인스턴스를 배포하고 관리하기 위해 operator를 사용합니다.
</Note>

Kubernetes operator에 대한 자세한 내용은 Kubernetes 문서의 [Operator pattern](https://kubernetes.io/docs/concepts/extend-kubernetes/operator/)을 참조하세요.

<div id="reasons-for-the-architecture-shift">
  ### 아키텍처 변경의 이유
</div>

기존에는 W&amp;B 애플리케이션을 Kubernetes 클러스터 내의 단일 Deployment와 Pod, 혹은 단일 Docker 컨테이너로 배포했습니다. W&amp;B는 과거부터 지금까지 Database와 Object Store를 외부로 분리해 두는 것을 권장해 왔습니다. Database와 Object Store를 외부화하면 애플리케이션의 상태를 디커플링할 수 있습니다.

애플리케이션이 성장함에 따라 모놀리식 컨테이너에서 분산 시스템(마이크로서비스)으로 전환해야 할 필요성이 분명해졌습니다. 이러한 변화는 백엔드 로직 처리를 더 쉽게 만들고, Kubernetes에 내장된 인프라 기능을 자연스럽게 활용할 수 있게 해 줍니다. 분산 시스템은 또한 W&amp;B가 의존하는 추가 기능을 위해 필요한 새로운 서비스를 배포하는 것도 지원합니다.

2024년 이전에는 Kubernetes 관련 변경 사항이 있을 때마다 [terraform-kubernetes-wandb](https://github.com/wandb/terraform-kubernetes-wandb) Terraform 모듈을 수동으로 업데이트해야 했습니다. Terraform 모듈을 업데이트해야 클라우드 제공업체 간 호환성을 보장하고, 필요한 Terraform 변수들을 구성하며, 각 백엔드나 Kubernetes 수준의 변경마다 `terraform apply`를 실행할 수 있었습니다.

이 과정은 W&amp;B Support가 각 고객의 Terraform 모듈 업그레이드를 개별적으로 도와야 했기 때문에 확장성이 없었습니다.

해결책은 중앙 [deploy.wandb.ai](https://deploy.wandb.ai) 서버에 연결하는 Operator를 구현하는 것이었습니다. 이 Operator는 지정된 릴리스 채널에 대한 최신 스펙 변경 사항을 요청해 적용합니다. 라이선스가 유효한 동안 업데이트를 계속 수신합니다. [Helm](https://helm.sh/)은 W&amp;B Operator의 배포 메커니즘으로 사용되며, 동시에 Operator가 W&amp;B Kubernetes 스택의 모든 구성 템플릿을 처리하는 수단으로도 사용됩니다. 말 그대로 Helm-ception입니다.

<div id="how-it-works">
  ### 작동 방식
</div>

오퍼레이터는 Helm 또는 소스에서 설치할 수 있습니다. 자세한 설치 방법은 [charts/operator](https://github.com/wandb/helm-charts/tree/main/charts/operator)를 참고하세요.

설치 과정에서는 `controller-manager`라는 deployment를 생성하고, `weightsandbiases.apps.wandb.com`이라는 이름의 [custom resource](https://kubernetes.io/docs/concepts/extend-kubernetes/api-extension/custom-resources/) definition (shortName: `wandb`)을 사용합니다. 이 리소스는 단일 `spec`을 받아 클러스터에 적용합니다:

```yaml
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: weightsandbiases.apps.wandb.com
```

`controller-manager`는 커스텀 리소스의 spec, 릴리스 채널, 그리고 사용자가 정의한 config를 기반으로 [charts/operator-wandb](https://github.com/wandb/helm-charts/tree/main/charts/operator-wandb)를 설치합니다. 구성 사양 계층 구조는 사용자에게 최대한의 구성 유연성을 제공하고, W&amp;B가 새로운 이미지, 구성, 기능, 그리고 Helm 업데이트를 자동으로 릴리스할 수 있도록 합니다.

구성 옵션은 [configuration specification hierarchy](#configuration-specification-hierarchy)와 [configuration reference](#configuration-reference-for-wb-operator)를 참고하세요.

배포는 여러 개의 파드(Pod)로 구성되며, 서비스마다 하나씩 존재합니다. 각 파드의 이름 앞에는 `wandb-` 접두사가 붙습니다.

<div id="configuration-specification-hierarchy">
  ### 구성 사양 계층 구조
</div>

구성 사양은 상위 수준의 사양이 하위 수준의 사양을 덮어쓰는 계층적 모델을 따릅니다. 동작 방식은 다음과 같습니다:

* **Release Channel 값**: 이 기본 수준 구성은 W&amp;B에서 해당 배포에 대해 설정한 Release Channel을 기준으로 기본 값과 구성을 정의합니다.
* **User Input 값**: 사용자는 System Console을 통해 Release Channel 사양에서 제공하는 기본 설정을 덮어쓸 수 있습니다.
* **Custom Resource 값**: 사용자에 의해 정의되는 가장 높은 수준의 사양입니다. 여기에서 지정한 모든 값은 User Input 및 Release Channel 사양보다 우선합니다. 구성 옵션에 대한 자세한 설명은 [Configuration Reference](#configuration-reference-for-wb-operator)를 참조하십시오.

이러한 계층적 모델을 사용하면 업그레이드와 변경을 체계적이고 관리 가능한 방식으로 유지하면서도 다양한 요구 사항에 맞게 구성을 유연하게 커스터마이즈할 수 있습니다.

<div id="run-the-container-as-an-un-privileged-user">
  #### 비특권 사용자로 컨테이너 실행
</div>

기본적으로 컨테이너는 `$UID` 999를 사용합니다. 오케스트레이터가 컨테이너를 루트가 아닌 사용자로 실행하도록 요구하는 경우 `$UID` &gt;= 100000 및 `$GID` 0을 지정하십시오.

<Note>
  파일 시스템 권한이 올바르게 동작하려면 W&amp;B는 루트 그룹(`$GID=0`)으로 시작해야 합니다.
</Note>

각 W&amp;B 구성 요소에 대해 security context를 구성하십시오. 예를 들어, API 구성 요소를 구성하려면 다음과 같이 합니다:

```yaml
api:
  install: true
  image:
    repository: wandb/megabinary
    tag: 0.74.1
  pod:
    securityContext:
      fsGroup: 10001
      fsGroupChangePolicy: Always
      runAsGroup: 0
      runAsNonRoot: true
      runAsUser: 10001
      seccompProfile:
        type: RuntimeDefault
  container:
    securityContext:
      allowPrivilegeEscalation: false
      capabilities:
        drop:
          - ALL
      privileged: false
      readOnlyRootFilesystem: false
```

필요한 경우 `app` 또는 `console`과 같은 다른 컴포넌트에도 사용자 지정 보안 컨텍스트를 구성하세요. 자세한 내용은 [사용자 지정 보안 컨텍스트](#custom-security-context)를 참조하세요.

<div id="deploy-wb-server-application">
  ## W&amp;B Server 애플리케이션 배포
</div>

<Note>
  **Helm과 함께 사용하는 W&amp;B Kubernetes Operator는** 클라우드, 온프레미스, 에어갭 환경을 포함한 모든 W&amp;B 셀프 관리형 배포에서 **권장되는 설치 방법입니다**.
</Note>

이 섹션에서는 W&amp;B Kubernetes Operator를 배포하는 다양한 방법을 설명합니다:

* **Helm CLI**: Helm 명령을 사용한 직접 배포
* **Helm Terraform Module**: 코드형 인프라(IaC)를 이용한 배포
* **W&amp;B Cloud Terraform Modules**: AWS, Google Cloud, Azure용 인프라 및 애플리케이션 일괄 배포

배포 방식별 추가 고려 사항은 다음 문서도 참고하세요:

* 데이터센터/베어메탈 환경에는 [온프레미스에 W&amp;B Platform 배포](/ko/platform/hosting/self-managed/bare-metal/)
* 분리된(네트워크 단절) 환경에는 [에어갭 인스턴스를 위한 Kubernetes Operator](/ko/platform/hosting/self-managed/operator-airgapped/)

<div id="deploy-wb-with-helm-cli">
  ### Helm CLI로 W&amp;B 배포하기
</div>

W&amp;B는 Kubernetes 클러스터에 W&amp;B Kubernetes 오퍼레이터를 배포하기 위한 Helm 차트를 제공합니다. 이 방법을 사용하면 Helm CLI 또는 ArgoCD와 같은 지속적 배포 도구로 W&amp;B Server를 배포할 수 있습니다. 위에서 언급한 요구 사항이 충족되었는지 확인하세요.

Helm CLI를 사용해 W&amp;B Kubernetes 오퍼레이터를 설치하려면 다음 단계를 따르세요:

1. W&amp;B Helm 리포지터리를 추가합니다. W&amp;B Helm 차트는 W&amp;B Helm 리포지터리에서 사용할 수 있습니다:
   ```shell
   helm repo add wandb https://charts.wandb.ai
   helm repo update
   ```

2. Kubernetes 클러스터에 오퍼레이터를 설치합니다:
   ```shell
   helm upgrade --install operator wandb/operator -n wandb-cr --create-namespace
   ```

3. W&amp;B 오퍼레이터 커스텀 리소스를 구성해 W&amp;B Server 설치가 실행되도록 설정합니다. W&amp;B 배포 구성이 포함된 `operator.yaml` 파일을 생성하세요. 사용 가능한 모든 옵션은 [Configuration Reference](#configuration-reference-for-wb-server)를 참조하세요.

   최소 구성 예시는 다음과 같습니다:

   ```yaml
   apiVersion: apps.wandb.com/v1
   kind: WeightsAndBiases
   metadata:
     labels:
       app.kubernetes.io/name: weightsandbiases
       app.kubernetes.io/instance: wandb
     name: wandb
     namespace: default
   spec:
     values:
       global:
         host: https://<HOST_URI>
         license: eyJhbGnUzaH...j9ZieKQ2x5GGfw
         bucket:
           <details depend on the provider>
         mysql:
           <redacted>
       ingress:
         annotations:
           <redacted>
   ```

4. 커스텀 구성을 사용해 오퍼레이터를 시작하여 W&amp;B Server 애플리케이션을 설치, 구성 및 관리할 수 있도록 합니다:

   ```shell
   kubectl apply -f operator.yaml
   ```

   설치가 완료될 때까지 기다리세요. 몇 분 정도 소요됩니다.

5. 웹 UI로 설치를 확인하려면, 먼저 첫 번째 관리자 사용자 계정을 생성한 후 [Verify the installation](#verify-the-installation)에 설명된 확인 단계를 따르세요.

<div id="deploy-wb-with-helm-terraform-module">
  ### Helm Terraform 모듈을 사용하여 W&amp;B 배포
</div>

이 방법은 Terraform의 인프라를 코드로 관리하는(Infrastructure as Code, IaC) 방식을 활용해 일관성과 재현성을 확보하면서, 특정 요구 사항에 맞춘 맞춤형 배포를 가능하게 합니다. 공식 W&amp;B Helm 기반 Terraform 모듈은 [여기](https://github.com/wandb/helm-charts/tree/main/charts/operator-wandb)에 있습니다.

아래 코드는 시작점으로 사용할 수 있으며, 프로덕션급 배포에 필요한 모든 구성 옵션을 포함합니다.

```hcl
module "wandb" {
  source  = "wandb/wandb/helm"

  spec = {
    values = {
      global = {
        host    = "https://<HOST_URI>"
        license = "eyJhbGnUzaH...j9ZieKQ2x5GGfw"

        bucket = {
          <details depend on the provider>
        }

        mysql = {
          <redacted>
        }
      }

      ingress = {
        annotations = {
          "a" = "b"
          "x" = "y"
        }
      }
    }
  }
}
```

구성 옵션은 [Configuration Reference](#configuration-reference-for-wb-operator)에서 설명한 것과 동일하지만, 구문은 HashiCorp 구성 언어(HCL)를 따라야 합니다. Terraform 모듈은 W&amp;B 커스텀 리소스 정의(CRD)를 생성합니다.

Weights &amp; Biases(W&amp;B)가 고객용 “Dedicated Cloud” 설치를 배포하기 위해 Helm Terraform 모듈을 어떻게 사용하는지 확인하려면, 다음 링크를 참조하세요:

* [AWS](https://github.com/wandb/terraform-aws-wandb/blob/45e1d746f53e78e73e68f911a1f8cad5408e74b6/main.tf#L225)
* [Azure](https://github.com/wandb/terraform-azurerm-wandb/blob/170e03136b6b6fc758102d59dacda99768854045/main.tf#L155)
* [Google Cloud](https://github.com/wandb/terraform-google-wandb/blob/49ddc3383df4cefc04337a2ae784f57ce2a2c699/main.tf#L189)

<div id="deploy-wb-with-wb-cloud-terraform-modules">
  ### W&amp;B Cloud Terraform 모듈을 사용해 W&amp;B 배포하기
</div>

W&amp;B는 AWS, Google Cloud, Azure용 Terraform 모듈 세트를 제공합니다. 이 모듈들은 Kubernetes 클러스터, 로드 밸런서, MySQL 데이터베이스 등 전체 인프라와 W&amp;B Server 애플리케이션까지 함께 배포합니다. W&amp;B Kubernetes Operator에는 아래 버전의 공식 W&amp;B 클라우드별 Terraform 모듈이 이미 사전 구성되어 있습니다:

| Terraform Registry                                                  | Source Code                                      | Version |
| ------------------------------------------------------------------- | ------------------------------------------------ | ------- |
| [AWS](https://registry.terraform.io/modules/wandb/wandb/aws/latest) | https://github.com/wandb/terraform-aws-wandb     | v4.0.0+ |
| [Azure](https://github.com/wandb/terraform-azurerm-wandb)           | https://github.com/wandb/terraform-azurerm-wandb | v2.0.0+ |
| [Google Cloud](https://github.com/wandb/terraform-google-wandb)              | https://github.com/wandb/terraform-google-wandb  | v2.0.0+ |

이 통합을 통해 W&amp;B Kubernetes Operator는 최소한의 설정만으로도 사용 중인 인스턴스에서 바로 사용할 수 있도록 준비되어 있으며, 클라우드 환경에서 W&amp;B Server를 배포하고 관리하기 위한 간소화된 프로세스를 제공합니다.

이 모듈을 사용하는 방법에 대한 자세한 설명은 문서의 [Self-Managed 설치 섹션](/ko/platform/hosting/hosting-options/self-managed#deploy-wb-server-within-self-managed-cloud-accounts)을 참고하십시오.

<div id="access-the-wb-management-console">
  ## W&amp;B Management Console에 액세스하기
</div>

W&amp;B Kubernetes operator에는 Management Console이 포함되어 있습니다. `${HOST_URI}/console` 에 위치하며, 예를 들어 `https://wandb.company-name.com/console` 과 같습니다.

Management Console에 로그인하는 방법은 두 가지가 있습니다:

<Tabs>
  <Tab title="옵션 1 (권장)">
    1. 브라우저에서 W&amp;B 애플리케이션을 열고 로그인합니다. `${HOST_URI}/` (예: `https://wandb.company-name.com/`)로 접속해 W&amp;B 애플리케이션에 로그인합니다.
    2. Management Console에 접근합니다. 오른쪽 상단 모서리에 있는 아이콘을 클릭한 다음 **System console** 을 클릭합니다. 관리자 권한이 있는 사용자만 **System console** 항목을 볼 수 있습니다.

       <Frame>
         <img src="/images/hosting/access_system_console_via_main_app.png" alt="System console access" />
       </Frame>
  </Tab>

  <Tab title="옵션 2">
    <Note>
      옵션 1이 동작하지 않을 때에만 아래 단계를 사용해 콘솔에 접근할 것을 W&amp;B에서 권장합니다.
    </Note>

    1. 브라우저에서 콘솔 애플리케이션을 엽니다. 위에서 설명한 URL을 열면 로그인 화면으로 리디렉션됩니다:
       <Frame>
         <img src="/images/hosting/access_system_console_directly.png" alt="Direct system console access" />
       </Frame>
    2. 설치 과정에서 생성된 Kubernetes Secret에서 비밀번호를 가져옵니다:
       ```shell
       kubectl get secret wandb-password -o jsonpath='{.data.password}' | base64 -d
       ```
       비밀번호를 복사합니다.
    3. 콘솔에 로그인합니다. 복사한 비밀번호를 붙여넣고 **Login** 을 클릭합니다.
  </Tab>

<div id="update-the-wb-kubernetes-operator">
  ## W&amp;B Kubernetes operator 업데이트
</div>

이 섹션에서는 W&amp;B Kubernetes operator를 업데이트하는 방법을 설명합니다.

<Note>
  * W&amp;B Kubernetes operator를 업데이트해도 W&amp;B 서버 애플리케이션은 업데이트되지 않습니다.
  * W&amp;B Kubernetes operator를 사용하지 않는 Helm chart를 사용하는 경우, 아래의 W&amp;B operator 업데이트 지침을 따르기 전에 [여기](#migrate-Self-Managed-instances-to-wb-operator)의 지침을 먼저 참조하십시오.
</Note>

아래 코드 스니펫을 복사하여 터미널에 붙여넣으십시오.

1. 먼저 [`helm repo update`](https://helm.sh/docs/helm/helm_repo_update/)로 레포지토리를 업데이트합니다:
   ```shell
   helm repo update
   ```

2. 다음으로 [`helm upgrade`](https://helm.sh/docs/helm/helm_upgrade/)로 Helm chart를 업데이트합니다:
   ```shell
   helm upgrade operator wandb/operator -n wandb-cr --reuse-values
   ```

<div id="update-the-wb-server-application">
  ## W&amp;B Server 애플리케이션 업데이트
</div>

W&amp;B Kubernetes operator를 사용하는 경우 더 이상 W&amp;B Server 애플리케이션을 직접 업데이트할 필요가 없습니다.

새 버전의 W&amp;B 소프트웨어가 릴리스되면 operator가 자동으로 W&amp;B Server 애플리케이션을 업데이트합니다.

<div id="migrate-to-operator-based-helm-chart">
  ### Operator 기반 Helm 차트로 마이그레이션
</div>

다음 단계에 따라 Operator 기반 Helm 차트로 마이그레이션합니다.

1. 현재 W&amp;B 설정을 가져옵니다. W&amp;B가 Operator 기반이 아닌 버전의 Helm 차트로 배포되었다면, 다음과 같이 values를 export합니다.
   ```shell
   helm get values wandb
   ```
   W&amp;B가 Kubernetes 매니페스트로 배포되었다면, 다음과 같이 values를 export합니다.
   ```shell
   kubectl get deployment wandb -o yaml
   ```
   이제 다음 단계에 필요한 모든 설정 값을 확보했습니다.

2. `operator.yaml`이라는 파일을 생성합니다. [Configuration Reference](#configuration-reference-for-wb-operator)에 설명된 형식을 따릅니다. 1단계에서 얻은 값을 사용합니다.

3. 현재 배포를 0개의 pod로 스케일 다운합니다. 이 단계에서 현재 배포를 중지합니다.
   ```shell
   kubectl scale --replicas=0 deployment wandb
   ```

4. Helm 차트 저장소를 업데이트합니다.
   ```shell
   helm repo update
   ```

5. 새 Helm 차트를 설치합니다.
   ```shell
   helm upgrade --install operator wandb/operator -n wandb-cr --create-namespace
   ```

6. 새 Helm 차트를 설정하고 W&amp;B 애플리케이션 배포를 트리거합니다. 새 설정을 적용합니다.
   ```shell
   kubectl apply -f operator.yaml
   ```
   배포가 완료되는 데 몇 분 정도 걸립니다.

7. 설치를 확인합니다. [Verify the installation](#verify-the-installation)의 단계를 따라 모든 것이 정상 동작하는지 확인합니다.

8. 이전 설치를 제거합니다. 이전 Helm 차트를 제거하거나 매니페스트로 생성된 리소스를 삭제합니다.

<div id="migrate-to-operator-based-terraform-helm-chart">
  ### Operator 기반 Terraform Helm 차트로 마이그레이션
</div>

다음 단계에 따라 Operator 기반 Helm 차트로 마이그레이션합니다:

1. Terraform 설정을 준비합니다. Terraform 설정에서 이전 배포에 사용하던 Terraform 코드를 [여기](#deploy-wb-with-helm-terraform-module)에 설명된 코드로 교체합니다. 이전과 동일한 변수 값을 사용합니다. .tfvars 파일이 있다면 변경하지 마세요.
2. Terraform 실행을 수행합니다. `terraform init`, `terraform plan`, `terraform apply`를 실행합니다.
3. 설치를 검증합니다. [설치 검증](#verify-the-installation)의 단계를 따라 모든 것이 정상적으로 동작하는지 확인합니다.
4. 이전 설치를 제거합니다. 이전 Helm 차트를 제거하거나, 매니페스트로 생성된 리소스를 삭제합니다.

<div id="configuration-reference-for-wb-server">
  ## W&amp;B Server 구성 참조
</div>

이 섹션에서는 W&amp;B Server 애플리케이션의 구성 옵션을 설명합니다. 애플리케이션은 [WeightsAndBiases](#how-it-works)라는 이름의 커스텀 리소스 정의(custom resource definition)를 통해 구성을 전달받습니다. 일부 구성 옵션은 아래의 구성으로 노출되며, 일부는 환경 변수로 설정해야 합니다.

이 문서에는 환경 변수 목록이 두 가지 있습니다: [기본](/ko/platform/hosting/env-vars/)과 [고급](/ko/platform/hosting/iam/advanced_env_vars/). 필요한 구성 옵션이 Helm 차트를 통해 노출되지 않는 경우에만 환경 변수를 사용하십시오.

<div id="basic-example">
  ### 기본 예시
</div>

이 예시는 W&amp;B에 필요한 최소한의 설정 값들을 정의합니다. 보다 현실적인 프로덕션 예시는 [완전한 예시](#complete-example)를 참고하세요.

이 YAML 파일은 버전, 환경 변수, 데이터베이스와 같은 외부 리소스, 그리고 기타 필요한 설정을 포함해, W&amp;B 배포의 원하는 상태를 정의합니다.

```yaml
apiVersion: apps.wandb.com/v1
kind: WeightsAndBiases
metadata:
  labels:
    app.kubernetes.io/name: weightsandbiases
    app.kubernetes.io/instance: wandb
  name: wandb
  namespace: default
spec:
  values:
    global:
      host: https://<HOST_URI>
      license: eyJhbGnUzaH...j9ZieKQ2x5GGfw
      bucket:
        <details depend on the provider>
      mysql:
        <redacted>
    ingress:
      annotations:
        <redacted>
```

전체 값 목록은 [W&amp;B Helm 차트 저장소](https://github.com/wandb/helm-charts/blob/main/charts/operator-wandb/values.yaml)에서 확인할 수 있습니다. **재정의가 필요한 값만 수정하세요**.

**기타 S3 호환 스토리지(Minio, Ceph 등)**

다른 S3 호환 스토리지를 사용하는 경우, 버킷을 다음과 같이 설정합니다:

```yaml
global:

    kmsKey: null
    path: wandb
    region: default
    accessKey: 5WOA500...P5DK7I
    secretKey: HDKYe4Q...JAp1YyjysnX
```

AWS 외부에 호스팅되는 S3 호환 스토리지를 사용하는 경우 `kmsKey`는 `null`이어야 합니다.

시크릿에서 `accessKey`와 `secretKey`를 참조하려면:

```yaml
global:

    # 예시 값입니다. 실제 값으로 교체하세요
    provider: s3
    name: storage.example.com
    kmsKey: null
    path: wandb
    region: default
    secret:
      secretName: bucket-secret
      accessKeyName: ACCESS_KEY
      secretKeyName: SECRET_KEY
```

<div id="ingress">
  ### 인그레스
</div>

Ingress 클래스를 확인하려면 이 FAQ [항목](#how-to-identify-the-kubernetes-ingress-class)을 참조하세요.

**TLS 미사용**

```yaml
global:
# 중요: Ingress는 YAML에서 'global'과 동일한 레벨에 있습니다 (하위 항목이 아님)
ingress:
  class: ""
```

**TLS 사용 시**

인증서를 포함하는 시크릿을 생성합니다

```console
kubectl create secret tls wandb-ingress-tls --key wandb-ingress-tls.key --cert wandb-ingress-tls.crt
```

Ingress 구성에서 해당 Secret을 참조하세요

```yaml
global:
# 중요: Ingress는 YAML에서 'global'과 동일한 레벨에 위치합니다 (하위 항목이 아님)
ingress:
  class: ""
  annotations:
    {}
    # kubernetes.io/ingress.class: nginx
    # kubernetes.io/tls-acme: "true"
  tls: 
    - secretName: wandb-ingress-tls
      hosts:
        - <HOST_URI>
```

Nginx를 사용하는 경우 다음 annotation을 추가해야 할 수 있습니다:

```
ingress:
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: 0
```

<div id="custom-kubernetes-serviceaccounts">

</div>

W&amp;B 파드를 실행할 사용자 지정 Kubernetes 서비스 계정을 지정합니다.

다음 스니펫은 배포의 일부로 지정한 이름의 서비스 계정을 생성합니다.

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: true

parquet:
  serviceAccount:
    name: custom-service-account
    create: true

global:
  ...
```

서브시스템 &quot;app&quot;과 &quot;parquet&quot;은(는) 지정된 서비스 계정에서 실행됩니다. 다른 서브시스템은 기본 서비스 계정에서 실행됩니다.

서비스 계정이 이미 클러스터에 존재하는 경우 `create: false`로 설정하세요:

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: false

parquet:
  serviceAccount:
    name: custom-service-account
    create: false
    
global:
  ...
```

app, parquet, console 등의 서로 다른 서브시스템별로 서비스 계정을 지정할 수 있습니다:

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: true

console:
  serviceAccount:
    name: custom-service-account
    create: true

global:
  ...
```

서비스 계정은 서브시스템마다 서로 다를 수 있습니다:

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: false

console:
  serviceAccount:
    name: another-custom-service-account
    create: true

global:
  ...
```

<div id="ldap">
  ### LDAP
</div>

<Warning>
  현재 Helm 차트에서는 LDAP 구성에 대한 지원이 제한적입니다. LDAP 구성을 위해 W&amp;B Support 또는 담당 AISE에 문의하세요.
</Warning>

LDAP를 구성하려면 `global.extraEnv`에 환경 변수를 설정하세요:

```yaml
global:
  extraEnv:
    LDAP_ADDRESS: ldaps://ldap.company.example.com
    LDAP_BASE_DN: cn=accounts,dc=company,dc=example,dc=com
    LDAP_USER_BASE_DN: cn=users,cn=accounts,dc=company,dc=example,dc=com
    LDAP_GROUP_BASE_DN: cn=groups,cn=accounts,dc=company,dc=example,dc=com
    LDAP_BIND_DN: uid=ldapbind,cn=sysaccounts,cn=etc,dc=company,dc=example,dc=com
    LDAP_BIND_PW: ********************
    LDAP_ATTRIBUTES: email=mail,name=cn
    LDAP_TLS_ENABLE: "true"
    LDAP_LOGIN: "true"
    LDAP_USER_OBJECT_CLASS: user
    LDAP_GROUP_OBJECT_CLASS: group
```

<accordion title="레거시 LDAP 구성">
  이 레거시 방식은 더 이상 권장되지 않습니다. 이 섹션은 참고용으로만 제공됩니다.

  **TLS 미사용**

  ```yaml
  global:
    ldap:
      enabled: true
      # "ldap://" 또는 "ldaps://"가 포함된 LDAP 서버 주소
      host:
      # 사용자를 찾기 위해 사용할 LDAP search base
      baseDN:
      # (anonymous bind를 사용하지 않는 경우) 바인드에 사용할 LDAP 사용자
      bindDN:
      # (anonymous bind를 사용하지 않는 경우) 바인드에 사용할 LDAP 비밀번호가 들어 있는 Secret 이름과 키
      bindPW:
      # 이메일 및 그룹 ID 속성 이름에 사용할 LDAP 속성으로, 쉼표로 구분된 문자열 값
      attributes:
      # 허용할 LDAP 그룹 목록
      groupAllowList:
      # LDAP TLS 활성화
      tls: false
  ```

  **TLS 사용**

  LDAP TLS 인증서 구성을 위해서는 인증서 내용을 포함하는 ConfigMap이 미리 생성되어 있어야 합니다.

  ConfigMap을 생성하려면 다음 명령을 사용할 수 있습니다:

  ```console
  kubectl create configmap ldap-tls-cert --from-file=certificate.crt
  ```

  그리고 아래 예시와 같이 YAML에서 해당 ConfigMap을 사용합니다.

  ```yaml
  global:
    ldap:
      enabled: true
      # "ldap://" 또는 "ldaps://"가 포함된 LDAP 서버 주소
      host:
      # 사용자를 찾기 위해 사용할 LDAP search base
      baseDN:
      # (anonymous bind를 사용하지 않는 경우) 바인드에 사용할 LDAP 사용자
      bindDN:
      # (anonymous bind를 사용하지 않는 경우) 바인드에 사용할 LDAP 비밀번호가 들어 있는 Secret 이름과 키
      bindPW:
      # 이메일 및 그룹 ID 속성 이름에 사용할 LDAP 속성으로, 쉼표로 구분된 문자열 값
      attributes:
      # 허용할 LDAP 그룹 목록
      groupAllowList:
      # LDAP TLS 활성화
      tls: true
      # LDAP 서버용 CA 인증서를 담고 있는 ConfigMap 이름과 키
      tlsCert:
        configMap:
          name: "ldap-tls-cert"
          key: "certificate.crt"
  ```
</accordion>

<div id="custom-security-context">
  ### 사용자 정의 보안 컨텍스트
</div>

각 W&amp;B 컴포넌트는 다음과 같은 형식의 사용자 정의 보안 컨텍스트 구성을 지원합니다.

```yaml
pod:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1001
    runAsGroup: 0
    fsGroup: 1001
    fsGroupChangePolicy: Always
    seccompProfile:
      type: RuntimeDefault
container:
  securityContext:
    capabilities:
      drop:
        - ALL
    readOnlyRootFilesystem: false
    allowPrivilegeEscalation: false 
```

<Note>
  `runAsGroup:`에 사용할 수 있는 유효한 값은 `0`뿐입니다. 다른 값을 사용하면 오류가 발생합니다.
</Note>

예를 들어 애플리케이션 파드를 구성하려면 설정에 `app` 섹션을 추가하세요.

```yaml
global:
  ...
app:
  pod:
    securityContext:
      runAsNonRoot: true
      runAsUser: 1001
      runAsGroup: 0
      fsGroup: 1001
      fsGroupChangePolicy: Always
      seccompProfile:
        type: RuntimeDefault
  container:
    securityContext:
      capabilities:
        drop:
          - ALL
      readOnlyRootFilesystem: false
      allowPrivilegeEscalation: false 
```

같은 개념이 `console`, `weave`, `weave-trace`, `parquet`에도 적용됩니다.

<div id="configuration-reference-for-wb-operator">
  ## W&amp;B Operator 구성 참조
</div>

이 섹션에서는 W&amp;B Kubernetes operator(`wandb-controller-manager`)에 대한 구성 옵션을 설명합니다. 구성은 YAML 파일 형태로 operator에 전달됩니다.

기본적으로 W&amp;B Kubernetes operator는 구성 파일이 필요하지 않습니다. 필요한 경우에만 구성 파일을 생성하십시오. 예를 들어 커스텀 인증 기관을 지정하거나 에어갭 환경에 배포해야 하는 경우 구성 파일이 필요할 수 있습니다.

스펙 커스터마이징 옵션의 전체 목록은 [Helm 저장소](https://github.com/wandb/helm-charts/blob/main/charts/operator/values.yaml)에서 확인할 수 있습니다.

<div id="what-is-the-purposerole-of-each-individual-pod">
  ### 각 개별 파드의 목적/역할은 무엇인가요?
</div>

* **`wandb-app`**: GraphQL API와 프런트엔드 애플리케이션을 포함하는 W&amp;B의 핵심 컴포넌트입니다. 플랫폼 기능의 대부분을 담당합니다.
* **`wandb-console`**: `/console` 경로를 통해 액세스하는 관리 콘솔입니다.
* **`wandb-otel`**: OpenTelemetry 에이전트로, Kubernetes 계층의 리소스에서 메트릭과 로그를 수집해 관리 콘솔에 표시합니다.
* **`wandb-prometheus`**: Prometheus 서버로, 다양한 컴포넌트에서 메트릭을 수집해 관리 콘솔에 표시합니다.
* **`wandb-parquet`**: 데이터베이스 데이터를 Parquet 형식으로 객체 스토리지에 내보내는, `wandb-app` 파드와 분리된 백엔드 마이크로서비스입니다.
* **`wandb-weave`**: UI에서 쿼리 테이블을 로드하고 여러 핵심 앱 기능을 지원하는 또 다른 백엔드 마이크로서비스입니다.
* **`wandb-weave-trace`**: LLM 기반 애플리케이션을 추적, 실험, 평가, 배포 및 개선하기 위한 프레임워크입니다. 이 프레임워크는 `wandb-app` 파드를 통해 액세스합니다.
