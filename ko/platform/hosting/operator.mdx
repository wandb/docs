<div id="wb-kubernetes-operator">
  ## W&amp;B Kubernetes Operator
</div>

W&amp;B Kubernetes Operator를 사용하면 Kubernetes에서 W&amp;B Server 인스턴스를 배포하고, 관리하고, 문제를 해결하며, 스케일링하는 작업을 간소화할 수 있습니다. 이 오퍼레이터를 W&amp;B 인스턴스를 위한 스마트한 보조 도구로 생각할 수 있습니다.

W&amp;B Server 아키텍처와 설계는 AI 개발 도구 기능을 확장하고, 고성능, 더 나은 확장성, 더 쉬운 관리를 위한 적절한 기본 구성 요소를 제공하기 위해 지속적으로 발전하고 있습니다. 이러한 발전은 컴퓨트 서비스, 관련 스토리지, 그리고 이들 간의 연결 전반에 적용됩니다. 다양한 배포 유형 전반에 걸쳐 지속적인 업데이트와 개선을 더 쉽게 하기 위해, W&amp;B는 Kubernetes 오퍼레이터를 사용합니다.

<Note>
  W&amp;B는 이 오퍼레이터를 사용해 AWS, Google Cloud, Azure 퍼블릭 클라우드에서 Dedicated Cloud 인스턴스를 배포하고 관리합니다.
</Note>

Kubernetes 오퍼레이터에 대한 자세한 내용은 Kubernetes 문서의 [Operator pattern](https://kubernetes.io/docs/concepts/extend-kubernetes/operator/)을 참조하세요.

<div id="reasons-for-the-architecture-shift">
  ### 아키텍처 전환의 이유
</div>

과거에는 W&amp;B 애플리케이션이 Kubernetes 클러스터 내의 단일 Deployment 및 Pod, 또는 단일 Docker 컨테이너로 배포되었습니다. W&amp;B는 과거부터 지금까지 데이터베이스와 오브젝트 스토어를 외부에 두는 것을 권장해 왔습니다. 데이터베이스와 오브젝트 스토어를 외부화하면 애플리케이션의 상태를 분리(디커플링)할 수 있습니다.

애플리케이션이 성장함에 따라 모놀리식 컨테이너에서 분산 시스템(마이크로서비스)으로 전환해야 할 필요성이 분명해졌습니다. 이러한 변경은 백엔드 로직 처리를 용이하게 하고, Kubernetes의 기본 제공 인프라 기능을 자연스럽게 도입할 수 있게 합니다. 분산 시스템은 또한 W&amp;B가 의존하는 추가 기능에 필수적인 신규 서비스를 배포하는 것도 지원합니다.

2024년 이전에는 Kubernetes 관련 변경이 있을 때마다 [terraform-kubernetes-wandb](https://github.com/wandb/terraform-kubernetes-wandb) Terraform 모듈을 수동으로 업데이트해야 했습니다. Terraform 모듈을 업데이트해야 클라우드 공급자 간 호환성을 보장하고, 필요한 Terraform 변수를 설정하며, 각 백엔드 또는 Kubernetes 수준의 변경에 대해 Terraform apply를 실행할 수 있습니다.

이 과정은 Terraform 모듈 업그레이드마다 W&amp;B Support가 각 고객을 도와야 했기 때문에 확장성이 떨어졌습니다.

이에 대한 해결책은 중앙 [deploy.wandb.ai](https://deploy.wandb.ai) 서버에 연결하는 오퍼레이터를 구현하는 것이었습니다. 이 오퍼레이터는 지정된 릴리스 채널에 대한 최신 사양 변경사항을 요청해 적용합니다. 라이선스가 유효한 동안 업데이트를 계속 받을 수 있습니다. [Helm](https://helm.sh/)은 W&amp;B 오퍼레이터의 배포 메커니즘이자, 오퍼레이터가 W&amp;B Kubernetes 스택의 모든 설정 템플릿을 처리하는 수단으로 사용됩니다. 일종의 Helm-ception입니다.

<div id="how-it-works">
  ### 작동 방식
</div>

이 operator는 Helm 또는 소스 코드에서 직접 설치할 수 있습니다. 자세한 지침은 [charts/operator](https://github.com/wandb/helm-charts/tree/main/charts/operator)를 참조하세요.

설치 과정에서는 `controller-manager`라는 deployment를 생성하고, `weightsandbiases.apps.wandb.com`(shortName: `wandb`)이라는 [custom resource](https://kubernetes.io/docs/concepts/extend-kubernetes/api-extension/custom-resources/) 정의를 사용합니다. 이 정의는 하나의 `spec`을 입력으로 받아 클러스터에 적용합니다:

```yaml
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: weightsandbiases.apps.wandb.com
```

`controller-manager`는 커스텀 리소스의 spec, 릴리스 채널, 그리고 사용자 정의 설정을 기반으로 [charts/operator-wandb](https://github.com/wandb/helm-charts/tree/main/charts/operator-wandb)를 설치합니다. 설정 명세 계층 구조를 통해 사용자는 설정을 최대한 유연하게 구성할 수 있으며, W&amp;B는 새로운 이미지, 설정, 기능, 그리고 Helm 업데이트를 자동으로 배포할 수 있습니다.

사용 가능한 설정 옵션은 [설정 명세 계층 구조](#configuration-specification-hierarchy) 및 [설정 참조](#configuration-reference-for-wb-operator)를 참조하세요.

이 배포는 서비스당 하나씩 여러 개의 파드(pod)로 구성됩니다. 각 파드의 이름에는 `wandb-` 접두사가 붙습니다.

<div id="configuration-specification-hierarchy">
  ### 설정 사양 계층 구조
</div>

설정 사양은 상위 수준 사양이 하위 수준 사양보다 우선 적용되는 계층형 모델을 따릅니다. 동작 방식은 다음과 같습니다:

* **Release Channel 값**: 이 기본 수준 설정은 W&amp;B가 배포에 대해 설정한 릴리스 채널을 기준으로 기본값과 설정을 지정합니다.
* **User Input 값**: 사용자는 System Console을 통해 Release Channel Spec에서 제공하는 기본 설정을 덮어쓸 수 있습니다.
* **Custom Resource 값**: 가장 높은 수준의 사양으로, 사용자로부터 제공됩니다. 여기에서 지정된 값은 User Input 및 Release Channel 사양보다 우선 적용됩니다. 설정 옵션에 대한 자세한 설명은 [Configuration Reference](#configuration-reference-for-wb-operator)를 참조하세요.

이 계층형 모델을 통해 설정은 다양한 요구사항을 충족할 수 있도록 유연하고 사용자 정의 가능하면서도, 업그레이드와 변경을 체계적이고 관리 가능한 방식으로 유지할 수 있습니다.

<div id="run-the-container-as-an-un-privileged-user">
  #### 비특권 사용자로 컨테이너 실행
</div>

기본적으로 컨테이너는 `$UID` 999로 실행됩니다. 오케스트레이터가 컨테이너를 non-root 사용자로 실행하도록 요구하는 경우 `$UID` &gt;= 100000 및 `$GID` 0을 지정합니다.

<Note>
  파일 시스템 권한이 정상적으로 동작하려면 W&amp;B는 루트 그룹(`$GID=0`)으로 시작해야 합니다.
</Note>

각 W&amp;B 컴포넌트에 대해 보안 컨텍스트(security context)를 구성합니다. 예를 들어, API 컴포넌트를 구성하려면 다음과 같이 합니다:

```yaml
api:
  install: true
  image:
    repository: wandb/megabinary
    tag: 0.74.1
  pod:
    securityContext:
      fsGroup: 10001
      fsGroupChangePolicy: Always
      runAsGroup: 0
      runAsNonRoot: true
      runAsUser: 10001
      seccompProfile:
        type: RuntimeDefault
  container:
    securityContext:
      allowPrivilegeEscalation: false
      capabilities:
        drop:
          - ALL
      privileged: false
      readOnlyRootFilesystem: false
```

필요한 경우 `app` 또는 `console`과 같은 다른 컴포넌트에 대해 사용자 지정 보안 컨텍스트를 구성하세요. 자세한 내용은 [Custom security context](#custom-security-context)를 참조하세요.

<div id="deploy-wb-server-application">
  ## W&amp;B Server 애플리케이션 배포
</div>

<Note>
  **Helm을 사용하는 W&amp;B Kubernetes Operator는** 클라우드, 온프레미스, 에어갭 환경을 포함한 모든 W&amp;B Self-Managed 배포에 대해 권장되는 설치 방법입니다.
</Note>

이 섹션에서는 W&amp;B Kubernetes Operator를 배포하는 다양한 방법을 설명합니다:

* **Helm CLI**: Helm 명령을 직접 사용한 배포
* **Helm Terraform Module**: 코드형 인프라(IaC) 방식의 배포
* **W&amp;B Cloud Terraform Modules**: AWS, Google Cloud, Azure용 인프라 + 애플리케이션 전체 배포

배포 방식별 고려 사항은 다음 문서도 참고하세요:

* 데이터센터/베어메탈 환경의 경우 [온프레미스에 W&amp;B Platform 배포](/ko/platform/hosting/self-managed/bare-metal/)
* 분리된 환경의 경우 [에어갭 인스턴스를 위한 Kubernetes operator](/ko/platform/hosting/self-managed/operator-airgapped/)

<div id="deploy-wb-with-helm-cli">
  ### Helm CLI로 W&amp;B 배포
</div>

W&amp;B는 Kubernetes 클러스터에 W&amp;B Kubernetes operator를 배포하기 위한 Helm 차트를 제공합니다. 이 방법을 사용하면 Helm CLI 또는 ArgoCD와 같은 지속적 배포 도구로 W&amp;B Server를 배포할 수 있습니다. 위에서 언급한 요구 사항이 충족되었는지 확인하세요.

Helm CLI로 W&amp;B Kubernetes Operator를 설치하려면 다음 단계를 따르세요:

1. W&amp;B Helm 저장소를 추가합니다. W&amp;B Helm 차트는 W&amp;B Helm 저장소에서 사용할 수 있습니다:
   ```shell
   helm repo add wandb https://charts.wandb.ai
   helm repo update
   ```

2. Kubernetes 클러스터에 Operator를 설치합니다:
   ```shell
   helm upgrade --install operator wandb/operator -n wandb-cr --create-namespace
   ```

3. W&amp;B operator 커스텀 리소스를 설정해 W&amp;B Server 설치가 시작되도록 합니다. `operator.yaml`이라는 파일을 생성하고 W&amp;B 배포 설정을 정의합니다. 사용 가능한 모든 옵션은 [Configuration Reference](#configuration-reference-for-wb-server)를 참조하세요.

   최소 예시 설정은 다음과 같습니다:

   ```yaml
   apiVersion: apps.wandb.com/v1
   kind: WeightsAndBiases
   metadata:
     labels:
       app.kubernetes.io/name: weightsandbiases
       app.kubernetes.io/instance: wandb
     name: wandb
     namespace: default
   spec:
     values:
       global:
         host: https://<HOST_URI>
         license: eyJhbGnUzaH...j9ZieKQ2x5GGfw
         bucket:
           <details depend on the provider>
         mysql:
           <redacted>
       ingress:
         annotations:
           <redacted>
   ```

4. 커스텀 설정으로 Operator를 시작하여 W&amp;B Server 애플리케이션을 설치, 설정 및 관리할 수 있도록 합니다:

   ```shell
   kubectl apply -f operator.yaml
   ```

   배포가 완료될 때까지 기다립니다. 몇 분 정도 소요됩니다.

5. 웹 UI를 사용해 설치를 검증하려면 먼저 첫 번째 관리자 계정을 생성한 다음, [Verify the installation](#verify-the-installation)에 설명된 검증 단계를 따르세요.

<div id="deploy-wb-with-helm-terraform-module">
  ### Helm Terraform Module로 W&amp;B 배포
</div>

이 방법은 Terraform의 인프라를 코드로 관리하는 방식(IaC)을 활용하여 일관성과 재현성을 확보하면서, 특정 요구 사항에 맞게 배포를 커스터마이징할 수 있습니다. 공식 W&amp;B Helm 기반 Terraform Module은 [여기](https://github.com/wandb/helm-charts/tree/main/charts/operator-wandb)에 있습니다.

다음 코드는 시작점으로 사용할 수 있으며, 프로덕션 환경급 배포에 필요한 모든 설정 옵션을 포함합니다.

```hcl
module "wandb" {
  source  = "wandb/wandb/helm"

  spec = {
    values = {
      global = {
        host    = "https://<HOST_URI>"
        license = "eyJhbGnUzaH...j9ZieKQ2x5GGfw"

        bucket = {
          <details depend on the provider>
        }

        mysql = {
          <redacted>
        }
      }

      ingress = {
        annotations = {
          "a" = "b"
          "x" = "y"
        }
      }
    }
  }
}
```

설정 옵션은 [Configuration Reference](#configuration-reference-for-wb-operator)에 설명된 것과 동일하지만, 구문은 HashiCorp Configuration Language(HCL)를 따라야 합니다. Terraform 모듈은 W&amp;B 사용자 지정 리소스 정의(CRD)를 생성합니다.

Weights &amp; Biases가 고객용 “Dedicated Cloud” 설치를 배포하는 데 Helm Terraform 모듈을 어떻게 사용하는지 확인하려면 다음 링크를 참조하십시오:

* [AWS](https://github.com/wandb/terraform-aws-wandb/blob/45e1d746f53e78e73e68f911a1f8cad5408e74b6/main.tf#L225)
* [Azure](https://github.com/wandb/terraform-azurerm-wandb/blob/170e03136b6b6fc758102d59dacda99768854045/main.tf#L155)
* [Google Cloud](https://github.com/wandb/terraform-google-wandb/blob/49ddc3383df4cefc04337a2ae784f57ce2a2c699/main.tf#L189)

<div id="deploy-wb-with-wb-cloud-terraform-modules">
  ### W&amp;B Cloud Terraform 모듈로 W&amp;B 배포
</div>

W&amp;B는 AWS, Google Cloud, Azure용 Terraform 모듈 세트를 제공합니다. 이 모듈들은 Kubernetes 클러스터, 로드 밸런서, MySQL 데이터베이스 등 전체 인프라와 W&amp;B Server 애플리케이션을 함께 배포합니다. W&amp;B Kubernetes Operator에는 아래 버전의 공식 W&amp;B 클라우드별 Terraform 모듈이 이미 포함되어 있습니다:

| Terraform Registry                                                  | Source Code                                      | Version |
| ------------------------------------------------------------------- | ------------------------------------------------ | ------- |
| [AWS](https://registry.terraform.io/modules/wandb/wandb/aws/latest) | https://github.com/wandb/terraform-aws-wandb     | v4.0.0+ |
| [Azure](https://github.com/wandb/terraform-azurerm-wandb)           | https://github.com/wandb/terraform-azurerm-wandb | v2.0.0+ |
| [Google Cloud](https://github.com/wandb/terraform-google-wandb)              | https://github.com/wandb/terraform-google-wandb  | v2.0.0+ |

이 인테그레이션을 통해 W&amp;B Kubernetes Operator는 최소한의 설정만으로도 인스턴스에서 바로 사용할 수 있으며, 클라우드 환경에서 W&amp;B Server를 배포하고 관리할 수 있는 간편한 워크플로를 제공합니다.

이 모듈들을 사용하는 방법에 대한 자세한 설명은 문서의 [Self-Managed installations 섹션](/ko/platform/hosting/hosting-options/self-managed#deploy-wb-server-within-self-managed-cloud-accounts)을 참조하세요.

<div id="access-the-wb-management-console">
  ## W&amp;B Management Console에 액세스하기

W&amp;B Kubernetes operator에는 management console(관리 콘솔)이 포함되어 있습니다. `${HOST_URI}/console` 경로에서 사용할 수 있으며, 예를 들어 `https://wandb.company-name.com/console`와 같습니다.

management console에 로그인하는 방법은 두 가지가 있습니다:

<Tabs>
  <Tab title="옵션 1 (권장)">
    1. 브라우저에서 W&amp;B 애플리케이션을 열고 로그인합니다. `${HOST_URI}/`(예: `https://wandb.company-name.com/`) 주소로 W&amp;B 애플리케이션에 로그인합니다.
    2. 콘솔에 액세스합니다. 오른쪽 상단 모서리의 아이콘을 클릭한 다음 **System console**을 클릭합니다. 관리자 권한이 있는 사용자만 **System console** 항목을 볼 수 있습니다.

       <Frame>
         <img src="/images/hosting/access_system_console_via_main_app.png" alt="System console 접근" />
       </Frame>
  </Tab>

  <Tab title="옵션 2">
    <Note>
      W&amp;B는 옵션 1이 작동하지 않을 때에만 다음 단계로 콘솔에 액세스할 것을 권장합니다.
    </Note>

    1. 브라우저에서 콘솔 애플리케이션을 엽니다. 위에서 설명한 URL을 열면 로그인 화면으로 리디렉션됩니다:

       비밀번호를 복사합니다.

<div id="update-the-wb-kubernetes-operator">
  ## W&amp;B Kubernetes operator 업데이트
</div>

이 섹션에서는 W&amp;B Kubernetes operator를 업데이트하는 방법을 설명합니다.

<Note>
  * W&amp;B Kubernetes operator를 업데이트해도 W&amp;B server 애플리케이션은 업데이트되지 않습니다.
  * Helm 차트에서 W&amp;B Kubernetes operator를 사용하지 않고 있다면, W&amp;B operator를 업데이트하기 위한 다음 안내를 따르기 전에 [여기](#migrate-Self-Managed-instances-to-wb-operator)의 지침을 먼저 확인하세요.
</Note>

아래 코드 스니펫을 복사해 터미널에 붙여넣으세요.

1. 먼저, [`helm repo update`](https://helm.sh/docs/helm/helm_repo_update/)로 리포지토리를 업데이트합니다:
   ```shell
   helm repo update
   ```

2. 다음으로, [`helm upgrade`](https://helm.sh/docs/helm/helm_upgrade/)를 사용해 Helm 차트를 업데이트합니다:
   ```shell
   helm upgrade operator wandb/operator -n wandb-cr --reuse-values
   ```

<div id="migrate-to-operator-based-helm-chart">
  ### Operator 기반 Helm 차트로 마이그레이션
</div>

다음 단계에 따라 Operator 기반 Helm 차트로 마이그레이션합니다:

1. 현재 W&amp;B 설정을 가져옵니다. W&amp;B가 Operator 기반이 아닌 버전의 Helm 차트로 배포되었다면, 다음과 같이 values를 export 합니다:
   ```shell
   helm get values wandb
   ```
   W&amp;B가 Kubernetes 매니페스트로 배포되었다면, 다음과 같이 values를 export 합니다:
   ```shell
   kubectl get deployment wandb -o yaml
   ```
   이제 다음 단계에 필요한 모든 설정 값을 확보했습니다.

2. `operator.yaml`이라는 파일을 생성합니다. [Configuration Reference](#configuration-reference-for-wb-operator)에 설명된 형식을 따릅니다. 1단계에서 확보한 값을 사용합니다.

3. 현재 디플로이먼트를 파드 수 0으로 스케일합니다. 이 단계에서 현재 디플로이먼트를 중지합니다.
   ```shell
   kubectl scale --replicas=0 deployment wandb
   ```

4. Helm 차트 리포지토리를 업데이트합니다:
   ```shell
   helm repo update
   ```

5. 새로운 Helm 차트를 설치합니다:
   ```shell
   helm upgrade --install operator wandb/operator -n wandb-cr --create-namespace
   ```

6. 새로운 Helm 차트를 설정하고 W&amp;B 애플리케이션 디플로이먼트를 트리거합니다. 새 설정을 적용합니다.
   ```shell
   kubectl apply -f operator.yaml
   ```
   디플로이먼트가 완료되는 데 몇 분 정도 걸립니다.

7. 설치를 검증합니다. [설치 검증](#verify-the-installation) 섹션의 단계를 따라 모든 것이 정상 동작하는지 확인합니다.

8. 이전 설치를 제거합니다. 기존 Helm 차트를 언인스톨하거나 매니페스트로 생성된 리소스를 삭제합니다.

<div id="migrate-to-operator-based-terraform-helm-chart">
  ### Operator 기반 Terraform Helm 차트로 마이그레이션
</div>

다음 단계에 따라 Operator 기반 Helm 차트로 마이그레이션합니다.

1. Terraform 구성을 준비합니다. Terraform 구성에서 기존 배포에 사용하던 Terraform 코드를 [여기](#deploy-wb-with-helm-terraform-module)에 설명된 코드로 교체합니다. 이전과 동일한 변수를 설정합니다. .tfvars 파일이 있다면 수정하지 마세요.
2. Terraform 작업을 실행합니다. `terraform init`, `plan`, `apply`를 실행합니다.
3. 설치를 확인합니다. [Verify the installation](#verify-the-installation)에 나와 있는 단계를 따라 모든 것이 정상적으로 동작하는지 확인합니다.
4. 기존 설치를 제거합니다. 기존 Helm 차트를 제거하거나 매니페스트로 생성한 리소스를 삭제합니다.

<div id="configuration-reference-for-wb-server">
  ## W&amp;B Server 설정 참조
</div>

이 섹션에서는 W&amp;B Server 애플리케이션의 설정 옵션을 설명합니다. 애플리케이션은 [WeightsAndBiases](#how-it-works)라는 커스텀 리소스 정의(CRD)를 통해 설정을 전달받습니다. 일부 설정 옵션은 아래 설정에서 직접 노출되며, 일부는 환경 변수로 설정해야 합니다.

문서에는 환경 변수가 [기본](/ko/platform/hosting/env-vars/) 및 [고급](/ko/platform/hosting/iam/advanced_env_vars/) 두 가지 목록으로 정리되어 있습니다. 필요한 설정 옵션이 Helm 차트를 통해 노출되지 않는 경우에만 환경 변수를 사용하십시오.

<div id="basic-example">
  ### 기본 예시
</div>

이 예시는 W&amp;B에 필요한 최소 설정 값을 정의합니다. 보다 현실적인 운영 환경 예시는 [완전한 예시](#complete-example)를 참조하세요.

이 YAML 파일은 버전, 환경 변수, 데이터베이스와 같은 외부 리소스 및 기타 필요한 설정을 포함해 W&amp;B 배포의 의도한 상태를 정의합니다.

```yaml
apiVersion: apps.wandb.com/v1
kind: WeightsAndBiases
metadata:
  labels:
    app.kubernetes.io/name: weightsandbiases
    app.kubernetes.io/instance: wandb
  name: wandb
  namespace: default
spec:
  values:
    global:
      host: https://<HOST_URI>
      license: eyJhbGnUzaH...j9ZieKQ2x5GGfw
      bucket:
        <details depend on the provider>
      mysql:
        <redacted>
    ingress:
      annotations:
        <redacted>
```

전체 설정 값 목록은 [W&amp;B Helm repository](https://github.com/wandb/helm-charts/blob/main/charts/operator-wandb/values.yaml)에서 확인할 수 있습니다. **재정의가 필요한 값만 변경하세요.**

```yaml
    name: storage.example.com
    kmsKey: null
    path: wandb
    region: default
    secret:
      secretName: bucket-secret
      accessKeyName: ACCESS_KEY
      secretKeyName: SECRET_KEY
```

<div id="ingress">
  ### Ingress
</div>

Ingress 클래스를 확인하려면 이 FAQ [항목](#how-to-identify-the-kubernetes-ingress-class)을 참조하세요.

**TLS 없이**

```yaml
global:
# 중요: Ingress는 YAML에서 'global'과 동일한 레벨에 있습니다 (하위 항목이 아님)
ingress:
  class: ""
```

**TLS 사용 시**

인증서가 포함된 시크릿을 생성합니다

```console
kubectl create secret tls wandb-ingress-tls --key wandb-ingress-tls.key --cert wandb-ingress-tls.crt
```

Ingress 설정에서 해당 Secret을 참조하세요

```yaml
global:
# 중요: Ingress는 YAML에서 'global'과 동일한 수준에 있습니다 (하위 항목이 아님)
ingress:
  class: ""
  annotations:
    {}
    # kubernetes.io/ingress.class: nginx
    # kubernetes.io/tls-acme: "true"
  tls: 
    - secretName: wandb-ingress-tls
      hosts:
        - <HOST_URI>
```

Nginx를 사용하는 경우 다음 애노테이션을 추가해야 할 수 있습니다:

```
ingress:
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: 0
```

  ### 사용자 지정 Kubernetes ServiceAccounts
</div>

W&amp;B 파드를 실행할 사용자 지정 Kubernetes 서비스 어카운트를 지정합니다.

다음 스니펫은 배포 시 지정한 이름으로 생성되는 서비스 어카운트를 정의합니다:

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: true

parquet:
  serviceAccount:
    name: custom-service-account
    create: true

global:
  ...
```

하위 시스템 &quot;app&quot;과 &quot;parquet&quot;는 지정된 서비스 계정에서 실행됩니다. 다른 하위 시스템은 기본 서비스 계정에서 실행됩니다.

서비스 계정이 이미 클러스터에 존재하는 경우 `create: false`로 설정하십시오:

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: false

parquet:
  serviceAccount:
    name: custom-service-account
    create: false
    
global:
  ...
```

app, parquet, console 등 서로 다른 서브시스템에 서비스 계정을 지정할 수 있습니다:

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: true

console:
  serviceAccount:
    name: custom-service-account
    create: true

global:
  ...
```

각 서브시스템마다 서비스 계정이 서로 다를 수 있습니다:

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: false

console:
  serviceAccount:
    name: another-custom-service-account
    create: true

global:
  ...
```

<div id="ldap">
  ### LDAP
</div>

<Warning>
  현재 Helm 차트에서 LDAP 설정 지원은 제한적입니다. LDAP 설정에 도움이 필요하면 W&amp;B Support 또는 AISE에 문의하십시오.
</Warning>

`global.extraEnv`에 환경 변수를 설정하여 LDAP를 설정합니다:

```yaml
global:
  extraEnv:
    LDAP_ADDRESS: ldaps://ldap.company.example.com
    LDAP_BASE_DN: cn=accounts,dc=company,dc=example,dc=com
    LDAP_USER_BASE_DN: cn=users,cn=accounts,dc=company,dc=example,dc=com
    LDAP_GROUP_BASE_DN: cn=groups,cn=accounts,dc=company,dc=example,dc=com
    LDAP_BIND_DN: uid=ldapbind,cn=sysaccounts,cn=etc,dc=company,dc=example,dc=com
    LDAP_BIND_PW: ********************
    LDAP_ATTRIBUTES: email=mail,name=cn
    LDAP_TLS_ENABLE: "true"
    LDAP_LOGIN: "true"
    LDAP_USER_OBJECT_CLASS: user
    LDAP_GROUP_OBJECT_CLASS: group
```

<accordion title="레거시 LDAP 설정">
  이 레거시 방식은 더 이상 권장되지 않습니다. 이 섹션은 참고용으로만 제공됩니다.

  **TLS 없이**

  ```yaml
  global:
    ldap:
      enabled: true
      # "ldap://" 또는 "ldaps://"를 포함한 LDAP 서버 주소
      host:
      # 사용자를 찾기 위한 LDAP 검색 base DN
      baseDN:
      # (anonymous bind를 사용하지 않는 경우) bind에 사용할 LDAP 사용자
      bindDN:
      # (anonymous bind를 사용하지 않는 경우) bind에 사용할 LDAP 비밀번호가 들어 있는 Secret 이름과 키
      bindPW:
      # 이메일 및 그룹 ID 속성 이름에 해당하는 LDAP 속성(쉼표로 구분된 문자열 값)
      attributes:
      # LDAP 그룹 허용 목록
      groupAllowList:
      # LDAP TLS 활성화
      tls: false
  ```

  **TLS 사용**

  LDAP TLS 인증서를 설정하려면 인증서 내용이 들어 있는 ConfigMap을 미리 생성해야 합니다.

  ConfigMap을 생성하려면 다음 명령어를 사용할 수 있습니다:

  ```console
  kubectl create configmap ldap-tls-cert --from-file=certificate.crt
  ```

  그리고 아래 예시처럼 YAML에서 해당 ConfigMap을 사용합니다.

  ```yaml
  global:
    ldap:
      enabled: true
      # "ldap://" 또는 "ldaps://"를 포함한 LDAP 서버 주소
      host:
      # 사용자를 찾기 위한 LDAP 검색 base DN
      baseDN:
      # (anonymous bind를 사용하지 않는 경우) bind에 사용할 LDAP 사용자
      bindDN:
      # (anonymous bind를 사용하지 않는 경우) bind에 사용할 LDAP 비밀번호가 들어 있는 Secret 이름과 키
      bindPW:
      # 이메일 및 그룹 ID 속성 이름에 해당하는 LDAP 속성(쉼표로 구분된 문자열 값)
      attributes:
      # LDAP 그룹 허용 목록
      groupAllowList:
      # LDAP TLS 활성화
      tls: true
      # LDAP 서버용 CA 인증서가 들어 있는 ConfigMap 이름과 키
      tlsCert:
        configMap:
          name: "ldap-tls-cert"
          key: "certificate.crt"
  ```
</accordion>

<div id="custom-security-context">
  ### 사용자 지정 보안 컨텍스트
</div>

각 W&amp;B 컴포넌트는 다음 형식의 사용자 지정 보안 컨텍스트 설정을 지원합니다:

```yaml
pod:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1001
    runAsGroup: 0
    fsGroup: 1001
    fsGroupChangePolicy: Always
    seccompProfile:
      type: RuntimeDefault
container:
  securityContext:
    capabilities:
      drop:
        - ALL
    readOnlyRootFilesystem: false
    allowPrivilegeEscalation: false 
```

<Note>
  `runAsGroup:`에 사용할 수 있는 유일한 유효한 값은 `0`입니다. 다른 값을 사용하면 오류가 발생합니다.
</Note>

예를 들어 애플리케이션 파드를 설정하려면 설정에 `app` 섹션을 추가하세요.

```yaml
global:
  ...
app:
  pod:
    securityContext:
      runAsNonRoot: true
      runAsUser: 1001
      runAsGroup: 0
      fsGroup: 1001
      fsGroupChangePolicy: Always
      seccompProfile:
        type: RuntimeDefault
  container:
    securityContext:
      capabilities:
        drop:
          - ALL
      readOnlyRootFilesystem: false
      allowPrivilegeEscalation: false 
```

동일한 개념이 `console`, `weave`, `weave-trace`, `parquet`에도 마찬가지로 적용됩니다.

<div id="configuration-reference-for-wb-operator">
  ## W&amp;B Operator 설정 참조
</div>

이 섹션에서는 W&amp;B Kubernetes 오퍼레이터(`wandb-controller-manager`)의 설정 옵션을 설명합니다. 오퍼레이터는 YAML 파일 형태로 설정을 입력받습니다.

기본적으로 W&amp;B Kubernetes 오퍼레이터에는 별도의 설정 파일이 필요하지 않습니다. 필요한 경우에만 설정 파일을 생성하십시오. 예를 들어, 사용자 정의 인증 기관(Certificate Authority)을 지정하거나, 에어갭(air-gapped) 환경에 배포하는 등의 경우 설정 파일이 필요할 수 있습니다.

spec 커스터마이징에 대한 전체 목록은 [Helm 리포지토리](https://github.com/wandb/helm-charts/blob/main/charts/operator/values.yaml)에서 확인하십시오.

<div id="what-is-the-purposerole-of-each-individual-pod">
  ### 각 개별 파드의 목적/역할은 무엇인가요?
</div>

* **`wandb-app`**: GraphQL API와 프론트엔드 애플리케이션을 포함하는 W&amp;B의 핵심 컴포넌트입니다. 플랫폼 기능의 대부분을 담당합니다.
* **`wandb-console`**: `/console`을 통해 액세스하는 관리 콘솔입니다.
* **`wandb-otel`**: OpenTelemetry 에이전트로, Kubernetes 계층의 리소스에서 메트릭과 로그를 수집하여 관리 콘솔에 표시합니다.
* **`wandb-prometheus`**: Prometheus 서버로, 다양한 컴포넌트에서 메트릭을 수집하여 관리 콘솔에 표시합니다.
* **`wandb-parquet`**: 데이터베이스 데이터를 Parquet 포맷으로 오브젝트 스토리지에 내보내는, `wandb-app` 파드와 분리된 백엔드 마이크로서비스입니다.
* **`wandb-weave`**: UI에서 쿼리 테이블을 로드하고 다양한 코어 앱 기능을 지원하는 또 다른 백엔드 마이크로서비스입니다.
* **`wandb-weave-trace`**: LLM 기반 애플리케이션을 추적, 실험, 평가, 배포 및 개선하기 위한 프레임워크입니다. 이 프레임워크에는 `wandb-app` 파드를 통해 액세스합니다.
