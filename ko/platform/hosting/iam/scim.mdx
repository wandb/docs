## 개요

SCIM(System for Cross-domain Identity Management) API를 사용하면 인스턴스 또는 조직 관리자가 W\&B 조직의 사용자, 그룹 및 커스텀 역할을 관리할 수 있습니다. SCIM 그룹은 W\&B Teams에 매핑됩니다.

W\&B의 SCIM API는 Okta를 포함한 주요 ID 공급자와 호환되며, 자동화된 사용자 프로비저닝 및 디프로비저닝을 지원합니다. Okta 및 기타 ID 공급자를 사용한 SSO 구성에 대해서는 [SSO 문서](/ko/platform/hosting/iam/sso)를 참조하세요.

SCIM API와 상호작용하는 방법을 보여주는 실용적인 Python 예제는 저희 [`wandb-scim`](https://github.com/wandb/examples/tree/master/wandb-scim) 저장소를 방문하세요.

### 지원되는 기능

* **필터링**: API는 `/Users` 및 `/Groups` 엔드포인트에 대한 필터링을 지원합니다
* **PATCH 작업**: 부분 리소스 업데이트를 위한 PATCH를 지원합니다
* **ETag 지원**: 충돌 감지를 위한 ETag를 사용한 조건부 업데이트
* **서비스 계정 인증**: 조직 서비스 계정이 API에 액세스할 수 있습니다

<Note>
  여러 Enterprise [Multi-tenant SaaS](/ko/platform/hosting/hosting-options/multi_tenant_cloud) 조직의 관리자인 경우, API 키를 사용하여 전송된 SCIM API 요청이 올바른 조직에 영향을 미치도록 SCIM API 요청이 전송될 조직을 구성해야 합니다. 프로필 이미지를 클릭한 다음 **사용자 설정**을 클릭하고 **기본 API 조직** 설정을 확인하세요.

  선택한 호스팅 옵션에 따라 이 페이지의 예제에서 사용되는 `<host-url>` 플레이스홀더의 값이 결정됩니다.

  또한 예제에서는 `abc` 및 `def`와 같은 사용자 ID를 사용합니다. 실제 요청 및 응답에서는 사용자 ID에 해시된 값이 사용됩니다.
</Note>

## 인증

조직 관리자는 **Bearer 토큰** 또는 **HTTP Basic** 자격 증명으로 인증할 수 있습니다. 두 방식 모두 키가 적용되는 곳에서 *동일한 API 키 문자열*을 사용합니다. 주요 차이점을 검토한 후 사용자 ID 또는 조직 범위의 서비스 계정을 선택하세요.

### 주요 차이점

* 사용 대상: 사용자는 대화형 일회성 관리 작업에 가장 적합하고, 서비스 계정은 자동화 및 통합(CI/CD, 프로비저닝 도구)에 가장 적합합니다.
* Credentials: Users send username and API key for Basic auth; service accounts send only an API key (no username) for Basic auth. For Bearer auth, send only the API key in the header (no username).
* Bearer 대 Basic: Bearer는 키를 그대로 사용하여 `Authorization: Bearer <API-KEY>`를 사용합니다. Basic은 `Authorization: Basic <base64(...)>`를 사용합니다(사용자는 `username:API-KEY`를 인코딩하고, 서비스 계정은 `:API-KEY`를 앞에 콜론과 빈 사용자 이름으로 인코딩합니다).
* 범위 및 권한: 인스턴스 또는 조직 관리자 사용자의 API 키 또는 [조직 범위의 서비스 계정](/ko/platform/hosting/iam/service-accounts#organization-scoped-service-accounts)의 키를 사용하세요. [팀 범위의 서비스 계정](/ko/platform/hosting/iam/service-accounts#team-scoped-service-accounts)의 키는 SCIM API에 인증할 수 없습니다. SCIM을 사용하는 서비스 계정은 조직 범위이며 헤드리스로, 자동화를 위한 명확한 감사 추적을 지원합니다.
* 자격 증명을 얻는 방법: 사용자는 사용자 설정에서 API 키를 복사하고, 조직 범위의 서비스 계정 키는 **서비스 계정** 탭 아래의 조직 대시보드에 있습니다.
* Multi-tenant Cloud: 둘 이상의 Multi-tenant Cloud 조직에 액세스할 수 있는 경우, SCIM API 호출이 의도한 조직으로 라우팅되도록 기본 API 조직을 설정해야 합니다.

### Bearer 토큰

API 키를 Bearer 토큰으로 전송합니다:

```bash
Authorization: Bearer <API-KEY>
```

`<API-KEY>` 값은 해당 주체의 HTTP Basic 인증에서 비밀번호로 사용하는 것과 동일한 문자열입니다. Bearer 요청에서는 키를 Base64로 인코딩하지 않습니다.

<Note>
  SCIM API에 대한 Bearer 인증은 W\&B Multi-tenant Cloud와 Dedicated Cloud 및 Self-Managed v0.79.0 이상에서 사용할 수 있습니다.
</Note>

다음 예제에서는 `<YOUR_API_KEY>`를 플레이스홀더로 사용합니다. 관리자 사용자 또는 조직 범위의 서비스 계정의 실제 키로 교체하세요.

**사용자 목록 조회**

```bash
curl -s -S \
  -H "Authorization: Bearer <YOUR_API_KEY>" \
  -H "Content-Type: application/scim+json" \
  "<host-url>/scim/Users"
```

**사용자 생성**

```bash
curl -s -S -X POST \
  -H "Authorization: Bearer <YOUR_API_KEY>" \
  -H "Content-Type: application/scim+json" \
  "<host-url>/scim/Users" \
  -d '{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
    "userName": "dev-user2",
    "emails": [{"primary": true, "value": "dev-user2@example.com"}]
  }'
```

자세한 내용은 [사용자 생성](#create-user)을 참조하세요.

### 사용자

대화형 관리 작업을 수행할 때는 개인 관리자 자격 증명을 사용하세요. HTTP `Authorization` 헤더를 `Basic <base64(username:API-KEY)>`로 구성하세요.

예를 들어, `demo:p@55w0rd`로 인증합니다:

```bash
Authorization: Basic ZGVtbzpwQDU1dzByZA==
```

### 서비스 계정

자동화 또는 통합에는 조직 범위의 서비스 계정을 사용하세요. HTTP `Authorization` 헤더를 `Basic <base64(:API-KEY)>`로 구성하세요(앞에 콜론과 빈 사용자 이름에 유의하세요). 서비스 계정 API 키는 **서비스 계정** 탭 아래의 조직 대시보드에서 찾을 수 있습니다. [조직 범위의 서비스 계정](/ko/platform/hosting/iam/service-accounts#organization-scoped-service-accounts)을 참조하세요.

예를 들어, API 키 `sa-p@55w0rd`로 인증합니다:

```bash
Authorization: Basic OnNhLXBANTV3MHJk
```

## 사용자 관리

SCIM 사용자 리소스는 W\&B 사용자에 매핑됩니다. 이 엔드포인트를 사용하여 조직의 사용자를 관리하세요.

### 사용자 조회

조직의 특정 사용자에 대한 정보를 검색합니다.

<Note>이 작업은 서비스 계정을 검색하지 않습니다.</Note>

#### 엔드포인트

* **URL**: `<host-url>/scim/Users/{id}`
* **메서드**: GET

#### 파라미터

| 파라미터 | 타입     | 필수 | 설명         |
| ---- | ------ | -- | ---------- |
| `id` | string | 예  | 사용자의 고유 ID |

    * **Dedicated Cloud / Self-Managed**: 사용자가 로그인하거나, W\&B App의 페이지를 열거나, run을 기록하거나, SDK를 사용하거나, 어떤 방식으로든 W\&B 서버와 상호작용하면 활성 상태입니다.

#### 사용자 필터링

The `/Users` 엔드포인트는 사용자 이름 또는 이메일로 사용자 필터링을 지원합니다:

* `userName eq "value"` - 사용자 이름으로 필터링
* `emails.value eq "value"` - 이메일 주소로 필터링

##### 예시

```bash
GET /scim/Users?filter=userName eq "john.doe"
GET /scim/Users?filter=emails.value eq "john@example.com"
```

### 사용자 생성

조직에 새 사용자를 생성합니다.

#### 엔드포인트

* **URL**: `<host-url>/scim/Users`
* **Method**: POST

#### 파라미터

| 파라미터         | 타입     | 필수 여부 | 설명                                                                                                  |
| ------------ | ------ | ----- | --------------------------------------------------------------------------------------------------- |
| `emails`     | array  | 예     | 이메일 객체의 배열. 기본 이메일을 포함해야 합니다                                                                        |
| `userName`   | string | 예     | 새 사용자의 사용자 이름                                                                                       |
| `modelsSeat` | string | 아니오   | Models 시트 레벨. `full`, `viewer`, 또는 `none` 중 하나. 기본값은 `full`. **Dedicated Cloud 및 Self-Managed 전용.** |
| `weaveRole`  | string | 아니오   | Weave 역할 레벨. `full`, `viewer`, 또는 `none` 중 하나. 기본값은 `full`. **Dedicated Cloud 및 Self-Managed 전용.**  |

#### 예시

<Tabs>
  <Tab title="Create User 요청 (Dedicated/Self-Managed)">
    ```bash
    POST /scim/Users
    ```

    ```json
    {
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User"
        ],
        "emails": [
            {
                "primary": true,
                "value": "dev-user2@example.com"
            }
        ],
        "userName": "dev-user2",
        "modelsSeat": "full",
        "weaveRole": "full"
    }
    ```
  </Tab>

  <Tab title="Create User 요청 (Multi-tenant)">
    ```bash
    POST /scim/Users
    ```

    ```json
    {
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User",

        ],
        "emails": [
            {
                "primary": true,
                "value": "dev-user2@example.com"
            }
        ],
        "userName": "dev-user2",
        "urn:ietf:params:scim:schemas:extension:teams:2.0:User": {
            "teams": ["my-team"]

    }
    ```

### 사용자 삭제

<Warning>
  **관리자 접근 권한 유지**

  인스턴스 또는 조직에 항상 최소 한 명의 관리자 사용자가 존재하도록 해야 합니다. 그렇지 않으면 어떤 사용자도 조직의 W\&B 계정을 구성하거나 유지 관리할 수 없게 됩니다. 조직이 SCIM 또는 다른 자동화된 프로세스를 사용하여 W\&B에서 사용자를 프로비저닝 해제하는 경우, 프로비저닝 해제 작업으로 인해 인스턴스 또는 조직에서 마지막 남은 관리자가 실수로 제거될 수 있습니다.

  운영 절차 개발 지원이 필요하거나 관리자 접근 권한을 복원하려면 [support](mailto:support@wandb.com)에 문의하세요.
</Warning>

조직에서 사용자를 완전히 삭제합니다.

<Note>이 작업은 사용자에게만 적용되며, 서비스 계정에는 적용되지 않습니다. 서비스 계정은 W\&B Team 설정에서 삭제하세요.</Note>

* **Method**: DELETE

| ---- | ------ | ----- | -------------- |
| `id` | string | 예     | 삭제할 사용자의 고유 ID |

### 조직 역할 할당

사용자에게 조직 수준의 역할을 할당합니다.

<Note>이 작업은 서비스 계정이 아닌 사용자에게만 적용됩니다. 커스텀 역할은 서비스 계정에 지원되지 않습니다.</Note>

#### Endpoint

* **URL**: `<host-url>/scim/Users/{id}`
* **Method**: PATCH

#### Parameters

| Parameter | Type   | Required | Description                 |
| --------- | ------ | -------- | --------------------------- |
| `id`      | string | Yes      | 사용자의 고유 ID                  |
| `op`      | string | Yes      | `replace`                   |
| `path`    | string | Yes      | `organizationRole`          |
| `value`   | string | Yes      | 역할 이름 (`admin` 또는 `member`) |

<Note>
  조직 범위의 `viewer` 역할은 더 이상 사용되지 않으며 UI에서 더 이상 할당할 수 없습니다. SCIM을 사용하여 `viewer` 역할을 사용자에게 할당하는 경우:

  * 해당 사용자는 조직에서 `member` 역할이 할당됩니다.
  * 해당 사용자의 `modelsSeat`은(는) `viewer` 대신 `full`으로 설정됩니다. 이를 통해 Models에 대한 보기 전용 액세스와 Registry에 대한 전체 액세스가 허용됩니다. Models 시트가 없는 경우 `Seat limit reached` 오류가 반환됩니다. 시트가 사용 가능한 경우 나중에 업데이트할 수 있습니다.
  * 해당 사용자의 `weaveRole`은(는) `viewer` 대신 `full`으로 설정됩니다. 이를 통해 Weave에 대한 보기 전용 액세스가 허용됩니다.
  * 해당 사용자의 기존 팀 및 프로젝트 역할이 모두 `viewer`으로 설정됩니다.
  * 해당 사용자는 조직 수준에서 표시되는 레지스트리에서 Registry `viewer` 역할이 할당됩니다.

  `member` 또는 `admin` 조직 역할을 할당해도 사용자의 `modelsSeat` 또는 `weaveRole`은(는) 변경되지 않습니다.
</Note>

#### Example

<Tabs>
  <Tab title="Assign Org Role Request">
    ```bash
    PATCH /scim/Users/abc
    ```

    ```json
    {
        "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
        "Operations": [
            {
                "op": "replace",
                "path": "organizationRole",
                "value": "admin"
            }
        ]
    }
    ```
  </Tab>

    ```bash
    (Status 200)
    ```

    ```json
            "location": "Users/abc"
    ```

### 팀 생성

* **Endpoint**: **`<host-url>/scim/Groups`**
* **Method**: POST
* **Description**: 새 팀 리소스를 생성합니다.
* **지원되는 필드**:

| Field           | Type               | Required                                |
| --------------- | ------------------ | --------------------------------------- |
| `displayName`   | String             | Yes                                     |
| `members`       | Multi-Valued Array | Yes (`value` 하위 필드는 필수이며 사용자 ID에 매핑됩니다) |
| `storageBucket` | Object             | No                                      |

팀 수준 [Bring your own bucket (BYOB)](/ko/platform/hosting/data-security/secure-storage-connector)을 팀 생성 시 `storageBucket` 객체를 포함하여 구성할 수 있습니다. 생략하면 팀은 기본 또는 인스턴스 수준 스토리지를 사용합니다. BYOB 가이드를 사용하여 버킷(정책, CORS, 자격 증명)을 프로비저닝하고 공급자별 스토리지 주소 형식을 결정하세요. `storageBucket` 객체에는 다음 하위 필드가 있습니다:

* **Required**: `name` (버킷 이름), `provider` (다음 중 하나: `COREWEAVE`, `AWS`, `AZURE`, `GCP`, 또는 `MINIO`). 값은 대소문자를 구분합니다. 표시된 대로 대문자를 사용하세요.
* **Optional**: `path` (버킷 내 경로 접두사), `kmsKeyId` (암호화용 KMS 키, 예: AWS), `awsExternalId` (AWS 크로스 계정 액세스), `azureTenantId` (Azure 테넌트 ID), `azureClientId` (Azure 관리 ID 클라이언트 ID).

W\&B는 팀을 생성하기 전에 버킷이 존재하고 접근 가능한지 확인합니다. 유효성 검사에 실패하면 SCIM 요청이 실패하고 팀이 생성되지 않습니다.

잘못된 `provider` 값은 허용된 값 목록이 포함된 SCIM 오류와 함께 `400 Bad Request`을 반환합니다.

### 팀 업데이트

* **Endpoint**: **`<host-url>/scim/Groups/{id}`**
* **Method**: PATCH
* **Description**: 기존 팀의 멤버십 목록을 업데이트합니다.
* **Supported Operations**: `add` member, `remove` member, `replace` members

<Note>
  - remove 작업은 RFC 7644 SCIM 프로토콜 사양을 따릅니다. 특정 사용자를 제거하려면 filter 구문을 사용하고, `members[value eq "{user_id}"]` 또는 `members` 팀에서 모든 사용자를 제거합니다.

    **User Identification**: `{user_id}` member 작업에서 다음 중 하나일 수 있습니다:

    * W\&B 사용자 ID
    * 이메일 주소 (예: "[user@example.com](mailto:user@example.com)")
  - 이 작업은 서비스 계정이 아닌 사용자에게만 적용됩니다. W\&B Team 설정에서 팀의 서비스 계정을 업데이트하세요.
</Note>

<Info>
  Replace `{team_id}` 를 실제 팀 ID로, `{user_id}` 를 요청에서 실제 사용자 ID 또는 이메일 주소로 교체하세요.
</Info>

## Role 리소스

SCIM role 리소스는 W\&B 커스텀 역할에 매핑됩니다. 앞서 언급했듯이, `/Roles` 엔드포인트는 공식 SCIM 스키마의 일부가 아니며, W\&B는 W\&B 조직에서 커스텀 역할의 자동화된 관리를 지원하기 위해 `/Roles` 엔드포인트를 추가합니다.

### 커스텀 역할 가져오기

역할의 고유 ID를 제공하여 커스텀 역할에 대한 정보를 검색합니다.

#### Endpoint

* **URL**: `<host-url>/scim/Roles/{id}`
* **Method**: GET

#### Example

    ```bash
    GET /scim/Roles/abc
    ```

### 커스텀 역할 생성

* **Endpoint**: **`<host-url>/scim/Roles`**
* **Method**: POST
* **Description**: W\&B 조직에 새 커스텀 역할을 생성합니다.
* **Supported Fields**:

| Field           | Type         | Required                                                                                                                               |
| --------------- | ------------ | -------------------------------------------------------------------------------------------------------------------------------------- |
| `name`          | String       | 커스텀 역할의 이름                                                                                                                             |
| `description`   | String       | 커스텀 역할에 대한 설명                                                                                                                          |
| `permissions`   | Object array | 각 객체에 `name` 형식의 값을 가진 `w&bobject:operation` 문자열 필드가 포함된 권한 객체 배열입니다. 예를 들어, W\&B runs에 대한 삭제 작업의 권한 객체는 `name` 를 `run:delete` 로 가집니다. |
| `inheritedFrom` | String       | 커스텀 역할이 상속할 사전 정의된 역할입니다. `member` 또는 `viewer` 중 하나일 수 있습니다.                                                                           |

#### Example

<Tabs>
  <Tab title="Request">
    ```bash
    POST /scim/Roles
    ```

    ```json
    {
        "schemas": ["urn:ietf:params:scim:schemas:core:2.0:Role"],
        "name": "Sample custom role",
        "description": "A sample custom role for example",
        "permissions": [
            {
                "name": "project:update"
            }
        ],
        "inheritedFrom": "member"
    }
    ```
  </Tab>

  <Tab title="Response">
    ```bash
    (Status 201)
    ```

### 커스텀 역할 업데이트

#### 역할에 권한 추가

* **Endpoint**: **`<host-url>/scim/Roles/{id}`**
* **Method**: PATCH
* **Description**: 기존 커스텀 역할에 권한을 추가합니다.

<Tabs>
  <Tab title="Request">
    ```bash
    PATCH /scim/Roles/{role_id}
    ```

    ```json
    {
        "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
        "Operations": [
            {
                "op": "add",
                "path": "permissions",
                "value": [
                    {
                        "name": "project:delete"
                    },
                    {
                        "name": "run:stop"
                    }
                ]
            }
        ]
    }
    ```
  </Tab>

  <Tab title="Response">
    ```bash
    (Status 200)
    ```

    새 권한이 추가된 업데이트된 역할을 반환합니다.
  </Tab>
</Tabs>

### 커스텀 역할 삭제

W\&B 조직에서 커스텀 역할을 삭제합니다. **주의하여 사용하세요**. 커스텀 역할이 상속한 사전 정의된 역할이 이제 해당 작업 이전에 커스텀 역할을 할당받은 모든 사용자에게 할당됩니다.

#### Endpoint

* **URL**: `<host-url>/scim/Roles/{id}`
* **Method**: DELETE

    ```bash
    DELETE /scim/Roles/abc
    ```
  </Tab>

  <Tab title="Response">
    ```bash
    (Status 204 No Content)
    ```
  </Tab>
</Tabs>

## 고급 기능

### ETag 지원

SCIM API는 동시 수정 충돌을 방지하기 위한 조건부 업데이트에 ETags를 지원합니다. ETags는 `ETag` 응답 헤더와 `meta.version` 필드에 반환됩니다.

#### ETags

Etag를 사용하려면:

1. **현재 ETag 가져오기**: 리소스를 GET할 때 응답의 ETag 헤더를 확인하세요
2. **조건부 업데이트**: 업데이트 시 `If-Match` 헤더에 ETag를 포함하세요

#### 예시

```
# Get user and note ETag
GET /scim/Users/abc
# Response includes: ETag: W/"xyz123"

# Update with ETag
PATCH /scim/Users/abc
If-Match: W/"xyz123"

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
    "Operations": [
        {
            "op": "replace",
            "path": "organizationRole",
            "value": "admin"
        }
    ]
}
```

`412 Precondition Failed` 오류 응답은 리소스를 검색한 이후 수정되었음을 나타냅니다.

### 오류 처리

SCIM API는 표준 SCIM 오류 응답을 반환합니다:

| 상태 코드 | 설명                             |
| ----- | ------------------------------ |
| `200` | 성공                             |
| `201` | 생성됨                            |
| `204` | 내용 없음 (삭제 성공)                  |
| `400` | 잘못된 요청 - 유효하지 않은 매개변수 또는 요청 본문 |
| `401` | 인증되지 않음 - 인증 실패                |
| `403` | 금지됨 - 권한 부족                    |
| `404` | 찾을 수 없음 - 리소스가 존재하지 않음         |
| `409` | 충돌 - 리소스가 이미 존재함               |
| `412` | 전제 조건 실패 - ETag 불일치            |
| `500` | 내부 서버 오류                       |

### 배포 유형별 구현 차이

W\&B는 두 가지 별도의 SCIM API 구현을 유지하며, 기능은 구현 간에 차이가 있습니다:

| 기능                        | Multi-tenant Cloud | Dedicated Cloud 및 Self-Managed |
| ------------------------- | ------------------ | ------------------------------ |
| 사용자 이메일 업데이트              | -                  | ✓                              |
| 사용자 표시 이름 업데이트            | -                  | ✓                              |
| 사용자 비활성화                  | ✓                  | ✓                              |
| 사용자 재활성화                  | -                  | ✓                              |
| 사용자당 여러 이메일               | ✓                  | -                              |
| 생성/업데이트 시 `modelsSeat` 설정 | -                  | ✓                              |
| 생성/업데이트 시 `weaveRole` 설정  | -                  | ✓                              |

## 제한 사항

* **최대 결과**: 요청당 9999개 항목.
* **단일 테넌트 환경**: 사용자당 하나의 이메일만 지원합니다.
* **팀 삭제**: SCIM을 통해 지원되지 않습니다 (W\&B 웹 인터페이스를 사용하세요).
* **사용자 재활성화**: Multi-tenant Cloud 환경에서는 지원되지 않습니다.
* **시트 제한**: 조직의 시트 제한에 도달하면 작업이 실패할 수 있습니다.
