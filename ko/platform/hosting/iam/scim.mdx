<div id="overview">
  ## Overview
</div>

SCIM(System for Cross-domain Identity Management) API를 사용하면 인스턴스 또는 조직 관리자가 W&amp;B 조직에서 사용자, 그룹, 커스텀 역할을 관리할 수 있습니다. SCIM 그룹은 W&amp;B Teams에 매핑됩니다.

W&amp;B의 SCIM API는 Okta를 비롯한 주요 ID 공급자와 호환되므로 사용자 프로비저닝 및 디프로비저닝을 자동화할 수 있습니다. Okta 및 기타 ID 공급자에 대한 SSO 설정은 [SSO documentation](/ko/platform/hosting/iam/sso/)을 참조하세요.

SCIM API와 상호작용하는 방법을 보여주는 실용적인 Python 예시는 [`wandb-scim`](https://github.com/wandb/examples/tree/master/wandb-scim) 저장소를 참조하세요.

<div id="supported-features">
  ### 지원되는 특성
</div>

* **Filtering**: API는 `/Users` 및 `/Groups` 엔드포인트에 대한 필터링을 지원합니다.
* **PATCH Operations**: 리소스를 부분적으로 업데이트하기 위한 PATCH를 지원합니다.
* **ETag Support**: 충돌 감지를 위해 ETag를 사용한 조건부 업데이트를 지원합니다.
* **Service Account Authentication**: 조직 서비스 계정이 API에 액세스할 수 있습니다.

<Note>
  여러 Enterprise [Multi-tenant SaaS](/ko/platform/hosting/hosting-options/multi_tenant_cloud) 조직의 관리자라면, SCIM API 요청을 보낼 조직을 반드시 먼저 지정해야 합니다. 그래야 API 키를 사용해 전송하는 SCIM API 요청이 올바른 조직에 적용됩니다. 프로필 이미지를 클릭한 다음 **User Settings**를 클릭하고 **Default API organization** 설정을 확인합니다.

  선택한 호스팅 옵션에 따라 이 페이지의 예시에서 사용하는 `<host-url>` 플레이스홀더 값이 결정됩니다.

  또한 예시에서는 `abc` 및 `def`와 같은 사용자 ID를 사용합니다. 실제 Request 및 Response에서는 사용자 ID에 대해 해시된 값이 사용됩니다.
</Note>

<div id="authentication">
  ## 인증
</div>

핵심 차이를 검토한 후, 사용자 ID 또는 서비스 계정을 사용해 인증 방식을 선택하세요.

<div id="key-differences">
  ### 주요 차이
</div>

* 사용 대상: Users는 대화형 일회성 관리자 액션에 가장 적합하고, service accounts는 자동화 및 인테그레이션(CI/CD, 프로비저닝 도구)에 가장 적합합니다.
* 자격 증명: Users는 사용자 이름과 API 키를 전송하고, service accounts는 API 키만 전송합니다(사용자 이름 없음).
* Authorization 헤더 페이로드: Users는 `username:API-KEY`를 인코딩하고, service accounts는 `:API-KEY`를 인코딩합니다(앞에 콜론 포함).
* 스코프 및 권한: 둘 다 관리자 권한이 필요합니다. service accounts는 조직 범위이며 헤드리스이므로, 자동화에 대해 더 명확한 감사 추적을 제공합니다.
* 자격 증명을 가져오는 위치: Users는 User Settings에서 API 키를 복사하고, service account 키는 조직의 Service account 탭에 있습니다.
* Multi-tenant Cloud: 둘 이상의 Multi-tenant Cloud 조직에 액세스할 수 있는 경우, SCIM API Call이 의도한 조직으로 라우팅되도록 Default API organization을 설정해야 합니다.

<div id="users">
  ### Users
</div>

대화형 관리 작업을 수행할 때는 개인 관리자 자격 증명을 사용하세요. HTTP `Authorization` 헤더를 `Basic <base64(username:API-KEY)>` 형식으로 구성하세요.

예를 들어, `demo:p@55w0rd`로 인증하려면 다음과 같습니다:

```bash
Authorization: Basic ZGVtbzpwQDU1dzByZA==
```

<div id="service-accounts">
  ### 서비스 계정
</div>

자동화 또는 인테그레이션에는 조직 범위 서비스 계정을 사용하세요. HTTP `Authorization` 헤더를 `Basic <base64(:API-KEY)>` 형식으로 구성하세요(앞에 콜론이 오고 사용자 이름은 비어 있습니다). 서비스 계정 API 키는 Organization Dashboard의 **Service account** 탭에서 찾을 수 있습니다. [조직 범위 서비스 계정](/ko/platform/hosting/iam/service-accounts/#organization-scoped-service-accounts)을 참고하세요.

예를 들어, API 키 `sa-p@55w0rd`로 인증하세요:

```bash
Authorization: Basic OnNhLXBANTV3MHJk
```

<div id="user-management">
  ## 사용자 관리
</div>

SCIM 사용자 리소스는 W&amp;B Users에 매핑됩니다. 조직의 사용자를 관리하려면 이러한 endpoints를 사용하세요.

<div id="get-user">
  ### 사용자 조회
</div>

조직 내 특정 사용자의 정보를 조회합니다.

<Note>이 오퍼레이션으로는 서비스 계정 정보를 조회할 수 없습니다.</Note>

<div id="endpoint">
  #### Endpoint
</div>

* **URL**: `<host-url>/scim/Users/{id}`
* **방법**: GET

<div id="parameters">
  #### 파라미터
</div>

| Parameter | 유형     | Required | 설명         |
| --------- | ------ | -------- | ---------- |
| `id`      | string | Yes      | 사용자의 고유 ID |

<div id="examples">
  #### 예시
</div>

<Tabs>
  <Tab title="User 조회 Request">
    ```bash
    GET /scim/Users/abc
    ```
  </Tab>

  <Tab title="User 조회 Response">
    ```bash
    (Status 200)
    ```

    ```json
    {
        "active": true,
        "daysActive": 42,
        "displayName": "Dev User 1",
        "emails": {
            "Value": "dev-user1@example.com",
            "Display": "",
            "Type": "",
            "Primary": true
        },
        "id": "abc",
        "lastActiveAt": "2023-10-15T14:32:10Z",
        "meta": {
            "resourceType": "User",
            "created": "2023-10-01T00:00:00Z",
            "lastModified": "2023-10-01T00:00:00Z",
            "location": "Users/abc"
        },
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User"
        ],
        "userName": "dev-user1"
    }
    ```

    Response에는 조직 내에서 사용자의 활동에 대한 세부 정보가 포함됩니다.

    * **`daysActive`**: 해당 사용자가 조직에서 활동한 전체 일수입니다.
    * **`lastActiveAt`**: 사용자의 가장 최근 활동 시점을 나타내는 ISO 8601 Timestamp입니다. 사용자가 한 번도 활동한 적이 없다면 `null`을 반환합니다.

    &quot;활성&quot;의 정의는 배포 유형에 따라 다릅니다.

    * **Dedicated Cloud / Self-Managed**: 사용자가 로그인하거나, W&amp;B App의 어떤 페이지든 열거나, Runs를 기록하거나, SDK를 사용하거나, 기타 어떤 방식으로든 W&amp;B Server와 상호작용하면 활성으로 간주됩니다.
    * **Multi-tenant Cloud**: 사용자가 2025년 5월 8일 이후에 조직 범위에서 감사 가능한 액션을 수행하면 활성으로 간주됩니다. 전체 목록은 [Audit logging actions](/ko/platform/hosting/monitoring-usage/audit-logging#actions)를 참조하세요.
  </Tab>
</Tabs>

<div id="list-users">
  ### 사용자 목록
</div>

조직 내 모든 사용자의 목록을 조회합니다.

<div id="filter-users">
  #### Users 필터링
</div>

`/Users` 엔드포인트에서는 사용자 이름이나 이메일로 사용자를 필터링할 수 있습니다.

* `userName eq "value"` - 사용자 이름으로 필터링
* `emails.value eq "value"` - 이메일 주소로 필터링

<div id="example">
  ##### 예시
</div>

```bash
GET /scim/Users?filter=userName eq "john.doe"
GET /scim/Users?filter=emails.value eq "john@example.com"
```

<div id="create-user">
  ### 사용자 생성
</div>

조직에 새 사용자를 추가합니다.

<div id="endpoint">
  #### Endpoint
</div>

* **URL**: `<host-url>/scim/Users`
* **방법**: POST

<div id="parameters">
  #### 파라미터
</div>

| Parameter  | Type   | Required | 설명                                     |
| ---------- | ------ | -------- | ----------------------------------------------- |
| `emails`   | array  | Yes      | 이메일 객체의 배열입니다. 기본 이메일(primary email)을 포함해야 합니다. |
| `userName` | string | Yes      | 새 사용자의 사용자 이름입니다.                               |

<div id="examples">
  #### 예시
</div>

<Tabs>
  <Tab title="사용자 생성 Request (Dedicated/Self-Managed)">
    ```bash
    POST /scim/Users
    ```

    ```json
    {
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User"
        ],
        "emails": [
            {
                "primary": true,
                "value": "dev-user2@example.com"
            }
        ],
        "userName": "dev-user2"
    }
    ```
  </Tab>

  <Tab title="사용자 생성 Request (Multi-tenant)">
    ```bash
    POST /scim/Users
    ```

    ```json
    {
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User",
            "urn:ietf:params:scim:schemas:extension:teams:2.0:User"
        ],
        "emails": [
            {
                "primary": true,
                "value": "dev-user2@example.com"
            }
        ],
        "userName": "dev-user2",
        "urn:ietf:params:scim:schemas:extension:teams:2.0:User": {
            "teams": ["my-team"]
        }
    }
    ```

<div id="delete-user">
  ### 사용자 삭제
</div>

<Warning>
  **관리자 액세스 유지**

  항상 인스턴스 또는 조직에 최소 한 명의 관리자 사용자가 있어야 합니다. 그렇지 않으면 어떤 사용자도 조직의 W&amp;B 계정을 설정하거나 유지 관리할 수 없습니다. 조직에서 SCIM 또는 기타 자동화된 프로세스를 사용해 W&amp;B에서 사용자를 프로비저닝 해제하는 경우, 프로비저닝 해제 오퍼레이션으로 인해 인스턴스 또는 조직에 마지막으로 남아 있던 관리자가 의도치 않게 제거될 수 있습니다.

  운영 절차를 수립하는 데 도움이 필요하거나 관리자 액세스를 복구해야 하는 경우 [지원팀](mailto:support@wandb.com)에 문의하십시오.
</Warning>

조직에서 사용자를 완전히 삭제합니다.

<Note>이 오퍼레이션은 서비스 계정이 아닌 사용자에게만 적용됩니다. 서비스 계정은 W&amp;B Team 설정에서 삭제하십시오.</Note>

<div id="endpoint">
  #### Endpoint
</div>

* **URL**: `<host-url>/scim/Users/{id}`

| ---- | ------ | ----- | -------------- |
| `id` | string | 예     | 삭제할 사용자의 고유 ID |

<div id="examples">

<Note>

</Note>

<div id="update-user-email">
  ### 사용자 이메일 업데이트
</div>

사용자의 기본 이메일 주소를 업데이트합니다.
**Multi-tenant Cloud에서는 지원되지 않습니다.** 이 환경에서는 사용자 계정을 조직에서 관리하지 않습니다.

<div id="endpoint">
  #### Endpoint
</div>

* **URL**: `<host-url>/scim/Users/{id}`
* **방법**: PATCH

<div id="parameters">
  #### 파라미터
</div>

| 파라미터    | 유형     | 필수 여부 | 설명              |
| ------- | ------ | ----- | --------------- |
| `id`    | string | 예     | 사용자의 고유 ID      |
| `op`    | string | 예     | `replace`       |
| `path`  | string | 예     | `emails`        |
| `value` | array  | 예     | 새 이메일 객체가 담긴 배열 |

<div id="examples">
  #### 예시
</div>

<Tabs>
  <Tab title="이메일 업데이트 Request">
    ```bash
    PATCH /scim/Users/abc
    ```

    ```json
    {
        "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
        "Operations": [
            {
                "op": "replace",
                "path": "emails",
                "value": [
                    {
                        "value": "newemail@example.com",
                        "primary": true
                    }
                ]
            }
        ]
    }
    ```
  </Tab>

  <Tab title="이메일 업데이트 Response">
    ```bash
    (Status 200)
    ```

    ```json
    {
        "active": true,
        "displayName": "Dev User 1",
        "emails": {
            "Value": "newemail@example.com",
            "Display": "",
            "Type": "",
            "Primary": true
        },
        "id": "abc",
        "meta": {
    ```

<div id="assign-organization-role">
  ### 조직 역할 부여
</div>

사용자에게 조직 수준 역할을 부여합니다.

<Note>이 오퍼레이션은 서비스 계정이 아닌 사용자에게만 적용됩니다. 서비스 계정에서는 커스텀 역할이 지원되지 않습니다.</Note>

<div id="endpoint">
  #### Endpoint
</div>

* **URL**: `<host-url>/scim/Users/{id}`
* **방법**: PATCH

<div id="parameters">
  #### 파라미터
</div>

| 파라미터    | 유형     | 필수 | 설명                         |
| ------- | ------ | -- | -------------------------- |
| `id`    | string | 예  | 사용자의 고유 ID                 |
| `op`    | string | 예  | `replace`                  |
| `path`  | string | 예  | `organizationRole`         |
| `value` | string | 예  | 역할 이름(`admin` 또는 `member`) |

<Note>
  조직 범위의 `viewer` 역할은 사용 중단되었으며 더 이상 UI에서 부여할 수 없습니다. SCIM을 사용해 사용자에게 `viewer` 역할을 부여하면:

  * 조직에서 해당 사용자에게 `member` 역할이 부여됩니다.
  * `full` 시트 대신 Models `viewer` 시트가 부여됩니다. 이렇게 하면 Models에는 보기 전용으로 액세스할 수 있고 Registry에는 전체 액세스할 수 있습니다. 사용 가능한 Models 시트가 없으면 `Seat limit reached` 오류가 로깅되고, 해당 멤버는 Models 액세스 없이 추가됩니다. 시트를 사용할 수 있게 되면 나중에 업데이트할 수 있습니다.
  * `full` 시트 대신 Weave `viewer` 시트가 부여됩니다. 이렇게 하면 Weave에는 보기 전용으로 액세스할 수 있습니다. 사용 가능한 Weave 시트가 없으면 `Seat limit reached` 오류가 로깅되고, 해당 멤버는 Weave 액세스 없이 추가됩니다. 시트를 사용할 수 있게 되면 나중에 업데이트할 수 있습니다.
  * 조직 수준에서 볼 수 있는 레지스트리에서 Registry `viewer` 역할이 부여됩니다.
</Note>

<div id="create-team">
  ### 팀 생성
</div>

* **Endpoint**: **`<host-url>/scim/Groups`**
* **방법**: POST
* **설명**: 새 팀 리소스를 생성합니다.
* **Supported Fields**:

| Field           | Type               | Required                                |
| --------------- | ------------------ | --------------------------------------- |
| `displayName`   | String             | Yes                                     |
| `members`       | Multi-Valued Array | Yes (`value` 하위 필드는 필수이며 사용자 ID에 매핑됩니다) |
| `storageBucket` | Object             | No                                      |

팀 생성 시 `storageBucket` 객체를 포함하면 팀 수준의 [Bring your own bucket (BYOB)](/ko/platform/hosting/data-security/secure-storage-connector)을 설정할 수 있습니다. 이를 생략하면 팀은 기본 저장소 또는 인스턴스 수준 저장소를 사용합니다. BYOB 가이드를 사용하여 버킷(정책, CORS, 자격 증명)을 프로비저닝하고 제공자별 저장소 주소 형식을 확인하세요. `storageBucket` 객체에는 다음 하위 필드가 있습니다.

* **Required**: `name`(버킷 이름), `provider`(`coreweave`, `aws`, `gcp`, `azure`, `minio` 중 하나). 대소문자를 구분합니다.
* **선택**: `path`(버킷 내 경로 접두사), `kmsKeyId`(암호화를 위한 KMS 키, 예: AWS), `awsExternalId`(AWS 교차 계정 액세스), `azureTenantId`(Azure 테넌트 ID), `azureClientId`(Azure 관리 ID 클라이언트 ID).

W&amp;B는 팀을 생성하기 전에 버킷이 존재하고 액세스 가능한지 검증합니다. 검증에 실패하면 SCIM Request가 실패하고 팀이 생성되지 않습니다.

유효하지 않은 `provider` 값은 허용된 값을 나열하는 SCIM 오류와 함께 `400 Bad Request`를 반환합니다.

<div id="update-team">
  ### 팀 업데이트
</div>

* **Endpoint**: **`<host-url>/scim/Groups/{id}`**
* **방법**: PATCH
* **설명**: 기존 팀의 멤버십 목록을 업데이트합니다.
* **지원되는 오퍼레이션**: `add` 멤버, `remove` 멤버, `replace` 멤버들

<Note>
  - remove 오퍼레이션은 RFC 7644 SCIM 프로토콜 사양을 따릅니다. 특정 사용자를 제거하려면 필터 구문 `members[value eq "{user_id}"]`을 사용하고, 팀에서 모든 사용자를 제거하려면 `members`를 사용하세요.

    **사용자 식별**: 멤버 오퍼레이션의 `{user_id}`는 다음 중 하나일 수 있습니다:

    * W&amp;B 사용자 ID
    * 이메일 주소(예: &quot;user@example.com&quot;)
  - 이 오퍼레이션은 사용자에 대해서만 작동하며 서비스 계정에는 작동하지 않습니다. 팀의 서비스 계정은 W&amp;B Team의 Settings에서 업데이트하세요.
</Note>

<Info>
  요청에서 `{team_id}`는 실제 팀 ID로, `{user_id}`는 실제 사용자 ID 또는 이메일 주소로 바꾸세요.
</Info>

<div id="create-custom-role">
  ### 커스텀 역할 생성
</div>

* **Endpoint**: **`<host-url>/scim/Roles`**
* **방법**: POST
* **설명**: W&amp;B 조직에서 새 커스텀 역할을 생성합니다.
* **지원되는 필드**:

| 필드              | 유형     | 필수 여부                                                                                                                                      |
| --------------- | ------ | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `name`          | String | 커스텀 역할의 이름                                                                                                                                 |
| `description`   | String | 커스텀 역할에 대한 설명                                                                                                                              |
| `permissions`   | 객체 배열  | 각 객체에 `w&bobject:operation` 형식의 값을 갖는 `name` string 필드가 포함된 권한 객체 배열입니다. 예를 들어, W&amp;B Runs에 대한 삭제 오퍼레이션의 권한 객체는 `name`이 `run:delete`입니다. |
| `inheritedFrom` | String | 커스텀 역할이 상속할 미리 정의된 역할입니다. `member` 또는 `viewer`일 수 있습니다.                                                                                    |

<div id="etag-support">
  ### ETag 지원
</div>

SCIM API는 동시 수정 충돌을 방지하기 위해 조건부 업데이트에서 ETag를 지원합니다. ETag는 `ETag` Response 헤더와 `meta.version` 필드에 반환됩니다.

<div id="etags">
  #### ETags
</div>

ETags를 사용하려면:

1. **현재 ETag 조회**: 리소스를 GET할 때 응답의 ETag 헤더를 확인하세요
2. **조건부 업데이트**: 업데이트할 때 `If-Match` 헤더에 ETag를 포함하세요

<div id="examples">
  #### 예시
</div>

```
# 사용자 조회 및 ETag 기록
GET /scim/Users/abc
# Response에 포함: ETag: W/"xyz123"

# ETag로 업데이트
PATCH /scim/Users/abc
If-Match: W/"xyz123"

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
    "Operations": [
        {
            "op": "replace",
            "path": "organizationRole",
            "value": "admin"
        }
    ]
}
```

`412 Precondition Failed` 오류 Response는 리소스를 조회한 이후 해당 리소스가 수정되었음을 나타냅니다.

<div id="error-handling">
  ### 오류 처리
</div>

SCIM API는 표준 SCIM 오류 Response를 반환합니다:

| Status Code | 설명                                       |
| ----------- | ---------------------------------------- |
| `200`       | 성공                                       |
| `201`       | 생성됨                                      |
| `204`       | No Content (삭제 성공)                       |
| `400`       | 잘못된 Request - 유효하지 않은 파라미터 또는 Request 본문 |
| `401`       | 권한 없음 - 인증 실패                            |
| `403`       | 금지됨 - 권한이 부족함                            |
| `404`       | 찾을 수 없음 - 리소스가 존재하지 않음                   |
| `409`       | 충돌 - 리소스가 이미 존재함                         |
| `412`       | 사전 조건 실패 - ETag 불일치                      |
| `500`       | 내부 서버 오류                                 |

<div id="implementation-differences-per-deployment-type">
  ### 배포 유형별 구현 차이
</div>

W&amp;B는 두 개의 별도 SCIM API 구현을 유지하며, 그 사이의 특성에는 차이가 있습니다:

| 특성                    | Dedicated Cloud | Self-Managed |
| --------------------- | --------------- | ------------ |
| 사용자 이메일 업데이트          | -               | ✓            |
| 사용자 Display name 업데이트 | -               | ✓            |
| 사용자 비활성화              | ✓               | ✓            |
| 사용자 재활성화              | -               | ✓            |
| 사용자당 여러 이메일           | ✓               | -            |

<div id="limitations">
  ## 제한 사항
</div>

* **최대 결과 수**: Request당 9999개 항목입니다.
* **Single-tenant 환경**: 사용자당 이메일 하나만 지원합니다.
* **팀 삭제**: SCIM을 통해서는 지원되지 않습니다(W&amp;B 웹 인터페이스를 사용하세요).
* **사용자 재활성화**: Multi-tenant Cloud 환경에서는 지원되지 않습니다.
* **시트 한도**: 조직 시트 한도에 도달하면 작업이 실패할 수 있습니다.
