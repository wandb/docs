---
title: Dedicated Cloud에서 데이터 암호화
---

W&amp;B는 각 클라우드의 customer-managed encryption key(CMEK) 기능을 활용하여, 모든 [Dedicated Cloud](/ko/platform/hosting/hosting-options/dedicated_cloud)에서 W&amp;B가 관리하는 데이터베이스와 오브젝트 스토리지를 암호화하기 위해 W&amp;B가 관리하는 클라우드 네이티브 키를 사용합니다. 이 경우 W&amp;B는 클라우드 제공업체의 `customer`로서 동작하면서, 동시에 여러분에게 서비스 형태로 W&amp;B 플랫폼을 제공합니다. W&amp;B가 관리하는 키를 사용한다는 것은 W&amp;B가 각 클라우드에서 데이터를 암호화하는 데 사용하는 키를 직접 제어한다는 의미이며, 이를 통해 모든 고객에게 매우 안전하고 보안성이 높은 플랫폼을 제공하겠다는 약속을 한층 더 강화합니다.

W&amp;B는 각 고객 인스턴스의 데이터를 암호화하기 위해 `unique key`를 사용하여 Dedicated Cloud 테넌트 간에 또 하나의 격리 계층을 제공합니다. 이 기능은 AWS, Azure, Google Cloud에서 사용할 수 있습니다.

<Note>
  W&amp;B가 2024년 8월 이전에 프로비저닝한 Google Cloud 및 Azure의 Dedicated Cloud 인스턴스는, W&amp;B가 관리하는 데이터베이스와 오브젝트 스토리지를 암호화하기 위해 기본 클라우드 제공업체 관리 키를 사용합니다. 2024년 8월부터 W&amp;B가 새로 생성하는 인스턴스만 관련 암호화를 위해 W&amp;B가 관리하는 클라우드 네이티브 키를 사용합니다.

  AWS의 Dedicated Cloud 인스턴스는 2024년 8월 이전부터 이미 암호화를 위해 W&amp;B가 관리하는 클라우드 네이티브 키를 사용하고 있습니다.
</Note>

W&amp;B는 일반적으로 고객이 Dedicated Cloud 인스턴스에서 W&amp;B가 관리하는 데이터베이스와 오브젝트 스토리지를 암호화하기 위해 자체 클라우드 네이티브 키를 가져와 사용하는 것을 허용하지 않습니다. 그 이유는 하나의 조직 내에 여러 팀과 다양한 역할의 인력이 존재하며, 이들이 여러 목적으로 클라우드 인프라에 접근할 수 있기 때문입니다. 일부 팀이나 인력은 조직의 기술 스택에서 W&amp;B가 핵심 구성 요소라는 맥락을 알지 못할 수 있고, 이로 인해 클라우드 네이티브 키를 완전히 삭제하거나 W&amp;B의 해당 키에 대한 접근 권한을 철회할 수 있습니다. 이러한 조치는 조직의 W&amp;B 인스턴스 내 모든 데이터를 손상시켜 복구 불가능한 상태에 이를 수 있습니다.

조직에서 AI 워크플로우용 Dedicated Cloud 사용을 승인받기 위해 W&amp;B가 관리하는 데이터베이스와 오브젝트 스토리지를 암호화하는 데 반드시 자체 클라우드 네이티브 키를 사용해야 하는 경우, W&amp;B는 예외적으로 해당 요청을 검토할 수 있습니다. 승인되는 경우, 암호화를 위한 귀사의 클라우드 네이티브 키 사용은 W&amp;B Dedicated Cloud의 `shared responsibility model`을 따르게 됩니다. Dedicated Cloud 인스턴스가 운영 중인 동안 조직 내 어떤 사용자가든 키를 삭제하거나, W&amp;B의 해당 키에 대한 접근을 어느 시점에든 철회하는 경우, 그로 인해 발생하는 데이터 손실 또는 손상에 대해 W&amp;B는 어떠한 책임도 지지 않으며, 그러한 데이터 복구에 대해서도 책임을 지지 않습니다.
