  <Accordion title="누가 내 아티팩트에 접근할 수 있나요?">
    아티팩트는 상위 프로젝트의 접근 권한을 상속합니다:

    * 비공개 프로젝트에서는 팀 구성원만 아티팩트에 접근할 수 있습니다.
    * 공개 프로젝트에서는 모든 사용자가 아티팩트를 읽을 수 있지만, 생성하거나 수정할 수 있는 것은 팀 구성원뿐입니다.
    * 오픈 프로젝트에서는 모든 사용자가 아티팩트를 읽고 쓸 수 있습니다.

    ## Artifacts 워크플로우

    이 섹션에서는 Artifacts를 관리하고 편집하기 위한 워크플로우를 설명합니다. 많은 워크플로우는 W&amp;B에 저장된 데이터에 접근할 수 있는 [W&amp;B API](/ko/models/track/public-api-guide)를 활용합니다. 이는 [클라이언트 라이브러리](/ko/models/ref/python/)의 구성 요소입니다.
  </Accordion>
</Update>

  <Accordion title="Plotly 또는 Bokeh 차트를 Tables에 어떻게 추가하나요?">
    Plotly 또는 Bokeh figure를 테이블에 직접 연동하는 기능은 지원되지 않습니다. 대신 figure를 HTML로 내보낸 후, 해당 HTML을 테이블에 포함하세요. 아래 예제는 인터랙티브한 Plotly 및 Bokeh 차트를 사용하여 이를 구현하는 방법을 보여줍니다.

    <Tabs>
      <Tab title="Plotly 사용하기">
        ```python
        import wandb
        import plotly.express as px

        # 새로운 run 초기화
        with wandb.init(project="log-plotly-fig-tables", name="plotly_html") as run:

            # 테이블 생성
            table = wandb.Table(columns=["plotly_figure"])

            # Plotly figure 경로 정의
            path_to_plotly_html = "./plotly_figure.html"

            # Plotly figure 생성
            fig = px.scatter(x=[0, 1, 2, 3, 4], y=[0, 1, 4, 9, 16])

            # Plotly figure를 HTML로 내보내기
            # auto_play를 False로 설정하면 애니메이션 Plotly 차트가 자동으로 재생되지 않습니다.
            fig.write_html(path_to_plotly_html, auto_play=False)

            # Plotly figure를 HTML 파일로 테이블에 추가
            table.add_data(wandb.Html(path_to_plotly_html))

            # 테이블 로깅
            run.log({"test_table": table})

        ```
      </Tab>

      <Tab title="Bokeh 사용하기">
        ```python
        from scipy.signal import spectrogram
        import holoviews as hv
        import panel as pn
        from scipy.io import wavfile
        import numpy as np
        from bokeh.resources import INLINE

        hv.extension("bokeh", logo=False)
        import wandb

        def save_audio_with_bokeh_plot_to_html(audio_path, html_file_name):
            sr, wav_data = wavfile.read(audio_path)
            duration = len(wav_data) / sr
            f, t, sxx = spectrogram(wav_data, sr)
            spec_gram = hv.Image((t, f, np.log10(sxx)), ["Time (s)", "Frequency (Hz)"]).opts(
                width=500, height=150, labelled=[]
            )
            audio = pn.pane.Audio(wav_data, sample_rate=sr, name="Audio", throttle=500)
            slider = pn.widgets.FloatSlider(end=duration, visible=False)
            line = hv.VLine(0).opts(color="white")
            slider.jslink(audio, value="time", bidirectional=True)
            slider.jslink(line, value="glyph.location")
            combined = pn.Row(audio, spec_gram * line, slider).save(html_file_name)

        html_file_name = "audio_with_plot.html"
        audio_path = "hello.wav"
        save_audio_with_bokeh_plot_to_html(audio_path, html_file_name)

        wandb_html = wandb.Html(html_file_name)
        with wandb.init(project="audio_test") as run:
            my_table = wandb.Table(columns=["audio_with_plot"], data=[[wandb_html], [wandb_html]])
            run.log({"audio_table": my_table})
        ```
      </Tab>
    </Tabs>

<Update tags={["Artifacts", "Storage"]}>
  <Accordion title="각 아티팩트 버전은 얼마나 많은 스토리지를 사용하나요?">
    두 아티팩트 버전 사이에서 변경된 파일만 스토리지 비용이 발생합니다.

    <Frame>
      <img src="/images/artifacts/artifacts-dedupe.PNG" alt="아티팩트 중복 제거" />
    </Frame>

    `cat.png`와 `dog.png` 두 개의 이미지 파일을 포함하는 `animals`라는 이름의 이미지 아티팩트를 가정해 봅시다:

    ```
    images
    |-- cat.png (2MB) # `v0`에 추가
    |-- dog.png (1MB) # `v0`에 추가
    ```

    이 아티팩트의 버전은 `v0`입니다.

    새 이미지 `rat.png`를 추가하면, 다음과 같은 내용을 가진 새 아티팩트 버전 `v1`이 생성됩니다:

    ```
    images
    |-- cat.png (2MB) # `v0`에 추가
    |-- dog.png (1MB) # `v0`에 추가
    |-- rat.png (3MB) # `v1`에 추가
    ```

    버전 `v1`은 총 6MB를 추적하지만, 나머지 3MB를 `v0`와 공유하므로 실제로는 3MB의 공간만 차지합니다. `v1`을 삭제하면 `rat.png`에 해당하는 3MB 스토리지가 회수됩니다. `v0`를 삭제하면 `cat.png`와 `dog.png`의 스토리지 비용이 `v1`로 이전되어, `v1`의 스토리지 사용량이 6MB로 증가합니다.
  </Accordion>
</Update>

  <Accordion title="여러 아키텍처와 run에서 아티팩트를 사용하는 방법은?">
    모델을 버전 관리하는 방법은 여러 가지가 있습니다. Artifacts는 특정 요구 사항에 맞게 모델 버전 관리를 할 수 있는 도구를 제공합니다. 여러 모델 아키텍처를 탐색하는 프로젝트의 일반적인 접근 방식은 아키텍처별로 아티팩트를 분리하는 것입니다. 다음 단계를 따르세요:

    1. 서로 다른 각 모델 아키텍처마다 새로운 아티팩트를 만드세요. run에서 `config`를 사용하는 것과 유사하게, 아키텍처에 대한 자세한 설명을 제공하기 위해 아티팩트의 `metadata` 속성을 사용하세요.
    2. 각 모델에 대해 `log_artifact`로 주기적으로 체크포인트를 로깅하세요. W&amp;B는 이러한 체크포인트의 이력을 구축하고, 가장 최근 체크포인트에 `latest` 별칭을 지정합니다. `architecture-name:latest`를 사용해 어떤 모델 아키텍처에 대해서든 최신 체크포인트를 참조할 수 있습니다.
  </Accordion>
</Update>

  <Accordion title="W&B 클라이언트는 Python 2를 지원하나요?">
    W&amp;B 클라이언트 라이브러리는 버전 0.10까지 Python 2.7과 Python 3를 모두 지원했습니다. Python 2의 서비스 종료로 인해 버전 0.11부터 Python 2.7 지원이 중단되었습니다. Python 2.7 시스템에서 `pip install --upgrade wandb`를 실행하면 0.10.x 시리즈의 새 릴리스만 설치됩니다. 0.10.x 시리즈에 대한 지원은 치명적인 버그 수정과 패치에만 한정됩니다. Python 2.7을 지원하는 0.10.x 시리즈의 마지막 버전은 0.10.33입니다.
  </Accordion>
</Update>

<Update tags={["Sweeps"]}>
  <Accordion title="스윕에서 사용자 정의 CLI 명령을 사용하려면 어떻게 해야 하나요?">
    트레이닝 설정이 커맨드라인 인수를 받도록 되어 있다면, 사용자 정의 CLI 명령과 함께 W&amp;B Sweeps를 사용할 수 있습니다.

    아래 예시에서는, 사용자가 `train.py`라는 이름의 Python 스크립트를 트레이닝하면서 스크립트가 파싱할 값을 제공하는 bash 터미널 코드 스니펫을 보여줍니다:

    ```bash
    /usr/bin/env python train.py -b \
        your-training-config \
        --batchsize 8 \
        --lr 0.00001
    ```

    사용자 정의 명령을 구현하려면 YAML 파일에서 `command` 키를 수정하세요. 이전 예시를 기준으로 설정은 다음과 같습니다:

    ```yaml
    program:
      train.py
    method: grid
    parameters:
      batch_size:
        value: 8
      lr:
        value: 0.0001
    command:
      - ${env}
      - python
      - ${program}
      - "-b"
      - your-training-config
      - ${args}
    ```

    `${args}` 키는 스윕 설정에 포함된 모든 파라미터를 `argparse` 형식인 `--param1 value1 --param2 value2`로 확장합니다.

    `argparse` 바깥의 추가 인수를 사용하려면, 다음과 같이 구현하세요:

    ```python
    parser = argparse.ArgumentParser()
    args, unknown = parser.parse_known_args()
    ```

    <Note>
      환경에 따라 `python`이 Python 2를 가리킬 수 있습니다. Python 3을 사용하도록 보장하려면, 명령 설정에서 `python3`를 사용하세요:

      ```yaml
      program:
        script.py
      command:
        - ${env}
        - python3
        - ${program}
        - ${args}
      ```
    </Note>
  </Accordion>
</Update>

  <Accordion title="네트워크 문제는 어떻게 처리하나요?">
    `wandb: Network error (ConnectionError), entering retry loop`와 같은 SSL 또는 네트워크 오류가 발생한 경우, 다음 방법을 시도하세요:

    1. SSL 인증서를 갱신합니다. Ubuntu 서버에서는 `update-ca-certificates`를 실행하세요. 유효한 SSL 인증서는 보안 위험을 줄이면서 트레이닝 로그를 동기화하는 데 필수적입니다.
    2. 네트워크 연결이 불안정한 경우, [optional environment variable](/ko/models/track/environment-variables#optional-environment-variables) `WANDB_MODE`를 `offline`으로 설정해 오프라인 모드에서 동작하게 한 뒤, 인터넷에 연결된 장치에서 나중에 파일을 동기화하세요.
    3. 로컬에서 실행되며 클라우드 서버와의 동기화를 피할 수 있는 [W&amp;B Private Hosting](/ko/platform/hosting/) 사용을 고려하세요.

    `SSL CERTIFICATE_VERIFY_FAILED` 오류의 경우, 회사 방화벽이 원인일 수 있습니다. 로컬 CA를 설정한 후 다음을 실행하세요:

    `export REQUESTS_CA_BUNDLE=/etc/ssl/certs/ca-certificates.crt`
  </Accordion>
</Update>

<Update tags={["Charts"]}>
  <Accordion title="`.log()`와 `.summary`의 차이는 무엇인가요?">
    `summary`는 테이블에 표시되고, `log`는 나중에 플로팅할 수 있도록 모든 값을 저장합니다.

    예를 들어 정확도가 변경될 때마다 `run.log()`를 호출합니다. 기본적으로 `run.log()`는 해당 메트릭에 대해 `summary` 값을 직접 설정하지 않는 한 `summary` 값을 업데이트합니다.

    산점도와 평행좌표 플롯은 `summary` 값을 사용하고, 선형 플롯은 `run.log()`로 기록된 모든 값을 표시합니다.

    일부 사용자는 가장 최근에 기록된 정확도가 아니라, 최적의 정확도를 반영하도록 `summary` 값을 수동으로 설정하는 것을 선호합니다.
  </Accordion>
</Update>

  <Accordion title="wandb.init 모드 간의 차이는 무엇인가요?">
    다음 모드가 제공됩니다:

    * `online` (기본): 클라이언트가 wandb 서버로 데이터를 전송합니다.
    * `offline`: 클라이언트가 wandb 서버로 데이터를 전송하는 대신 머신의 로컬에 데이터를 저장합니다. 나중에 데이터를 동기화하려면 [`wandb sync`](/ko/models/ref/cli/wandb-sync) 명령을 사용하세요.
    * `disabled`: 클라이언트가 모킹된 객체를 반환하여 동작을 시뮬레이션하며, 네트워크 통신을 모두 차단합니다. 모든 로깅은 비활성화되지만, 모든 API 메서드 스텁은 여전히 호출할 수 있습니다. 이 모드는 일반적으로 테스트에 사용됩니다.

<Update tags={["Tensorboard"]}>
  <Accordion title="W&B는 TensorBoard와 어떻게 다른가요?">
    W&amp;B는 TensorBoard와 통합되어 실험 추적 도구를 개선합니다. W&amp;B의 창업자들은 TensorBoard 사용자들이 겪는 공통적인 불편을 해소하기 위해 W&amp;B를 만들었습니다. 주요 개선 사항은 다음과 같습니다:

    1. **모델 재현성**: W&amp;B는 실험, 탐색, 모델 재현을 용이하게 합니다. 메트릭, 하이퍼파라미터, 코드 버전, 모델 체크포인트를 모두 캡처하여 재현성을 보장합니다.

    2. **자동 정리**: W&amp;B는 시도했던 모든 모델에 대한 개요를 제공하여 프로젝트 인수인계와 휴가 시기를 원활하게 합니다. 이를 통해 오래된 실험을 다시 실행하지 않아도 되어 시간을 절약할 수 있습니다.

    3. **빠른 인테그레이션**: 5분 안에 프로젝트에 W&amp;B를 인테그레이션할 수 있습니다. 무료 오픈 소스 Python 패키지를 설치하고 코드 몇 줄만 추가하면 됩니다. 로깅된 메트릭과 기록은 각 모델 run과 함께 표시됩니다.

    4. **중앙화된 대시보드**: 트레이닝이 어디에서 이루어지든 — 로컬, 연구실 클러스터, 클라우드 스팟 인스턴스 — 일관된 대시보드에 접근할 수 있습니다. 서로 다른 머신 간에 TensorBoard 파일을 관리할 필요가 없습니다.

    5. **강력한 필터링 테이블**: 다양한 모델에서 나온 결과를 검색, 필터, 정렬, 그룹화하여 효율적으로 확인할 수 있습니다. 이렇게 해서 서로 다른 작업별로 최고의 성능을 내는 모델을 쉽게 식별할 수 있으며, 이는 TensorBoard가 대규모 프로젝트에서 자주 어려움을 겪는 부분입니다.

    6. **협업 도구**: W&amp;B는 복잡한 머신러닝 프로젝트에서의 협업을 강화합니다. 프로젝트 링크를 공유하고, 결과 공유를 위해 비공개 팀을 활용할 수 있습니다. 작업 로그나 발표용으로 인터랙티브 시각화와 마크다운 설명이 포함된 리포트를 만들 수 있습니다.
  </Accordion>
</Update>

    ## 1. 항상 오류 처리를 구현하세요

    API 호출을 `try-except` 블록으로 감싸세요:

    ```python
    import openai

    try:
        response = client.chat.completions.create(
            model="meta-llama/Llama-3.1-8B-Instruct",
            messages=messages
        )
    except Exception as e:
        print(f"Error: {e}")
        # 여기에서 오류를 적절히 처리하세요
    ```

    ## 2. 지수 백오프를 사용하는 재시도 로직을 사용하세요

    ```python
    import time
    from typing import Optional

    def call_inference_with_retry(
        client, 
        messages, 
        model: str,
        max_retries: int = 3,
        base_delay: float = 1.0
    ) -> Optional[str]:
        for attempt in range(max_retries):
            try:
                response = client.chat.completions.create(
                    model=model,
                    messages=messages
                )
                return response.choices[0].message.content
            except Exception as e:
                if attempt == max_retries - 1:
                    raise
                
                # 지수 백오프를 사용해 지연 시간 계산
                delay = base_delay * (2 ** attempt)
                print(f"Attempt {attempt + 1} failed, retrying in {delay}s...")
                time.sleep(delay)
        
        return None
    ```

    ## 3. 사용량을 모니터링하세요

    * W&amp;B Billing 페이지에서 크레딧 사용량을 추적하세요
    * 한도에 도달하기 전에 알림을 설정하세요
    * 애플리케이션에서 API 사용량을 로그로 남기세요

    ```python
    def handle_inference_error(error):

            raise ValueError("API 키와 프로젝트 설정을 확인하세요")

            # 크레딧 소진
            raise ValueError("크레딧이 부족합니다")
        elif "429" in error_str:

            return "retry"
        elif "500" in error_str or "503" in error_str:

        else:
            # 알 수 없는 오류
            raise
    ```

  <Accordion title="W&B Inference에서 서버 오류(500, 503)를 어떻게 해결하나요?">
    서버 오류는 W&amp;B Inference 서비스에 일시적인 문제가 있음을 나타냅니다.

    ## 오류 유형

    ### 500 - Internal Server Error

    **메시지:** &quot;The server had an error while processing your request&quot;

    서버 측에서 발생한 일시적인 내부 오류입니다.

    ### 503 - Service Overloaded

    **메시지:** &quot;The engine is currently overloaded, please try again later&quot;

    서비스에 과도한 트래픽이 발생한 상태입니다.

    ## 서버 오류 처리 방법

    1. **재시도하기 전에 대기하세요.**
       * 500 오류: 30–60초 대기
       * 503 오류: 60–120초 대기

    2. **지수 백오프를 사용하세요.**
       ```python
       import time
       import openai

       def call_with_retry(client, messages, model, max_retries=5):
           for attempt in range(max_retries):
               try:
                   return client.chat.completions.create(
                       model=model,
                       messages=messages
                   )
               except Exception as e:
                   if "500" in str(e) or "503" in str(e):
                       if attempt < max_retries - 1:
                           wait_time = min(60, (2 ** attempt))
                           time.sleep(wait_time)
                       else:
                           raise
                   else:
                       raise
       ```

    3. **적절한 타임아웃을 설정하세요.**
       * HTTP 클라이언트의 타임아웃 값을 늘립니다.
       * 더 나은 처리를 위해 비동기(async) 작업을 고려합니다.

    다음과 같은 경우 지원팀에 문의하세요.

    * 오류가 10분 이상 지속될 때
    * 특정 시간대에 반복적인 실패 패턴이 나타날 때

  <Accordion title="하나의 스크립트에서 여러 run을 실행하려면 어떻게 하나요?">
    하나의 스크립트에서 여러 run을 기록하려면, 새 run을 시작하기 전에 이전 run을 먼저 종료해야 합니다.

    권장 방법은 `wandb.init()`을 컨텍스트 매니저로 사용하는 것입니다. 이렇게 하면 스크립트에서 예외가 발생했을 때 run을 종료하고 실패로 표시합니다:

    ```python
    import wandb

    for x in range(10):
        with wandb.init() as run:
            for y in range(100):
                run.log({"metric": x + y})
    ```

    `run.finish()`를 명시적으로 호출할 수도 있습니다:

    ```python
    import wandb

    for x in range(10):
        run = wandb.init()

        try:
            for y in range(100):
                run.log({"metric": x + y})

        except Exception:
            run.finish(exit_code=1)
            raise

        finally:
            run.finish()
    ```

    ## 여러 활성 run

    wandb 0.19.10부터는 `reinit` 설정을 `"create_new"`로 지정하여 동시에 활성 상태인 여러 run을 만들 수 있습니다.

    ```python
    import wandb

    with wandb.init(reinit="create_new") as tracking_run:
        for x in range(10):
            with wandb.init(reinit="create_new") as run:
                for y in range(100):
                    run.log({"x_plus_y": x + y})

                tracking_run.log({"x": x})
    ```

    `reinit="create_new"`에 대한 자세한 내용과 W&amp;B 인테그레이션 관련 주의사항은 [Multiple runs per process](/ko/models/runs/initialize-run)를 참고하세요.
  </Accordion>
</Update>

    ## 유효한 메트릭 이름

    * **허용 문자**: 알파벳 (A-Z, a-z), 숫자 (0-9), 밑줄 (&#95;)
    * **시작 문자**: 이름은 알파벳 또는 밑줄로 시작해야 함
    * **패턴**: 메트릭 이름은 `/^[_a-zA-Z][_a-zA-Z0-9]*$/` 패턴과 일치해야 함

    이 규칙을 따르지 않는 메트릭은 W&amp;B UI에서 정렬하거나 필터링할 수 없을 수 있습니다.

    ## 예시

    **유효한 메트릭 이름:**

    ```python
    with wandb.init() as run:
      run.log({"accuracy": 0.9, "val_loss": 0.1, "epoch_5": 5})
      run.log({"modelAccuracy": 0.95, "learning_rate": 0.001})
    ```

    **유효하지 않은 메트릭 이름 (다음은 피하세요):**

    ```python
    with wandb.init() as run:
      run.log({"acc,val": 0.9})  # 쉼표 포함
      run.log({"loss-train": 0.1})  # 하이픈 포함
      run.log({"test acc": 0.95})  # 공백 포함
      run.log({"5_fold_cv": 0.8})  # 숫자로 시작
    ```

    ## 권장 해결 방법

    유효하지 않은 문자를 밑줄과 같은 유효한 문자로 바꾸세요:

    * `"test acc"` 대신 `"test_acc"` 사용
    * `"loss-train"` 대신 `"loss_train"` 사용
    * `"acc,val"` 대신 `"acc_val"` 사용

    자세한 내용은 [메트릭 이름 제약 사항](/ko/models/track/log/#metric-naming-constraints)을 참고하세요.
  </Accordion>
</Update>

    <Tabs>

        ```python
        # 하이퍼파라미터 기본값 설정
        config_defaults = {"lr": 0.1, "batch_size": 256}

        # run을 시작하며 스윕이 덮어쓸 수 있는
        # 기본값을 제공합니다
        with wandb.init(config=config_defaults) as run:
            # 여기에 트레이닝 코드를 추가하세요
            ...
        ```
      </Tab>

      <Tab title="config.setdefaults()">
        ```python
        # 하이퍼파라미터 기본값 설정
        config_defaults = {"lr": 0.1, "batch_size": 256}

        # run 시작
        with wandb.init() as run:
            # 스윕에서 설정하지 않은 값들을 업데이트
            run.config.setdefaults(config_defaults)

            # 여기에 트레이닝 코드를 추가하세요
        ```
      </Tab>
    </Tabs>
  </Accordion>

  <Accordion title="로그한 것보다 적은 데이터 포인트가 보이는 이유는 무엇인가요?">
    `Step`이 아닌 다른 X축으로 메트릭을 시각화하면 더 적은 데이터 포인트가 나타날 수 있습니다. 메트릭이 동기화된 상태를 유지하려면 동일한 `Step`에서 로그되어야 합니다. 샘플 간 보간을 수행할 때는 동일한 `Step`에서 로그된 메트릭만 샘플링됩니다.

    **지침**

    메트릭을 하나의 `log()` 호출에 묶어 주세요. 예를 들어, 다음과 같이 하는 대신:

    ```python
    import wandb
    with wandb.init() as run:
        run.log({"Precision": precision})
        ...
        run.log({"Recall": recall})
    ```

    다음과 같이 사용하세요:

    ```python
    import wandb
    with wandb.init() as run:
        run.log({"Precision": precision, "Recall": recall})
    ```

    step 파라미터를 수동으로 제어하려면, 코드에서 메트릭을 다음과 같이 동기화하세요:

    ```python
    with wandb.init() as run:
        step = 100  # 예시 step 값
        # Precision과 Recall을 동일한 step에서 로그
        run.log({"Precision": precision, "Recall": recall}, step=step)
    ```

    두 `log()` 호출에서 `step` 값이 동일해야 메트릭이 같은 step 아래에 로그되고 함께 샘플링됩니다. `step` 값은 각 호출에서 단조 증가해야 합니다. 그렇지 않으면 `step` 값은 무시됩니다.
  </Accordion>
</Update>

    * 콜백 URI를 `https://wandb.auth0.com/login/callback`으로 설정합니다.

    설정을 완료한 후, 애플리케이션의 `Client ID`와 `Issuer URL`을 고객 성공 매니저(CSM)에게 전달하세요. W&amp;B는 이 정보를 사용해 Auth0 연결을 설정하고 SSO를 활성화합니다.

      `wandb.init(mode="disabled")` 를 사용해도 W&amp;B가 `WANDB_CACHE_DIR` 에 아티팩트를 저장하는 것은 차단되지 않습니다.
    </Note>
  </Accordion>
</Update>

  <Accordion title="wandb.init는 제 트레이닝 프로세스에 무엇을 하나요?">
    트레이닝 스크립트에서 `wandb.init()` 가 실행되면, API 호출을 통해 서버에 run 객체가 생성됩니다. 메트릭을 스트리밍하고 수집하는 새 프로세스가 시작되어, 메인 프로세스는 정상적으로 동작할 수 있습니다. 스크립트는 로컬 파일에 쓰기를 수행하는 동안, 별도의 프로세스가 시스템 메트릭을 포함한 데이터를 서버로 스트리밍합니다. 스트리밍을 끄려면 트레이닝 디렉터리에서 `wandb off` 를 실행하거나 `WANDB_MODE` 환경 변수를 `offline` 으로 설정하세요.
  </Accordion>
</Update>

<Update tags={["Artifacts", "Environment Variables"]}>
  <Accordion title="아티팩트는 어디에 다운로드되며, 이 위치를 어떻게 변경할 수 있나요?">
    기본적으로 아티팩트는 `artifacts/` 폴더에 다운로드됩니다. 위치를 변경하려면:

    * [`wandb.Artifact().download`](/ko/models/ref/python/public-api/api)에 인자로 경로를 전달합니다:

      ```python
      wandb.Artifact().download(root="<path_to_download>")
      ```

    * `WANDB_ARTIFACT_DIR` [환경 변수](/ko/models/track/environment-variables)를 설정합니다:

      ```python
      import os
      os.environ["WANDB_ARTIFACT_DIR"] = "<path_to_download>"
      ```
  </Accordion>
</Update>

<Update tags={["Experiments", "Runs"]}>
  <Accordion title="CSV 메트릭 내보내기에서 스텝이 누락되는 이유는 무엇인가요?">
    내보내기 제한으로 인해 전체 run 히스토리를 CSV로 내보내거나 `run.history` API로 가져오지 못할 수도 있습니다. 전체 run 히스토리에 접근하려면 Parquet 형식으로 run 히스토리 아티팩트를 다운로드하세요:

    ```python
    import wandb
    import pandas as pd

    run = wandb.init()
    artifact = run.use_artifact('<entity>/<project>/<run-id>-history:v0', type='wandb-history')
    artifact_dir = artifact.download()
    df = pd.read_parquet('<path to .parquet file>')
    ```
  </Accordion>
</Update>
