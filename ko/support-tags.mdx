  <Accordion title="누가 내 아티팩트에 접근할 수 있나요?">
    아티팩트는 상위 프로젝트의 액세스 권한을 그대로 따릅니다:

    * 비공개 프로젝트에서는 팀 구성원만 아티팩트에 접근할 수 있습니다.
    * 공개 프로젝트에서는 모든 사용자가 아티팩트를 읽을 수 있고, 팀 구성원만 생성하거나 수정할 수 있습니다.
    * 개방형 프로젝트에서는 모든 사용자가 아티팩트를 읽고 쓸 수 있습니다.

    ## Artifacts 워크플로

    이 섹션에서는 Artifacts를 관리하고 편집하기 위한 워크플로를 설명합니다. 많은 워크플로는 W&amp;B에 저장된 데이터에 접근할 수 있도록 하는 [클라이언트 라이브러리](/ko/models/ref/python/)의 구성 요소인 [W&amp;B API](/ko/models/track/public-api-guide)를 활용합니다.
  </Accordion>
</Update>

  <Accordion title="Plotly 또는 Bokeh 차트를 Tables에 어떻게 추가하나요?">
    Plotly 또는 Bokeh figure를 Tables에 직접 통합하는 기능은 지원되지 않습니다. 대신 figure를 HTML로 내보낸 후 해당 HTML을 테이블에 포함하세요. 아래 예시는 대화형 Plotly 및 Bokeh 차트를 사용해 이를 구현하는 방법을 보여줍니다.

    <Tabs>
      <Tab title="Plotly 사용하기">
        ```python
        import wandb
        import plotly.express as px

        # 새 실행 초기화
        with wandb.init(project="log-plotly-fig-tables", name="plotly_html") as run:

            # 테이블 생성
            table = wandb.Table(columns=["plotly_figure"])

            # Plotly figure 경로 정의
            path_to_plotly_html = "./plotly_figure.html"

            # Plotly figure 생성
            fig = px.scatter(x=[0, 1, 2, 3, 4], y=[0, 1, 4, 9, 16])

            # Plotly figure를 HTML로 내보내기
            # auto_play를 False로 설정하면 애니메이션 Plotly 차트가 자동 재생되지 않습니다
            fig.write_html(path_to_plotly_html, auto_play=False)

            # Plotly figure를 HTML 파일로 테이블에 추가
            table.add_data(wandb.Html(path_to_plotly_html))

            # 테이블 로깅
            run.log({"test_table": table})

        ```
      </Tab>

      <Tab title="Bokeh 사용하기">
        ```python
        from scipy.signal import spectrogram
        import holoviews as hv
        import panel as pn
        from scipy.io import wavfile
        import numpy as np
        from bokeh.resources import INLINE

        hv.extension("bokeh", logo=False)
        import wandb

        def save_audio_with_bokeh_plot_to_html(audio_path, html_file_name):
            sr, wav_data = wavfile.read(audio_path)
            duration = len(wav_data) / sr
            f, t, sxx = spectrogram(wav_data, sr)
            spec_gram = hv.Image((t, f, np.log10(sxx)), ["Time (s)", "Frequency (Hz)"]).opts(
                width=500, height=150, labelled=[]
            )
            audio = pn.pane.Audio(wav_data, sample_rate=sr, name="Audio", throttle=500)
            slider = pn.widgets.FloatSlider(end=duration, visible=False)
            line = hv.VLine(0).opts(color="white")
            slider.jslink(audio, value="time", bidirectional=True)
            slider.jslink(line, value="glyph.location")
            combined = pn.Row(audio, spec_gram * line, slider).save(html_file_name)

        html_file_name = "audio_with_plot.html"
        audio_path = "hello.wav"
        save_audio_with_bokeh_plot_to_html(audio_path, html_file_name)

        wandb_html = wandb.Html(html_file_name)
        with wandb.init(project="audio_test") as run:
            my_table = wandb.Table(columns=["audio_with_plot"], data=[[wandb_html], [wandb_html]])
            run.log({"audio_table": my_table})
        ```
      </Tab>
    </Tabs>
  </Accordion>

<Update tags={["Artifacts", "Storage"]}>
  <Accordion title="각 아티팩트 버전은 얼마나 많은 스토리지를 사용하나요?">
    두 아티팩트 버전 간에 변경된 파일에만 스토리지 비용이 발생합니다.

    <Frame>
      <img src="/images/artifacts/artifacts-dedupe.PNG" alt="Artifact 중복 제거" />
    </Frame>

    두 개의 이미지 파일 `cat.png`와 `dog.png`를 포함하는 `animals`라는 이름의 이미지 아티팩트를 가정해 봅시다:

    ```
    images
    |-- cat.png (2MB) # `v0`에 추가됨
    |-- dog.png (1MB) # `v0`에 추가됨
    ```

    이 아티팩트에는 버전 `v0`이 부여됩니다.

    새 이미지 `rat.png`를 추가하면, 다음과 같은 내용을 가진 새 아티팩트 버전 `v1`이 생성됩니다:

    ```
    images
    |-- cat.png (2MB) # `v0`에 추가됨
    |-- dog.png (1MB) # `v0`에 추가됨
    |-- rat.png (3MB) # `v1`에 추가됨
    ```

    버전 `v1`은 총 6MB를 추적하지만, 나머지 3MB를 `v0`와 공유하므로 실제로는 3MB의 공간만 차지합니다. `v1`을 삭제하면 `rat.png`에 해당하는 3MB의 스토리지가 회수됩니다. `v0`을 삭제하면 `cat.png`와 `dog.png`의 스토리지 비용이 `v1`으로 이전되어, `v1`의 스토리지 크기가 6MB로 증가합니다.
  </Accordion>
</Update>

<Update tags={["Artifacts"]}>
  <Accordion title="여러 아키텍처와 실행에서 아티팩트를 사용하는 방법은?">
    모델을 버전 관리하는 방법에는 여러 가지가 있습니다. 아티팩트는 특정 요구 사항에 맞춘 모델 버전 관리 도구를 제공합니다. 여러 모델 아키텍처를 탐색하는 프로젝트에서 일반적으로 사용하는 접근 방식은 아키텍처별로 아티팩트를 분리하는 것입니다. 다음 단계를 고려해 보세요:

    1. 서로 다른 각 모델 아키텍처에 대해 새 아티팩트를 생성합니다. 아티팩트의 `metadata` 속성을 사용해 아키텍처에 대한 자세한 설명을 제공하세요. 이는 실행의 `config`를 사용하는 방식과 유사합니다.
    2. 각 모델에 대해 `log_artifact`로 주기적으로 체크포인트를 로그합니다. W&amp;B는 이러한 체크포인트의 이력을 구축하고, 가장 최근 체크포인트에 `latest` 별칭을 부여합니다. `architecture-name:latest`를 사용해 어떤 모델 아키텍처에 대해서도 최신 체크포인트를 참조할 수 있습니다.
  </Accordion>
</Update>

<Update tags={["Sweeps"]}>
  <Accordion title="Sweeps에서 커스텀 CLI 명령을 사용하려면 어떻게 해야 하나요?">
    훈련 설정이 커맨드라인 인수를 전달하도록 구성되어 있다면, W&amp;B Sweeps에서 커스텀 CLI 명령을 사용할 수 있습니다.

    아래 예시는 사용자가 `train.py`라는 이름의 Python 스크립트를 bash 터미널에서 실행하면서, 스크립트가 파싱할 값을 함께 제공하는 코드 스니펫입니다:

    ```bash
    /usr/bin/env python train.py -b \
        your-training-config \
        --batchsize 8 \
        --lr 0.00001
    ```

    커스텀 명령을 사용하려면 YAML 파일의 `command` 키를 수정하면 됩니다. 앞의 예시를 바탕으로 한 설정은 다음과 같습니다:

    ```yaml
    program:
      train.py
    method: grid
    parameters:
      batch_size:
        value: 8
      lr:
        value: 0.0001
    command:
      - ${env}
      - python
      - ${program}
      - "-b"
      - your-training-config
      - ${args}
    ```

    `${args}` 키는 sweep 설정에 정의된 모든 매개변수를 `argparse` 형식인 `--param1 value1 --param2 value2`와 같이 확장합니다.

    `argparse` 외부의 추가 인자가 필요하다면, 다음과 같이 구현하세요:

    ```python
    parser = argparse.ArgumentParser()
    args, unknown = parser.parse_known_args()
    ```

    <Note>
      환경에 따라 `python`이 Python 2를 가리킬 수 있습니다. Python 3이 실행되도록 하려면, 명령 설정에서 `python3`를 사용하세요:

      ```yaml
      program:
        script.py
      command:
        - ${env}
        - python3
        - ${program}
        - ${args}
      ```
    </Note>
  </Accordion>
</Update>

  <Accordion title="네트워크 문제를 어떻게 처리해야 하나요?">
    `wandb: Network error (ConnectionError), entering retry loop`와 같은 SSL 또는 네트워크 오류가 발생하면 다음 방법을 시도하세요:

    1. SSL 인증서를 업데이트합니다. Ubuntu 서버에서는 `update-ca-certificates`를 실행하세요. 유효한 SSL 인증서는 보안 위험을 줄이기 위해 훈련 로그를 동기화하는 데 필수적입니다.
    2. 네트워크 연결이 불안정하다면, [optional environment variable](/ko/models/track/environment-variables#optional-environment-variables) `WANDB_MODE`를 `offline`으로 설정하여 오프라인 모드로 실행한 뒤, 인터넷에 연결된 장치에서 나중에 파일을 동기화하세요.
    3. 로컬에서 동작하며 클라우드 서버로의 동기화를 피할 수 있는 [W&amp;B Private Hosting](/ko/platform/hosting/) 사용을 고려하세요.

    `SSL CERTIFICATE_VERIFY_FAILED` 오류의 경우, 회사 방화벽에서 문제가 발생했을 수 있습니다. 로컬 CA를 설정하고 다음을 실행하세요:

    `export REQUESTS_CA_BUNDLE=/etc/ssl/certs/ca-certificates.crt`
  </Accordion>
</Update>

  <Accordion title="`.log()`와 `.summary`의 차이점은 무엇인가요?">
    `.summary`는 테이블에 표시되고, `.log`는 향후 플롯/그래프 생성을 위해 모든 값을 저장합니다.

    예를 들어, 정확도가 변경될 때마다 `run.log()`를 호출합니다. 기본적으로 `run.log()`는 해당 메트릭에 대해 요약 값을 수동으로 설정하지 않는 한 summary 값을 업데이트합니다.

    산점도와 평행좌표 플롯은 summary 값을 사용하고, 선 그래프(line plot)는 `run.log`에 의해 기록된 모든 값을 표시합니다.

    일부 사용자는 가장 최근에 기록된 정확도 대신 최적의 정확도를 반영하도록 summary를 수동으로 설정하는 것을 선호합니다.
  </Accordion>
</Update>

  <Accordion title="W&B는 TensorBoard와 어떻게 다른가요?">
    W&amp;B는 TensorBoard와 통합되고 실험 추적 기능을 보완합니다. W&amp;B 창립자는 TensorBoard 사용자들이 겪는 공통적인 불편함을 해소하기 위해 W&amp;B를 만들었습니다. 주요 개선 사항은 다음과 같습니다:

    1. **모델 재현성**: W&amp;B는 실험, 탐색, 모델 재현을 용이하게 합니다. 메트릭, 하이퍼파라미터, 코드 버전 정보를 기록하고 모델 체크포인트를 저장하여 재현성을 보장합니다.

    2. **자동 정리**: W&amp;B는 시도했던 모든 모델에 대한 개요를 제공하여 프로젝트 인수인계와 휴가 기간의 업무를 원활하게 합니다. 이를 통해 오래된 실험을 다시 실행하는 일을 방지하여 시간을 절약할 수 있습니다.

    3. **빠른 통합**: W&amp;B를 프로젝트에 5분 안에 통합할 수 있습니다. 무료 오픈 소스 Python 패키지를 설치하고 몇 줄의 코드를 추가하세요. 기록된 메트릭과 로그는 각 모델 실행과 함께 표시됩니다.

    4. **중앙 집중식 대시보드**: 학습이 어디에서 이루어지든(로컬, 연구실 클러스터, 클라우드 스팟 인스턴스 등) 일관된 대시보드에 접근할 수 있습니다. 서로 다른 머신 간에 TensorBoard 파일을 관리할 필요가 없습니다.

    5. **강력한 필터링 테이블**: 다양한 모델의 결과를 검색, 필터, 정렬, 그룹화하여 효율적으로 탐색할 수 있습니다. TensorBoard가 대규모 프로젝트에서 자주 어려움을 겪는, 여러 작업에 대해 최고의 성능을 내는 모델을 쉽게 식별할 수 있습니다.

    6. **협업 도구**: W&amp;B는 복잡한 머신러닝 프로젝트에서의 협업을 강화합니다. 프로젝트 링크를 공유하고, 결과 공유를 위한 비공개 팀을 활용할 수 있습니다. 작업 로그나 프레젠테이션용으로 인터랙티브 시각화와 마크다운 설명이 포함된 리포트를 생성하세요.

  <Accordion title="누가 리포트를 편집하고 공유할 수 있나요?">
    개인이 가진 비공개 프로젝트 내에서 생성된 리포트는 해당 사용자에게만 표시됩니다. 사용자는 자신의 프로젝트를 팀 또는 공개 대상으로 공유할 수 있습니다.

    ## 1. 항상 오류 처리 로직을 구현하세요

    API 호출을 try-except 블록으로 감싸세요:

    ```python
    import openai

    try:
        response = client.chat.completions.create(
            model="meta-llama/Llama-3.1-8B-Instruct",
            messages=messages
        )
    except Exception as e:
        print(f"Error: {e}")
        # 오류를 적절히 처리하세요
    ```

    ## 2. 지수 백오프를 사용하는 재시도 로직을 구현하세요

    ```python
    import time
    from typing import Optional

    def call_inference_with_retry(
        client, 
        messages, 
        model: str,
        max_retries: int = 3,
        base_delay: float = 1.0
    ) -> Optional[str]:
        for attempt in range(max_retries):
            try:
                response = client.chat.completions.create(
                    model=model,
                    messages=messages
                )
                return response.choices[0].message.content
            except Exception as e:
                if attempt == max_retries - 1:
                    raise
                
                # 지수 백오프로 대기 시간 계산
                delay = base_delay * (2 ** attempt)
                print(f"Attempt {attempt + 1} failed, retrying in {delay}s...")
                time.sleep(delay)
        
        return None
    ```

    ## 3. 사용량을 모니터링하세요

    * W&amp;B Billing 페이지에서 크레딧 사용량을 추적하세요

    def handle_inference_error(error):

        if "401" in error_str:

        elif "402" in error_str:

  <Accordion title="W&B Inference에서 서버 오류(500, 503)를 어떻게 해결하나요?">

    ## 오류 유형

    ### 500 - Internal Server Error

    **Message:** &quot;The server had an error while processing your request&quot;

    서버 측에서 발생한 일시적인 내부 오류입니다.

    ### 503 - Service Overloaded

    **Message:** &quot;The engine is currently overloaded, please try again later&quot;

    서비스에 트래픽이 과도하게 몰린 상태입니다.

    ## 서버 오류 처리 방법

    1. **재시도 전에 대기합니다.**
       * 500 오류: 30–60초 대기
       * 503 오류: 60–120초 대기

    2. **지수 백오프를 사용합니다.**
       ```python
       import time
       import openai

       def call_with_retry(client, messages, model, max_retries=5):
           for attempt in range(max_retries):
               try:
                   return client.chat.completions.create(
                       model=model,
                       messages=messages
                   )
               except Exception as e:
                   if "500" in str(e) or "503" in str(e):
                       if attempt < max_retries - 1:
                           wait_time = min(60, (2 ** attempt))
                           time.sleep(wait_time)
                       else:
                           raise
                   else:
                       raise
       ```

    3. **적절한 타임아웃을 설정합니다.**
       * HTTP 클라이언트의 타임아웃 값을 늘립니다.
       * 더 나은 처리를 위해 비동기 작업을 고려합니다.

    ## 언제 지원팀에 문의해야 하나요?

    다음과 같은 경우 지원팀에 문의하세요.

    * 오류가 10분 이상 지속되는 경우
    * 특정 시간대에 반복적인 실패 패턴이 보이는 경우
    * 오류 메시지에 추가 상세 정보가 포함된 경우

    * 오류가 발생한 시간
    * 코드 스니펫 (API 키는 제거)
    * W&amp;B 엔터티 및 프로젝트 이름

  <Accordion title="하나의 스크립트에서 여러 실행을 시작하려면 어떻게 하나요?">
    하나의 스크립트 안에서 여러 실행을 기록하려면, 새 실행을 시작하기 전에
    이전 실행을 먼저 종료해야 합니다.

    가장 권장되는 방법은 `wandb.init()`을 컨텍스트 매니저로 사용하는 것입니다.
    이렇게 하면 스크립트에서 예외가 발생할 경우 실행을 종료하고 실패로 표시합니다:

    ```python
    import wandb

    for x in range(10):
        with wandb.init() as run:
            for y in range(100):
                run.log({"metric": x + y})
    ```

    `run.finish()`를 명시적으로 호출할 수도 있습니다:

    ```python
    import wandb

    for x in range(10):
        run = wandb.init()

        try:
            for y in range(100):
                run.log({"metric": x + y})

        except Exception:
            run.finish(exit_code=1)
            raise

        finally:
            run.finish()
    ```

    ## 여러 개의 활성 실행

    wandb 0.19.10부터는 동시에 활성 상태인 여러 실행을 생성하기 위해
    `reinit` 설정을 `"create_new"`로 지정할 수 있습니다.

    ```python
    import wandb

    with wandb.init(reinit="create_new") as tracking_run:
        for x in range(10):
            with wandb.init(reinit="create_new") as run:
                for y in range(100):
                    run.log({"x_plus_y": x + y})

                tracking_run.log({"x": x})
    ```

    `reinit="create_new"`에 대한 자세한 내용과 W&amp;B 통합과 관련된 주의사항은
    [프로세스당 여러 실행](/ko/models/runs/initialize-run)을 참고하세요.
  </Accordion>
</Update>

  <Accordion title="특정 문자가 포함된 메트릭은 정렬하거나 필터링할 수 없는 이유가 무엇인가요?">
    W&amp;B의 메트릭 이름은 UI에서 올바르게 정렬하고 필터링할 수 있도록 GraphQL 명명 규칙을 따라야 합니다.

    ## 유효한 메트릭 이름

    * **허용되는 문자**: 문자(A-Z, a-z), 숫자(0-9), 밑줄(&#95;)
    * **시작 문자**: 이름은 문자 또는 밑줄로 시작해야 함
    * **패턴**: 메트릭 이름은 `/^[_a-zA-Z][_a-zA-Z0-9]*$/` 패턴과 일치해야 함

    이 규칙을 따르지 않는 메트릭은 W&amp;B UI에서 정렬하거나 필터링할 수 없을 수 있습니다.

    ## 예시

    **유효한 메트릭 이름:**

    ```python
    with wandb.init() as run:
      run.log({"accuracy": 0.9, "val_loss": 0.1, "epoch_5": 5})
      run.log({"modelAccuracy": 0.95, "learning_rate": 0.001})
    ```

    **유효하지 않은 메트릭 이름(다음은 피하십시오):**

    ```python
    with wandb.init() as run:
      run.log({"acc,val": 0.9})  # 쉼표 포함
      run.log({"loss-train": 0.1})  # 하이픈 포함
      run.log({"test acc": 0.95})  # 공백 포함
      run.log({"5_fold_cv": 0.8})  # 숫자로 시작
    ```

    ## 권장 해결 방법

    유효하지 않은 문자를 밑줄과 같은 유효한 문자로 대체하십시오:

    * `"test acc"` 대신 `"test_acc"` 사용
    * `"loss-train"` 대신 `"loss_train"` 사용
    * `"acc,val"` 대신 `"acc_val"` 사용

    자세한 내용은 [Metric naming constraints](/ko/models/track/log/#metric-naming-constraints)를 참조하십시오.
  </Accordion>

    <Tabs>

        ```python
        # Set default values for hyperparameters
        config_defaults = {"lr": 0.1, "batch_size": 256}

        # Start a run and provide defaults
        # that a sweep can override
        with wandb.init(config=config_defaults) as run:
            # Add training code here
            ...
        ```
      </Tab>

      <Tab title="config.setdefaults()">
        ```python
        # Set default values for hyperparameters
        config_defaults = {"lr": 0.1, "batch_size": 256}

        # Start a run
        with wandb.init() as run:
            # Update any values not set by the sweep
            run.config.setdefaults(config_defaults)

            # Add training code here
        ```
      </Tab>
    </Tabs>
  </Accordion>

    * **엔터티 및 프로젝트 이름 확인**: 코드에서 W&amp;B 엔터티와 프로젝트 이름의 철자와 대소문자가 정확한지 확인합니다.

      import wandb

      export WANDB_API_KEY='your_api_key'

<Update tags={["Anonymous"]}>
  <Accordion title="계정이 없는 사람은 실행 결과를 어떻게 볼 수 있나요?">
    누군가 `anonymous="allow"`로 스크립트를 실행하면:

    1. **임시 계정 자동 생성**: W&amp;B는 로그인된 계정이 있는지 확인합니다. 없으면 새 익명 계정을 만들고, 해당 세션을 위한 API 키를 저장합니다.
    2. **결과를 빠르게 로깅**: 사용자는 스크립트를 반복해서 실행하고, W&amp;B 대시보드에서 즉시 결과를 볼 수 있습니다. 이러한 소유자가 없는 익명 실행은 7일 동안 유지됩니다.
    3. **유용할 때 데이터 소유권 주장**: 사용자가 W&amp;B에서 유용한 결과를 찾으면, 페이지 상단 배너의 버튼을 클릭해 실행 데이터를 실제 계정으로 저장할 수 있습니다. 소유권을 주장하지 않으면 실행 데이터는 7일 후 삭제됩니다.

    <Warning>
      **익명 실행 링크는 민감한 정보입니다**. 이 링크를 가진 누구나 7일 동안 실험 결과를 보고 소유권을 주장할 수 있으므로, 신뢰할 수 있는 사람하고만 링크를 공유하세요. 작성자의 신원을 숨긴 채로 공개적으로 결과를 공유하려면 support@wandb.com으로 연락해 도움을 받으세요.
    </Warning>

    W&amp;B 사용자가 스크립트를 찾아 실행하면, 일반적인 실행과 마찬가지로 결과가 해당 사용자의 계정에 정상적으로 로깅됩니다.
  </Accordion>
</Update>

  <Accordion title="로그한 것보다 데이터 포인트가 적게 보이는 이유는 무엇인가요?">
    `Step`이 아닌 다른 X축에 대해 지표를 시각화할 경우, 데이터 포인트가 더 적게 보일 수 있습니다. 지표는 동일한 `Step`에서 로그되어야 동기화된 상태를 유지합니다. 샘플링할 때는 동일한 `Step`에서 로그된 지표만 사용하고, 샘플 사이를 보간합니다.

    **가이드라인**

    지표를 하나의 `log()` 호출에 묶어서 로그하세요. 예를 들어, 다음과 같이 하는 대신:

    ```python
    import wandb
    with wandb.init() as run:
        run.log({"Precision": precision})
        ...
        run.log({"Recall": recall})
    ```

    다음과 같이 사용하세요:

    ```python
    import wandb
    with wandb.init() as run:
        run.log({"Precision": precision, "Recall": recall})
    ```

    `step` 파라미터를 수동으로 제어하려면, 코드에서 다음과 같이 지표를 동기화하세요:

    ```python
    with wandb.init() as run:
        step = 100  # 예시 step 값
        # Precision과 Recall을 같은 step에서 로그합니다
        run.log({"Precision": precision, "Recall": recall}, step=step)
    ```

    두 `log()` 호출에서 `step` 값이 동일해야 지표가 같은 step 아래에 로그되고 함께 샘플링됩니다. `step` 값은 각 호출에서 단조 증가해야 합니다. 그렇지 않으면 `step` 값은 무시됩니다.
  </Accordion>

  <Accordion title="서비스 계정이란 무엇이며, 왜 유용한가요?">
    **서비스 계정**은 사람(사용자)이 아닌 머신 ID를 나타내며, 팀과 프로젝트 전반에 걸친 반복 작업을 자동화할 수 있습니다. 서비스 계정은 CI/CD 파이프라인, 자동화된 학습 작업, 그 밖의 머신 간 워크플로에 적합합니다.

    서비스 계정의 주요 이점:

    * **라이선스 소모 없음**: 서비스 계정은 사용자 시트나 라이선스를 소모하지 않습니다.
    * **전용 API 키**: 자동화된 워크플로를 위한 보안 자격 증명
    * **사용자 귀속**: 자동화된 실행을 사람 사용자에게 선택적으로 귀속 가능
    * **엔터프라이즈 준비 완료**: 대규모 프로덕션 자동화를 위해 설계됨
    * **위임된 작업 수행**: 서비스 계정은 이를 생성한 사용자 또는 조직을 대신하여 작업합니다.

    기타 여러 용도 중 하나로, 서비스 계정은 주기적인 재학습, 야간 빌드 등 wandb에 로깅되는 자동화된 작업을 추적하는 데 유용합니다. 원한다면, 이러한 머신에서 시작된 실행에 대해 [environment variables](/ko/models/track/environment-variables) `WANDB_USERNAME` 또는 `WANDB_USER_EMAIL`을 사용해 사용자 이름을 연결할 수 있습니다.

    서비스 계정에 대한 모범 사례와 상세 설정 방법을 포함한 종합적인 정보는 [Use service accounts to automate workflows](/ko/platform/hosting/iam/service-accounts)를 참고하세요. 팀 환경에서 서비스 계정이 어떻게 동작하는지에 대한 정보는 [Team Service Account Behavior](/ko/platform/app/settings-page/teams#team-service-account-behavior)를 참고하세요.

    <Note>

JSON Web Tokens (JWT)를 발급할 수 있는 ID 공급자에서 관리되는 서비스 ID를 사용해 W&amp;B 작업을 자동화하고자 하는 경우 External service accounts를 사용하세요.

    설정을 완료한 후, 애플리케이션의 `Client ID`와 `Issuer URL`을 고객 성공 매니저(CSM)에게 전달하세요. W&amp;B는 이 정보를 사용해 Auth0 연결을 구성하고 SSO를 활성화합니다.

  <Accordion title="코드를 테스트할 때 wandb를 끌 수 있나요?">
    테스트 목적을 위해 W&amp;B를 no-operation (NOOP, 아무 작업도 하지 않음) 모드로 설정하려면 `wandb.init(mode="disabled")`를 사용하거나 `WANDB_MODE=disabled`를 설정하세요.

    <Note>
      `wandb.init(mode="disabled")`를 사용해도 W&amp;B가 `WANDB_CACHE_DIR`에 아티팩트를 저장하는 것은 막지 못합니다.
    </Note>
  </Accordion>
</Update>

<Update tags={["Billing"]}>
  <Accordion title="결제 수단은 어떻게 업데이트하나요?">
    결제 수단을 업데이트하려면 다음 단계를 따르세요:

    1. **프로필 페이지로 이동**: 먼저 사용자 프로필 페이지로 이동합니다.
    2. **조직 선택**: 계정 선택기에서 관련 Organization을 선택합니다.
    3. **Billing 설정으로 이동**: **Account** 아래에서 **Billing**을 선택합니다.
    4. **새 결제 수단 추가**:
       * **Add payment method**를 클릭합니다.
       * 새 카드 정보를 입력하고, 이를 **primary** 결제 수단으로 설정하는 옵션을 선택합니다.

    > **참고:** 결제를 관리하려면 조직에서 결제 관리자(billing admin)로 지정되어 있어야 합니다.
  </Accordion>
</Update>

<Update tags={["Environment Variables", "Experiments"]}>
  <Accordion title="wandb.init은 내 학습 프로세스에 어떤 영향을 주나요?">
    학습 스크립트에서 `wandb.init()`이 실행되면, API 호출을 통해 서버에 실행(run) 객체가 생성됩니다. 새로운 프로세스가 시작되어 메트릭을 스트리밍하고 수집하며, 기본 프로세스는 정상적으로 동작할 수 있습니다. 스크립트는 로컬 파일에 기록하고, 별도의 프로세스가 시스템 메트릭을 포함한 데이터를 서버로 스트리밍합니다. 스트리밍을 끄려면 학습 디렉터리에서 `wandb off`를 실행하거나 `WANDB_MODE` 환경 변수를 `offline`으로 설정하세요.
  </Accordion>
</Update>

<Update tags={["Sweeps"]}>
  <Accordion title="Sweep이 실행 중일 때 Python 파일을 수정하면 어떻게 되나요?">
    Sweep이 실행 중일 때:

    * Sweep이 사용하는 `train.py` 스크립트가 변경되더라도, Sweep은 원래의 `train.py`를 계속 사용합니다.
    * `helper.py` 스크립트의 헬퍼 함수처럼 `train.py` 스크립트가 참조하는 파일이 변경되면, Sweep은 업데이트된 `helper.py`를 사용하기 시작합니다.
  </Accordion>
</Update>

<Update tags={["Artifacts", "Environment Variables"]}>
  <Accordion title="아티팩트는 어디에 다운로드되며, 어떻게 제어할 수 있나요?">
    기본적으로 아티팩트는 `artifacts/` 폴더에 다운로드됩니다. 위치를 변경하려면 다음과 같이 하세요:

    * [`wandb.Artifact().download`](/ko/models/ref/python/public-api/api)에 경로를 전달합니다:

      ```python
      wandb.Artifact().download(root="<path_to_download>")
      ```

    * `WANDB_ARTIFACT_DIR` [환경 변수](/ko/models/track/environment-variables)를 설정합니다:

      ```python
      import os
      os.environ["WANDB_ARTIFACT_DIR"] = "<path_to_download>"
      ```
  </Accordion>
</Update>

<Update tags={["Experiments", "Runs"]}>
  <Accordion title="CSV 메트릭 내보내기에서 스텝이 누락되는 이유는 무엇인가요?">
    내보내기 제한으로 인해 전체 실행 이력이 CSV로 내보내지 않거나 `run.history` API로 반환되지 않을 수 있습니다. 전체 실행 이력에 액세스하려면 Parquet 형식으로 실행 이력 아티팩트를 다운로드하세요:

    ```python
    import wandb
    import pandas as pd

    run = wandb.init()
    artifact = run.use_artifact('<entity>/<project>/<run-id>-history:v0', type='wandb-history')
    artifact_dir = artifact.download()
    df = pd.read_parquet('<path to .parquet file>')
    ```
  </Accordion>
</Update>
