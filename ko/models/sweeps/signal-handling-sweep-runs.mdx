이 페이지에서는 W&amp;B Sweeps가 시스템 시그널과 프로세스 종료 코드를 처리하는 방식에 대한 자세한 내용을 제공합니다. 이를 통해 SLURM, EC2 Spot, Google Cloud 선점형 VM과 같은 선점형 환경에서 스윕을 안정적으로 실행할 수 있습니다. 다음 섹션에서는 키보드로 run을 깔끔하게 중단하는 방법을 설명하고, run이 다시 큐에 들어가는 동작을 이해하고 예측하는 데 도움이 되는 세부 정보를 제공합니다. 선점되었을 때 run이 어떻게 다시 큐에 들어가는지에 대한 자세한 내용은 [선점형 Sweeps run 재개](/ko/models/runs/resuming#resume-preemptible-sweeps-runs)를 참조하세요.

<div id="exit-status-and-signals">
  ## 종료 상태와 시그널
</div>

W&amp;B는 트레이닝 프로세스의 종료 상태를 기준으로 run을 다시 큐에 넣을지, 그리고 run 상태를 어떻게 기록할지를 결정합니다.

**종료 코드 규약:**

* **종료 코드 0**: run이 성공적으로 완료된 것으로 간주되며 다시 큐에 들어가지 않습니다.
* **0이 아닌 종료 코드**: run은 실패했거나 선점된 것으로 처리됩니다. [`mark_preempting()`](/ko/models/ref/python/experiments/run#mark_preempting)을 사용하면 W&amp;B가 해당 run을 다시 큐에 넣으므로, 다른 에이전트(또는 재시작 후 동일한 에이전트)가 이를 재개할 수 있습니다.

이 규칙은 프로세스가 시그널 핸들러에서 종료되든, 예외로 종료되든, 명시적으로 `sys.exit()`를 호출해 종료되든 동일하게 적용됩니다. 이 규약을 이해하고 이를 전제로 동작하도록 구성하는 것은 선점형 환경이나 클러스터 환경에서 매우 중요합니다.

프로세스가 [**처리 가능한** 시그널](#catchable-signals-and-preemption)로 종료되는 경우, run을 다시 큐에 넣으려면 핸들러를 실행한 뒤 [`wandb.run.mark_preempting()`](/ko/models/ref/python/experiments/run#mark_preempting)을 호출하고, 정리 작업(예: 체크포인트 저장)을 수행한 후 0이 아닌 코드로 종료할 수 있습니다. 시그널로 인한 종료에는 일반적으로 `sys.exit(128 + signum)` 관례를 사용합니다. W&amp;B는 해당 종료 코드를 기록하며, 동일한 [requeue 규칙](/ko/models/runs/resuming#resume-preemptible-sweeps-runs)이 적용됩니다. 프로세스가 운영 체제 커널에 의해 [**`SIGKILL`**](#sigkill-uncatchable)로 종료되면 종료 훅을 실행할 수 없으므로 최종 summary가 기록되지 않으며, run이 crashed 또는 killed로 표시될 수 있습니다. 그래도 에이전트는 다음 run을 시작합니다.

<div id="stale-runs-and-server-side-timeouts">
  ## 유휴 run 및 서버 측 타임아웃
</div>

run이 오랫동안 완료되지 않고 새 메트릭을 기록하지 않으면(대략 5분 정도), W&amp;B Server는 해당 run을 **crashed**로 표시합니다. 이는 트레이닝 프로세스가 멈추거나, 메트릭 로깅이 중단되거나, 정상적으로 종료되지 않은 경우(예: `SIGKILL` 이후) 자주 발생합니다. 메트릭을 일정한 간격으로 기록하거나 정의된 종료 코드로 종료하면 Run 상태를 실제 발생한 상황에 맞게 유지하는 데 도움이 됩니다.

<div id="catchable-signals-and-preemption">
  ## 처리 가능한 시그널과 선점
</div>

트레이닝 스크립트에서 맞춤형 시그널 핸들러를 등록할 수 있습니다. 처리 가능한 시그널이 전달되면 핸들러가 실행되며, 이미 W&amp;B로 전송된 메트릭은 유지됩니다. 또한 에이전트가 프로세스 종료를 감지하고 다음 run을 시작합니다.

**모범 사례:**

* 핸들러는 가능한 한 초기에 등록하세요(예: 기본 트레이닝 루프에 들어가기 전).
* 핸들러에서는 선점 후 run이 다시 큐에 들어가야 하는 경우 [`wandb.run.mark_preempting()`](/ko/models/ref/python/experiments/run#mark_preempting)을 호출하고, 정리 작업(예: 체크포인트 저장)을 수행한 다음, 0이 아닌 종료 코드로 종료하세요.

다음 예제에서는 `SIGUSR1`(클러스터에서 선점 시 일반적으로 사용하는 시그널)과 `SIGTERM`에 대한 핸들러를 등록합니다. `SIGINT`는 대화형 용도(예: 터미널에서 수동으로 취소)에 사용할 수 있도록 그대로 둡니다. 핸들러는 `wandb.run.mark_preempting()`을 호출한 뒤 `128 + signum`으로 종료합니다:

```python
import signal
import sys
import wandb


def signal_handler(signum, frame):
    if wandb.run is not None:
        # 선택 사항: 모델 체크포인트 저장, 버퍼 플러시 등.
        print(f"Preempted with signal: {signal.Signals(signum).name}.")
        wandb.run.mark_preempting()
    sys.exit(128 + signum)


def train():
    signal.signal(signal.SIGUSR1, signal_handler)
    signal.signal(signal.SIGTERM, signal_handler)

    with wandb.init() as run:
        config = wandb.config
        for epoch in range(100):
            # 트레이닝 step; 필요에 따라 wandb.log(...) 호출
            pass


if __name__ == "__main__":
    train()
```

<div id="sigkill-uncatchable">
  ## `SIGKILL` (포착 불가)
</div>

`SIGKILL`은 포착하거나 무시할 수 없습니다. 프로세스는 핸들러나 atexit 콜백을 실행할 기회 없이 즉시 종료됩니다. W&amp;B는 해당 run에 대한 최종 summary를 기록할 수 없습니다. 에이전트는 여전히 복구되어 스윕을 계속하지만, 해당 run의 데이터는 불완전합니다. `SIGKILL`은 최후의 수단으로만 사용하세요. 정상적인 종료가 필요할 때는 `SIGTERM` 또는 `SIGINT`를 우선 사용하세요.

<div id="forwarding-signals-from-agent-to-child">
  ## 에이전트에서 자식 프로세스로 시그널 전달
</div>

[`wandb agent`](/ko/models/ref/cli/wandb-agent) CLI를 사용하면 에이전트가 트레이닝 스크립트를 **자식 프로세스**로 실행합니다. **에이전트**가 중단되면(예: Ctrl+C를 누르거나 스케줄러가 작업에 `SIGTERM`을 보낼 때) **자식**(트레이닝 프로세스)은 기본적으로 시그널을 받지 못합니다. 따라서 트레이닝 스크립트는 핸들러를 실행하거나 `mark_preempting()`을 호출할 수 없습니다. 이 내용은 [GitHub #3667](https://github.com/wandb/wandb/issues/3667)에 설명되어 있습니다.

자식 프로세스가 정상적으로 종료되고 핸들러에서 `wandb.run.mark_preempting()`을 호출할 수 있도록 하려면, CLI 에이전트를 `--forward-signals`와 함께 실행하세요:

```bash
wandb agent --forward-signals entity/project/sweep_ID
```

Python API의 [`wandb.agent()`](/ko/models/ref/python/functions/agent)에서는 시그널 포워딩이 **지원되지 않습니다**. 이 방식에서는 트레이닝 함수를 별도의 자식 프로세스가 아니라 스레드에서 실행하므로, 동일한 포워딩 동작은 적용되지 않습니다.

CLI 에이전트가 포워딩이 활성화된 상태에서 `SIGINT` 또는 `SIGTERM`을 받으면, 해당 시그널을 자식 프로세스로 전달합니다. 그러면 트레이닝 스크립트의 핸들러가 실행되어 필요에 따라 `wandb.run.mark_preempting()`과 [`wandb.finish()`](/ko/models/ref/python/experiments/run#finish)를 0이 아닌 종료 코드와 함께 호출하고, 0이 아닌 코드로 종료할 수 있습니다. 에이전트 프로세스에서 Ctrl+C를 두 번 누르면 기본적으로 에이전트는 `SIGTERM`을 받습니다. `--forward-signals`를 사용하면 `SIGINT`를 자식 프로세스로 전달하여 핸들러가 실행되도록 할 수 있습니다.

자세한 내용은 [wandb agent](/ko/models/ref/cli/wandb-agent) CLI 레퍼런스를 참조하세요.

<div id="preemptible-clusters-like-slurm">
  ## `SLURM`과 같은 선점형 클러스터
</div>

선점이 발생하면 **트레이닝 프로세스**가 시그널을 받아 run을 선점 중으로 표시하고, run이 다시 큐에 들어가도록 0이 아닌 코드로 종료해야 합니다. 그러면 새 에이전트(또는 작업이 다시 큐에 들어간 뒤 동일한 에이전트)가 해당 run을 재개할 수 있습니다.

**트레이닝 프로세스가 시그널을 받도록 하세요:**

1. **스케줄러가 에이전트에 시그널을 보내는 경우**: `wandb agent --forward-signals`로 에이전트를 실행하세요. 그러면 스케줄러(또는 사용자)가 에이전트에 시그널을 보낼 때 에이전트가 그 시그널을 자식 프로세스에 전달합니다. 그러면 자식 프로세스의 핸들러에서 `wandb.run.mark_preempting()`와 [`wandb.finish(exit_code=...)`](/ko/models/ref/python/experiments/run#finish)를 0이 아닌 코드와 함께 호출하고, `sys.exit(128 + signum)`(또는 다른 0이 아닌 종료 코드)을 호출할 수 있습니다.
2. **스케줄러가 시작 스크립트에 시그널을 보내는 경우(에이전트에 직접 보내지 않음)**: 시작 스크립트가 선점 시그널을 트레이닝 프로세스에 직접 보내도록 하세요. 예를 들어, 트레이닝 스크립트가 자신의 프로세스 ID를 파일에 기록하고, 시작 스크립트가 클러스터 시그널(예: `SIGUSR1`)를 트랩한 다음 `kill -SIGUSR1 $(cat $PID_FILE)`를 실행해 트레이닝 프로세스의 핸들러가 실행되도록 할 수 있습니다.

**트레이닝 스크립트에서:** 클러스터에서 사용하는 시그널(예: `SIGTERM` 또는 `SIGUSR1`)에 대한 핸들러를 등록하세요. 핸들러에서는 활성 run이 있으면 `wandb.run.mark_preempting()`를 호출한 다음, run이 다시 큐에 들어가도록 0이 아닌 종료 코드와 `sys.exit(128 + signum)`(또는 다른 0이 아닌 코드)로 run을 종료하세요. run이 언제 다시 큐에 들어가는지와 이것이 `mark_preempting()`과 어떻게 상호작용하는지는 [Resume preemptible Sweeps runs](/ko/models/runs/resuming#resume-preemptible-sweeps-runs)을 참조하세요.

**스윕 상태:** 에이전트를 시작하기 전에 `wandb sweep entity/project/sweep_ID --resume`를 실행해 스윕이 재개 모드가 되도록 하고, 다시 큐에 들어간 run을 할당하도록 하세요.

**다중 에이전트 조정:** 많은 에이전트가 동시에 실행되면(SLURM 배열 작업 등) 동일한 선점된 run을 가져가려고 경합할 수 있습니다. 이는 알려진 제한 사항입니다. 이 잠재적인 문제를 완화하려면 에이전트 시작 시점을 엇갈리게 하거나 lock과 같은 외부 조정 메커니즘을 사용하세요.

<div id="wandb-sweep-cancel">
  ## `wandb sweep --cancel`
</div>

스윕은 OS 시그널이 아니라 W&amp;B API를 사용해 취소합니다. `wandb sweep --cancel entity/project/sweep_ID`와 같은 명령을 실행하세요. 서버가 에이전트에 종료하도록 지시하면, 에이전트는 실행 중인 자식 프로세스를 종료한 뒤 중지됩니다. 취소가 적용되기까지는 잠시 지연될 수 있습니다(대략 에이전트의 API 폴링 간격 정도).

취소 시 run에 **`SIGKILL`**이 전달됩니다. 자식 프로세스는 사용자 정의 시그널 핸들러를 실행할 기회가 없습니다. 이는 Sweeps UI의 **Cancel** 컨트롤을 사용할 때도 마찬가지입니다. 전체 스윕을 중지하고 취소됨으로 표시하려면 `--cancel`을 사용하세요. 현재 run을 정상 종료하려면 run에 처리 가능한 시그널을 보내세요(또는 CLI 에이전트에서 `--forward-signals`를 사용한 다음 에이전트에 시그널을 보내세요). 스윕을 정상적으로 완료하려면 `--cancel` 대신 [`wandb sweep --stop`](/ko/models/sweeps/pause-resume-and-cancel-sweeps#stop-a-sweep)을 사용하세요.

일시 중지, 재개, 중지, 취소 옵션은 [Manage sweeps](/ko/models/sweeps/pause-resume-and-cancel-sweeps)를 참조하세요.

<div id="killing-the-agent-vs-the-run">
  ## 에이전트 종료와 run 종료의 차이
</div>

**에이전트** 프로세스에 시그널을 보내면(자식 트레이닝 프로세스가 아니라) 에이전트는 종료될 수 있지만 자식 프로세스는 고아 프로세스로 계속 실행될 수 있습니다. 고아 프로세스가 터미널에 계속 출력할 수 있으며, Enter를 누르기 전까지 셸에 새 프롬프트가 표시되지 않을 수 있습니다.

CLI 에이전트와 함께 `--forward-signals`를 사용하지 않는 한, 에이전트를 중지해도 자식 트레이닝 프로세스가 중지된다고 보장할 수는 없습니다.

에이전트가 종료되었는지 확인하려면 프롬프트 표시 여부에 의존하지 말고 `ps -p <agent_pid>` 또는 `pgrep -f "wandb agent"` 같은 OS 명령어를 대신 사용하세요.

<div id="reference-mark_preempting-and-final-run-state">
  ## 레퍼런스: `mark_preempting()` 및 최종 run 상태
</div>

아래 표는 `mark_preempting()`을 **언제** 호출하는지와 프로세스가 어떻게 종료되는지에 따라 run 상태가 어떻게 달라지는지 요약합니다. 이 표는 트레이닝 프로그램을 서브프로세스로 사용하는 [`wandb agent`](/ko/models/ref/cli/wandb-agent) CLI를 사용한다고 가정합니다.

| 시나리오                                        | `mark_preempting()` 호출하지 않음 | 시그널 핸들러에서 `mark_preempting()`을 호출한 뒤 0이 아닌 코드로 종료 | `init()` 직후 항상 `mark_preempting()` 호출 |
| ------------------------------------------- | --------------------------- | ------------------------------------------------- | ------------------------------------- |
| run이 종료 코드 0으로 정상적으로 완료됨                    | FINISHED                    | FINISHED                                          | FINISHED                              |
| run이 0이 아닌 종료 코드로 실패함                       | FAILED                      | FAILED                                            | PREEMPTED                             |
| run이 `SIGKILL`을 수신함                         | 약 5분 후 CRASHED              | 약 5분 후 CRASHED (포착 불가)                            | 약 5분 후 PREEMPTED                      |
| run이 `SIGINT`를 수신함                          | KILLED                      | PREEMPTED (`SIGINT` 핸들러가 있는 경우)                   | PREEMPTED                             |
| run이 다른 시그널(예: `SIGTERM` 또는 `SIGUSR1`)을 수신함 | 약 5분 후 CRASHED              | PREEMPTED (해당 시그널용 핸들러가 있는 경우)                    | 약 5분 후 PREEMPTED                      |

시그널 핸들러 내부에서만 `mark_preempting()`을 호출하면 `SIGKILL`처럼 핸들러가 전혀 실행되지 않는 경우까지는 처리할 수 없습니다.

항상 `wandb.init()` 직후에 `mark_preempting()`을 호출하면 모든 실패가 선점으로 처리될 수 있으므로, 버그나 잘못된 설정의 경우에도 run이 반복해서 다시 큐에 들어갈 수 있습니다.

선점 시그널이 명확히 정의된 환경에서는 일반적으로 `init()` 직후 무조건 호출하는 방식이 아니라, `mark_preempting()`을 호출한 뒤 0이 아닌 코드로 종료하는 **시그널 핸들러**를 사용합니다.
