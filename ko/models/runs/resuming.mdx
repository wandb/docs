title: run 재개
---

`wandb.init()`에서 `resume` 파라미터를 설정해 run이 중지되거나 크래시될 경우 W&amp;B가 어떻게 응답할지 지정합니다. run을 초기화하면 W&amp;B는 run ID가 이미 존재하는지 확인하고, `resume` 값에 따라 정의된 동작을 적용합니다.

다음 표는 `resume` 파라미터에 전달된 인수와 run ID의 존재 여부에 따라 W&amp;B의 동작을 보여줍니다.

|Argument | Description | Run ID exists| Run ID does not exist | Use case |
| --- | --- | -- | --| -- |
| `"must"` | W&amp;B는 run ID로 지정된 run을 반드시 재개해야 합니다. | W&amp;B는 동일한 run ID로 run을 재개합니다. 마지막 step부터 재개합니다. | W&amp;B는 오류를 발생시킵니다. | 반드시 동일한 run ID를 사용해야 하는 run을 재개할 때. |
| `"allow"`| run ID가 존재하면 W&amp;B가 run을 재개하도록 허용합니다. | W&amp;B는 동일한 run ID로 run을 재개합니다. 마지막 step부터 재개합니다. | W&amp;B는 지정된 run ID로 새 run을 초기화합니다. | 기존 run을 덮어쓰지 않고 run을 재개할 때. |
| `"never"`| W&amp;B가 run ID로 지정된 run을 절대 재개하지 않도록 합니다. | 지정된 ID를 가진 run이 이미 존재하면 오류를 발생시킵니다. | W&amp;B는 지정된 run ID로 새 run을 초기화합니다. | |
| `"auto"`| run ID가 존재하면 W&amp;B가 자동으로 run 재개를 시도하도록 허용합니다. 실패한 프로세스와 동일한 디렉터리에서 run을 다시 시작합니다. | W&amp;B는 동일한 run ID로 run을 재개합니다. | W&amp;B는 지정된 run ID로 새 run을 초기화합니다. | run이 자동으로 재개되도록 설정할 때. |

<Note>
  **`auto`와 `allow`를 언제 사용해야 하나요?**

  W&amp;B는 `resume="allow"`를 사용하고, 재개하려는 특정 run ID를 지정할 것을 권장합니다.

  `resume="auto"` 옵션은 run ID를 지정할 필요가 없지만, 동일한 디렉터리에서 여러 run이 실패했거나 파일 디렉터리 구조가 변경된 경우 예기치 않은 동작이 발생할 수 있습니다. 또한 `resume="auto"`를 사용할 때는 실패한 프로세스와 동일한 디렉터리에서 run을 다시 시작해야 합니다.
</Note>

아래의 모든 예제에서 `<>`로 둘러싸인 값은 실제 값으로 바꾸세요.
<Tip>[재개된 run의 라이브 데모 보기](https://wandb.ai/wandb/resume-run/workspace?nw=nwuserjuliarose).</Tip>

<div id="resume-a-run-that-must-use-the-same-run-id">
  ## 동일한 run ID를 사용해야 하는 run 재개하기
</div>

run이 중지되거나 크래시가 발생하거나 실패한 경우, 동일한 run ID를 사용해 이를 재개할 수 있습니다. 그러려면 run을 초기화하고 다음을 지정하세요:

* `resume` 파라미터를 `"must"`로 설정합니다 (`resume="must"`)
* 중단되었거나 크래시된 run의 run ID를 제공합니다

다음 코드 스니펫은 W&amp;B Python SDK로 이를 수행하는 방법을 보여 줍니다:

```python
with wandb.init(entity="<entity>", project="<project>", resume="must") as run:
        # Your training code here
```

<Warning>
  여러 프로세스가 동시에 동일한 `id`를 사용하면 예기치 않은 결과가 발생할 수 있습니다.

  여러 프로세스를 관리하는 방법에 대한 자세한 내용은 [분산 트레이닝 실험 로깅](/ko/models/track/log/distributed-training/)을 참조하세요.
</Warning>

<div id="resume-a-run-without-overriding-the-existing-run">
  ## 기존 run을 덮어쓰지 않고 재개하기
</div>

중단되었거나 크래시된 run을 기존 run을 덮어쓰지 않고 재개할 수 있습니다. 이는 프로세스가 정상적으로 종료되지 않았을 때 특히 유용합니다. 다음 번에 W&amp;B를 시작하면 W&amp;B는 마지막 스텝부터 로깅을 다시 시작합니다.

W&amp;B에서 run을 초기화할 때 `resume` 파라미터를 `"allow"` (`resume="allow"`)로 설정하세요. 중단되었거나 크래시된 run의 run ID를 제공해야 합니다. 다음 코드 스니펫은 W&amp;B Python SDK로 이를 수행하는 방법을 보여 줍니다:

```python
import wandb

with wandb.init(entity="<entity>", project="<project>", id="<run ID>", resume="allow") as run:
        # 트레이닝 코드를 여기에 작성하세요
```

<div id="enable-runs-to-automatically-resume">
  ## run을 자동으로 재개하도록 설정하기
</div>

다음 코드 예시는 Python SDK 또는 환경 변수를 사용해 run을 자동으로 재개하도록 설정하는 방법을 보여줍니다.

<Tabs>
  <Tab title="W&B Python SDK">
    run을 초기화할 때 `resume` 매개변수에 `auto`를 인수로 전달하십시오. 실패한 프로세스와 동일한 디렉터리에서 run을 다시 시작해야 합니다.

    다음 코드 예시를 복사하여 붙여넣으십시오. `<>`로 둘러싸인 값을 모두 사용자 환경에 맞게 바꾸십시오:

    ```python
    with wandb.init(entity="<entity>", project="<project>", id="<run ID>", resume="auto") as run:
            # 여기에 트레이닝 코드를 작성합니다
    ```
  </Tab>

  <Tab title="Shell script">
    다음 예시는 bash 스크립트에서 W&amp;B `WANDB_RUN_ID` 변수를 지정하는 방법을 보여줍니다:

    ```bash title="run_experiment.sh"
    RUN_ID="$1"

    WANDB_RESUME=auto WANDB_RUN_ID="$RUN_ID" python eval.py
    ```

    터미널에서 W&amp;B run ID와 함께 셸 스크립트를 실행할 수 있습니다. 다음 코드 예시는 run ID `akj172`를 전달합니다:

    ```bash
    sh run_experiment.sh akj172 
    ```
  </Tab>
</Tabs>

<Warning>
  자동 재개는 실패한 프로세스와 동일한 파일 시스템에서 프로세스를 다시 시작하는 경우에만 동작합니다.
</Warning>

예를 들어 `Users/AwesomeEmployee/Desktop/ImageClassify/training/` 디렉터리에서 `train.py`라는 Python 스크립트를 실행한다고 가정해 보겠습니다. `train.py` 스크립트는 자동 재개가 가능하도록 run을 생성합니다. 이후 트레이닝 스크립트가 중지되었다면, 이 run을 재개하려면 `Users/AwesomeEmployee/Desktop/ImageClassify/training/` 디렉터리에서 다시 `train.py` 스크립트를 실행해야 합니다.

<Note>
  파일 시스템을 공유할 수 없다면 `WANDB_RUN_ID` 환경 변수를 지정하거나 W&amp;B Python SDK를 사용해 run ID를 전달하십시오. run ID에 대한 자세한 내용은 &quot;What are runs?&quot; 페이지의 [Custom run IDs](./#custom-run-ids) 섹션을 참고하십시오.
</Note>

<div id="resume-preemptible-sweeps-runs">
  ## 선점형 Sweeps run 재개
</div>

중단된 [sweep](/ko/models/sweeps/) run을 자동으로 다시 큐에 넣습니다. 이는 선점이 발생할 수 있는 컴퓨팅 환경(예: 선점형 큐의 SLURM 작업, EC2 스팟 인스턴스, Google Cloud 선점형 VM)에서 sweep 에이전트를 실행하는 경우에 특히 유용합니다.

중단된 sweep run을 자동으로 다시 큐에 넣으려면 [`mark_preempting`](/ko/models/ref/python/experiments/run#mark_preempting) 함수를 사용하세요. 예를 들면 다음과 같습니다.

```python
with wandb.init() as run:
    run.mark_preempting()
```

다음 표는 sweep run의 종료 상태에 따라 W&amp;B가 run을 어떻게 처리하는지 설명합니다.

|Status| Behavior |
|------| ---------|
|Status code 0| run이 성공적으로 종료된 것으로 간주되며 다시 큐에 들어가지 않습니다.|
|Nonzero status| W&amp;B는 해당 run을 sweep에 연결된 run 큐에 자동으로 추가합니다.|
|No status| run이 sweep run 큐에 추가됩니다. Sweep 에이전트는 큐가 빌 때까지 run 큐에서 run을 가져옵니다. 큐가 비면, sweep 큐는 sweep 검색 알고리즘을 기반으로 새 run 생성을 재개합니다.|
