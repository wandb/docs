프로젝트 또는 레지스트리 내에서 특정 이벤트가 발생하면 자동화를 시작할 수 있습니다. 이 페이지에서는 각 범위 내에서 자동화를 트리거할 수 있는 이벤트를 설명합니다. 자동화에 대한 자세한 내용은 [Automations Overview](/ko/models/automations) 또는 [자동화 만들기](/ko/models/automations/create-automations)에서 확인하세요.

<div id="registry">
  ## 레지스트리
</div>

이 섹션에서는 [레지스트리](/ko/models/registry) 자동화의 스코프와 이벤트를 설명합니다.

<div id="scopes">
  ### 스코프
</div>

[레지스트리](/ko/models/registry) 자동화는 향후 추가되는 컬렉션을 포함하여, 특정 레지스트리 내의 모든 컬렉션에서 발생하는 이벤트를 감시합니다.

### 이벤트 <a id="registry-events" aria-label="레지스트리 이벤트" />

레지스트리 자동화는 다음 이벤트를 감시할 수 있습니다:

* **새 버전이 collection에 링크됨**: 새 모델 또는 데이터셋이 레지스트리에 추가되면 이를 테스트하고 검증합니다.
* **아티팩트 별칭이 추가됨**: 새 아티팩트 버전에 특정 별칭이 적용되면 워크플로의 특정 step을 트리거합니다. 예를 들어, 모델에 `프로덕션` 별칭이 적용되면 배포합니다.

<div id="project">
  ## 프로젝트
</div>

이 섹션에서는 [프로젝트](/ko/models/track/project-page)의 자동화 스코프와 이벤트를 설명합니다.

<div id="scopes">
  ### 스코프
</div>

프로젝트 수준 자동화는 프로젝트의 모든 컬렉션에서 발생하는 이벤트를 감시합니다. 지정한 이벤트에 따라 자동화의 스코프를 더 제한할 수 있습니다.

<div id="artifact-events">
  ### 아티팩트 이벤트
</div>

이 섹션에서는 자동화를 트리거할 수 있는 아티팩트 관련 이벤트를 설명합니다.

* **아티팩트에 새 버전이 추가됨**: 아티팩트의 각 버전에 반복 액션을 적용합니다. 예를 들어, 새 데이터셋 아티팩트 버전이 생성되면 트레이닝 작업을 시작합니다.
* **아티팩트 별칭이 추가됨**: 프로젝트의 아티팩트 버전에, 지정한 **Alias regex**와 일치하는 별칭이 적용되면 워크플로의 특정 step을 트리거합니다. 예를 들어, 아티팩트에 `test-set-quality-check` 별칭이 적용되면 일련의 다운스트림 처리 step을 실행하거나, 새 아티팩트 버전에 `latest` 별칭이 있을 때마다 워크플로를 실행할 수 있습니다. 특정 시점에는 하나의 아티팩트 버전에만 해당 별칭을 가질 수 있습니다.
* **아티팩트 태그가 추가됨**: 프로젝트의 아티팩트 버전에, 지정한 **Tag regex**와 일치하는 태그가 적용되면 워크플로의 특정 step을 트리거합니다. 예를 들어, `^europe.*`를 지정하면 `europe` 문자열로 시작하는 태그가 아티팩트 버전에 추가될 때 지역별 워크플로를 트리거할 수 있습니다. 아티팩트 태그는 그룹화와 필터링에 사용되며, 하나의 태그를 여러 아티팩트 버전에 동시에 부여할 수 있습니다.

<div id="run-events">
  ### Run 이벤트
</div>

자동화는 [run의 status](/ko/models/runs/run-states) 변경 또는 [metric 값](/ko/models/track/log#what-data-is-logged-with-specific-wb-api-calls) 변경으로 트리거될 수 있습니다.

<div id="run-status-change">
  #### Run 상태 변경
</div>

<Note>
  * <MultiTenantCloudOnly />
  * **Killed** 상태인 run은 자동화를 트리거할 수 없습니다. 이 상태는 관리자 사용자가 run을 강제로 중지했음을 나타냅니다.
</Note>

run의 [상태](/ko/models/runs/run-states)가 **Running**, **Finished** 또는 **Failed**로 변경될 때 워크플로를 트리거하세요. 선택적으로 사용자 또는 run name 필터를 지정해 자동화를 트리거할 수 있는 run을 더 제한할 수 있습니다.

<Frame>
  ![run 상태 변경 자동화를 보여주는 스크린샷](/images/automations/run_status_change.png)
</Frame>

run 상태는 전체 run의 속성이므로, run 상태 자동화는 Workspace가 아니라 **Automations** 페이지에서만 생성할 수 있습니다.

<div id="run-metrics-change">
  #### Run 메트릭 변경
</div>

<Note>
  <MultiTenantCloudOnly />
</Note>

Run 이력의 메트릭이나 CPU 사용량 비율을 추적하는 `cpu`와 같은 [시스템 메트릭](/ko/models/ref/python/experiments/system-metrics)에 로깅된 값을 기준으로 워크플로를 트리거하세요. W&amp;B는 시스템 메트릭을 15초마다 자동으로 로깅합니다.

프로젝트의 **Automations** 탭이나 Workspace의 선형 플롯 패널에서 직접 Run 메트릭 자동화를 만들 수 있습니다.

Run 메트릭 자동화를 설정하려면 메트릭 값을 지정한 임곗값과 어떻게 비교할지 설정합니다. 사용할 수 있는 옵션은 이벤트 유형과 지정한 필터에 따라 달라집니다.

선택적으로 사용자 또는 run name 필터를 지정하여 자동화를 트리거할 수 있는 Runs를 추가로 제한할 수 있습니다.

<div id="threshold">
  ##### 임곗값
</div>

**Run 메트릭 임곗값 충족** 이벤트의 경우, 다음을 설정합니다:

1. 고려할 가장 최근에 로깅된 값의 범위(기본값은 5).
2. 범위 내에서 **Average**, **Min**, 또는 **Max** 값을 평가할지 여부.
3. 수행할 비교:
   * Above
   * Above or equal to
   * Below
   * Below or equal to
   * Not equal to
   * Equal to

예를 들어, 평균 `accuracy`가 `.6`보다 크면 자동화를 트리거하세요.

<Frame>
  ![Run 메트릭 임곗값 자동화를 보여주는 스크린샷](/images/automations/run_metrics_threshold_automation.png)
</Frame>

<div id="change-threshold">
  ##### 변화 임곗값
</div>

**Run metrics change threshold met** 이벤트의 경우, 자동화는 시작 여부를 확인하기 위해 두 개의 값 &quot;window&quot;를 사용합니다.

* 고려할 최근 로깅된 값의 *현재 window* (기본값은 10)
* 고려할 최근 로깅된 값의 *이전 window* (기본값은 50)

현재 window와 이전 window는 연속되며 서로 겹치지 않습니다.

자동화를 생성하려면 다음을 설정합니다.

1. 현재 window의 로깅된 값 (기본값은 10)
2. 이전 window의 로깅된 값 (기본값은 50)
3. 값을 상대값으로 평가할지 절댓값으로 평가할지 여부 (기본값은 **Relative**)
4. 수행할 비교:
   * 최소한 다음 값만큼 증가
   * 최소한 다음 값만큼 감소
   * 최소한 다음 값만큼 증가 또는 감소

예를 들어, 평균 `loss`가 최소 `.25`만큼 감소할 때 자동화를 트리거합니다.

<Frame>
  ![run 메트릭 변화 임곗값 자동화를 보여주는 스크린샷](/images/automations/run_metrics_change_threshold_automation.png)
</Frame>

<div id="run-metrics-z-score-change">
  #### Run metrics z-score change
</div>

<Note>
  <MultiTenantCloudOnly />
</Note>

W&amp;B는 메트릭의 z-score(표준 점수)가 지정된 임곗값을 초과하면 자동화를 트리거할 수 있습니다. z-score는 프로젝트에서 구성 가능한 Runs 범위(기본값: run 30개) 전반에 걸쳐 해당 메트릭의 평균으로부터 값이 표준편차 몇 개만큼 떨어져 있는지를 나타냅니다.

z-score를 이벤트 트리거로 사용하려면 **Run metrics z-score threshold met** 이벤트를 선택하세요.

z-score 기준으로 작동하는 자동화는 절대 임곗값을 일일이 확인하지 않아도 비정상적인 성능 변화를 팀에 알려줍니다. 이러한 임곗값은 모델이나 트레이닝 프로세스가 발전함에 따라 달라질 수 있습니다.

프로젝트의 **Automations** 탭이나 Workspace의 선형 플롯 패널에서 직접 run 메트릭 z-score 자동화를 만들 수 있습니다.

z-score 자동화를 만들려면 다음을 설정합니다.

1. 양의 부동소수점 값(예: 2.0)으로 표현되는 대상 z-score 임곗값
2. 평균값을 계산하는 데 사용할 로깅된 값의 범위(기본값: 30)
3. 수행할 비교:
   * Above(성능이 비정상적으로 높을 때 트리거)
   * Below(성능이 비정상적으로 낮을 때 트리거)
   * Above 또는 Below 중 하나

예를 들어 `accuracy`의 z-score가 2를 초과할 때 자동화를 트리거하도록 설정할 수 있으며, 이는 해당 run이 프로젝트의 다른 run보다 훨씬 더 좋은 성능을 보이고 있음을 의미합니다.

**z-score 값 이해하기:**

* z-score가 0이면 메트릭이 평균 수준임을 의미합니다.
* z-score가 +2.0이면 메트릭이 평균보다 표준편차 2만큼 높음을 의미합니다.
* z-score가 -2.0이면 메트릭이 평균보다 표준편차 2만큼 낮음을 의미합니다.
* ±2를 벗어나는 값은 통계적으로 유의미한 이상값으로 간주되는 경우가 많습니다.

<div id="run-filters">
  #### Run 필터
</div>

이 섹션에서는 자동화가 평가할 Runs를 어떻게 선택하는지 설명합니다.

* 기본적으로 프로젝트의 모든 run은 이벤트가 발생하면 자동화를 트리거합니다. 다음 필터 중 하나를 설정하여 어떤 run이 자동화를 트리거할지 제한할 수 있습니다.

  * **한 사용자의 Runs로 필터링**: 지정된 사용자가 생성한 Runs만 포함합니다.
  * **run name으로 필터링**: 이름이 지정된 정규식과 일치하는 Runs만 포함합니다.

  자세한 내용은 [자동화 생성](/ko/models/automations/create-automations)을 참조하세요.
* 각 run은 개별적으로 고려되며 잠재적으로 자동화를 트리거할 수 있습니다.
* 각 run의 값은 별도의 window에 배치되고 임곗값과도 별도로 비교됩니다.
* 24시간 동안 특정 자동화는 run당 최대 한 번만 실행될 수 있습니다.
