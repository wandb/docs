자동화는 프로젝트 또는 Registry 내에서 특정 이벤트가 발생할 때 시작되도록 설정할 수 있습니다. 이 페이지에서는 각 스코프에서 자동화를 트리거할 수 있는 이벤트를 설명합니다. 자동화에 대해 더 알아보려면 [Automations 개요](/ko/models/automations) 또는 [자동화 생성](/ko/models/automations/create-automations)을 참고하세요.

<div id="registry">
  ## Registry
</div>

이 섹션에서는 [Registry](/ko/models/registry)에서 자동화에 사용되는 스코프와 이벤트를 설명합니다.

<div id="scopes">
  ### 범위
</div>

[Registry](/ko/models/registry) 자동화는 특정 레지스트리 내의 모든 컬렉션에서 발생하는 이벤트를 모니터링하며, 이후에 추가되는 컬렉션도 포함합니다.

### 이벤트 <a id="registry-events" aria-label="Registry events" />

Registry 자동화는 다음 이벤트를 감지할 수 있습니다:

* **새 버전이 컬렉션에 연결됨**: Registry에 새 모델이나 데이터셋이 추가될 때 이를 테스트하고 검증합니다.
* **아티팩트 별칭이 추가됨**: 새 아티팩트 버전에 특정 별칭이 적용되었을 때 워크플로의 특정 단계를 트리거합니다. 예를 들어 `production` 별칭이 적용되면 모델을 배포합니다.

<div id="project">
  ## 프로젝트
</div>

이 섹션에서는 [프로젝트](/ko/models/track/project-page) 내 자동화의 범위와 이벤트를 설명합니다.

<div id="scopes">
  ### 범위
</div>

프로젝트 수준 자동화는 프로젝트 내 모든 컬렉션에서 발생하는 이벤트를 모니터링합니다. 지정한 이벤트에 따라 자동화의 범위를 더 좁힐 수 있습니다.

<div id="artifact-events">
  ### Artifact 이벤트
</div>

이 섹션에서는 자동화를 트리거하는 Artifact 관련 이벤트를 설명합니다.

* **새 버전이 아티팩트에 추가됨**: 아티팩트의 각 버전에 반복 작업을 적용합니다. 예를 들어, 새로운 데이터셋 아티팩트 버전이 생성될 때 학습 작업을 시작합니다.
* **아티팩트 별칭(alias)이 추가됨**: 프로젝트의 새로운 아티팩트 버전에, 지정한 **Alias regex**와 일치하는 별칭이 적용될 때 워크플로의 특정 단계를 트리거합니다. 예를 들어, 아티팩트에 `test-set-quality-check` 별칭이 적용되면 후속 처리 단계를 일련으로 실행하거나, 새로운 아티팩트 버전에 `latest` 별칭이 추가될 때마다 워크플로를 실행하도록 설정할 수 있습니다. 동시에 하나의 아티팩트 버전에만 특정 별칭을 사용할 수 있습니다.
* **아티팩트 태그가 추가됨**: 프로젝트의 아티팩트 버전에, 지정한 **Tag regex**와 일치하는 태그가 적용될 때 워크플로의 특정 단계를 트리거합니다. 예를 들어, `^europe.*`를 지정해 `europe` 문자열로 시작하는 태그가 아티팩트 버전에 추가될 때 지역별(geo-specific) 워크플로를 트리거할 수 있습니다. 아티팩트 태그는 그룹화와 필터링에 사용되며, 하나의 태그를 여러 아티팩트 버전에 동시에 할당할 수 있습니다.

<div id="run-events">
  ### 실행 이벤트
</div>

자동화는 [실행 상태](/ko/models/runs/run-states)가 변경되거나 [메트릭 값](/ko/models/track/log#what-data-is-logged-with-specific-wb-api-calls)이 변경될 때 트리거될 수 있습니다.

<div id="run-status-change">
  #### 실행 상태 변경
</div>

<Note>
  * <MultiTenantCloudOnly />
  * 상태가 **Killed**인 실행은 자동화를 트리거할 수 없습니다. 이 상태는 실행이 관리자가 강제로 중지했음을 나타냅니다.
</Note>

실행의 [상태](/ko/models/runs/run-states)가 **Running**, **Finished**, 또는 **Failed**로 변경될 때 워크플로를 트리거합니다. 필요한 경우 사용자나 실행 이름 필터를 지정하여 자동화를 트리거할 수 있는 실행을 추가로 제한할 수 있습니다.

<Frame>
  ![Screenshot showing a run status change automation](/images/automations/run_status_change.png)
</Frame>

실행 상태는 전체 실행의 속성이므로, 실행 상태 자동화는 워크스페이스가 아니라 **Automations** 페이지에서만 생성할 수 있습니다.

<div id="run-metrics-change">
  #### 실행 지표 변경
</div>

<Note>
  <MultiTenantCloudOnly />
</Note>

실행의 히스토리에 있는 지표이든, CPU 사용률 비율을 추적하는 `cpu` 같은 [시스템 지표](/ko/models/ref/python/experiments/system-metrics)이든, 지표에 기록된 값을 기반으로 워크플로를 트리거합니다. W&amp;B는 시스템 지표를 15초마다 자동으로 기록합니다.

프로젝트의 **Automations** 탭이나 워크스페이스의 선 그래프(line plot) 패널에서 바로 실행 지표 자동화를 생성할 수 있습니다.

실행 지표 자동화를 설정하려면, 지표 값과 지정한 임계값을 어떻게 비교할지 구성합니다. 사용할 수 있는 옵션은 이벤트 유형과 지정한 필터에 따라 달라집니다.

필요하다면 사용자 또는 실행 이름 필터를 지정하여, 자동화를 트리거할 수 있는 실행을 추가로 제한할 수 있습니다.

<div id="threshold">
  ##### 임계값
</div>

**Run metrics threshold met** 이벤트에 대해 다음을 설정합니다:

1. 평가에 사용할, 가장 최근에 기록된 값들의 개수(기본값은 5).
2. 이 값들 중에서 **Average**, **Min**, 또는 **Max** 값 중 어떤 값을 평가할지.
3. 수행할 비교 방식:
   * Above (초과)
   * Above or equal to (이상)
   * Below (미만)
   * Below or equal to (이하)
   * Not equal to (같지 않음)
   * Equal to (같음)

예를 들어, `accuracy`의 평균값이 `.6`보다 클 때 자동화를 트리거하도록 설정할 수 있습니다.

<Frame>
  ![실행 메트릭 임계값 자동화를 보여 주는 스크린샷](/images/automations/run_metrics_threshold_automation.png)
</Frame>

<div id="change-threshold">
  ##### 변경 임계값
</div>

**실행 지표 변경 임계값 충족** 이벤트의 경우, 자동화는 시작할지 여부를 확인하기 위해 두 개의 값 &quot;윈도우&quot;를 사용합니다:

* 고려할 최근 로그 값의 *현재 윈도우* (기본값 10)
* 고려할 최근 로그 값의 *이전 윈도우* (기본값 50)

현재 윈도우와 이전 윈도우는 연속적이며 서로 겹치지 않습니다.

자동화를 생성하려면 다음을 설정합니다:

1. 로그된 값의 현재 윈도우 (기본값 10)
2. 로그된 값의 이전 윈도우 (기본값 50)
3. 값을 상대값으로 평가할지 절대값으로 평가할지 여부 (기본값은 **Relative**)
4. 사용할 비교 방식:
   * 최소한 …만큼 증가
   * 최소한 …만큼 감소
   * 최소한 …만큼 증가 또는 감소

예를 들어, 평균 `loss`가 최소 `.25`만큼 감소할 때 자동화를 트리거하도록 설정할 수 있습니다.

</Frame>

<div id="run-metrics-z-score-change">
  #### 실행 메트릭 z-점수 변경
</div>

<Note>
  <MultiTenantCloudOnly />
</Note>

W&amp;B는 메트릭의 z-점수(표준 점수)가 지정된 임계값을 초과할 때 자동화를 트리거할 수 있습니다. z-점수는 프로젝트 내에서 구성 가능한 실행 구간(기본값은 30회 실행) 전반에 걸쳐 해당 메트릭 값이 평균으로부터 몇 개의 표준편차만큼 떨어져 있는지를 측정합니다.

z-점수를 이벤트 트리거로 사용하려면 **Run metrics z-score threshold met** 이벤트를 선택합니다.

z-점수를 기반으로 한 자동화는 절대 임계값을 확인할 필요 없이, 팀이 이례적인 성능을 파악할 수 있도록 도와줍니다. 절대 임계값은 모델이나 학습 프로세스가 발전함에 따라 달라질 수 있습니다.

실행 메트릭 z-점수 자동화는 프로젝트의 **Automations** 탭 또는 워크스페이스의 선 그래프 패널에서 직접 생성할 수 있습니다.

z-점수 자동화를 생성하려면 다음을 설정합니다:

1. 양의 부동 소수점 값으로 표현된 대상 z-점수 임계값(예: 2.0)
2. 평균 값을 결정하는, 로그된 값이 포함되는 구간(기본값은 30)
3. 수행할 비교:
   * Above (성능이 이례적으로 높을 때 트리거)
   * Below (성능이 이례적으로 낮을 때 트리거)
   * Either above or below

예를 들어, `accuracy`의 z-점수가 2를 초과할 때 자동화를 트리거하도록 설정할 수 있습니다. 이는 해당 실행이 프로젝트의 다른 실행보다 유의미하게 더 좋은 성능을 내고 있음을 의미합니다.

**z-점수 값 이해하기:**

* z-점수가 0이면 메트릭이 평균과 같습니다.
* z-점수가 +2.0이면 메트릭이 평균보다 2 표준편차만큼 높습니다.
* z-점수가 -2.0이면 메트릭이 평균보다 2 표준편차만큼 낮습니다.
* ±2를 넘는 값은 통계적으로 유의미한 이상치로 간주되는 경우가 많습니다.

<div id="run-filters">
  #### 실행 필터
</div>

이 섹션에서는 자동화가 평가 대상으로 실행을 어떻게 선택하는지 설명합니다.

* 기본적으로 이벤트가 발생하면 프로젝트 내의 모든 실행이 자동화를 트리거합니다. 다음 필터 중 하나를 구성하여 어떤 실행이 자동화를 트리거할지 제한할 수 있습니다:

  * **한 사용자의 실행으로 필터링**: 지정된 사용자가 생성한 실행만 포함합니다.
  * **실행 이름으로 필터링**: 지정된 정규 표현식과 이름이 일치하는 실행만 포함합니다.

  자세한 내용은 [자동화 생성](/ko/models/automations/create-automations)을 참조하세요.
* 각 실행은 개별적으로 고려되며, 자동화를 트리거할 수 있습니다.
* 각 실행의 값은 별도의 윈도우(window)에 넣어지고 임계값과 개별적으로 비교됩니다.
* 24시간 동안 특정 자동화는 각 실행마다 최대 한 번만 트리거될 수 있습니다.
