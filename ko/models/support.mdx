    아티팩트는 상위 프로젝트의 접근 권한을 상속합니다:

    * 비공개 프로젝트에서는 팀 구성원만 아티팩트에 접근할 수 있습니다.
    * 공개 프로젝트에서는 모든 사용자가 아티팩트를 읽을 수 있고, 생성 및 수정은 팀 구성원만 할 수 있습니다.
    * 오픈 프로젝트에서는 모든 사용자가 아티팩트를 읽고 쓸 수 있습니다.

    ## Artifacts Workflows

    이 섹션에서는 Artifact를 관리하고 편집하기 위한 워크플로를 설명합니다. 많은 워크플로는 [W&amp;B API](/ko/models/track/public-api-guide)를 활용하며, 이는 W&amp;B에 저장된 데이터에 접근할 수 있게 해 주는 [클라이언트 라이브러리](/ko/models/ref/python/)의 구성 요소입니다.

  <Accordion title="Tables에 Plotly 또는 Bokeh 차트를 어떻게 추가하나요?">
    Plotly 또는 Bokeh figure를 테이블에 직접 넣는 기능은 지원되지 않습니다. 대신 figure를 HTML로 내보낸 다음, 해당 HTML을 테이블에 포함하세요. 아래 예시는 인터랙티브 Plotly 및 Bokeh 차트를 사용해 이를 구현하는 방법을 보여줍니다.

    <Tabs>
      <Tab title="Plotly 사용하기">
        ```python
        import wandb
        import plotly.express as px

        # 새 실행을 초기화합니다
        with wandb.init(project="log-plotly-fig-tables", name="plotly_html") as run:

            # 테이블을 생성합니다
            table = wandb.Table(columns=["plotly_figure"])

            # Plotly figure의 경로를 정의합니다
            path_to_plotly_html = "./plotly_figure.html"

            # Plotly figure를 생성합니다
            fig = px.scatter(x=[0, 1, 2, 3, 4], y=[0, 1, 4, 9, 16])

            # Plotly figure를 HTML로 내보냅니다
            # auto_play를 False로 설정하면 애니메이션 Plotly 차트가 자동 재생되지 않습니다
            fig.write_html(path_to_plotly_html, auto_play=False)

            # Plotly figure를 HTML 파일로 테이블에 추가합니다
            table.add_data(wandb.Html(path_to_plotly_html))

            # 테이블을 로그합니다
            run.log({"test_table": table})

        ```
      </Tab>

      <Tab title="Bokeh 사용하기">
        ```python
        from scipy.signal import spectrogram
        import holoviews as hv
        import panel as pn
        from scipy.io import wavfile
        import numpy as np
        from bokeh.resources import INLINE

        hv.extension("bokeh", logo=False)
        import wandb

        def save_audio_with_bokeh_plot_to_html(audio_path, html_file_name):
            sr, wav_data = wavfile.read(audio_path)
            duration = len(wav_data) / sr
            f, t, sxx = spectrogram(wav_data, sr)
            spec_gram = hv.Image((t, f, np.log10(sxx)), ["Time (s)", "Frequency (Hz)"]).opts(
                width=500, height=150, labelled=[]
            )
            audio = pn.pane.Audio(wav_data, sample_rate=sr, name="Audio", throttle=500)
            slider = pn.widgets.FloatSlider(end=duration, visible=False)
            line = hv.VLine(0).opts(color="white")
            slider.jslink(audio, value="time", bidirectional=True)
            slider.jslink(line, value="glyph.location")
            combined = pn.Row(audio, spec_gram * line, slider).save(html_file_name)

        html_file_name = "audio_with_plot.html"
        audio_path = "hello.wav"
        save_audio_with_bokeh_plot_to_html(audio_path, html_file_name)

        wandb_html = wandb.Html(html_file_name)
        with wandb.init(project="audio_test") as run:
            my_table = wandb.Table(columns=["audio_with_plot"], data=[[wandb_html], [wandb_html]])
            run.log({"audio_table": my_table})
        ```
      </Tab>
    </Tabs>

<Update tags={["Artifacts", "Storage"]}>
  <Accordion title="각 아티팩트 버전은 얼마나 많은 스토리지를 사용하나요?">
    서로 다른 두 아티팩트 버전 사이에서 변경된 파일에 대해서만 스토리지 비용이 발생합니다.

    <Frame>
      <img src="/images/artifacts/artifacts-dedupe.PNG" alt="Artifact 중복 제거" />
    </Frame>

    `cat.png`와 `dog.png` 두 개의 이미지 파일을 포함하는 `animals`라는 이름의 이미지 아티팩트를 가정해 봅시다:

    ```
    images
    |-- cat.png (2MB) # `v0`에서 추가됨
    |-- dog.png (1MB) # `v0`에서 추가됨
    ```

    이 아티팩트는 버전 `v0`을 부여받습니다.

    새 이미지 `rat.png`를 추가하면, 다음과 같은 내용을 가진 새 아티팩트 버전 `v1`이 생성됩니다:

    ```
    images
    |-- cat.png (2MB) # `v0`에서 추가됨
    |-- dog.png (1MB) # `v0`에서 추가됨
    |-- rat.png (3MB) # `v1`에서 추가됨
    ```

    버전 `v1`은 총 6MB를 추적하지만, `v0`과 3MB를 공유하므로 실제로는 3MB의 공간만 차지합니다. `v1`을 삭제하면 `rat.png`에 해당하는 3MB 스토리지가 회수됩니다. `v0`을 삭제하면 `cat.png`와 `dog.png`의 스토리지 비용이 `v1`으로 이전되어, `v1`의 스토리지 크기가 6MB로 증가합니다.
  </Accordion>
</Update>

<Update tags={["Artifacts"]}>
  <Accordion title="여러 아키텍처와 실행에서 아티팩트를 사용하는 방법은?">
    모델을 버전 관리하는 방법은 여러 가지가 있습니다. 아티팩트는 특정 요구 사항에 맞춘 모델 버전 관리 도구를 제공합니다. 여러 모델 아키텍처를 탐색하는 프로젝트의 일반적인 접근 방식은 아키텍처별로 아티팩트를 분리하는 것입니다. 다음 단계를 고려해 보세요:

    1. 각 서로 다른 모델 아키텍처에 대해 새 아티팩트를 생성하세요. 실행에 `config`를 사용하는 것과 유사하게, 아티팩트의 `metadata` 속성을 사용해 아키텍처에 대한 자세한 설명을 추가하세요.
    2. 각 모델에 대해 `log_artifact`를 사용해 주기적으로 체크포인트를 로깅합니다. W&amp;B는 이러한 체크포인트의 히스토리를 생성하고, 가장 최근 체크포인트에 `latest` 별칭을 붙입니다. 어떤 모델 아키텍처든 최신 체크포인트를 참조할 때는 `architecture-name:latest`를 사용하세요.
  </Accordion>
</Update>

  <Accordion title="Sweeps에서 커스텀 CLI 명령을 사용하려면 어떻게 하나요?">
    학습 설정이 커맨드라인 인자를 전달하는 방식이라면, W&amp;B Sweeps를 커스텀 CLI 명령과 함께 사용할 수 있습니다.

    아래 예시에서 코드 스니펫은 사용자가 `train.py`라는 이름의 Python 스크립트를 학습시키면서, 스크립트가 파싱할 값을 전달하는 bash 터미널을 보여줍니다:

    ```bash
    /usr/bin/env python train.py -b \
        your-training-config \
        --batchsize 8 \
        --lr 0.00001
    ```

    커스텀 명령을 적용하려면 YAML 파일의 `command` 키를 수정하세요. 위 예시를 기준으로 한 설정은 다음과 같습니다:

    ```yaml
    program:
      train.py
    method: grid
    parameters:
      batch_size:
        value: 8
      lr:
        value: 0.0001
    command:
      - ${env}
      - python
      - ${program}
      - "-b"
      - your-training-config
      - ${args}
    ```

    `${args}` 키는 sweep 설정의 모든 파라미터를 `argparse` 형식인 `--param1 value1 --param2 value2`로 확장합니다.

    `argparse` 외의 추가 인자를 사용하려면, 다음을 구현하세요:

    ```python
    parser = argparse.ArgumentParser()
    args, unknown = parser.parse_known_args()
    ```

    <Note>
      환경에 따라 `python`이 Python 2를 가리킬 수 있습니다. Python 3이 실행되도록 보장하려면, 명령 설정에서 `python3`를 사용하세요:

      ```yaml
      program:
        script.py
      command:
        - ${env}
        - python3
        - ${program}
        - ${args}
      ```
    </Note>
  </Accordion>

    1. SSL 인증서를 갱신합니다. Ubuntu 서버에서는 `update-ca-certificates`를 실행합니다. 보안 위험을 줄이기 위해 학습 로그를 동기화하려면 유효한 SSL 인증서가 필수입니다.

    3. 로컬에서 동작하고 클라우드 서버로 동기화를 수행하지 않는 [W&amp;B Private Hosting](/ko/platform/hosting/) 사용을 고려하세요.

  <Accordion title="`.log()`와 `.summary`의 차이는 무엇인가요?">
    `summary`는 테이블에 표시되고, `log`는 이후 플로팅을 위해 모든 값을 저장합니다.

    산점도와 평행 좌표 플롯은 summary 값을 사용하고, 선 그래프는 `run.log`로 기록된 모든 값을 표시합니다.

    일부 사용자는 가장 최근에 기록된 정확도 대신 최적 정확도를 반영하도록 summary를 직접 설정하는 것을 선호합니다.

    6. **협업 도구**: W&amp;B는 복잡한 머신 러닝 프로젝트를 위한 협업을 강화합니다. 프로젝트 링크를 공유하고 비공개 팀을 활용해 결과를 공유할 수 있습니다. 대화형 시각화와 마크다운 설명으로 구성된 리포트를 만들어 작업 로그나 프레젠테이션에 사용할 수 있습니다.

    ```python
    import base64
    import requests

    def encode_base64(username, key):
        auth_string = f'{username}:{key}'
        return base64.b64encode(auth_string.encode('utf-8')).decode('utf-8')

    username = ''  # 조직 관리자 사용자 이름
    key = ''  # API 키
    scim_base_url = 'https://api.wandb.ai/scim/v2'
    users_endpoint = f'{scim_base_url}/Users'
    headers = {
        'Authorization': f'Basic {encode_base64(username, key)}',
        'Content-Type': 'application/scim+json'
    }

    response = requests.get(users_endpoint, headers=headers)
    users = []
    for user in response.json()['Resources']:
        users.append([user['userName'], user['emails']['Value']])
    ```

    필요에 따라 출력 결과를 저장하도록 스크립트를 수정하세요.

    W&amp;B는 각 실행에서 기록한 아티팩트와 각 실행에서 사용한 아티팩트를 추적하여 아티팩트 그래프를 구성합니다. 이 그래프는 실행과 아티팩트를 나타내는 노드로 이루어진 이분 그래프이면서, 방향성을 가진 비순환 그래프입니다. 예시는 [여기](https://wandb.ai/shawn/detectron2-11/artifacts/dataset/furniture-small-val/06d5ddd4deeb2a6ebdd5/graph)에서 볼 수 있습니다(그래프를 확장하려면 &quot;Explode&quot;를 클릭하세요).

    <Tabs>
      <Tab title="Artifact에서 시작">
        ```python

        artifact = api.artifact("project/artifact:alias")

        # 아티팩트에서 그래프를 위쪽으로 탐색:
        producer_run = artifact.logged_by()
        # 아티팩트에서 그래프를 아래쪽으로 탐색:
        consumer_runs = artifact.used_by()

        # 실행에서 그래프를 아래쪽으로 탐색:
        next_artifacts = consumer_runs[0].logged_artifacts()
        # 실행에서 그래프를 위쪽으로 탐색:
        previous_artifacts = producer_run.used_artifacts()
        ```
      </Tab>

      <Tab title="Run에서 시작">

        ```python
        # 실행에서 그래프를 위쪽으로 탐색:
        consumed_artifacts = run.used_artifacts()

        # 아티팩트에서 그래프를 위쪽으로 탐색:
        earlier_run = consumed_artifacts[0].logged_by()
        # 아티팩트에서 그래프를 아래쪽으로 탐색:
        consumer_runs = produced_artifacts[0].used_by()
        ```

  <Accordion title="로컬에서 `wandb` 파일의 위치를 어떻게 정의하나요?">
    * `WANDB_DIR=<path>` 또는 `wandb.init(dir=<path>)`: 학습 스크립트에 대해 생성되는 `wandb` 폴더의 위치를 제어합니다. 기본값은 `./wandb`입니다. 이 폴더는 실행의 데이터와 로그를 저장합니다.
    * `WANDB_ARTIFACT_DIR=<path>` 또는 `wandb.Artifact().download(root="<path>")`: 아티팩트가 다운로드되는 위치를 제어합니다. 기본값은 `./artifacts`입니다.
    * `WANDB_CACHE_DIR=<path>`: `wandb.Artifact`를 호출할 때 아티팩트가 생성되고 저장되는 위치입니다. 기본값은 `~/.cache/wandb`입니다.
    * `WANDB_CONFIG_DIR=<path>`: config 파일이 저장되는 위치입니다. 기본값은 `~/.config/wandb`입니다.
    * `WANDB_DATA_DIR=<PATH>`: 업로드 중 아티팩트를 스테이징하는 데 사용되는 위치를 제어합니다. 기본값은 `~/.cache/wandb-data/`입니다.
  </Accordion>

    ## 1. 항상 오류 처리를 구현하세요

    API 호출을 try-except 블록으로 감싸세요:

    ```python
    import openai

    try:
        response = client.chat.completions.create(
            model="meta-llama/Llama-3.1-8B-Instruct",
            messages=messages
        )
    except Exception as e:
        print(f"Error: {e}")
        # 오류를 적절히 처리합니다
    ```

    ## 2. 지수 백오프를 사용하는 재시도 로직을 구현하세요

    ```python
    import time
    from typing import Optional

    def call_inference_with_retry(
        client, 
        messages, 
        model: str,
        max_retries: int = 3,
        base_delay: float = 1.0
    ) -> Optional[str]:
        for attempt in range(max_retries):
            try:
                response = client.chat.completions.create(
                    model=model,
                    messages=messages
                )
                return response.choices[0].message.content
            except Exception as e:
                if attempt == max_retries - 1:
                    raise
                
                # 지수 백오프를 사용해 지연 시간을 계산합니다
                delay = base_delay * (2 ** attempt)
                print(f"Attempt {attempt + 1} failed, retrying in {delay}s...")
                time.sleep(delay)
        
        return None
    ```

    ## 3. 사용량을 모니터링하세요

    * W&amp;B Billing 페이지에서 크레딧 사용량을 추적하세요
    * 한도에 도달하기 전에 알림을 설정하세요
    * 애플리케이션에서 API 사용량을 로깅하세요

    ## 4. 특정 오류 코드를 처리하세요

    ```python
    def handle_inference_error(error):

            raise ValueError("API 키와 프로젝트 구성을 확인하세요")
        elif "402" in error_str:
            # 크레딧 소진
            raise ValueError("크레딧이 부족합니다")
        elif "429" in error_str:

            return "retry"
        elif "500" in error_str or "503" in error_str:
            # 서버 오류
    ```

    ## 서버 오류 처리 방법

    1. **재시도 전에 대기**

       * 503 오류: 60–120초 대기

       ```python
       import time
       import openai

       def call_with_retry(client, messages, model, max_retries=5):
           for attempt in range(max_retries):
               try:
                   return client.chat.completions.create(
                       model=model,
                       messages=messages
                   )
               except Exception as e:
                   if "500" in str(e) or "503" in str(e):
                       if attempt < max_retries - 1:
                           wait_time = min(60, (2 ** attempt))
                           time.sleep(wait_time)
                       else:
                           raise
                   else:
                       raise
       ```

    3. **적절한 타임아웃 설정**
       * HTTP 클라이언트의 타임아웃 값을 늘립니다.

  <Accordion title="하나의 스크립트에서 여러 실행을 어떻게 시작하나요?">
    하나의 스크립트 안에서 여러 실행을 기록하려면, 새로운 실행을 시작하기 전에 이전 실행을 반드시 종료하세요.

    권장되는 방법은 `wandb.init()`를 컨텍스트 매니저로 사용하는 것입니다. 이렇게 하면 스크립트에서 예외가 발생할 경우 실행을 종료하고 실패로 표시합니다:

    ```python
    import wandb

    for x in range(10):
        with wandb.init() as run:
            for y in range(100):
                run.log({"metric": x + y})
    ```

    `run.finish()`를 명시적으로 호출할 수도 있습니다:

    ```python
    import wandb

    for x in range(10):
        run = wandb.init()

        try:
            for y in range(100):
                run.log({"metric": x + y})

        except Exception:
            run.finish(exit_code=1)
            raise

        finally:
            run.finish()
    ```

    ## 여러 활성 실행

    wandb 0.19.10부터는 `reinit` 설정을 `"create_new"`로 지정해 여러 개의 실행을 동시에 활성 상태로 둘 수 있습니다.

    ```python
    import wandb

    with wandb.init(reinit="create_new") as tracking_run:
        for x in range(10):
            with wandb.init(reinit="create_new") as run:
                for y in range(100):
                    run.log({"x_plus_y": x + y})

                tracking_run.log({"x": x})
    ```

    `reinit="create_new"`에 대한 자세한 설명과 W&amp;B 통합 시 주의사항은
    [프로세스당 여러 실행](/ko/models/runs/initialize-run)을 참고하세요.
  </Accordion>
</Update>

  <Accordion title="왜 특정 문자가 포함된 메트릭은 정렬하거나 필터링할 수 없나요?">
    W&amp;B의 메트릭 이름은 UI에서 제대로 정렬·필터링될 수 있도록 GraphQL 네이밍 규칙을 따라야 합니다.

    ## 유효한 메트릭 이름

    * **허용된 문자**: 알파벳(A-Z, a-z), 숫자(0-9), 밑줄(&#95;)
    * **시작 문자**: 이름은 문자 또는 밑줄로 시작해야 함
    * **패턴**: 메트릭 이름은 `/^[_a-zA-Z][_a-zA-Z0-9]*$/` 패턴을 따라야 함

    이 규칙을 따르지 않는 메트릭은 W&amp;B UI에서 정렬하거나 필터링할 수 없을 수 있습니다.

    ## 예시

    **유효한 메트릭 이름:**

    ```python
    with wandb.init() as run:
      run.log({"accuracy": 0.9, "val_loss": 0.1, "epoch_5": 5})
      run.log({"modelAccuracy": 0.95, "learning_rate": 0.001})
    ```

    **유효하지 않은 메트릭 이름(다음은 피하세요):**

    ```python
    with wandb.init() as run:
      run.log({"acc,val": 0.9})  # 콤마 포함
      run.log({"loss-train": 0.1})  # 하이픈 포함
      run.log({"test acc": 0.95})  # 공백 포함
      run.log({"5_fold_cv": 0.8})  # 숫자로 시작
    ```

    ## 권장 해결 방법

    유효하지 않은 문자를 밑줄 등 유효한 문자로 바꾸세요:

    * `"test acc"` 대신 `"test_acc"` 사용
    * `"loss-train"` 대신 `"loss_train"` 사용
    * `"acc,val"` 대신 `"acc_val"` 사용

    자세한 내용은 [Metric naming constraints](/ko/models/track/log/#metric-naming-constraints)를 참조하세요.
  </Accordion>

    Sweep 설정에서 `(run.config())`을 사용해 하이퍼파라미터 이름과 값을 가져올 수 있으며, 이는 딕셔너리처럼 동작합니다.

    Sweep 외부의 실행에서는 `wandb.init()`의 `config` 인자에 딕셔너리를 전달하여 `wandb.Run.config()` 값을 설정합니다. Sweep에서는 `wandb.init()`에 전달된 설정이 기본값 역할을 하며, Sweep이 이를 덮어쓸 수 있습니다.

    명시적인 동작을 위해 `wandb.Run.config.setdefaults()`를 사용하세요. 아래 코드 예시는 두 가지 방법을 모두 보여줍니다:

    <Tabs>
      <Tab title="wandb.init()">
        ```python
        # 하이퍼파라미터 기본값 설정
        config_defaults = {"lr": 0.1, "batch_size": 256}

        # 실행을 시작하고 Sweep이 덮어쓸 수 있는
        # 기본값을 제공합니다
        with wandb.init(config=config_defaults) as run:
            # 여기에 학습 코드를 추가하세요
            ...
        ```
      </Tab>

      <Tab title="config.setdefaults()">
        ```python
        # 하이퍼파라미터 기본값 설정
        config_defaults = {"lr": 0.1, "batch_size": 256}

        # 실행 시작
        with wandb.init() as run:
            # Sweep에서 설정하지 않은 값들을 업데이트
            run.config.setdefaults(config_defaults)

            # 여기에 학습 코드를 추가하세요
        ```
      </Tab>
    </Tabs>
  </Accordion>
</Update>

    프로젝트를 더 엄격한 공개 범위 설정으로 업데이트하는 경우, 일부 사용자가 다시 프로젝트에 접근할 수 있도록 해당 사용자들을 다시 초대해야 할 수 있습니다.

<Update tags={["Anonymous"]}>
  <Accordion title="계정이 없는 사람은 실행 결과를 어떻게 볼 수 있나요?">
    누군가가 `anonymous="allow"`로 스크립트를 실행하면:

    1. **임시 계정 자동 생성**: W&amp;B는 로그인된 계정이 있는지 확인합니다. 계정이 없으면, W&amp;B는 새 익명 계정을 만들고 해당 세션을 위한 API 키를 저장합니다.
    2. **결과를 빠르게 로깅**: 사용자는 스크립트를 여러 번 실행하고 W&amp;B 대시보드에서 결과를 즉시 확인할 수 있습니다. 이렇게 소유되지 않은 익명 실행은 7일 동안 유지됩니다.
    3. **유용해졌을 때 데이터 소유**: 사용자가 W&amp;B에서 가치 있는 결과를 찾으면, 페이지 상단 배너의 버튼을 클릭해 실행 데이터를 실제 계정으로 가져올 수 있습니다. 소유하지 않으면 실행 데이터는 7일 후 삭제됩니다.

    <Warning>
      **익명 실행 링크는 민감한 정보입니다**. 이 링크를 가진 누구나 7일 동안 실험 결과를 조회하고 소유할 수 있으므로, 신뢰할 수 있는 사람에게만 링크를 공유하세요. 작성자의 신원을 숨긴 채로 결과를 공개적으로 공유하고 싶다면, support@wandb.com으로 문의해 도움을 받으세요.
    </Warning>

    W&amp;B 사용자가 해당 스크립트를 찾아 실행하면, 일반적인 실행과 마찬가지로 결과가 그 사용자의 계정에 정상적으로 로깅됩니다.
  </Accordion>
</Update>

  <Accordion title="로그한 것보다 데이터 포인트가 적게 보이는 이유는 무엇인가요?">
    X축을 `Step`이 아닌 다른 값으로 설정해 메트릭을 시각화하면, 표시되는 데이터 포인트 수가 줄어듭니다. 메트릭이 동기화된 상태를 유지하려면 동일한 `Step`에서 로그되어야 합니다. 샘플 간 보간 시, 동일한 `Step`에서 로그된 메트릭만 샘플링됩니다.

    **가이드라인**

    메트릭은 하나의 `log()` 호출에 묶어서 로그하세요. 예를 들어, 다음처럼 사용하는 대신:

    ```python
    import wandb
    with wandb.init() as run:
        run.log({"Precision": precision})
        ...
        run.log({"Recall": recall})
    ```

    다음과 같이 사용하세요:

    ```python
    import wandb
    with wandb.init() as run:
        run.log({"Precision": precision, "Recall": recall})
    ```

    `step` 파라미터를 수동으로 제어하려면, 코드에서 메트릭을 다음과 같이 동기화하세요:

    ```python
    with wandb.init() as run:
        step = 100  # 예시 step 값
        # 같은 step에서 Precision과 Recall을 로그
        run.log({"Precision": precision, "Recall": recall}, step=step)
    ```

    두 `log()` 호출에서 `step` 값이 동일해야 메트릭이 같은 step 아래에 로그되고 함께 샘플링됩니다. `step` 값은 각 호출에서 단조 증가해야 합니다. 그렇지 않으면 `step` 값은 무시됩니다.
  </Accordion>

    * **위임 작업**: 서비스 계정은 이를 생성한 사용자 또는 조직을 대신해 작업을 수행합니다.

<Update tags={["Environment Variables", "Experiments"]}>
  <Accordion title="wandb.init는 내 학습 프로세스에 어떤 영향을 주나요?">
    학습 스크립트에서 `wandb.init()`이 실행되면, API 호출을 통해 서버에 실행 객체가 생성됩니다. 새로운 프로세스가 시작되어 메트릭을 스트리밍하고 수집하며, 기본 프로세스는 정상적으로 동작합니다. 스크립트는 로컬 파일에 기록하고, 별도의 프로세스는 시스템 메트릭을 포함한 데이터를 서버로 스트리밍합니다. 스트리밍을 끄려면 학습 디렉터리에서 `wandb off`를 실행하거나 `WANDB_MODE` 환경 변수를 `offline`으로 설정하세요.
  </Accordion>
</Update>

<Update tags={["Artifacts", "Environment Variables"]}>
  <Accordion title="아티팩트는 어디에 다운로드되며, 이 위치를 어떻게 제어할 수 있나요?">
    기본적으로 아티팩트는 `artifacts/` 폴더에 다운로드됩니다. 위치를 변경하려면 다음과 같이 하면 됩니다:

    * [`wandb.Artifact().download`](/ko/models/ref/python/public-api/api)에 경로를 전달합니다:

      ```python
      wandb.Artifact().download(root="<path_to_download>")
      ```

    * `WANDB_ARTIFACT_DIR` [환경 변수](/ko/models/track/environment-variables)를 설정합니다:

      ```python
      import os
      os.environ["WANDB_ARTIFACT_DIR"] = "<path_to_download>"
      ```
  </Accordion>
</Update>

  <Accordion title="CSV 메트릭 내보내기에서 스텝이 일부 누락되는 이유는 무엇인가요?">
    내보내기 제한 때문에 전체 실행 기록을 CSV로 내보내거나 `run.history` API를 사용해 모두 가져오지 못할 수 있습니다. 전체 실행 기록에 접근하려면, Parquet 형식으로 실행 기록 아티팩트를 다운로드하세요:

    ```python
    import wandb
    import pandas as pd

    run = wandb.init()
    artifact = run.use_artifact('<entity>/<project>/<run-id>-history:v0', type='wandb-history')
    artifact_dir = artifact.download()
    df = pd.read_parquet('<path to .parquet file>')
    ```
