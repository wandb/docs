이 가이드는 Python 라이브러리에 Weights &amp; Biases (W&amp;B)를 통합하는 방법을 설명합니다.

트레이닝 프레임워크, SDK 또는 재사용 가능한 라이브러리와 같이 복잡한 코드베이스에 W&amp;B를 통합하는 경우에는 다음 권장 사항을 따르세요.

<Tip>
  W&amp;B가 처음이라면 계속하기 전에 핵심 가이드(예: [Experiment Tracking](/ko/models/track/))를 먼저 살펴보세요.
</Tip>

아래에서는 작업 중인 코드베이스가 단일 Python 트레이닝 스크립트나 Jupyter Notebook보다 더 복잡할 때 유용한 핵심 팁과 모범 사례를 다룹니다.

<div id="decide-how-users-install-wb">
  ## 사용자가 W&amp;B를 설치하는 방식 결정하기
</div>

시작하기 전에 W&amp;B를 라이브러리의 필수 의존성으로 할지, 선택적 기능으로 제공할지 결정하세요.

<div id="require-wb-as-a-dependency">
  ### W&amp;B를 필수 의존성으로 요구하기
</div>

W&amp;B가 라이브러리의 핵심 기능에 필수적이라면, W&amp;B Python SDK(`wandb`)를 의존성 목록에 추가하세요:

```txt
torch==1.8.0 
...
wandb==0.13.*
```

<div id="make-wb-optional-on-installation">
  ### 설치 시 W&amp;B를 선택 사항으로 만들기
</div>

W&amp;B가 선택 기능이라면, 라이브러리가 W&amp;B가 설치되지 않은 상태에서도 실행될 수 있도록 하세요.

Python에서 `wandb`를 조건부로 임포트하거나, `pyproject.toml`에서 선택적 의존성으로 선언할 수 있습니다.

<Tabs>
  <Tab title="Python">
    `wandb` 사용 가능 여부를 확인하고, 사용자가 W&amp;B 특성을 활성화했지만 설치하지 않은 경우에는 명확한 오류를 발생시키세요:

    ```python
    try:
        import wandb
        _WANDB_AVAILABLE = True
    except ImportError:
        _WANDB_AVAILABLE = False
    ```
  </Tab>

  <Tab title="pyproject.toml">
    `pyproject.toml` 파일에서 `wandb`를 선택적 의존성으로 선언하세요:

    ```toml
    [project]
    name = "my_awesome_lib"
    version = "0.1.0"
    dependencies = [
        "torch",
        "sklearn"
    ]

    [project.optional-dependencies]
    dev = [
        "wandb"
    ]
    ```
  </Tab>
</Tabs>

<div id="start-a-run">
  ## run 시작하기
</div>

*run*은 트레이닝 실험과 같은 단일 계산 단위를 나타냅니다. 대부분의 라이브러리는 트레이닝 작업마다 run을 하나씩 생성합니다. run에 대한 자세한 내용은 [W&amp;B Runs](/ko/models/runs/)를 참조하세요.

[`wandb.init()`](/ko/models/ref/python/functions/init)으로 run을 초기화하고 프로젝트 이름과 팀 entity(팀 이름)를 지정하세요. 프로젝트를 지정하지 않으면 W&amp;B는 run을 &quot;uncategorized&quot;라는 기본 프로젝트에 저장합니다.:

```python
with wandb.init(project="<project_name>", entity="<entity>") as run:
    ...
```

W&amp;B는 오류가 발생하더라도 run이 정상적으로 종료되도록 컨텍스트 매니저를 사용할 것을 권장합니다. 컨텍스트 매니저를 사용하지 않는 경우에는 `run.finish()`를 호출해 run을 종료하고 모든 데이터를 W&amp;B에 로깅해야 합니다.

<Tip>
  **`wandb.init`은 언제 호출해야 하나요?**

  가능한 한 빨리 `wandb.init()`을 호출하세요. W&amp;B는 stdout, stderr 및 오류 메시지를 캡처하므로 디버깅이 더 쉬워집니다.

  관련된 모든 정보가 run에 캡처되도록 전체 트레이닝 루프를 `wandb.init` 컨텍스트 매니저로 감싸세요. 여기에는 디버깅에 매우 중요한 오류 메시지도 포함됩니다.
</Tip>

<div id="set-wandb-as-an-optional-dependency">

사용자가 라이브러리를 사용할 때 `wandb`를 선택으로 만들고 싶다면, 다음 중 하나를 사용할 수 있습니다:

* 다음과 같이 `wandb` 플래그를 정의하세요:

* 또는 `wandb.init`에서 `wandb`를 `disabled`로 설정하세요:

  <Tab title="Bash">

  </Tab>
</Tabs>

* 또는 `wandb`를 오프라인으로 설정하세요 - 이렇게 해도 `wandb`는 계속 실행되지만, 인터넷을 통해 W&amp;B와 다시 통신을 시도하지는 않습니다:

<Tabs>
  <Tab title="환경 변수">

<div id="define-a-run-config">
  ## run 설정 정의하기
</div>

run을 초기화할 때 설정 사전을 전달하여 하이퍼파라미터와 기타 메타데이터를 W&amp;B에 로깅합니다.

W&amp;B App에서 설정 파라미터를 기준으로 run을 비교하고 Runs table에서 필터링할 수 있습니다. 또한 이러한 파라미터를 사용하여 W&amp;B App에서 run을 그룹화할 수도 있습니다.

예를 들어 아래 이미지에서 배치 크기(batch&#95;size)는 설정 파라미터로 정의되었으며, Runs table의 첫 번째 열에서 확인할 수 있습니다. 이를 통해 사용자는 배치 크기에 따라 run을 필터링하고 비교할 수 있습니다:

<Frame>
  <img src="/images/integrations/integrations_add_any_lib_runs_page.png" alt="W&B Runs table" />
</Frame>

일반적인 설정 파라미터 값의 예는 다음과 같습니다:

* 모델 이름, 버전, 아키텍처 파라미터, 하이퍼파라미터
* 데이터셋 이름, 버전, 트레이닝 또는 검증 예시 수
* 학습률, 배치 크기, 옵티마이저와 같은 트레이닝 파라미터

다음 코드 스니펫은 설정을 로깅하는 방법을 보여줍니다:

```python
config = {"batch_size": 32, ...}
with wandb.init(..., config=config) as run:
    ...
```

<div id="update-the-run-config">
  ### run 설정 업데이트
</div>

초기화 시점에 값을 사용할 수 없는 경우, 나중에 `wandb.Run.config.update`를 사용해 설정을 업데이트하세요. 예를 들어, 모델을 인스턴스화한 후 해당 모델의 파라미터를 추가할 수 있습니다:

```python
with wandb.init(...) as run:
    model = MyModel(...)
    run.config.update({"model_parameters": 3500})
```

자세한 내용은 [Experiments 설정](/ko/models/track/config/)을 참조하세요.

<div id="log-metrics-and-data">
  ## 메트릭과 데이터 로깅하기
</div>

<div id="log-metrics">
  ### 메트릭 로깅
</div>

키가 메트릭 이름인 딕셔너리를 생성하세요. 이 딕셔너리 객체를 [`wandb.Run.log()`](/ko/models/ref/python/experiments/run#method-run-log)에 전달해 W&amp;B에 로깅하세요:

```python
NUM_EPOCHS = 10

for epoch in range(NUM_EPOCHS):
    for input, ground_truth in data: 
        prediction = model(input) 
        loss = loss_fn(prediction, ground_truth) 
        metrics = { "loss": loss } 
        run.log(metrics)
```

W&amp;B App에서 관련 메트릭을 그룹화하려면 메트릭 이름 접두사를 사용하세요. 일반적인 접두사로는 각각 트레이닝 메트릭과 검증 메트릭에 사용하는 `train/` 및 `val/`가 있지만, 사용 사례에 맞는 어떤 접두사든 사용할 수 있습니다.

이렇게 하면 프로젝트의 Workspace에서 트레이닝 메트릭과 검증 메트릭 또는 구분하려는 다른 메트릭 유형에 대해 별도의 섹션이 생성됩니다:

```python
with wandb.init(...) as run:
    metrics = {
        "train/loss": 0.4,
        "train/learning_rate": 0.4,
        "val/loss": 0.5, 
        "val/accuracy": 0.7
    }
    run.log(metrics)
```

</Frame>

자세한 내용은 [`wandb.Run.log()`](/ko/models/ref/python/experiments/run#method-run-log)을 참조하세요.

<div id="control-the-x-axis">
  ### x-axis 제어하기
</div>

같은 트레이닝 step에 대해 `wandb.Run.log()`를 여러 Call 수행하면, wandb SDK는 `wandb.Run.log()`를 Call할 때마다 내부 step 카운터를 증가시킵니다. 이 카운터는 트레이닝 루프의 트레이닝 step과 일치하지 않을 수 있습니다.

이 상황을 방지하려면 `wandb.init`를 호출한 직후 한 번 `run.define_metric`으로 x-axis step을 명시적으로 정의하세요:

```python
with wandb.init(...) as run:
    run.define_metric("*", step_metric="global_step")
```

glob 패턴인 `*`는 모든 메트릭이 차트에서 x-axis로 `global_step`을 사용함을 의미합니다. 특정 메트릭만 `global_step` 기준으로 로깅하려면 대신 해당 메트릭을 지정할 수 있습니다:

```python
run.define_metric("train/loss", step_metric="global_step")
```

이제 `wandb.Run.log()`를 호출할 때마다 메트릭, `step` 메트릭, 그리고 `global_step`을 기록하세요:

```python
for step, (input, ground_truth) in enumerate(data):
    ...
    run.log({"global_step": step, "train/loss": 0.1})
    run.log({"global_step": step, "eval/loss": 0.2})
```

독립적인 step 변수에 액세스할 수 없는 경우, 예를 들어 검증 루프 동안 &quot;global&#95;step&quot;을 사용 가능하지 않은 경우에는, 이전에 &quot;global&#95;step&quot;에 대해 로깅된 값을 wandb가 자동으로 사용합니다. 이 경우 필요할 때 정의되어 있도록 metric의 초기값을 로깅해야 합니다.

<div id="log-media-and-structured-data">
  ### 미디어 및 구조화된 데이터 로깅
</div>

스칼라 외에도 이미지, 테이블, 텍스트, 오디오, 비디오 등을 로깅할 수 있습니다.

데이터를 로깅할 때는 다음 사항을 고려하세요:

* 메트릭을 얼마나 자주 로깅해야 하나요? 선택 사항이어야 하나요?
* 시각화에 어떤 유형의 데이터가 유용할까요?
  * 이미지의 경우 시간에 따른 변화를 확인할 수 있도록 샘플 예측, 세그멘테이션 마스크 등을 로깅할 수 있습니다.
  * 텍스트의 경우 나중에 탐색할 수 있도록 샘플 예측이 포함된 테이블을 로깅할 수 있습니다.

<div id="track-models-and-datasets-with-artifacts">
  ## 아티팩트로 모델과 데이터셋 추적하기
</div>

[W&amp;B Artifacts](/ko/models/artifacts/)를 사용해 모델과 데이터셋을 추적하고 버전 관리를 수행합니다. Artifacts는 머신 러닝 자산에 대한 저장소와 버전 관리를 제공하며, 데이터와 모델이 어떻게 서로 연관되어 있는지 보여 주는 리니지를 자동으로 추적합니다.

<Frame>
  <img src="/images/integrations/integrations_add_any_lib_dag.png" alt="W&B에 저장된 데이터셋과 모델 체크포인트" />
</Frame>

라이브러리에 Artifacts를 통합할 때는 다음 사항을 고려하십시오.

* (선택 사항으로 제공하고 싶다면) 모델 체크포인트와 데이터셋 중 무엇을 아티팩트로 로깅할지.
* 아티팩트 입력 레퍼런스(예: `entity/project/artifact`).
* 모델 체크포인트 또는 데이터셋을 로깅하는 빈도. 예를 들어, 매 에포크마다, 500 step마다 등.

<div id="log-model-checkpoints">
  ### 모델 체크포인트 로깅
</div>

모델 체크포인트를 W&amp;B에 로깅하세요. 일반적인 방법은 W&amp;B가 생성한 고유한 run ID를 아티팩트 이름의 일부로 사용해 체크포인트를 아티팩트로 로깅하는 것입니다.

```python
metadata = {"eval/accuracy": 0.8, "train/steps": 800} 

artifact = wandb.Artifact(
                name=f"model-{run.id}", 
                metadata=metadata, 
                type="model"
                ) 
artifact.add_dir("output_model") # 모델 가중치가 저장되어 있는 로컬 디렉터리

aliases = ["best", "epoch_10"] 
run.log_artifact(artifact, aliases=aliases)
```

이전 코드 스니펫은 모델 체크포인트를 아티팩트로 로깅하고 평가 정확도 및 트레이닝 step과 같은 메타데이터를 추가하는 방법을 보여줍니다. 이 아티팩트에는 고유한 run ID를 포함하는 이름이 지정되며, 쉽게 참조할 수 있도록 [맞춤형 별칭](/ko/models/artifacts/create-a-custom-alias/) 태그가 지정됩니다.

<div id="log-input-artifacts">
  ### 입력 아티팩트 로깅하기
</div>

입력으로 사용하는 데이터셋 또는 사전 학습된 모델을 로깅합니다:

```python
dataset = wandb.Artifact(name="flowers", type="dataset")
dataset.add_file("flowers.npy")
run.use_artifact(dataset)
```

이전 코드 스니펫은 &quot;flowers&quot;라는 데이터셋에 대한 아티팩트를 생성하고 그 안에 파일을 추가합니다. 그런 다음 `run.use_artifact()`를 사용해 해당 아티팩트를 현재 run과 연결하여, W&amp;B가 run에서 사용된 데이터셋의 리니지를 추적할 수 있도록 합니다.

<div id="download-artifacts">
  ### 아티팩트 다운로드
</div>

이전에 W&amp;B에 로깅된 아티팩트를 다운로드하여 트레이닝 또는 Inference 코드에서 사용하세요.

run 컨텍스트가 있는 경우, [`wandb.Run.use_artifact()`](/ko/models/ref/python/experiments/run)를 사용해 W&amp;B의 아티팩트를 참조한 다음 [`wandb.Artifact.download()`](/ko/models/ref/python/experiments/artifact)를 호출하여 로컬 디렉터리로 다운로드하세요.

```python
with wandb.init(...) as run:
    artifact = run.use_artifact("user/project/artifact:latest")
    local_path = artifact.download()
```

run을 초기화하지 않고 아티팩트를 참조하고 다운로드하려면 [W&amp;B Public API](/ko/models/ref/python/public-api/)를 사용하세요. 이는 분산 환경이나 Inference를 수행하는 경우처럼 새 run을 생성하고 싶지 않을 수 있는 시나리오에서 유용합니다.

```python
import wandb
artifact = wandb.Api().artifact("user/project/artifact:latest")
local_path = artifact.download()
```

자세한 내용은 [Artifacts를 다운로드하고 사용하기](/ko/models/artifacts/download-and-use-an-artifact/)를 참조하세요.
