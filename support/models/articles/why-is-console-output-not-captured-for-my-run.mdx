---
title: "Why is console output not captured for my run?"
keywords: ["Logs", "Runs"]
---

W&B captures your script's stdout and stderr output and stores it as `output.log` on the run's Files tab. If the log file is empty, missing, or incomplete, there are several possible causes.

**Output printed before `wandb.init()`**

W&B begins capturing console output when `wandb.init()` is called. Anything printed before that point—such as import-time messages, argument parsing output, or environment checks—is not captured. Move `wandb.init()` as early in your script as possible, before other print statements.

**Console capture is disabled**

Console capture can be turned off via settings or environment variable:

```python
# Disabled via settings
wandb.init(settings=wandb.Settings(console="off"))
```

```bash
# Disabled via environment variable
WANDB_CONSOLE=off python my_script.py
```

Check whether either of these is set in your environment or launch configuration. To re-enable, remove the setting or set `WANDB_CONSOLE=wrap`.

**Jupyter notebooks capture differently**

In Jupyter notebooks, stdout is routed through the notebook kernel, not the standard Python stream, so W&B may not capture cell output in the same way. Console capture in notebooks can be incomplete or delayed. To ensure logs are captured, write explicitly to `sys.stdout` or log messages using `wandb.log()` with string values instead of relying on console capture.

**Multiprocessing and subprocess output**

W&B only hooks into the stdout and stderr of the process that called `wandb.init()`. Output from child processes spawned with `multiprocessing`, `subprocess`, or similar mechanisms is not automatically captured. To log output from worker processes, write to the parent process's stream or use explicit `wandb.log()` calls from the rank-0 process.

**Run crashed before flush**

If your script crashes or is killed before W&B has a chance to flush the log buffer to the server, the `output.log` file on the UI may be truncated or absent. A local copy is always written to `wandb/run-<timestamp>-<run-id>/logs/output.log`; check there for the full output even if the server-side file is missing.

**Long-running jobs hitting the 10,000-line display limit**

The W&B UI displays up to 10,000 lines in the Logs panel. If your job produces more output than that, the display is truncated but the full file is still stored. Download `output.log` from the Files tab or via the API to access all lines:

```python
import wandb
api = wandb.Api()
run = api.run("entity/project/run_id")
run.file("output.log").download(replace=True)
```
