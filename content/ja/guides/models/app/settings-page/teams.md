W&B Teams を使って、ML チームのための中央ワークスペースとして、より良いモデルを素早く構築しましょう。

* **チームで試した全ての実験管理** をトラッキングし、重複作業を防ぎます。
* **トレーニング済みのモデルを保存して再現** できます。
* **進捗や結果を上司やコラボレーターと共有** できます。
* **退化を検知し、パフォーマンスが低下した際には即座にアラート** を受け取れます。
* **モデルのパフォーマンスをベンチマークし、バージョンごとに比較** できます。

## コラボレーションチームを作成する

1. [W&B アカウントへのサインアップまたはログイン](https://app.wandb.ai/login?signup=true) を行います（無料です）。
2. ナビゲーションバーで **Invite Team** をクリックします。
3. チームを作成し、コラボレーターを招待します。
4. チームの設定については [Manage team settings]({{< relref "team-settings.md#privacy" >}}) を参照してください。

{{% alert %}}
**注意**: 新しいチームを作成できるのは、組織の管理者のみです。
{{% /alert %}}

## チームプロフィールの作成

チームのプロフィールページには、イントロダクションや、公開またはチームメンバーに表示される Reports や Projects を掲載できます。Reports や Projects、外部リンクを使ってチームを紹介しましょう。

* **優れた研究成果** を公開 Reports でアピールし、来訪者にハイライトできます
* **活発な Projects を目立たせ**、他のメンバーが素早く見つけられるようにします
* **コラボレーターを見つけるために**、会社や研究室のウェブサイト、発表論文への外部リンクを追加できます

## チームメンバーの削除

チーム管理者はチームの設定ページを開き、離脱するメンバーの名前の横にある削除ボタンをクリックできます。なお、そのユーザーがログした Run は、退会後もチーム内に残ります。

## チームの役割と権限の管理

チームへの招待時に、メンバーの役割を選択できます。選択できる役割は以下の通りです。

- **Admin**: チームの Admin は、他の Admin やチームメンバーの追加・削除が可能です。全てのプロジェクトの操作や完全な削除権限があります（Runs, Projects, Artifacts, Sweeps の削除などを含みます）。
- **Member**: 通常のチームメンバーです。デフォルトでは、Admin のみがチームメンバーを招待できます。この挙動を変更したい場合は [Manage team settings]({{< relref "team-settings.md#privacy" >}}) を参照してください。

チームメンバーは、自分が作成した Run のみ削除できます。例えば、AさんとBさんがいる場合、Bさんが自分の Project から Aさんの Project に Run を移動した場合でも、Aさんはその Run を削除できません。Admin であれば、全てのチームメンバーが作成した Run や Sweep Run の管理が可能です。
- **View-Only（エンタープライズ限定機能）**: View-Only メンバーは、チーム内の Runs、Reports、Workspace などの閲覧のみが可能です。Reports をフォローしたりコメントすることはできますが、プロジェクト概要や Reports、Runs の作成・編集・削除はできません。
- **カスタムロール（エンタープライズ限定機能）**: カスタムロールは、組織の Admin が **View-Only** または **Member** のいずれかのロールをベースに追加権限を加えて作成できる柔軟な権限管理です。チーム Admin は、これらのカスタムロールをチームメンバーに割り当てることができます。詳細は [Introducing Custom Roles for W&B Teams](https://wandb.ai/wandb_fc/announcements/reports/Introducing-Custom-Roles-for-W-B-Teams--Vmlldzo2MTMxMjQ3) を参照してください。
- **サービスアカウント（エンタープライズ限定機能）**: 詳しくは [Use service accounts to automate workflows]({{< relref "/guides/hosting/iam/authentication/service-accounts.md" >}}) をご覧ください。

{{% alert %}}
チームには Admin が複数人在籍することを推奨します。主要 Admin が不在時にも管理作業を継続できるため、ベストプラクティスです。
{{% /alert %}}

### チームサービスアカウントの挙動

* トレーニング環境でチームを設定した場合、そのチームのサービスアカウントを利用して、プライベート/パブリックいずれのプロジェクトにも Run をログできます。さらに、**WANDB_USERNAME** または **WANDB_USER_EMAIL** 変数が環境に設定され、そのユーザーがチームに所属していれば、Run を該当ユーザーに紐づけることができます。
* トレーニング環境でチームを設定せずにサービスアカウントを利用する場合、Run はそのサービスアカウントの親チーム内の指定プロジェクトにログされます。この場合も同様に、**WANDB_USERNAME** または **WANDB_USER_EMAIL** 変数が環境に存在し、該当ユーザーが親チームに所属していれば、そのユーザーに紐づけることが可能です。
* サービスアカウントは、自身の親チームとは異なるチームのプライベートプロジェクトへ Run をログすることはできません。ただし、プロジェクトの公開設定が「Open」の場合のみ、Run をログできます。

## 詳細な設定

### セキュアストレージコネクタ

チーム単位のセキュアストレージコネクタ機能により、各チームが自身のクラウドストレージバケットを W&B で利用できます。これにより、機密性が高いデータや厳格なコンプライアンス要件のあるチームでも、データアクセスコントロールとデータ分離が強化されます。詳細は [Secure Storage Connector]({{< relref "/guides/hosting/data-security/secure-storage-connector.md" >}}) をご参照ください。
