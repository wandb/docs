Model Context Protocol (MCP) は、AI アプリケーションが大規模言語モデル (LLM) と情報を交換するための標準化された通信プロトコルです。ハードウェアの互換性を革新したユニバーサル コネクターと同様に、MCP は LLM がさまざまなデータソースにアクセスし、外部ツールとやり取りするためのインターフェースを提供します。これにより、各種新しいサービスごとにカスタム インテグレーションを実装する必要がなくなります。

Weave のインテグレーションにより、MCP クライアントと MCP サーバー間のアクティビティをトレースできます。これにより、MCP ベースのシステム全体にわたるツール呼び出し、リソース アクセス、プロンプト生成の詳細な可視性が得られます。

<div id="how-it-works">
  ## 仕組み
</div>

<Warning>
  現在、このインテグレーションはクライアント側とサーバー側の操作を個別に捕捉しますが、それらの相互作用をエンドツーエンドで可視化する機能は提供されていません。MCP に OpenTelemetry トレースのサポートを追加し、エンドツーエンドの可観測性を実現するための提案が現在進行中です。詳細は [GitHub discussion #269](https://github.com/modelcontextprotocol/modelcontextprotocol/discussions/269) を参照してください。
</Warning>

Weave インテグレーションは、[`weave.op()`](../tracking/ops) デコレーターを使ってコア methodをパッチすることで、Model Context Protocol (MCP) の主要コンポーネントを自動的にトレースします。具体的には、[`mcp.server.fastmcp.FastMCP`](https://github.com/modelcontextprotocol/python-sdk/blob/b4c7db6a50a5c88bae1db5c1f7fba44d16eebc6e/src/mcp/server/fastmcp/server.py#L109) および [`mcp.ClientSession`](https://github.com/modelcontextprotocol/python-sdk/blob/b4c7db6a50a5c88bae1db5c1f7fba44d16eebc6e/src/mcp/client/session.py#L84) クラスのmethodをパッチします。

このインテグレーションにより、Weave は以下の MCP コンポーネントをトレースします：

* [Tools](https://modelcontextprotocol.io/specification/2025-06-18/server/tools)
* [Resources](https://modelcontextprotocol.io/specification/2025-06-18/server/resources)
* [Prompts](https://modelcontextprotocol.io/specification/2025-06-18/server/prompts)

<div id="use-the-integration">
  ## インテグレーションを使用する
</div>

Weave インテグレーションは、MCP サーバーとクライアントの両方で動作します。インストール後、`weave` をインポートする1行と、それを初期化する1行のコードを追加するだけで、トレースを有効にできます。

<div id="prerequisites">
  ### 前提条件
</div>

作業を始める前に、必須のパッケージをインストールしてください：

```bash
pip install -qq "mcp[cli]" weave
```


<div id="configuration">
  ### 設定
</div>

環境変数を使用して MCP インテグレーションを設定します：

* `MCP_TRACE_LIST_OPERATIONS`: `true` に設定すると、サーバー側とクライアント側の両方で、リスト操作 (`list_tools`、`list_resources`、`list_prompts`) をトレースします。

<div id="server-side-integration">
  ### サーバーサイド インテグレーション
</div>

MCP サーバーをトレースするには、既存の `FastMCP` の設定に 2 行追加します。1 行は Weave をインポートするため、もう 1 行はクライアントを初期化するためです。追加すると、ツール、リソース、プロンプトの各操作が自動的にトレースされます。

```python lines
# Weave をインポート（トレースに必須）
import weave
from mcp.server.fastmcp import FastMCP

# プロジェクト名を指定して Weave を初期化
weave_client = weave.init("my-project")

# MCP サーバーを設定する
mcp = FastMCP("Demo")

# ツールを定義（この呼び出しはトレースされます）
@mcp.tool()
def add(a: int, b: int) -> int:
    """2 つの数を加算します。"""
    return a + b

# リソースを定義（この呼び出しはトレースされます）
@mcp.resource("greeting://{name}")
def get_greeting(name: str) -> str:
    """パーソナライズされた挨拶を返します。"""
    return f"Hello, {name}!"

# プロンプトを定義（この呼び出しはトレースされます）
@mcp.prompt()
def review_code(code: str) -> str:
    """コードレビュー用のプロンプトを返します。"""
    return f"Please review this code:\n\n{code}"

# サーバーを起動
mcp.run(transport="stdio")
```


<div id="client-side-integration">
  ### クライアント側でのインテグレーション
</div>

クライアント側では、トレースに必要な変更は 2 つだけです。Weave をインポートし、初期化します。ツールの呼び出し、リソースへのアクセス、プロンプトのリクエストはすべて自動的にトレースされます。

```python lines
# Weave をインポート（トレースに必須）
import weave
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

# プロジェクト名で Weave を初期化する
weave_client = weave.init("my-project")

# MCP クライアントを設定して実行する
async with stdio_client(server_params) as (read, write):
    async with ClientSession(read, write) as session:
        # セッションを初期化する
        await session.initialize()
        
        # ツールを呼び出す（トレースされる）
        result = await session.call_tool("add", arguments={"a": 1, "b": 2})
        
        # リソースを読み取る（トレースされる）
        resource = await session.read_resource("greeting://user")
        
        # プロンプトを取得する（トレースされる）
        prompt = await session.get_prompt("review_code", arguments={"code": "print('Hello')"})
```

1. docs リポジトリをクローンし、`mcp_demo` の例ディレクトリに移動します：

   ```bash
   cd docs/weave/examples/mcp_demo
   ```

2. 必須の依存関係を手動でインストールします：

   ```bash
   pip install mcp[cli] weave
   ```

3. デモを実行します：

   ```bash
   python example_client.py example_server.py
   ```

   このコマンドはクライアントとサーバーの両方を起動します。クライアントは対話型の CLI を開始し、そこでさまざまな機能をテストできます。

クライアントインターフェースは、以下のコマンドをサポートしています。

| Command                 | Description                                                                 |
| ----------------------- | --------------------------------------------------------------------------- |
| `tools`                 | 利用可能なツールを一覧表示する                                                             |
| `resources`             | 利用可能なリソースを一覧表示する                                                            |
| `prompts`               | 利用可能なプロンプトを一覧表示する                                                           |

| `greeting <name>`       | パーソナライズされた挨拶を取得する                                                           |
| `user <id>`             | 指定した ID のユーザープロフィールを取得する                                                    |
| `config`                | アプリケーションの設定を取得する                                                            |
| `code-review <code>`    | コードレビュー用のプロンプトを生成する                                                         |
| `debug <error>`         | デバッグ用のプロンプトを生成する                                                            |
| `demo`                  | 利用可能なすべての機能の完全なデモを実行します。各機能が順次実行され、Weave UI にインタラクションの完全なトレースタイムラインが表示されます。 |
| `q`                     | セッションを終了する                                                                  |

<div id="understanding-the-example">
  ### 例の理解
</div>

`example_server.py` のサーバーは、以下を定義しています：

* *ツール*: `add()`、`calculate_bmi()`、`fetch_weather()` などの関数
* *リソース*: `greeting://{name}`、`config://app`、`users://{id}/profile` などのエンドポイント
* *プロンプト*: `review_code()` や `debug_error()` などのテンプレート

`weave.init()` でクライアントを初期化すると、Weave によりサーバー側のすべての操作が自動的にトレースされます。

`example_client.py` のクライアントは、以下を実演しています：

* MCP サーバーへの接続
* 利用可能なツール、リソース、プロンプトの発見
* パラメーターを指定したツールの呼び出し
* リソース URI からの読み取り
* 引数付きプロンプトの生成
* カスタムな method/関数を使った [`weave.op()`](../tracking/ops) の使用例

Weave は、すべてのクライアント側の呼び出しをトレースすることで、クライアントとサーバー間のやり取りの全体像を提供します。

<div id="why-is-mcp-tracing-needed">
  ### なぜ MCP トレースが必要なのですか？
</div>

LLM アプリケーション開発者として、あなたは次の 3 つのカテゴリのいずれかに該当します。

* *MCP サーバーサイド開発者*: MCP クライアントに対して複数のツール、リソース、プロンプトを公開したい場合です。既存のアプリケーションのツールやリソースなどを公開している、エージェントを構築している、またはオーケストレーターエージェントにより複数のエージェントをオーケストレーションしている状況です。

* *MCP クライアントサイド開発者*: クライアントサイドのアプリケーションを複数の MCP サーバーに接続したい場合です。クライアントサイドのロジックの中核部分は、どのツールを呼び出すか、どのリソースを取得するかを決定するための LLM 呼び出しです。

* *MCP サーバー兼クライアント開発者*: サーバーとクライアントの両方を開発している場合です。

最初の 2 つのカテゴリのいずれかに該当する場合、各ツールがいつ呼び出されるか、実行フローがどのようなものか、サーバーまたはクライアントサイドのロジック内の各コンポーネントごとのトークン数やレイテンシを把握したいと思うでしょう。

サーバーとクライアントの両方を開発している場合、統一されたトレースのタイムラインを確認できることで、サーバーおよびクライアントサイドのロジックの両方を迅速にイテレーションできます。

いずれの場合でも、オブザーバビリティ レイヤーにより、以下のことが可能になります。

* アプリケーションを迅速にイテレーションする
* ワークフローや実行ロジックを監査する
* ボトルネックを特定する
