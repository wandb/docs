<div id="trace-pages-load-slowly">
  ## Trace ページの読み込みが遅い
</div>

Trace ページの読み込みが遅い場合は、表示する行数を減らすことで読み込み時間を短縮できます。デフォルト値は `50` です。行数は UI から、またはクエリパラメーターを使って調整できます。

<div id="adjust-via-the-ui-recommended">
  ### UI から調整する（推奨）
</div>

Traces ページ右下の **Per page** コントロールを使用して、表示する行数を調整します。デフォルトの `50` に加えて、`10`、`25`、または `100` に設定できます。

<div id="use-query-parameters">
  ### クエリパラメーターを使用する
</div>

手動で設定したい場合は、クエリ URL 内の `pageSize` クエリパラメーターを、最大値である `100` 未満の任意の値に変更できます。

<div id="server-response-caching">
  ## サーバーレスポンスのキャッシュ
</div>

Weave は、同じクエリを繰り返し実行する場合や、ネットワーク帯域幅が限られている環境でのパフォーマンスを向上させるために、サーバーからのレスポンスをキャッシュする機能を提供します。現在はデフォルトでは無効になっていますが、将来のリリースではこの機能がデフォルトの動作になる予定です。

<div id="when-to-use-caching">
  ### キャッシュを使用するタイミング
</div>

サーバー レスポンスのキャッシュは、次のような場合に特に有用です：

- 同じクエリを頻繁に実行する場合
- ネットワーク帯域幅が限られている場合
- レイテンシーの高い環境で作業している場合
- オフラインで開発しており、後で使用するためにレスポンスをキャッシュしたい場合

この機能は、データセットに対して繰り返し評価を実行する場合に特に便利で、run 間でそのデータセットをキャッシュできるようになります。

<div id="how-to-enable-caching">
  ### キャッシュを有効にする方法
</div>

キャッシュを有効にするには、次の環境変数を設定できます。

```bash
# サーバーレスポンスのキャッシュを有効にする
export WEAVE_USE_SERVER_CACHE=true

# キャッシュサイズの上限を設定する（デフォルトは 1GB）
export WEAVE_SERVER_CACHE_SIZE_LIMIT=1000000000

# キャッシュディレクトリを設定する（省略可能、デフォルトは一時ディレクトリ）
export WEAVE_SERVER_CACHE_DIR=/path/to/cache
```


<div id="caching-behavior">
  ### キャッシュの動作
</div>

技術的には、この機能ではサーバーに対する冪等なリクエストがキャッシュされます。具体的には、次のものがキャッシュ対象になります:

- `obj_read`
- `table_query`
- `table_query_stats`
- `refs_read_batch`
- `file_content_read`

<div id="cache-size-and-storage-details">
  ### キャッシュサイズとストレージの詳細
</div>

キャッシュサイズは `WEAVE_SERVER_CACHE_SIZE_LIMIT`（バイト単位）で制御されます。実際に使用されるディスク容量は、次の 3 つの要素から構成されます。

1. 一定の 32KB のチェックサムファイル
2. 実行中のクライアントごとに最大約 4MB の Write-Ahead Log (WAL) ファイル（プログラム終了時に自動的に削除されます）
3. メインのデータベースファイル（少なくとも 32KB、最大で `WEAVE_SERVER_CACHE_SIZE_LIMIT`）

使用されるディスク容量の合計:

- 実行中 >= 32KB + 約 4MB + キャッシュサイズ
- 終了後 >= 32KB + キャッシュサイズ

たとえば、キャッシュサイズの上限を 5MB に設定した場合:

- 実行中: 最大約 9MB
- 終了後: 最大約 5MB

<div id="trace-data-is-truncated">
  ## トレースデータが切り捨てられる
</div>

Weave UI では、トレースデータが大きい場合に一部が途中で切り捨てられて表示されることがあります。これは、デフォルトのトレース出力が生のカスタム Python オブジェクトであり、Weave がシリアライズする方法が分からないために発生します。

大きなトレースデータが切り捨てられないようにするには、文字列の辞書を定義して、すべてのトレースデータを返すようにします。

```python lines
import weave

class MyObj:
    def __init__(self, x: int):
        self.x = x

    def __repr__(self):
        return f"MyObj(x={self.x})"

    def to_dict(self):
        return {"x": self.x}

@weave.op()
def make_my_obj():
    x = "s" * 10_000
    return MyObj(x)
```


<div id="long-eval-clean-up-times">
  ## 評価のクリーンアップに時間がかかる場合
</div>

大規模なデータセットで評価を実行する際のパフォーマンスを改善するには、次の 2 つの方法を組み合わせて使用してください。

<div id="flushing">
  ### フラッシュ処理
</div>

大きなデータセットで評価を実行する場合、バックグラウンドスレッドでデータセットがアップロードされている間、プログラムが実行を再開するまで長い時間がかかることがあります。これは一般的に、バックグラウンドでのクリーンアップが完了する前にメインスレッドの実行が終了した場合に発生します。`client.flush()` を呼び出すと、すべてのバックグラウンドタスクをメインスレッドで処理するよう強制し、メインスレッドの実行中も並列処理が行われるようにします。これは、ユーザーコードの処理がサーバーへのデータアップロード完了より先に終わってしまう場合に、パフォーマンスの改善につながる可能性があります。

例:

```python lines
client = weave.init("fast-upload")

# ... 評価のセットアップ
result = evaluation.Evaluate(dataset_id="my_dataset_id")

client.flush()
```


<div id="increasing-client-parallelism">
  ### クライアントの並列処理数の増加
</div>

クライアントの並列処理数は環境に応じて自動的に決定されますが、次の環境変数を使って手動で設定できます。

* `WEAVE_CLIENT_PARALLELISM`: 並列処理に利用可能なスレッド数です。この値を増やすと並列処理に利用できるスレッド数が増え、データセットのアップロードなどのバックグラウンドタスクのパフォーマンスが向上する可能性があります。

これは、`weave.init()` の `settings` 引数を使用してプログラムから設定することもできます。

```python lines
client = weave.init("fast-upload", settings={"client_parallelism": 100})
```


<div id="os-errors">
  ## OS エラー
</div>

<div id="errno-24-too-many-open-files">
  ### `[Errno 24]: Too many open files`
</div>

このエラーは、オペレーティングシステムで設定されている上限を開いているファイル数が超えた場合に発生します。Weave では、大規模な画像データセットを扱っていることが原因となる場合があります。Weave は画像処理に `PIL` を使用しており、プログラムの実行中はファイルディスクリプタを開いたままにします。

この問題を解決するには、`ulimit` を使用して開くことができるファイル数のシステム上限を `65,536` に引き上げてください。

```bash
ulimit -n 65536
```


<div id="trace-data-loss-in-worker-processes">
  ## ワーカープロセスにおけるトレースデータの損失
</div>

Weave は、アプリケーションのパフォーマンスへの影響を最小限に抑えるために、バックグラウンドスレッドでトレースデータをアップロードします。しかし、マルチプロセッシングやタスクキューシステム、あるいは Celery のようなワーカープロセスを使用している場合、ワーカープロセスがバックグラウンドスレッドによるトレースのアップロード完了前に終了すると、トレースデータが失われます。

ワーカープロセスでデータ損失を防ぐには、ワーカータスクが完了する前に `client.flush()` または `client.finish()` を呼び出して、バックグラウンドのアップロードが完了するよう確実にしてください。

これらのメソッドには異なる目的があります。

* `weave.flush()`: シンプルでサイレントなフラッシュ。Weave をワーカープロセスや CI 環境に統合している場合に推奨されます。
* `weave.finish()`: プログレスバーやステータスコールバックによる進捗フィードバックを含みます。対話的なスクリプトやノートブックでの使用に推奨されます。

両方のメソッドは、すべてのバックグラウンドアップロードが完了するまでブロックするため、ワーカープロセスが終了するときにトレースデータが失われないことを保証します。
次の例は `client.finish()` の使用方法を示します。

```python lines {15}
from celery import Celery
import weave

app = Celery('tasks')

@app.task
def process_task(input_data):
    weave.init("your-team-name/your-project-name")
    
    try:
        # トレースを作成するタスクロジック
        result = your_processing_function(input_data)
        
        # タスク完了前にすべてのトレースをアップロードする
        weave.finish()
        
        return result
    finally:
        pass
```

また、`with` コンテキストマネージャーを使用して、ブロックを抜ける際に自動的に `weave.finish()` を呼び出すこともできます。

```python lines {1}
with weave.init("your-team-name/your-project-name") as client:
    result = your_processing_function(input_data)

    return result
```

`weave.flush()` を使用することで、アプリケーションのパフォーマンスをさらに向上させることもできます。詳しくは [Flushing](#flushing) を参照してください。
