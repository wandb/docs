サービスアカウントは、人間ではないマシンユーザーを表し、1 つの Teams 内の複数のプロジェクトにまたがって、または複数の Teams 間で、一般的なタスクを自動的に実行できます。サービスアカウントは、CI/CD パイプライン、自動化されたトレーニングジョブ、その他のマシン間ワークフローに最適です。

<div id="overview">
  ## 概要
</div>

サービスアカウントは、個人のユーザー認証情報やハードコードされた認証情報を使わずに、W&amp;B のワークフローを安全に自動化する方法を提供します。サービスアカウントは、次の 2 つのスコープで作成できます。

* **組織スコープ**: 組織管理者が作成し、すべての Teams にアクセスできます。
* **チームスコープ**: チーム管理者が作成し、特定のチームにアクセスが限定されます。

サービスアカウントの APIキー を使用すると、呼び出し元はそのサービスアカウントのスコープ内にあるプロジェクトに対して読み取りまたは書き込みを行えます。これにより、W&amp;B Models での実験管理や W&amp;B Weave でのトレースのログに使用する自動化ワークフローを一元的に管理できます。

サービスアカウントは、特に次のような用途で役立ちます。

* **CI/CD パイプライン**: GitHub Actions、GitLab CI、Jenkins からモデルのトレーニング run を自動的にログする
* **スケジュールジョブ**: 夜間のモデル再トレーニング、定期的な評価 run、データ検証ワークフロー
* **本番環境のモニタリング**: 本番システムから推論メトリクスやモデル性能をログする
* **Jupyter ノートブック**: JupyterHub や Google Colab 環境で共有されるノートブック
* **Kubernetes ジョブ**: K8s クラスターで実行される自動化ワークフロー
* **Airflow/Prefect/Dagster**: ML パイプラインのオーケストレーションツール

<Note>
  サービスアカウントは、[Dedicated Cloud](/ja/platform/hosting/hosting-options/dedicated-cloud)、エンタープライズライセンス付きの [Self-Managed instances](/ja/platform/hosting/hosting-options/self-managed)、および [Multi-tenant Cloud](/ja/platform/hosting/hosting-options/multi_tenant_cloud) のエンタープライズアカウントで利用できます。
</Note>

  ## 組織スコープのサービスアカウント
</div>

チームスコープのサービスアカウントを使用するモデル トレーニング環境または生成 AI アプリ環境でチームを設定しない場合、モデルの run や Weave トレースは、サービスアカウントの親チーム内にある指定のプロジェクトにログされます。この場合、参照先のユーザーがサービスアカウントの親チームのメンバーでない限り、`WANDB_USERNAME` または `WANDB_USER_EMAIL` 変数を使用したユーザー属性の付与は *機能しません*。

<Warning>
  チームスコープのサービスアカウントは、親チームとは異なるチームにある [チームスコープまたは 制限付きスコープのプロジェクト](/ja/platform/hosting/iam/access-management/restricted-projects/#visibility-scopes) に run をログすることはできませんが、別のチーム内の Open 公開範囲のプロジェクトには run をログできます。
</Warning>

<div id="external-service-accounts">
  ### 外部サービスアカウント
</div>

組み込みのサービスアカウントに加えて、W&amp;B は JSON Web Token (JWT) を発行できるアイデンティティプロバイダー (IdP) を用いた [アイデンティティ フェデレーション](./identity_federation#external-service-accounts) を利用することで、W&amp;B SDK と CLI でチームスコープの外部サービスアカウントもサポートしています。

<div id="best-practices">
  ## ベストプラクティス
</div>

組織でサービスアカウントを安全かつ効率的に使用するために、以下の推奨事項に従ってください。

* **シークレットマネージャーを使用する**: サービスアカウントのAPIキーは、プレーンテキストの設定ファイルではなく、安全なシークレット管理システム (例: AWS Secrets Manager、HashiCorp Vault、Azure Key Vault) に保存してください。

* **最小権限の原則**: 必要なプロジェクトのみにアクセスを制限するため、可能な場合は組織スコープのアカウントではなく、チームスコープのサービスアカウントを作成してください。

* **ユースケースごとに一意のサービスアカウント**: 監査性を高め、きめ細かなアクセス制御を可能にするため、異なるオートメーションワークフローごとに個別のサービスアカウントを作成してください (例: CI/CD 用に 1 つ、定期的な再トレーニング用にもう 1 つ) 。

* **定期的な監査**: アクティブなサービスアカウントを定期的に確認し、不要になったものは削除してください。サービスアカウントのアクティビティを監視するには、監査ログを確認してください。

* **安全なAPIキーの処理**:
  * APIキーをバージョン管理にコミットしない
  * アプリケーションにキーを渡すには環境変数を使用する
  * 誤って公開された場合はキーをローテーションする

* **命名規則**: サービスアカウントの目的が分かる説明的な名前を使用してください。
  * 良い例: `ci-model-training`, `nightly-eval-pipeline`, `prod-inference-monitor`
  * 避ける例: `service-account-1`, `test-sa`, `temp`

* **ユーザー属性の付与**: 複数のチームメンバーが同じオートメーションワークフローを使用する場合は、各 run を誰がトリガーしたかを追跡するために `WANDB_USERNAME` または `WANDB_USER_EMAIL` を設定してください。
  ```bash
  export WANDB_API_KEY="<service_account_key>"
  export WANDB_USERNAME="john.doe@company.com"
  ```

* **環境設定**: チームスコープのサービスアカウントでは、Runs が正しいチームにログされるよう、常に `WANDB_ENTITY` を設定してください。
  ```bash
  export WANDB_ENTITY="ml-team"
  export WANDB_PROJECT="production-models"
  ```

* **エラー処理**: サービスアカウント認証情報に関する問題を迅速に特定できるように、認証失敗に対する適切なエラー処理とアラートを実装してください。

* **ドキュメント**: 以下についてのドキュメントを維持してください。
  * 存在するサービスアカウントとその目的
  * 各サービスアカウントを使用するシステム/ワークフロー
  * 各アカウントを担当するチームの連絡先情報

<div id="troubleshooting">
  ## トラブルシューティング
</div>

一般的な問題と解決策:

* **&quot;Unauthorized&quot; エラー**: APIキーが正しく設定されており、サービスアカウントが対象のプロジェクトに 액세스 できることを確認してください
* **Runs が表示されない**: `WANDB_ENTITY` が正しいチーム名に設定されていることを確認してください
* **ユーザー属性の付与が機能しない**: `WANDB_USERNAME` で指定したユーザーがチームのメンバーであることを確認してください
* **制限付きプロジェクトへのアクセスが拒否される**: 制限付きプロジェクトのアクセスリストにサービスアカウントを明示的に追加してください
