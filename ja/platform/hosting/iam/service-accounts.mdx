サービスアカウントは、人間ではなくマシンによるユーザーを表し、1 つの Teams 内の複数のプロジェクト間や複数の Teams 間で共通のタスクを自動的に実行できます。サービスアカウントは、CI/CD パイプライン、自動トレーニングジョブ、その他のマシン同士のワークフローに最適です。

<div id="overview">
  ## 概要
</div>

サービスアカウントは、個人のユーザー認証情報やハードコードされた認証情報を使わずに、W&amp;B のワークフローを自動化するための安全な手段を提供します。サービスアカウントは次の 2 つのスコープで作成できます。

* **Organization スコープ**: 組織管理者によって作成され、すべてのチームにまたがるアクセス権を持つ
* **Team スコープ**: チーム管理者によって作成され、特定のチームに限定されたアクセス権を持つ

サービスアカウントの API キー によって、呼び出し元はそのサービスアカウントのスコープ内にあるプロジェクトに対して読み取りや書き込みを行えます。これにより、W&amp;B Models での実験管理用の自動化ワークフローや、W&amp;B Weave でのトレースのログ記録を一元的に管理できます。

サービスアカウントは特に次のような用途に有用です。

* **CI/CD パイプライン**: GitHub Actions、GitLab CI、Jenkins からのモデル トレーニング run を自動でログする
* **スケジュールされたジョブ**: 毎晩のモデル再トレーニング、定期的な評価 run、データ検証ワークフロー
* **本番監視**: 本番システムから推論メトリクスやモデルパフォーマンスをログする
* **Jupyter ノートブック**: JupyterHub や Google Colab 環境で共有されるノートブック
* **Kubernetes ジョブ**: K8s クラスターで動作する自動化ワークフロー
* **Airflow/Prefect/Dagster**: ML パイプラインのオーケストレーションツール

<Note>
  サービスアカウントは、[Dedicated Cloud](/ja/platform/hosting/hosting-options/dedicated_cloud)、エンタープライズライセンス付きの [Self-Managed インスタンス](/ja/platform/hosting/hosting-options/self-managed)、および [Multi-tenant Cloud](/ja/platform/hosting/hosting-options/multi_tenant_cloud) のエンタープライズアカウントで利用できます。
</Note>

  ## 組織スコープのサービスアカウント
</div>

組織にスコープされたサービスアカウントは、チームに関係なく、その組織内のすべてのプロジェクトに対して、[制限付きプロジェクト](/ja/platform/hosting/iam/access-management/restricted-projects/#visibility-scopes) を除き、読み取りおよび書き込み権限を持ちます。組織スコープのサービスアカウントが制限付きプロジェクトにアクセスできるようにするには、そのプロジェクトの管理者が明示的にサービスアカウントをそのプロジェクトに追加する必要があります。

team-scoped service account を使用する model training 環境や Generative AI アプリケーション環境で team を設定しない場合、モデルの run や Weave traces は、その service account の親 team 内にある指定された project にログされます。このような状況では、参照されているユーザーが service account の親 team の一員でない限り、`WANDB_USERNAME` や `WANDB_USER_EMAIL` 変数を使ったユーザー属性の付与は *機能しません*。

<Warning>
  team-scoped service account は、親 team とは異なる team にある [team または restricted-scoped project](/ja/platform/hosting/iam/access-management/restricted-projects/#visibility-scopes) には run をログできませんが、別の team にある公開範囲が open の project には run をログできます。
</Warning>

<div id="external-service-accounts">
  ### 外部サービスアカウント
</div>

組み込みのサービスアカウントに加えて、W&amp;B は JSON Web Token (JWT) を発行できるアイデンティティプロバイダー (IdP) を用いた [Identity federation](./identity_federation#external-service-accounts) を利用することで、W&amp;B SDK と CLI でチーム単位の外部サービスアカウントもサポートしています。

<div id="best-practices">
  ## ベストプラクティス
</div>

組織内でサービスアカウントを安全かつ効率的に利用するために、次の推奨事項に従ってください。

* **シークレットマネージャーを使用する**: サービスアカウントの API キー は、プレーンテキストの設定ファイルではなく、AWS Secrets Manager、HashiCorp Vault、Azure Key Vault などの安全なシークレット管理システムに保存します。

* **最小権限の原則**: 必要なプロジェクトへのアクセスに限定するため、可能な限り組織スコープのアカウントではなくチームスコープのサービスアカウントを作成します。

* **ユースケースごとに固有のサービスアカウントを使用する**: 監査性の向上ときめ細かなアクセス制御を可能にするため、CI/CD 用、定期的な再トレーニング用など、自動化ワークフローごとに別々のサービスアカウントを作成します。

* **定期的な監査**: アクティブなサービスアカウントを定期的に確認し、使用されていないものを削除します。監査ログを確認してサービスアカウントのアクティビティを監視します。

* **API キー の安全な取り扱い**:
  * API キーをバージョン管理にコミットしない
  * 環境変数を使ってアプリケーションにキーを渡す
  * キーが誤って漏えいした場合にはローテーションする

* **命名規則**: サービスアカウントの目的が分かる説明的な名前を使用します:
  * 良い例: `ci-model-training`, `nightly-eval-pipeline`, `prod-inference-monitor`
  * 避ける例: `service-account-1`, `test-sa`, `temp`

* **ユーザー帰属の明確化**: 複数のチームメンバーが同じ自動化ワークフローを使用する場合、それぞれの run を誰がトリガーしたかを追跡できるように、`WANDB_USERNAME` または `WANDB_USER_EMAIL` を設定します:
  ```bash
  export WANDB_API_KEY="<service_account_key>"
  export WANDB_USERNAME="john.doe@company.com"
  ```

* **環境設定**: チームスコープのサービスアカウントでは、run が正しいチームにログされるよう、必ず `WANDB_ENTITY` を設定します:
  ```bash
  export WANDB_ENTITY="ml-team"
  export WANDB_PROJECT="production-models"
  ```

* **エラーハンドリング**: サービスアカウントの認証情報に関する問題を迅速に特定できるよう、認証失敗時の適切なエラーハンドリングとアラートを実装します。

* **ドキュメント化**: 次の内容をドキュメントとして整備・維持します:
  * どのサービスアカウントが存在し、それぞれの目的は何か
  * 各サービスアカウントをどのシステム／ワークフローが使用しているか
  * 各アカウントを担当するチームの連絡先情報

<div id="troubleshooting">
  ## トラブルシューティング
</div>

よくある問題とその解決策：

* **&quot;Unauthorized&quot; エラー**: API キー が正しく設定されており、サービスアカウントが対象のプロジェクトにアクセスできることを確認します
* **Runs が表示されない**: `WANDB_ENTITY` が正しいチーム名に設定されているか確認します
* **ユーザーの紐付けが機能しない**: `WANDB_USERNAME` で指定したユーザーがそのチームのメンバーであることを確認します
* **制限付きプロジェクトへのアクセスが拒否される**: サービスアカウントを制限付きプロジェクトのアクセスリストに明示的に追加します
