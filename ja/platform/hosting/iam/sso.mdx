W&amp;B は OpenID Connect (OIDC) 互換のアイデンティティプロバイダーをサポートしており、Okta、Keycloak、Auth0、Google、Entra などの外部アイデンティティプロバイダーを通じて、ユーザー ID とグループメンバーシップを管理できます。

<div id="openid-connect-oidc">
  ## OpenID Connect (OIDC)
</div>

W&amp;B は、外部のアイデンティティプロバイダー (IdP) と連携するために、次の OIDC 認証フローをサポートしています。

1. Implicit Flow with Form Post
2. Authorization Code Flow with Proof Key for Code Exchange (PKCE)

これらのフローではユーザーを認証し、アクセス制御の管理に必要なアイデンティティ情報を ID トークンの形式で W&amp;B に提供します。

ID トークンは JWT であり、ユーザーの氏名、ユーザー名、メールアドレス、所属グループなどのアイデンティティ情報が含まれます。W&amp;B はこのトークンを使用してユーザーを認証し、システム内の適切なロールまたはグループにマッピングします。

W&amp;B では、アクセス トークンはユーザーに代わって API へのリクエストを認可するために使用されますが、主に必要なのはユーザー認証とアイデンティティ情報であるため、W&amp;B が必要とするのは ID トークンのみです。

環境変数を使用して、[Dedicated Cloud](/ja/platform/hosting/hosting-options/dedicated-cloud) または [Self-Managed](/ja/platform/hosting/hosting-options/self-managed) インスタンスの [IAM オプションを設定](/ja/platform/hosting/iam/advanced_env_vars) できます。

[Dedicated Cloud](/ja/platform/hosting/hosting-options/dedicated-cloud) または [Self-Managed](/ja/platform/hosting/hosting-options/self-managed) の W&amp;B インストール向けにアイデンティティプロバイダーを設定するには、各種 IdP について以下のガイドラインに従ってください。SaaS 版の W&amp;B を使用している場合は、組織向けの Auth0 テナントの設定について [support@wandb.com](mailto:support@wandb.com) までお問い合わせください。

<div id="configure-your-idp">
  ## IdP を設定する
</div>

このセクションでは、OIDC 用にアイデンティティプロバイダー (IdP) を設定する方法を説明します。詳細は、ご利用の IdP に対応するタブを選択して確認してください。

<Tabs>
  <Tab title="Cognito">
    認可のために AWS Cognito を設定するには、以下の手順に従ってください。

    1. まず、AWS アカウントにサインインし、[AWS Cognito](https://aws.amazon.com/cognito/) アプリにアクセスします。

       <Frame>
         <img src="/images/hosting/setup_aws_cognito.png" alt="AWS Cognito の設定" />
       </Frame>

    2. IdP でアプリケーションを設定するために、許可されたコールバック URL を指定します。
       * コールバック URL として `http(s)://YOUR-W&B-HOST/oidc/callback` を追加します。`YOUR-W&B-HOST` は W&amp;B ホストのパスに置き換えてください。

    3. IdP がユニバーサルログアウトをサポートしている場合は、ログアウト URL を `http(s)://YOUR-W&B-HOST` に設定します。`YOUR-W&B-HOST` は W&amp;B ホストのパスに置き換えてください。

       たとえば、アプリケーションが `https://wandb.mycompany.com` で実行されている場合は、`YOUR-W&B-HOST` を `wandb.mycompany.com` に置き換えます。

       以下の画像は、AWS Cognito で許可されたコールバック URL とサインアウト URL を指定する方法を示しています。

       <Frame>
         <img src="/images/hosting/setup_aws_cognito_ui_settings.png" alt="ホスト設定" />
       </Frame>

       *wandb/local* はデフォルトで、[`implicit` grant と `form_post` response type](https://auth0.com/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) を使用します。

       また、[PKCE Code Exchange](https://www.oauth.com/oauth2-servers/pkce/) フローを使用する `authorization_code` grant を実行するように *wandb/local* を設定することもできます。

    4. 1 つ以上の OAuth grant type を選択して、AWS Cognito がアプリにトークンをどのように配信するかを設定します。

    5. W&amp;B では特定の OpenID Connect (OIDC) スコープが必要です。AWS Cognito アプリで以下を選択します。

       * &quot;openid&quot;
       * &quot;profile&quot;
       * &quot;email&quot;

       たとえば、AWS Cognito アプリの UI は次の画像のようになります。

       <Frame>
         <img src="/images/hosting/setup_aws_required_fields.png" alt="必須フィールド" />
       </Frame>

       設定ページで **Auth Method** を選択するか、OIDC&#95;AUTH&#95;METHOD 環境変数を設定して、どの grant を使用するかを *wandb/local* に指定します。

       Auth Method は `pkce` に設定する必要があります。

    6. Client ID と OIDC issuer の URL が必要です。OpenID discovery document は `$OIDC_ISSUER/.well-known/openid-configuration` で利用できる必要があります

       たとえば、**User Pools** セクション内の **App Integration** タブにある Cognito IdP URL に User Pool ID を追加することで、issuer URL を生成できます。

       <Frame>
         <img src="/images/hosting/setup_aws_cognito_issuer_url.png" alt="AWS Cognito issuer URL" />
       </Frame>

       IDP URL に &quot;Cognito domain&quot; を使用しないでください。Cognito は `https://cognito-idp.$REGION.amazonaws.com/$USER_POOL_ID` でディスカバリードキュメントを提供します

    次に、[W&amp;B で SSO を設定する](#set-up-sso-in-w%26b)。
  </Tab>

    2. 左側で **Applications** を選択し、続けて再度 **Applications** を選択します。

       <Frame>
         <img src="/images/hosting/okta_create_new_app_integration.png" alt="Create App integration ボタン" />
       </Frame>

    4. &quot;Create a new app integration,&quot; という画面で、**OIDC - OpenID Connect** と **Single-Page Application** を選択します。次に、&quot;Next.&quot; をクリックします。
       <Frame>

    OIDC Issuer URL の形式は次のとおりです: `https://COMPANY.okta.com`。COMPANY を対応する値に置き換えてください。これも控えておいてください。

  <Tab title="Entra">
    Azure AD (Entra ID) は、W&amp;B 向けに2つの OIDC 設定モードをサポートしています。セキュリティ要件に合った設定を選択してください。

    <Warning>
      設定を混在させないでください。Azure AD で &quot;Single-page application&quot; を選択した場合は、client secret を指定しないでください。client secret が必要な場合は、プラットフォームのタイプとして &quot;Web&quot; を選択する必要があります。

<div id="set-up-sso-in-wb">
  ## W&amp;B で SSO を設定する
</div>

SSO を設定するには、管理者権限と次の情報が必要です。

* OIDC Client ID
* OIDC Auth method (`implicit` または `pkce`)
* OIDC Issuer URL
* OIDC Client Secret (任意。IdP の設定方法によって異なります)

IdP で OIDC Client Secret が必要な場合は、[環境変数](/ja/platform/hosting/env-vars) `GORILLA_OIDC_SECRET` を指定します。

* W&amp;B App で **System Console** &gt; **Settings** &gt; **Advanced** &gt; **User Spec** に移動し、以下のように `extraENV` セクションに `GORILLA_OIDC_SECRET` を追加します。
* Helm では、以下のように `values.global.extraEnv` を設定します。
  ```yaml
  values:
  global:
      extraEnv:
      GORILLA_OIDC_SECRET="<your_secret>"
  ```

<Note>
  SSO の設定後にインスタンスにログインできなくなった場合は、環境変数 `LOCAL_RESTORE=true` を設定してインスタンスを再起動できます。これにより、一時パスワードがコンテナーのログに出力され、SSO が無効になります。SSO の問題を解消したら、SSO を再度有効にするため、その環境変数を削除する必要があります。
</Note>

<div id="security-assertion-markup-language-saml">
  ## Security Assertion Markup Language (SAML)
</div>

W&amp;B は SAML をサポートしていません。
