W&amp;B による OpenID Connect (OIDC) 互換のアイデンティティプロバイダーのサポートにより、Okta、Keycloak、Auth0、Google、Entra などの外部アイデンティティプロバイダーを通じて、ユーザー ID とグループメンバーシップを管理できます。

<div id="openid-connect-oidc">
  ## OpenID Connect (OIDC)
</div>

W&amp;B は、外部の Identity Provider (IdP) と連携するために、次の OIDC 認証フローをサポートしています。

1. Implicit Flow with Form Post
2. Authorization Code Flow with Proof Key for Code Exchange (PKCE)

これらのフローはユーザーを認証し、アクセス制御を管理するために必要な ID トークンという形で、W&amp;B に必要なアイデンティティ情報を提供します。

ID トークンは JWT であり、ユーザーの氏名、ユーザー名、メールアドレス、グループメンバーシップなどのアイデンティティ情報を含みます。W&amp;B はこのトークンを使用してユーザーを認証し、システム内の適切なロールやグループにマッピングします。

W&amp;B のコンテキストでは、アクセストークンはユーザーに代わって API へのリクエストを許可しますが、W&amp;B の主な関心事はユーザー認証とアイデンティティであるため、必要なのは ID トークンだけです。

環境変数を使用して [IAM オプションを構成](/ja/platform/hosting/iam/advanced_env_vars) し、[Dedicated Cloud](/ja/platform/hosting/hosting-options/dedicated-cloud) または [Self-Managed](/ja/platform/hosting/hosting-options/self-managed) インスタンス向けに設定できます。

[Dedicated Cloud](/ja/platform/hosting/hosting-options/dedicated-cloud) または [Self-Managed](/ja/platform/hosting/hosting-options/self-managed) の W&amp;B インストールで Identity Provider を構成する際は、さまざまな IdP 向けに以下のガイドラインに従ってください。SaaS 版の W&amp;B を使用している場合は、組織向けの Auth0 テナント構成について [support@wandb.com](mailto:support@wandb.com) までお問い合わせください。

<div id="configure-your-idp">
  ## IdP を構成する
</div>

このセクションでは、OIDC 用のアイデンティティプロバイダー (IdP) を構成する方法を説明します。利用している IdP に対応するタブを選択して、詳細を確認してください。

<Tabs>
  <Tab title="Cognito">
    認可のために AWS Cognito をセットアップするには、次の手順に従います。

    1. まず AWS アカウントにサインインし、[AWS Cognito](https://aws.amazon.com/cognito/) アプリに移動します。

       <Frame>
         <img src="/images/hosting/setup_aws_cognito.png" alt="AWS Cognito のセットアップ" />
       </Frame>

    2. IdP 内でアプリケーションを設定するために、許可されたコールバック URL を指定します。
       * コールバック URL として `http(s)://YOUR-W&B-HOST/oidc/callback` を追加します。`YOUR-W&B-HOST` をお使いの W&amp;B ホストのパスに置き換えてください。

    3. IdP がユニバーサルログアウトをサポートしている場合、ログアウト URL を `http(s)://YOUR-W&B-HOST` に設定します。`YOUR-W&B-HOST` をお使いの W&amp;B ホストのパスに置き換えてください。

       たとえば、アプリケーションが `https://wandb.mycompany.com` で動作している場合、`YOUR-W&B-HOST` を `wandb.mycompany.com` に置き換えます。

       下の画像は、AWS Cognito で許可されるコールバック URL とサインアウト URL を指定する方法を示しています。

       <Frame>
         <img src="/images/hosting/setup_aws_cognito_ui_settings.png" alt="ホスト設定" />
       </Frame>

       *wandb/local* はデフォルトで、[`implicit` grant と `form_post` レスポンスタイプ](https://auth0.com/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) を使用します。

       また、*wandb/local* を設定して、[PKCE Code Exchange](https://www.oauth.com/oauth2-servers/pkce/) フローを使用する `authorization_code` grant を実行させることもできます。

    4. 1 つ以上の OAuth grant タイプを選択して、AWS Cognito がトークンをアプリに配信する方法を設定します。

    5. W&amp;B には特定の OpenID Connect (OIDC) スコープが必要です。AWS Cognito アプリで次の項目を選択します。

       * &quot;openid&quot;
       * &quot;profile&quot;
       * &quot;email&quot;

       たとえば、AWS Cognito アプリの UI は次の画像のような見た目になります。

       <Frame>
         <img src="/images/hosting/setup_aws_required_fields.png" alt="必須フィールド" />
       </Frame>

       設定ページで **Auth Method** を選択するか、環境変数 `OIDC_AUTH_METHOD` を設定して、どの grant タイプを使用するかを *wandb/local* に指定します。

       Auth Method は `pkce` に設定する必要があります。

    6. クライアント ID と OIDC issuer の URL が必要です。OpenID ディスカバリー ドキュメントは `$OIDC_ISSUER/.well-known/openid-configuration` で利用可能でなければなりません。

       たとえば、**User Pools** セクション内の **App Integration** タブにある Cognito IdP URL に User Pool ID を追加することで、Issuer URL を生成できます。

       <Frame>
         <img src="/images/hosting/setup_aws_cognito_issuer_url.png" alt="AWS Cognito issuer URL" />
       </Frame>

       IdP URL として &quot;Cognito domain&quot; を使用しないでください。Cognito はディスカバリー ドキュメントを `https://cognito-idp.$REGION.amazonaws.com/$USER_POOL_ID` で提供します。

    次に、[W&amp;B で SSO をセットアップ](#set-up-sso-in-w%26b)します。
  </Tab>

    3. &quot;Create App integration&quot; をクリックします。
       <Frame>
         <img src="/images/hosting/okta_create_new_app_integration.png" alt="Create App integration ボタン" />

    4. &quot;Create a new app integration&quot; という画面で、**OIDC - OpenID Connect** と **Single-Page Application** を選択し、その後 &quot;Next&quot; をクリックします。
       <Frame>

    OIDC issuer URL の形式は次のとおりです: `https://COMPANY.okta.com`。COMPANY を該当する値に置き換えてメモしておきます。

  <Tab title="Entra">
    Azure AD (Entra ID) は W&amp;B 向けに2つの OIDC 設定モードをサポートしています。セキュリティ要件に合った設定を選択してください。

    * [Public Client (パブリッククライアント) ](#public-client)：クライアントシークレットを使わずに PKCE を使用します。設定が簡単で、多くのデプロイに適しています。
    * [Confidential Client](#confidential-client): セキュリティを強化するため、クライアントシークレット付きの PKCE を使用します。`GORILLA_OIDC_SECRET` 環境変数を設定する必要がある場合に必須です。

    <Warning>
      設定を混在させないでください。Azure AD で「Single-page application」を選択した場合、クライアント シークレットは入力しないでください。クライアント シークレットが必要な場合は、プラットフォームの種類として「Web」を選択する必要があります。
    </Warning>

<div id="set-up-sso-in-wb">
  ## W&amp;B で SSO を設定する
</div>

SSO を設定するには、管理者権限と次の情報が必要です。

* OIDC Client ID
* OIDC Auth method (`implicit` または `pkce`)
* OIDC Issuer URL
* OIDC Client Secret (任意。IdP の設定方法に依存)

IdP が OIDC Client Secret を要求する場合は、[environment variables](/ja/platform/hosting/env-vars) の `GORILLA_OIDC_SECRET` を指定します。

* W&amp;B App で **System Console** &gt; **Settings** &gt; **Advanced** &gt; **User Spec** に移動し、`extraENV` セクションに `GORILLA_OIDC_SECRET` を次のように追加します。
* Helm では、`values.global.extraEnv` を次のように設定します。
  ```yaml
  values:
  global:
      extraEnv:
      GORILLA_OIDC_SECRET="<your_secret>"
  ```

<Note>
  SSO を設定した後にインスタンスへログインできなくなった場合は、環境変数 `LOCAL_RESTORE=true` を設定してインスタンスを再起動してください。これにより一時パスワードがコンテナのログに出力され、SSO が無効化されます。SSO に関する問題を解決したら、SSO を再度有効化するために必ずこの環境変数を削除してください。
</Note>

<div id="security-assertion-markup-language-saml">
  ## Security Assertion Markup Language (SAML)
</div>

W&amp;B は SAML をサポートしていません。
