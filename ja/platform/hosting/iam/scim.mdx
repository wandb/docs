<div id="overview">
  ## 概要
</div>

System for Cross-domain Identity Management (SCIM) API を使用すると、インスタンスまたは組織の管理者が、自身の W&amp;B 組織内のユーザー、グループ、およびカスタムロールを管理できます。SCIM グループは W&amp;B Teams に対応します。

W&amp;B の SCIM API は Okta を含む主要なアイデンティティプロバイダーと互換性があり、ユーザーの自動プロビジョニングおよびデプロビジョニングが可能です。Okta やその他のアイデンティティプロバイダーを用いた SSO 設定については、[SSO ドキュメント](/ja/platform/hosting/iam/sso/) を参照してください。

SCIM API との連携方法を示した実践的な Python の例については、[`wandb-scim`](https://github.com/wandb/examples/tree/master/wandb-scim) リポジトリを参照してください。

<div id="supported-features">
  ### サポートされている機能
</div>

* **フィルタリング**: API は `/Users` と `/Groups` エンドポイントでのフィルタリングをサポートします
* **PATCH 操作**: リソースの部分的な更新のための PATCH をサポートします
* **ETag サポート**: 競合検出のために ETag を用いた条件付き更新をサポートします
* **サービスアカウントによる認証**: 組織のサービスアカウントが API にアクセスできます

<Note>
  複数の Enterprise [Multi-tenant SaaS](/ja/platform/hosting/hosting-options/multi_tenant_cloud) 組織の管理者である場合は、SCIM API リクエストの送信先となる組織を必ず設定し、あなたの API キーを使用して送信された SCIM API リクエストが正しい組織に対して行われるようにする必要があります。プロフィール画像をクリックし、**User Settings** をクリックして、**Default API organization** 設定を確認してください。

  選択したホスティングオプションによって、このページの例で使用されている `<host-url>` プレースホルダーの値が決まります。

  また、例では `abc` や `def` といったユーザー ID を使用しています。実際のリクエストおよびレスポンスでは、ユーザー ID にはハッシュ化された値が使用されます。
</Note>

<div id="authentication">
  ## 認証
</div>

主な違いを確認したうえで、認証方法としてユーザー ID を使うかサービスアカウントを使うかを選択します。

<div id="key-differences">
  ### 主な違い
</div>

* 想定される利用者: ユーザー は対話的な単発の管理作業に最適であり、サービスアカウント は自動化や インテグレーション (CI/CD、プロビジョニングツール) に最適です。
* 認証情報: ユーザー は username と API キーを送信し、サービスアカウント は API キー のみを送信します (username は不要) 。
* Authorization ヘッダーのペイロード: ユーザー は `username:API-KEY` をエンコードし、サービスアカウント は `:API-KEY` (先頭にコロン) をエンコードします。
* スコープと権限: どちらも管理者権限が必要ですが、サービスアカウント は組織スコープでヘッドレスなため、自動化に対してより明確な監査証跡を提供します。
* 認証情報の取得場所: ユーザー は自分の API キーを User Settings からコピーし、サービスアカウント キーは組織の Service account タブにあります。
* Multi-tenant Cloud: 複数の Multi-tenant Cloud 組織にアクセスできる場合、SCIM API 呼び出しが意図した組織にルーティングされるように、Default API organization を必ず設定する必要があります。

<div id="users">
  ### Users
</div>

対話的な管理タスクを実行する際は、自身の管理者認証情報を使用してください。HTTP `Authorization` ヘッダーは `Basic <base64(username:API-KEY)>` という形式に構成します。

例えば、`demo:p@55w0rd` として認証を行います。

```bash
Authorization: Basic ZGVtbzpwQDU1dzByZA==
```

<div id="service-accounts">
  ### サービスアカウント
</div>

自動化やインテグレーションには、組織スコープのサービスアカウントを使用します。HTTP `Authorization` ヘッダーは `Basic <base64(:API-KEY)>` の形式で設定します (先頭のコロンと空のユーザー名に注意してください) 。サービスアカウントの API キーは、組織のダッシュボード内の **Service account** タブで確認できます。詳しくは、[Organization-scoped service accounts](/ja/platform/hosting/iam/service-accounts/#organization-scoped-service-accounts) を参照してください。

例えば、API キー `sa-p@55w0rd` を使って認証を行う場合:

```bash
Authorization: Basic OnNhLXBANTV3MHJk
```

<div id="user-management">
  ## ユーザー管理
</div>

SCIM のユーザーリソースは W&amp;B のユーザーに対応します。これらのエンドポイントを使用して、組織内のユーザーを管理してください。

<div id="get-user">
  ### ユーザーを取得
</div>

組織内の特定のユーザーに関する情報を取得します。

<Note>この操作ではサービス アカウントの情報は取得されません。</Note>

<div id="endpoint">
  #### エンドポイント
</div>

* **URL**: `<host-url>/scim/Users/{id}`
* **メソッド**: GET

<div id="parameters">
  #### パラメーター
</div>

| パラメーター | 型 | 必須 | 説明 |
|-----------|------|----------|-------------|
| `id` | string | Yes | ユーザーの一意な ID |

<div id="examples">
  #### 例
</div>

<Tabs>
  <Tab title="ユーザー取得リクエスト">
    ```bash
    GET /scim/Users/abc
    ```
  </Tab>

  <Tab title="ユーザー取得レスポンス">
    ```bash
    (Status 200)
    ```

    ```json
    {
        "active": true,
        "daysActive": 42,
        "displayName": "Dev User 1",
        "emails": {
            "Value": "dev-user1@example.com",
            "Display": "",
            "Type": "",
            "Primary": true
        },
        "id": "abc",
        "lastActiveAt": "2023-10-15T14:32:10Z",
        "meta": {
            "resourceType": "User",
            "created": "2023-10-01T00:00:00Z",
            "lastModified": "2023-10-01T00:00:00Z",
            "location": "Users/abc"
        },
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User"
        ],
        "userName": "dev-user1"
    }
    ```

    レスポンスには、組織内でのユーザーのアクティビティに関する詳細が含まれます:

    * **`daysActive`**: ユーザーが組織内でアクティブだった合計日数。
    * **`lastActiveAt`**: ユーザーの直近のアクティビティの ISO 8601 タイムスタンプ。ユーザーが一度もアクティブでなかった場合は `null` を返します。

    「アクティブ」の定義はデプロイタイプによって異なります:

    * **Dedicated Cloud / Self-Managed**: ユーザーがサインインし、W&amp;B App 内の任意のページを開く、run をログに記録する、SDK を使用する、または何らかの形で W&amp;B サーバーとやり取りした場合、そのユーザーはアクティブとみなされます。
    * **Multi-tenant Cloud**: ユーザーが 2025-05-08 以降に、その組織をスコープとした監査可能なアクションを行った場合、そのユーザーはアクティブとみなされます。完全な一覧は [Audit logging actions](/ja/platform/hosting/monitoring-usage/audit-logging#actions) を参照してください。
  </Tab>
</Tabs>

<div id="list-users">
  ### ユーザーの一覧
</div>

組織内のすべてのユーザーの一覧を取得します。

<Note>この操作では、サービス アカウントは取得されません。</Note>

<div id="filter-users">
  #### ユーザーを絞り込む
</div>

`/Users` エンドポイントでは、ユーザー名またはメールアドレスでユーザーを絞り込むことができます：

* `userName eq "value"` - ユーザー名で絞り込み
* `emails.value eq "value"` - メールアドレスで絞り込み

<div id="example">
  ##### 例
</div>

```bash
GET /scim/Users?filter=userName eq "john.doe"
GET /scim/Users?filter=emails.value eq "john@example.com"
```

<div id="create-user">
  ### ユーザーを作成
</div>

組織内で新しいユーザーを作成します。

<div id="endpoint">
  #### エンドポイント
</div>

* **URL**: `<host-url>/scim/Users`
* **メソッド**: POST

<div id="parameters">
  #### パラメーター
</div>

| パラメーター | 型 | 必須 | 説明 |
|-----------|------|----------|-------------|
| `emails` | array | Yes | email オブジェクトの配列。primary email アドレスを 1 つ含める必要があります |
| `userName` | string | Yes | 新しいユーザーのユーザー名 |

<div id="examples">
  #### 例
</div>

<Tabs>
  <Tab title="ユーザー作成リクエスト（Dedicated／セルフマネージド）">
    ```bash
    POST /scim/Users
    ```

    ```json
    {
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User"
        ],
        "emails": [
            {
                "primary": true,
                "value": "dev-user2@example.com"
            }
        ],
        "userName": "dev-user2"
    }
    ```
  </Tab>

  <Tab title="ユーザー作成リクエスト（マルチテナント）">
    ```bash
    POST /scim/Users
    ```

    ```json
    {
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User",
            "urn:ietf:params:scim:schemas:extension:teams:2.0:User"
        ],
        "emails": [
            {
                "primary": true,
                "value": "dev-user2@example.com"
            }
        ],
        "userName": "dev-user2",
        "urn:ietf:params:scim:schemas:extension:teams:2.0:User": {
            "teams": ["my-team"]

    }
    ```

<div id="delete-user">
  ### ユーザーの削除
</div>

<Warning>
  **管理者アクセスを維持する**

  インスタンスまたは組織内に、常に少なくとも 1 人の管理者ユーザーが存在するようにする必要があります。そうしないと、どのユーザーも組織の W&amp;B アカウントを設定または維持できなくなります。組織が SCIM やその他の自動化されたプロセスを使用して W&amp;B からユーザーのデプロビジョニングを行っている場合、デプロビジョニング操作によって、インスタンスまたは組織から最後に残っている管理者が意図せず削除される可能性があります。

  運用手順の策定に関する支援や、管理者アクセスの復旧については、[support](mailto:support@wandb.com) までお問い合わせください。
</Warning>

組織からユーザーを完全に削除します。

<Note>この操作はユーザーに対してのみ有効で、サービスアカウントには使用できません。サービスアカウントは W&amp;B チーム の設定で削除してください。</Note>

<div id="endpoint">
  #### エンドポイント
</div>

* **URL**: `<host-url>/scim/Users/{id}`
* **メソッド**: DELETE

<div id="parameters">
  #### パラメーター
</div>

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `id` | string | Yes | 削除対象のユーザーの一意の ID |

<div id="examples">
  #### 例
</div>

<Tabs>
  <Tab title="ユーザー削除リクエスト">
    ```bash
    DELETE /scim/Users/abc
    ```
  </Tab>

  <Tab title="ユーザー削除レスポンス">
    ```bash
    (Status 204)
    ```
  </Tab>
</Tabs>

<Note>
  ユーザーを一時的に無効化する場合は、`PATCH` エンドポイントを使用する [Deactivate user](#deactivate-user) API を参照してください。
</Note>

<div id="deactivate-user">
  ### ユーザーの無効化
</div>

組織内のユーザーを無効化します。実際の挙動はデプロイ種別によって異なります。

* **Dedicated Cloud** / **Self-Managed**: ユーザーの `active` フィールドを `false` に設定します。無効化したユーザーの組織へのアクセスを復元するには、[ユーザーの再有効化](#reactivate-user) を参照してください。
* **Multi-tenant Cloud**: ユーザーを組織から削除します。ユーザーのアクセスを復元するには、組織にユーザーを再追加してください。[ユーザーの作成](#create-user-request-multi-tenant) を参照してください。Multi-tenant Cloud では、ユーザーのアカウントは組織によって管理されません。

<Note>この操作はユーザーに対してのみ有効で、サービスアカウントには使用できません。サービスアカウントの無効化はサポートされていません。チームのサービスアカウントは W&amp;B Team の設定で管理してください。</Note>

<div id="assign-organization-role">
  ### 組織ロールの割り当て
</div>

ユーザーに組織レベルのロールを割り当てます。

<Note>この操作はユーザーに対してのみ有効で、サービスアカウントには使用できません。サービスアカウントではカスタムロールはサポートされていません。</Note>

<div id="endpoint">
  #### エンドポイント
</div>

* **URL**: `<host-url>/scim/Users/{id}`
* **Method**: PATCH

<div id="parameters">
  #### パラメーター
</div>

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `id` | string | Yes | ユーザーの一意の ID |
| `op` | string | Yes | `replace` |
| `path` | string | Yes | `organizationRole` |
| `value` | string | Yes | ロール名 (`admin` または `member`)  |

<Note>
  組織スコープの `viewer` ロールは非推奨であり、UI から割り当てることはできません。SCIM を使用してユーザーに `viewer` ロールを割り当てた場合:

  * 組織内では `member` ロールが割り当てられます。
  * Models の `full` seat の代わりに、Models の `viewer` seat が割り当てられます。これにより、Models への閲覧専用アクセスと Registry へのフルアクセスが可能になります。利用可能な Models seat がない場合は、`Seat limit reached` エラーがログに記録され、そのメンバーは Models へのアクセスなしで追加されます。seat が利用可能になった場合は、後から更新できます。
  * Weave の `full` seat の代わりに、Weave の `viewer` seat が割り当てられます。これにより、Weave への閲覧専用アクセスが可能になります。利用可能な Weave seat がない場合は、`Seat limit reached` エラーがログに記録され、そのメンバーは Weave へのアクセスなしで追加されます。seat が利用可能になった場合は、後から更新できます。
  * 組織レベルで可視なレジストリでは、Registry の `viewer` ロールが割り当てられます。
</Note>

| --------------- | ------------------ | --------------------------------------------- |
| `displayName`   | String             | Yes                                           |
| `members`       | Multi-Valued Array | Yes (`value` サブフィールドは必須で、その値はユーザー ID に対応します)  |
| `storageBucket` | Object             | No                                            |

`storageBucket` オブジェクトを含めることで、チーム作成時にチームレベルの [独自バケットの利用 (BYOB)](/ja/platform/hosting/data-security/secure-storage-connector) を設定できます。省略した場合、チームはデフォルトまたはインスタンスレベルのストレージを使用します。BYOB ガイドを参照して、バケット (ポリシー、CORS、認証情報) を準備し、プロバイダーごとのストレージアドレス形式を確認してください。`storageBucket` オブジェクトには、次のサブフィールドがあります。

* **Required**: `name` (バケット名) 、`provider` (`coreweave`、`aws`、`gcp`、`azure`、`minio` のいずれか) 。大文字と小文字は区別されます。
* **Optional**: `path` (バケット内のパスプレフィックス) 、`kmsKeyId` (暗号化用の KMS キー。たとえば AWS で使用) 、`awsExternalId` (AWS クロスアカウントアクセス) 、`azureTenantId` (Azure テナント ID) 、`azureClientId` (Azure マネージド アイデンティティ クライアント ID) 。

W&amp;B は、チームを作成する前に、バケットが存在しアクセス可能であることを検証します。検証に失敗した場合、SCIM リクエストは失敗し、チームは作成されません。

無効な `provider` 値を指定すると、許可されている値を一覧表示する SCIM エラーとともに `400 Bad Request` が返されます。

<div id="update-team">
  ### チームを更新する
</div>

* **Endpoint**: **`<host-url>/scim/Groups/{id}`**
* **Method**: PATCH
* **Description**: 既存のチームのメンバーシップ一覧を更新します。
* **Supported Operations**: `add` メンバー、`remove` メンバー、`replace` メンバー

<Note>
  - remove 操作は RFC 7644 SCIM プロトコル仕様に従います。特定のユーザーを削除するにはフィルター構文 `members[value eq "{user_id}"]` を使用し、チームからすべてのユーザーを削除するには `members` を使用します。

    **ユーザーの識別**: メンバー操作内の `{user_id}` には、次のいずれかを指定できます:

    * W&amp;B ユーザー ID
    * メールアドレス (例: &quot;user@example.com&quot;)
  - これらの操作が適用されるのはユーザーのみで、サービスアカウントには使用できません。チームのサービスアカウントを更新するには、W&amp;B Team の設定から行ってください。
</Note>

<Info>
  リクエスト内の `{team_id}` は実際のチーム ID に、`{user_id}` は実際のユーザー ID またはメールアドレスに置き換えてください。
</Info>

<div id="etag-support">
  ### ETag サポート
</div>

SCIM API は、同時更新による競合を防ぐための条件付き更新に ETag をサポートしています。ETag は `ETag` レスポンスヘッダーと `meta.version` フィールドで返されます。

<div id="etags">
  #### ETags
</div>

ETags を使用するには：

1. **現在の ETag を取得**: リソースを GET するとき、レスポンスの ETag ヘッダーを確認する
2. **条件付き更新**: 更新時に `If-Match` ヘッダーに ETag を含める

<div id="examples">
  #### 例
</div>

```
# ユーザーを取得して ETag を記録する
GET /scim/Users/abc
# レスポンスに含まれる: ETag: W/"xyz123"

# ETag を使用して更新する
PATCH /scim/Users/abc
If-Match: W/"xyz123"

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
    "Operations": [
        {
            "op": "replace",
            "path": "organizationRole",
            "value": "admin"
        }
    ]
}
```

`412 Precondition Failed` エラー応答は、そのリソースが取得された後に変更されたことを示します。

<div id="error-handling">
  ### エラー処理
</div>

SCIM API は標準的な SCIM エラー応答を返します。

| Status Code | Description |
|-------------|-------------|
| `200` | Success |
| `201` | Created |
| `204` | No Content (削除成功) |
| `400` | Bad Request - パラメーターまたはリクエストボディが無効 |
| `401` | Unauthorized - 認証に失敗しました |
| `403` | Forbidden - 権限が不足しています |
| `404` | Not Found - リソースが存在しません |
| `409` | Conflict - リソースが既に存在します |
| `412` | Precondition Failed - ETag が一致しません |
| `500` | Internal Server Error |

<div id="implementation-differences-per-deployment-type">
  ### デプロイタイプごとの実装の違い
</div>

W&amp;B は 2 つの SCIM API 実装を提供しており、それぞれで利用できる機能が異なります。

| 機能                      | 専用クラウド       | セルフマネージド |
|---------------------------|-----------------|--------------|
| ユーザーのメールアドレスを更新 |        -        |    ✓   |
| ユーザーの表示名を更新          |        -        |    ✓   |
| ユーザーの無効化               |    ✓      |    ✓   |
| ユーザーの再有効化             |        -        |    ✓   |
| 1 ユーザーあたり複数のメールアドレス |    ✓      |      -       |

<div id="limitations">
  ## 制限
</div>

* **最大結果数**: 1 回のリクエストにつき 9999 件。
* **シングルテナント環境**: ユーザーごとにメールアドレスは 1 つのみサポートされます。
* **チーム削除**: SCIM 経由ではサポートされません (W&amp;B の Web インターフェースを使用してください) 。
* **ユーザーの再有効化**: Multi-tenant Cloud 環境ではサポートされません。
* **シート数の上限**: 組織のシート数の上限に達している場合、操作が失敗することがあります。
