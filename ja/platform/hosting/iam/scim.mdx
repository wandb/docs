## 概要

System for Cross-domain Identity Management (SCIM) APIを使用すると、インスタンスまたは組織の管理者がW\&B組織内のユーザー、グループ、カスタムロールを管理できます。SCIMグループはW\&B Teamsにマッピングされます。

W\&BのSCIM APIは、Oktaを含む主要なIDプロバイダーと互換性があり、ユーザーの自動プロビジョニングおよびデプロビジョニングを可能にします。Oktaおよびその他のIDプロバイダーとのSSO設定については、[SSOドキュメント](/ja/platform/hosting/iam/sso)をご覧ください。

SCIM APIとのやり取りを示す実用的なPythonの例については、[`wandb-scim`](https://github.com/wandb/examples/tree/master/wandb-scim) リポジトリをご覧ください。

### サポートされている機能

* **フィルタリング**: APIは`/Users` および`/Groups` エンドポイントのフィルタリングをサポートしています
* **PATCH操作**: 部分的なリソース更新のためのPATCHをサポートしています
* **ETagサポート**: 競合検出のためのETagを使用した条件付き更新
* **サービスアカウント認証**: 組織のサービスアカウントがAPIにアクセスできます

<Note>
  複数のEnterprise[Multi-tenant SaaS](/ja/platform/hosting/hosting-options/multi_tenant_cloud) 組織の管理者である場合、SCIM APIリクエストの送信先となる組織を設定して、APIキーを使用して送信されたSCIM APIリクエストが正しい組織に影響するようにする必要があります。プロフィール画像をクリックし、**ユーザー設定** をクリックして、設定**デフォルトAPI組織**を確認してください。

  選択したホスティングオプションによって、このページの例で使用される`<host-url>` プレースホルダーの値が決まります。

  また、例ではユーザーIDとして`abc` や`def` などが使用されています。実際のリクエストとレスポンスでは、ユーザーIDにハッシュ値が使用されます。
</Note>

## 認証

組織の管理者は**Bearerトークン** または**HTTP Basic** 認証情報で認証できます。どちらのスタイルも、キーが適用される場所で*同じAPIキー文字列* を使用します。主な違いを確認した上で、ユーザーIDまたは組織スコープのサービスアカウントを選択してください。

### 主な違い

* 使用すべき対象: ユーザーはインタラクティブな単発の管理操作に最適です。サービスアカウントは自動化および統合（CI/CD、プロビジョニングツール）に最適です。
* Credentials: Users send username and API key for Basic auth; service accounts send only an API key (no username) for Basic auth. For Bearer auth, send only the API key in the header (no username).
* BearerとBasicの違い: Bearerは`Authorization: Bearer <API-KEY>` にキーをそのまま使用します。Basicは`Authorization: Basic <base64(...)>` を使用します（ユーザーは`username:API-KEY` をエンコードします。サービスアカウントは`:API-KEY` を先頭にコロンと空のユーザー名でエンコードします）。
* スコープと権限: インスタンスまたは組織の管理者ユーザー、あるいは[組織スコープのサービスアカウント](/ja/platform/hosting/iam/service-accounts#organization-scoped-service-accounts) のAPIキーを使用してください。[チームスコープのサービスアカウント](/ja/platform/hosting/iam/service-accounts#team-scoped-service-accounts) のキーはSCIM APIに認証できません。SCIMを使用するサービスアカウントは組織スコープでヘッドレスであり、自動化のための明確な監査証跡をサポートします。
* 認証情報の取得場所: ユーザーはユーザー設定からAPIキーをコピーします。組織スコープのサービスアカウントキーは、組織ダッシュボードの**サービスアカウント** タブにあります。
* Multi-tenant Cloud: 複数のMulti-tenant Cloud組織にアクセスできる場合、SCIM API呼び出しが意図した組織にルーティングされるように、デフォルトAPI組織を設定する必要があります。

### Bearerトークン

APIキーをBearerトークンとして送信します:

```bash
Authorization: Bearer <API-KEY>
```

`<API-KEY>` の値は、そのプリンシパルのHTTP Basic認証でパスワードとして使用するのと同じ文字列です。BearerリクエストではキーをBase64エンコードする必要はありません。

<Note>
  SCIM APIのBearer認証は、W\&B Multi-tenant Cloud、およびDedicated CloudとSelf-Managed v0.79.0以降で利用可能です。
</Note>

以下の例では`<YOUR_API_KEY>` をプレースホルダーとして使用しています。管理者ユーザーまたは組織スコープのサービスアカウントの実際のキーに置き換えてください。

**ユーザーの一覧表示**

```bash
curl -s -S \
  -H "Authorization: Bearer <YOUR_API_KEY>" \
  -H "Content-Type: application/scim+json" \
  "<host-url>/scim/Users"
```

**ユーザーの作成**

```bash
curl -s -S -X POST \
  -H "Authorization: Bearer <YOUR_API_KEY>" \
  -H "Content-Type: application/scim+json" \
  "<host-url>/scim/Users" \
  -d '{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
    "userName": "dev-user2",
    "emails": [{"primary": true, "value": "dev-user2@example.com"}]
  }'
```

詳細については、[ユーザーの作成](#create-user)をご覧ください。

### ユーザー

インタラクティブな管理タスクを実行する際は、個人の管理者認証情報を使用してください。HTTP`Authorization` ヘッダーを`Basic <base64(username:API-KEY)>` として構築してください。

例えば、`demo:p@55w0rd` として認証する場合:

```bash
Authorization: Basic ZGVtbzpwQDU1dzByZA==
```

### サービスアカウント

自動化または統合には、組織スコープのサービスアカウントを使用してください。HTTP`Authorization` ヘッダーを`Basic <base64(:API-KEY)>` として構築してください（先頭のコロンと空のユーザー名に注意）。サービスアカウントのAPIキーは、組織ダッシュボードの**サービスアカウント** タブで確認できます。[組織スコープのサービスアカウント](/ja/platform/hosting/iam/service-accounts#organization-scoped-service-accounts)を参照してください。

例えば、APIキー`sa-p@55w0rd` で認証する場合:

```bash
Authorization: Basic OnNhLXBANTV3MHJk
```

## ユーザー管理

SCIMユーザーリソースはW\&Bユーザーにマッピングされます。これらのエンドポイントを使用して、組織内のユーザーを管理します。

### ユーザーの取得

組織内の特定のユーザーの情報を取得します。

<Note>この操作ではサービスアカウントは取得されません。</Note>

#### エンドポイント

* **URL**: `<host-url>/scim/Users/{id}`
* **メソッド**: GET

#### パラメーター

| パラメーター | 型      | 必須 | 説明         |
| ------ | ------ | -- | ---------- |
| `id`   | string | はい | ユーザーの一意のID |

#### 例


    * **Dedicated Cloud / Self-Managed**: ユーザーがサインイン、W\&B Appのページを開く、runのログ記録、SDKの使用、またはW\&Bサーバーとの何らかのやり取りを行った場合にアクティブとみなされます。
    * **Multi-tenant Cloud**: 2025年5月8日以降に組織にスコープされた監査可能なアクションを実行した場合にアクティブとみなされます。完全なリストについては[監査ログアクション](/ja/platform/hosting/monitoring-usage/audit-logging#actions)をご覧ください。

</Tabs>

### ユーザーの一覧表示

組織内のすべてのユーザーのリストを取得します。

<Note>この操作ではサービスアカウントは取得されません。</Note>

#### ユーザーのフィルタリング

The `/Users` エンドポイントは、ユーザー名またはメールアドレスによるユーザーのフィルタリングをサポートしています：

* `userName eq "value"` - ユーザー名でフィルタリング
* `emails.value eq "value"` - メールアドレスでフィルタリング

##### 例

```bash
GET /scim/Users?filter=userName eq "john.doe"
GET /scim/Users?filter=emails.value eq "john@example.com"
```

### ユーザーの作成

組織に新しいユーザーを作成します。

#### エンドポイント

* **URL**: `<host-url>/scim/Users`
* **メソッド**: POST

#### パラメーター

| パラメーター       | 型      | 必須  | 説明                                                                                                     |
| ------------ | ------ | --- | ------------------------------------------------------------------------------------------------------ |
| `emails`     | array  | はい  | メールオブジェクトの配列。プライマリメールを含める必要があります                                                                       |
| `userName`   | string | はい  | 新しいユーザーのユーザー名                                                                                          |
| `modelsSeat` | string | いいえ | Models シートレベル。`full`、`viewer`、または `none` のいずれか。デフォルトは `full`。 **Dedicated Cloud および Self-Managed のみ。** |
| `weaveRole`  | string | いいえ | Weave ロールレベル。`full`、`viewer`、または `none` のいずれか。デフォルトは `full`。 **Dedicated Cloud および Self-Managed のみ。**  |

#### 例

<Tabs>
  <Tab title="Create User リクエスト（Dedicated/Self-Managed）">
    ```bash
    POST /scim/Users
    ```

    ```json
    {
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User"
        ],
        "emails": [
            {
                "primary": true,
                "value": "dev-user2@example.com"
            }
        ],
        "userName": "dev-user2",
        "modelsSeat": "full",
        "weaveRole": "full"
    }
    ```
  </Tab>

  <Tab title="Create User リクエスト（Multi-tenant）">
    ```bash
    POST /scim/Users
    ```

    ```json
    {
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User",
            "urn:ietf:params:scim:schemas:extension:teams:2.0:User"
        ],
        "emails": [
            {
                "primary": true,
                "value": "dev-user2@example.com"
            }
        ],
        "userName": "dev-user2",
        "urn:ietf:params:scim:schemas:extension:teams:2.0:User": {
            "teams": ["my-team"]
        }
    }
    ```
  </Tab>

### ユーザーの削除

<Warning>
  **管理者アクセスの維持**

  インスタンスまたは組織に常に少なくとも1人の管理者ユーザーが存在することを確認する必要があります。そうしないと、組織の W\&B アカウントを設定または管理できるユーザーがいなくなります。組織が SCIM または別の自動化プロセスを使用して W\&B からユーザーをプロビジョニング解除する場合、プロビジョニング解除操作によってインスタンスまたは組織から最後の管理者が誤って削除される可能性があります。

  運用手順の開発に関するサポート、または管理者アクセスの復元については、[サポート](mailto:support@wandb.com)にお問い合わせください。
</Warning>

組織からユーザーを完全に削除します。

<Note>この操作はユーザーにのみ有効で、サービスアカウントには使用できません。サービスアカウントを削除するには、W\&B Team の設定で行ってください。</Note>

#### エンドポイント

* **URL**: `<host-url>/scim/Users/{id}`
* **メソッド**: DELETE

#### パラメーター

| パラメーター | 型      | 必須 | 説明             |
| ------ | ------ | -- | -------------- |
| `id`   | string | はい | 削除するユーザーの一意のID |

#### 例

<Tabs>
  <Tab title="Delete User リクエスト">

  </Tab>

<Note>
  ユーザーを一時的に無効化するには、[ユーザーの無効化](#deactivate-user) API（`PATCH` エンドポイントを使用）を参照してください。
</Note>

### ユーザーメールの更新

ユーザーのプライマリメールアドレスを更新します。
**Multi-tenant Cloud ではサポートされていません**（ユーザーのアカウントは組織によって管理されていないため）。

#### エンドポイント

* **URL**: `<host-url>/scim/Users/{id}`
* **メソッド**: PATCH

#### パラメーター

| パラメーター  | 型      | 必須 | 説明                |
| ------- | ------ | -- | ----------------- |
| `id`    | string | はい | ユーザーの一意のID        |
| `op`    | string | はい | `replace`         |
| `path`  | string | はい | `emails`          |
| `value` | array  | はい | 新しいメールオブジェクトを含む配列 |

#### 例

### 組織ロールの割り当て

ユーザーに組織レベルのロールを割り当てます。

<Note>この操作はユーザーにのみ有効で、サービスアカウントには使用できません。カスタムロールはサービスアカウントではサポートされていません。</Note>

#### Endpoint

* **URL**: `<host-url>/scim/Users/{id}`
* **Method**: PATCH

#### Parameters

| Parameter | Type   | Required | Description                 |
| --------- | ------ | -------- | --------------------------- |
| `id`      | string | Yes      | ユーザーの一意のID                  |
| `op`      | string | Yes      | `replace`                   |
| `path`    | string | Yes      | `organizationRole`          |
| `value`   | string | Yes      | ロール名 (`admin` または `member`) |

<Note>
  組織スコープの `viewer` ロールは非推奨となり、UI では割り当てられなくなりました。SCIM を使用して `viewer` ロールをユーザーに割り当てた場合:

  * 組織内で `member` ロールが割り当てられます。
  * ユーザーの `modelsSeat` は `viewer` ではなく `full` に設定されます。これにより、Models への閲覧専用アクセスと Registry へのフルアクセスが許可されます。Models のシートが利用できない場合、`Seat limit reached` エラーが返されます。シートが利用可能になった後で更新できます。
  * ユーザーの `weaveRole` は `viewer` ではなく `full` に設定されます。これにより、Weave への閲覧専用アクセスが許可されます。
  * 既存のすべてのチームおよびプロジェクトロールは `viewer` に設定されます。
  * 組織レベルで表示可能なレジストリにおいて、Registry の `viewer` ロールが割り当てられます。

  `member` または `admin` 組織ロールを割り当てても、ユーザーの `modelsSeat` または `weaveRole` は変更されません。
</Note>

### Teamの作成

* **エンドポイント**: **`<host-url>/scim/Groups`**
* **メソッド**: POST
* **説明**: 新しいteamリソースを作成します。
* **サポートされているフィールド**:

| フィールド           | 型                  | 必須                                        |
| --------------- | ------------------ | ----------------------------------------- |
| `displayName`   | String             | Yes                                       |
| `members`       | Multi-Valued Array | Yes（`value` サブフィールドは必須で、ユーザーIDにマッピングされます） |
| `storageBucket` | Object             | No                                        |

team作成時に [Bring your own bucket (BYOB)](/ja/platform/hosting/data-security/secure-storage-connector) をteamレベルで設定するには、`storageBucket` オブジェクトを含めます。省略した場合、teamはデフォルトまたはインスタンスレベルのストレージを使用します。BYOBガイドを参照してバケットのプロビジョニング（ポリシー、CORS、認証情報）を行い、プロバイダーごとのストレージアドレス形式を確認してください。`storageBucket` オブジェクトには以下のサブフィールドがあります：

* **必須**: `name`（バケット名）、`provider`（次のいずれか `COREWEAVE`、`AWS`、`AZURE`、`GCP`、または `MINIO`）。値は大文字・小文字を区別します。表示されている通り大文字を使用してください。
* **オプション**: `path`（バケット内のパスプレフィックス）、`kmsKeyId`（暗号化用のKMSキー、例：AWS向け）、`awsExternalId`（AWSクロスアカウントアクセス）、`azureTenantId`（AzureテナントID）、`azureClientId`（AzureマネージドアイデンティティのクライアントID）。

W\&Bはteamを作成する前に、バケットが存在し到達可能であることを検証します。検証に失敗した場合、SCIMリクエストは失敗し、teamは作成されません。

無効な `provider` 値は `400 Bad Request` を返し、許可された値を一覧表示するSCIMエラーが含まれます。

### チームを更新する

* **Endpoint**: **`<host-url>/scim/Groups/{id}`**
* **Method**: PATCH
* **Description**: 既存のチームのメンバーシップリストを更新します。
* **Supported Operations**: `add` member, `remove` member, `replace` members

<Note>
  - remove操作はRFC 7644 SCIMプロトコル仕様に従います。特定のユーザーを削除するにはfilter構文 `members[value eq "{user_id}"]` を使用し、チームからすべてのユーザーを削除するには `members` を使用します。

    **User Identification**: `{user_id}` はmember操作において以下のいずれかを指定できます:

    * W\&B ユーザーID
    * メールアドレス（例: "[user@example.com](mailto:user@example.com)")
  - これらの操作はユーザーにのみ有効で、サービスアカウントには使用できません。チームのサービスアカウントはW\&B Teamの設定で更新してください。
</Note>

<Info>
  &#x20;を実際のチームIDに、`{team_id}` を実際のチームIDに、`{user_id}` をリクエスト内の実際のユーザーIDまたはメールアドレスに置き換えてください。
</Info>

## 高度な機能

### ETag サポート

SCIM APIは、同時変更の競合を防ぐための条件付き更新にETagをサポートしています。ETagは`ETag`レスポンスヘッダーと`meta.version`フィールドに返されます。

#### ETags

Etagを使用するには:

1. **現在のETagを取得する**: リソースをGETする際に、レスポンスのETagヘッダーを確認してください
2. **条件付き更新**: 更新時に`If-Match`ヘッダーにETagを含めてください

#### 例

```
# Get user and note ETag
GET /scim/Users/abc
# Response includes: ETag: W/"xyz123"

# Update with ETag
PATCH /scim/Users/abc
If-Match: W/"xyz123"

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
    "Operations": [
        {
            "op": "replace",
            "path": "organizationRole",
            "value": "admin"
        }
    ]
}
```

`412 Precondition Failed`エラーレスポンスは、リソースが取得後に変更されたことを示します。

### エラー処理

SCIM APIは標準のSCIMエラーレスポンスを返します:

| ステータスコード | 説明                             |
| -------- | ------------------------------ |
| `200`    | 成功                             |
| `201`    | 作成済み                           |
| `204`    | コンテンツなし（削除成功）                  |
| `400`    | 不正なリクエスト - 無効なパラメータまたはリクエストボディ |
| `401`    | 未認証 - 認証失敗                     |
| `403`    | 禁止 - 権限不足                      |
| `404`    | 見つかりません - リソースが存在しません          |
| `409`    | 競合 - リソースが既に存在します              |
| `412`    | 前提条件失敗 - ETagの不一致              |
| `500`    | 内部サーバーエラー                      |

### デプロイメントタイプ別の実装の違い

W\&Bは2つの独立したSCIM API実装を維持しており、それらの間で機能が異なります:

| 機能                       | マルチテナントクラウド | 専用クラウドおよびセルフマネージド |
| ------------------------ | ----------- | ----------------- |
| ユーザーメールの更新               | -           | ✓                 |
| ユーザー表示名の更新               | -           | ✓                 |
| ユーザーの無効化                 | ✓           | ✓                 |
| ユーザーの再有効化                | -           | ✓                 |
| ユーザーごとの複数メール             | ✓           | -                 |
| 作成/更新時に`modelsSeat`を設定する | -           | ✓                 |
| 作成/更新時に`weaveRole`を設定する  | -           | ✓                 |

## 制限事項

* **最大結果数**: リクエストあたり9999件。
* **シングルテナント環境**: ユーザーごとに1つのメールのみサポートします。
* **チームの削除**: SCIMではサポートされていません（W\&BのWebインターフェースを使用してください）。
* **ユーザーの再有効化**: マルチテナントクラウド環境ではサポートされていません。
* **シート制限**: 組織のシート制限に達した場合、操作が失敗することがあります。
