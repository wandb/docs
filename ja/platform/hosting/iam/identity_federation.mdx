---
title: SDK でアイデンティティ フェデレーションを使用する
description: "JSON Web Token (JWT) を使用したアイデンティティ フェデレーションにより、APIキーなしで W&B SDK および CLI で認証します。"
---

W&amp;B SDK で組織の認証情報を使用してサインインするには、アイデンティティ フェデレーションを使用します。W&amp;B 組織管理者が組織に対して SSO を設定している場合は、すでに組織の認証情報を使用して W&amp;B App UI にサインインしています。その意味で、アイデンティティ フェデレーションは W&amp;B SDK 向けの SSO に似ていますが、JSON Web Token (JWT) を直接使用する点が異なります。アイデンティティ フェデレーションは、APIキーの代替として使用できます。

[RFC 7523](https://datatracker.ietf.org/doc/html/rfc7523) は、SDK でのアイデンティティ フェデレーションの基盤となる仕様です。

<Note>
  アイデンティティ フェデレーションは、すべてのプラットフォーム タイプ (SaaS Cloud、専用クラウド、セルフマネージド インスタンス) において、`Enterprise` プランで `Preview` として利用できます。ご不明な点があれば、W&amp;B チームにお問い合わせください。
</Note>

<Note>
  このドキュメントでは、`identity provider` と `JWT issuer` という用語を同じ意味で使用します。この機能の文脈では、どちらも同じものを指します。
</Note>

<div id="jwt-issuer-setup">
  ## JWT issuer の設定
</div>

最初の手順として、組織管理者は、ご利用の W&amp;B 組織と公開アクセス可能な JWT issuer の間でフェデレーションを設定する必要があります。

* 組織のダッシュボードの **Settings** タブにアクセスします
* **Authentication** オプションで、**Set up JWT Issuer** をクリックします
* テキストボックスに JWT issuer URL を追加し、**Create** をクリックします

W&amp;B は、`${ISSUER_URL}/.well-known/openid-configuration` のパスで OIDC discovery document を自動的に探し、discovery document 内の関連 URL から JSON Web Key Set (JWKS) を検索しようとします。JWKS は JWT のリアルタイム検証に使用され、それらが該当する identity provider によって発行されたことを確認します。

<div id="using-the-jwt-to-access-wb">
  ## JWT を使用して W&amp;B に 액세스する
</div>

W&amp;B 組織に対して JWT issuer が設定されると、ユーザーはその identity provider によって発行された JWT を使用して、該当する W&amp;B プロジェクトに 액세スできるようになります。JWT を使用する仕組みは次のとおりです。

* 組織で利用可能ないずれかの方法を使用して、identity provider にサインインする必要があります。プロバイダーによっては API や SDK を使用して自動的に 액세스できる場合もあれば、該当する UI からしか 액세スできない場合もあります。詳細については、W&amp;B 組織管理者または JWT issuer の所有者にお問い合わせください。
* identity provider にサインインして JWT を取得したら、安全な場所にあるファイルに保存し、その絶対パスを環境変数 `WANDB_IDENTITY_TOKEN_FILE` に設定します。
* W&amp;B SDK または CLI を使用して W&amp;B プロジェクトに 액세スします。SDK または CLI は自動的に JWT を検出し、JWT が正常に検証されると、それを W&amp;B アクセストークンと交換します。W&amp;B アクセストークンは、run、メトリクス、Artifacts などをログに記録するための関連 API に 액세スし、AI ワークフローを実行できるようにするために使用されます。アクセストークンはデフォルトで `~/.config/wandb/credentials.json` に保存されます。環境変数 `WANDB_CREDENTIALS_FILE` を指定することで、このパスを変更できます。

<Note>
  JWT は、API キーやパスワードなどの長期間有効な認証情報の欠点に対処するための、短期間のみ有効な認証情報です。identity provider で設定されている JWT の有効期限に応じて、JWT を継続的に更新し、環境変数 `WANDB_IDENTITY_TOKEN_FILE` で参照されるファイルに保存されていることを確認する必要があります。

  W&amp;B アクセストークンにもデフォルトの有効期限があり、期限が切れると SDK または CLI は JWT を使用して自動的にリフレッシュを試みます。その時点でユーザーの JWT も期限切れになっており更新されていない場合は、認証エラーが発生する可能性があります。可能であれば、JWT の取得と有効期限切れ後の更新の仕組みは、W&amp;B SDK または CLI を使用する AI ワークロードの一部として実装してください。
</Note>

<div id="jwt-validation">
  ### JWT 検証
</div>

JWT を W&amp;B アクセス トークンに交換し、その後プロジェクトに액세스するワークフローの一部として、JWT は以下の検証を受けます。

* JWT の署名は、W&amp;B 組織レベルの JWKS を使って検証されます。これは最初の防御線であり、ここで失敗する場合は、JWKS 自体か JWT の署名方法に問題があることを意味します。
* JWT の `iss` クレームは、組織レベルで設定された issuer URL と一致している必要があります。
* JWT の `sub` クレームは、W&amp;B 組織で設定されているユーザーのメールアドレスと一致している必要があります。
* JWT の `aud` クレームは、AI ワークフローの一部として액세스しているプロジェクトが属する W&amp;B 組織の名と一致している必要があります。[専用クラウド](/ja/platform/hosting/hosting-options/dedicated-cloud) または [セルフマネージド](/ja/platform/hosting/hosting-options/self-managed) インスタンスの場合、インスタンスレベルの環境変数 `SKIP_AUDIENCE_VALIDATION` を `true` に設定して audience クレームの検証をスキップするか、`wandb` を audience として使用できます。
* JWT の `exp` クレームは、トークンが有効か、期限切れで更新が必要かどうかを確認するためにチェックされます。

<div id="external-service-accounts">
  ## 外部サービスアカウント
</div>

W&amp;B では、長期間有効な APIキーを使用する組み込みサービスアカウントを以前からサポートしています。SDK と CLI のアイデンティティ フェデレーション機能により、認証に JWT を使用できる外部サービスアカウントも利用できます。ただし、それらは組織レベルで設定されたものと同じ issuer によって発行されている必要があります。チーム管理者は、組み込みサービスアカウントと同様に、チームのスコープ内で外部サービスアカウントを設定できます。

外部サービスアカウントを設定するには、次の手順に従います。

* チームの **Service Accounts** タブにアクセスする
* **New service account** をクリックする
* サービスアカウントの名を入力し、`Authentication Method` として `Federated Identity` を選択し、`Subject` を入力して、**Create** をクリックする

外部サービスアカウントの JWT 内の `sub` クレームは、チームレベルの **Service Accounts** タブでチーム管理者が subject として設定する値と同一である必要があります。このクレームは、[JWT 検証](#jwt-validation) の一部として検証されます。`aud` クレームの要件は、人間のユーザー JWT の場合と同様です。

[外部サービスアカウントの JWT を使用して W&amp;B に 액세スする](#using-the-jwt-to-access-wb)場合、通常は、初期 JWT の生成と継続的な更新を自動化する ワークフローのほうが簡単です。外部サービスアカウントを使用してログされた Runs を人間のユーザーに帰属させたい場合は、組み込みサービスアカウントの場合と同様に、AI ワークフロー用に環境変数 `WANDB_USERNAME` または `WANDB_USER_EMAIL` を設定できます。

<Note>
  W&amp;B は、柔軟性と単純さのバランスを取るために、データの機微性が異なる AI ワークロード全体で、組み込みサービスアカウントと外部サービスアカウントを組み合わせて使用することを推奨します。
</Note>
