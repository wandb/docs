---
title: SDK でアイデンティティ フェデレーションを使用する
description: "API キーを使わずに、JSON Web Tokens (JWTs) を使用して W&B SDK と CLI で認証するためにアイデンティティ フェデレーションを使用します。"
---

W&amp;B SDK を通じて組織の認証情報を使用してサインインするには、アイデンティティ フェデレーションを使用します。W&amp;B 組織管理者が組織に対して SSO を設定している場合は、すでに組織の認証情報を使って W&amp;B アプリ UI にサインインしています。その意味で、アイデンティティ フェデレーションは W&amp;B SDK 向けの SSO のようなものですが、JSON Web Tokens (JWTs) を直接使用します。アイデンティティ フェデレーションは、API キーの代替として使用できます。

[RFC 7523](https://datatracker.ietf.org/doc/html/rfc7523) は、SDK でのアイデンティティ フェデレーションの基盤となる仕様です。

<Note>
  アイデンティティ フェデレーションは、すべてのプラットフォームタイプ (SaaS Cloud、Dedicated Cloud、Self-Managed インスタンス) の `Enterprise` プランで `Preview` として利用できます。ご不明な点があれば、W&amp;B チームまでお問い合わせください。
</Note>

<Note>
  このドキュメントでは、`identity provider` と `JWT issuer` という用語は同じ意味で使用されます。この機能の文脈では、どちらもまったく同じものを指します。
</Note>

<div id="jwt-issuer-setup">
  ## JWT issuer のセットアップ
</div>

最初のステップとして、組織管理者はお使いの W&amp;B 組織と公開アクセス可能な JWT issuer の間でフェデレーションを設定する必要があります。

* 組織のダッシュボードの **Settings** タブに移動します
* **Authentication** オプションで **Set up JWT Issuer** をクリックします
* テキストボックスに JWT issuer の URL を入力し、**Create** をクリックします

W&amp;B は自動的に、パス `${ISSUER_URL}/.well-known/openid-configuration` で OIDC の discovery document を探し、その discovery document 内の適切な URL から JSON Web Key Set (JWKS) を取得します。JWKS は、JWT が対象の identity provider によって発行されたものであることをリアルタイムで検証するために使用されます。

<div id="using-the-jwt-to-access-wb">
  ## JWT を使用して W&amp;B にアクセスする
</div>

JWT issuer があなたの W&amp;B 組織向けにセットアップされると、ユーザーはその IdP が発行した JWT を使って、該当する W&amp;B プロジェクトにアクセスできるようになります。JWT を使用する仕組みは次のとおりです。

* 組織で利用可能ないずれかの方法を用いて IdP にサインインする必要があります。プロバイダーによっては API や SDK を用いて自動的にアクセスできる場合もあれば、該当する UI からしかアクセスできない場合もあります。詳細については、あなたの W&amp;B 組織の管理者、または JWT issuer の所有者に問い合わせてください。
* IdP にサインインした後で JWT を取得したら、それを安全な場所にあるファイルに保存し、その絶対パスを環境変数 `WANDB_IDENTITY_TOKEN_FILE` に設定します。
* W&amp;B SDK または CLI を使って W&amp;B プロジェクトにアクセスします。SDK または CLI は自動的に JWT を検出し、JWT が正常に検証された後に W&amp;B アクセストークンと交換します。W&amp;B アクセストークンは run のログ記録、メトリクス、 Artifacts など、AI ワークフローを有効にするための関連 API にアクセスするために使用されます。アクセストークンはデフォルトで `~/.config/wandb/credentials.json` パスに保存されます。環境変数 `WANDB_CREDENTIALS_FILE` を指定することで、このパスを変更できます。

<Note>
  JWT は、API キーやパスワードなどの長期間有効な認証情報の欠点に対処するために設計された、短期間のみ有効な認証情報です。IdP で設定されている JWT の有効期限に応じて、JWT を継続的に更新し、環境変数 `WANDB_IDENTITY_TOKEN_FILE` で指定されたファイルに常に保存されていることを確実にする必要があります。

  W&amp;B アクセストークンにもデフォルトの有効期限があり、その期限が切れると、SDK や CLI は JWT を用いて自動的にリフレッシュを試みます。その時点でユーザーの JWT も期限切れとなっていて更新されていない場合、認証エラーが発生する可能性があります。可能であれば、JWT の取得および有効期限切れ後の更新の仕組みは、W&amp;B SDK または CLI を使用する AI ワークロードの一部として実装することが望ましいです。
</Note>

<div id="jwt-validation">
  ### JWT 検証
</div>

JWT を W&amp;B アクセス トークンに交換し、その後 プロジェクト にアクセスするワークフローの一部として、JWT は次の検証を受けます。

* JWT の署名は、W&amp;B 組織レベルの JWKS を使って検証されます。これは最初の防御線であり、ここで失敗する場合は、JWKS 自体か JWT の署名方法に問題があることを意味します。
* JWT の `iss` クレームは、組織レベルで設定された issuer URL と一致している必要があります。
* JWT の `sub` クレームは、W&amp;B 組織で設定されているユーザーのメールアドレスと一致している必要があります。
* JWT の `aud` クレームは、AI ワークフローの一部としてアクセスしている プロジェクト が属する W&amp;B 組織の名前と一致している必要があります。[Dedicated Cloud](/ja/platform/hosting/hosting-options/dedicated-cloud) または [Self-Managed](/ja/platform/hosting/hosting-options/self-managed) インスタンスの場合、インスタンス レベルの環境変数 `SKIP_AUDIENCE_VALIDATION` を `true` に設定して audience クレームの検証をスキップするか、`wandb` を audience として使用できます。
* JWT の `exp` クレームは、トークンが有効か、期限切れで更新が必要かどうかを確認するためにチェックされます。

<div id="external-service-accounts">
  ## 外部サービスアカウント
</div>

W&amp;B では、長期間有効な API キーを持つ組み込みサービスアカウントを以前からサポートしています。SDK と CLI のアイデンティティフェデレーション機能により、組織レベルで設定された発行者と同一の発行者から発行される限り、認証に JWT を使用する外部サービスアカウントも利用できます。チーム管理者は、組み込みサービスアカウントと同様に、チームのスコープ内で外部サービスアカウントを設定できます。

外部サービスアカウントを設定するには:

* チームの **Service Accounts** タブに移動します
* **New service account** をクリックします
* サービスアカウントの名前を入力し、`Authentication Method` として `Federated Identity` を選択し、`Subject` を指定してから **Create** をクリックします

外部サービスアカウントの JWT に含まれる `sub` クレームは、チームレベルの **Service Accounts** タブでチーム管理者がサブジェクトとして設定した値と同じでなければなりません。このクレームは [JWT 検証](#jwt-validation) の一部として検証されます。`aud` クレームの要件は、人間ユーザー向け JWT の場合と同様です。

[外部サービスアカウントの JWT を使って W&amp;B にアクセスする](#using-the-jwt-to-access-wb) 場合、初回の JWT を生成し継続的にリフレッシュするワークフローを自動化する方が、一般的に容易です。外部サービスアカウントを使って記録された run を人間のユーザーに帰属させたい場合は、組み込みサービスアカウントの場合と同様に、AI ワークフローで環境変数 `WANDB_USERNAME` または `WANDB_USER_EMAIL` を設定できます。

<Note>
  W&amp;B では、柔軟性とシンプルさのバランスをとるために、データの機密度が異なるさまざまな AI ワークロードにおいて、組み込みサービスアカウントと外部サービスアカウントを併用することを推奨しています。
</Note>
