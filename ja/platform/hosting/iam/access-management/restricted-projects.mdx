---
description: 可視性スコープとプロジェクトレベルのロールを使ってプロジェクトへのアクセスを管理する
title: プロジェクトのアクセス制御を管理する
---

W&B プロジェクトのスコープを定義し、そのプロジェクトで W&B の run を閲覧・編集・送信できるユーザーを制限します。

W&B チーム内の任意のプロジェクトについて、複数のコントロールを組み合わせてアクセスレベルを設定できます。**Visibility scope** は、より上位の仕組みです。これを使って、どのユーザーグループがプロジェクト内の run を閲覧または送信できるかを制御します。可視性スコープが _Team_ または _Restricted_ のプロジェクトでは、**Project level roles** を使用して、そのプロジェクト内で各ユーザーが持つアクセスレベルを制御できます。

<Note>
プロジェクトのオーナー、チーム管理者、または組織管理者は、プロジェクトの可視性を設定および変更できます。
</Note>

<div id="visibility-scopes">
  ## 可視性スコープ
</div>

選択できるプロジェクトの可視性スコープは 4 種類あります。公開範囲が広いものから狭いものの順に並べると、次のとおりです。

| Scope | 説明 | 
| ----- | ----- |
| Open |プロジェクトについて知っている人は誰でも、そのプロジェクトを閲覧し、run やレポートを送信できます。|
| Public |プロジェクトについて知っている人は誰でも、そのプロジェクトを閲覧できます。run やレポートを送信できるのは、あなたのチームだけです。|
| Team | 親チームのメンバーのみが、そのプロジェクトを閲覧し、run やレポートを送信できます。チーム外のユーザーはプロジェクトへアクセスできません。 |
| Restricted| 親チームから招待されたメンバーだけが、そのプロジェクトを閲覧し、run やレポートを送信できます。|

<Note>
機密性の高いデータや秘密情報に関連するワークフローで共同作業したい場合は、プロジェクトのスコープを **Restricted** に設定してください。チーム内で Restricted プロジェクトを作成すると、そのチームから特定のメンバーを招待または追加して、関連する Experiments、アーティファクト、レポートなどで共同作業できます。 

他のプロジェクトスコープとは異なり、チームの全メンバーが自動的に Restricted プロジェクトへアクセスできるわけではありません。一方で、必要に応じてチーム管理者は Restricted プロジェクトに参加できます。
</Note>

<div id="set-visibility-scope-on-a-new-or-existing-project">
  ### 新規または既存のプロジェクトの可視性スコープを設定する
</div>

プロジェクトを作成するとき、または後から編集するときに、そのプロジェクトの可視性スコープを設定します。

<Note>
* プロジェクトのオーナーまたはチーム管理者のみが、その可視性スコープを設定または編集できます。
* チーム管理者がチームのプライバシー設定内で **Make all future team projects private (public sharing not allowed)** を有効にすると、そのチームでは **Open** と **Public** のプロジェクト可視性スコープが無効になります。この場合、チームで使用できるのは **Team** と **Restricted** のスコープのみです。
</Note>

<div id="set-visibility-scope-when-you-create-a-new-project">
  #### 新しいプロジェクトを作成するときに可視性スコープを設定する
</div>

1. SaaS Cloud、Dedicated Cloud、または Self-Managed インスタンス上の自分の W&B 組織にアクセスします。
2. 左側のサイドバー内の **My projects** セクションで **Create a new project** ボタンをクリックします。あるいは、チームの **Projects** タブに移動し、右上隅の **Create new project** ボタンをクリックします。
3. 親チームを選択し、プロジェクト名を入力した後、**Project Visibility** ドロップダウンから希望するスコープを選択します。
   <Frame>
       <img src="/images/hosting/restricted_project_add_new.gif" alt="制限付きプロジェクトの作成"  />
   </Frame>

   可視性として **Restricted** を選択した場合は、次の手順を実行します。 

4. **Invite team members** フィールドに、1 人以上の W&B チームメンバーの名前を入力します。プロジェクトで共同作業するために必要なメンバーだけを追加します。
   <Frame>
       <img src="/images/hosting/restricted_project_2.png" alt="制限付きプロジェクトの設定"  />
   </Frame>

   <Note>
   後から、そのプロジェクトの **Users** タブからメンバーを追加または削除できます。
   </Note>

<div id="edit-visibility-scope-of-an-existing-project">
  #### 既存プロジェクトの可視性スコープを編集する
</div>

1. 対象の W&B プロジェクトに移動します。
2. 左側のサイドバーで **Overview** タブを選択します。
3. 右上隅の **Edit Project Details** ボタンをクリックします。  
4. **Project Visibility** ドロップダウンから、希望するスコープを選択します。
   <Frame>
       <img src="/images/hosting/restricted_project_edit.gif" alt="制限付きプロジェクト設定の編集"  />
   </Frame>

   可視性として **Restricted** を選択した場合は、次の手順も実施します。

5. プロジェクトの **Users** タブに移動し、**Add user** ボタンをクリックして、特定のユーザーを制限付きプロジェクトに招待します。

<Warning>
* プロジェクトの可視性スコープを **Team** から **Restricted** に変更すると、必要なチームメンバーをそのプロジェクトに招待しない限り、チームの全メンバーはそのプロジェクトへのアクセス権を失います。
* プロジェクトの可視性スコープを **Restricted** から **Team** に変更すると、チームの全メンバーがそのプロジェクトへのアクセス権を得ます。
* 制限付きプロジェクトのユーザーリストからチームメンバーを削除すると、そのメンバーはそのプロジェクトへのアクセス権を失います。
</Warning>

<div id="other-key-things-to-note-for-restricted-scope">
  ### 制限付きスコープに関するその他の重要な注意点
</div>

* 制限付きプロジェクトでチームレベルのサービスアカウントを使用したい場合は、そのサービスアカウントを明示的にそのプロジェクトに招待または追加する必要があります。そうしない限り、チームレベルのサービスアカウントはデフォルトで制限付きプロジェクトにアクセスできません。
* 制限付きプロジェクトから run を移動することはできませんが、制限なしのプロジェクトから制限付きプロジェクトへ run を移動することはできます。
* チームのプライバシー設定 **Make all future team projects private (public sharing not allowed)** に関係なく、制限付きプロジェクトの可視性を **Team** スコープのみに変更できます。
* 制限付きプロジェクトのオーナーが親チームに所属していない場合、チーム管理者はプロジェクトの運用を円滑に行うために、オーナーを変更する必要があります。

<div id="project-level-roles">
  ## プロジェクトレベルのロール
</div>

チーム内の _Team_ または _Restricted_ スコープのプロジェクトでは、ユーザーごとに特定のロールを割り当てることができます。このロールは、そのユーザーのチームレベルのロールとは異なる場合があります。たとえば、あるユーザーがチームレベルで _Member_ ロールを持っている場合でも、そのチーム内の _Team_ または _Restricted_ スコープの各プロジェクトで、そのユーザーに _View-Only_ や _Admin_、その他利用可能なカスタムロールを割り当てることができます。

<Note>
プロジェクトレベルのロールは、SaaS Cloud、Dedicated Cloud、および Self-Managed インスタンスでプレビュー提供されています。
</Note>

<div id="assign-project-level-role-to-a-user">
  ### ユーザーにプロジェクトレベルのロールを割り当てる
</div>

1. 対象の W&B プロジェクトに移動します。
2. 左側のサイドバーで **Overview** タブを選択します。
3. プロジェクト内の **Users** タブに移動します。
4. **Project Role** フィールドで、対象ユーザーに現在割り当てられているロールをクリックします。利用可能な他のロールがドロップダウンで表示されます。
5. ドロップダウンから別のロールを選択します。変更は即座に保存されます。

<Note>
ユーザーのプロジェクトレベルのロールを、チームレベルのロールと異なるものに変更した場合、違いを示すためにプロジェクトレベルのロールには **\*** が付きます。
</Note>

<div id="other-key-things-to-note-for-project-level-roles">
  ### プロジェクトレベルのロールに関して注意すべきその他の重要な点
</div>

* デフォルトでは、_team_ または _restricted_ スコープのプロジェクト内のすべてのユーザーのプロジェクトレベルのロールは、それぞれのチームレベルのロールを**継承**します。
* チームレベルで _View-only_ ロールを持つユーザーのプロジェクトレベルのロールを変更することは**できません**。
* 特定のプロジェクト内で、ユーザーのプロジェクトレベルのロールがチームレベルのロールと**同じで**、後にチーム管理者がチームレベルのロールを変更した場合、該当するプロジェクトのロールも自動的に変更され、チームレベルのロールに追従します。
* 特定のプロジェクト内で、ユーザーのプロジェクトレベルのロールをチームレベルのロールと**異なるように**変更した場合、後にチーム管理者がチームレベルのロールを変更しても、該当するプロジェクトレベルのロールはそのまま維持されます。
* プロジェクトレベルのロールがチームレベルのロールと異なっている状態で、そのユーザーを _restricted_ プロジェクトから削除し、その後しばらくしてから再度そのユーザーをプロジェクトに追加した場合、デフォルトの挙動により、そのユーザーはチームレベルのロールを継承します。必要に応じて、再度プロジェクトレベルのロールをチームレベルのロールと異なるように変更する必要があります。
