<div id="introduction">
  ## はじめに
</div>

このガイドでは、エアギャップ環境、完全に分離された環境、または制限付きネットワークの、お客様管理 (customer-managed) 環境に W&amp;B Platform をデプロイするための手順を順を追って説明します。エアギャップ環境でのデプロイは、次のようなケースで一般的です。

* セキュアな政府機関施設
* 厳格なネットワーク分離を行う金融機関
* コンプライアンス要件を持つ医療機関
* 産業制御システム (ICS) 環境
* 機密ネットワークを持つ研究施設

必要な W&amp;B のイメージとチャートをホストするために、内部コンテナ レジストリと Helm リポジトリを使用します。Kubernetes クラスターに適切なアクセス権を持つシェル コンソールで、これらのコマンドを実行してください。

Kubernetes アプリケーションをデプロイするために使用している任意の CI/CD ツールに合わせて、これらのコマンドを調整できます。

インターネット接続のある標準的なオンプレミス Kubernetes デプロイについては、[Kubernetes Operator を使用して W&amp;B をデプロイする](/ja/platform/hosting/self-managed/operator) を参照してください。

<div id="air-gapped-specific-requirements">
  ### エアギャップ環境特有の要件
</div>

上記の標準要件に加えて、エアギャップ環境でのデプロイには次のものが必要です。

* **内部コンテナレジストリ**: 必要なすべての W&amp;B イメージを格納したプライベート コンテナレジストリ（Harbor、JFrog Artifactory、Nexus など）へのアクセス
* **内部 Helm リポジトリ**: W&amp;B Helm チャートを格納したプライベート Helm チャートリポジトリへのアクセス
* **イメージ転送手段**: インターネットに接続されたシステムからエアギャップ環境のレジストリへコンテナイメージを転送する方法
* **ライセンスファイル**: 有効な W&amp;B Enterprise ライセンス。ライセンスを取得するには（たとえばインターネットに接続されたマシンから）、Requirements ページの [License](/ja/platform/hosting/self-managed/requirements#license) セクションを参照するか、W&amp;B のアカウントチームにお問い合わせください。

ネットワークおよびロードバランサーの設定を含むインフラストラクチャ要件の全体像については、[reference architecture](/ja/platform/hosting/self-managed/ref-arch#infrastructure-requirements) を参照してください。

  ## エアギャップ環境を準備する

<div id="step-1-set-up-internal-container-registry">
  ### ステップ 1: 内部コンテナ レジストリをセットアップする
</div>

エアギャップ環境でのデプロイを成功させるには、必要なすべてのコンテナ イメージがエアギャップ環境のコンテナ レジストリで利用可能である必要があります。

<Note>
  W&amp;B Operator の要件を把握し、コンテナ レジストリ内のイメージを定期的に更新して維持する責任はあなたにあります。必要なコンテナ イメージとバージョンの最新一覧については Helm chart を参照するか、[W&amp;B Support](mailto:support@wandb.com) または担当の W&amp;B サポートエンジニアにお問い合わせください。
</Note>

<div id="core-wb-component-containers">
  #### W&amp;B のコアコンポーネントコンテナ
</div>

次のコア イメージが必要です。

* [`docker.io/wandb/controller`](https://hub.docker.com/r/wandb/controller) - W&amp;B Kubernetes Operator
* [`docker.io/wandb/local`](https://hub.docker.com/r/wandb/local) - W&amp;B アプリケーションサーバー
* [`docker.io/wandb/console`](https://hub.docker.com/r/wandb/console) - W&amp;B 管理コンソール
* [`docker.io/wandb/megabinary`](https://hub.docker.com/r/wandb/megabinary) - W&amp;B マイクロサービス群（API、Executor、Glue、Parquet）

<div id="dependency-containers">
  #### 依存関係コンテナー
</div>

次のサードパーティ製依存関係コンテナーイメージが必要です：

* [`docker.io/bitnamilegacy/redis`](https://hub.docker.com/r/bitnamilegacy/redis) - テストおよび開発時のローカル Redis デプロイメントに必要です。本番環境における Redis の要件については、「前提条件」の [Redis セクション](#redis) を参照してください。
* [`docker.io/otel/opentelemetry-collector-contrib`](https://hub.docker.com/r/otel/opentelemetry-collector-contrib) - メトリクスとログを収集するための OpenTelemetry エージェント
* [`quay.io/prometheus/prometheus`](https://quay.io/repository/prometheus/prometheus) - メトリクス収集用の Prometheus
* [`quay.io/prometheus-operator/prometheus-config-reloader`](https://quay.io/repository/prometheus-operator/prometheus-config-reloader) - Prometheus の依存コンポーネント

<div id="get-the-complete-image-list">
  #### 完全なイメージ一覧を取得する
</div>

Helm chart から必要なイメージとバージョンの完全な一覧を抽出するには、次の手順を実行します。

1. インターネットに接続されたシステム上で、[W&amp;B Helm charts リポジトリ](https://github.com/wandb/helm-charts) から W&amp;B Helm charts をダウンロードします。

   ```bash
   # helm-charts リポジトリをクローン
   git clone https://github.com/wandb/helm-charts.git
   cd helm-charts
   ```

2. `values.yaml` ファイルを参照して、すべてのコンテナイメージとそのバージョンを確認します。

   ```bash
   # operator chart からイメージ参照を抽出
   helm show values charts/operator | grep -E "repository:|tag:" | grep -v "^#"

   # platform chart からイメージ参照を抽出
   helm show values charts/operator-wandb | grep -E "repository:|tag:" | grep -v "^#"
   ```

   または、次のコマンドを使用して、バージョンタグなしでリポジトリ名のみを抽出することもできます。

   ```bash
   helm show values charts/operator-wandb \
     | awk -F': *' '/^[[:space:]]*repository:/{print $2}' \
     | grep -v "^#" \
     | sort -u
   ```

   リポジトリの一覧は次のようになります。

   ```text
   wandb/controller
   wandb/local
   wandb/console
   wandb/megabinary
   wandb/weave-python
   wandb/weave-trace
   otel/opentelemetry-collector-contrib
   prometheus/prometheus
   prometheus-operator/prometheus-config-reloader
   bitnamilegacy/redis
   ```

   各イメージの具体的なバージョンタグを取得するには、最初に示したコマンド（`grep -E "repository:|tag:"`）を使用します。このコマンドは、リポジトリ名とそれに対応するバージョンタグの両方を表示します。

<div id="transfer-images-to-air-gapped-registry">
  #### イメージをエアギャップ環境のレジストリに転送する
</div>

1. インターネットに接続されたシステム上で、必要なイメージをすべて pull して保存します。

   <Note>
     以下の例にあるバージョン番号は、上記のステップ 2 で Helm chart を確認した際の実際のバージョンに置き換えてください。ここで示しているバージョンはあくまで例であり、今後古くなる可能性があります。
   </Note>

   バージョンを一貫して管理するためにシェル変数を使用します:

   ```bash
   # バージョン変数を設定する（Helm chart のバージョンに合わせて更新する）
   CONTROLLER_VERSION="1.13.3"
   APP_VERSION="0.59.2"
   CONSOLE_VERSION="2.12.2"

   # イメージを pull する
   docker pull wandb/controller:${CONTROLLER_VERSION}
   docker pull wandb/local:${APP_VERSION}
   docker pull wandb/console:${CONSOLE_VERSION}
   docker pull wandb/megabinary:${APP_VERSION}
   # ... 他の必要なイメージもすべて、対応するバージョンで pull する

   # イメージを .tar ファイルに保存する
   docker save wandb/controller:${CONTROLLER_VERSION} -o wandb-controller-${CONTROLLER_VERSION}.tar
   docker save wandb/local:${APP_VERSION} -o wandb-local-${APP_VERSION}.tar
   docker save wandb/console:${CONSOLE_VERSION} -o wandb-console-${CONSOLE_VERSION}.tar
   docker save wandb/megabinary:${APP_VERSION} -o wandb-megabinary-${APP_VERSION}.tar
   # ... 他のすべてのイメージも保存する
   ```

2. 承認された方法（USB ドライブ、セキュアなファイル転送など）を使って `.tar` ファイルをエアギャップ環境に転送します。

3. エアギャップ環境内で、イメージをロードし、内部レジストリに push します:

   ```bash
   # 上で使用したものと同じバージョン変数を設定する
   CONTROLLER_VERSION="1.13.3"
   APP_VERSION="0.59.2"
   CONSOLE_VERSION="2.12.2"
   INTERNAL_REGISTRY="registry.yourdomain.com"

   # イメージをロードする
   docker load -i wandb-controller-${CONTROLLER_VERSION}.tar
   docker load -i wandb-local-${APP_VERSION}.tar
   docker load -i wandb-console-${CONSOLE_VERSION}.tar
   docker load -i wandb-megabinary-${APP_VERSION}.tar
   # ... 他のすべてのイメージもロードする

   # 内部レジストリ用にタグ付けする
   docker tag wandb/controller:${CONTROLLER_VERSION} ${INTERNAL_REGISTRY}/wandb/controller:${CONTROLLER_VERSION}
   docker tag wandb/local:${APP_VERSION} ${INTERNAL_REGISTRY}/wandb/local:${APP_VERSION}
   docker tag wandb/console:${CONSOLE_VERSION} ${INTERNAL_REGISTRY}/wandb/console:${CONSOLE_VERSION}
   docker tag wandb/megabinary:${APP_VERSION} ${INTERNAL_REGISTRY}/wandb/megabinary:${APP_VERSION}
   # ... 他のすべてのイメージにタグを付ける

   # 内部レジストリに push する
   docker push ${INTERNAL_REGISTRY}/wandb/controller:${CONTROLLER_VERSION}
   docker push ${INTERNAL_REGISTRY}/wandb/local:${APP_VERSION}
   docker push ${INTERNAL_REGISTRY}/wandb/console:${CONSOLE_VERSION}
   docker push ${INTERNAL_REGISTRY}/wandb/megabinary:${APP_VERSION}
   # ... 他のすべてのイメージも push する
   ```

<div id="step-2-set-up-internal-helm-chart-repository">
  ### ステップ 2: 内部 Helm チャートリポジトリをセットアップする
</div>

コンテナイメージと合わせて、次の Helm チャートが内部 Helm リポジトリで利用可能であることを確認してください。

* [W&amp;B Operator チャート](https://github.com/wandb/helm-charts/tree/main/charts/operator)
* [W&amp;B Platform チャート](https://github.com/wandb/helm-charts/tree/main/charts/operator-wandb)

1. インターネットに接続されたシステム上で、チャートをダウンロードします：

   ```bash
   # W&B Helm リポジトリを追加
   helm repo add wandb https://wandb.github.io/helm-charts
   helm repo update

   # チャートをダウンロード
   helm pull wandb/operator --version 1.13.3
   helm pull wandb/operator-wandb --version 0.18.0
   ```

2. `.tgz` チャートファイルをエアギャップ環境へ転送し、リポジトリの手順に従って内部 Helm リポジトリにアップロードします。

   `operator` チャートは W&amp;B Kubernetes Operator (Controller Manager) をデプロイします。`operator-wandb` チャートは、Custom Resource (CR) で指定された values を使用して W&amp;B Platform をデプロイします。

<div id="step-3-configure-helm-repository-access">
  ### ステップ 3: Helm リポジトリへのアクセスを設定する
</div>

1. エアギャップ環境で、Helm に内部リポジトリを使用するように設定する:

   ```bash
   helm repo add local-repo https://charts.yourdomain.com
   helm repo update
   ```

2. チャートが利用可能であることを確認する:

   ```bash
   helm search repo local-repo/operator
   helm search repo local-repo/operator-wandb
   ```

<div id="deploy-wb-in-air-gapped-environment">
  ## エアギャップ環境で W&amp;B をデプロイする
</div>

<div id="step-4-install-the-kubernetes-operator">
  ### ステップ 4: Kubernetes Operator をインストールする
</div>

W&amp;B Kubernetes Operator（controller manager）は W&amp;B プラットフォーム コンポーネントを管理します。エアギャップ環境にインストールするには、内部コンテナレジストリを使用するように設定します。

1. 次の内容で `values.yaml` ファイルを作成します:

   ```yaml
   image:
     repository: registry.yourdomain.com/wandb/controller
     tag: 1.13.3

   airgapped: true
   ```

   <Note>
     `repository` と `tag` は、ステップ 1 で内部レジストリに転送した実際のバージョンに置き換えてください。ここに示しているバージョン（`1.13.3`）は一例であり、今後古くなる可能性があります。
   </Note>

2. Operator と Custom Resource Definition (CRD) をインストールします:

   ```bash
   helm upgrade --install operator local-repo/operator \
     --namespace wandb \
     --create-namespace \
     --values values.yaml
   ```

3. Operator が起動していることを確認します:

   ```bash
   kubectl get pods -n wandb
   ```

   `Running` 状態の Operator Pod が表示されるはずです。

サポートされている設定値の詳細については、[Kubernetes Operator の GitHub リポジトリの values ファイル](https://github.com/wandb/helm-charts/blob/main/charts/operator/values.yaml) を参照してください。

MySQL の設定パラメーターについては、[リファレンスアーキテクチャの MySQL 設定パラメーター セクション](/ja/platform/hosting/self-managed/ref-arch#mysql-configuration-parameters)を参照してください。

<div id="step-6-configure-wb-custom-resource">
  ### ステップ 6: W&amp;B Custom Resource を設定する
</div>

W&amp;B Kubernetes Operator をインストールしたら、Custom Resource (CR) を設定して、社内の Helm リポジトリとコンテナレジストリを参照するようにします。

この設定により、Kubernetes Operator が W&amp;B プラットフォームの必要なコンポーネントをデプロイする際に、社内レジストリおよびリポジトリを使用するようになります。

<Note>
  以下の設定例に含まれるイメージのバージョンタグは、時間の経過とともに古くなります。ステップ 1 で社内レジストリに転送した実際のバージョンに合わせて、すべての `tag:` の値を置き換えてください。
</Note>

次の内容で `wandb.yaml` という名前のファイルを作成します:

```yaml
apiVersion: apps.wandb.com/v1
kind: WeightsAndBiases
metadata:
  labels:
    app.kubernetes.io/instance: wandb
    app.kubernetes.io/name: weightsandbiases
  name: wandb
  namespace: wandb

spec:
  chart:
    url: https://charts.yourdomain.com
    name: operator-wandb
    version: 0.18.0

  values:
    global:
      host: https://wandb.yourdomain.com
      license: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
      
      bucket:
        accessKey: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
        secretKey: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
        name: s3.yourdomain.com:9000
        path: wandb
        provider: s3
        region: us-east-1
      
      mysql:
        database: wandb
        host: mysql.yourdomain.com
        password: <your-mysql-password>
        port: 3306
        user: wandb
      
      redis:
        host: redis.yourdomain.com
        port: 6379
        password: <your-redis-password>
      
      api:
        enabled: true
      
      glue:
        enabled: true
      
      executor:
        enabled: true
      
      extraEnv:
        ENABLE_REGISTRY_UI: 'true'

    # すべてのコンポーネントイメージを内部レジストリを使用するよう設定する
    app:
      image:
        repository: registry.yourdomain.com/wandb/local
        tag: 0.59.2

    console:
      image:
        repository: registry.yourdomain.com/wandb/console
        tag: 2.12.2

    api:
      image:
        repository: registry.yourdomain.com/wandb/megabinary
        tag: 0.59.2

    executor:
      image:
        repository: registry.yourdomain.com/wandb/megabinary
        tag: 0.59.2

    glue:
      image:
        repository: registry.yourdomain.com/wandb/megabinary
        tag: 0.59.2

    parquet:
      image:
        repository: registry.yourdomain.com/wandb/megabinary
        tag: 0.59.2

    weave:
      image:
        repository: registry.yourdomain.com/wandb/weave-python
        tag: 0.59.2

    otel:
      image:
        repository: registry.yourdomain.com/otel/opentelemetry-collector-contrib
        tag: 0.97.0

    prometheus:
      server:
        image:
          repository: registry.yourdomain.com/prometheus/prometheus
          tag: v2.47.0
      configmapReload:
        prometheus:
          image:
            repository: registry.yourdomain.com/prometheus-operator/prometheus-config-reloader
            tag: v0.67.0

    ingress:
      annotations:
        nginx.ingress.kubernetes.io/proxy-body-size: 0
      class: nginx
```

<Note>
  すべてのプレースホルダー値（ホスト名、パスワード、タグなど）を、実際の設定値に置き換えてください。上記の例には、最も一般的に使用されるコンポーネントが含まれています。
</Note>

デプロイ要件によっては、次のような追加コンポーネント用のイメージ リポジトリを設定する必要がある場合もあります。

* `settingsMigrationJob`
* `weave-trace`
* `filestream`
* `flat-runs-table`

設定可能なコンポーネントの完全な一覧については、[W&amp;B Helm リポジトリ values ファイル](https://github.com/wandb/helm-charts/blob/main/charts/operator-wandb/values.yaml)を参照してください。

<div id="step-7-deploy-the-wb-platform">
  ### ステップ 7: W&amp;B プラットフォームをデプロイする
</div>

1. プラットフォームをデプロイするには W&amp;B Custom Resource を適用します:

   ```bash
   kubectl apply -f wandb.yaml
   ```

2. デプロイの進行状況を監視します:

   ```bash
   # 作成される Pod を監視
   kubectl get pods -n wandb --watch

   # デプロイのステータスを確認
   kubectl get weightsandbiases -n wandb

   # operator のログを表示
   kubectl logs -n wandb deployment/wandb-operator-controller-manager
   ```

   operator が必要なコンポーネントをすべて作成するため、デプロイには数分かかる場合があります。

<div id="openshift-configuration">
  ## OpenShift の設定
</div>

W&amp;B は、エアギャップ環境の OpenShift Kubernetes クラスターへのデプロイを完全にサポートしています。OpenShift のより厳格なセキュリティポリシーにより、OpenShift へのデプロイには追加のセキュリティコンテキスト設定が必要になります。

<div id="openshift-security-context-constraints">
  ### OpenShift セキュリティコンテキスト制約
</div>

OpenShift は Security Context Constraints (SCC) を使用して pod の権限を制御します。デフォルトでは、OpenShift は pod に `restricted` SCC を割り当てます。これにより root としての実行が禁止され、特定のユーザー ID で実行することが求められます。

<div id="option-1-use-restricted-scc-recommended">
  #### オプション 1: restricted SCC を使用する（推奨）
</div>

カスタムリソースで適切な security context を設定して、restricted SCC で動作するように W&amp;B コンポーネントを構成します。

```yaml
spec:
  values:
    # すべての Pod のセキュリティコンテキストを設定する
    app:
      podSecurityContext:
        fsGroup: 1000
        runAsUser: 1000
        runAsNonRoot: true
      securityContext:
        allowPrivilegeEscalation: false
        capabilities:
          drop:
            - ALL
        runAsNonRoot: true
        seccompProfile:
          type: RuntimeDefault

    console:
      podSecurityContext:
        fsGroup: 1000
        runAsUser: 1000
        runAsNonRoot: true
      securityContext:
        allowPrivilegeEscalation: false
        capabilities:
          drop:
            - ALL
        runAsNonRoot: true
        seccompProfile:
          type: RuntimeDefault

    # 他のコンポーネントにも繰り返す: api, executor, glue, parquet, weave
```

<div id="option-2-create-custom-scc-if-required">
  #### オプション 2: カスタム SCC を作成する（必要な場合）
</div>

デプロイに `restricted` SCC では利用できない機能が必要な場合は、カスタム SCC を作成します。

```yaml
apiVersion: security.openshift.io/v1
kind: SecurityContextConstraints
metadata:
  name: wandb-scc
allowHostDirVolumePlugin: false
allowHostIPC: false
allowHostNetwork: false
allowHostPID: false
allowHostPorts: false
allowPrivilegeEscalation: false
allowPrivilegedContainer: false
allowedCapabilities: []
defaultAddCapabilities: []
fsGroup:
  type: MustRunAs
  ranges:
    - min: 1000
      max: 65535
readOnlyRootFilesystem: false
requiredDropCapabilities:
  - ALL
runAsUser:
  type: MustRunAsRange
  uidRangeMin: 1000
  uidRangeMax: 65535
seLinuxContext:
  type: MustRunAs
supplementalGroups:
  type: RunAsAny
volumes:
  - configMap
  - downwardAPI
  - emptyDir
  - persistentVolumeClaim
  - projected
  - secret
```

1. SCC を適用します:

   ```bash
   oc apply -f wandb-scc.yaml
   ```

2. SCC を W&amp;B の ServiceAccount にバインドします:

   ```bash
   oc adm policy add-scc-to-user wandb-scc -z wandb-app -n wandb
   oc adm policy add-scc-to-user wandb-scc -z wandb-console -n wandb
   ```

<div id="openshift-routes">
  ### OpenShift ルート
</div>

OpenShift は標準的な Kubernetes Ingress の代わりに Route を使用します。W&amp;B を OpenShift Route を使用するように構成します:

```yaml
spec:
  values:
    ingress:
      enabled: false
    
    route:
      enabled: true
      host: wandb.apps.openshift.yourdomain.com
      tls:
        enabled: true
        termination: edge
        insecureEdgeTerminationPolicy: Redirect
```

<div id="openshift-image-pull-configuration">
  ### OpenShift イメージプル設定
</div>

OpenShift クラスターが認証付きの内部イメージレジストリを使用している場合は、次のように設定します。

1. イメージプル用シークレットを作成します。

   ```bash
   kubectl create secret docker-registry wandb-registry-secret \
     --docker-server=registry.yourdomain.com \
     --docker-username=<username> \
     --docker-password=<password> \
     --namespace=wandb
   ```

2. Custom Resource でそのシークレットを参照します。

   ```yaml
   spec:
     values:
       imagePullSecrets:
         - name: wandb-registry-secret
   ```

  name: wandb

    route:
      enabled: true

    imagePullSecrets:

    app:

            - ALL

<div id="additional-air-gapped-verification">
  ### 追加のエアギャップ検証
</div>

エアギャップ環境でのデプロイの場合、次の点も確認してください:

1. **イメージプル**: すべての Pod が内部レジストリからイメージを正常にプルできていることを確認します:

   ```bash
   kubectl get pods -n wandb -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.phase}{"\t"}{.status.containerStatuses[*].image}{"\n"}{end}'
   ```

   すべてのイメージが内部レジストリを参照しており、すべての Pod が `Running` 状態である必要があります。

2. **外部接続**: エアギャップモードでは W&amp;B が外部への接続を試みていないことを確認します（エアギャップモードでは外部接続は行われないはずです）:

   ```bash
   kubectl logs -n wandb deployment/wandb-app --tail=100 | grep -i "connection"
   ```

3. **ライセンス検証**: W&amp;B コンソールにアクセスし、ライセンスが有効になっていることを確認します。

<div id="troubleshooting">
  ## トラブルシューティング
</div>

<div id="image-pull-errors">
  ### イメージ pull エラー
</div>

Pod がイメージを pull できない場合は、次を確認してください。

1. 内部レジストリにイメージが存在することを確認する
2. image pull シークレットが正しく構成されているか確認する
3. Kubernetes ノードからレジストリへのネットワーク接続を確認する
4. レジストリの認証情報を確認する

   ```bash
   # イメージの pull を手動でテストする
   kubectl run test-pull --image=registry.yourdomain.com/wandb/local:0.59.2 --namespace=wandb
   kubectl logs test-pull -n wandb
   kubectl delete pod test-pull -n wandb
   ```

<div id="openshift-scc-errors">
  ### OpenShift SCC エラー
</div>

OpenShift で pod が権限エラーによって失敗する場合:

```bash
# 使用中の SCC を確認する
oc get pod <pod-name> -n wandb -o yaml | grep scc

# サービスアカウントの権限を確認する
oc describe scc wandb-scc
oc get rolebinding -n wandb
```

<div id="helm-chart-not-found">
  ### Helm チャートが見つからない
</div>

operator が platform チャートを見つけられない場合は、次を確認してください:

1. Custom Resource 内のチャートリポジトリ URL を確認する
2. operator の Pod から内部 Helm リポジトリへ到達できることを確認する
3. 次のコマンドで、リポジトリ内にチャートが存在することを確認する:

   ```bash
   helm search repo local-repo/operator-wandb
   ```

<div id="frequently-asked-questions">
  ## よくある質問
</div>

<div id="can-i-use-a-different-ingress-class">
  ### 別の ingress クラスを使用できますか？
</div>

はい、Custom Resource 内の ingress 設定を変更して ingress クラスを設定してください。

```yaml
spec:
  values:
    ingress:
      class: your-ingress-class
```

<div id="how-do-i-handle-certificate-bundles-with-multiple-certificates">
  ### 複数の証明書を含む証明書バンドルはどのように扱えばよいですか？
</div>

証明書を分割し、`customCACerts` セクションで複数のエントリとして指定します。

```yaml
spec:
  values:
    customCACerts:
      cert1.crt: |
        -----BEGIN CERTIFICATE-----
        ...
        -----END CERTIFICATE-----
      cert2.crt: |
        -----BEGIN CERTIFICATE-----
        ...
        -----END CERTIFICATE-----
```

<div id="how-do-i-prevent-automatic-updates">
  ### 自動アップデートを防ぐにはどうすればよいですか？
</div>

W&amp;B が自動的に更新されないように、オペレーターを次のように設定します:

1. オペレーターのインストール時に `airgapped: true` を設定する（これにより自動更新チェックが無効になります）
2. Custom Resource 内の `spec.chart.version` を手動で更新してバージョン更新を制御する
3. 必要に応じて、W&amp;B System Console から自動アップデートを無効化する

詳しくは [Disable automatic app version updates](/ja/platform/hosting/self-managed/disable-automatic-app-version-updates) を参照してください。

<Note>
  W&amp;B は、Self-Managed インスタンスを利用しているお客様に対し、サポートを維持し、最新の機能やパフォーマンス向上および修正を受け取るために、少なくとも四半期に一度はデプロイメントを最新リリースに更新することを強く推奨します。W&amp;B は、メジャーリリースについては初回リリース日から 12 か月間サポートします。[Release policies and processes](/ja/release-notes/release-policies) を参照してください。
</Note>

<div id="does-the-deployment-work-with-no-connection-to-public-repositories">
  ### デプロイはパブリック リポジトリへの接続が一切なくても動作しますか？
</div>

はい。`airgapped: true` が operator の設定で指定されている場合、Kubernetes operator は内部リソースのみを使用し、パブリック リポジトリへの接続を試みません。

<div id="how-do-i-update-wb-in-an-air-gapped-environment">
  ### エアギャップ環境で W&amp;B を更新するにはどうすればよいですか？
</div>

W&amp;B を更新するには、次の手順を実行します。

1. インターネットに接続されたシステムで新しいコンテナーイメージをプルする
2. イメージをエアギャップレジストリに転送する
3. 新しい Helm チャートを社内リポジトリにアップロードする
4. Custom Resource 内の `spec.chart.version` とイメージタグを更新する
5. 更新した Custom Resource を適用する

   これにより、オペレーターが W&amp;B コンポーネントのローリングアップデートを実行します。

<div id="next-steps">
  ## 次のステップ
</div>

デプロイが正常に完了したら、以下を実施してください。

1. **ユーザー認証を構成する**: [SSO](/ja/platform/hosting/iam/sso) などの認証方式を設定する
2. **モニタリングを設定する**: W&amp;B インスタンスおよびインフラストラクチャのモニタリングを構成する
3. **アップデートを計画する**: [Server upgrade process](/ja/platform/hosting/server-upgrade-process) を確認し、アップデートの実施頻度を決める
4. **バックアップを構成する**: MySQL データベースのバックアップ手順を定義する
5. **プロセスを文書化する**: エアギャップ環境における独自のアップデート手順向けに runbook を作成する
