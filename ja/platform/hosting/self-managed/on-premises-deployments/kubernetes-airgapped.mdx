<div id="introduction">
  ## はじめに
</div>

このガイドでは、에어갭環境、完全に分離された環境、または制限付きネットワークのセルフマネージド環境に W&amp;B プラットフォームをデプロイするための手順を順を追って説明します。에어갭環境でのデプロイは、次のようなケースで一般的です。

* セキュリティの厳しい政府施設
* 厳格なネットワーク分離を行う金融機関
* コンプライアンス要件を持つ医療機関
* 産業制御システム (ICS) 環境
* 機密ネットワークを持つ研究施設

必要な W&amp;B のイメージとチャートをホストするために、内部コンテナレジストリと Helm リポジトリを使用します。Kubernetes クラスターに適切な액세스権を持つシェルコンソールで、これらのコマンドを実行してください。

Kubernetes アプリケーションをデプロイするために使用している任意の CI/CD ツールに合わせて、これらのコマンドを調整できます。

インターネット接続のある標準的なオンプレミス Kubernetes デプロイについては、[Operator を使用して W&amp;B をデプロイする](/ja/platform/hosting/self-managed/operator) を参照してください。

<div id="air-gapped-specific-requirements">
  ### 에어갭環境固有の要件
</div>

上記の標準要件に加えて、에어갭環境でのデプロイには次が必要です。

* **内部コンテナー Registry**: 必要なすべての W&amp;B コンテナーイメージを含む非公開のコンテナー Registry へのアクセス (Harbor、JFrog Artifactory、Nexus など)
* **内部 Helm リポジトリ**: W&amp;B Helm チャートを含む非公開の Helm リポジトリ へのアクセス
* **イメージ転送機能**: インターネット接続されたシステムから에어갭環境の Registry にコンテナーイメージを転送するための method
* **License ファイル**: 有効な W&amp;B Enterprise license。ライセンスを取得するには (たとえば、インターネット接続されたマシンから取得する場合) 、要件ページの [License](/ja/platform/hosting/self-managed/requirements#license) セクションを参照するか、W&amp;B account team にお問い合わせください。

ネットワークやロードバランサーの設定を含む完全なインフラストラクチャー要件については、[リファレンスアーキテクチャ](/ja/platform/hosting/self-managed/ref-arch#infrastructure-requirements) を参照してください。

<div id="step-1-set-up-internal-container-registry">
  ### ステップ 1: 内部コンテナレジストリを設定する
</div>

에어갭環境でのデプロイを成功させるには、必要なコンテナーイメージをすべて에어갭環境内のコンテナレジストリで利用できるようにしておく必要があります。

<Note>
  W&amp;B Operator の要件を把握し、コンテナレジストリ内のコンテナーイメージを定期的に更新して維持する責任はお客様にあります。必須のコンテナーイメージとバージョンの最新一覧については、Helm チャートを参照するか、[W&amp;B Support](mailto:support@wandb.com) または担当の W&amp;B サポートエンジニアにお問い合わせください。
</Note>

<div id="core-wb-component-containers">
  #### W&amp;B のコアコンポーネントコンテナー
</div>

次のコア イメージが必須です。

* [`docker.io/wandb/controller`](https://hub.docker.com/r/wandb/controller) - W&amp;B Kubernetes Operator
* [`docker.io/wandb/local`](https://hub.docker.com/r/wandb/local) - W&amp;B アプリケーションサーバー
* [`docker.io/wandb/console`](https://hub.docker.com/r/wandb/console) - W&amp;B 管理コンソール
* [`docker.io/wandb/megabinary`](https://hub.docker.com/r/wandb/megabinary) - W&amp;B マイクロサービス群 (API、Executor、Glue、Parquet)

<div id="dependency-containers">
  #### 依存関係コンテナー
</div>

次のサードパーティ製依存関係コンテナーイメージが必須です：

* [`docker.io/bitnamilegacy/redis`](https://hub.docker.com/r/bitnamilegacy/redis) - テストおよび開発時のローカル Redis デプロイに必須です。本番 Redis の要件については、「前提条件」の [Redis セクション](#redis) を参照してください。
* [`docker.io/otel/opentelemetry-collector-contrib`](https://hub.docker.com/r/otel/opentelemetry-collector-contrib) - メトリクスとログを収集するための OpenTelemetry エージェント
* [`quay.io/prometheus/prometheus`](https://quay.io/repository/prometheus/prometheus) - メトリクス収集用の Prometheus
* [`quay.io/prometheus-operator/prometheus-config-reloader`](https://quay.io/repository/prometheus-operator/prometheus-config-reloader) - Prometheus の依存コンポーネント

<div id="get-the-complete-image-list">
  #### 完全なコンテナーイメージ一覧を取得する
</div>

Helm チャートから必要なコンテナーイメージとバージョンの完全な一覧を抽出するには、次の手順を実行してください。

1. インターネットに接続されたシステムで、[W&amp;B Helm charts repository](https://github.com/wandb/helm-charts) から W&amp;B Helm charts をダウンロードします。

   ```bash
   # helm-charts リポジトリをクローン
   git clone https://github.com/wandb/helm-charts.git
   cd helm-charts
   ```

2. `values.yaml` ファイルを確認して、すべてのコンテナーイメージとそのバージョンを特定します。

   ```bash
   # operator チャートからイメージ参照を抽出
   helm show values charts/operator | grep -E "repository:|tag:" | grep -v "^#"

   # platform チャートからイメージ参照を抽出
   helm show values charts/operator-wandb | grep -E "repository:|tag:" | grep -v "^#"
   ```

   または、次のコマンドを使用して、バージョンタグを含まないリポジトリ名だけを抽出することもできます。

   ```bash
   helm show values charts/operator-wandb \
     | awk -F': *' '/^[[:space:]]*repository:/{print $2}' \
     | grep -v "^#" \
     | sort -u
   ```

   リポジトリの一覧は、次のようになります。

   ```text
   wandb/controller
   wandb/local
   wandb/console
   wandb/megabinary
   wandb/weave-python
   wandb/weave-trace
   otel/opentelemetry-collector-contrib
   prometheus/prometheus
   prometheus-operator/prometheus-config-reloader
   bitnamilegacy/redis
   ```

   各イメージの具体的なバージョンタグを取得するには、最初のコマンド (`grep -E "repository:|tag:"`) を使用します。このコマンドでは、リポジトリ名と対応するバージョンタグの両方が表示されます。

<div id="transfer-images-to-air-gapped-registry">
  #### イメージを에어갭 Registry に転送する
</div>

1. インターネット接続されたシステムで、必要なイメージをすべて pull して保存します。

   <Note>
     以下の例のバージョン番号は、上記 step 2 の Helm チャート の確認結果にある実際のバージョンに置き換えてください。ここに示しているバージョンは例であり、時間の経過とともに古くなります。
   </Note>

   バージョンを一貫して管理するため、シェル変数を使用します。

   ```bash
   # バージョン変数を設定する（Helm チャート のバージョンに基づいて更新）
   CONTROLLER_VERSION="1.13.3"
   APP_VERSION="0.59.2"
   CONSOLE_VERSION="2.12.2"

   # イメージを pull
   docker pull wandb/controller:${CONTROLLER_VERSION}
   docker pull wandb/local:${APP_VERSION}
   docker pull wandb/console:${CONSOLE_VERSION}
   docker pull wandb/megabinary:${APP_VERSION}
   # ... 他の必須イメージもそれぞれのバージョンで pull

   # イメージを .tar ファイルとして保存
   docker save wandb/controller:${CONTROLLER_VERSION} -o wandb-controller-${CONTROLLER_VERSION}.tar
   docker save wandb/local:${APP_VERSION} -o wandb-local-${APP_VERSION}.tar
   docker save wandb/console:${CONSOLE_VERSION} -o wandb-console-${CONSOLE_VERSION}.tar
   docker save wandb/megabinary:${APP_VERSION} -o wandb-megabinary-${APP_VERSION}.tar
   # ... 他のすべてのイメージも保存
   ```

2. 承認済みの方法 (USB ドライブ、セキュアなファイル転送など) を使用して、`.tar` ファイルを에어갭環境に転送します。

3. 에어갭環境でイメージをロードし、内部 Registry に push します。

   ```bash
   # 上で使用したものと同じバージョン変数を設定
   CONTROLLER_VERSION="1.13.3"
   APP_VERSION="0.59.2"
   CONSOLE_VERSION="2.12.2"
   INTERNAL_REGISTRY="registry.yourdomain.com"

   # イメージをロード
   docker load -i wandb-controller-${CONTROLLER_VERSION}.tar
   docker load -i wandb-local-${APP_VERSION}.tar
   docker load -i wandb-console-${CONSOLE_VERSION}.tar
   docker load -i wandb-megabinary-${APP_VERSION}.tar
   # ... 他のすべてのイメージもロード

   # 内部 Registry 用にタグ付け
   docker tag wandb/controller:${CONTROLLER_VERSION} ${INTERNAL_REGISTRY}/wandb/controller:${CONTROLLER_VERSION}
   docker tag wandb/local:${APP_VERSION} ${INTERNAL_REGISTRY}/wandb/local:${APP_VERSION}
   docker tag wandb/console:${CONSOLE_VERSION} ${INTERNAL_REGISTRY}/wandb/console:${CONSOLE_VERSION}
   docker tag wandb/megabinary:${APP_VERSION} ${INTERNAL_REGISTRY}/wandb/megabinary:${APP_VERSION}
   # ... 他のすべてのイメージにもタグ付け

   # 内部 Registry に push
   docker push ${INTERNAL_REGISTRY}/wandb/controller:${CONTROLLER_VERSION}
   docker push ${INTERNAL_REGISTRY}/wandb/local:${APP_VERSION}
   docker push ${INTERNAL_REGISTRY}/wandb/console:${CONSOLE_VERSION}
   docker push ${INTERNAL_REGISTRY}/wandb/megabinary:${APP_VERSION}
   # ... 他のすべてのイメージも push
   ```

<div id="step-2-set-up-internal-helm-chart-repository">
  ### ステップ 2: 内部 Helm チャートリポジトリを設定する
</div>

コンテナーイメージに加えて、次の Helm チャートが内部 Helm リポジトリで利用可能であることを確認してください。

* [W&amp;B Operator チャート](https://github.com/wandb/helm-charts/tree/main/charts/operator)
* [W&amp;B Platform チャート](https://github.com/wandb/helm-charts/tree/main/charts/operator-wandb)

1. インターネットに接続されたシステムで、チャートをダウンロードします。

   ```bash
   # Add W&B Helm repository
   helm repo add wandb https://wandb.github.io/helm-charts
   helm repo update

   # Download the charts
   helm pull wandb/operator --version 1.13.3
   helm pull wandb/operator-wandb --version 0.18.0
   ```

2. `.tgz` のチャートファイルを에어갭環境に転送し、リポジトリの手順に従って内部 Helm リポジトリにアップロードします。

   `operator` チャートは W&amp;B Kubernetes Operator (Controller Manager) をデプロイします。`operator-wandb` チャートは、Custom Resource (CR) で設定した値を使用して W&amp;B Platform をデプロイします。

<div id="step-3-configure-helm-repository-access">
  ### ステップ 3: Helm リポジトリへの액세스を設定する
</div>

1. 에어갭環境で、Helm に内部リポジトリを使用するように設定する:

   ```bash
   helm repo add local-repo https://charts.yourdomain.com
   helm repo update
   ```

2. チャートが사용 가능하다ことを確認する:

   ```bash
   helm search repo local-repo/operator
   helm search repo local-repo/operator-wandb
   ```

<div id="deploy-wb-in-air-gapped-environment">
  ## 에어갭環境で W&amp;B をデプロイする
</div>

<div id="step-4-install-the-kubernetes-operator">
  ### ステップ 4: Kubernetes Operator をインストールする
</div>

W&amp;B Kubernetes Operator (controller manager) は、W&amp;B プラットフォームの컴포넌트を管理します。에어갭 環境にインストールするには、内部コンテナRegistryを使用するように設定します。

1. 次の内容で `values.yaml` ファイルを作成します。

   ```yaml
   image:
     repository: registry.yourdomain.com/wandb/controller
     tag: 1.13.3

   airgapped: true
   ```

   <Note>
     `repository` と `tag` は、ステップ 1 で内部レジストリに転送した実際のバージョンに置き換えてください。ここに示しているバージョン (`1.13.3`) はあくまで例であり、いずれ古くなります。
   </Note>

2. Operator と Custom Resource Definition (CRD) をインストールします。

   ```bash
   helm upgrade --install operator local-repo/operator \
     --namespace wandb \
     --create-namespace \
     --values values.yaml
   ```

3. Operator が実行中であることを確認します。

   ```bash
   kubectl get pods -n wandb
   ```

   `Running` 状態の Operator Pod が表示されるはずです。

サポートされる値の詳細については、[Kubernetes operator GitHub Registry の values ファイル](https://github.com/wandb/helm-charts/blob/main/charts/operator/values.yaml)を参照してください。

<div id="step-5-set-up-mysql-database">

<SelfManagedMysqlDatabaseCreation />

MySQL の設定パラメーターについては、[リファレンスアーキテクチャの MySQL 設定パラメーター セクション](/ja/platform/hosting/self-managed/ref-arch#mysql-configuration-parameters)を参照してください。

<div id="step-6-configure-wb-custom-resource">
  ### ステップ 6: W&amp;B Custom Resource を設定する
</div>

W&amp;B Kubernetes Operator をインストールしたら、Custom Resource (CR) を設定して、社内の Helm リポジトリとコンテナー Registryを参照するようにします。

この設定により、Kubernetes Operator は W&amp;B プラットフォームの必須컴포넌트をデプロイするときに、社内のRegistryとリポジトリを使用します。

<Note>
  以下の設定例には、いずれ古くなるイメージのバージョン タグが含まれています。ステップ 1 で社内レジストリに転送した実際のバージョンに、すべての `tag:` の値を置き換えてください。
</Note>

次の内容で `wandb.yaml` という名前のファイルを作成してください。

```yaml
apiVersion: apps.wandb.com/v1
kind: WeightsAndBiases
metadata:
  labels:
    app.kubernetes.io/instance: wandb
    app.kubernetes.io/name: weightsandbiases
  name: wandb
  namespace: wandb

spec:
  chart:
    url: https://charts.yourdomain.com
    name: operator-wandb
    version: 0.18.0

  values:
    global:
      host: https://wandb.yourdomain.com
      license: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
      
      bucket:
        accessKey: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
        secretKey: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
        name: s3.yourdomain.com:9000
        path: wandb
        provider: s3
        region: us-east-1
      
      mysql:
        database: wandb
        host: mysql.yourdomain.com
        password: <your-mysql-password>
        port: 3306
        user: wandb
      
      redis:
        host: redis.yourdomain.com
        port: 6379
        password: <your-redis-password>
      
      api:
        enabled: true
      
      glue:
        enabled: true
      
      executor:
        enabled: true
      
      extraEnv:
        ENABLE_REGISTRY_UI: 'true'

    # すべての컴포넌트イメージを内部レジストリを使用するよう設定する
    app:
      image:
        repository: registry.yourdomain.com/wandb/local
        tag: 0.59.2

    console:
      image:
        repository: registry.yourdomain.com/wandb/console
        tag: 2.12.2

    api:
      image:
        repository: registry.yourdomain.com/wandb/megabinary
        tag: 0.59.2

    executor:
      image:
        repository: registry.yourdomain.com/wandb/megabinary
        tag: 0.59.2

    glue:
      image:
        repository: registry.yourdomain.com/wandb/megabinary
        tag: 0.59.2

    parquet:
      image:
        repository: registry.yourdomain.com/wandb/megabinary
        tag: 0.59.2

    weave:
      image:
        repository: registry.yourdomain.com/wandb/weave-python
        tag: 0.59.2

    otel:
      image:
        repository: registry.yourdomain.com/otel/opentelemetry-collector-contrib
        tag: 0.97.0

    prometheus:
      server:
        image:
          repository: registry.yourdomain.com/prometheus/prometheus
          tag: v2.47.0
      configmapReload:
        prometheus:
          image:
            repository: registry.yourdomain.com/prometheus-operator/prometheus-config-reloader
            tag: v0.67.0

    ingress:
      annotations:
        nginx.ingress.kubernetes.io/proxy-body-size: "0"
      class: nginx
```

<Note>
  すべてのプレースホルダー値 (ホスト名、パスワード、タグなど) を、実際の設定値に置き換えてください。上記の例には、最も一般的に使用される컴포넌트が含まれています。
</Note>

デプロイ要件によっては、次のような追加컴포넌트用のイメージ リポジトリを設定する必要がある場合もあります。

* `settingsMigrationJob`
* `weave-trace`
* `filestream`
* `flat-runs-table`

設定可能な컴포넌트の完全な一覧については、[W&amp;B Helm リポジトリ values ファイル](https://github.com/wandb/helm-charts/blob/main/charts/operator-wandb/values.yaml)を参照してください。

<div id="step-7-deploy-the-wb-platform">
  ### ステップ 7: W&amp;B プラットフォームをデプロイする
</div>

1. プラットフォームをデプロイするには、W&amp;B Custom Resource を適用します:

   ```bash
   kubectl apply -f wandb.yaml
   ```

2. デプロイの進行状況を監視します:

   ```bash
   # ポッドが作成されるのを監視
   kubectl get pods -n wandb --watch

   # デプロイのステータスを確認
   kubectl get weightsandbiases -n wandb

   # オペレーターのログを表示
   kubectl logs -n wandb deployment/wandb-operator-controller-manager
   ```

   オペレーターが必要な컴포넌트をすべて作成するため、デプロイには数分かかる場合があります。

<div id="openshift-configuration">
  ## OpenShift の設定
</div>

W&amp;B は、에어갭環境の OpenShift Kubernetes クラスターへのデプロイを完全にサポートしています。OpenShift のより厳格なセキュリティポリシーにより、OpenShift へのデプロイには追加のセキュリティコンテキスト設定が必要になります。

<div id="openshift-security-context-constraints">
  ### OpenShift セキュリティコンテキスト制約
</div>

OpenShift は Security Context Constraints (SCC) を使用して pod の権限を制御します。デフォルトでは、OpenShift は pod に `restricted` SCC を割り当てます。これにより root としての実行が禁止され、特定のユーザー ID で実行することが求められます。

<div id="option-1-use-restricted-scc-recommended">
  #### オプション 1: restricted SCC を使用する (推奨)
</div>

Custom Resource で適切なセキュリティコンテキストを設定して、W&amp;B コンポーネントが restricted SCC で実行されるようにします。

```yaml
spec:
  values:
    # すべての Pod のセキュリティコンテキストを設定する
    app:
      podSecurityContext:
        fsGroup: 1000
        runAsUser: 1000
        runAsNonRoot: true
      securityContext:
        allowPrivilegeEscalation: false
        capabilities:
          drop:
            - ALL
        runAsNonRoot: true
        seccompProfile:
          type: RuntimeDefault

    console:
      podSecurityContext:
        fsGroup: 1000
        runAsUser: 1000
        runAsNonRoot: true
      securityContext:
        allowPrivilegeEscalation: false
        capabilities:
          drop:
            - ALL
        runAsNonRoot: true
        seccompProfile:
          type: RuntimeDefault

    # 他のコンポーネントにも繰り返す: api, executor, glue, parquet, weave
```

<div id="option-2-create-custom-scc-if-required">
  #### オプション 2: カスタム SCC を作成する (必須の場合)
</div>

デプロイで `restricted` SCC では利用できない機能が必要な場合は、カスタム SCC を作成します。

```yaml
apiVersion: security.openshift.io/v1
kind: SecurityContextConstraints
metadata:
  name: wandb-scc
allowHostDirVolumePlugin: false
allowHostIPC: false
allowHostNetwork: false
allowHostPID: false
allowHostPorts: false
allowPrivilegeEscalation: false
allowPrivilegedContainer: false
allowedCapabilities: []
defaultAddCapabilities: []
fsGroup:
  type: MustRunAs
  ranges:
    - min: 1000
      max: 65535
readOnlyRootFilesystem: false
requiredDropCapabilities:
  - ALL
runAsUser:
  type: MustRunAsRange
  uidRangeMin: 1000
  uidRangeMax: 65535
seLinuxContext:
  type: MustRunAs
supplementalGroups:
  type: RunAsAny
volumes:
  - configMap
  - downwardAPI
  - emptyDir
  - persistentVolumeClaim
  - projected
  - secret
```

1. SCC を適用します:

   ```bash
   oc apply -f wandb-scc.yaml
   ```

2. SCC を W&amp;B のサービスアカウントにバインドします:

   ```bash
   oc adm policy add-scc-to-user wandb-scc -z wandb-app -n wandb
   oc adm policy add-scc-to-user wandb-scc -z wandb-console -n wandb
   ```

<div id="openshift-routes">
  ### OpenShift ルート
</div>

OpenShift は標準的な Kubernetes Ingress の代わりに Route を使用します。W&amp;B で OpenShift Route を使用するように設定します:

```yaml
spec:
  values:
    ingress:
      enabled: false
    
    route:
      enabled: true
      host: wandb.apps.openshift.yourdomain.com
      tls:
        enabled: true
        termination: edge
        insecureEdgeTerminationPolicy: Redirect
```

<div id="openshift-image-pull-configuration">
  ### OpenShift イメージプル設定
</div>

OpenShift クラスターが認証付きの内部イメージRegistryを使用している場合は、次のように設定します。

1. イメージプル secret を作成します。

   ```bash
   kubectl create secret docker-registry wandb-registry-secret \
     --docker-server=registry.yourdomain.com \
     --docker-username=<username> \
     --docker-password=<password> \
     --namespace=wandb
   ```

2. Custom Resource でその secret を参照します。

   ```yaml
   spec:
     values:
       imagePullSecrets:
         - name: wandb-registry-secret
   ```

```yaml

kind: WeightsAndBiases
metadata:
  name: wandb
  namespace: wandb

spec:
  chart:
    url: https://charts.yourdomain.com
    name: operator-wandb
    version: 0.18.0

      bucket:
        accessKey: <your-access-key>

        host: mysql.yourdomain.com

        port: 3306
        user: wandb
      
      redis:
        host: redis.yourdomain.com
        port: 6379
        password: <your-redis-password>

    # OpenShift 固有: Ingress の代わりに Routes を使用
    ingress:
      enabled: false
    
    route:
      enabled: true
      host: wandb.apps.openshift.yourdomain.com
      tls:
        enabled: true
        termination: edge

    # 内部Registry用のイメージプル secret
    imagePullSecrets:
      - name: wandb-registry-secret

    # OpenShift の制限付き SCC 用のセキュリティコンテキスト
    app:
      image:
        repository: registry.yourdomain.com/wandb/local
        tag: 0.59.2
      podSecurityContext:
        fsGroup: 1000
        runAsUser: 1000

            - ALL
```

<div id="additional-air-gapped-verification">
  ### 追加の에어갭検証
</div>

에어갭環境でのデプロイの場合、次の点も確認してください:

1. **イメージプル**: すべての Pod が内部Registryからイメージを正常にプルできていることを確認します:

   ```bash
   kubectl get pods -n wandb -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.phase}{"\t"}{.status.containerStatuses[*].image}{"\n"}{end}'
   ```

   すべてのイメージが内部Registryを参照しており、すべての Pod が `Running` 状態である必要があります。

2. **外部接続**: 에어갭モードでは W&amp;B が外部への接続を試みていないことを確認します (에어갭モードでは外部接続は行われないはずです) :

   ```bash
   kubectl logs -n wandb deployment/wandb-app --tail=100 | grep -i "connection"
   ```

3. **ライセンス検証**: W&amp;B コンソールに액세스し、ライセンスが有効になっていることを確認します。

<div id="troubleshooting">
  ## トラブルシューティング
</div>

<div id="image-pull-errors">
  ### イメージプル エラー
</div>

Pod がイメージをプルできない場合は、次を確認してください。

1. 内部Registryにイメージが存在することを確認する
2. イメージプル secret が正しく設定されているか確認する
3. Kubernetes ノードからRegistryへのネットワーク接続を確認する
4. Registryの認証情報を確認する

   ```bash
   # イメージプルを手動でテストする
   kubectl run test-pull --image=registry.yourdomain.com/wandb/local:0.59.2 --namespace=wandb
   kubectl logs test-pull -n wandb
   kubectl delete pod test-pull -n wandb
   ```

<div id="openshift-scc-errors">
  ### OpenShift SCC エラー
</div>

OpenShift で pod が権限エラーで失敗する場合:

```bash
# 使用中の SCC を確認する
oc get pod <pod-name> -n wandb -o yaml | grep scc

# サービスアカウントの権限を確認する
oc describe scc wandb-scc
oc get rolebinding -n wandb
```

<div id="helm-chart-not-found">
  ### Helm チャートが見つからない
</div>

operator が platform チャートを見つけられない場合は、次を確認してください:

1. Custom Resource 内のチャートリポジトリ URL を確認する
2. operator の Pod から内部 Helm リポジトリへ到達できることを確認する
3. 次のコマンドで、リポジトリ内にチャートが存在することを確認する:

   ```bash
   helm search repo local-repo/operator-wandb
   ```

<div id="frequently-asked-questions">
  ## よくある質問
</div>

<div id="can-i-use-a-different-ingress-class">
  ### 別の ingress クラスを使用できますか？
</div>

はい、Custom Resource 内の ingress 設定を変更して ingress クラスを設定してください。

```yaml
spec:
  values:
    ingress:
      class: your-ingress-class
```

<div id="how-do-i-handle-certificate-bundles-with-multiple-certificates">
  ### 複数の証明書を含む証明書バンドルはどうすればよいですか？
</div>

証明書を分割し、`customCACerts` セクションで複数のエントリとして指定します。

```yaml
spec:
  values:
    customCACerts:
      cert1.crt: |
        -----BEGIN CERTIFICATE-----
        ...
        -----END CERTIFICATE-----
      cert2.crt: |
        -----BEGIN CERTIFICATE-----
        ...
        -----END CERTIFICATE-----
```

<div id="how-do-i-prevent-automatic-updates">
  ### 自動アップデートを防ぐにはどうすればよいですか？
</div>

W&amp;B を自動的に更新しないようにオペレーターを設定します：

1. オペレーターのインストール時に `airgapped: true` を設定します (これにより自動更新チェックが無効になります)
2. Custom Resource 内の `spec.chart.version` を手動で更新してバージョン更新を制御します
3. オプションで、W&amp;B System Console から自動アップデートを無効にします

詳細は [Disable automatic app version updates](/ja/platform/hosting/self-managed/disable-automatic-app-version-updates) を参照してください。

<Note>
  W&amp;B は、セルフマネージド インスタンスを利用しているお客様に対し、サポートを維持し、最新の機能、パフォーマンス改善、および修正を受け取るために、少なくとも四半期に一度はデプロイを最新リリースに更新することを強く推奨します。W&amp;B は、メジャーリリースを初回リリース日から 12 か月間サポートします。[Release policies and processes](/ja/release-notes/release-policies) を参照してください。
</Note>

<div id="does-the-deployment-work-with-no-connection-to-public-repositories">
  ### デプロイはパブリック リポジトリへの接続が一切なくても動作しますか？
</div>

はい。`airgapped: true` が Operator の設定で指定されている場合、Operator は内部リソースのみを使用し、パブリック リポジトリへの接続を試みません。

<div id="how-do-i-update-wb-in-an-air-gapped-environment">
  ### 에어갭 環境で W&amp;B を更新するにはどうすればよいですか？
</div>

W&amp;B を更新するには、次の手順を実行します。

1. インターネットに接続されたシステムで新しいコンテナーイメージをプルします
2. コンテナーイメージを 에어갭 Registry に転送します
3. 新しい Helm チャートを内部リポジトリにアップロードします
4. Custom Resource で `spec.chart.version` とコンテナーイメージのタグを更新します
5. 更新した Custom Resource を適用します

   operator により、W&amp;B 컴포넌트 のローリングアップデートが実行されます。

<div id="next-steps">
  ## 次のステップ
</div>

デプロイが正常に完了したら、以下の手順を実行してください。

1. **ユーザー認証を設定する**: [SSO](/ja/platform/hosting/iam/sso) などの認証方法を設定する
2. **監視を設定する**: W&amp;B インスタンスとインフラストラクチャーの監視を設定する
3. **更新を計画する**: [Server upgrade process](/ja/platform/hosting/server-upgrade-process) を確認し、更新のサイクルを確立する
4. **バックアップを設定する**: MySQL データベースのバックアップ手順を設定する
5. **手順を文書化する**: 에어갭環境特有の更新手順向けに runbook を作成する
