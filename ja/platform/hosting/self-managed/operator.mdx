<div id="overview">
  ## 概要
</div>

W&amp;B Kubernetes Operator は、Kubernetes（クラウドまたはオンプレミス環境）上に W&amp;B Server をデプロイするための推奨される方法です。Operator の概要、W&amp;B がそれを採用している理由、および設定階層がどのように機能するかについては、[Self-Managed](/ja/platform/hosting/hosting-options/self-managed#about-the-wb-kubernetes-operator) を参照してください。

<div id="before-you-begin">
  ## はじめに
</div>

Kubernetes Operator を使用して W&amp;B をデプロイする前に、インフラストラクチャがすべての要件を満たしていることを確認してください。

1. **インフラストラクチャ要件を確認する**: 詳細については、[Self-Managed infrastructure requirements](/ja/platform/hosting/self-managed/requirements/) ページを参照してください。このページでは次の内容を包括的に説明しています:

* ソフトウェアバージョン要件 (Kubernetes、MySQL、Redis、Helm)
  * ハードウェア要件 (CPU アーキテクチャ、推奨構成)
  * Kubernetes クラスターの設定
  * ネットワーク、SSL/TLS、および DNS 要件

2. **W&amp;B Server ライセンスを取得する**: Requirements ページの [License](/ja/platform/hosting/self-managed/requirements#license) セクションを参照してください。

3. **外部サービスをプロビジョニングする**: デプロイ前に MySQL、Redis、およびオブジェクトストレージをセットアップしてください。

4. **その他の参考情報**: より詳しい背景情報については、[reference architecture](/ja/platform/hosting/self-managed/ref-arch/) ページを参照してください。

<div id="openshift-kubernetes-clusters">
  ### OpenShift Kubernetes クラスター
</div>

W&amp;B は、クラウド環境、オンプレミス環境、およびエアギャップ環境の [OpenShift Kubernetes クラスター](https://www.redhat.com/en/technologies/cloud-computing/openshift) へのデプロイをサポートしています。

<Note>
  W&amp;B では、公式の W&amp;B Helm チャートを使用してインストールすることを推奨します。
</Note>

<div id="run-the-container-as-an-un-privileged-user">
  #### 非特権ユーザーとしてコンテナを実行する
</div>

デフォルトでは、コンテナの `$UID` は 999 です。オーケストレーターがコンテナを非 root ユーザーで実行することを要求する場合は、`$UID` を 100000 以上、`$GID` を 0 に指定します。

<Note>
  ファイルシステムのパーミッションを正しく機能させるために、W&amp;B は root グループ（`$GID=0`）として起動する必要があります。
</Note>

各 W&amp;B コンポーネントのセキュリティコンテキストを設定します。たとえば、API コンポーネントを設定するには次のようにします。

```yaml
api:
  install: true
  image:
    repository: wandb/megabinary
    tag: 0.74.1  # 実際のバージョンに置き換えてください
  pod:
    securityContext:
      fsGroup: 10001
      fsGroupChangePolicy: Always
      runAsGroup: 0
      runAsNonRoot: true
      runAsUser: 10001
      seccompProfile:
        type: RuntimeDefault
  container:
    securityContext:
      allowPrivilegeEscalation: false
      capabilities:
        drop:
          - ALL
      privileged: false
      readOnlyRootFilesystem: false
```

必要に応じて、`app` や `console` などの他のコンポーネントに対してカスタム セキュリティコンテキストを設定してください。詳細は [Custom security context](#custom-security-context) を参照してください。

<div id="deploy-wb-server-application">
  ## W&amp;B Server アプリケーションをデプロイする
</div>

<Note>
  **W&amp;B Kubernetes Operator と Helm を使用したインストール方法が推奨です**。これは、クラウド、オンプレミス、エアギャップ環境を含むすべての W&amp;B Self-Managed デプロイに推奨されるインストール方法です。
</Note>

デプロイ方法を選択してください:

<Tabs>
  <Tab title="Helm CLI">
    W&amp;B は、W&amp;B Kubernetes operator を Kubernetes クラスターにデプロイするための Helm チャートを提供しています。この方法により、Helm CLI や ArgoCD のような継続的デリバリー ツールを使って W&amp;B Server をデプロイできます。

    デプロイに固有の考慮事項については、[環境固有の考慮事項](#environment-specific-considerations) および [パブリッククラウド上で Terraform を使ってデプロイ](#deploy-with-terraform-on-public-cloud) を参照してください。分離環境の場合は、[Air-Gapped Kubernetes へのデプロイ](/ja/platform/hosting/self-managed/on-premises-deployments/kubernetes-airgapped/) を参照してください。

    Helm CLI を使用して W&amp;B Kubernetes Operator をインストールするには、次の手順に従います。

    1. W&amp;B Helm リポジトリを追加します。W&amp;B Helm チャートは W&amp;B Helm リポジトリで利用可能です。
       ```shell
       helm repo add wandb https://charts.wandb.ai
       helm repo update
       ```

    2. Kubernetes クラスターに Operator をインストールします。
       ```shell
       helm upgrade --install operator wandb/operator -n wandb-cr --create-namespace
       ```

    3. W&amp;B operator のカスタムリソースを設定して、W&amp;B Server のインストールをトリガーします。W&amp;B デプロイの設定を含む `operator.yaml` という名前のファイルを作成します。利用可能なすべてのオプションについては、[設定リファレンス](#configuration-reference-for-wb-server) を参照してください。

       ここでは最小構成の例を示します。

       ```yaml
       apiVersion: apps.wandb.com/v1
       kind: WeightsAndBiases
       metadata:
         labels:
           app.kubernetes.io/name: weightsandbiases
           app.kubernetes.io/instance: wandb
         name: wandb
         namespace: default
       spec:
         values:
           global:
             host: https://<HOST_URI>
             license: eyJhbGnUzaH...j9ZieKQ2x5GGfw
             bucket:
               <details depend on the provider>
             mysql:
               <redacted>
           ingress:
             annotations:
               <redacted>
       ```

    4. カスタム設定を使用して Operator を起動し、W&amp;B Server アプリケーションをインストール、設定、管理できるようにします。

       ```shell
       kubectl apply -f operator.yaml
       ```

       デプロイが完了するまで待ちます。数分かかる場合があります。

    5. Web UI を使用してインストールを検証するには、最初の管理者ユーザーアカウントを作成し、[インストールの検証](#verify-the-installation) に記載されている検証手順に従ってください。
  </Tab>

  <Tab title="Terraform">
    Terraform を使用して Infrastructure as Code として W&amp;B をデプロイします。次のいずれかを選択します:

    * **Helm Terraform Module**: 既存の Kubernetes インフラストラクチャにオペレーターをデプロイします
    * **Cloud Terraform Modules**: AWS、Google Cloud、Azure 向けのインフラストラクチャとアプリケーションをまとめてデプロイします

    デプロイメント固有の考慮事項については、[Environment-specific considerations](#environment-specific-considerations) および [Deploy with Terraform on public cloud](#deploy-with-terraform-on-public-cloud) を参照してください。エアギャップなどのインターネット非接続環境の場合は、[Deploy on Air-Gapped Kubernetes](/ja/platform/hosting/self-managed/on-premises-deployments/kubernetes-airgapped/) を参照してください。

    #### Helm Terraform Module

    この方法では、Terraform の Infrastructure as Code アプローチを活用して、特定の要件に合わせてカスタマイズ可能なデプロイメントを行うことができ、一貫性と再現性を確保できます。公式の W&amp;B の Helm ベースの Terraform Module は [こちら](https://registry.terraform.io/modules/wandb/wandb/helm/latest) にあります。

    次のコードは出発点として使用でき、本番環境向けデプロイメントに必要なすべての設定オプションを含みます:

    ```hcl
    module "wandb" {
      source  = "wandb/wandb/helm"

      spec = {
        values = {
          global = {
            host    = "https://<HOST_URI>"
            license = "eyJhbGnUzaH...j9ZieKQ2x5GGfw"

            bucket = {
              <details depend on the provider>
            }

            mysql = {
              <redacted>
            }
          }

          ingress = {
            annotations = {
              "a" = "b"
              "x" = "y"
            }
          }
        }
      }
    }
    ```

    設定オプションは [Configuration Reference](#configuration-reference-for-wb-server) で説明されているものと同じですが、構文は HashiCorp Configuration Language (HCL) に従う必要があります。Terraform モジュールは W&amp;B の Custom Resource Definition (CRD) を作成します。

    W&amp;B 自身が Dedicated Cloud インストールを顧客向けにデプロイするために Helm Terraform モジュールをどのように使用しているかを確認するには、次のリンクを参照してください。

    * [AWS](https://github.com/wandb/terraform-aws-wandb/blob/45e1d746f53e78e73e68f911a1f8cad5408e74b6/main.tf#L225)
    * [Azure](https://github.com/wandb/terraform-azurerm-wandb/blob/170e03136b6b6fc758102d59dacda99768854045/main.tf#L155)
    * [Google Cloud](https://github.com/wandb/terraform-google-wandb/blob/49ddc3383df4cefc04337a2ae784f57ce2a2c699/main.tf#L189)

    #### Cloud Terraform Modules

    W&amp;B は AWS、Google Cloud、Azure 向けに一連の Terraform モジュールを提供しています。これらのモジュールは、Kubernetes クラスター、ロードバランサー、MySQL データベースなどを含むインフラストラクチャ全体と W&amp;B Server アプリケーションをデプロイします。W&amp;B Kubernetes Operator には、以下のバージョンの公式な W&amp;B クラウド別 Terraform モジュールがすでにあらかじめ組み込まれています。

    | Terraform Registry                                                  | Source Code                                      | Version |
    | ------------------------------------------------------------------- | ------------------------------------------------ | ------- |
    | [AWS](https://registry.terraform.io/modules/wandb/wandb/aws/latest) | https://github.com/wandb/terraform-aws-wandb     | v4.0.0+ |
    | [Azure](https://github.com/wandb/terraform-azurerm-wandb)           | https://github.com/wandb/terraform-azurerm-wandb | v2.0.0+ |
    | [Google Cloud](https://github.com/wandb/terraform-google-wandb)     | https://github.com/wandb/terraform-google-wandb  | v2.0.0+ |

    このインテグレーションにより、最小限のセットアップでお使いのインスタンスに対して W&amp;B Kubernetes Operator をすぐに利用できるようになり、クラウド環境での W&amp;B Server のデプロイと管理までのプロセスがスムーズになります。

    これらのクラウド別モジュールの詳細な使用手順については、以下の [Deploy with Terraform on public cloud](#deploy-with-terraform-on-public-cloud) を参照してください。
  </Tab>
</Tabs>

<div id="environment-specific-considerations">
  ## 環境ごとの考慮事項
</div>

Kubernetes は、オンプレミスかクラウドかに関係なく基本的に同じです。主な違いは、名称やマネージドサービスにあります（たとえば、MySQL と RDS、S3 とオンプレミスのオブジェクトストレージなど）。このセクションでは、環境ごとに異なる考慮事項について説明します。

<div id="on-premises-and-bare-metal">
  ### オンプレミスおよびベアメタル
</div>

オンプレミス環境やベアメタル環境で Kubernetes をデプロイする際は、以下の点に留意してください。

<div id="load-balancer-configuration">
  #### ロードバランサーの設定
</div>

オンプレミスの Kubernetes クラスターでは、通常、ロードバランサーの設定を手動で行う必要があります。主な選択肢は次のとおりです。

* **外部ロードバランサー**: 既存のハードウェアまたはソフトウェアのロードバランサー (F5、HAProxy など) を設定する
* **Nginx Ingress Controller**: NodePort またはホストネットワーク構成で nginx-ingress-controller をデプロイする
* **MetalLB**: ベアメタルの Kubernetes クラスター向けに、MetalLB がロードバランサー サービスを提供する

ロードバランサーの設定に関する詳細な例については、[Reference Architecture のネットワーキング セクション](/ja/platform/hosting/self-managed/ref-arch#networking)を参照してください。

<div id="persistent-storage">
  #### 永続ストレージ
</div>

Kubernetes クラスターに永続ボリューム用の StorageClass が構成されていることを確認してください。W&amp;B コンポーネントは、キャッシュや一時データの保存のために永続ストレージを必要とする場合があります。

オンプレミス環境で一般的なストレージ オプションは次のとおりです:

* NFS ベースの StorageClass
* Ceph/Rook ストレージ
* ローカル永続ボリューム
* エンタープライズ向けストレージ ソリューション（NetApp、Pure Storage など）

<div id="dns-and-certificate-management">
  #### DNS と証明書の管理
</div>

オンプレミス デプロイの場合:

* 内部 DNS レコードを W&amp;B ホスト名を指すように設定する
* 内部 Certificate Authority (CA) から SSL/TLS 証明書を発行する
* 自己署名証明書を使用する場合は、オペレーターが CA 証明書を信頼するように設定する

証明書の設定の詳細については、[SSL/TLS の要件](/ja/platform/hosting/self-managed/requirements#ssl-tls) を参照してください。

<div id="openshift-deployments">
  #### OpenShift デプロイ
</div>

W&amp;B は OpenShift Kubernetes クラスターへのデプロイを完全にサポートしています。OpenShift デプロイでは、OpenShift のより厳格なセキュリティポリシーにより、追加のセキュリティコンテキストの設定が必要です。

OpenShift 固有の設定の詳細については、上記の [OpenShift Kubernetes クラスター](#openshift-kubernetes-clusters) を参照してください。エアギャップ環境における OpenShift の包括的な例については、[Deploy on Air-Gapped Kubernetes](/ja/platform/hosting/self-managed/on-premises-deployments/kubernetes-airgapped#openshift-configuration) を参照してください。

<div id="object-storage-for-on-premises-and-s3-compatible">
  #### オンプレミスおよび S3 互換環境向けのオブジェクトストレージ
</div>

オブジェクトストレージバケットをプロビジョニングしたら（[Object storage provisioning](/ja/platform/hosting/data-security/secure-storage-connector) を参照）、W&amp;B Custom Resource で設定します。

**AWS S3（オンプレミス）**

オンプレミス環境で AWS S3（Outposts または互換ストレージ経由）を利用する場合：

```yaml
bucket:
  kmsKey: <kms key arn>  # 省略可能な暗号化用 KMS キー
  name: <bucket name>    # 例: wandb
  path: ""               # 空文字列のままにしてください
  provider: s3
  region: <region>       # 例: us-east-1
```

**S3 互換ストレージ (MinIO、Ceph、NetApp など)**

S3 互換ストレージシステムを使用する場合:

```yaml
bucket:
  kmsKey: null
  name: <s3 endpoint>    # 例: s3.example.com:9000
  path: <bucket name>    # 例: wandb
  provider: s3
  region: <region>       # 例: us-east-1
```

S3 互換ストレージで TLS を有効化するには、バケットパスの末尾に `?tls=true` を追加します。

```yaml
bucket:
  path: "wandb?tls=true"
```

<Warning>
  証明書は信頼されている必要があります。自己署名証明書を使用する場合は追加の設定が必要です。詳細は [SSL/TLS requirements](/ja/platform/hosting/self-managed/requirements#ssl-tls) を参照してください。
</Warning>

**オンプレミスのオブジェクトストレージに関する重要な考慮事項**

独自のオブジェクトストレージを運用する場合、次の点を考慮してください。

1. **ストレージ容量とパフォーマンス**: ディスク容量を注意深く監視してください。平均的な W&amp;B の利用では、数十〜数百 GB 程度になります。大規模な利用では、ストレージ消費がペタバイト単位に達する可能性があります。
2. **フォールトトレランス**: 最低限、物理ディスクには RAID アレイを使用してください。S3 互換ストレージの場合は、分散構成または高可用性構成を使用してください。
3. **可用性**: ストレージが利用可能な状態を維持できているかを確認できるよう、モニタリングを設定してください。

**MinIO に関する考慮事項**

<Warning>
  MinIO Open Source は現在 [maintenance mode](https://github.com/minio/minio) にあり、積極的な開発は行われていません。事前コンパイル済みバイナリは提供されなくなっており、重大なセキュリティ修正のみがケースバイケースで検討されます。本番環境のデプロイには、W&amp;B はマネージドオブジェクトストレージサービス、または [MinIO Enterprise (AIStor)](https://min.io/product/aistor) の利用を推奨します。
</Warning>

オンプレミスのオブジェクトストレージ向けエンタープライズ代替製品としては、次のようなものがあります。

* [Amazon S3 on Outposts](https://aws.amazon.com/s3/outposts/)
* [NetApp StorageGRID](https://www.netapp.com/data-storage/storagegrid/)
* MinIO Enterprise (AIStor)
* [Dell ObjectScale](https://www.dell.com/en-us/shop/cty/sf/objectscale)

既存の MinIO デプロイメントまたは MinIO Enterprise を使用している場合は、MinIO クライアントを使用してバケットを作成できます。

```bash
mc config host add local http://$MINIO_HOST:$MINIO_PORT "$MINIO_ACCESS_KEY" "$MINIO_SECRET_KEY" --api s3v4
mc mb --region=us-east-1 local/wandb-files
```

  ### Terraform を使用したパブリッククラウド
</div>

AWS、Google Cloud、または Azure 上でインフラストラクチャとアプリケーションをまとめてデプロイする手順の全体像については、以下の [Deploy with Terraform on public cloud](#deploy-with-terraform-on-public-cloud) を参照してください。

<div id="deploy-with-terraform-on-public-cloud">
  ## パブリッククラウド上で Terraform を使ってデプロイ
</div>

<Note>
  W&amp;B は、[W&amp;B Multi-tenant Cloud](/ja/platform/hosting/hosting-options/multi_tenant_cloud) や [W&amp;B Dedicated Cloud](/ja/platform/hosting/hosting-options/dedicated_cloud/) のようなフルマネージドのデプロイ オプションを推奨します。W&amp;B のフルマネージドサービスは、必要な設定が最小限または不要で、簡単かつ安全に利用できます。
</Note>

W&amp;B は、パブリッククラウドプロバイダー上にプラットフォームをデプロイするための Terraform モジュールを提供しています。これらのモジュールは、インフラストラクチャのプロビジョニングと W&amp;B Server のインストールを自動化します。

開始する前に、Terraform 用の [remote backends](https://developer.hashicorp.com/terraform/language/backend) のいずれかを選択し、[State File](https://developer.hashicorp.com/terraform/language/state) を保存することを W&amp;B は推奨します。State File は、すべてのコンポーネントを再作成せずに、デプロイのアップグレードや変更を行うために必要なリソースです。

クラウドプロバイダーを選択してください。

<Tabs>
  <Tab title="AWS">
    W&amp;B は、AWS 上にプラットフォームをデプロイするには [W&amp;B Server AWS Terraform Module](https://registry.terraform.io/modules/wandb/wandb/aws/latest) の使用を推奨しています。

    Terraform Module は以下の必須コンポーネントをデプロイします：

    * ロードバランサー
    * AWS Identity &amp; Access Management (IAM)
    * AWS Key Management System (KMS)
    * Amazon Aurora MySQL
    * Amazon VPC
    * Amazon S3
    * Amazon Route 53
    * Amazon Certificate Manager (ACM)
    * Amazon Elastic Load Balancing (ALB)
    * Amazon Secrets Manager

    1. 開発環境を準備する。
       * [Terraform](https://developer.hashicorp.com/terraform/tutorials/aws-get-started/install-cli) をインストールします
       * W&amp;B では、バージョン管理のために Git リポジトリを作成することを推奨しています。

    2. `terraform.tfvars` ファイルを作成します。

       ```bash
       namespace                  = "wandb"
       license                    = "xxxxxxxxxxyyyyyyyyyyyzzzzzzz"
       subdomain                  = "wandb-aws"
       domain_name                = "wandb.ml"
       zone_id                    = "xxxxxxxxxxxxxxxx"
       allowed_inbound_cidr       = ["0.0.0.0/0"]
       allowed_inbound_ipv6_cidr  = ["::/0"]
       eks_cluster_version        = "1.29"
       ```

       `namespace` 変数は、Terraform によって作成されるすべてのリソース名の先頭に付与される文字列なので、デプロイする前に `tvfars` ファイル内で変数を定義しておいてください。

       `subdomain` と `domain` を組み合わせたものが、W&amp;B を構成するための FQDN になります。上記の例では、W&amp;B の FQDN は `wandb-aws.wandb.ml` となり、その FQDN レコードが作成される DNS `zone_id` を指定します。

       `allowed_inbound_cidr` と `allowed_inbound_ipv6_cidr` も設定する必要があります。モジュールでは、これは必須の入力です。次の例では、任意の送信元から W&amp;B インストールへのアクセスを許可します。

       このファイルでは、AWS 上に W&amp;B をデプロイするために必要な Terraform および Terraform プロバイダーのバージョンを指定します。

       ```bash
       provider "aws" {
         region = "eu-central-1"

         default_tags {
           tags = {
             GithubRepo = "terraform-aws-wandb"
             GithubOrg  = "wandb"
             Enviroment = "Example"
             Example    = "PublicDnsExternal"
           }
         }
       }
       ```

       AWS プロバイダーを設定するには、[Terraform の公式ドキュメント](https://registry.terraform.io/providers/hashicorp/aws/latest/docs#provider-configuration)を参照してください。

       必須ではありませんが強く推奨される手順として、このドキュメントの冒頭で説明した [remote backend 設定](https://developer.hashicorp.com/terraform/language/backend) を追加してください。

       `terraform.tfvars` で設定した各オプションごとに、Terraform ではそれぞれに対応する変数宣言が必要です。

       ```
       variable "namespace" {
         type        = string
         description = "Name prefix used for resources"
       }

       variable "domain_name" {
         type        = string
         description = "Domain name used to access instance."
       }

       variable "subdomain" {
         type        = string
         default     = null
         description = "Subdomain for accessing the Weights & Biases UI."
       }

       variable "license" {
         type = string
       }

       variable "zone_id" {
         type        = string
         description = "Domain for creating the Weights & Biases subdomain on."
       }

       variable "allowed_inbound_cidr" {
        description = "CIDRs allowed to access wandb-server."
        nullable    = false
        type        = list(string)
       }

       variable "allowed_inbound_ipv6_cidr" {
        description = "CIDRs allowed to access wandb-server."
        nullable    = false
        type        = list(string)
       }

       variable "eks_cluster_version" {
        description = "EKS cluster kubernetes version"
        nullable    = false
        type        = string
       }
       ```

       ```

         source  = "wandb/wandb/aws"
         version = "~>7.0"

         namespace   = var.namespace
         domain_name = var.domain_name
         subdomain   = var.subdomain
         zone_id     = var.zone_id

         allowed_inbound_cidr           = var.allowed_inbound_cidr
         allowed_inbound_ipv6_cidr      = var.allowed_inbound_ipv6_cidr

         public_access                  = true
         external_dns                   = true
         kubernetes_public_access       = true
         kubernetes_public_access_cidrs = ["0.0.0.0/0"]
         eks_cluster_version            = var.eks_cluster_version
       }

        data "aws_eks_cluster" "eks_cluster_id" {
          name = module.wandb_infra.cluster_name
        }

        data "aws_eks_cluster_auth" "eks_cluster_auth" {
          name = module.wandb_infra.cluster_name
        }

        provider "kubernetes" {
          host                   = data.aws_eks_cluster.eks_cluster_id.endpoint
          cluster_ca_certificate = base64decode(data.aws_eks_cluster.eks_cluster_id.certificate_authority.0.data)
          token                  = data.aws_eks_cluster_auth.eks_cluster_auth.token
        }


        provider "helm" {
          kubernetes {
            host                   = data.aws_eks_cluster.eks_cluster_id.endpoint
            cluster_ca_certificate = base64decode(data.aws_eks_cluster.eks_cluster_id.certificate_authority.0.data)
            token                  = data.aws_eks_cluster_auth.eks_cluster_auth.token
          }
        }

        output "url" {
          value = module.wandb_infra.url
        }

        output "bucket" {
          value = module.wandb_infra.bucket_name
        }
       ```

       ```
       terraform apply -var-file=terraform.tfvars
       ```

      W&amp;B にはブローカーが組み込まれているため、これは必須ではありません。このオプションを指定してもパフォーマンスは向上しません。
    </Note>

    ```bash
    # 必要なサービスをすべて起動する
    module "wandb" {
      source  = "wandb/wandb/azurerm"
      version = "~> 1.2"

      namespace   = var.namespace
      location    = var.location
      license     = var.license
      domain_name = var.domain_name
      subdomain   = var.subdomain

      use_internal_queue = false
      [...]
    }
    ```

    ### 追加リソース

    * [Azure 用 Terraform モジュールのドキュメント](https://registry.terraform.io/modules/wandb/wandb/azurerm/latest)
    * [Azure 用 Terraform モジュールのソースコード](https://github.com/wandb/terraform-azurerm-wandb)
  </Tab>
</Tabs>

<div id="migrate-to-operator-based-helm-chart">
  ### Operator ベースの Helm チャートに移行する
</div>

次の手順に従って、Operator ベースの Helm チャートに移行します。

1. 現在の W&amp;B 設定を取得します。W&amp;B が Operator を使用しないバージョンの Helm チャートでデプロイされている場合は、次のように値をエクスポートします：
   ```shell
   helm get values wandb
   ```
   W&amp;B が Kubernetes マニフェストでデプロイされている場合は、次のように値をエクスポートします：
   ```shell
   kubectl get deployment wandb -o yaml
   ```
   これで次のステップに必要なすべての設定値がそろいました。

2. `operator.yaml` というファイルを作成します。[Configuration Reference](#configuration-reference-for-wb-operator) に記載されている形式に従い、手順 1 で取得した値を使用します。

3. 現在の Deployment を pod 数 0 にスケールします。このステップで現在の Deployment を停止します。
   ```shell
   kubectl scale --replicas=0 deployment wandb
   ```

4. Helm チャート リポジトリを更新します：
   ```shell
   helm repo update
   ```

5. 新しい Helm チャートをインストールします：
   ```shell
   helm upgrade --install operator wandb/operator -n wandb-cr --create-namespace
   ```

6. 新しい Helm チャートを設定し、W&amp;B アプリケーションのデプロイをトリガーします。新しい設定を適用します。
   ```shell
   kubectl apply -f operator.yaml
   ```
   デプロイが完了するまで数分かかります。

7. インストールを検証します。[Verify the installation](#verify-the-installation) の手順に従って、すべてが正しく動作していることを確認します。

8. 古いインストールを削除します。古い Helm チャートをアンインストールするか、マニフェストで作成されたリソースを削除します。

<div id="migrate-to-operator-based-terraform-helm-chart">
  ### Operator ベースの Terraform Helm チャートへ移行する
</div>

以下の手順に従って、Operator ベースの Helm チャートに移行します。

1. Terraform 設定ファイルを準備します。Terraform 設定ファイル内の旧デプロイメントの Terraform コードを、[こちら](#deploy-wb-with-helm-terraform-module) で説明しているコードに置き換えます。以前と同じ変数を設定します。.tfvars ファイルがある場合は変更しないでください。
2. Terraform を実行します。`terraform init`、`plan`、`apply` を実行します。
3. インストールを検証します。[インストールの検証](#verify-the-installation) の手順に従って、すべてが正しく動作していることを確認します。
4. 旧インストールを削除します。古い Helm チャートをアンインストールするか、マニフェストから作成されたリソースを削除します。

<div id="ingress">
  ### Ingress
</div>

Ingress クラスを特定するには、この FAQ の[項目](#how-to-identify-the-kubernetes-ingress-class)を参照してください。

**TLS なし**

```yaml
global:
# 重要: Ingress は YAML 内で 'global' と同じレベルにあります（子要素ではありません）
ingress:
  class: ""
```

**TLS を使用する場合**

```console
kubectl create secret tls wandb-ingress-tls --key wandb-ingress-tls.key --cert wandb-ingress-tls.crt
```

ingress の設定で secret を参照する

```yaml
global:
# 重要: Ingress は YAML 内で 'global' と同じレベルにあります（子要素ではありません）
ingress:
  class: ""
  annotations:
    {}
    # kubernetes.io/ingress.class: nginx
    # kubernetes.io/tls-acme: "true"
  tls: 
    - secretName: wandb-ingress-tls
      hosts:
        - <HOST_URI>
```

Nginx の場合には、次の annotation を追加する必要が生じる場合があります:

```
ingress:
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: 0
```

<div id="custom-kubernetes-serviceaccounts">
  ### カスタム Kubernetes ServiceAccount
</div>

W&amp;B の Pod を実行するためのカスタム Kubernetes ServiceAccount を指定します。

次のスニペットでは、指定した名前でデプロイの一部として ServiceAccount を作成します。

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: true

parquet:
  serviceAccount:
    name: custom-service-account
    create: true

global:
  ...
```

サブシステムの &quot;app&quot; と &quot;parquet&quot; は、指定されたサービスアカウントで実行されます。他のサブシステムは、デフォルトのサービスアカウントで実行されます。

すでにクラスターにサービスアカウントが存在する場合は、`create: false` を設定します。

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: false

parquet:
  serviceAccount:
    name: custom-service-account
    create: false
    
global:
  ...
```

app や parquet、console などのサブシステムごとに、サービスアカウントを指定できます。

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: true

console:
  serviceAccount:
    name: custom-service-account
    create: true

global:
  ...
```

サービスアカウントは、サブシステムごとに異なっていてもかまいません。

```yaml
app:
  serviceAccount:
    name: custom-service-account
    create: false

console:
  serviceAccount:
    name: another-custom-service-account
    create: true

global:
  ...
```

<div id="ldap">
  ### LDAP
</div>

<Warning>
  現在の Helm チャートでは LDAP 設定のサポートは制限されています。LDAP の設定については W&amp;B Support または AISE に連絡してください。
</Warning>

`global.extraEnv` に環境変数を設定して LDAP を構成します。

```yaml
global:
  extraEnv:
    LDAP_ADDRESS: ldaps://ldap.company.example.com
    LDAP_BASE_DN: cn=accounts,dc=company,dc=example,dc=com
    LDAP_USER_BASE_DN: cn=users,cn=accounts,dc=company,dc=example,dc=com
    LDAP_GROUP_BASE_DN: cn=groups,cn=accounts,dc=company,dc=example,dc=com
    LDAP_BIND_DN: uid=ldapbind,cn=sysaccounts,cn=etc,dc=company,dc=example,dc=com
    LDAP_BIND_PW: ********************
    LDAP_ATTRIBUTES: email=mail,name=cn
    LDAP_TLS_ENABLE: "true"
    LDAP_LOGIN: "true"
    LDAP_USER_OBJECT_CLASS: user
    LDAP_GROUP_OBJECT_CLASS: group
```

<accordion title="レガシー LDAP 設定">
  このレガシーなアプローチは現在推奨されません。このセクションは参考用に掲載しています。

  **TLS なし**

  ```yaml
  global:
    ldap:
      enabled: true
      # "ldap://" または "ldaps://" を含む LDAP サーバーアドレス
      host:
      # ユーザー検索に使用する LDAP 検索ベース
      baseDN:
      # 匿名バインドを使用しない場合にバインドに使用する LDAP ユーザー
      bindDN:
      # 匿名バインドを使用しない場合に、バインドに使用する LDAP パスワードを含む Secret 名とキー
      bindPW:
      # メールアドレスおよびグループ ID の属性名。カンマ区切りの文字列値として指定します。
      attributes:
      # LDAP グループの許可リスト
      groupAllowList:
      # LDAP TLS を有効化
      tls: false
  ```

  **TLS あり**

  LDAP TLS 証明書の設定では、証明書の内容を含む ConfigMap を事前に作成しておく必要があります。

  ConfigMap を作成するには、次のコマンドを使用します。

  ```console
  kubectl create configmap ldap-tls-cert --from-file=certificate.crt
  ```

  その後、以下の例のように YAML 内でその ConfigMap を使用します。

  ```yaml
  global:
    ldap:
      enabled: true
      # "ldap://" または "ldaps://" を含む LDAP サーバーアドレス
      host:
      # ユーザー検索に使用する LDAP 検索ベース
      baseDN:
      # 匿名バインドを使用しない場合にバインドに使用する LDAP ユーザー
      bindDN:
      # 匿名バインドを使用しない場合に、バインドに使用する LDAP パスワードを含む Secret 名とキー
      bindPW:
      # メールアドレスおよびグループ ID の属性名。カンマ区切りの文字列値として指定します。
      attributes:
      # LDAP グループの許可リスト
      groupAllowList:
      # LDAP TLS を有効化
      tls: true
      # LDAP サーバー用 CA 証明書を含む ConfigMap 名とキー
      tlsCert:
        configMap:
          name: "ldap-tls-cert"
          key: "certificate.crt"
  ```
</accordion>

<div id="custom-security-context">
  ### カスタム セキュリティコンテキスト
</div>

各 W&amp;B コンポーネントでは、次の形式のカスタム セキュリティコンテキストの設定をサポートしています。

```yaml
pod:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1001
    runAsGroup: 0
    fsGroup: 1001
    fsGroupChangePolicy: Always
    seccompProfile:
      type: RuntimeDefault
container:
  securityContext:
    capabilities:
      drop:
        - ALL
    readOnlyRootFilesystem: false
    allowPrivilegeEscalation: false 
```

<Note>
  `runAsGroup:` の唯一の有効な値は `0` です。それ以外の値はエラーになります。
</Note>

たとえば、アプリケーション Pod を設定するには、設定に `app` セクションを追加します。

```yaml
global:
  ...
app:
  pod:
    securityContext:
      runAsNonRoot: true
      runAsUser: 1001
      runAsGroup: 0
      fsGroup: 1001
      fsGroupChangePolicy: Always
      seccompProfile:
        type: RuntimeDefault
  container:
    securityContext:
      capabilities:
        drop:
          - ALL
      readOnlyRootFilesystem: false
      allowPrivilegeEscalation: false 
```

同じ考え方が `console`、`weave`、`weave-trace`、`parquet` にも当てはまります。

<div id="what-is-the-purposerole-of-each-individual-pod">
  ### 各 pod 個別の目的／役割は何ですか？
</div>

* **`wandb-app`**: GraphQL API とフロントエンドアプリケーションを含む W&amp;B の中核コンポーネントです。プラットフォームのほとんどの機能を提供します。
* **`wandb-console`**: `/console` からアクセスする管理コンソールです。
* **`wandb-otel`**: OpenTelemetry エージェントで、Kubernetes レイヤー上のリソースからメトリクスとログを収集し、管理コンソールに表示します。
* **`wandb-prometheus`**: Prometheus サーバーで、各種コンポーネントからメトリクスを収集して管理コンソールに表示します。
* **`wandb-parquet`**: データベースのデータを Parquet 形式でオブジェクトストレージにエクスポートする、`wandb-app` pod とは別のバックエンドマイクロサービスです。
* **`wandb-weave`**: UI でクエリテーブルを読み込み、さまざまなコアアプリケーション機能をサポートする、別のバックエンドマイクロサービスです。
* **`wandb-weave-trace`**: LLM ベースのアプリケーションをトラッキング、実験、評価、デプロイ、および改善するためのフレームワークです。このフレームワークには `wandb-app` pod 経由でアクセスします。
