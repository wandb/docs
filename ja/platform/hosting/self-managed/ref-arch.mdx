このページでは、W\&B デプロイメントのリファレンスアーキテクチャについて説明し、プラットフォームの本番デプロイメントをサポートするために推奨されるインフラストラクチャとリソースの概要を示します。

W\&B に選択したデプロイメント環境によっては、さまざまなサービスがデプロイメントの回復性を高めるのに役立ちます。

たとえば、主要なクラウドプロバイダーは、データベースの設定、メンテナンス、高可用性、および回復性の複雑さを軽減するのに役立つ堅牢なマネージドデータベースサービスを提供しています。

このリファレンスアーキテクチャは、いくつかの一般的なデプロイメントシナリオに対応し、最適なパフォーマンスと信頼性のためにクラウドベンダーサービスと W\&B デプロイメントを統合する方法を示します。

## 始める前に

本番環境でアプリケーションを実行することには固有の課題があり、W\&B も例外ではありません。プロセスの合理化を目指していますが、固有のアーキテクチャや設計上の決定によっては、特定の複雑さが生じる場合があります。通常、本番デプロイメントの管理には、ハードウェア、オペレーティングシステム、ネットワーキング、ストレージ、セキュリティ、W\&B プラットフォーム自体、およびその他の依存関係など、さまざまなコンポーネントの監視が含まれます。この責任は、環境の初期セットアップとその継続的なメンテナンスの両方に及びます。

W\&B でのセルフマネージドアプローチがチームと特定の要件に適しているかどうかを慎重に検討してください。

本番グレードのアプリケーションを実行および維持する方法についての深い理解は、セルフマネージド W\&B をデプロイする前の重要な前提条件です。チームがサポートを必要とする場合、プロフェッショナルサービスチームとパートナーが実装と最適化のサポートを提供しています。

自分で管理する代わりに W\&B を実行するためのマネージドソリューションの詳細については、[W\&B Multi-tenant Cloud](/ja/platform/hosting/hosting-options/multi_tenant_cloud) および [W\&B Dedicated Cloud](/ja/platform/hosting/hosting-options/dedicated-cloud)を参照してください。

### アプリケーション層

アプリケーション層は、ノード障害に対する回復性を備えたマルチノード Kubernetes クラスターで構成されています。Kubernetes クラスターは W\&B の pod を実行および維持します。

### ストレージ層

ストレージ層は、MySQL データベースとオブジェクトストレージで構成されています。MySQL データベースはメタデータを保存し、オブジェクトストレージはモデルやデータセットなどの artifact を保存します。

## インフラストラクチャ要件

以下のセクションでは、Kubernetes クラスターの詳細、MySQL、Redis、オブジェクトストレージ、ソフトウェアバージョン、ネットワーキング、DNS、ロードバランサーと ingress、SSL/TLS、およびサポートされている CPU アーキテクチャなど、W\&B デプロイメントのさまざまな側面に関する要件を詳しく説明します。

### Kubernetes

W\&B Server アプリケーションは、[Kubernetes Operator](/ja/platform/hosting/self-managed/operator) として複数の pod をデプロイします。このため、W\&B には以下を備えた Kubernetes クラスターが必要です：

* 完全に設定され機能している Ingress コントローラー。
* Persistent Volume をプロビジョニングする機能。

W\&B は、[OpenShift Kubernetes クラスター](https://www.redhat.com/en/technologies/cloud-computing/openshift) でのクラウド、オンプレミス、およびエアギャップ環境へのデプロイメントをサポートしています。特定の設定手順については、Operator ガイドの[OpenShift セクション](/ja/platform/hosting/self-managed/operator#openshift-kubernetes-clusters)を参照してください。

### MySQL

W\&B はメタデータを MySQL データベースに保存します。データベースのパフォーマンスとストレージ要件は、モデルパラメーターの形状および関連するメタデータによって異なります。たとえば、より多くのトレーニング run を追跡するにつれてデータベースのサイズが増大し、run テーブル、ユーザーワークスペース、およびレポートのクエリに基づいてデータベースへの負荷が増加します。

**W\&B はマネージドデータベースサービスの使用を強く推奨します**（AWS RDS Aurora MySQL、Google Cloud SQL for MySQL、Azure Database for MySQL など）本番デプロイメント向けに。マネージドサービスは、自動バックアップ、モニタリング、高可用性、パッチ適用を提供し、運用の複雑さを大幅に軽減します。特定のサービスの推奨事項については、以下の[クラウドプロバイダーインスタンスの推奨事項](#cloud-provider-instance-recommendations)セクションを参照してください。

セルフマネージド MySQL データベースをデプロイすることを選択した場合は、以下を考慮してください：

* **バックアップ**：データベースを定期的に別の施設にバックアップする必要があります。W\&B は、少なくとも 1 週間の保持期間で毎日バックアップすることを推奨します。
* **パフォーマンス**：データベースには、SSD や高速 NAS などの高速ストレージハードウェアが必要です。
* **モニタリング**：データベースには適切な CPU リソースが必要です。データベースサーバーの CPU 負荷を監視してください。CPU 使用率がシステムの 90% を超えた状態が 5 分以上続く場合は、CPU 容量の追加を検討してください。
* **可用性**：可用性と耐久性の要件を満たすために、W\&B は、プライマリデプロイメントからすべての更新をリアルタイムでストリーミングし、プライマリサーバーがクラッシュ、破損、または継続的なダウンタイムが発生した場合にフェイルオーバーできる準備が整った、別のマシン上のホットスタンバイデプロイメントを設定することを推奨します。

#### MySQL トポロジー

本番環境では、クラウドプロバイダーがフェイルオーバー、バックアップ、およびパッチ適用を処理するため、マネージド MySQL サービスが高可用性への最もシンプルな方法です。プロバイダーの高可用性オプション（例：AWS の Aurora Multi-AZ）を使用してください。

セルフマネージド MySQL を実行する場合は、リアルタイムのレプリケーションストリームを受信し、障害時に引き継ぐことができるホットスタンバイを備えたプライマリデータベースを使用してください。W\&B は、アプリケーションデータベースのマルチマスタートポロジーや読み取り専用レプリカをサポートしていません。

#### MySQL データベースの作成

MySQL データベースとユーザーを手動で作成する手順については、[ベアメタルガイドの MySQL データベースセクション](/ja/platform/hosting/self-managed/operator#mysql-database)を参照してください。

#### MySQL 設定パラメーター

独自の MySQL インスタンスを実行している場合は、以下の設定で MySQL を構成してください：

```
binlog_format = 'ROW'
binlog_row_image = 'MINIMAL'
innodb_flush_log_at_trx_commit = 1
innodb_online_alter_log_max_size = 268435456
max_prepared_stmt_count = 1048576
sort_buffer_size = '67108864'
sync_binlog = 1
```

これらの設定は、最適なパフォーマンスと信頼性のために W\&B によって検証されています。

### Redis

W\&B は、ジョブキューイングとデータキャッシングのために W\&B のコンポーネントが使用するシングルノード Redis 7.x デプロイメントに依存しています。テストや概念実証の開発時の利便性のために、W\&B Self-Managed にはローカル Redis デプロイメントが含まれていますが、これは本番デプロイメントには適していません。

W\&B は以下の環境の Redis インスタンスに接続できます：

* [AWS Elasticache](https://aws.amazon.com/elasticache/)
* [Google Cloud Memory Store](https://cloud.google.com/memorystore?hl=en)
* [Azure Cache for Redis](https://azure.microsoft.com/en-us/products/cache)
* クラウドまたはオンプレミスインフラストラクチャでホストされた Redis デプロイメント

### オブジェクトストレージ

W\&B には、以下のいずれかにデプロイされた、事前署名 URL と CORS サポートを備えたオブジェクトストレージが必要です：

* [CoreWeave AI Object Storage](https://docs.coreweave.com/products/storage/object-storage) は、AI ワークロード向けに最適化された高性能な S3 互換オブジェクトストレージサービスです。
* [Amazon S3](https://aws.amazon.com/s3/) は、業界をリードするスケーラビリティ、データ可用性、セキュリティ、およびパフォーマンスを提供するオブジェクトストレージサービスです。
* [Google Cloud Storage](https://cloud.google.com/storage) は、大規模な非構造化データを保存するためのマネージドサービスです。
* [Azure Blob Storage](https://azure.microsoft.com/en-us/products/storage/blobs) は、テキスト、バイナリデータ、画像、動画、ログなどの大量の非構造化データを保存するためのクラウドベースのオブジェクトストレージソリューションです。
* S3 互換ストレージ（例：[MinIO Enterprise (AIStor)](https://www.min.io/product/aistor)、[NetApp StorageGRID](https://www.netapp.com/data-storage/storagegrid/)、またはクラウドやオンプレミスインフラストラクチャでホストされているその他のエンタープライズグレードのソリューション。

### DNS

W\&B デプロイメントの完全修飾ドメイン名（FQDN）は、A レコードを使用してイングレス/ロードバランサーの IP アドレスに解決される必要があります。

### ロードバランサーとイングレス

W\&B Kubernetes Operator は、Kubernetes イングレスコントローラーを使用してサービスを公開できます。このコントローラーは、異なるポートを持つ URL パスに基づいてサービスエンドポイントにルーティングします。イングレスコントローラーは、機械学習ペイロードを実行するすべてのマシン、またはウェブブラウザーを通じてサービスにアクセスするすべてのマシンからアクセス可能である必要があります。

#### イングレスコントローラーの要件

Kubernetes クラスターには `IngressClass` が必要です。一般的なイングレスコントローラーのオプションには以下が含まれます：

* [Nginx Ingress Controller](https://kubernetes.github.io/ingress-nginx/)
* [Istio](https://istio.io)
* [Traefik](https://traefik.io/)
* クラウドプロバイダーのイングレスコントローラー（AWS ALB、GCP Ingress、Azure Application Gateway）

#### W\&B サービスルーティング

W\&B Operator はパスに基づいて複数のバックエンドサービスへのリクエストを自動的にルーティングします：

| パス          | サービス                | デフォルトポート | 目的                      |
| ----------- | ------------------- | -------- | ----------------------- |
| `/`         | `wandb-app`         | 8080     | メイン Web アプリケーション UI     |
| `/api`      | `wandb-api`         | 8081     | API サービス                |
| `/graphql`  | `wandb-api`         | 8081     | GraphQL API エンドポイント     |
| `/graphql2` | `wandb-api`         | 8081     | GraphQL API v2 エンドポイント  |
| `/console`  | `wandb-console`     | 8082     | System Console          |
| `/traces`   | `wandb-weave-trace` | 8722     | Weave トレーシングサービス（有効な場合） |

#### イングレス設定の例

以下は W\&B Operator によって作成されたイングレスリソースの例を示しています：

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: wandb
  namespace: wandb
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: "0"
spec:
  ingressClassName: nginx
  rules:
  - host: wandb.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: wandb-app
            port:
              number: 8080
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: wandb-api
            port:
              number: 8081
      - path: /graphql
        pathType: Prefix
        backend:
          service:
            name: wandb-api
            port:
              number: 8081
      - path: /graphql2
        pathType: Prefix
        backend:
          service:
            name: wandb-api
            port:
              number: 8081
      - path: /console
        pathType: Prefix
        backend:
          service:
            name: wandb-console
            port:
              number: 8082
  tls:
  - hosts:
    - wandb.example.com
    secretName: wandb-tls
```

<Note>
  W\&B Operator はイングレス設定を自動的に作成および管理します。通常、イングレスリソースを手動で作成する必要はありません。クラスターに機能するイングレスコントローラーと適切な `IngressClass` が設定されていることを確認してください。
</Note>

## デプロイメント方法

### Recommended: W\&B Kubernetes Operator with Helm

W\&B Self-Managed の推奨インストール方法は、**W\&B Kubernetes Operator**を Helm 経由でデプロイすることです。このアプローチには以下の利点があります：

* W\&B コンポーネントの自動更新と管理
* 設定とデプロイメントの簡素化
* すべてのデプロイメントシナリオのサポート（クラウド、オンプレミス、エアギャップ）

詳細なインストール手順については、以下を参照してください：

* [W\&B Platform をオンプレミスにデプロイする](/ja/platform/hosting/self-managed/operator) - 主要インストールガイド
* [エアギャップインスタンス用の Kubernetes Operator](/ja/platform/hosting/self-managed/on-premises-deployments/kubernetes-airgapped) - 切断された環境向け

### インフラストラクチャのプロビジョニング

Terraform は W\&B 本番デプロイメントのインフラストラクチャをプロビジョニングするための推奨方法です。Terraform を使用して、必要なリソース、他のリソースへの参照、および依存関係を定義します。W\&B は主要なクラウドプロバイダー向けの Terraform モジュールを提供しています。詳細については、[Self-Managed クラウドアカウント内に W\&B Server をデプロイする](/ja/platform/hosting/hosting-options/self-managed#deploy-wb-server-within-Self-Managed-cloud-accounts)を参照してください。

## サイジング

デプロイメントを計画する際の出発点として、以下の一般的なガイドラインを使用してください。W\&B は、新しいデプロイメントのすべてのコンポーネントを綿密に監視し、観察された使用パターンに基づいて調整を行うことを推奨します。本番デプロイメントを継続的に監視し、最適なパフォーマンスを維持するために必要に応じて調整を行ってください。

キャパシティを計画する際には、W\&B Operator ワークロード用の Kubernetes クラスターと、メタデータ用の MySQL データベースという 2 つのコアコンポーネントをサイジングします。推奨事項は **環境**（テスト/開発または本番）によって異なり、Kubernetes のみの場合は **製品の組み合わせ**（Models のみ、Weave のみ、または Models と Weave）によっても異なります。W\&B は、テスト/開発および本番の両方で最低 3 つのワーカーノードから始め、本番ではクラスターオートスケーリングを有効にすることを推奨します。
