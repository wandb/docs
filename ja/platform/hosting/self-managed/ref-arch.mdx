このページでは、W&amp;B デプロイメント向けのリファレンスアーキテクチャを説明し、本番環境でのプラットフォーム運用を支える推奨インフラストラクチャとリソースの概要を示します。

選択した W&amp;B のデプロイメント環境に応じて、さまざまなサービスを利用してデプロイメントのレジリエンスを高めることができます。

たとえば、大手クラウドプロバイダーは、データベースの設定、メンテナンス、高可用性、およびレジリエンスに関する複雑さを軽減する堅牢なマネージドデータベースサービスを提供しています。

このリファレンスアーキテクチャでは、代表的なデプロイメントシナリオをいくつか取り上げ、W&amp;B デプロイメントをクラウドベンダーのサービスと統合して、パフォーマンスと信頼性を最適化する方法を示します。

<div id="before-you-start">
  ## はじめに
</div>

本番環境でアプリケーションを実行する場合には固有の課題がつきものであり、W&amp;B も例外ではありません。可能な限りプロセスを簡素化することを目指していますが、個別のアーキテクチャや設計上の判断によっては、特有の複雑さが生じる場合があります。一般的に、本番環境へのデプロイを管理するには、ハードウェア、オペレーティングシステム、ネットワーク、ストレージ、セキュリティ、W&amp;B プラットフォーム本体、その他の依存関係など、さまざまなコンポーネントを管理する必要があります。この責任は、環境の初期セットアップだけでなく、その後の継続的な運用・保守にも及びます。

W&amp;B の Self-Managed アプローチが、チームおよび特定の要件に適しているかどうかを慎重に検討してください。

Self-Managed W&amp;B をデプロイするための重要な前提条件として、本番運用レベルのアプリケーションを実行・維持する方法について十分に理解していることが求められます。チームに支援が必要な場合は、W&amp;B Professional Services チームおよびパートナーが、導入と最適化に向けたサポートを提供します。

W&amp;B を自前で管理する代わりに、マネージドな形で利用する方法について詳しく知りたい場合は、[W&amp;B Multi-tenant Cloud](/ja/platform/hosting/hosting-options/multi_tenant_cloud) および [W&amp;B Dedicated Cloud](/ja/platform/hosting/hosting-options/dedicated_cloud) を参照してください。

<div id="application-layer">
  ### アプリケーションレイヤー
</div>

アプリケーションレイヤーは複数ノードで構成された Kubernetes クラスターで構成されており、ノード障害に対する耐障害性を備えています。Kubernetes クラスターは W&amp;B の Pod を実行し、管理します。

<div id="storage-layer">
  ### ストレージレイヤー
</div>

ストレージレイヤーは、MySQL データベースとオブジェクトストレージで構成されています。MySQL データベースにはメタデータが保存され、オブジェクトストレージには モデル や データセット などの Artifacts が保存されます。

<div id="infrastructure-requirements">
  ## インフラストラクチャ要件
</div>

### Kubernetes

W&amp;B Server アプリケーションは、複数の pod をデプロイする [Kubernetes Operator](/ja/platform/hosting/operator/) として稼働します。このため、W&amp;B には次の要件を満たす Kubernetes クラスターが必要です。

* 完全に設定され、正常に動作している Ingress コントローラー。
* Persistent Volume をプロビジョニングできる機能。

W&amp;B は、クラウド、オンプレミス、およびエアギャップ環境における [OpenShift Kubernetes クラスター](https://www.redhat.com/en/technologies/cloud-computing/openshift) 上でのデプロイをサポートします。具体的な設定手順については、Operator ガイドの [OpenShift セクション](/ja/platform/hosting/operator/#openshift-kubernetes-clusters) を参照してください。

### MySQL

W&amp;B はメタデータを MySQL データベースに保存します。データベースのパフォーマンスとストレージ要件は、モデル パラメーターや関連メタデータの構造やサイズに依存します。たとえば、より多くのトレーニング run をトラッキングするとデータベースのサイズが増加し、run テーブル、ユーザー ワークスペース、およびレポートに対するクエリが増えると、それに応じてデータベースへの負荷も増加します。

**W&amp;B は本番環境でのデプロイにはマネージド データベース サービスの利用を強く推奨します**（AWS RDS Aurora MySQL、Google Cloud SQL for MySQL、Azure Database for MySQL など）。マネージド サービスは、自動バックアップ、モニタリング、高可用性、パッチ適用などを提供し、運用の複雑さを大幅に軽減します。具体的なサービス推奨については、以下の [Cloud provider instance recommendations](#cloud-provider-instance-recommendations) セクションを参照してください。

セルフマネージドの MySQL データベースをデプロイする場合は、次の点を検討してください。

* **バックアップ**: データベースは定期的に別の環境にバックアップする必要があります。W&amp;B は、少なくとも 1 週間分の保持期間を設けた、毎日のバックアップを推奨します。
* **パフォーマンス**: データベースには SSD や高速 NAS など、高速なストレージ ハードウェアが必要です。
* **モニタリング**: データベースには十分な CPU リソースが必要です。データベース サーバーの CPU 負荷をモニタリングしてください。CPU 使用率がシステム全体の 90% を超える状態が 5 分以上継続する場合は、CPU キャパシティの追加を検討してください。
* **可用性**: 可用性と耐久性の要件を満たすために、W&amp;B は、すべての更新をプライマリ デプロイメントからリアルタイムでストリーミングし、プライマリ サーバーがクラッシュしたり、破損したり、長時間のダウンタイムが発生した場合にフェイルオーバーできるよう、別マシン上にホット スタンバイ Server デプロイメントを構成することを推奨します。なお、W&amp;B はマルチマスター トポロジーや読み取り専用レプリカをサポートしていません。

  #### MySQL データベースの作成
</div>

MySQL データベースとユーザーを手動で作成する方法については、[ベアメタルガイドの MySQL データベース セクション](/ja/platform/hosting/self-managed/bare-metal/#mysql-database)を参照してください。

<div id="mysql-configuration-parameters">
  #### MySQL の設定パラメーター
</div>

独自の MySQL インスタンスを実行している場合は、MySQL を次の設定で構成してください。

```
binlog_format = 'ROW'
binlog_row_image = 'MINIMAL'
innodb_flush_log_at_trx_commit = 1
innodb_online_alter_log_max_size = 268435456
max_prepared_stmt_count = 1048576
sort_buffer_size = '67108864'
sync_binlog = 1
```

これらの設定は、W&amp;B により最適なパフォーマンスと信頼性が得られることを確認済みです。

<div id="redis">
  ### Redis
</div>

W&amp;B は、ジョブキューイングとデータキャッシュのために W&amp;B の各コンポーネントが利用する、Redis 7.x の単一ノード デプロイメントに依存します。概念実証のテストおよび開発を容易にするために、W&amp;B Self-Managed にはローカルの Redis デプロイメントが含まれていますが、これは本番環境での利用には適していません。

W&amp;B は、次の環境にある Redis インスタンスに接続できます。

* [AWS Elasticache](https://aws.amazon.com/elasticache/)
* [Google Cloud Memory Store](https://cloud.google.com/memorystore?hl=en)
* [Azure Cache for Redis](https://azure.microsoft.com/en-us/products/cache)
* お使いのクラウド環境またはオンプレミス インフラストラクチャでホストされている Redis デプロイメント

<div id="object-storage">
  ### オブジェクトストレージ
</div>

W&amp;B には、事前署名付き URL と CORS をサポートするオブジェクトストレージが必要で、次のいずれかとしてデプロイされている必要があります。

* [CoreWeave AI Object Storage](https://docs.coreweave.com/products/storage/object-storage) は、AI ワークロード向けに最適化された高性能な S3 互換オブジェクトストレージサービスです。
* [Amazon S3](https://aws.amazon.com/s3/) は、業界最高水準のスケーラビリティ、データ可用性、セキュリティ、およびパフォーマンスを提供するオブジェクトストレージサービスです。
* [Google Cloud Storage](https://cloud.google.com/storage) は、非構造化データを大規模に保存するためのマネージドサービスです。
* [Azure Blob Storage](https://azure.microsoft.com/en-us/products/storage/blobs) は、テキスト、バイナリデータ、画像、動画、ログなどの大量の非構造化データを保存するためのクラウドベースのオブジェクトストレージソリューションです。
* S3 互換ストレージとしての [MinIO Enterprise (AIStor)](https://www.min.io/product/aistor) や [NetApp StorageGRID](https://www.netapp.com/data-storage/storagegrid/)、あるいはお使いのクラウド環境またはオンプレミス インフラストラクチャ上でホストされるその他のエンタープライズグレードのソリューション。

<div id="load-balancer-and-ingress">
  ### ロードバランサーと ingress
</div>

W&amp;B Kubernetes Operator は、Kubernetes ingress controller を使用してサービスを公開できます。ingress controller は、URL パスおよび異なるポートに基づいてサービスエンドポイントへルーティングします。ingress controller には、機械学習のペイロードを実行するすべてのマシン、および Web ブラウザー経由でサービスにアクセスするすべてのマシンからアクセス可能である必要があります。

<div id="ingress-controller-requirements">
  #### Ingress controller の要件
</div>

Kubernetes クラスターには利用可能な `IngressClass` が必要です。一般的な Ingress controller のオプションとしては次のものがあります。

* [Nginx Ingress Controller](https://kubernetes.github.io/ingress-nginx/)
* [Istio](https://istio.io)
* [Traefik](https://traefik.io/)
* クラウドプロバイダーの Ingress controller（AWS ALB、GCP Ingress、Azure Application Gateway）

<div id="wb-service-routing">
  #### W&amp;B サービスのルーティング
</div>

W&amp;B Operator は、パスに基づいて複数のバックエンド サービスにリクエストを自動的にルーティングします。

| パス | サービス | デフォルト ポート | 用途 |
|------|---------|------|---------|
| `/` | `wandb-app` | 8080 | メインの Web アプリケーション UI |
| `/api` | `wandb-api` | 8081 | API サービス |
| `/graphql` | `wandb-api` | 8081 | GraphQL API エンドポイント |
| `/graphql2` | `wandb-api` | 8081 | GraphQL API v2 エンドポイント |
| `/console` | `wandb-console` | 8082 | システム コンソール |
| `/traces` | `wandb-weave-trace` | 8722 | Weave トレーシング サービス（有効化されている場合） |

<div id="example-ingress-configuration">
  #### ingress 設定の例
</div>

以下は、W&amp;B Operator によって作成される ingress リソースの例です。

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: wandb
  namespace: wandb
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: "0"
spec:
  ingressClassName: nginx
  rules:
  - host: wandb.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: wandb-app
            port:
              number: 8080
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: wandb-api
            port:
              number: 8081
      - path: /graphql
        pathType: Prefix
        backend:
          service:
            name: wandb-api
            port:
              number: 8081
      - path: /graphql2
        pathType: Prefix
        backend:
          service:
            name: wandb-api
            port:
              number: 8081
      - path: /console
        pathType: Prefix
        backend:
          service:
            name: wandb-console
            port:
              number: 8082
  tls:
  - hosts:
    - wandb.example.com
    secretName: wandb-tls
```

<Note>
  W&amp;B Operator は ingress の設定を自動的に作成して管理します。通常は手動で ingress リソースを作成する必要はありません。クラスターに正常に動作する ingress コントローラーがあり、適切な `IngressClass` が設定されていることを確認してください。
</Note>

<div id="recommended-wb-kubernetes-operator-with-helm">
  ### 推奨: W&amp;B Kubernetes Operator と Helm
</div>

W&amp;B Self-Managed の推奨インストール方法は、Helm を用いてデプロイする **W&amp;B Kubernetes Operator** を使用する方法です。この方法には次の利点があります。

* W&amp;B コンポーネントの自動アップデートと管理
* 設定とデプロイの簡素化
* すべてのデプロイシナリオ（クラウド、オンプレミス、エアギャップ）のサポート

詳細なインストール手順については、次を参照してください。

* [Run W&amp;B Server on Kubernetes (Operator)](/ja/platform/hosting/operator/) - メインのインストールガイド
* [Deploy W&amp;B Platform On-premises](/ja/platform/hosting/self-managed/bare-metal/) - データセンター／ベアメタル環境に特有の考慮事項向け
* [Kubernetes operator for air-gapped instances](/ja/platform/hosting/self-managed/operator-airgapped/) - インターネットから分離された環境向け

<div id="infrastructure-provisioning">
  ### インフラプロビジョニング
</div>

W&amp;B の本番デプロイメント用インフラのプロビジョニングには、Terraform の利用を推奨します。Terraform を使用すると、必要なリソース、それらと他のリソースとの参照関係、および依存関係を定義できます。W&amp;B は主要なクラウドプロバイダー向けに Terraform モジュールを提供しています。詳細については、[セルフマネージド クラウドアカウント内に W&amp;B Server をデプロイする](/ja/platform/hosting/hosting-options/self-managed#deploy-wb-server-within-Self-Managed-cloud-accounts) を参照してください。
