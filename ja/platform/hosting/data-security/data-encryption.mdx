---
title: Dedicated Cloud におけるデータ暗号化
description: "W&B がクラウドネイティブキーと customer-managed encryption key (CMEK) ポリシーを使用して、Dedicated Cloud 内のデータをどのように暗号化するかを説明します。"
---

W&amp;B は、各クラウドの customer-managed encryption key (CMEK) 機能を利用して、すべての [Dedicated Cloud](/ja/platform/hosting/hosting-options/dedicated-cloud) で W&amp;B が管理するデータベースおよびオブジェクトストレージを、W&amp;B 管理のクラウドネイティブキーで暗号化します。この場合、W&amp;B はクラウドプロバイダーの `customer` として行動しつつ、W&amp;B プラットフォームをサービスとしてお客様に提供します。W&amp;B 管理のキーを使用することで、各クラウドでデータの暗号化に使用するキーを W&amp;B 自身が制御できます。これにより、すべてのお客様に対して高い安全性とセキュリティを備えたプラットフォームを提供するという W&amp;B の取り組みがさらに強化されます。

W&amp;B は、各顧客インスタンス内のデータを暗号化するために `unique key` を使用し、Dedicated Cloud テナント間の分離をさらに強化しています。この機能は AWS、Azure、Google Cloud で利用できます。

<Note>
  Dedicated Cloud インスタンスのうち、2024 年 8 月以前に W&amp;B が Google Cloud および Azure 上へプロビジョニングしたものでは、W&amp;B 管理のデータベースおよびオブジェクトストレージの暗号化に、クラウドプロバイダー管理のデフォルトキーを使用しています。2024 年 8 月以降に W&amp;B が新規作成するインスタンスのみ、該当する暗号化に W&amp;B 管理のクラウドネイティブキーを使用しています。

  AWS 上の Dedicated Cloud インスタンスでは、2024 年 8 月以前から W&amp;B 管理のクラウドネイティブキーを暗号化に使用しています。
</Note>

W&amp;B は通常、お客様が独自のクラウドネイティブキーを持ち込み、その Dedicated Cloud インスタンス内にある W&amp;B 管理のデータベースおよびオブジェクトストレージを暗号化することを許可していません。これは、組織内の複数のチームや担当者が、さまざまな理由でそのクラウドインフラストラクチャーにアクセスできる可能性があるためです。それらのチームや担当者の中には、組織のテクノロジースタックにおける重要なコンポーネントとしての W&amp;B の位置づけを十分に把握していない人がいる可能性があり、その結果、クラウドネイティブキーを完全に削除したり、W&amp;B のアクセス権を取り消したりするおそれがあります。そのような操作により、組織の W&amp;B インスタンス内のすべてのデータが破損し、復旧不能な状態になる可能性があります。

組織で AI ワークフローに Dedicated Cloud を使用する承認を得るために、独自のクラウドネイティブキーを使用して W&amp;B 管理のデータベースおよびオブジェクトストレージを暗号化する必要がある場合は、W&amp;B が例外的に個別審査することがあります。承認された場合、暗号化におけるお客様のクラウドネイティブキーの使用は、W&amp;B Dedicated Cloud の `shared responsibility model` に従うことになります。Dedicated Cloud インスタンスの稼働中に、組織内のいずれかのユーザーがお客様のキーを削除したり、そのキーに対する W&amp;B のアクセス権を取り消したりした場合、W&amp;B はその結果生じるデータ損失や破損について責任を負わず、そのようなデータの復旧責任も負いません。
