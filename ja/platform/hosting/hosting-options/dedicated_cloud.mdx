W&amp;B Dedicated Cloud は、W&amp;B の AWS、Google Cloud、または Azure のクラウド アカウント上にデプロイされる、シングルテナント構成のフルマネージド プラットフォームです。各 Dedicated Cloud インスタンスは、他の W&amp;B Dedicated Cloud インスタンスとは分離されたネットワーク、コンピュートリソース、およびストレージを持ちます。W&amp;B 専用のメタデータおよびデータは、分離されたクラウド ストレージに保存され、分離されたクラウド コンピュート サービスを用いて処理されます。

W&amp;B Dedicated Cloud は、[各クラウドプロバイダーごとに複数のグローバル リージョン](./dedicated_regions) で利用可能です。

<div id="compliance">
  ## コンプライアンス
</div>

* **SOC 2**: W&amp;B Dedicated Cloud のホスティングプラットフォームは、[Service and Organization Controls (SOC) 2 Type 2](https://www.aicpa-cima.com/topic/audit-assurance/audit-and-assurance-greater-than-soc-2) の要件を満たしており、これは [Auditing Standards Board of the American Institute of Certified Public Accountants (AICPA)](https://www.aicpa-cima.com/home) によって公開されています。SOC 2 レポートは、サービス組織のセキュリティ、可用性、処理の完全性、機密性、およびプライバシーに関する管理策を評価するものです。W&amp;B Dedicated Cloud は、コンプライアンスが継続していることを検証するため、定期的な社内および社外監査の対象となります。SOC 2 レポートおよびその他のセキュリティおよびコンプライアンス文書の入手については、[W&amp;B Security Portal](https://security.wandb.ai/) をご覧ください。
* **HIPAA**: 適切に設定された場合、W&amp;B Dedicated Cloud は、[Health Insurance Portability and Accountability Act of 1996 (HIPAA)](https://www.hhs.gov/hipaa/for-professionals/index.html) の要件を満たします。HIPAA 準拠は、W&amp;B、顧客、およびデプロイメントに関与するあらゆるサードパーティサービスが負う共同責任となります。HIPAA の適用を受ける組織は、W&amp;B との間で **Business Associate Agreement** を締結している必要があります。詳細情報については、[W&amp;B Security Portal](https://security.wandb.ai/) をご覧ください。

<div id="data-security">
  ## データセキュリティ
</div>

[secure storage connector](/ja/platform/hosting/data-security/secure-storage-connector) を使用して、[インスタンス レベルおよびチーム レベル](/ja/platform/hosting/data-security/secure-storage-connector#configuration-options) で BYOB (Bring Your Own Bucket) として、モデル、データセット などのファイルを保存できます。

W&amp;B Multi-tenant Cloud と同様に、複数の Teams 向けに 1 つのバケットを設定することも、Teams ごとに別々のバケットを使用することもできます。特定の Team に対して secure storage connector を設定しなかった場合、そのデータはインスタンス レベルのバケットに保存されます。

<Frame>
  <img src="/images/hosting/dedicated_cloud_arch.png" alt="Dedicated Cloud architecture diagram" />
</Frame>

secure storage connector を用いた BYOB に加えて、[IP allowlisting](/ja/platform/hosting/data-security/ip-allowlisting) を使用することで、信頼されたネットワーク ロケーションからの Dedicated Cloud インスタンスへのアクセスにのみ制限できます。

[クラウド プロバイダーのセキュアな接続ソリューション](/ja/platform/hosting/data-security/private-connectivity) を使用して、Dedicated Cloud インスタンスにプライベート接続することができます。

お客様は、ご自身のデプロイメントが所属組織のポリシーおよび、該当する場合は [Security Technical Implementation Guidelines (STIG)](https://en.wikipedia.org/wiki/Security_Technical_Implementation_Guide) に準拠していることを確保する責任を負います。

<div id="identity-and-access-management-iam">
  ## Identity and access management (IAM)
</div>

W&amp;B Organization 内で安全な認証と効果的な認可を実現するために、Identity and access management (IAM) 機能を使用します。Dedicated Cloud インスタンスで利用できる IAM 機能は次のとおりです。

* [SSO using OpenID Connect (OIDC)](/ja/platform/hosting/iam/sso) または [LDAP](/ja/platform/hosting/iam/ldap) で認証します。
* 組織レベルおよびチーム内のスコープで、[適切なユーザー ロールを設定](/ja/platform/hosting/iam/access-management/manage-organization#assign-or-update-a-users-role) します。
* [restricted projects](/ja/platform/hosting/iam/access-management/restricted-projects) を使用して W&amp;B プロジェクトのスコープを定義し、そのプロジェクトを閲覧・編集し、W&amp;B run を送信できるユーザーを制限します。
* [identity federation](/ja/platform/hosting/iam/identity_federation) と JSON Web Token を活用して W&amp;B API にアクセスします。

<div id="monitor">
  ## 監視
</div>

[Audit logs](/ja/platform/hosting/monitoring-usage/audit-logging) を使用して Teams 内の ユーザー のアクティビティを追跡し、エンタープライズ向けガバナンス要件への準拠に役立てます。また、Dedicated Cloud インスタンスでは [W&amp;B Organization Dashboard](/ja/platform/hosting/monitoring-usage/org_dashboard) から組織の利用状況を確認できます。

<div id="maintenance">
  ## メンテナンス
</div>

W&amp;B Multi-tenant Cloud と同様に、Dedicated Cloud では W&amp;B プラットフォームのプロビジョニングおよび保守に伴う運用負荷やコストは発生しません。

Dedicated Cloud における W&amp;B のアップデートの管理方法については、[サーバーリリースプロセス](/ja/platform/hosting/server-upgrade-process) を参照してください。

<div id="compliance">
  ## コンプライアンス
</div>

W&amp;B Dedicated Cloud のセキュリティコントロールは、社内および第三者による定期的な監査を受けています。製品評価のために必要なセキュリティおよびコンプライアンス関連文書は、[W&amp;B Security Portal](https://security.wandb.ai/) からリクエストしてください。

<div id="data-retention-policy">
  ## データ保持ポリシー
</div>

既定では、Dedicated Cloud インスタンスは、削除後 7 日間、次の項目を保持します。

* Runs と履歴
* メディア、設定ファイル、ログファイルなどの、 Artifacts 以外の run ファイル
* Artifacts と Artifacts 参照

この期間が経過するまでは、これらの項目を復元できます。復元が必要な場合は、[support](mailto:support@wandb.ai) か担当の AISE までお問い合わせください。

データ保持要件を満たすために、Dedicated Cloud インスタンスのデータ保持期間を変更できます。ユースケースに応じて、詳細は **Environment variable** タブまたは **Helm** タブのいずれかを選択してください。

<Tabs>
  <Tab title="Environment variable">
    データ保持ポリシーを変更するには、環境変数 `GORILLA_DATA_RETENTION_PERIOD` を保持する時間数で設定します。たとえば、削除済みデータを 14 日間 (336 時間) 保持するには、次のようにします。

    ```bash
    export GORILLA_DATA_RETENTION_PERIOD="336h"
    ```
  </Tab>

  <Tab title="Helm">
    データ保持ポリシーを変更するには、Helm の値 `env.dataRetentionPeriod` を、保持する時間数を指定して設定します。たとえば、削除済みデータを 14 日間 (336 時間) 保持するには、次のようにします。

    ```helm
    env: dataRetentionPeriod: "336h"
    ```
  </Tab>
</Tabs>
