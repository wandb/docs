---
title: Launch キューを監視する
---

対話型の **Queue monitoring ダッシュボード** を使うと、Launch キューが高負荷状態かアイドル状態かを確認したり、実行中のワークロードを可視化したり、非効率なジョブを特定したりできます。Launch キューのダッシュボードは、計算用ハードウェアやクラウドリソースを有効に活用できているかどうかを判断するのに特に役立ちます。

より詳細な分析のために、このページから W&B の experiment tracking Workspace や Datadog、NVIDIA Base Command、各種クラウドコンソールなどの外部インフラ監視サービスへアクセスできます。

<Note>
Queue monitoring ダッシュボードは現在、W&B Multi-tenant Cloud デプロイオプションでのみ利用できます。
</Note>

<div id="dashboard-and-plots">
  ## ダッシュボードとプロット
</div>

過去 7 日間に行われたキューのアクティビティを表示するには、**Monitor** タブを使用します。左側のパネルで、時間範囲、グルーピング、フィルタを設定します。

ダッシュボードには、パフォーマンスと効率に関する一般的な問いに答える複数のプロットが含まれています。以降のセクションでは、キューのダッシュボードにおける UI 要素について説明します。

<div id="job-status">
  ### ジョブステータス
</div>

**Job status** プロットでは、各時間間隔ごとに、実行中・保留中・キュー待ち・完了しているジョブの数を表示します。キューがアイドル状態になっている期間を特定するには、この **Job status** プロットを使用します。

<Frame>
  <img src="/images/launch/launch_obs_jobstatus.png" alt="ジョブステータスのタイムライン" />
</Frame>

たとえば、固定リソース（DGX BasePod など）があるとします。その固定リソースに対するキューがアイドル状態になっている場合は、Sweeps などの優先度の低いプリエンプティブルな Launch ジョブを実行する好機であることが示唆されます。

一方、クラウドリソースを使用していて、周期的にアクティビティが急増していることがわかる場合もあります。アクティビティの周期的な急増は、特定の時間帯にリソースを予約することでコストを節約できる可能性を示唆します。

プロットの右側には、どの色が[Launch ジョブのステータス](./launch-view-jobs#check-the-status-of-a-job)を表しているかを示す凡例があります。

<Note>
  `Queued` の項目は、ワークロードを他のキューに移す機会を示している可能性があります。失敗の急増は、Launch ジョブのセットアップについて支援が必要なユーザーを特定するのに役立ちます。
</Note>

<div id="queued-time">
  ### キュー待機時間
</div>

**Queued time** プロットは、指定した日付または時間範囲において、Launch ジョブがキューで待機していた時間（秒）を表示します。 

<Frame>
    <img src="/images/launch/launch_obs_queuedtime.png" alt="Queued time メトリクス"  />
</Frame>

x 軸には指定した時間範囲が表示され、y 軸には Launch ジョブが Launch キューで待機していた時間（秒）が表示されます。たとえば、ある日に 10 個の Launch ジョブがキューに投入されているとします。これら 10 個の Launch ジョブがそれぞれ平均 60 秒待機していた場合、**Queued time** プロットには 600 秒と表示されます。

<Note>
**Queued time** プロットを使用して、長いキュー待機時間の影響を受けているユーザーを特定します。
</Note>

左側のバーにある `Grouping` コントロールを使用して、ジョブごとの色をカスタマイズします。

これは特に、どのユーザーやジョブがキュー容量の不足による影響を強く受けているかを特定するのに役立ちます。

<div id="job-runs">
  ### ジョブの run
</div>

<Frame>
    <img src="/images/launch/launch_obs_jobruns2.png" alt="ジョブ run のタイムライン"  />
</Frame>

このプロットは、ある期間内に実行されたすべてのジョブの開始と終了を、各 run ごとに異なる色で表示します。これにより、特定の時点でキューがどのようなワークロードを処理していたかを一目で把握できます。  

パネル右下の Select ツールを使ってジョブをドラッグして選択し、下のテーブルに詳細情報を表示します。

<div id="cpu-and-gpu-usage">
  ### CPU と GPU の使用状況
</div>

**GPU use by a job**、**CPU use by a job**、**GPU memory by job**、**System memory by job** を使用して、Launch ジョブの効率を確認します。 

<Frame>
    <img src="/images/launch/launch_obs_gpu.png" alt="GPU usage metrics"  />
</Frame>

たとえば、**GPU memory by job** を使用して、ある W&B run の完了に長い時間がかかったかどうか、また CPU コアの使用率が低かったかどうかを確認できます。

各プロットの x 軸には、Launch ジョブによって作成された W&B run の実行時間が秒単位で表示されます。データポイント上にマウスカーソルを置くと、run ID、run が属しているプロジェクト、その W&B run を作成した Launch ジョブなど、W&B run に関する情報が表示されます。

<div id="errors">
  ### エラー
</div>

**Errors** パネルには、特定の Launch キューで発生したエラーが表示されます。より具体的には、エラーが発生したタイムスタンプ、エラー元の Launch ジョブ名、生成されたエラーメッセージが表示されます。デフォルトでは、エラーは新しいものから古いものの順に並びます。

<Frame>
    <img src="/images/launch/launch_obs_errors.png" alt="Error logs panel"  />
</Frame>

**Errors** パネルを使用して、影響を受けているユーザーを特定し、ブロックを解除します。 

<div id="external-links">
  ## 外部リンク
</div>

キューの可観測性ダッシュボードのビューはすべてのキュータイプで共通ですが、多くの場合、環境固有のモニターに直接ジャンプできると便利です。これを行うには、キューの可観測性ダッシュボードからコンソール上の環境固有モニターへのリンクを追加します。

ページの下部にある `Manage Links` をクリックしてパネルを開きます。目的のページの完全な URL を追加します。続いて、ラベルを追加します。追加したリンクは **External Links** セクションに表示されます。
