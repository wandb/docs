---
title: Launch キューをモニタリングする
description: "インタラクティブなダッシュボードで Launch キューのワークロードをモニタリングし、非効率なジョブを特定して、リソース使用状況を分析します。"
---

インタラクティブな **Queue monitoring ダッシュボード** を使用して、Launch キューが高負荷で使われているのかアイドル状態なのかを確認し、実行中のワークロードを可視化して、非効率なジョブを特定できます。Launch キューのダッシュボードは、計算用ハードウェアやクラウドリソースを有効に活用できているかどうかを判断する際に特に有用です。

さらに詳しく分析するには、このページから W&amp;B の実験管理 Workspace や Datadog、NVIDIA Base Command、各種クラウドコンソールなどの外部インフラストラクチャ監視プロバイダーにアクセスできます。

<Note>
  Queue monitoring ダッシュボードは現在、W&amp;B Multi-tenant Cloud デプロイ オプションでのみ利用可能です。
</Note>

<div id="dashboard-and-plots">
  ## ダッシュボードとプロット
</div>

**Monitor** タブを使用して、過去 7 日間に発生したキューのアクティビティを表示します。左側のパネルを使って、時間範囲、グループ化、およびフィルターを設定できます。

ダッシュボードには、パフォーマンスと効率に関するよくある疑問に答える複数のプロットが含まれています。以降のセクションでは、キューのダッシュボードにおける UI 要素について説明します。

<div id="job-status">
  ### ジョブステータス
</div>

**Job status** プロットは、各時間間隔において実行中、保留、キュー待ち、完了済みのジョブ数を表示します。キューが遊休状態になっている期間を特定するには、**Job status** プロットを使用します。

<Frame>
  <img src="/images/launch/launch_obs_jobstatus.png" alt="ジョブステータスのタイムライン" />
</Frame>

たとえば、DGX BasePod などの固定リソースを保有しているとします。固定リソースに対してキューが遊休状態になっているのを観測した場合、sweep などの優先度の低いプリエンプト可能な Launch ジョブを実行できる可能性を示唆しています。

一方で、クラウド リソースを使用していて、周期的にアクティビティが急増するのが見られる場合、その周期的なバーストは、特定の時間帯にリソースを予約することでコストを節約できる可能性を示しているかもしれません。

プロットの右側には、どの色が [Launch ジョブのステータス](./launch-view-jobs#check-the-status-of-a-job)を表しているかを示す凡例があります。

<Note>
  `Queued` の項目は、ワークロードを別のキューに移す機会を示している可能性があります。失敗数のスパイクは、Launch ジョブのセットアップについて支援が必要かもしれないユーザーを特定するのに役立ちます。
</Note>

<div id="queued-time">
  ### Queued time
</div>

**Queued time** プロットは、指定した日付または期間において、Launch ジョブがキューに入っていた時間（秒単位）を表示します。 

<Frame>
    <img src="/images/launch/launch_obs_queuedtime.png" alt="Queued time メトリクス"  />
</Frame>

x 軸には指定した時間枠が、y 軸には Launch ジョブが Launch キュー上で待機していた時間（秒単位）が表示されます。たとえば、ある日に 10 個の Launch ジョブがキューに入っているとします。**Queue time** プロットには、それら 10 個の Launch ジョブがそれぞれ平均 60 秒待機している場合、合計 600 秒と表示されます。

<Note>
キュー時間の長さによる影響を受けているユーザーを特定するには、**Queued time** プロットを使用します。
</Note>

左側のバーにある `Grouping` コントロールを使って、それぞれのジョブの色をカスタマイズできます。

これは、どのユーザーとジョブがキュー容量不足の影響を強く受けているかを特定するのに特に役立ちます。

<div id="job-runs">
  ### ジョブ実行
</div>

<Frame>
    <img src="/images/launch/launch_obs_jobruns2.png" alt="Job runs timeline"  />
</Frame>

このグラフでは、ある期間に実行されたすべてのジョブの開始時刻と終了時刻が、それぞれの実行ごとに異なる色で表示されます。これにより、特定の時点でキューがどのようなワークロードを処理していたかを一目で把握できます。  

パネル右下にある Select ツールでジョブ上をドラッグして選択し、下のテーブルに詳細を表示します。

<div id="cpu-and-gpu-usage">
  ### CPU と GPU の使用状況
</div>

**GPU use by a job**、**CPU use by a job**、**GPU memory by job**、**System memory by job** を使用して、Launch ジョブの効率を把握します。 

<Frame>
    <img src="/images/launch/launch_obs_gpu.png" alt="GPU usage metrics"  />
</Frame>

たとえば、**GPU memory by job** を使用して、ある W&B run の完了までに長時間かかったかどうか、また CPU コアの使用率が低かったかどうかを確認できます。

各プロットの x 軸には、Launch ジョブによって作成された W&B run の継続時間（秒）が表示されます。データポイント上にマウスカーソルを載せると、run ID、その run が属するプロジェクト、W&B run を作成した Launch ジョブなど、W&B run に関する情報を確認できます。

<div id="errors">
  ### エラー
</div>

**Errors** パネルには、指定した Launch キューで発生したエラーが表示されます。より具体的には、Errors パネルには、エラーが発生した時刻のタイムスタンプ、そのエラーが発生した Launch ジョブの名前、および生成されたエラーメッセージが表示されます。デフォルトでは、エラーは新しいものから古いものへと並びます。

<Frame>
    <img src="/images/launch/launch_obs_errors.png" alt="エラーログパネル"  />
</Frame>

**Errors** パネルを使用して、ユーザーの問題を特定して解消します。 

<div id="external-links">
  ## 外部リンク
</div>

queue observability ダッシュボードの表示はすべてのキュータイプで一貫していますが、多くの場合、環境ごとのモニターに直接移動できると便利です。これを実現するには、queue observability ダッシュボードから console へのリンクを追加します。

ページ下部の `Manage Links` をクリックしてパネルを開きます。目的のページのフル URL を追加します。次に、ラベルを追加します。追加したリンクは **External Links** セクションに表示されます。
