---
title: Launch queueの監視
description: "インタラクティブなダッシュボードでLaunch queueのワークロードを監視し、非効率なジョブを特定してリソース使用状況を分析します。"
---

インタラクティブな **Queue monitoring ダッシュボード** を使用すると、Launch queueが高負荷状態またはアイドル状態にある時間帯の確認、実行中のワークロードの可視化、非効率なジョブの特定が可能です。Launch queueのダッシュボードは、コンピューティングハードウェアやクラウドリソースを効果的に活用できているかどうかを判断する際に特に役立ちます。

より詳細な分析のために、このページからW&amp;Bの実験管理Workspaceや、Datadog、NVIDIA Base Command、クラウドコンソールなどの外部インフラストラクチャー監視プロバイダーへのリンクが提供されています。

<Note>
  Queue monitoring ダッシュボードは現在、W&amp;B Multi-tenant Cloudデプロイオプションでのみ使用可能です。
</Note>

<div id="dashboard-and-plots">
  ## ダッシュボードとプロット
</div>

**Monitor** タブを使用して、過去7日間に発生したキューのアクティビティを確認できます。左パネルを使用して、時間範囲、グループ化、フィルターを制御します。

ダッシュボードには、パフォーマンスと効率に関するよくある質問に答えるいくつかのプロットが含まれています。以降のセクションでは、キューダッシュボードのUI要素について説明します。

<div id="job-status">
  ### ジョブのステータス
</div>

**Job status** プロットは、各時間間隔においてrunning、pending、queued、または完了しているジョブの数を示します。キューのアイドル期間を特定するには、**Job status** プロットを使用してください。

<Frame>
  <img src="/images/launch/launch_obs_jobstatus.png" alt="Job status timeline" />
</Frame>

たとえば、固定リソース (DGX BasePod など) がある場合を考えてみましょう。固定リソースでキューがアイドル状態になっているのが確認できた場合、sweeps などの優先度の低いプリエンプティブな launch ジョブを実行する機会があることを示している可能性があります。

一方、クラウドリソースを使用していて、定期的なアクティビティのバーストが見られる場合は、特定の時間帯にリソースを予約することでコストを削減できる機会があることを示している可能性があります。

プロットの右側には、各色が [launch ジョブのステータス](./launch-view-jobs#check-the-status-of-a-job) を表すキーが表示されます。

<Note>
  `Queued` のアイテムは、ワークロードを他のキューにシフトする機会があることを示している場合があります。失敗のスパイクは、launch ジョブの設定についてサポートが必要なユーザーを特定するのに役立ちます。
</Note>

{/* 範囲を選択すると下のプロットに詳細が表示されます。ページ全体にフィルターを適用するにはズームしてください。 */}


<div id="queued-time">
  ### キュー待機時間
</div>

**Queued time** プロットは、指定した日付または時間範囲において、Launch ジョブがキューに入っていた時間 (秒単位) を表示します。

<Frame>
  <img src="/images/launch/launch_obs_queuedtime.png" alt="Queued time metrics" />
</Frame>

X 軸には指定した時間範囲が表示され、Y 軸には Launch ジョブが Launch queue に入っていた時間 (秒単位) が表示されます。たとえば、ある日に 10 件の Launch ジョブがキューに入っているとします。それらの 10 件の Launch ジョブがそれぞれ平均 60 秒待機した場合、**Queue time** プロットには 600 秒と表示されます。

<Note>
  **Queued time** プロットを使用して、キュー待機時間が長いことで影響を受けているユーザーを特定してください。
</Note>

左バーの `Grouping` コントロールを使用して、各ジョブの色をカスタマイズできます。

これは、キューのリソース不足によって影響を受けているユーザーやジョブを特定する際に特に役立ちます。

<div id="job-runs">
  ### ジョブ run
</div>

<Frame>
  <img src="/images/launch/launch_obs_jobruns2.png" alt="Job runs timeline" />
</Frame>

このプロットは、特定の期間内に実行されたすべてのジョブの開始と終了を、run ごとに異なる色で表示します。これにより、特定の時点でキューがどのようなワークロードを処理していたかを一目で確認できます。

パネル右下の Select ツールを使用して、ジョブをブラシ選択すると、下の表に詳細が表示されます。

<div id="cpu-and-gpu-usage">
  ### CPUとGPUの使用状況
</div>

**GPU use by a job**、**CPU use by a job**、**GPU memory by job**、**System memory by job** を使用して、Launch ジョブの効率を確認できます。

<Frame>
  <img src="/images/launch/launch_obs_gpu.png" alt="GPU usage metrics" />
</Frame>

たとえば、**GPU memory by job** を使用して、W&amp;B の run が完了するまでに時間がかかったかどうか、またCPUコアの使用率が低かったかどうかを確認できます。

各プロットのx軸は、W&amp;B の run (Launch ジョブによって作成) の実行時間 (秒単位) を示しています。データポイントにマウスをホバーすると、run ID、その run が属するプロジェクト、W&amp;B の run を作成した Launch ジョブなど、W&amp;B の run に関する情報を確認できます。

<div id="errors">
  ### Errors
</div>

**Errors** パネルには、特定の Launch queue で発生したエラーが表示されます。具体的には、エラーが発生した日時のタイムスタンプ、エラーの発生元となった Launch ジョブの名前、および生成されたエラーメッセージが表示されます。デフォルトでは、エラーは新しい順に並べられます。

<Frame>
  <img src="/images/launch/launch_obs_errors.png" alt="Error logs panel" />
</Frame>

**Errors** パネルを使用して、問題を特定し、ユーザーのブロックを解除してください。 

<div id="external-links">
  ## 外部リンク
</div>

キューの可観測性ダッシュボードのビューはすべてのキュータイプで一貫していますが、多くの場合、環境固有のモニターに直接アクセスできると便利です。これを実現するには、キューの可観測性ダッシュボードからコンソールへのリンクを直接追加します。

ページ下部の `Manage Links` をクリックしてパネルを開きます。追加したいページの完全な URL を入力し、次にラベルを追加します。追加したリンクは **External Links** セクションに表示されます。
