このガイドでは、Python ライブラリに Weights &amp; Biases (W&amp;B) を統合する方法を説明します。

W&amp;B をトレーニング用フレームワーク、SDK、再利用可能なライブラリなどの複雑なコードベースに統合する場合は、以下の推奨事項に従ってください。

<Tip>
  W&amp;B を初めて利用する場合は、先にコアガイド (たとえば [Experiment Tracking (実験管理)](/ja/models/track/)) に目を通してください。
</Tip>

ここでは、扱っているコードベースが単一の Python トレーニングスクリプトや Jupyter Notebook よりも複雑な場合に役立つヒントとベストプラクティスを紹介します。

<div id="decide-how-users-install-wb">
  ## ユーザーに W&amp;B をどのようにインストールしてもらうかを決める
</div>

開始する前に、ライブラリで W&amp;B を必須の依存関係とするか、オプション機能として提供するかを決めてください。

<div id="require-wb-as-a-dependency">
  ### W&amp;B を必須の依存関係にする
</div>

W&amp;B がライブラリの中核機能である場合は、W&amp;B Python SDK (`wandb`) を依存関係に追加してください。

```txt
torch==1.8.0 
...
wandb==0.13.*
```

<div id="make-wb-optional-on-installation">
  ### インストール時に W&amp;B をオプション機能にする
</div>

W&amp;B をオプション機能とする場合、ライブラリが W&amp;B をインストールしていなくても動作するようにしてください。

Python で `wandb` を条件付きでインポートするか、`pyproject.toml` でオプションの依存関係として宣言できます。

<Tabs>
  <Tab title="Python">
    `wandb` が利用可能かどうかを検出し、ユーザーが W&amp;B 機能を有効にしたのにインストールされていない場合に、明確なエラーを発生させてください：

    ```python
    try:
        import wandb
        _WANDB_AVAILABLE = True
    except ImportError:
        _WANDB_AVAILABLE = False
    ```
  </Tab>

  <Tab title="pyproject.toml">
    `pyproject.toml` ファイルに `wandb` をオプションの依存関係として宣言してください：

    ```toml
    [project]
    name = "my_awesome_lib"
    version = "0.1.0"
    dependencies = [
        "torch",
        "sklearn"
    ]

    [project.optional-dependencies]
    dev = [
        "wandb"
    ]
    ```
  </Tab>
</Tabs>

<div id="authenticate-users">
  ## ユーザーを認証する
</div>

W&amp;B では、ユーザーとマシンの認証に APIキーを使用します。

<div id="create-an-api-key">
  ### APIキーを発行する
</div>

APIキーは、クライアントまたはマシンをW&amp;Bに対して認証するために使用します。APIキーはユーザーのプロフィールから発行できます。

<div id="install-and-log-in-to-wb">
  ### W&amp;B をインストールしてログインする
</div>

`wandb` ライブラリをローカルにインストールしてログインするには、次の手順に従います。

<Tabs>
  <Tab title="コマンドライン">


    2. `wandb` ライブラリをインストールしてログインします。


  </Tab>

  <Tab title="Python">


  </Tab>

  <Tab title="Python notebook">

<div id="start-a-run">
  ## run を開始する
</div>

*run* は、トレーニング実験などの単一の計算単位を表します。ほとんどのライブラリでは、トレーニングジョブごとに 1 つの run が作成されます。run の詳細については、[W&amp;B Runs](/ja/models/runs/) を参照してください。

[`wandb.init()`](/ja/models/ref/python/functions/init) を使用して run を初期化し、プロジェクト名とチーム entity (チーム名) を指定します。プロジェクトを指定しない場合、W&amp;B は run を「uncategorized」という名前のデフォルトプロジェクトに保存します。:

```python
with wandb.init(project="<project_name>", entity="<entity>") as run:
    ...
```

W&amp;B では、エラーが発生した場合でも run が適切に終了されるように、コンテキストマネージャーを使用することを推奨しています。コンテキストマネージャーを使用しない場合は、`run.finish()` を呼び出して run を終了し、すべてのデータを W&amp;B にログする必要があります。

<Tip>
  **`wandb.init` を呼び出すタイミング**

  `wandb.init()` はできるだけ早く呼び出してください。W&amp;B は stdout、stderr、およびエラーメッセージを取得するため、デバッグが容易になります。

  関連するすべての情報が run で取得されるように、トレーニングループ全体を `wandb.init` のコンテキストマネージャーで囲んでください。これには、デバッグで重要になる可能性があるエラーメッセージも含まれます。
</Tip>

<div id="set-wandb-as-an-optional-dependency">
  ### `wandb` をオプションの依存関係として設定する
</div>

あなたのライブラリを使用するユーザーにとって `wandb` をオプションにしたい場合は、次のいずれかを行います。

* たとえば、次のような `wandb` フラグを定義します。

<Tabs>
  <Tab title="Python">
    ```python
    trainer = my_trainer(..., use_wandb=True)
    ```
  </Tab>

  <Tab title="Bash">
    ```bash
    python train.py ... --use-wandb
    ```
  </Tab>
</Tabs>

* または、`wandb.init` で `wandb` を `disabled` に設定します。

<Tabs>
  <Tab title="Python">
    ```python
    wandb.init(mode="disabled")
    ```
  </Tab>

  <Tab title="Bash">
    ```bash
    export WANDB_MODE=disabled
    ```

    または

    ```bash
    wandb disabled
    ```
  </Tab>
</Tabs>

* または、`wandb` をオフラインに設定します。なお、この場合も `wandb` 自体は実行されますが、インターネット経由で W&amp;B と通信しようとはしません。

<Tabs>
  <Tab title="Environment Variable">
    ```bash
    export WANDB_MODE=offline
    ```

    または

    ```python
    os.environ['WANDB_MODE'] = 'offline'
    ```
  </Tab>

  <Tab title="Bash">
    ```bash
    wandb offline
    ```
  </Tab>
</Tabs>

<div id="define-a-run-config">
  ## run の設定を定義する
</div>

run を初期化するときに設定用の辞書を渡して、ハイパーパラメーターやその他のメタデータを W&amp;B にログします。

W&amp;B App を使うと、設定パラメーターに基づいて run を比較したり、Runs テーブル内でフィルタリングしたりできます。また、これらのパラメーターを使って W&amp;B App 内で run をグループ化することもできます。

たとえば次の画像では、バッチサイズ (`bathch_size`) が設定パラメーターとして定義されており、Runs テーブルの列 (最初の列) に表示されています。これにより ユーザーはバッチサイズに基づいて run をフィルタリングおよび比較できます。

<Frame>
  <img src="/images/integrations/integrations_add_any_lib_runs_page.png" alt="W&B Runs テーブル" />
</Frame>

代表的な設定パラメーター値には次のようなものがあります。

* モデル名、バージョン、アーキテクチャのパラメーターおよびハイパーパラメーター
* データセット名、バージョン、トレーニングまたは検証用サンプル数
* 学習率、バッチサイズ、オプティマイザーなどのトレーニングパラメーター

次のコードスニペットは、設定をログする方法を示しています。

```python
config = {"batch_size": 32, ...}
with wandb.init(..., config=config) as run:
    ...
```

<div id="update-the-run-config">
  ### run 設定 を更新する
</div>

初期化時点で値を利用できない場合は、後から `wandb.Run.config.update` を使って設定を更新します。たとえば、モデルをインスタンス化した後に、そのパラメーターを追加したい場合などです。

```python
with wandb.init(...) as run:
    model = MyModel(...)
    run.config.update({"model_parameters": 3500})
```

詳細は [Experiments の設定](/ja/models/track/config/) を参照してください.

<div id="log-metrics-and-data">
  ## メトリクスとデータをログする
</div>

<div id="log-metrics">
  ### メトリクスをログする
</div>

メトリクス名をキーとする辞書を作成します。この辞書オブジェクトを [`wandb.Run.log()`](/ja/models/ref/python/experiments/run#method-run-log) に渡して、W&amp;B にログします。

```python
NUM_EPOCHS = 10

for epoch in range(NUM_EPOCHS):
    for input, ground_truth in data: 
        prediction = model(input) 
        loss = loss_fn(prediction, ground_truth) 
        metrics = { "loss": loss } 
        run.log(metrics)
```

W&amp;B App で関連するメトリクスをグループ化するには、メトリクス名のプレフィックスを使用します。一般的なプレフィックスには、トレーニング メトリクス用の `train/` と、検証メトリクス用の `val/` がありますが、ユースケースに応じて任意のプレフィックスを使用できます。

これにより、プロジェクトの Workspace 内に、トレーニング メトリクス用、検証メトリクス用、または分けて表示したいその他のメトリクス タイプ用の個別のセクションが作成されます。

```python
with wandb.init(...) as run:
    metrics = {
        "train/loss": 0.4,
        "train/learning_rate": 0.4,
        "val/loss": 0.5, 
        "val/accuracy": 0.7
    }
    run.log(metrics)
```

<Frame>
  <img src="/images/integrations/integrations_add_any_lib_log.png" alt="W&B Workspace" />
</Frame>

詳細については、[`wandb.Run.log()`](/ja/models/ref/python/experiments/run#method-run-log) を参照してください。

<div id="control-the-x-axis">
  ### x 軸を制御する
</div>

同じトレーニングstepに対して `wandb.Run.log()` を複数回呼び出すと、wandb SDK は `wandb.Run.log()` の呼び出しごとに内部のstepカウンターを 1 ずつ増やします。このカウンターは、トレーニングループ内のトレーニングstepと一致しないことがあります。

この状況を避けるには、`wandb.init` を呼び出した直後に一度だけ、`run.define_metric` を使って x 軸のstepを明示的に定義してください。

```python
with wandb.init(...) as run:
    run.define_metric("*", step_metric="global_step")
```

グロブパターン `*` は、すべてのメトリクスでチャートの x 軸として `global_step` が使用されることを意味します。`global_step` に対してログするメトリクスを特定のものだけにしたい場合は、代わりにそれらを個別に指定できます。

```python
run.define_metric("train/loss", step_metric="global_step")
```

次に、`wandb.Run.log()` を呼び出すたびに、メトリクス、`step` メトリクス、`global_step` をログしてください。

```python
for step, (input, ground_truth) in enumerate(data):
    ...
    run.log({"global_step": step, "train/loss": 0.1})
    run.log({"global_step": step, "eval/loss": 0.2})
```

独立したstep変数にアクセスできない場合、たとえば検証ループ中に &quot;global&#95;step&quot; が利用できないときは、wandb によって &quot;global&#95;step&quot; に対して以前にログされた値が自動的に使用されます。この場合は、必要なときにそのメトリクスが定義済みになっているよう、初期値をあらかじめログしておいてください。

<div id="log-media-and-structured-data">
  ### メディアと構造化データをログする
</div>

スカラー値に加えて、画像、表、テキスト、オーディオ、動画などもログできます。

データをログする際には、次の点を検討してください。

* メトリクスはどのくらいの頻度でログすべきか？ 任意にすべきか？
* どのような種類のデータが可視化に役立つか？
  * 画像の場合、サンプル予測やセグメンテーションマスクなどをログして、時間経過に伴う変化を確認できます。
  * テキストの場合、後から探索できるようにサンプル予測の表をログできます。

例については、[Log objects and media](/ja/models/track/log) を参照してください。

<div id="support-distributed-training">
  ## 分散トレーニングのサポート
</div>

分散環境をサポートしているフレームワークでは、次のいずれかのワークフローを利用できます。

* メインプロセスからのみログする (推奨) 。
* すべてのプロセスからログし、共通の `group` 名で Runs をグループ化する。

<div id="track-models-and-datasets-with-artifacts">
  ## アーティファクト でモデルとデータセットをトラッキングする
</div>

[W&amp;B Artifacts](/ja/models/artifacts/) を使用して、モデルとデータセットをトラッキングし、バージョン管理します。アーティファクト は機械学習アセットのストレージとバージョン管理を提供し、データとモデルのリネージを自動的にトラッキングして、それらの関係を可視化します。

<Frame>
  <img src="/images/integrations/integrations_add_any_lib_dag.png" alt="W&B に保存されたデータセットとモデル チェックポイント" />
</Frame>

アーティファクト をライブラリに統合する際は、次の点を検討してください。

* モデル チェックポイントやデータセットを アーティファクト としてログするかどうか (必要に応じてオプションにする場合など) 。
* アーティファクト の入力参照 (例: `entity/project/artifact`) 。
* モデル チェックポイントやデータセットをログする頻度。たとえば、各エポックごと、500 step ごとなどです.

<div id="log-model-checkpoints">
  ### モデルのチェックポイントをログする
</div>

モデルのチェックポイントを W&amp;B にログします。一般的な方法は、W&amp;B が生成する一意の run ID を アーティファクト の名前の一部として使用し、チェックポイントを アーティファクト としてログすることです。

```python
metadata = {"eval/accuracy": 0.8, "train/steps": 800} 

artifact = wandb.Artifact(
                name=f"model-{run.id}", 
                metadata=metadata, 
                type="model"
                ) 
artifact.add_dir("output_model") # モデルの重みが保存されているローカルディレクトリ

aliases = ["best", "epoch_10"] 
run.log_artifact(artifact, aliases=aliases)
```

前の코드 스니펫は、モデル チェックポイントを アーティファクト としてログし、評価精度やトレーニングstepなどのメタデータを追加する方法を示しています。 アーティファクト には一意の run ID を含む名前が付けられ、参照しやすいように [カスタムエイリアス](/ja/models/artifacts/create-a-custom-alias/) がタグとして付けられています.

<div id="log-input-artifacts">
  ### 入力 アーティファクト をログする
</div>

入力として使用するデータセットや事前学習済みモデルをログします。

```python
dataset = wandb.Artifact(name="flowers", type="dataset")
dataset.add_file("flowers.npy")
run.use_artifact(dataset)
```

前の코드 스니펫では、&quot;flowers&quot; という名前のデータセット用アーティファクトを作成し、その中にファイルを追加します。その後、`run.use_artifact()` を使用してそのアーティファクトを現在の run に関連付けます。こうすることで、その run で使用されたデータセットのリネージを W&amp;B がトラッキングできるようになります.

<div id="download-artifacts">
  ### アーティファクト をダウンロードする
</div>

以前 W&amp;B にログした アーティファクト をダウンロードして、トレーニングまたは推論のコードで使用します。

run コンテキストがある場合は、[`wandb.Run.use_artifact()`](/ja/models/ref/python/experiments/run) を使用して W&amp;B のアーティファクトを参照し、その後 [`wandb.Artifact.download()`](/ja/models/ref/python/experiments/artifact) を呼び出してローカルディレクトリにダウンロードします。

```python
with wandb.init(...) as run:
    artifact = run.use_artifact("user/project/artifact:latest")
    local_path = artifact.download()
```

run を初期化せずにアーティファクトを参照またはダウンロードするには、[W&amp;B Public API](/ja/models/ref/python/public-api/) を使用します。これは、分散環境や推論時など、新しい run を作成したくない場合に便利です。

```python
import wandb
artifact = wandb.Api().artifact("user/project/artifact:latest")
local_path = artifact.download()
```

詳しくは、[アーティファクト をダウンロードして使用する](/ja/models/artifacts/download-and-use-an-artifact/) を参照してください.

<div id="tune-hyper-parameters">
  ## ハイパーパラメーターを調整する
</div>

ライブラリがハイパーパラメーターチューニングをサポートしている場合は、[W&amp;B Sweeps](/ja/models/sweeps/) を統合して、Experiments を管理・可視化できます。
