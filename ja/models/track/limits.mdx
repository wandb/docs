{/* ## ページを高速に保つためのベストプラクティス */}

以下の推奨範囲内でログすることで、W&amp;B 内のページをより高速かつ応答性の高い状態に保てます。

<div id="logging-considerations">
  ## ロギングに関する考慮事項
</div>

実験のメトリクスをトラッキングするには `wandb.Run.log()` を使用します。

<div id="distinct-metric-count">
  ### 個別のメトリクス数
</div>

パフォーマンスを高く保つため、1 つのプロジェクト内の個別のメトリクスの総数は 10,000 未満に抑えてください。

```python
import wandb

with wandb.init() as run:
    run.log(
        {
            "a": 1,  # "a" は個別のメトリクス
            "b": {
                "c": "hello",  # "b.c" は個別のメトリクス
                "d": [1, 2, 3],  # "b.d" は個別のメトリクス
            },
        }
    )
```

<Note>
  W&amp;B はネストされた値を自動的にフラット化します。つまり、dict を渡すと、W&amp;B はそれをドット区切りの名前に変換します。config の値では、名前に含められるドットは最大 3 個までサポートされます。summary の値では、ドットは最大 4 個までサポートされます。
</Note>

メトリクス名は、GraphQL による命名上の制約に従う必要があります。詳しくは [Metric naming constraints](/ja/models/track/log/#metric-naming-constraints) を参照してください。

{/* ### 同じメトリクス名でメディアをログする
  関連するメディアは同じメトリクス名にログしてください:

  ```python
  for i, img in enumerate(images):
    # 非推奨
    run.log({f"pred_img_{i}": wandb.Image(image)})

    # 推奨
    run.log({"pred_imgs": [wandb.Image(image) for image in images]})
  ``` */}

Workspace が突然遅くなった場合は、最近の run で意図せず何千もの新しいメトリクスをログしていないか確認してください。(これは、数千個のプロットがあるのに 1～2 個の run しか表示されていないセクションを探すと、最も簡単に見つけられます。) そのような run がある場合は、それらの run を削除し、必要なメトリクスだけをログするように再作成することを検討してください。

<div id="value-width">
  ### 値のサイズ
</div>

1 つのログ値のサイズは 1 MB 未満、1 回の `wandb.Run.log()` 呼び出しの合計サイズは 25 MB 未満にしてください。この制限は、`wandb.Image` や `wandb.Audio` などの `wandb.Media` 型には適用されません。

```python
import wandb

with wandb.init(project="wide-values") as run:

    # 非推奨
    run.log({"wide_key": range(10000000)})

    # 非推奨
    with open("large_file.json", "r") as f:
        large_data = json.load(f)
        run.log(large_data)
```

値の幅が大きいと、そのような値を持つメトリクスだけでなく、その run 内のすべてのメトリクスのプロット読み込み時間に影響することがあります。

<Note>
  推奨される幅を超える値をログしても、データは保存され、トラッキングも行われます。ただし、プロットの読み込みが遅くなる場合があります。
</Note>

<div id="metric-frequency">
  ### メトリクスのログ頻度
</div>

ログするメトリクスに応じて、適切なログ頻度を選択してください。一般的な目安として、値の幅が広いものほどログ頻度を低くし、幅が狭いものほど高い頻度でログします。W&amp;B では、次を推奨しています。

* スカラー: メトリクスごとのログポイント数は 100,000 未満
* メディア: メトリクスごとのログポイント数は 50,000 未満
* ヒストグラム: メトリクスごとのログポイント数は 10,000 未満

```python
import wandb

with wandb.init(project="metric-frequency") as run:
    # 非推奨
    run.log(
        {
            "scalar": 1,  # 100,000 スカラー
            "media": wandb.Image(...),  # 100,000 画像
            "histogram": wandb.Histogram(...),  # 100,000 ヒストグラム
        }
    )

    # 推奨
    run.log(
        {
            "scalar": 1,  # 100,000 スカラー
        },
        commit=True,
    )  # バッチ化されたステップごとのメトリクスをまとめてコミット

    run.log(
        {
            "media": wandb.Image(...),  # 50,000 画像
        },
        commit=False,
    )
    
    run.log(
        {
            "histogram": wandb.Histogram(...),  # 10,000 ヒストグラム
        },
        commit=False,
    )
```

{/* 特定の step における API Call の総数を最小限に抑えるには、`commit=False` を渡して `run.log` の Call をバッチ処理します。詳細は `run.log` の [ドキュメント](/models/ref/python/experiments/run/#method-runlog) を参照してください。 */}

<Note>
  ガイドラインを超えても、W&amp;B はログデータを引き続き受け付けますが、ページの読み込みが遅くなる場合があります。
</Note>

<div id="config-size">
  ### Config サイズ
</div>

run config の合計サイズは 10 MB 未満にしてください。大きな値をログすると、プロジェクトの Workspace や Runs table での操作が遅くなる可能性があります。

```python
import wandb 

# 推奨
with wandb.init(
    project="config-size",
    config={
        "lr": 0.1,
        "batch_size": 32,
        "epochs": 4,
    }
) as run:
    # Your training code here
    pass

# 非推奨
with wandb.init(
    project="config-size",
    config={
        "large_list": list(range(10000000)),  # 大きなリスト
        "large_string": "a" * 10000000,  # 大きな文字列
    }
) as run:
    # トレーニングコードをここに記述
    pass

# 非推奨
with open("large_config.json", "r") as f:
    large_config = json.load(f)
    wandb.init(config=large_config)
```

<div id="workspace-considerations">
  ## Workspace に関する考慮事項
</div>

<div id="run-count">
  ### Run 数
</div>

読み込み時間を短く抑えるために、1 つのプロジェクト内の合計 run 数は、次の範囲内に収めてください。

* SaaS Cloud では 100,000
* 専用クラウド または セルフマネージド では 10,000

run 数がこれらのしきい値を超えると、プロジェクト Workspace や Runs table に関わる操作、特に run をグループ化する場合や、run 中に多数の異なるメトリクスを収集する場合に、処理が遅くなる可能性があります。[Metric count](#metric-count) のセクションも参照してください。

チームが同じ run のセット (たとえば最近の run) に頻繁に 액세스する場合は、[使用頻度の低い run をまとめて移動する](/ja/models/runs/manage-runs/)ことで新しい「archive」プロジェクトに退避させ、作業中のプロジェクトには少数の run のみを残すことを検討してください。

<div id="workspace-performance">
  ### Workspace のパフォーマンス
</div>

このセクションでは、Workspace のパフォーマンスを最適化するためのヒントを紹介します。

<div id="panel-count">
  #### パネル数
</div>

既定では、Workspace は *automatic* で、ログされた各キーに対して標準パネルを生成します。大規模なプロジェクトの Workspace に多数のログされたキーのパネルが含まれている場合、Workspace の読み込みや使用が遅くなることがあります。パフォーマンスを改善するには、次のことを実行できます。

1. Workspace を Manual モードにリセットします。このモードには、既定ではパネルが含まれません。
2. [Quick add](/ja/models/app/features/panels/#quick-add) を使用して、可視化に必要なログされたキーのパネルだけを選択して追加します。

<Note>
  未使用のパネルを 1 つずつ削除しても、パフォーマンスへの影響はほとんどありません。代わりに、Workspace をリセットし、必要なパネルだけを選択して追加し直してください。
</Note>

Workspace の設定の詳細については、[Panels](/ja/models/app/features/panels/) を参照してください。

<div id="section-count">
  #### セクション数
</div>

Workspace内に何百ものセクションがあると、パフォーマンスが低下する可能性があります。メトリクスを大まかなグループごとにセクション化し、各メトリクスごとに 1 つずつセクションを作成するようなアンチパターンは避けてください。

セクションが多すぎてパフォーマンスが低下している場合は、Workspace設定で、セクションをサフィックスではなくプレフィックスで作成するように変更することを検討してください。これにより、セクション数を減らし、パフォーマンスを向上できる可能性があります。

<div id="metric-count">
  ### メトリクス数
</div>

1 run あたりで 5,000〜100,000 個のメトリクスを記録する場合は、W&amp;B では [manual workspace](/ja/models/app/features/panels/#workspace-modes) の使用を推奨します。Manual モードでは、異なるメトリクスのセットを探索する際に、パネルをまとめて簡単に追加または削除できます。プロットを必要なものに絞ることで、Workspace の読み込みが速くなります。プロットされていないメトリクスも、通常どおり収集および保存されます。

Workspace を Manual モードにリセットするには、Workspace のアクション `...` メニューをクリックし、**Reset workspace** をクリックします。Workspace をリセットしても、run に保存されているメトリクスには影響しません。詳しくは [workspace panel management](/ja/models/app/features/panels/) を参照してください。

<div id="file-count">
  ### ファイル数
</div>

1 つの run にアップロードするファイルの総数は 1,000 未満に保ってください。大量のファイルをログする必要がある場合は、W&amp;B Artifacts を使用できます。1 つの run で 1,000 個を超えるファイルをアップロードすると、run ページの表示が遅くなる可能性があります。

<div id="reports-vs-workspaces">
  ### Reports と Workspaces
</div>

レポートは、パネル、テキスト、メディアを任意に配置して自由形式で構成できるため、同僚とインサイトを簡単に共有できます。

一方、Workspace は、数十から数千のメトリクスを、数百から数十万の run にわたって高密度かつ高性能に分析できます。Workspace は、レポートと比較して、캐싱、クエリ、読み込みといった機能が最適化されています。Workspace は、プレゼンテーションよりも主に分析に使用されるプロジェクトや、20 個以上のプロットをまとめて表示する必要がある場合に推奨されます。

<div id="python-script-performance">
  ## Python スクリプトのパフォーマンス
</div>

Python スクリプトのパフォーマンスが低下する要因はいくつかあります:

1. データのサイズが大きすぎる。データ サイズが大きいと、トレーニング ループに 1 ms を超えるオーバーヘッドが発生する可能性があります。
2. ネットワークの速度と、W&amp;B バックエンドの構成方法。
3. `wandb.Run.log()` を 1 秒間に何度も呼び出している場合。これは、`wandb.Run.log()` が呼び出されるたびにトレーニング ループにわずかなレイテンシーが追加されるためです。

<Note>
  頻繁なロギングがトレーニング run を遅くしていませんか？ロギング戦略を変更してパフォーマンスを向上させる方法については、[この Colab](https://wandb.me/log-hf-colab) を確認してください。
</Note>

W&amp;B はレート制限以外の制限は設けていません。W&amp;B Python SDK は、制限を超えたリクエストに対して、自動的に指数関数的な「バックオフ」と「再試行」を行います。W&amp;B Python SDK は、コマンドライン上で「Network failure」と応答します。無償アカウントの場合、利用量が妥当な閾値を大きく超えた極端なケースでは、W&amp;B から連絡する場合があります。

<div id="rate-limits">
  ## レート制限
</div>

W&amp;B SaaS Cloud API では、システムの整合性を維持し、可用性を確保するためにレート制限を実装しています。この措置により、共有インフラストラクチャー内のリソースを 1 人のユーザーが独占することを防ぎ、すべてのユーザーがサービスを利用できるようにしています。さまざまな理由により、より低いレート制限が適用される場合があります。

<Note>
  レート制限は変更される場合があります。
</Note>

レート制限に達すると、HTTP `429` の `Rate limit exceeded` エラーを受け取り、レスポンスには[レート制限の HTTP ヘッダー](#rate-limit-http-headers)が含まれます。

<div id="rate-limit-http-headers">
  ### レート制限の HTTP ヘッダー
</div>

前述の表では、レート制限の HTTP ヘッダーについて説明しています。

| ヘッダー名               | 説明                                                   |
| ------------------- | ---------------------------------------------------- |
| RateLimit-Limit     | 時間ウィンドウごとに利用可能なクォータ量で、0 ～ 1000 の範囲にスケーリングされた値です      |
| RateLimit-Remaining | 現在のレート制限ウィンドウ内で残っているクォータ量で、0 ～ 1000 の範囲にスケーリングされた値です |
| RateLimit-Reset     | 現在のクォータがリセットされるまでの秒数です                               |

<div id="rate-limits-on-metric-logging-api">
  ### メトリクスのロギング API のレート制限
</div>

`wandb.Run.log()` は、トレーニングデータを W&amp;B に記録します。この API は、オンライン時または [オフライン同期](/ja/models/ref/cli/wandb-sync) を通じて使用されます。いずれの場合も、一定のローリングウィンドウ内でレート制限のクォータが適用されます。これには、リクエストの合計サイズとリクエストレートに対する制限が含まれます。後者は、一定時間内のリクエスト数を指します。

W&amp;B では、レート制限は W&amp;B プロジェクトごとに適用されます。そのため、1 つのチームに 3 つのプロジェクトがある場合、各プロジェクトに個別のレート制限クォータがあります。[Paid plans](https://wandb.ai/site/pricing) のユーザーは、Free プランよりも高いレート制限が適用されます。

レート制限に達すると、HTTP `429` `Rate limit exceeded` エラーが返され、レスポンスには [レート制限の HTTP ヘッダー](#rate-limit-http-headers) が含まれます。

<div id="suggestions-for-staying-under-the-metrics-logging-api-rate-limit">
  ### メトリクスのロギング API のレート制限内に収めるための方法
</div>

レート制限を超えると、レート制限がリセットされるまで `run.finish()` が遅延する可能性があります。これを避けるために、次の方法を検討してください。

* W&amp;B Python SDK のバージョンを更新する: 最新バージョンの W&amp;B Python SDK を使用していることを確認してください。W&amp;B Python SDK は定期的に更新されており、リクエストを適切に再試行し、クォータ使用量を最適化する仕組みが強化されています。
* メトリクスのログ頻度を下げる:
  クォータを節約するために、メトリクスをログする頻度を抑えてください。たとえば、毎エポックではなく 5 エポックごとにメトリクスをログするようにコードを変更できます。

```python
import wandb
import random

with wandb.init(project="basic-intro") as run:
    for epoch in range(10):
        # トレーニングと評価をシミュレート
        accuracy = 1 - 2 ** -epoch - random.random() / epoch
        loss = 2 ** -epoch + random.random() / epoch

        # 5 エポックごとにメトリクスをログ
        if epoch % 5 == 0:
            run.log({"acc": accuracy, "loss": loss})
```

* 手動でのデータ同期: レート制限が適用されると、W&amp;B は run データをローカルに保存します。コマンド `wandb sync <run-file-path>` を使用して、データを手動で同期できます。詳細については、[`wandb sync`](/ja/models/ref/cli/wandb-sync) リファレンスを参照してください。

<div id="rate-limits-on-graphql-api">
  ### GraphQL API のレート制限
</div>

W&amp;B Models UI と SDK の [public API](/ja/models/ref/python/public-api/api) は、データのクエリや変更を行うために、サーバーに GraphQL リクエストを送信します。SaaS Cloud 内のすべての GraphQL リクエストに対して、W&amp;B は未認証のリクエストには IP アドレス単位で、認証済みのリクエストにはユーザー単位でレート制限を適用します。この制限は、固定時間ウィンドウ内のリクエスト率 (1 秒あたりのリクエスト数) に基づいており、デフォルトの制限値は利用中の料金プランによって決まります。プロジェクト パスを指定する関連 SDK リクエスト (たとえば Reports、Runs、Artifacts) については、W&amp;B はデータベース クエリ時間に基づいて、プロジェクト単位でレート制限を適用します。

[Teams and Enterprise plans](https://wandb.ai/site/pricing) のユーザーには、Free プランよりも高いレート制限が適用されます。
W&amp;B Models SDK の public API の使用中にレート制限に達すると、標準出力にエラーを示す関連メッセージが表示されます。

レート制限に達すると、HTTP `429` `Rate limit exceeded` エラーが返され、レスポンスには [レート制限の HTTP ヘッダー](#rate-limit-http-headers) が含まれます。

<div id="suggestions-for-staying-under-the-graphql-api-rate-limit">
  #### GraphQL API レート制限を超えないための推奨事項
</div>

W&amp;B Models SDK の [Public API](/ja/models/ref/python/public-api/api) を使って大量のデータを取得する場合は、リクエストの間隔を少なくとも 1 秒あけることを検討してください。HTTP `429` `Rate limit exceeded` エラーを受け取った場合、またはレスポンスヘッダーで `RateLimit-Remaining=0` を確認した場合は、`RateLimit-Reset` で指定されている秒数だけ待機してから再試行してください。

<div id="browser-considerations">
  ## ブラウザに関する考慮事項
</div>

W&amp;B App はメモリを多く消費することがあり、Chrome で最も快適に動作します。お使いのコンピュータのメモリ容量によっては、W&amp;B を 3 つ以上のタブで同時に開くと、パフォーマンスが低下する可能性があります。想定外に動作が遅い場合は、他のタブやアプリケーションを閉じることを検討してください。

<div id="reporting-performance-issues-to-wb">
  ## W&amp;B へのパフォーマンス問題の報告
</div>

W&amp;B はパフォーマンスを重視しており、遅延の報告はすべて調査しています。調査を迅速に進めるため、読み込み時間が長い場合を報告する際は、主要なメトリクスとパフォーマンスイベントを取得できる W&amp;B の組み込みパフォーマンスロガーを使用することを検討してください。読み込みが遅いページの URL に `&PERF_LOGGING` パラメーターを追加し、その後、コンソールの出力をアカウントチームまたはサポートと共有してください。
