---
description: W&B を使用して、複数の GPU を用いた分散トレーニング実験をログします。
title: 分散トレーニング実験をログする
---

分散トレーニング実験では、複数のマシンまたはクライアントを並行して使用し、モデルをトレーニングします。W&amp;B は分散トレーニング実験のトラッキングを支援します。ユースケースに応じて、以下のアプローチのいずれかを使用して分散トレーニング実験をトラッキングしてください：

* **単一プロセスをトラッキングする**: W&amp;B でランク 0 プロセス (「リーダー」または「コーディネーター」とも呼ばれる) をトラッキングします。これは、[PyTorch Distributed Data Parallel](https://pytorch.org/docs/stable/generated/torch.nn.parallel.DistributedDataParallel.html#torch.nn.parallel.DistributedDataParallel) (DDP) クラスを使用して分散トレーニング実験をログする一般的な方法です。
* **複数プロセスをトラッキングする**: 複数のプロセスの場合、以下のいずれかを実行できます：
  * プロセスごとに 1 つの run を使用して、各プロセスを個別にトラッキングします。オプションで W&amp;B App UI でそれらをグループ化できます。
  * すべてのプロセスを単一の run にトラッキングします。

<Tip>
  **同時接続**

  各同時接続は、コンピュート、メモリ、およびネットワークリソースを消費します。メトリクスを定期的にログしない空のクライアント接続であっても、システムメトリクス更新をプッシュし、チャートの読み込み時のパフォーマンスが低下します。

  W&amp;B は、ワークロードに適した最大同時クライアント接続数を制限し、リソース使用量を時間経過で監視することを推奨します。W&amp;B は、**専用クラウド** で 300 の同時クライアント接続のハードリミットでテストを行いました。

  **マルチテナント クラウド** 組織では、分散トレーニングのクライアント接続は、通常のトレーニング run と同様の [rate limits](/ja/models/track/limits#rate-limits) に従います。[Teams および Enterprise plans](https://wandb.ai/site/pricing) のユーザー は、Free プランよりも高い rate limits を受け取ります。
</Tip>

<div id="track-a-single-process">
  ## 単一のプロセスをトラッキングする
</div>

このセクションでは、ランク 0 プロセスで利用可能な値とメトリクスをトラッキングする方法を説明します。この手法により、単一のプロセスから利用可能なメトリクスのみをトラッキングできます。典型的なメトリクスには、GPU/CPU の利用率、共有の検証セットでの動作、グラディエントとパラメーター、および代表的なデータ例での損失値が含まれます。

ランク 0 プロセス内で、[`wandb.init()`](/ja/models/ref/python/functions/init) を使用して W&amp;B の run を初期化し、その run に対して experiments を [`wandb.log`](/ja/models/ref/python/experiments/run/#method-runlog) でログします。

以下の [サンプル Python スクリプト (`log-ddp.py`)](https://github.com/wandb/examples/blob/master/examples/pytorch/pytorch-ddp/log-ddp.py) は、PyTorch DDP を使用して単一のマシン上の 2 つの GPU でメトリクスをトラッキングする一例を示しています。[PyTorch DDP](https://pytorch.org/tutorials/intermediate/ddp_tutorial.html) (`torch.nn` 内の `DistributedDataParallel`) は、分散トレーニングで人気のライブラリです。基本的な原則は任意の分散トレーニング環境に適用されますが、実装は異なる場合があります。

Python スクリプト:

1. `torch.distributed.launch` で複数のプロセスを開始します。
2. `--local_rank` コマンドライン引数でランクを確認します。
3. ランクが 0 に設定されている場合、[`train()`](https://github.com/wandb/examples/blob/master/examples/pytorch/pytorch-ddp/log-ddp.py#L24) 関数内で条件付きで `wandb` ロギングを設定します。

```python
if __name__ == "__main__":
    # 引数を取得
    args = parse_args()

    if args.local_rank == 0:  # メイン プロセスでのみ
        # wandb run を初期化
        run = wandb.init(
            entity=args.entity,
            project=args.project,
        )
        # DDP を使用してモデルをトレーニング
        train(args, run)
    else:
        train(args)
```

<div id="track-multiple-processes">
  ## 複数のプロセスをトラッキングする
</div>

W&amp;B を使用して、以下のいずれかのアプローチで複数のプロセスをトラッキングします：

* [各プロセスを個別にトラッキングする](/ja/models/track/log/distributed-training/#track-each-process-separately) 各プロセスごとに run を 1 つずつ作成することにより。
* [すべてのプロセスを単一の run にトラッキングする](/ja/models/track/log/distributed-training/#track-all-processes-to-a-single-run)。

<div id="track-each-process-separately">
  ### 各プロセスを個別にトラッキングする
</div>

このセクションでは、各プロセスごとに run を作成することで、各プロセスを個別にトラッキングする方法を説明します。各 run 内で、メトリクスやアーティファクトなどをその run にログします。トレーニングの終了時に `wandb.Run.finish()` を呼び出して、run が完了したことをマークし、全てのプロセスが適切に終了するようにします。

複数の実験にわたる run をトラッキングするのが難しい場合があります。これを緩和するために、W&amp;B を初期化する際 (`wandb.init(group='group-name')`) に `group` パラメーターに値を指定して、どの run が特定の experiment に属するかをトラッキングします。実験におけるトレーニングと評価の W&amp;B Runs をトラッキングする方法の詳細については、[Group Runs](/ja/models/runs/grouping/) を参照してください。

<Note>
  **個別のプロセスからのメトリクスをトラッキングしたい場合にこのアプローチを使用してください**。典型的な例として、各ノード上のデータと予測 (データ分布のデバッグ用) や、メインノード外の個別バッチ上のメトリクスが挙げられます。このアプローチは、全ノードからのシステムメトリクスを取得したり、メインノードで利用可能な要約統計を取得したりするために必要ありません。
</Note>

以下の Python コード スニペットは、W&amp;B を初期化する際に group パラメーターを設定する方法を示しています：

```python
if __name__ == "__main__":
    # 引数を取得
    args = parse_args()
    # run を初期化
    run = wandb.init(
        entity=args.entity,
        project=args.project,
        group="DDP",  # 実験の全 run を1つのグループにまとめる
    )
    # DDP でモデルをトレーニング
    train(args, run)

    run.finish()  # run を完了としてマーク
```

W&amp;B App UI を探索して、複数のプロセスからトラッキングされたメトリクスの [例のダッシュボード](https://wandb.ai/ayush-thakur/DDP?workspace=user-noahluna) をご覧ください。左サイドバーに 2 つの W&amp;B Runs がグループ化されていることに注意してください。グループをクリックすると、実験専用のグループページが表示されます。専用のグループページでは、各プロセスからのメトリクスを個別に表示します。

<div id="organize-distributed-runs">
  ### 分散 runs の整理
</div>

W&amp;B を初期化する際に `job_type` パラメーターを設定してください (`wandb.init(job_type='type-name')`) 。これにより、ノードを機能に基づいて分類できます。例えば、主な調整ノードと複数の報告ワーカーノードがある場合です。主な調整ノードに対しては `job_type` を `main` に、報告ワーカーノードに対しては `worker` に設定できます：

```python
   # メインの調整ノード
   with wandb.init(project="<project>", job_type="main", group="experiment_1") as run:
        # トレーニング コード

   # レポート ワーカーノード
   with wandb.init(project="<project>", job_type="worker", group="experiment_1") as run:
        # トレーニング コード
```

`job_type` をノードに設定したら、ワークスペース内で [保存済みビュー](/ja/models/track/workspaces/#create-a-new-saved-workspace-view) を作成して run を整理できます。右上の **...** アクションメニューをクリックし、**新しいビューとして保存** をクリックします。

例として、以下の保存済みビューを作成できます：

* **Default view**: ノイズを低減するためにワーカーノードを除外
  * **フィルター** をクリックし、**Job Type** を `worker` に設定します。

  * レポートノードのみを表示します

  * **Debug view**: トラブルシューティングのためにワーカーノードに焦点を当てる
    * **フィルター** をクリックし、**Job Type** `==` `worker` に設定し、**State** を `IN` `crashed` に設定します。
    * クラッシュしたかエラー状態にあるワーカーノードのみを表示します

  * **All nodes view**: すべてをまとめて表示
    * フィルターなし
    * 包括的な監視に便利です

保存済みビューを開くには、プロジェクトのサイドバーで **Workspaces** をクリックし、次にメニューをクリックします。Workspaces はリストの上部に表示され、保存済みビューは下部に表示されます。

<div id="track-all-processes-to-a-single-run">
  ### すべてのプロセスを単一の run にトラッキングする
</div>

<Warning>
  `x_` で始まるパラメーター (例: `x_label`) は現在パブリック プレビュー段階です。[W&amp;B リポジトリ](https://github.com/wandb/wandb) で GitHub issue を作成し、フィードバックをお寄せください。
</Warning>

<Note>
  **要件**

  複数のプロセスを単一の run にトラッキングするには、以下が必要です：

  * W&amp;B Python SDK バージョン `v0.19.9` 以上。

  * W&amp;B Server v0.68 以上。
</Note>

この方法では、プライマリノードと1つ以上のワーカーノードを使用します。プライマリノード内で W&amp;B run を初期化します。各ワーカーノードに対して、プライマリノードで使用される run ID を使用して run を初期化します。トレーニング中に、各ワーカーノードはプライマリノードと同じ run ID にログします。W&amp;B はすべてのノードからのメトリクスを集約し、W&amp;B App UI に表示します。

プライマリノード内で、[`wandb.init()`](/ja/models/ref/python/functions/init) を使用して W&amp;B run を初期化します。`settings` パラメーターに `wandb.Settings` オブジェクトを渡します (`wandb.init(settings=wandb.Settings()`) で、以下の設定を行います：

1. 共有モードを有効にするために、`mode` パラメーターを `"shared"` に設定します。
2. [`x_label`](https://github.com/wandb/wandb/blob/main/wandb/sdk/wandb_settings.py#L638) の一意のラベルを指定します。`x_label` に指定する値を使用して、W&amp;B App UI のログとシステムメトリクスでデータがどのノードから来ているかを識別します。指定しない場合、W&amp;B はホスト名とランダムハッシュを使用してラベルを作成します。
3. これがプライマリノードであることを示すために、[`x_primary`](https://github.com/wandb/wandb/blob/main/wandb/sdk/wandb_settings.py#L660) パラメーターを `True` に設定します。
4. W&amp;B がメトリクスをトラッキングする GPU を指定するために、`x_stats_gpu_device_ids` に GPU インデックスのリスト ([0,1,2]) をオプションで提供します。リストを提供しない場合、W&amp;B はマシン上のすべての GPU のメトリクスをトラッキングします。

プライマリノードの run ID をメモしておきます。各ワーカーノードはプライマリノードの run ID が必要です。

<Note>
  `x_primary=True` はプライマリノードをワーカーノードと区別します。プライマリノードは、設定ファイル、テレメトリなど、ノード間で共有されるファイルをアップロードする唯一のノードです。ワーカーノードはこれらのファイルをアップロードしません。
</Note>

各ワーカーノードに対して、[`wandb.init()`](/ja/models/ref/python/functions/init) を使用して W&amp;B run を初期化し、以下のものを提供します：

1. `settings` パラメーターに `wandb.Settings` オブジェクトを渡します (`wandb.init(settings=wandb.Settings()`) で：
   * 共有モードを有効にするために、`mode` パラメーターを `"shared"` に設定します。
   * `x_label` の一意のラベルを指定します。`x_label` に指定する値を使用して、W&amp;B App UI のログとシステムメトリクスでデータがどのノードから来ているかを識別します。指定しない場合、W&amp;B はホスト名とランダムハッシュを使用してラベルを作成します。
   * これがワーカーノードであることを示すために、`x_primary` パラメーターを `False` に設定します。
2. プライマリノードで使用される run ID を `id` パラメーターに渡します。
3. オプションで [`x_update_finish_state`](https://github.com/wandb/wandb/blob/main/wandb/sdk/wandb_settings.py#L772) を `False` に設定します。これにより、非プライマリノードが [run の状態](/ja/models/runs/run-states#run-states) を早期に `finished` に更新するのを防ぎ、run 状態が一貫してプライマリノードによって管理されることを保証します。

<Note>
  * すべてのノードで同じ entity とプロジェクトを使用してください。これにより、正しい run ID が検出されることを保証します。
  * 各ワーカーノードでプライマリノードの run ID を設定するための環境変数を定義することを検討してください。
</Note>

以下のサンプルコードは、複数のプロセスを単一の run にトラッキングするための高レベル要件を示しています：

```python
import wandb

entity = "<team_entity>"
project = "<project_name>"

# プライマリノードで run を初期化します
run = wandb.init(
    entity=entity,
    project=project,
	settings=wandb.Settings(
        x_label="rank_0", 
        mode="shared", 
        x_primary=True,
        x_stats_gpu_device_ids=[0, 1],  # (オプション) GPU 0 と 1 のメトリクスのみをトラッキングします
</OLD_STRING>
<OLD_STRING>
# Note the run ID of the primary node.
# Each worker node needs this run ID.
</OLD_STRING>
<NEW_STRING>
# プライマリノードの run ID を記録します。
# 各ワーカーノードはこの run ID が必要です。
        )
)

# Note the run ID of the primary node.
# Each worker node needs this run ID.
run_id = run.id

# プライマリノードの run ID を使用してワーカーノードで run を初期化します
run = wandb.init(
    entity=entity, # プライマリノードと同じ entity を使用します
    project=project, # プライマリノードと同じプロジェクトを使用します
	settings=wandb.Settings(x_label="rank_1", mode="shared", x_primary=False),
	id=run_id,
)

# プライマリノードの run ID を使用してワーカーノードで run を初期化します
run = wandb.init(
    entity=entity, # プライマリノードと同じ entity を使用します
    project=project, # プライマリノードと同じプロジェクトを使用します
	settings=wandb.Settings(x_label="rank_2", mode="shared", x_primary=False),
	id=run_id,
)
```

実際の例では、各ワーカーノードが別々のマシン上にある場合があります。

以下のコードスニペットは、高度な分散型ユースケースにおける一般的なシナリオを示しています。

<div id="spawn-process">
  ### Spawn プロセス
</div>

spawn されたプロセスで run を開始する場合、メイン関数で `wandb.setup()` methodを使用してください：

```python
import multiprocessing as mp

def do_work(n):
    with wandb.init(config=dict(n=n)) as run:
        run.log(dict(this=n * n))

def main():
    wandb.setup()
    pool = mp.Pool(processes=4)
    pool.map(do_work, range(4))


if __name__ == "__main__":
    main()
```

<div id="share-a-run">
  ### run を共有する
</div>

プロセス間で run を共有するには、run オブジェクトを引数として渡します：

```python
def do_work(run):
    with wandb.init() as run:
        run.log(dict(this=1))

def main():
    run = wandb.init()
    p = mp.Process(target=do_work, kwargs=dict(run=run))
    p.start()
    p.join()
    run.finish()  # run を完了済みとしてマークする


if __name__ == "__main__":
    main()
```

W&amp;B はログの順序を保証できません。スクリプトの作者が同期を行う必要があります。

<div id="troubleshooting">
  ## トラブルシューティング
</div>

W&amp;B を使用した分散トレーニングでよく発生する 2 つの問題があります：

1. **トレーニング開始時にハングする** - 分散トレーニングのマルチプロセス処理と `wandb` のマルチプロセス処理が干渉すると、`wandb` プロセスがハングすることがあります。
2. **トレーニング終了時にハングする** - `wandb` プロセスが終了のタイミングを知らない場合、トレーニング ジョブがハングすることがあります。Python スクリプトの最後に `wandb.Run.finish()` API を呼び出して、W&amp;B に run が終了したことを知らせてください。`wandb.Run.finish()` API はデータのアップロードを完了し、W&amp;B を終了させます。
   W&amp;B は、`wandb service` コマンドを使用して分散ジョブの信頼性を向上させることを推奨します。上記のトレーニングの問題の両方は、wandb service が利用できない W&amp;B SDK のバージョンでよく発生します。

<div id="enable-wb-service">
  ### W&amp;B Service を有効にする
</div>

W&amp;B SDK のバージョンによっては、W&amp;B Service がデフォルトで有効になっている場合があります。

<div id="wb-sdk-0130-and-above">
  #### W&amp;B SDK 0.13.0 以降
</div>

W&amp;B SDK のバージョン `0.13.0` 以降では、W&amp;B Service はデフォルトで有効になっています。

<div id="wb-sdk-0125-and-above">
  #### W&amp;B SDK 0.12.5 以降
</div>

W&amp;B SDK バージョン 0.12.5 以降で W&amp;B Service を有効にするには、Python スクリプトを修正してください。main 関数内で `wandb.require` methodを使用し、文字列 `"service"` を渡します：

```python
if __name__ == "__main__":
    main()


def main():
    wandb.require("service")
    # スクリプトの残りの部分をここに記述
```

最適なエクスペリエンスを得るために、最新バージョンへのアップグレードをおすすめします。

**W&amp;B SDK 0.12.4 およびそれ以下**

W&amp;B SDK バージョン 0.12.4 およびそれ以下を使用している場合、代わりにマルチスレッドを使用するために、`WANDB_START_METHOD` 環境変数を `"thread"` に設定してください。
