### <kbd>fonction</kbd> `init`


Démarrez un nouveau run pour suivre et effectuer la journalisation dans W&amp;B.

Dans un pipeline d’entraînement ML, vous pouvez ajouter `wandb.init()` au début de votre script d’entraînement ainsi que de votre script d’évaluation, et chacun sera suivi en tant que run dans W&amp;B.

`wandb.init()` lance un nouveau processus en arrière-plan pour la journalisation de données dans un run, et synchronise également les données avec https://wandb.ai par défaut, afin que vous puissiez voir vos résultats en temps réel. Lorsque vous avez terminé la journalisation de données, appelez `wandb.Run.finish()` pour mettre fin au run. Si vous n’appelez pas `run.finish()`, le run se terminera à la fermeture de votre script.

Les ID de run ne doivent contenir aucun des caractères spéciaux suivants `/ \ # ? % :`

**Arguments :**


* `entity`:  Le nom d&#39;utilisateur ou le nom de l&#39;équipe auquel les runs sont enregistrés.  L&#39;entité doit déjà exister. Assurez-vous donc de créer votre compte  ou votre équipe dans l&#39;interface utilisateur avant de commencer à enregistrer des runs. Si elle n&#39;est pas spécifiée, le  run utilisera par défaut votre entité par défaut. Pour modifier l&#39;entité par défaut,  accédez à vos paramètres et mettez à jour  &quot;Emplacement par défaut pour créer de nouveaux projets&quot; sous &quot;Équipe par défaut&quot;.
* `project`:  Le nom du projet auquel ce run sera enregistré.  S’il n’est pas spécifié, nous utilisons une heuristique pour déduire le nom du projet à partir du système, par exemple en vérifiant la racine git ou le fichier  du programme en cours. Si nous ne pouvons pas déduire le nom du projet, le projet sera défini par défaut sur  `"uncategorized"`.
* `dir`:  Le chemin absolu du répertoire où sont stockés les journaux d&#39;expérience et les fichiers de métadonnées. S&#39;il n&#39;est pas spécifié, le répertoire `./wandb` est utilisé par défaut. Notez que cela n&#39;affecte pas l&#39;emplacement où les artefacts sont stockés lors de l&#39;appel à `download()`.
* `id`:  Identifiant unique de ce run, utilisé pour le reprendre. Il doit être unique  au sein du projet et ne peut pas être réutilisé une fois le run supprimé. Pour  un nom descriptif court, utilisez le champ `name` ;  pour enregistrer des hyperparamètres afin de les comparer entre les runs, utilisez `config`.
* `name` : Un nom d&#39;affichage court pour ce run, qui apparaît dans l&#39;interface utilisateur pour vous aider à l&#39;identifier. Par défaut, nous générons un nom aléatoire de deux mots, ce qui permet de faire facilement le lien entre les runs dans les tableaux et les graphiques. Le fait de conserver des noms de run brefs améliore la lisibilité dans les légendes des graphiques et les tableaux. Pour enregistrer les hyperparamètres, nous vous recommandons d&#39;utiliser le champ `config`.
* `notes`:  Une description détaillée du run, semblable à un message de commit dans Git. Utilisez cet argument pour capturer tout contexte ou détail susceptible de vous aider à vous rappeler l&#39;objectif ou la configuration de ce run par la suite.
* `tags` :  Une liste de tags permettant d’identifier ce run dans l’interface utilisateur. Les tags sont utiles pour organiser les runs ou ajouter des identifiants temporaires comme &quot;baseline&quot; ou &quot;production&quot;. Vous pouvez facilement ajouter ou supprimer des tags, ou filtrer par tag dans l’interface utilisateur.  Si vous reprenez un run, les tags fournis ici remplaceront tous les tags existants. Pour ajouter des tags à un run repris sans écraser les tags actuels, utilisez `run.tags += ("new_tag",)` après l’appel à `run = wandb.init()`.
* `config`:  Définit `wandb.config`, un objet semblable à un dictionnaire permettant de stocker les paramètres d’entrée de votre run, tels que les hyperparamètres du modèle ou les paramètres de prétraitement des données.  La configuration apparaît dans l’interface utilisateur sur une page d’Aperçu, ce qui vous permet de regrouper, filtrer et trier les runs en fonction de ces paramètres.  Les clés ne doivent pas contenir de points (`.`), et les valeurs doivent être inférieures à 10 MB.  Si un dictionnaire, `argparse.Namespace` ou `absl.flags.FLAGS` est fourni, les paires clé-valeur sont chargées directement dans `wandb.config`.  Si une chaîne de caractères est fournie, elle est interprétée comme le chemin d’accès à un fichier YAML, à partir duquel les valeurs de configuration sont chargées dans `wandb.config`.
* `config_exclude_keys` : une liste de clés spécifiques à exclure de `wandb.config`.
* `config_include_keys`:  Une liste de clés spécifiques à inclure dans `wandb.config`.
* `allow_val_change`:  Contrôle si les valeurs de configuration peuvent être modifiées après leur définition initiale. Par défaut, une exception est levée si une valeur de configuration est remplacée. Pour suivre des variables qui changent pendant l’entraînement, comme un taux d’apprentissage, pensez plutôt à utiliser `wandb.log()`. Par défaut, cette valeur est `False` dans les scripts et `True` dans les environnements notebook.
* `group`:  Spécifiez un nom de groupe pour organiser des runs individuels dans le cadre d&#39;une  expérience plus large. Cela est utile dans des cas comme la validation croisée ou l&#39;exécution de  multiples jobs qui entraînent et évaluent un modèle sur différents jeux de test.  Le regroupement vous permet de gérer collectivement des runs associés dans l&#39;interface utilisateur,  ce qui facilite le basculement entre eux et l&#39;examen des résultats comme une expérience unifiée.
* `job_type` : spécifiez le type de run, ce qui est particulièrement utile pour organiser les runs d’un groupe dans le cadre d’une expérience plus vaste. Par exemple, dans un groupe, vous pouvez attribuer aux runs des types de job tels que &quot;train&quot; et &quot;eval&quot;. Définir des types de job vous permet de filtrer et de regrouper facilement des runs similaires dans l’interface utilisateur, afin de faciliter les comparaisons directes.
* `mode` : spécifie comment les données du run sont gérées, avec les options suivantes :
  * `"online"` (par défaut) : Active la synchronisation en direct avec W&amp;B si une connexion réseau est disponible, avec des mises à jour des visualisations en temps réel.
  * `"offline"` : Adapté aux environnements isolés du réseau ou hors ligne ; les données sont enregistrées localement et peuvent être synchronisées plus tard. Assurez-vous que le dossier du run est conservé pour permettre une synchronisation ultérieure.
  * `"disabled"` : Désactive toutes les fonctionnalités de W&amp;B, ce qui transforme les méthodes du run en opérations sans effet. Généralement utilisé lors des tests pour contourner les opérations W&amp;B.
  * `"shared"` : (Il s&#39;agit d&#39;une fonctionnalité expérimentale). Permet à plusieurs processus, éventuellement sur des machines différentes, de journaliser simultanément dans le même run. Dans cette approche, vous utilisez un nœud principal et un ou plusieurs nœuds worker pour journaliser des données dans le même run. Sur le nœud principal, vous initialisez un run. Pour chaque nœud worker, initialisez un run à l&#39;aide de l&#39;ID de run utilisé par le nœud principal.
* `force` : détermine si une connexion à W&amp;B est requise pour exécuter le script. Si `True`, l’utilisateur doit être connecté à W&amp;B ; sinon, le script ne s’exécutera pas. Si `False` (par défaut), le script peut s’exécuter sans connexion, en basculant en mode hors ligne si l’utilisateur n’est pas connecté.
* `reinit` :  Abréviation du paramètre « reinit ». Détermine le comportement de `wandb.init()` lorsqu’un run est actif.
* `resume` : contrôle le comportement lors de la reprise d’un run avec l’`id` spécifié. Les options disponibles sont :
  * `"allow"` : Si un run avec l’`id` spécifié existe, il reprendra à la dernière `étape` ; sinon, un nouveau run sera créé.
  * `"never"` : Si un run avec l’`id` spécifié existe, une erreur sera levée. Si aucun run correspondant n’est trouvé, un nouveau run sera créé.
  * `"must"` : Si un run avec l’`id` spécifié existe, il reprendra à la dernière `étape`. Si aucun run n’est trouvé, une erreur sera levée.
  * `"auto"` : Reprend automatiquement le run précédent s’il s’est arrêté de manière inattendue sur cette machine ; sinon, démarre un nouveau run.
  * `True` : Obsolète. Utilisez `"auto"` à la place.
  * `False` : Obsolète. Utilisez le comportement par défaut (en laissant `resume` non défini) pour toujours démarrer un nouveau run. Si `resume` est défini, `fork_from` et `resume_from` ne peuvent pas être utilisés. Lorsque `resume` n’est pas défini, le système démarrera toujours un nouveau run.
* `resume_from` :  Spécifie un point dans un run précédent à partir duquel reprendre un run,  au format `{run_id}?_step={step}`. Cela permet aux utilisateurs de tronquer  l&#39;historique enregistré pour un run à une étape intermédiaire, puis de reprendre l&#39;enregistrement  à partir de cette étape. Le run cible doit appartenir au même projet.  Si un argument `id` est également fourni, l&#39;argument `resume_from` sera  prioritaire.  `resume`, `resume_from` et `fork_from` ne peuvent pas être utilisés ensemble : un seul  peut être utilisé à la fois.  Notez que cette fonctionnalité est en version bêta et pourra évoluer à l&#39;avenir.
* `fork_from`:  Spécifie un point dans un run précédent à partir duquel créer un nouveau  run, en utilisant le format `{id}?_step={step}`. Cela crée un nouveau run qui  reprend la journalisation à partir de l’étape spécifiée dans l’historique du run cible.  Le run cible doit faire partie du projet actuel.  Si un argument `id` est également fourni, il doit être différent de l’argument  `fork_from` ; une erreur est levée s’ils sont identiques.  `resume`, `resume_from` et `fork_from` ne peuvent pas être utilisés ensemble : un seul  d’entre eux peut être utilisé à la fois.  Notez que cette fonctionnalité est en bêta et peut évoluer à l’avenir.
* `save_code` : permet d’enregistrer le script principal ou le notebook dans W&amp;B, ce qui améliore la reproductibilité des expériences et permet de comparer le code entre les runs dans l’interface utilisateur. Par défaut, cette option est désactivée, mais vous pouvez modifier ce paramètre par défaut pour l’activer sur votre page de paramètres.
* `tensorboard`:  Obsolète. Utilisez plutôt `sync_tensorboard`.
* `sync_tensorboard`:  Active la synchronisation automatique des journaux W&amp;B depuis TensorBoard ou TensorBoardX, en enregistrant les fichiers d’événements pertinents afin de pouvoir les consulter dans l’interface W&amp;B.
* `monitor_gym`:  Permet d’enregistrer automatiquement des vidéos de l’environnement lors de l’utilisation d’OpenAI Gym.
* `settings` : spécifie un dictionnaire ou un objet `wandb.Settings` contenant des paramètres avancés pour le run. 

**Retourne :**
Un objet `Run`.

**Exceptions levées :**

* `Error` :  Si une erreur interne ou inconnue se produit lors de l’initialisation du run.
* `AuthenticationError` :  Si l’utilisateur n’a pas fourni d’identifiants valides.
* `CommError` :  Si un problème de communication est survenu avec le serveur WandB.
* `UsageError` :  Si l’utilisateur a fourni des arguments non valides.
* `KeyboardInterrupt` :  Si l’utilisateur interrompt le run.

**Exemples :**
`wandb.init()` renvoie un objet `Run`. Utilisez l’objet run pour la journalisation des données, enregistrer des artefacts et gérer le cycle de vie du run.

```python
import wandb

config = {"lr": 0.01, "batch_size": 32}
with wandb.init(config=config) as run:
    # Journalisation de la précision et de la perte dans le run
    acc = 0.95  # Exemple de précision
    loss = 0.05  # Exemple de perte
    run.log({"accuracy": acc, "loss": loss})
```
