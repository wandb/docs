## <kbd>class</kbd> `Run`

Une unité de calcul journalisée par W&amp;B. Il s&#39;agit généralement d&#39;une expérience de ML.

Appelez [`wandb.init()`](https://docs.wandb.ai/ref/python/init/) pour créer un nouveau run. `wandb.init()` démarre un nouveau run et renvoie un objet `wandb.Run`. Chaque run est associé à un ID unique (ID du run). W&amp;B recommande d&#39;utiliser un gestionnaire de contexte (instruction `with`) pour terminer automatiquement le run.

Pour les expériences d&#39;entraînement distribué, vous pouvez soit suivre chaque processus séparément avec un run par processus, soit suivre tous les processus dans un seul run. Voir [Journaliser des expériences d&#39;entraînement distribué](https://docs.wandb.ai/guides/track/log/distributed-training) pour plus d&#39;informations.

Vous pouvez journaliser des données dans un run avec `wandb.Run.log()`. Tout ce que vous journalisez avec `wandb.Run.log()` est envoyé à ce run. Voir [Créer une expérience](https://docs.wandb.ai/guides/track/create-an-experiment/) ou la page de référence de l&#39;API [`wandb.init`](https://docs.wandb.ai/ref/python/init/) pour plus d&#39;informations.

Il existe également un autre objet `Run` dans l&#39;espace de noms [`wandb.apis.public`](https://docs.wandb.ai/ref/python/public-api/api/). Utilisez cet objet pour interagir avec des runs déjà créés.

**Attributs :**

* `summary`:  (Summary) Résumé du run, qui est un objet de type dictionnaire. Pour plus d&#39;informations, voir
* `[Journaliser les métriques de résumé](https`: //docs.wandb.ai/guides/track/log/log-summary/).

**Exemples :**
Créez un run avec `wandb.init()` :

```python
import wandb

# Démarrer un nouveau run et journaliser des données
# Utiliser le gestionnaire de contexte (instruction `with`) pour terminer automatiquement le run
with wandb.init(entity="entity", project="project") as run:
    run.log({"accuracy": acc, "loss": loss})
```

Personnalisez les métriques enregistrées avec `wandb.Run.log()`.

**Arguments :**

* `step_sync`:  Insère automatiquement la dernière valeur de step&#95;metric dans `wandb.Run.log()` si elle n’est pas fournie explicitement. La valeur par défaut est True si step&#95;metric est spécifié.
* `hidden`:  Masquez cette métrique dans les graphiques automatiques.
* `summary`:  Spécifiez les métriques agrégées ajoutées au résumé. Les agrégations prises en charge incluent &quot;min&quot;, &quot;max&quot;, &quot;mean&quot;, &quot;last&quot;, &quot;first&quot;, &quot;best&quot;, &quot;copy&quot; et &quot;none&quot;. &quot;none&quot; empêche la génération d’un résumé. &quot;best&quot; s’utilise avec le paramètre goal, mais &quot;best&quot; est obsolète et ne doit pas être utilisé ; utilisez plutôt &quot;min&quot; ou &quot;max&quot;. &quot;copy&quot; est obsolète et ne doit pas être utilisé.
* `goal`:  Spécifiez comment interpréter le type de résumé &quot;best&quot;. Les options prises en charge sont &quot;minimize&quot; et &quot;maximize&quot;. &quot;goal&quot; est obsolète et ne doit pas être utilisé ; utilisez plutôt &quot;min&quot; ou &quot;max&quot;.
* `overwrite`:  Si false, cet appel est fusionné avec les appels `define_metric` précédents pour la même métrique en utilisant leurs valeurs pour tous les paramètres non spécifiés. Si true, les paramètres non spécifiés remplacent les valeurs définies par les appels précédents.

**Retourne :**
Un objet qui représente cet appel, mais qui peut sinon être ignoré.

### <kbd>méthode</kbd> `Run.finish`

```python
finish(exit_code: 'int | None' = None, quiet: 'bool | None' = None) → None
```

Terminer un run et téléverser toutes les données restantes.

Marque la fin d’un run W&amp;B et garantit que toutes les données sont synchronisées avec le serveur. L’état final du run est déterminé par ses conditions de sortie et son statut de synchronisation.

États du run :

* En cours d’exécution : run actif qui consigne des données et/ou envoie des heartbeats.
* Planté : run qui a cessé d’envoyer des heartbeats de manière inattendue.
* Terminé : run terminé avec succès (`exit_code=0`) et dont toutes les données sont synchronisées.
* Échoué : run terminé avec des erreurs (`exit_code!=0`).
* Interrompu : le run a été arrêté de force avant d’avoir pu se terminer.

**Arguments :**

* `exit_code` :  Entier indiquant le statut de sortie du run. Utilisez 0 pour indiquer une réussite ; toute autre valeur marque le run comme échoué.
* `quiet` :  Obsolète. Configurez la verbosité de la journalisation à l’aide de `wandb.Settings(quiet=...)`.

### <kbd>méthode</kbd> `Run.finish_artifact`

```python
finish_artifact(
    artifact_or_path: 'Artifact | str',
    name: 'str | None' = None,
    type: 'str | None' = None,
    aliases: 'list[str] | None' = None,
    distributed_id: 'str | None' = None
) → Artifact
```

Finalise un artefact non finalisé en sortie d’un run.

Les « upserts » suivants avec le même ID distribué créeront une nouvelle version.

**Arguments :**

* `artifact_or_path` :  Chemin vers le contenu de cet artefact. Il peut prendre l’une des formes suivantes :
  * `/local/directory`
  * `/local/directory/file.txt`
  * `s3://bucket/path`  Vous pouvez également passer un objet `Artifact` créé en appelant `wandb.Artifact`.
* `name` :  Nom d’artefact. Peut être préfixé par entité/projet. Les noms valides peuvent prendre les formes suivantes :
  * name:version
  * name:alias
  * digest  Si ce paramètre n’est pas spécifié, la valeur par défaut sera le nom de base du chemin, précédé de l’ID du run actuel.
* `type` :  Type d’artefact à journaliser ; par exemple `dataset`, `model`
* `aliases` :  Alias à appliquer à cet artefact. La valeur par défaut est `["latest"]`
* `distributed_id` :  Chaîne unique partagée par tous les jobs distribués. Si None, la valeur par défaut est le nom du groupe du run.

**Retourne :**
Un objet `Artifact`.

### <kbd>méthode</kbd> `Run.link_artifact`

```python
link_artifact(
    artifact: 'Artifact',
    target_path: 'str',
    aliases: 'list[str] | None' = None
) → Artifact
```

Liez l’artefact à une collection.

Le terme « link » désigne les pointeurs qui relient l’emplacement où W&amp;B stocke l’artefact à celui où l’artefact est accessible dans le registre. W&amp;B ne duplique pas les artefacts lorsque vous liez un artefact à une collection.

Consultez les artefacts liés dans l’interface utilisateur du registre pour la collection spécifiée.

**Arguments :**

* `artifact`:  L’objet artefact à lier à la collection.
* `target_path`:  Le chemin de la collection. Il se compose du préfixe &quot;wandb-registry-&quot;, ainsi que du nom du registre et du nom de la collection : `wandb-registry-{REGISTRY_NAME}/{COLLECTION_NAME}`.
* `aliases`:  Ajoutez un ou plusieurs alias à l’artefact lié. L’alias &quot;latest&quot; est automatiquement appliqué à l’artefact le plus récent que vous liez.

**Retourne :**
L’artefact lié.

### <kbd>méthode</kbd> `Run.link_model`

```python
link_model(
    path: 'StrPath',
    registered_model_name: 'str',
    name: 'str | None' = None,
    aliases: 'list[str] | None' = None
) → Artifact | None
```

Journalisez une version d’artefact de modèle et liez-la à un modèle enregistré dans le registre des modèles.

Les versions de modèle liées sont visibles dans l’interface utilisateur pour le modèle enregistré spécifié.

Cette méthode effectue les opérations suivantes :

* Vérifie si l’artefact de modèle &#39;name&#39; a déjà été journalisé. Si c’est le cas, elle utilise la version de l’artefact qui correspond aux fichiers situés à &#39;path&#39;, ou en journalise une nouvelle version. Sinon, elle journalise les fichiers sous &#39;path&#39; comme un nouvel artefact de modèle nommé &#39;name&#39; et de type &#39;model&#39;.
* Vérifie si un modèle enregistré portant le nom &#39;registered&#95;model&#95;name&#39; existe dans le projet &#39;model-registry&#39;. Sinon, elle crée un nouveau modèle enregistré nommé &#39;registered&#95;model&#95;name&#39;.
* Lie la version de l’artefact de modèle &#39;name&#39; au modèle enregistré &#39;registered&#95;model&#95;name&#39;.
* Associe les alias de la liste &#39;aliases&#39; à la version de l’artefact de modèle nouvellement liée.

**Arguments :**

* `path`:  (str) Chemin vers le contenu de ce modèle. Il peut prendre l’une des formes suivantes :
  * `/local/directory`
  * `/local/directory/file.txt`
  * `s3://bucket/path`
* `registered_model_name`:  Nom du modèle enregistré auquel le modèle doit être lié. Un modèle enregistré est une collection de versions de modèle liées au registre des modèles, représentant généralement une tâche de ML spécifique à une équipe. L’entité à laquelle appartient ce modèle enregistré sera déduite du run.
* `name`:  Nom de l’artefact de modèle dans lequel les fichiers de &#39;path&#39; seront journalisés. Par défaut, si ce paramètre n’est pas spécifié, il s’agira du nom de base du chemin, préfixé par l’ID du run actuel.
* `aliases`:  Alias qui seront appliqués uniquement à cet artefact lié dans le modèle enregistré. L’alias &quot;latest&quot; sera toujours appliqué à la dernière version d’un artefact lié.

**Exceptions levées :**

* `AssertionError`:  Si registered&#95;model&#95;name est un chemin, ou si l’artefact de modèle &#39;name&#39; est d’un type qui ne contient pas la sous-chaîne &#39;model&#39;.
* `ValueError`:  Si name contient des caractères spéciaux non valides.

**Retourne :**
L’artefact lié si l’opération de liaison a réussi, sinon `None`.

### <kbd>méthode</kbd> `Run.log`

```python
log(
    data: 'dict[str, Any]',
    step: 'int | None' = None,
    commit: 'bool | None' = None
) → None
```

Téléverser les données du run.

Utilisez `log` pour journaliser les données des runs, comme des scalaires, des images, des vidéos, des histogrammes, des graphiques et des tableaux. Voir [Journaliser des objets et des médias](https://docs.wandb.ai/guides/track/log) pour des extraits de code, des bonnes pratiques, etc.

Utilisation de base :

```python
import wandb

with wandb.init() as run:
     run.log({"train-loss": 0.5, "accuracy": 0.9})
```

L’extrait de code précédent enregistre la perte et la précision dans l’historique du run, et met à jour les valeurs de synthèse de ces métriques.

Visualisez les données journalisées dans un espace de travail sur [wandb.ai](https://wandb.ai), localement sur une [instance auto-hébergée](https://docs.wandb.ai/guides/hosting) de l’application W&amp;B, ou exportez les données pour les visualiser et les explorer en local, par exemple dans un notebook Jupyter, à l’aide de l’[API publique](https://docs.wandb.ai/guides/track/public-api-guide).

Les valeurs journalisées ne doivent pas forcément être des scalaires. Vous pouvez journaliser n’importe quel [type de données pris en charge par W&amp;B](https://docs.wandb.ai/ref/python/data-types/), comme des images, de l’audio, de la vidéo, etc. Par exemple, vous pouvez utiliser `wandb.Table` pour journaliser des données structurées. Voir le tutoriel [Log tables, visualize and query data](https://docs.wandb.ai/guides/models/tables/tables-walkthrough) pour plus de détails.

W&amp;B organise les métriques dont le nom contient une barre oblique (`/`) en sections, nommées d’après le texte qui précède la dernière barre oblique. Par exemple, ce qui suit crée deux sections nommées &quot;train&quot; et &quot;validate&quot; :

```python
with wandb.init() as run:
     # Consigner les métriques dans la section "train".
     run.log(
         {
             "train/accuracy": 0.9,
             "train/loss": 30,
             "validate/accuracy": 0.8,
             "validate/loss": 20,
         }
     )
```

Un seul niveau d’imbrication est pris en charge ; `run.log({"a/b/c": 1})` produit une section appelée &quot;a&quot;.

`run.log()` n’est pas conçu pour être appelé plus de quelques fois par seconde. Pour des performances optimales, limitez la journalisation à une fois toutes les N itérations, ou collectez des données sur plusieurs itérations et consignez-les en une seule étape.

Par défaut, chaque appel à `log` crée une nouvelle &quot;étape&quot;. L’étape doit toujours augmenter, et il n’est pas possible de consigner des données dans une étape antérieure. Vous pouvez utiliser n’importe quelle métrique comme axe X dans les graphiques. Voir [Axes de journalisation personnalisés](https://docs.wandb.ai/guides/track/log/customize-logging-axes/) pour plus de détails.

Dans de nombreux cas, il est préférable de traiter l’étape W&amp;B comme un horodatage plutôt que comme une étape d’entraînement.

```python
with wandb.init() as run:
     # Exemple : journaliser une métrique "epoch" à utiliser comme axe X.
     run.log({"epoch": 40, "train-loss": 0.5})
```

Il est possible d’utiliser plusieurs appels à `wandb.Run.log()` pour journaliser sur la même étape avec les paramètres `step` et `commit`. Les éléments suivants sont équivalents :

```python
with wandb.init() as run:
     # Utilisation normale :
     run.log({"train-loss": 0.5, "accuracy": 0.8})
     run.log({"train-loss": 0.4, "accuracy": 0.9})

     # Étape implicite sans auto-incrémentation :
     run.log({"train-loss": 0.5}, commit=False)
     run.log({"accuracy": 0.8})
     run.log({"train-loss": 0.4}, commit=False)
     run.log({"accuracy": 0.9})

     # Étape explicite :
     run.log({"train-loss": 0.5}, step=current_step)
     run.log({"accuracy": 0.8}, step=current_step)
     current_step += 1
     run.log({"train-loss": 0.4}, step=current_step)
     run.log({"accuracy": 0.9}, step=current_step, commit=True)
```

**Arguments :**


* `data`:  Un `dict` avec des clés `str` et des valeurs sérialisables
* `objets Python, notamment`:  `int`, `float` et `string` ; n’importe lequel des `wandb.data_types` ; des listes, tuples et tableaux NumPy d’objets Python sérialisables ; d’autres `dict` de cette structure.
* `step`:  Le numéro d’étape à journaliser. Si `None`, une étape implicite auto-incrémentée est utilisée. Voir les notes dans la description.
* `commit`:  Si la valeur est true, finalisez et téléversez l’étape. Si la valeur est false, les données sont accumulées pour l’étape. Voir les notes dans la description. Si `step` est `None`, la valeur par défaut est `commit=True` ; sinon, la valeur par défaut est `commit=False`.

### <kbd>méthode</kbd> `Run.log_artifact`

```python
log_artifact(
    artifact_or_path: 'Artifact | StrPath',
    name: 'str | None' = None,
    type: 'str | None' = None,
    aliases: 'list[str] | None' = None,
    tags: 'list[str] | None' = None
) → Artifact
```

Déclare un artefact comme sortie d’un run.

**Arguments :**

* `artifact_or_path`:  (str ou Artifact) Chemin vers le contenu de cet artefact. Il peut prendre les formes suivantes :
  * `/local/directory`
  * `/local/directory/file.txt`
  * `s3://bucket/path`  Vous pouvez également passer un objet `Artifact` créé en appelant `wandb.Artifact`.
* `name`:  (str, facultatif) Nom d’artefact. Les noms valides peuvent prendre les formes suivantes :
  * name:version
  * name:alias
  * digest  Si ce paramètre n’est pas spécifié, la valeur par défaut sera le nom de base du chemin, préfixé par l’ID du run actuel.
* `type`:  (str) Type d’artefact à journaliser ; par exemple `dataset`, `model`
* `aliases`:  (liste, facultatif) Alias à appliquer à cet artefact. La valeur par défaut est `["latest"]`
* `tags`:  (liste, facultatif) Étiquettes à appliquer à cet artefact, le cas échéant.

**Retourne :**
Un objet `Artifact`.

### <kbd>méthode</kbd> `Run.log_code`

```python
log_code(
    root: 'str | None' = '.',
    name: 'str | None' = None,
    include_fn: 'Callable[[str, str], bool] | Callable[[str], bool]' = <function _is_py_requirements_or_dockerfile at 0x10998dda0>,
    exclude_fn: 'Callable[[str, str], bool] | Callable[[str], bool]' = <function exclude_wandb_fn at 0x10b11bb00>
) → Artifact | None
```

Enregistrez l’état actuel de votre code dans un artefact W&amp;B.

Par défaut, cette méthode parcourt le répertoire actuel et journalise tous les fichiers qui se terminent par `.py`.

**Arguments :**

* `root`:  Le chemin relatif (par rapport à `os.getcwd()`) ou absolu à partir duquel rechercher récursivement le code.
* `name`:  (str, facultatif) Le nom de notre artefact de code. Par défaut, l’artefact sera nommé `source-$PROJECT_ID-$ENTRYPOINT_RELPATH`. Dans certains cas, vous pouvez vouloir que plusieurs runs partagent le même artefact. Spécifier un nom vous permet de le faire.
* `include_fn`:  Un callable qui accepte un chemin de fichier et (facultativement) un chemin racine, puis renvoie True s’il doit être inclus et False sinon. Cette
* `defaults to `lambda path, root`:  path.endswith(".py")`.
* `exclude_fn`:  Un callable qui accepte un chemin de fichier et (facultativement) un chemin racine, puis renvoie `True` s’il doit être exclu et `False` sinon. Par défaut, cette option utilise une fonction qui exclut tous les fichiers des répertoires `<root>/.wandb/` et `<root>/wandb/`.

**Exemples :**
Utilisation de base

```python
import wandb

with wandb.init() as run:
    run.log_code()
```

Utilisation avancée

```python
import wandb

with wandb.init() as run:
    run.log_code(
         root="../",
         include_fn=lambda path: path.endswith(".py") or path.endswith(".ipynb"),
         exclude_fn=lambda path, root: os.path.relpath(path, root).startswith(
             "cache/"
         ),
    )
```

**Retourne :**
Un objet `Artifact` si le code a été journalisé

### <kbd>méthode</kbd> `Run.restore`

```python
restore(
    name: 'str',
    run_path: 'str | None' = None,
    replace: 'bool' = False,
    root: 'str | None' = None
) → None | TextIO
```

Télécharge le fichier spécifié depuis le stockage dans le cloud.

Le fichier est placé dans le répertoire actuel ou dans le répertoire du run. Par défaut, le fichier n’est téléchargé que s’il n’existe pas déjà.

**Arguments :**

* `name`:  Le nom du fichier.
* `run_path`:  Chemin facultatif vers un run depuis lequel récupérer des fichiers, c.-à-d. `username/project_name/run_id`  si wandb.init n’a pas été appelé, ce paramètre est requis.
* `replace`:  Indique s’il faut télécharger le fichier même s’il existe déjà localement
* `root`:  Le répertoire dans lequel télécharger le fichier.  La valeur par défaut est le répertoire actuel ou le répertoire du run si wandb.init a été appelé.

**Retourne :**
`None` si le fichier est introuvable, sinon un objet de fichier ouvert en lecture.

**Exceptions levées :**

* `CommError`:  Si W&amp;B ne peut pas se connecter au backend W&amp;B.
* `ValueError`:  Si le fichier est introuvable ou si `run_path` est introuvable.

### <kbd>méthode</kbd> `Run.save`

```python
save(
    glob_str: 'str | os.PathLike',
    base_path: 'str | os.PathLike | None' = None,
    policy: 'PolicyName' = 'live'
) → bool | list[str]
```

Synchronise un ou plusieurs fichiers avec W&amp;B.

Les chemins relatifs sont interprétés par rapport au répertoire de travail courant.

Un glob Unix, tel que &quot;myfiles/*&quot;, est développé au moment de l&#39;appel à `save`, quelle que soit la valeur de `policy`. En particulier, les nouveaux fichiers ne sont pas pris en compte automatiquement.

Vous pouvez fournir un `base_path` pour contrôler la structure de répertoires des fichiers téléversés. Il doit être un préfixe de `glob_str`, et la structure de répertoires située en dessous est conservée.

Lorsqu&#39;un chemin absolu ou un glob est fourni sans `base_path`, un niveau de répertoire est conservé, comme dans l&#39;exemple ci-dessus.

Les fichiers sont automatiquement dédupliqués : appeler `save()` plusieurs fois sur le même fichier sans modification ne le retéléversera pas.

**Arguments :**

* `glob_str`:  Un chemin relatif ou absolu, ou un glob Unix.
* `base_path`:  Un chemin utilisé pour déduire une structure de répertoires ; voir les exemples.
* `policy`:  L&#39;une des valeurs `live`, `now` ou `end`.
  * live: téléverser le fichier au fur et à mesure de ses modifications, en écrasant la version précédente
  * now: téléverser le fichier une seule fois immédiatement
  * end: téléverser le fichier lorsque le run se termine

**Retourne :**
Les chemins vers les liens symboliques créés pour les fichiers correspondants.

Pour des raisons historiques, cette méthode peut renvoyer un booléen dans du code existant.

```python
import wandb

run = wandb.init()

run.save("these/are/myfiles/*")
# => Enregistre les fichiers dans un dossier "these/are/myfiles/" dans le run.

run.save("these/are/myfiles/*", base_path="these")
# => Enregistre les fichiers dans un dossier "are/myfiles/" dans le run.

run.save("/Users/username/Documents/run123/*.txt")
# => Enregistre les fichiers dans un dossier "run123/" dans le run. Voir la note ci-dessous.

run.save("/Users/username/Documents/run123/*.txt", base_path="/Users")
# => Enregistre les fichiers dans un dossier "username/Documents/run123/" dans le run.

run.save("files/*/saveme.txt")
# => Enregistre chaque fichier "saveme.txt" dans un sous-répertoire approprié
#    de "files/".

# Terminer explicitement le run car aucun gestionnaire de contexte n'est utilisé.
run.finish()
```

### <kbd>méthode</kbd> `Run.upsert_artifact`

```python
upsert_artifact(
    artifact_or_path: 'Artifact | str',
    name: 'str | None' = None,
    type: 'str | None' = None,
    aliases: 'list[str] | None' = None,
    distributed_id: 'str | None' = None
) → Artifact
```

Déclarez un artefact non finalisé en tant que sortie d’un run (ou ajoutez-y du contenu).

Notez que vous devez appeler run.finish&#95;artifact() pour finaliser l’artefact. Cela est utile lorsque des jobs distribués doivent tous contribuer au même artefact.

**Arguments :**

* `artifact_or_path`:  Chemin vers le contenu de cet artefact, pouvant prendre les formes suivantes :
  * `/local/directory`
  * `/local/directory/file.txt`
  * `s3://bucket/path`
* `name`:  Un nom d’artefact. Peut être préfixé par &quot;entité/projet&quot;. Par défaut, si aucun nom n’est spécifié, il s’agit du nom de base du chemin précédé de l’ID du run actuel. Les noms valides peuvent prendre les formes suivantes :
  * name:version
  * name:alias
  * digest
* `type`:  Le type d’artefact à journaliser. Exemples courants : `dataset`, `model`.
* `aliases`:  Alias à appliquer à cet artefact. La valeur par défaut est `["latest"]`.
* `distributed_id`:  Chaîne unique partagée par tous les jobs distribués. Si `None`, la valeur par défaut est le nom du groupe du run.

**Retourne :**
Un objet `Artifact`.

### <kbd>méthode</kbd> `Run.use_artifact`

```python
use_artifact(
    artifact_or_name: 'str | Artifact',
    type: 'str | None' = None,
    aliases: 'list[str] | None' = None,
    use_as: 'str | None' = None
) → Artifact
```

Déclare un artefact en tant qu&#39;entrée d&#39;un run.

Appelez `download` ou `file` sur l&#39;objet renvoyé pour récupérer le contenu en local.

**Arguments :**

* `artifact_or_name`:  Le nom de l&#39;artefact à utiliser. Il peut être préfixé par le nom du projet dans lequel l&#39;artefact a été enregistré  (&quot;entity&quot; ou &quot;entity/project&quot;). Si aucune  entité n&#39;est indiquée dans le nom, l&#39;entité du run ou de la configuration de l&#39;API est utilisée.  Les noms valides peuvent prendre les formes suivantes
  * nom:version
  * nom:alias
* `type`:  Le type d&#39;artefact à utiliser.
* `aliases`:  Les alias à appliquer à cet artefact
* `use_as`:  Cet argument est obsolète et n&#39;a aucun effet.

**Retourne :**
Un objet `Artifact`.

**Exemples :**

```python
import wandb

run = wandb.init(project="<example>")

# Utiliser un artefact par nom et alias
artifact_a = run.use_artifact(artifact_or_name="<name>:<alias>")

# Utiliser un artefact par nom et version
artifact_b = run.use_artifact(artifact_or_name="<name>:v<version>")

# Utiliser un artefact par entité/projet/nom:alias
artifact_c = run.use_artifact(
    artifact_or_name="<entity>/<project>/<name>:<alias>"
)

# Utiliser un artefact par entité/projet/nom:version
artifact_d = run.use_artifact(
    artifact_or_name="<entity>/<project>/<name>:v<version>"
)

# Terminer explicitement le run car aucun gestionnaire de contexte n'est utilisé.
run.finish()
```

### <kbd>méthode</kbd> `Run.watch`

```python
watch(
    models: 'torch.nn.Module | Sequence[torch.nn.Module]',
    criterion: 'torch.F | None' = None,
    log: "Literal['gradients', 'parameters', 'all'] | None" = 'gradients',
    log_freq: 'int' = 1000,
    idx: 'int | None' = None,
    log_graph: 'bool' = False
) → None
```

Se connecte au modèle PyTorch fourni pour surveiller les gradients et le graphe de calcul du modèle.

Cette fonction peut suivre les paramètres, les gradients ou les deux pendant l’entraînement.

**Arguments :**

* `models`:  Un modèle unique ou une séquence de modèles à surveiller.
* `criterion`:  La fonction de perte en cours d’optimisation (facultatif).
* `log`:  Indique s’il faut journaliser `"gradients"`, `"parameters"` ou `"all"`. Définissez cette valeur sur None pour désactiver la journalisation. (default=&quot;gradients&quot;).
* `log_freq`:  Fréquence (en lots) de journalisation des gradients et des paramètres. (default=1000)
* `idx`:  Index utilisé lors du suivi de modèles multiples avec `wandb.watch`. (default=None)
* `log_graph`:  Indique s’il faut journaliser le graphe de calcul du modèle. (default=False)

**Exceptions levées :**
ValueError:  Si `wandb.init()` n’a pas été appelé ou si l’un des modèles n’est pas une instance de `torch.nn.Module`.
