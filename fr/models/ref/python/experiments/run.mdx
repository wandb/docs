## <kbd>classe</kbd> `Run`

Une unité de calcul enregistrée par W&amp;B. Il s&#39;agit généralement d&#39;une expérience de ML.

Appelez [`wandb.init()`](https://docs.wandb.ai/ref/python/init/) pour créer un nouveau run. `wandb.init()` démarre un nouveau run et renvoie un objet `wandb.Run`. Chaque run est associé à un ID unique (ID du run). W&amp;B recommande d&#39;utiliser un gestionnaire de contexte (instruction `with`) pour terminer automatiquement le run.

Pour les expériences d&#39;entraînement distribué, vous pouvez soit suivre chaque processus séparément, avec un run par processus, soit suivre tous les processus dans un seul run. Voir [Journaliser des expériences d&#39;entraînement distribué](https://docs.wandb.ai/guides/track/log/distributed-training) pour plus d&#39;informations.

Vous pouvez enregistrer des données sur un run avec `wandb.Run.log()`. Tout ce que vous enregistrez avec `wandb.Run.log()` est envoyé à ce run. Voir [Créer une expérience](https://docs.wandb.ai/guides/track/create-an-experiment/) ou la page de référence de l&#39;API [`wandb.init`](https://docs.wandb.ai/ref/python/init/) pour plus d&#39;informations.

Il existe également un autre objet `Run` dans l&#39;espace de noms [`wandb.apis.public`](https://docs.wandb.ai/ref/python/public-api/api/). Utilisez cet objet pour interagir avec des runs déjà créés.

**Attributs :**

* `summary`:  (Summary) Synthèse du run, sous la forme d&#39;un objet de type dictionnaire. Pour plus d&#39;informations, voir
* `[Journaliser les métriques de synthèse](https`: //docs.wandb.ai/guides/track/log/log-summary/).

**Exemples :**
Créez un run avec `wandb.init()` :

```python
import wandb

# Démarrer un nouveau run et journaliser des données
# Utiliser le gestionnaire de contexte (instruction `with`) pour terminer automatiquement le run
with wandb.init(entity="entity", project="project") as run:
    run.log({"accuracy": acc, "loss": loss})
```

define_metric(
    name: 'str',

    step_sync: 'bool | None' = None,

Personnalisez les métriques enregistrées avec `wandb.Run.log()`.

**Arguments :**

* `step_sync`:  Insère automatiquement la dernière valeur de step&#95;metric dans `wandb.Run.log()` si elle n’est pas fournie explicitement. La valeur par défaut est True si step&#95;metric est spécifié.
* `hidden`:  Masquez cette métrique dans les graphiques automatiques.
* `summary`:  Spécifiez les métriques agrégées ajoutées à la synthèse. Les agrégations prises en charge incluent &quot;min&quot;, &quot;max&quot;, &quot;mean&quot;, &quot;last&quot;, &quot;first&quot;, &quot;best&quot;, &quot;copy&quot; et &quot;none&quot;. &quot;none&quot; empêche la génération d’une synthèse. &quot;best&quot; s’utilise avec le paramètre goal, mais &quot;best&quot; est obsolète et ne doit pas être utilisé ; utilisez plutôt &quot;min&quot; ou &quot;max&quot;. &quot;copy&quot; est obsolète et ne doit pas être utilisé.
* `goal`:  Spécifiez comment interpréter le type de synthèse &quot;best&quot;. Les options prises en charge sont &quot;minimize&quot; et &quot;maximize&quot;. &quot;goal&quot; est obsolète et ne doit pas être utilisé ; utilisez plutôt &quot;min&quot; ou &quot;max&quot;.
* `overwrite`:  Si false, cet appel est fusionné avec les appels `define_metric` précédents pour la même métrique en utilisant leurs valeurs pour tous les paramètres non spécifiés. Si true, les paramètres non spécifiés remplacent les valeurs définies par les appels précédents.

**Retourne :**
Un objet qui représente cet appel, mais qui peut sinon être ignoré.

### <kbd>méthode</kbd> `Run.finish`

```python
finish(exit_code: 'int | None' = None, quiet: 'bool | None' = None) → None
```

Terminer un run et téléverser toutes les données restantes.

Marque la fin d&#39;un run W&amp;B et garantit que toutes les données sont synchronisées avec le serveur. L&#39;état final du run est déterminé par ses conditions de sortie et son statut de synchronisation.

États des runs :

* En cours d&#39;exécution : run actif qui consigne des données et/ou envoie des heartbeats.
* Planté : run qui a cessé d&#39;envoyer des heartbeats de manière inattendue.
* Terminé : run terminé avec succès (`exit_code=0`) et dont toutes les données sont synchronisées.
* Échoué : run terminé avec des erreurs (`exit_code!=0`).
* Interrompu : le run a été arrêté de force avant d&#39;avoir pu se terminer.

**Arguments :**

* `exit_code` :  Entier indiquant le statut de sortie du run. Utilisez 0 pour indiquer une réussite ; toute autre valeur marque le run comme échoué.
* `quiet` :  Obsolète. Configurez la verbosité de la journalisation à l&#39;aide de `wandb.Settings(quiet=...)`.

### <kbd>méthode</kbd> `Run.finish_artifact`

```python
finish_artifact(
    artifact_or_path: 'Artifact | str',
    name: 'str | None' = None,
    type: 'str | None' = None,
    aliases: 'list[str] | None' = None,
    distributed_id: 'str | None' = None
) → Artifact
```

Finalise un artefact non finalisé en sortie d&#39;un run.

Les « upserts » suivants avec le même ID distribué créeront une nouvelle version.

**Arguments :**

* `artifact_or_path` :  Chemin vers le contenu de cet artefact. Il peut prendre l&#39;une des formes suivantes :
  * `/local/directory`
  * `/local/directory/file.txt`
  * `s3://bucket/path`  Vous pouvez également passer un objet `Artifact` créé en appelant `wandb.Artifact`.
* `name` :  Nom d&#39;artefact. Peut être préfixé par entité/projet. Les noms valides peuvent prendre les formes suivantes :
  * name:version
  * name:alias
  * digest  Si ce paramètre n&#39;est pas spécifié, la valeur par défaut sera le nom de base du chemin, précédé de l&#39;ID du run actuel.
* `type` :  Type d&#39;artefact à journaliser ; par exemple `dataset`, `model`
* `aliases` :  Alias à appliquer à cet artefact. La valeur par défaut est `["latest"]`
* `distributed_id` :  Chaîne unique partagée par tous les jobs distribués. Si None, la valeur par défaut est le nom du groupe du run.

**Retourne :**
Un objet `Artifact`.

### <kbd>méthode</kbd> `Run.link_artifact`

```python
link_artifact(
    artifact: 'Artifact',
    target_path: 'str',
    aliases: 'list[str] | None' = None
) → Artifact
```

Liez l’artefact à une collection.

Le terme « lien » désigne les pointeurs qui relient l’emplacement où W&amp;B stocke l’artefact à celui où l’artefact est accessible dans le registre. W&amp;B ne duplique pas les artefacts lorsque vous liez un artefact à une collection.

Consultez les artefacts liés dans l’interface utilisateur du registre pour la collection spécifiée.

**Arguments :**

* `artifact`:  L’objet artefact à lier à la collection.
* `target_path`:  Le chemin de la collection. Il se compose du préfixe &quot;wandb-registry-&quot;, ainsi que du nom du registre et du nom de la collection : `wandb-registry-{REGISTRY_NAME}/{COLLECTION_NAME}`.
* `aliases`:  Ajoutez un ou plusieurs alias à l’artefact lié. L’alias &quot;latest&quot; est automatiquement appliqué à l’artefact le plus récent que vous liez.

**Retourne :**
L’artefact lié.

### <kbd>méthode</kbd> `Run.link_model`

```python
link_model(
    path: 'StrPath',
    registered_model_name: 'str',
    name: 'str | None' = None,
    aliases: 'list[str] | None' = None
) → Artifact | None
```

Enregistrez une version d’artefact de modèle et liez-la à un modèle enregistré dans le registre de modèles.

Les versions de modèle liées sont visibles dans l’interface utilisateur pour le modèle enregistré spécifié.

Cette méthode effectue les opérations suivantes :

* Vérifie si l’artefact de modèle &#39;name&#39; a été enregistré. Si c’est le cas, utilise la version d’artefact correspondant aux fichiers situés à &#39;path&#39; ou enregistre une nouvelle version. Sinon, enregistre les fichiers sous &#39;path&#39; comme un nouvel artefact de modèle, &#39;name&#39;, de type &#39;model&#39;.
* Vérifie si un modèle enregistré nommé &#39;registered&#95;model&#95;name&#39; existe dans le projet &#39;model-registry&#39;. Sinon, crée un nouveau modèle enregistré nommé &#39;registered&#95;model&#95;name&#39;.
* Lie la version de l’artefact de modèle &#39;name&#39; au modèle enregistré &#39;registered&#95;model&#95;name&#39;.
* Attache les alias de la liste &#39;aliases&#39; à la version d’artefact de modèle nouvellement liée.

**Arguments :**

* `path`:  (str) Chemin vers le contenu de ce modèle, qui peut prendre les formes suivantes :
  * `/local/directory`
  * `/local/directory/file.txt`
  * `s3://bucket/path`
* `registered_model_name`:  Le nom du modèle enregistré auquel le modèle doit être lié. Un modèle enregistré est une collection de versions de modèle liées au registre de modèles, représentant généralement une tâche de ML spécifique à une équipe. L’entité à laquelle appartient ce modèle enregistré est dérivée du run.
* `name`:  Le nom de l’artefact de modèle dans lequel les fichiers de &#39;path&#39; seront enregistrés. Par défaut, il s’agit du nom de base du chemin, préfixé par l’ID du run actuel s’il n’est pas spécifié.
* `aliases`:  Alias appliqués uniquement à cet artefact lié dans le modèle enregistré. L’alias &quot;latest&quot; est toujours appliqué à la dernière version d’un artefact lié.

**Exceptions levées :**

* `AssertionError`:  Si registered&#95;model&#95;name est un chemin, ou si l’artefact de modèle &#39;name&#39; est d’un type qui ne contient pas la sous-chaîne &#39;model&#39;.
* `ValueError`:  Si name contient des caractères spéciaux non valides.

**Retourne :**
L’artefact lié si la liaison a réussi, sinon `None`.

### <kbd>méthode</kbd> `Run.log`

```python
log(
    data: 'dict[str, Any]',
    step: 'int | None' = None,
    commit: 'bool | None' = None
) → None
```

Téléverser les données du run.

Utilisez `log` pour journaliser les données des runs, comme des scalaires, des images, des vidéos, des histogrammes, des graphiques et des tableaux. Voir [Journaliser des objets et des médias](https://docs.wandb.ai/guides/track/log) pour des extraits de code, des bonnes pratiques, et plus encore.

Utilisation de base :

```python
import wandb

with wandb.init() as run:
     run.log({"train-loss": 0.5, "accuracy": 0.9})
```

L&#39;extrait de code précédent enregistre la perte et la précision dans l&#39;historique du run, et met à jour les valeurs de synthèse de ces métriques.

Visualisez les données journalisées dans un Workspace sur [wandb.ai](https://wandb.ai), localement sur une [instance auto-hébergée](https://docs.wandb.ai/guides/hosting) de l&#39;application W&amp;B, ou exportez les données pour les visualiser et les explorer en local, par exemple dans un notebook Jupyter, à l&#39;aide de l&#39;[API publique](https://docs.wandb.ai/guides/track/public-api-guide).

Les valeurs journalisées ne doivent pas forcément être des scalaires. Vous pouvez journaliser n&#39;importe quel [type de données pris en charge par W&amp;B](https://docs.wandb.ai/ref/python/data-types/), comme des images, de l&#39;audio, de la vidéo, etc. Par exemple, vous pouvez utiliser `wandb.Table` pour journaliser des données structurées. Voir le tutoriel [Log tables, visualize and query data](https://docs.wandb.ai/guides/models/tables/tables-walkthrough) pour plus de détails.

W&amp;B organise les métriques dont le nom contient une barre oblique (`/`) en sections, nommées d&#39;après le texte qui précède la dernière barre oblique. Par exemple, ce qui suit crée deux sections nommées &quot;train&quot; et &quot;validate&quot; :

```python
with wandb.init() as run:
     # Consigner les métriques dans la section "train".
     run.log(
         {
             "train/accuracy": 0.9,
             "train/loss": 30,
             "validate/accuracy": 0.8,
             "validate/loss": 20,
         }
     )
```

Un seul niveau d&#39;imbrication est pris en charge ; `run.log({"a/b/c": 1})` produit une section nommée &quot;a&quot;.

`run.log()` n&#39;est pas conçu pour être appelé plus de quelques fois par seconde. Pour des performances optimales, limitez la journalisation à une fois toutes les N itérations, ou collectez des données sur plusieurs itérations et journalisez-les en une seule étape.

Par défaut, chaque appel à `log` crée une nouvelle &quot;étape&quot;. L&#39;étape doit toujours être croissante, et il n&#39;est pas possible de journaliser des données à une étape précédente. Vous pouvez utiliser n&#39;importe quelle métrique comme axe X dans les graphiques. Voir [Axes de journalisation personnalisés](https://docs.wandb.ai/guides/track/log/customize-logging-axes/) pour plus de détails.

Dans de nombreux cas, il est préférable de traiter l&#39;étape W&amp;B comme un horodatage plutôt que comme une étape d&#39;entraînement.

```python
with wandb.init() as run:
     # Exemple : journaliser une métrique "epoch" à utiliser comme axe X.
     run.log({"epoch": 40, "train-loss": 0.5})
```

Il est possible d&#39;utiliser plusieurs appels à `wandb.Run.log()` pour journaliser dans la même étape à l&#39;aide des paramètres `step` et `commit`. Les éléments suivants sont équivalents :

```python
with wandb.init() as run:
     # Utilisation normale :
     run.log({"train-loss": 0.5, "accuracy": 0.8})
     run.log({"train-loss": 0.4, "accuracy": 0.9})

     # Étape implicite sans auto-incrémentation :
     run.log({"train-loss": 0.5}, commit=False)
     run.log({"accuracy": 0.8})
     run.log({"train-loss": 0.4}, commit=False)
     run.log({"accuracy": 0.9})

     # Étape explicite :
     run.log({"train-loss": 0.5}, step=current_step)
     run.log({"accuracy": 0.8}, step=current_step)
     current_step += 1
     run.log({"train-loss": 0.4}, step=current_step)
     run.log({"accuracy": 0.9}, step=current_step, commit=True)
```

**Arguments :**


* `data`:  Un `dict` avec des clés de type `str` et des valeurs sérialisables
* `objets Python, notamment`:  `int`, `float` et `chaîne de caractères` ; n&#39;importe lequel des `wandb.data_types` ; des listes, des tuples et des tableaux NumPy d&#39;objets Python sérialisables ; d&#39;autres `dict` de cette structure.
* `step`:  Le numéro d&#39;étape à journaliser. Si `None`, une étape implicite auto-incrémentée est utilisée. Voir les notes dans la description.
* `commit`:  Si la valeur est true, finalisez et téléversez l&#39;étape. Si la valeur est false, accumulez les données de l&#39;étape. Voir les notes dans la description. Si `step` est `None`, la valeur par défaut est `commit=True` ; sinon, la valeur par défaut est `commit=False`.

**Exemples :**
Pour plus d&#39;exemples et des exemples plus détaillés, voir [nos guides de journalisation](https://docs.wandb.com/guides/track/log).

### <kbd>méthode</kbd> `Run.save`

```python
save(
    glob_str: 'str | os.PathLike',
    base_path: 'str | os.PathLike | None' = None,
    policy: 'PolicyName' = 'live'
) → bool | list[str]
```

Synchronisez un ou plusieurs fichiers vers W&amp;B.

Les chemins relatifs sont relatifs au répertoire de travail actuel.

Un glob Unix, tel que &quot;myfiles/*&quot;, est développé au moment où `save` est appelé, quelle que soit la valeur de `policy`. En particulier, les nouveaux fichiers ne sont pas détectés automatiquement.

Un `base_path` peut être fourni pour contrôler la structure de répertoires des fichiers téléversés. Il doit être un préfixe de `glob_str`, et la structure de répertoires située en dessous est préservée.

Lorsqu&#39;un chemin absolu ou un glob est fourni sans `base_path`, un niveau de répertoire est préservé, comme dans l&#39;exemple ci-dessus.

Les fichiers sont automatiquement dédupliqués : appeler `save()` plusieurs fois sur le même fichier sans modification ne le téléversera pas de nouveau.

**Arguments :**

* `glob_str`:  Un chemin relatif ou absolu, ou un glob Unix.
* `base_path`:  Un chemin à utiliser pour déduire une structure de répertoires ; voir les exemples.
* `policy`:  L&#39;une des valeurs suivantes : `live`, `now` ou `end`.
  * live: téléverse le fichier à mesure qu&#39;il change, en remplaçant la version précédente
  * now: téléverse le fichier une fois immédiatement
  * end: téléverse le fichier lorsque le run se termine

**Retourne :**
Les chemins vers les liens symboliques créés pour les fichiers correspondants.

Pour des raisons historiques, cela peut renvoyer un booléen dans du code hérité.

```python
import wandb

run = wandb.init()

run.save("these/are/myfiles/*")
# => Saves files in a "these/are/myfiles/" folder in the run.

run.save("these/are/myfiles/*", base_path="these")
# => Saves files in an "are/myfiles/" folder in the run.

run.save("/Users/username/Documents/run123/*.txt")
# => Saves files in a "run123/" folder in the run. See note below.

run.save("/Users/username/Documents/run123/*.txt", base_path="/Users")
# => Saves files in a "username/Documents/run123/" folder in the run.

run.save("files/*/saveme.txt")
# => Enregistre chaque fichier "saveme.txt" dans un sous-répertoire approprié
#    de "files/".

# Explicitly finish the run since a context manager is not used.
run.finish()
```
