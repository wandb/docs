Lors d’une expérience d’entraînement distribué, vous entraînez un modèle à l’aide de plusieurs machines ou clients en parallèle. W&amp;B peut vous aider à suivre des expériences d’entraînement distribué. Selon votre cas d’usage, suivez les expériences d’entraînement distribué à l’aide de l’une des approches suivantes :

* **Suivre un seul processus** : suivez un processus de rang 0 (également appelé &quot;leader&quot; ou &quot;coordinateur&quot;) avec W&amp;B. Il s’agit d’une solution courante pour la journalisation d’expériences d’entraînement distribué avec la classe [PyTorch Distributed Data Parallel](https://pytorch.org/docs/stable/generated/torch.nn.parallel.DistributedDataParallel.html#torch.nn.parallel.DistributedDataParallel) (DDP).
* **Suivre plusieurs processus** : pour plusieurs processus, vous pouvez soit :
  * Suivre chaque processus séparément en utilisant un run par processus. Vous pouvez éventuellement les regrouper dans l’interface de l’application W&amp;B.
  * Suivre tous les processus dans un seul run.

<Tip>
  **Connexions simultanées**

  Chaque connexion simultanée consomme des ressources de calcul, de mémoire et réseau. Même les connexions client vides qui ne consignent pas régulièrement de métriques envoient tout de même des mises à jour des métriques système, ce qui ralentit le chargement des graphiques.

  W&amp;B recommande de limiter le nombre maximal de connexions client simultanées selon votre charge de travail et de surveiller l’utilisation des ressources au fil du temps. W&amp;B a testé une limite stricte de 300 connexions client simultanées dans **Cloud dédié**.

  Dans les organisations **Cloud mutualisé**, les connexions client pour l’entraînement distribué sont soumises aux mêmes [limites de débit](/fr/models/track/limits#rate-limits) que les runs d’entraînement classiques. Les Users des [plans Teams et Enterprise](https://wandb.ai/site/pricing) bénéficient de limites de débit plus élevées que ceux du plan Free.
</Tip>

<div id="track-a-single-process">
  ## Suivre un seul processus
</div>

Cette section décrit comment suivre les valeurs et les métriques disponibles pour votre processus de rang 0. Utilisez cette approche pour suivre uniquement les métriques disponibles à partir d’un seul processus. Les métriques typiques incluent l’utilisation du GPU/CPU, le comportement sur un ensemble de validation partagé, les gradients et les paramètres, ainsi que les valeurs de perte sur des exemples de données représentatifs.

Dans le processus de rang 0, initialisez un run W&amp;B avec [`wandb.init()`](/fr/models/ref/python/functions/init) et journalisez les expériences ([`wandb.log`](/fr/models/ref/python/experiments/run/#method-runlog)) dans ce run.

L’[exemple de script Python (`log-ddp.py`)](https://github.com/wandb/examples/blob/master/examples/pytorch/pytorch-ddp/log-ddp.py) suivant montre une façon de suivre des métriques sur deux GPU sur une seule machine à l’aide de PyTorch DDP. [PyTorch DDP](https://pytorch.org/tutorials/intermediate/ddp_tutorial.html) (`DistributedDataParallel` in`torch.nn`) est une bibliothèque populaire pour l’entraînement distribué. Les principes de base s’appliquent à toute configuration d’entraînement distribué, mais l’implémentation peut différer.

Le script Python :

1. Démarre plusieurs processus avec `torch.distributed.launch`.
2. Vérifie le rang avec l’argument de ligne de commande `--local_rank`.
3. Si le rang est défini sur 0, configure conditionnellement la journalisation `wandb` dans la fonction [`train()`](https://github.com/wandb/examples/blob/master/examples/pytorch/pytorch-ddp/log-ddp.py#L24).

```python
if __name__ == "__main__":
    # Obtenir les arguments
    args = parse_args()

    if args.local_rank == 0:  # uniquement sur le processus principal
        # Initialiser le run wandb
        run = wandb.init(
            entity=args.entity,
            project=args.project,
        )
        # Entraîner le modèle avec DDP
        train(args, run)
    else:
        train(args)
```

<div id="track-multiple-processes">
  ## Suivre plusieurs processus
</div>

Suivez plusieurs processus avec W&amp;B en utilisant l&#39;une des approches suivantes :

* [Suivre chaque processus séparément](/fr/models/track/log/distributed-training/#track-each-process-separately) en créant un run pour chaque processus.
* [Suivre tous les processus dans un seul run](/fr/models/track/log/distributed-training/#track-all-processes-to-a-single-run).

<div id="track-each-process-separately">
  ### Suivre chaque processus séparément
</div>

Cette section explique comment suivre chaque processus séparément en créant un run pour chacun d’eux. Dans chaque run, vous consignez les métriques, les artefacts, etc. dans le run correspondant. Appelez `wandb.Run.finish()` à la fin de l’entraînement pour indiquer que le run est terminé, afin que tous les processus se ferment correctement.

Il peut être difficile de suivre les runs sur plusieurs expériences. Pour y remédier, fournissez une valeur au paramètre `group` lorsque vous initialisez W&amp;B (`wandb.init(group='group-name')`) afin d’identifier à quelle expérience appartient chaque run. Pour plus d’informations sur la façon de suivre les Runs W&amp;B d’entraînement et d’évaluation dans les expériences, voir [Group Runs](/fr/models/runs/grouping/).

<Note>
  **Utilisez cette approche si vous voulez suivre les métriques de processus individuels**. Les exemples typiques incluent les données et les prédictions sur chaque nœud (pour déboguer la distribution des données) ainsi que les métriques sur des lots individuels en dehors du nœud principal. Cette approche n’est pas nécessaire pour obtenir les métriques système de tous les nœuds ni pour obtenir les statistiques de synthèse disponibles sur le nœud principal.
</Note>

L’extrait de code Python suivant montre comment définir le paramètre `group` lorsque vous initialisez W&amp;B :

```python
if __name__ == "__main__":
    # Obtenir les arguments
    args = parse_args()
    # Initialiser le run
    run = wandb.init(
        entity=args.entity,
        project=args.project,
        group="DDP",  # tous les runs de l'expérience dans un seul groupe
    )
    # Entraîner le modèle avec DDP
    train(args, run)

    run.finish()  # marquer le run comme terminé
```

<div id="organize-distributed-runs">
  ### Organiser les runs distribués
</div>

Définissez le paramètre `job_type` lorsque vous initialisez W&amp;B (`wandb.init(job_type='type-name')`) afin de catégoriser vos nœuds selon leur fonction. Par exemple, vous pouvez avoir un nœud coordinateur principal et plusieurs nœuds worker chargés du reporting. Vous pouvez définir `job_type` sur `main` pour le nœud coordinateur principal et sur `worker` pour les nœuds worker chargés du reporting :

```python
   # Nœud coordinateur principal
   with wandb.init(project="<project>", job_type="main", group="experiment_1") as run:
        # Code d'entraînement

   # Nœuds worker de reporting
   with wandb.init(project="<project>", job_type="worker", group="experiment_1") as run:
        # Code d'entraînement
```

Une fois que vous avez défini le `job_type` pour vos nœuds, vous pouvez créer des [vues enregistrées](/fr/models/track/workspaces/#create-a-new-saved-workspace-view) dans votre workspace pour organiser vos runs. Cliquez sur le menu d’actions **...** en haut à droite, puis cliquez sur **Save as new view**.

Par exemple, vous pouvez créer les vues enregistrées suivantes :

* **Vue par défaut** : Filtrez les nœuds worker pour réduire le bruit
  * Cliquez sur **Filtre**, puis définissez **Job Type** sur `worker`.

  * Affiche uniquement vos nœuds de reporting

  * **Vue de débogage** : Concentrez-vous sur les nœuds worker pour le dépannage
    * Cliquez sur **Filtre**, puis définissez **Job Type** sur `==` `worker` et **État** sur  `IN` `crashed`.
    * Affiche uniquement les nœuds worker qui ont planté ou sont en état d’erreur

  * **Vue de tous les nœuds** : Voir l’ensemble
    * Aucun filtre
    * Utile pour une surveillance complète

Pour ouvrir une vue enregistrée, cliquez sur **Workspaces** dans la barre latérale du projet, puis cliquez sur le menu. Les Workspaces apparaissent en haut de la liste et les vues enregistrées apparaissent en bas.

<div id="track-all-processes-to-a-single-run">
  ### Suivre tous les processus dans un seul run
</div>

<Warning>
  Les paramètres préfixés par `x_` (comme `x_label`) sont en préversion publique. Créez une [issue GitHub dans le dépôt W&amp;B](https://github.com/wandb/wandb) pour nous faire part de votre retour.
</Warning>

<Note>
  **Prérequis**

  Pour suivre plusieurs processus dans un seul run, vous devez disposer de :

  * du SDK Python W&amp;B en version `v0.19.9` ou ultérieure.

  * du serveur W&amp;B en version v0.68 ou ultérieure.
</Note>

Dans cette approche, vous utilisez un nœud principal et un ou plusieurs nœuds worker. Sur le nœud principal, vous initialisez un run W&amp;B. Pour chaque nœud worker, initialisez un run en utilisant l’ID du run du nœud principal. Pendant l’entraînement, chaque nœud worker journalise avec le même ID de run que le nœud principal. W&amp;B agrège les métriques de tous les nœuds et les affiche dans l’interface de l’application W&amp;B.

Sur le nœud principal, initialisez un run W&amp;B avec [`wandb.init()`](/fr/models/ref/python/functions/init). Passez un objet `wandb.Settings` au paramètre `settings` (`wandb.init(settings=wandb.Settings()`) avec les éléments suivants :

1. Le paramètre `mode` défini sur `"shared"` pour activer le mode partagé.
2. Une étiquette unique pour [`x_label`](https://github.com/wandb/wandb/blob/main/wandb/sdk/wandb_settings.py#L638). La valeur que vous spécifiez pour `x_label` permet d’identifier de quel nœud proviennent les données dans les logs et les métriques système de l’interface de l’application W&amp;B. Si elle n’est pas spécifiée, W&amp;B crée une étiquette pour vous à l’aide du nom d’hôte et d’un hachage aléatoire.
3. Définissez le paramètre [`x_primary`](https://github.com/wandb/wandb/blob/main/wandb/sdk/wandb_settings.py#L660) sur `True` pour indiquer qu’il s’agit du nœud principal.
4. Facultativement, fournissez une liste d’index GPU ([0,1,2]) à `x_stats_gpu_device_ids` pour préciser les GPU dont W&amp;B doit suivre les métriques. Si vous ne fournissez pas de liste, W&amp;B suit les métriques de tous les GPU de la machine.

Notez l’ID du run du nœud principal. Chaque nœud worker en a besoin.

<Note>
  `x_primary=True` permet de distinguer un nœud principal des nœuds worker. Les nœuds principaux sont les seuls à téléverser les fichiers partagés entre les nœuds, comme les fichiers de configuration, la télémétrie, entre autres. Les nœuds worker ne téléversent pas ces fichiers.
</Note>

Pour chaque nœud worker, initialisez un run W&amp;B avec [`wandb.init()`](/fr/models/ref/python/functions/init) et fournissez les éléments suivants :

1. Un objet `wandb.Settings` au paramètre `settings` (`wandb.init(settings=wandb.Settings()`) avec :
   * Le paramètre `mode` défini sur `"shared"` pour activer le mode partagé.
   * Une étiquette unique pour `x_label`. La valeur que vous spécifiez pour `x_label` permet d’identifier de quel nœud proviennent les données dans les logs et les métriques système de l’interface de l’application W&amp;B. Si elle n’est pas spécifiée, W&amp;B crée une étiquette pour vous à l’aide du nom d’hôte et d’un hachage aléatoire.
   * Définissez le paramètre `x_primary` sur `False` pour indiquer qu’il s’agit d’un nœud worker.
2. Passez l’ID du run utilisé par le nœud principal au paramètre `id`.
3. Facultativement, définissez [`x_update_finish_state`](https://github.com/wandb/wandb/blob/main/wandb/sdk/wandb_settings.py#L772) sur `False`. Cela empêche les nœuds non principaux de mettre à jour prématurément l’[état du run](/fr/models/runs/run-states#run-states) vers `finished`, ce qui garantit que l’état du run reste cohérent et géré par le nœud principal.

<Note>
  * Utilisez la même entité et le même projet pour tous les nœuds. Cela permet de s’assurer que le bon ID du run est trouvé.
  * Pensez à définir une variable d’environnement sur chaque nœud worker pour définir l’ID du run du nœud principal.
</Note>

L’exemple de code suivant illustre les prérequis généraux pour suivre plusieurs processus dans un seul run :

```python
import wandb

entity = "<team_entity>"
project = "<project_name>"

# Initialiser un run dans le nœud principal
run = wandb.init(
    entity=entity,
    project=project,
	settings=wandb.Settings(
        x_label="rank_0", 
        mode="shared", 
        x_primary=True,
        x_stats_gpu_device_ids=[0, 1],  # (Facultatif) Suivre uniquement les métriques pour les GPU 0 et 1
        )
)

# Noter l'ID du run du nœud principal.
# Chaque nœud worker a besoin de cet ID du run.
run_id = run.id

# Initialiser un run dans un nœud worker en utilisant l'ID du run du nœud principal
run = wandb.init(
    entity=entity, # Utiliser la même entité que le nœud principal
    project=project, # Utiliser le même projet que le nœud principal
	settings=wandb.Settings(x_label="rank_1", mode="shared", x_primary=False),
	id=run_id,
)

# Initialiser un run dans un nœud worker en utilisant l'ID du run du nœud principal
run = wandb.init(
    entity=entity, # Utiliser la même entité que le nœud principal
    project=project, # Utiliser le même projet que le nœud principal
	settings=wandb.Settings(x_label="rank_2", mode="shared", x_primary=False),
	id=run_id,
)
```

Dans un exemple concret, chaque nœud worker peut se trouver sur une machine distincte.

<div id="example-use-cases">
  ## Exemples de cas d&#39;utilisation
</div>

Les extraits de code suivants illustrent des scénarios courants pour des usages distribués avancés.

<div id="spawn-process">
  ### Processus enfant
</div>

Utilisez la méthode `wandb.setup()` dans votre fonction principale si vous lancez un run dans un processus enfant :

```python
import multiprocessing as mp

def do_work(n):
    with wandb.init(config=dict(n=n)) as run:
        run.log(dict(this=n * n))

def main():
    wandb.setup()
    pool = mp.Pool(processes=4)
    pool.map(do_work, range(4))


if __name__ == "__main__":
    main()
```

<div id="share-a-run">
  ### Partager un run
</div>

Passez un objet run comme argument pour partager des runs entre des processus :

```python
def do_work(run):
    with wandb.init() as run:
        run.log(dict(this=1))

def main():
    run = wandb.init()
    p = mp.Process(target=do_work, kwargs=dict(run=run))
    p.start()
    p.join()
    run.finish()  # marquer le run comme terminé


if __name__ == "__main__":
    main()
```

W&amp;B ne peut pas garantir l’ordre de journalisation. La synchronisation doit être effectuée par l’auteur du script.

<div id="troubleshooting">
  ## Dépannage
</div>

Il existe deux problèmes courants que vous pouvez rencontrer lorsque vous utilisez W&amp;B avec l’entraînement distribué :

1. **Blocage au début de l’entraînement** - Un processus `wandb` peut se bloquer si le multiprocessing de `wandb` interfère avec celui de l’entraînement distribué.
2. **Blocage à la fin de l’entraînement** - Une tâche d’entraînement peut se bloquer si le processus `wandb` ne sait pas à quel moment il doit se terminer. Appelez l’API `wandb.Run.finish()` à la fin de votre script Python pour indiquer à W&amp;B que le run est terminé. L’API `wandb.Run.finish()` terminera le téléversement des données et entraînera la fermeture de W&amp;B.
   W&amp;B recommande d’utiliser la commande `wandb service` pour améliorer la fiabilité de vos jobs distribués. Ces deux problèmes d’entraînement se rencontrent fréquemment dans les versions du SDK W&amp;B où wandb service n’est pas disponible.

<div id="enable-wb-service">
  ### Activer le service W&amp;B
</div>

Selon la version du SDK W&amp;B que vous utilisez, le service W&amp;B peut déjà être activé par défaut.

<div id="wb-sdk-0130-and-above">
  #### SDK W&amp;B 0.13.0 et versions ultérieures
</div>

Le service W&amp;B est activé par défaut à partir de la version `0.13.0` du SDK W&amp;B.

<div id="wb-sdk-0125-and-above">
  #### W&amp;B SDK 0.12.5 et versions ultérieures
</div>

Modifiez votre script Python pour activer le service W&amp;B avec la version 0.12.5 du SDK W&amp;B et les versions ultérieures. Utilisez la méthode `wandb.require` et passez la chaîne de caractères `"service"` dans votre fonction principale :

```python
if __name__ == "__main__":
    main()


def main():
    wandb.require("service")
    # reste-de-votre-script-ici
```

Pour une expérience optimale, nous vous recommandons de passer à la dernière version.

**SDK W&amp;B 0.12.4 et versions antérieures**

Si vous utilisez le SDK W&amp;B version 0.12.4 ou antérieure, définissez la variable d&#39;environnement `WANDB_START_METHOD` sur `"thread"` pour utiliser le multithreading à la place.
