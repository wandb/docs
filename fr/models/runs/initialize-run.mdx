Initialisez un Run W&amp;B avec [`wandb.init()`](/fr/models/ref/python/functions/init).

Par défaut, W&amp;B suppose que chaque processus Python n&#39;a qu&#39;un seul Run actif à la fois lorsque vous appelez `wandb.init()`. Si vous appelez de nouveau `wandb.init()`, W&amp;B renvoie soit le même Run, soit termine l&#39;ancien Run avant d&#39;en démarrer un nouveau. La façon dont W&amp;B gère les appels multiples à `wandb.init()` dans un même processus dépend de l&#39;environnement (notebook ou hors notebook) et de la configuration `reinit`.

Pour gérer plusieurs Runs actifs dans un même processus, voir [Plusieurs Runs dans un même processus](/fr/models/runs/initialize-run#multiple-runs-in-one-process).

<Note>
  W&amp;B recommande d&#39;utiliser un bloc `with` lors de l&#39;appel à `wandb.init()`. Cela garantit que W&amp;B finalise correctement le Run et téléverse toutes les données à la fin du bloc.
</Note>

<div id="single-run-per-process">
  ## Un seul run par processus
</div>

L&#39;extrait de code suivant montre comment importer le SDK Python W&amp;B et initialiser un run.

```python title="basic.py"
import wandb

with wandb.init(entity="nico", project="awesome-project") as run:
    # Votre logique d'entraînement ici
```

L’extrait de code produit la sortie suivante :

```bash
🚀 Voir le run exalted-darkness-6 à : 
https://wandb.ai/nico/awesome-project/runs/pgbn9y21
Trouver les Logs dans : wandb/run-20241106_090747-pgbn9y21/logs
```

La sortie montre que W&amp;B journalise le run `exalted-darkness-6` dans le projet `awesome-project` sous l&#39;entité `nico`. `pgbn9y21` est l&#39;ID du run unique que W&amp;B génère pour ce run.

<div id="multiple-runs-in-one-process">
  ## Plusieurs runs dans un même processus
</div>

Gérez plusieurs runs dans un même processus Python. Cela est utile pour les flux de travail dans lesquels vous souhaitez garder un processus principal actif tout en créant des processus secondaires de courte durée pour des sous-tâches. Voici quelques cas d’usage :

* Garder un seul run « principal » actif tout au long d’un script, tout en lançant des runs « secondaires » de courte durée pour des évaluations ou des sous-tâches.
* Orchestrer des sous-expériences dans un seul fichier.
* Effectuer la journalisation depuis un processus « principal » vers plusieurs runs représentant différentes tâches ou périodes.

Par défaut, W&amp;B suppose que chaque processus Python n’a qu’un seul run actif à la fois lorsque vous appelez `wandb.init()`. Si vous appelez `wandb.init()` à nouveau, W&amp;B renvoie soit le même run, soit termine l’ancien run avant d’en démarrer un nouveau, selon la configuration.

Ce guide explique comment utiliser `reinit` pour modifier le comportement de `wandb.init()` afin d’autoriser plusieurs runs dans un même processus Python.

<Note>
  **Exigences**

  Pour gérer plusieurs runs dans un même processus Python, vous devez disposer de la version `v0.19.10` ou ultérieure du SDK Python W&amp;B.
</Note>

<div id="reinit-options">
  ### Options de `reinit`
</div>

Utilisez le paramètre `reinit` pour configurer la façon dont W&amp;B gère les appels multiples à `wandb.init()`. Le tableau suivant décrit les arguments valides et leurs effets :

|                   | Description                                                                                                                                                                                                                                                                                                                                                                                               | Crée un run ? | Exemple de cas d&#39;utilisation                                                                                                                                       |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `create_new`      | Créer un nouveau run avec `wandb.init()` sans terminer les runs actifs existants. W&amp;B ne bascule pas automatiquement le `wandb.Run` global vers les nouveaux runs. Vous devez conserver vous-même une référence à chaque objet run. Voir l&#39;[exemple de plusieurs runs dans un même processus](/fr/models/runs/initialize-run/#example-multiple-runs-in-one-process) ci-dessous pour plus de détails. | Oui           | Idéal pour créer et gérer des processus concurrents. Par exemple, un run « principal » qui reste actif pendant que vous démarrez ou terminez des runs « secondaires ». |
| `finish_previous` | Terminez tous les runs actifs avec `run.finish()` avant d&#39;en créer un nouveau avec `wandb.init()`. Il s&#39;agit du comportement par défaut dans les environnements hors notebook.                                                                                                                                                                                                                    | Oui           | Idéal lorsque vous souhaitez séparer des sous-processus séquentiels en runs individuels distincts.                                                                     |
| `return_previous` | Renvoie le run inachevé le plus récent. Il s&#39;agit du comportement par défaut dans les environnements notebook.                                                                                                                                                                                                                                                                                        | Non           |                                                                                                                                                                        |

<Note>
  W&amp;B ne prend pas en charge le mode `create_new` pour les [intégrations W&amp;B](/fr/models/integrations) qui supposent un seul run global, comme Hugging Face Trainer, les callbacks Keras et PyTorch Lightning. Si vous utilisez ces intégrations, vous devez exécuter chaque sous-expérience dans un processus distinct.
</Note>

<div id="specifying-reinit">
  ### Spécification de `reinit`
</div>

{/* Il existe plusieurs façons de créer et de gérer plusieurs runs au sein d’un même processus Python : */}

* Utilisez directement `wandb.init()` avec l’argument `reinit` :
  ```python
  import wandb
  with wandb.init(reinit="<create_new|finish_previous|return_previous>") as run:
      # Votre code ici
  ```

* Utilisez `wandb.init()` et passez un objet `wandb.Settings` au paramètre `settings`. Spécifiez `reinit` dans l’objet `Settings` :

  ```python
  import wandb
  with wandb.init(settings=wandb.Settings(reinit="<create_new|finish_previous|return_previous>")) as run:
      # Votre code ici
  ```

* Utilisez `wandb.setup()` pour définir globalement l’option `reinit` pour tous les runs du processus en cours. Cette approche est utile si vous souhaitez configurer le comportement une seule fois et l’appliquer à tous les appels suivants de `wandb.init()` dans ce processus.

  ```python
  import wandb
  with wandb.setup(wandb.Settings(reinit="<create_new|finish_previous|return_previous>")) as run:
       # Votre code ici
  ```

* Spécifiez la valeur souhaitée pour `reinit` dans la variable d’environnement `WANDB_REINIT`. La définition de cette variable d’environnement applique l’option `reinit` aux appels à `wandb.init()`.

  ```bash
  export WANDB_REINIT="<create_new|finish_previous|return_previous>"
  ```

L’extrait de code suivant donne un aperçu général de la manière de configurer W&amp;B pour créer un nouveau run à chaque appel à `wandb.init()` :

```python
import wandb

wandb.setup(wandb.Settings(reinit="create_new"))

with wandb.init() as experiment_results_run:
    # Ce run sera utilisé pour journaliser les résultats de chaque expérience.
    # Vous pouvez le considérer comme un run parent qui collecte les résultats
      with wandb.init() as run:
         # La fonction do_experiment() journalise des métriques détaillées
         # dans le run donné et renvoie des métriques de résultats que
         # vous souhaitez suivre séparément.
         experiment_results = do_experiment(run)

         # Après chaque expérience, journalisez ses résultats dans un run
         # parent. Chaque point dans les graphiques du run parent correspond
         # aux résultats d'une expérience.
         experiment_results_run.log(experiment_results)
```

<div id="example-concurrent-processes">
  ### Exemple : processus concurrents
</div>

Supposons que vous souhaitiez créer un processus principal qui reste ouvert pendant toute la durée de vie du script, tout en lançant périodiquement des processus secondaires de courte durée sans terminer le processus principal. Par exemple, ce schéma peut être utile si vous souhaitez entraîner un modèle dans le run principal, mais calculer des évaluations ou effectuer d&#39;autres tâches dans des runs distincts.

Pour y parvenir, utilisez `reinit="create_new"` et initialisez plusieurs runs. Pour cet exemple, supposons que &quot;Run A&quot; soit le processus principal qui reste ouvert tout au long du script, tandis que &quot;Run B1&quot; et &quot;Run B2&quot; sont des runs secondaires de courte durée pour des tâches comme l&#39;évaluation.

Le flux de travail global peut ressembler à ceci :

1. Initialisez le processus principal Run A avec `wandb.init()` et journalisez les métriques d&#39;entraînement.
2. Initialisez Run B1 (avec `wandb.init()`), journalisez des données, puis terminez-le.
3. Journalisez davantage de données dans Run A.
4. Initialisez Run B2, journalisez des données, puis terminez-le.
5. Continuez la journalisation dans Run A.
6. Enfin, terminez Run A à la fin.

L&#39;exemple de code Python suivant illustre ce flux de travail :

```python
import wandb

def train(name: str) -> None:
    """Effectue une itération d'entraînement dans son propre run W&B.

    L'utilisation d'un bloc 'with wandb.init()' avec `reinit="create_new"` garantit que
    ce sous-run d'entraînement peut être créé même si un autre run (comme notre run de
    suivi principal) est déjà actif.
    """
    with wandb.init(
        project="my_project",
        name=name,
        reinit="create_new"
    ) as run:
        # Dans un vrai script, vous exécuteriez vos étapes d'entraînement dans ce bloc.
        run.log({"train_loss": 0.42})  # Remplacez par vos métriques réelles

def evaluate_loss_accuracy() -> (float, float):
    """Renvoie la perte et la précision du modèle actuel.
    
    Remplacez cet espace réservé par votre logique d'évaluation réelle.
    """
    return 0.27, 0.91  # Exemples de valeurs de métriques

# Crée un run « principal » qui reste actif tout au long de plusieurs étapes d'entraînement/évaluation.
with wandb.init(
    project="my_project",
    name="tracking_run",
    reinit="create_new"
) as tracking_run:
    # 1) Entraîner une fois sous un sous-run nommé 'training_1'
    train("training_1")
    loss, accuracy = evaluate_loss_accuracy()
    tracking_run.log({"eval_loss": loss, "eval_accuracy": accuracy})

    # 2) Entraîner à nouveau sous un sous-run nommé 'training_2'
    train("training_2")
    loss, accuracy = evaluate_loss_accuracy()
    tracking_run.log({"eval_loss": loss, "eval_accuracy": accuracy})
    
    # Le 'tracking_run' se termine automatiquement à la fin de ce bloc 'with'.
```

Notez trois points clés de l’exemple précédent :

1. `reinit="create_new"` crée un nouveau run chaque fois que vous appelez `wandb.init()`.
2. Vous conservez des références à chaque run. `wandb.run` ne pointe pas automatiquement vers le nouveau run créé avec `reinit="create_new"`. Stockez les nouveaux runs dans des variables comme `run_a`, `run_b1`, etc., et appelez `.log()` ou `.finish()` sur ces objets selon les besoins.
3. Vous pouvez terminer les sous-runs quand vous le souhaitez tout en gardant le run principal ouvert jusque-là.
4. Terminez vos runs avec `run.finish()` lorsque vous avez terminé d’y faire la journalisation des données. Cela garantit que toutes les données sont téléversées et que le run est correctement fermé.
