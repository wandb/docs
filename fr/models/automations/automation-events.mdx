Une automatisation peut démarrer lorsqu’un événement spécifique se produit dans un projet ou un registre. Cette page décrit les événements qui peuvent déclencher une automatisation dans chaque portée. Pour en savoir plus sur les automatisations, consultez l’[Aperçu des automatisations](/fr/models/automations) ou [Créer une automatisation](/fr/models/automations/create-automations).

<div id="registry">
  ## Registre
</div>

Cette section décrit les portées et les événements d&#39;une automatisation dans un [registre](/fr/models/registry).

<div id="scopes">
  ### Portées
</div>

Une automatisation du [registre](/fr/models/registry) surveille l’événement sur toute collection d’un registre donné, y compris sur les collections ajoutées ultérieurement.

### Événements <a id="registry-events" aria-label="Événements du registre" />

Une automatisation du registre peut surveiller les événements suivants :

* **Une nouvelle version est liée à une collection** : testez et validez de nouveaux modèles ou jeux de données lorsqu’ils sont ajoutés à un registre.
* **Un alias d’artefact est ajouté** : déclenchez une étape spécifique de votre flux de travail lorsqu’une nouvelle version d’artefact reçoit un alias spécifique. Par exemple, déployez un modèle lorsque l’alias `production` lui est attribué.

<div id="project">
  ## Projet
</div>

Cette section décrit les portées et les événements d’une automatisation dans un [projet](/fr/models/track/project-page).

<div id="scopes">
  ### Portées
</div>

Une automatisation au niveau du projet surveille l&#39;événement survenant dans n&#39;importe quelle collection du projet. Selon l&#39;événement que vous spécifiez, vous pouvez restreindre davantage la portée de l&#39;automatisation.

<div id="artifact-events">
  ### Événements d&#39;artefact
</div>

Cette section décrit les événements liés à un artefact qui peuvent déclencher une automatisation.

* **Une nouvelle version est ajoutée à un artefact** : appliquez des actions récurrentes à chaque version d&#39;un artefact. Par exemple, démarrez une tâche d&#39;entraînement lorsqu&#39;une nouvelle version d&#39;un artefact de jeu de données est créée.
* **Un alias d&#39;artefact est ajouté** : déclenchez une étape spécifique de votre flux de travail lorsqu&#39;une nouvelle version d&#39;artefact dans un projet se voit attribuer un alias correspondant à l&#39;**Alias regex** que vous spécifiez. Par exemple, exécutez une série d&#39;étapes de traitement en aval lorsqu&#39;un artefact se voit attribuer l&#39;alias `test-set-quality-check`, ou lancez un flux de travail chaque fois qu&#39;une nouvelle version d&#39;artefact se voit attribuer l&#39;alias `latest`. Une seule version d&#39;artefact peut avoir un alias donné à un instant donné.
* **Une balise d&#39;artefact est ajoutée** : déclenchez une étape spécifique de votre flux de travail lorsqu&#39;une version d&#39;artefact dans un projet se voit attribuer une balise correspondant au **Tag regex** que vous spécifiez. Par exemple, spécifiez `^europe.*` pour déclencher un flux de travail propre à une zone géographique lorsqu&#39;une balise commençant par la chaîne `europe` est ajoutée à une version d&#39;artefact. Les balises d&#39;artefact servent au regroupement et au filtrage, et une balise donnée peut être attribuée simultanément à plusieurs versions d&#39;artefact.

<div id="run-events">
  ### Événements de run
</div>

Une automatisation peut être déclenchée par une modification du [statut d’un run](/fr/models/runs/run-states) ou de la [valeur d’une métrique](/fr/models/track/log#what-data-is-logged-with-specific-wb-api-calls).

<div id="run-status-change">
  #### Changement du statut d’un run
</div>

<Note>
  * <MultiTenantCloudOnly />
  * Un run ayant le statut **Killed** ne peut pas déclencher d’automatisation. Ce statut indique que le run a été arrêté de force par un administrateur.
</Note>

Déclenchez un flux de travail lorsqu’un run change de [statut](/fr/models/runs/run-states) pour passer à **Running**, **Finished** ou **Failed**. Vous pouvez également restreindre davantage les runs pouvant déclencher une automatisation en spécifiant un filtre sur l’utilisateur ou le nom de l’exécution.

<Frame>
  ![Capture d'écran montrant une automatisation de changement du statut d’un run](/images/automations/run_status_change.png)
</Frame>

Comme le statut est une propriété du run dans son ensemble, vous pouvez créer une automatisation de statut de run uniquement depuis la page **Automations**, et non depuis un espace de travail.

<div id="run-metrics-change">
  #### Changement des métriques de run
</div>

<Note>
  <MultiTenantCloudOnly />
</Note>

Déclenchez un flux de travail basé sur une valeur enregistrée pour une métrique, qu&#39;il s&#39;agisse d&#39;une métrique dans l&#39;historique d&#39;un run ou d&#39;une [métrique système](/fr/models/ref/python/experiments/system-metrics) telle que `cpu`, qui suit le pourcentage d&#39;utilisation du processeur. W&amp;B enregistre automatiquement les métriques système toutes les 15 secondes.

Vous pouvez créer une automatisation de métriques de run depuis l&#39;onglet **Automations** du projet ou directement depuis un panneau de graphique linéaire dans un espace de travail.

Pour configurer une automatisation de métrique de run, définissez comment comparer la valeur de la métrique au seuil que vous indiquez. Les options disponibles dépendent du type d&#39;événement et des filtres que vous spécifiez.

Vous pouvez également limiter davantage les runs susceptibles de déclencher une automatisation en spécifiant un filtre sur l&#39;utilisateur ou le nom de l&#39;exécution.

<div id="threshold">
  ##### Seuil
</div>

Pour les événements **Seuil des métriques de run atteint**, vous configurez les éléments suivants :

1. La fenêtre des valeurs enregistrées les plus récentes à prendre en compte (5 par défaut).
2. S&#39;il faut évaluer la valeur **moyenne**, **minimale** ou **maximale** dans la fenêtre.
3. La comparaison à effectuer :
   * Supérieur à
   * Supérieur ou égal à
   * Inférieur à
   * Inférieur ou égal à
   * Différent de
   * Égal à

Par exemple, déclenchez une automatisation lorsque la moyenne de `accuracy` est supérieure à `.6`.

<Frame>
  ![Capture d’écran montrant une automatisation de seuil de métriques de run](/images/automations/run_metrics_threshold_automation.png)
</Frame>

<div id="change-threshold">
  ##### Seuil de variation
</div>

Pour les événements **Seuil de variation des métriques du run atteint**, l&#39;automatisation utilise deux « fenêtres » de valeurs pour déterminer si elle doit se déclencher :

* la *fenêtre actuelle* des valeurs récemment enregistrées à prendre en compte (valeurs par défaut : 10).
* la *fenêtre précédente* des valeurs récemment enregistrées à prendre en compte (valeurs par défaut : 50).

Les fenêtres actuelle et précédente sont consécutives et ne se chevauchent pas.

Pour créer l&#39;automatisation, vous configurez :

1. la fenêtre actuelle des valeurs enregistrées (valeurs par défaut : 10).
2. la fenêtre précédente des valeurs enregistrées (valeurs par défaut : 50).
3. si les valeurs doivent être évaluées en relatif ou en absolu (valeurs par défaut : **Relatif**).
4. la comparaison à effectuer :
   * Augmente d&#39;au moins
   * Diminue d&#39;au moins
   * Augmente ou diminue d&#39;au moins

Par exemple, déclenchez une automatisation lorsque la moyenne de `loss` diminue d&#39;au moins `.25`.

  ![Capture d'écran montrant une automatisation de seuil de variation des métriques du run](/images/automations/run_metrics_change_threshold_automation.png)

<div id="run-metrics-z-score-change">
  #### Variation du z-score des métriques de run
</div>

<Note>
  <MultiTenantCloudOnly />
</Note>

W&amp;B peut déclencher une automatisation lorsque le z-score (score standard) d’une métrique dépasse un seuil spécifié. Un z-score mesure le nombre d’écarts-types entre une valeur et la moyenne de cette métrique sur une fenêtre configurable de runs du projet (30 runs par défaut).

Pour utiliser un z-score comme déclencheur d’événement, sélectionnez l’événement **seuil de z-score des métriques de run atteint**.

Les automatisations basées sur le z-score permettent à votre équipe d’être informée des performances inhabituelles sans avoir à vérifier des seuils absolus, qui peuvent varier à mesure que votre modèle ou votre processus d’entraînement évolue.

Vous pouvez créer une automatisation de z-score pour les métriques de run depuis l’onglet **Automations** du projet ou directement depuis un panneau de graphique linéaire dans un espace de travail.

Pour créer une automatisation de z-score, vous configurez :

1. Le seuil cible du z-score, exprimé sous la forme d’un nombre à virgule flottante positif (par exemple, 2.0).
2. La fenêtre de valeurs enregistrées qui détermine la moyenne (30 par défaut).
3. La comparaison à effectuer :
   * Au-dessus (déclenchement lorsque les performances sont inhabituellement élevées)
   * En dessous (déclenchement lorsque les performances sont inhabituellement faibles)
   * Au-dessus ou en dessous

Par exemple, déclenchez une automatisation lorsque `accuracy` a un z-score supérieur à 2, ce qui signifie que le run obtient des performances nettement meilleures que les autres runs du projet.

**Comprendre les valeurs de z-score :**

* Un z-score de 0 signifie que la métrique correspond à la moyenne.
* Un z-score de +2.0 signifie que la métrique se situe à 2 écarts-types au-dessus de la moyenne.
* Un z-score de -2.0 signifie que la métrique se situe à 2 écarts-types en dessous de la moyenne.
* Les valeurs au-delà de ±2 sont souvent considérées comme des valeurs aberrantes statistiquement significatives.

<div id="run-filters">
  #### Filtres de runs
</div>

Cette section décrit comment l&#39;automatisation sélectionne les runs à évaluer.

* Par défaut, tout run du projet déclenche l&#39;automatisation lorsque l&#39;événement se produit. Vous pouvez limiter les runs qui déclenchent une automatisation en configurant l&#39;un des filtres suivants :

  * **Filtrer les runs d&#39;un seul utilisateur** : inclure uniquement les runs créés par l&#39;utilisateur spécifié.
  * **Filtrer par nom de l&#39;exécution** : inclure uniquement les runs dont le nom correspond à l&#39;expression régulière indiquée.

  Pour plus de détails, voir [Créer des automatisations](/fr/models/automations/create-automations).
* Chaque run est traité individuellement et peut potentiellement déclencher l&#39;automatisation.
* Les valeurs de chaque run sont placées dans une fenêtre distincte et comparées séparément au seuil.
* Sur une période de 24 heures, une automatisation donnée ne peut se déclencher qu&#39;une seule fois par run.
