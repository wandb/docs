Ce guide explique comment intégrer Weights &amp; Biases (W&amp;B) dans une bibliothèque Python.

Suivez ces recommandations si vous intégrez W&amp;B dans une base de code complexe, comme un framework d’entraînement, un SDK ou une bibliothèque réutilisable.

<Tip>
  Si vous débutez avec W&amp;B, consultez les guides essentiels (par exemple, [Suivi des expériences](/fr/models/track/)) avant de continuer.
</Tip>

Nous présentons ci-dessous les principaux conseils et bonnes pratiques à suivre lorsque la base de code sur laquelle vous travaillez est plus complexe qu’un simple script d’entraînement Python ou un notebook Jupyter.

<div id="decide-how-users-install-wb">
  ## Choisissez comment les utilisateurs installent W&amp;B
</div>

Avant de commencer, déterminez si W&amp;B doit être une dépendance requise ou une fonctionnalité facultative de votre bibliothèque.

<div id="require-wb-as-a-dependency">
  ### Ajouter W&amp;B comme dépendance requise
</div>

Si W&amp;B joue un rôle central dans le fonctionnement de votre bibliothèque, ajoutez le SDK Python W&amp;B (`wandb`) à vos dépendances :

```txt
torch==1.8.0 
...
wandb==0.13.*
```

<div id="make-wb-optional-on-installation">
  ### Rendre W&amp;B facultatif lors de l’installation
</div>

Si W&amp;B est une fonctionnalité facultative, permettez à votre bibliothèque de fonctionner sans qu’il soit installé.

Vous pouvez soit importer `wandb` conditionnellement en Python, soit le déclarer comme dépendance facultative dans `pyproject.toml`.

<Tabs>
  <Tab title="Python">
    Détectez si `wandb` est disponible et levez une erreur explicite si un utilisateur active les fonctionnalités W&amp;B sans l’avoir installé :

    ```python
    try:
        import wandb
        _WANDB_AVAILABLE = True
    except ImportError:
        _WANDB_AVAILABLE = False
    ```
  </Tab>

  <Tab title="pyproject.toml">
    Déclarez `wandb` comme dépendance facultative dans votre fichier `pyproject.toml` :

    ```toml
    [project]
    name = "my_awesome_lib"
    version = "0.1.0"
    dependencies = [
        "torch",
        "sklearn"
    ]

    [project.optional-dependencies]
    dev = [
        "wandb"
    ]
    ```
  </Tab>
</Tabs>

<div id="authenticate-users">
  ## Authentifier les utilisateurs
</div>

W&amp;B utilise des clés API pour authentifier les utilisateurs et les machines.

<div id="create-an-api-key">
  ### Créer une clé API
</div>

Une clé API permet d’authentifier un client ou une machine auprès de W&amp;B. Vous pouvez générer une clé API depuis votre profil utilisateur.

2. Sélectionnez **Paramètres utilisateur**, puis faites défiler la page jusqu’à la section **Clés API**.

<div id="install-and-log-in-to-wb">
  ### Installer et vous connecter à W&amp;B
</div>

Pour installer la bibliothèque `wandb` localement et vous connecter :

<Tabs>
  <Tab title="Ligne de commande">
    1. Définissez la [variable d&#39;environnement](/fr/models/track/environment-variables/) `WANDB_API_KEY` avec votre clé API :

       ```bash
       export WANDB_API_KEY=<your_api_key>
       ```

    2. Installez la bibliothèque `wandb` et connectez-vous :

       ```bash
       pip install wandb

       wandb login
       ```
  </Tab>

  <Tab title="Python">
    1. Ouvrez votre terminal et installez le SDK Python.
       ```bash
       pip install wandb
       ```

    2. Connectez-vous à W&amp;B depuis votre script Python ou votre notebook. Vous serez invité à saisir
       votre clé API.
       ```python
       import wandb
       wandb.login()
       ```
  </Tab>

  <Tab title="Notebook Python">
    Copiez-collez l&#39;extrait de code suivant dans une cellule de votre notebook Jupyter, puis exécutez-le. Vous serez invité à saisir votre clé API.

    ```notebook
    !pip install wandb

    import wandb
    wandb.login()
    ```
  </Tab>
</Tabs>

<div id="start-a-run">
  ## Démarrer un run
</div>

Un *run* représente une unité de calcul, comme une expérience d&#39;entraînement. La plupart des bibliothèques créent un run par tâche d&#39;entraînement. Pour en savoir plus sur les Runs, voir [W&amp;B Runs](/fr/models/runs/).

Initialisez un run avec [`wandb.init()`](/fr/models/ref/python/functions/init) et spécifiez le nom de votre projet ainsi que votre entité d’équipe (nom de l’équipe). Si vous ne spécifiez pas de projet, W&amp;B stocke votre run dans un projet par défaut appelé &quot;uncategorized&quot;.:

```python
with wandb.init(project="<project_name>", entity="<entity>") as run:
    ...
```

W&amp;B recommande d&#39;utiliser un gestionnaire de contexte pour garantir que votre run est correctement fermé, même en cas d&#39;erreur. Si vous n&#39;utilisez pas de gestionnaire de contexte, vous devez appeler `run.finish()` pour fermer le run et journaliser toutes les données dans W&amp;B.

<Tip>
  **Quand appeler `wandb.init`**

  Appelez `wandb.init()` le plus tôt possible. W&amp;B capture stdout, stderr et les messages d&#39;erreur, ce qui facilite le débogage.

  Enveloppez l&#39;intégralité de votre boucle d&#39;entraînement dans un gestionnaire de contexte `wandb.init` afin de garantir que toutes les informations pertinentes sont capturées dans le run. Cela inclut les messages d&#39;erreur, qui peuvent être essentiels pour le débogage.
</Tip>

<div id="set-wandb-as-an-optional-dependency">
  ### Définir `wandb` comme dépendance facultative
</div>

Si vous souhaitez rendre `wandb` facultatif pour les utilisateurs de votre bibliothèque, vous avez deux possibilités :

* Définir un indicateur `wandb`, par exemple :

<Tabs>
  <Tab title="Python">
    ```python
    trainer = my_trainer(..., use_wandb=True)
    ```
  </Tab>

  <Tab title="Bash">
    ```bash
    python train.py ... --use-wandb
    ```
  </Tab>
</Tabs>

* Ou définir `wandb` sur `disabled` dans `wandb.init` :

<Tabs>
  <Tab title="Python">
    ```python
    wandb.init(mode="disabled")
    ```
  </Tab>

  <Tab title="Bash">
    ```bash
    export WANDB_MODE=disabled
    ```

    ou

    ```bash
    wandb disabled
    ```
  </Tab>
</Tabs>

* Ou définir `wandb` en mode hors ligne ; notez que cela exécutera toujours `wandb`, mais qu’il n’essaiera simplement pas de communiquer avec W&amp;B via Internet :

<Tabs>
  <Tab title="Environment Variable">
    ```bash
    export WANDB_MODE=offline
    ```

    ou

    ```python
    os.environ['WANDB_MODE'] = 'offline'
    ```
  </Tab>

  <Tab title="Bash">
    ```bash
    wandb offline
    ```
  </Tab>
</Tabs>

<div id="define-a-run-config">
  ## Définir la configuration d’un run
</div>

Fournissez un dictionnaire de configuration lorsque vous initialisez votre run afin de journaliser les hyperparamètres et d’autres métadonnées dans W&amp;B.

Utilisez l’application W&amp;B pour comparer les runs en fonction de leurs paramètres de configuration et les filtrer dans le tableau Runs. Vous pouvez également utiliser ces paramètres pour regrouper des runs dans l’application W&amp;B.

Par exemple, dans l’image suivante, la taille de lot (bathch&#95;size) a été définie comme paramètre de configuration et est visible (voir la première colonne) dans le tableau Runs. Cela permet aux utilisateurs de filtrer et de comparer les runs en fonction de leur taille de lot :

<Frame>
  <img src="/images/integrations/integrations_add_any_lib_runs_page.png" alt="tableau Runs de W&B" />
</Frame>

Les valeurs typiques des paramètres de configuration incluent :

* Nom du modèle, version, paramètres d’architecture et hyperparamètres.
* Nom du jeu de données, version, nombre d’exemples d’entraînement ou de validation.
* Paramètres d’entraînement tels que le taux d’apprentissage, la taille de lot et l’optimiseur.

L’extrait de code suivant montre comment journaliser une configuration :

```python
config = {"batch_size": 32, ...}
with wandb.init(..., config=config) as run:
    ...
```

<div id="update-the-run-config">
  ### Mettre à jour la configuration du run
</div>

Si certaines valeurs ne sont pas disponibles au moment de l’initialisation, mettez à jour la configuration ultérieurement avec `wandb.Run.config.update`. Par exemple, vous pouvez ajouter les paramètres d’un modèle après l’instanciation de celui-ci :

```python
with wandb.init(...) as run:
    model = MyModel(...)
    run.config.update({"model_parameters": 3500})
```

Pour plus de détails, voir [Configurer les expériences](/fr/models/track/config/).

<div id="log-metrics-and-data">
  ## Journaliser les métriques et les données
</div>

<div id="log-metrics">
  ### Journaliser des métriques
</div>

Créez un dictionnaire dans lequel chaque clé correspond au nom de la métrique. Passez cet objet dictionnaire à [`wandb.Run.log()`](/fr/models/ref/python/experiments/run#method-run-log) pour journaliser ces métriques dans W&amp;B :

```python
NUM_EPOCHS = 10

for epoch in range(NUM_EPOCHS):
    for input, ground_truth in data: 
        prediction = model(input) 
        loss = loss_fn(prediction, ground_truth) 
        metrics = { "loss": loss } 
        run.log(metrics)
```

Utilisez des préfixes de noms de métriques pour regrouper les métriques associées dans l’application W&amp;B. Les préfixes courants incluent `train/` et `val/` pour les métriques d&#39;entraînement et de validation, respectivement, mais vous pouvez utiliser n&#39;importe quel préfixe adapté à votre cas d&#39;utilisation.

Cela créera des sections distinctes dans l&#39;espace de travail de votre projet pour les métriques d&#39;entraînement et de validation, ou pour d&#39;autres types de métriques que vous souhaitez séparer :

```python
with wandb.init(...) as run:
    metrics = {
        "train/loss": 0.4,
        "train/learning_rate": 0.4,
        "val/loss": 0.5, 
        "val/accuracy": 0.7
    }
    run.log(metrics)
```

Voir [`wandb.Run.log()`](/fr/models/ref/python/experiments/run#method-run-log) pour en savoir plus.

<div id="control-the-x-axis">
  ### Contrôler l’axe X
</div>

Si vous effectuez des appels multiples à `wandb.Run.log()` pour la même étape d’entraînement, le SDK wandb incrémente un compteur interne d’étapes à chaque appel à `wandb.Run.log()`. Ce compteur peut ne pas correspondre à l’étape d’entraînement de votre boucle d’entraînement.

Pour éviter cette situation, définissez explicitement l’étape de l’axe X avec `run.define_metric`, une seule fois, immédiatement après avoir appelé `wandb.init` :

```python
with wandb.init(...) as run:
    run.define_metric("*", step_metric="global_step")
```

Le motif glob, `*`, signifie que chaque métrique utilisera `global_step` comme axe des x dans vos graphiques. Si vous souhaitez que seules certaines métriques soient journalisées en fonction de `global_step`, vous pouvez les spécifier à la place :

```python
run.define_metric("train/loss", step_metric="global_step")
```

Maintenant, journalisez vos métriques, votre métrique `step` et votre `global_step` chaque fois que vous appelez `wandb.Run.log()` :

```python
for step, (input, ground_truth) in enumerate(data):
    ...
    run.log({"global_step": step, "train/loss": 0.1})
    run.log({"global_step": step, "eval/loss": 0.2})
```

Si vous n’avez pas accès à la variable d’étape indépendante, par exemple si &quot;global&#95;step&quot; n’est pas disponible dans votre boucle de validation, wandb utilise automatiquement la valeur précédemment journalisée pour &quot;global&#95;step&quot;. Dans ce cas, assurez-vous de journaliser une valeur initiale pour la métrique afin qu’elle soit déjà définie au moment où elle est nécessaire.

<div id="log-media-and-structured-data">
  ### Journaliser les médias et les données structurées
</div>

En plus des scalaires, vous pouvez journaliser des images, des tableaux, du texte, de l’audio, de la vidéo, et bien plus encore.

Voici quelques considérations à garder à l’esprit lorsque vous journalisez des données :

* À quelle fréquence la métrique doit-elle être enregistrée ? Doit-elle être facultative ?
* Quel type de données peut être utile pour la visualisation ?
  * Pour les images, vous pouvez journaliser des exemples de prédictions, des masques de segmentation, etc., afin de voir l’évolution au fil du temps.
  * Pour le texte, vous pouvez journaliser des tableaux d’exemples de prédictions pour une exploration ultérieure.

Voir [Journaliser des objets et des médias](/fr/models/track/log) pour des exemples.

Pour les frameworks qui prennent en charge les environnements distribués, vous pouvez adapter l’un des flux de travail suivants :

<div id="track-models-and-datasets-with-artifacts">
  ## Suivre les modèles et les jeux de données avec des artefacts
</div>

Utilisez [W&amp;B Artifacts](/fr/models/artifacts/) pour suivre et gérer les versions des modèles et des jeux de données. Les artefacts assurent le stockage et la gestion des versions des ressources de machine learning, et suivent automatiquement leur traçabilité afin de montrer comment les données et les modèles sont liés.

<Frame>
  <img src="/images/integrations/integrations_add_any_lib_dag.png" alt="Jeux de données stockés et points de contrôle du modèle dans W&B" />
</Frame>

Tenez compte des éléments suivants lorsque vous intégrez des artefacts à votre bibliothèque :

* S&#39;il faut journaliser des points de contrôle du modèle ou des jeux de données en tant qu&#39;artefacts (si vous souhaitez rendre cela facultatif).
* Références d&#39;entrée d&#39;artefact (par exemple, `entity/project/artifact`).
* Fréquence de journalisation des points de contrôle du modèle ou des jeux de données. Par exemple, à chaque époque, toutes les 500 étapes, etc.

<div id="log-model-checkpoints">
  ### Journaliser les points de contrôle du modèle
</div>

Journalisez les points de contrôle du modèle dans W&amp;B. Une approche courante consiste à journaliser les points de contrôle en tant qu&#39;artefacts, en utilisant l&#39;ID unique du run généré par W&amp;B dans le nom de l&#39;artefact.

```python
metadata = {"eval/accuracy": 0.8, "train/steps": 800} 

artifact = wandb.Artifact(
                name=f"model-{run.id}", 
                metadata=metadata, 
                type="model"
                ) 
artifact.add_dir("output_model") # répertoire local où sont stockés les poids du modèle

aliases = ["best", "epoch_10"] 
run.log_artifact(artifact, aliases=aliases)
```

L’extrait de code précédent montre comment journaliser un point de contrôle du modèle comme artefact et y ajouter des métadonnées, telles que la précision lors de l’évaluation et les étapes d’entraînement. L’artefact reçoit un nom qui inclut l’identifiant unique du run, et il est associé à des [alias personnalisés](/fr/models/artifacts/create-a-custom-alias/) pour le retrouver facilement.

<div id="log-input-artifacts">
  ### Journaliser les artefacts en entrée
</div>

Journalisez les jeux de données ou les modèles préentraînés utilisés en entrée :

```python
dataset = wandb.Artifact(name="flowers", type="dataset")
dataset.add_file("flowers.npy")
run.use_artifact(dataset)
```

L’extrait de code précédent crée un artefact pour un jeu de données nommé &quot;flowers&quot; et y ajoute un fichier. L’artefact est ensuite associé au run en cours à l’aide de `run.use_artifact()`, ce qui permet à W&amp;B de suivre la traçabilité du jeu de données utilisé dans ce run.

<div id="download-artifacts">
  ### Télécharger des artefacts
</div>

Téléchargez des artefacts précédemment journalisés depuis W&amp;B afin de les utiliser dans votre code d’entraînement ou d’inférence.

Si vous disposez d’un contexte de run, utilisez [`wandb.Run.use_artifact()`](/fr/models/ref/python/experiments/run) pour faire référence à un artefact dans W&amp;B, puis appelez [`wandb.Artifact.download()`](/fr/models/ref/python/experiments/artifact) pour le télécharger dans un répertoire local.

```python
with wandb.init(...) as run:
    artifact = run.use_artifact("user/project/artifact:latest")
    local_path = artifact.download()
```

Utilisez l&#39;[API publique W&amp;B](/fr/models/ref/python/public-api/) pour faire référence à un artefact et le télécharger sans initialiser de run. Cela est utile dans des scénarios tels que les environnements distribués ou lors d&#39;opérations d&#39;Inférence, où vous ne souhaitez pas forcément créer un nouveau run.

```python
import wandb
artifact = wandb.Api().artifact("user/project/artifact:latest")
local_path = artifact.download()
```

Voir [Télécharger et utiliser des artefacts](/fr/models/artifacts/download-and-use-an-artifact/) pour en savoir plus.
