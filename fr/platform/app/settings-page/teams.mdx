Utilisez W&amp;B Teams comme espace de travail central pour aider votre équipe ML à créer de meilleurs modèles plus rapidement. Les Administrateurs de l’organisation et les Administrateurs d’équipe peuvent consulter et modifier les paramètres d’une équipe.

* **Suivez toutes les expériences** menées par votre équipe pour ne jamais refaire le même travail.
* **Enregistrez et reproduisez** des modèles déjà entraînés.
* **Partagez les progrès** et les résultats avec votre responsable et vos collaborateurs.
* **Détectez les régressions** et recevez immédiatement une alerte lorsque les performances baissent.
* **Évaluez la performance du modèle** et comparez les versions du modèle.

<div id="create-a-collaborative-team">
  ## Créer une équipe collaborative
</div>

1. [Inscrivez-vous ou connectez-vous](https://app.wandb.ai/login?signup=true) à votre compte W&amp;B gratuit.
2. Cliquez sur **Invite Team** dans la barre de navigation.
3. Créez votre équipe et invitez des collaborateurs.
4. Configurez votre équipe à l’aide des paramètres décrits ci-dessous.

<Note>
  **Remarque** : seul l’administrateur d’une organisation peut créer une nouvelle équipe.
</Note>

<div id="team-configuration">
  ## Configuration de l’équipe
</div>

Vous pouvez configurer différents paramètres pour votre équipe, notamment les membres, les alertes et les paramètres de confidentialité.

Pour gérer les paramètres de votre équipe, accédez à la section **Teams** dans le menu de gauche et cliquez sur l’équipe dont vous souhaitez modifier les paramètres. Vous pouvez mettre à jour les paramètres suivants :

<div id="members">
  ### Membres
</div>

La section Membres affiche une liste de toutes les invitations en attente ainsi que des membres qui ont accepté l’invitation à rejoindre l’équipe. La liste affiche, pour chaque membre, son nom, son nom d’utilisateur, son adresse e-mail, son rôle dans l’équipe et ses privilèges d’accès à Models et à W&amp;B Weave, hérités de l’organisation. Vous pouvez choisir parmi les rôles d’équipe standard **Admin**, **membre** et **lecture seule**. Si votre organisation a créé des [rôles personnalisés](/fr/platform/hosting/iam/access-management/manage-organization#add-and-manage-custom-roles), vous pouvez attribuer un rôle personnalisé à la place.

Voir [Ajouter et gérer les équipes](/fr/platform/hosting/iam/access-management/manage-organization#add-and-manage-teams) pour savoir comment créer une équipe, gérer les équipes, ainsi que gérer les membres et les rôles d’équipe. Pour configurer qui peut inviter de nouveaux membres et définir les autres paramètres de confidentialité de l’équipe, se référer à [Confidentialité](#privacy).

Pour retirer un membre de l’équipe, les administrateurs peuvent ouvrir la page des paramètres de l’équipe et cliquer sur le bouton de suppression à côté du nom du membre concerné. Tous les runs journalisés pour l’équipe restent après le départ d’un utilisateur.

<div id="avatar">
  ### Avatar
</div>

Pour définir un avatar pour votre équipe :

1. Accédez à la section **Teams** dans le menu de gauche et cliquez sur l’équipe à laquelle vous souhaitez ajouter un avatar. Cela ouvre la page d’aperçu de l’équipe.
2. Survolez l’image d’avatar par défaut de l’équipe, dans l’angle supérieur gauche de la page, puis cliquez sur le bouton **Téléverser une photo**. Cela ouvre une boîte de dialogue de sélection de fichier.
3. Dans la boîte de dialogue, sélectionnez l’image que vous souhaitez utiliser, puis cliquez sur **Ouvrir**. Cela téléverse la photo pour votre équipe et la définit comme avatar de l’équipe.

<div id="alerts">
  ### Alertes
</div>

Vous pouvez configurer des alertes pour notifier votre équipe lorsqu’un run échoue ou se termine. Les alertes peuvent être envoyées par e-mail ou via Slack, et vous pouvez les personnaliser en fonction de vos besoins.

Dans la section **Alertes d&#39;équipe**, basculez l’interrupteur situé à côté du type d’événement pour lequel vous souhaitez recevoir des alertes. Weights &amp; Biases propose les types d’événements suivants :

* **Runs terminés** : indique si un run Weights &amp; Biases s’est terminé avec succès.
* **Run planté** : indique qu’un run n’a pas réussi à se terminer.
* **wandb.alert()** : une alerte personnalisée pouvant être déclenchée par script. Voir [Envoyer des alertes avec `wandb.Run.alert()`](/fr/models/runs/alert) pour plus d’informations.

<div id="slack-notifications">
  ### Notifications Slack
</div>

Configurez les destinations Slack vers lesquelles les [automatisations](/fr/models/automations/) de votre équipe peuvent envoyer des notifications lorsqu&#39;un événement se produit dans un registre ou un projet, par exemple lorsqu&#39;un nouvel artefact est créé ou lorsqu&#39;une métrique de run atteint un seuil défini. Se référer à [Créer une automatisation Slack](/fr/models/automations/create-automations/slack).

<div id="privacy">
  ### Confidentialité
</div>

Les paramètres de confidentialité de l’équipe vous permettent de gérer la visibilité et les options de partage au sein de votre équipe. Seuls les Administrateurs de l’organisation peuvent modifier les paramètres de confidentialité.

Vous pouvez modifier les paramètres de confidentialité suivants :

* Masquer cette équipe pour tous les non-membres.
* Rendre privés tous les futurs projets de l’équipe (le partage public n’est pas autorisé).
* Autoriser n’importe quel membre de l’équipe à inviter d’autres membres de l’équipe (et pas seulement les administrateurs).
* Désactiver le partage public à l’extérieur de l’équipe pour les rapports des projets privés. Cela désactive les liens magiques existants.
* Recommander automatiquement aux nouveaux utilisateurs dont le domaine d’e-mail correspond de rejoindre cette équipe lors de l’inscription.
* Activer l’enregistrement du code par défaut.

Vous pouvez voir les paramètres de confidentialité de tous les projets de l’équipe sur la page des paramètres de l’équipe : `app.wandb.ai/teams/your-team-name`

<div id="create-a-team-profile">
  ## Créer un profil d’équipe
</div>

Vous pouvez personnaliser la page de profil de votre équipe pour y afficher une présentation et mettre en avant des rapports et des projets visibles par le public ou par les membres de l’équipe. Présentez des rapports, des projets et des liens externes.

* **Mettez en avant vos meilleures recherches** auprès des visiteurs en présentant vos meilleurs rapports publics

<div id="team-roles-and-permissions">
  ## Rôles et autorisations d’équipe
</div>

Sélectionnez un rôle d’équipe lorsque vous invitez des collègues à rejoindre une équipe. Les rôles d’équipe suivants sont disponibles :

* **Admin** : Les Administrateurs d’équipe peuvent ajouter et supprimer d’autres administrateurs ou membres de l’équipe. Ils sont autorisés à modifier tous les projets et disposent d’autorisations de suppression complètes. Cela inclut notamment la suppression de runs, de projets, d’artefacts et de balayages.
* **Membre** : Un membre standard de l’équipe. Par défaut, seul un administrateur peut inviter un membre de l’équipe. Pour modifier ce comportement, se référer à [Paramètres de confidentialité](#privacy).
* **Lecture seule (fonctionnalité réservée à Enterprise)** : Les membres en lecture seule peuvent voir les ressources de l’équipe, telles que les runs, les rapports et les espaces de travail. Ils peuvent suivre les rapports et y ajouter des commentaires, mais ils ne peuvent pas créer, modifier ou supprimer l’aperçu du projet, les rapports ou les runs.
* **Rôles personnalisés (fonctionnalité réservée à Enterprise)** : Les rôles personnalisés permettent aux Administrateurs de l’organisation de créer de nouveaux rôles basés sur les rôles **Lecture seule** ou **Membre**, avec des autorisations supplémentaires pour mettre en place un contrôle d’accès granulaire. Les Administrateurs d’équipe peuvent ensuite attribuer l’un de ces rôles personnalisés aux utilisateurs de leurs équipes respectives. Se référer à [Présentation des rôles personnalisés pour les équipes W&amp;B](https://wandb.ai/wandb_fc/announcements/reports/Introducing-Custom-Roles-for-W-B-Teams--Vmlldzo2MTMxMjQ3) pour plus de détails.

Un membre de l’équipe ne peut supprimer que les runs qu’il a créés. Supposons que vous ayez deux membres, A et B. Le membre B déplace un run du projet de l’équipe B vers un autre projet appartenant au membre A. Le membre A ne peut pas supprimer le run que le membre B a déplacé vers le projet du membre A. Un administrateur peut gérer les runs ainsi que les runs issus de balayages créés par n’importe quel membre de l’équipe.

<div id="service-accounts">
  ### Comptes de service
</div>

En plus des rôles utilisateur, les équipes peuvent également utiliser des **comptes de service** à des fins d’automatisation. Les comptes de service ne sont pas des utilisateurs, mais des identités non humaines utilisées pour des flux de travail automatisés. Pour plus d’informations, se référer à [Utiliser des comptes de service pour automatiser les flux de travail](/fr/platform/hosting/iam/service-accounts).

<Note>
  W&amp;B recommande d’attribuer au moins deux administrateurs à une équipe afin de garantir la continuité des opérations administratives lorsque l’administrateur principal n’est pas disponible.
</Note>

<div id="team-service-account-behavior">
  ### Comportement des comptes de service d’équipe
</div>

* Lorsque vous configurez une équipe dans votre environnement d’entraînement, vous pouvez utiliser un compte de service de cette équipe pour journaliser des runs dans des projets privés ou publics de cette équipe. De plus, vous pouvez attribuer ces runs à un utilisateur si la variable **WANDB&#95;USERNAME** ou **WANDB&#95;USER&#95;EMAIL** existe dans votre environnement et si l’utilisateur référencé fait partie de cette équipe.
* Lorsque vous **ne** configurez **pas** d’équipe dans votre environnement d’entraînement et que vous utilisez un compte de service, les runs sont journalisés dans le projet spécifié au sein de l’équipe parente de ce compte de service. Dans ce cas également, vous pouvez attribuer les runs à un utilisateur si la variable **WANDB&#95;USERNAME** ou **WANDB&#95;USER&#95;EMAIL** existe dans votre environnement et si l’utilisateur référencé fait partie de l’équipe parente du compte de service.
* Un compte de service ne peut pas journaliser de runs dans un projet privé d’une équipe différente de son équipe parente. Un compte de service peut journaliser des runs dans un projet uniquement si la visibilité du projet est définie sur `Open`.

<div id="secure-storage-connector">
  ### Connecteur de stockage sécurisé
</div>

Le connecteur de stockage sécurisé au niveau de l’équipe permet aux équipes d’utiliser leur propre bucket de stockage cloud avec W&amp;B. Cela offre un meilleur contrôle d’accès aux données et une meilleure isolation des données aux équipes qui traitent des données très sensibles ou qui sont soumises à des exigences strictes en matière de conformité. Pour plus d’informations, se référer à [Secure Storage Connector](/fr/platform/hosting/data-security/secure-storage-connector).
