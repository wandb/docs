---
title: Configurer Launch
description: "Configurez W&B Launch en configurant une file d'attente et en déployant un agent pour soumettre des jobs à votre infrastructure de calcul."
---

Cette page décrit les étapes générales requises pour configurer W&amp;B Launch :

1. **Configurer une file d&#39;attente** : Les files d&#39;attente fonctionnent en FIFO et disposent d&#39;une configuration de la file d’attente. La configuration d&#39;une file d&#39;attente détermine où et comment les jobs sont exécutés sur une ressource cible.
2. **Configurer un agent** : Les agents s&#39;exécutent sur votre machine ou votre infrastructure et interrogent une ou plusieurs files d&#39;attente pour récupérer les jobs Launch. Lorsqu&#39;un job est récupéré, l&#39;agent s&#39;assure que l&#39;image est construite et disponible. L&#39;agent soumet ensuite le job à la ressource cible.

<div id="set-up-a-queue">
  ## Configurer une file d&#39;attente
</div>

Les files d&#39;attente Launch doivent être configurées de manière à pointer vers une ressource cible spécifique, ainsi qu’avec toute configuration supplémentaire propre à cette ressource. Par exemple, une file d&#39;attente Launch qui pointe vers un cluster Kubernetes peut inclure des variables d&#39;environnement ou définir un espace de noms personnalisé dans sa configuration de la file d’attente. Lorsque vous créez une file d&#39;attente, vous devez spécifier à la fois la ressource cible à utiliser et la configuration associée à cette ressource.

Lorsqu&#39;un agent reçoit un job depuis une file d&#39;attente, il reçoit également la configuration de la file d’attente. Lorsque l&#39;agent soumet le job à la ressource cible, il inclut la configuration de la file d’attente ainsi que les éventuelles substitutions définies dans le job lui-même. Par exemple, vous pouvez utiliser une configuration de job pour spécifier le type d&#39;instance Amazon SageMaker uniquement pour ce job. Dans ce cas, il est courant d&#39;utiliser les [modèles de configuration de file d&#39;attente](/fr/platform/launch/setup-queue-advanced/#configure-queue-template) comme interface destinée à l&#39;utilisateur final.

<div id="create-a-queue">
  ### Créer une file d&#39;attente
</div>

1. Accédez à l&#39;application Launch sur [wandb.ai/launch](https://wandb.ai/launch).
2. Cliquez sur le bouton **create queue** en haut à droite de l&#39;écran.

<Frame>
  <img src="/images/launch/create-queue.gif" alt="Création d'une file d'attente Launch" />
</Frame>

3. Dans le menu déroulant **Entité**, sélectionnez l&#39;entité à laquelle la file d&#39;attente sera rattachée.
4. Saisissez un nom pour votre file d&#39;attente dans le champ **Queue**.
5. Dans le menu déroulant **Resource**, sélectionnez la ressource de calcul que les jobs ajoutés à cette file d&#39;attente doivent utiliser.
6. Choisissez si vous souhaitez autoriser la **Prioritization** pour cette file d&#39;attente. Si la priorisation est activée, un utilisateur de votre équipe peut définir une priorité pour son job Launch au moment de l&#39;ajouter à la file d&#39;attente. Les jobs ayant une priorité plus élevée sont exécutés avant ceux ayant une priorité plus faible.
7. Fournissez une configuration de ressource au format JSON ou YAML dans le champ **Configuration**. La structure et la sémantique de votre document de configuration dépendent du type de ressource vers lequel pointe la file d&#39;attente. Pour plus de détails, voir la page de configuration dédiée à votre ressource cible.

<div id="set-up-a-launch-agent">
  ## Configurer un agent Launch
</div>

Les agents Launch sont des processus de longue durée qui interrogent une ou plusieurs files d&#39;attente Launch à la recherche de jobs. Les agents Launch retirent les jobs des files d&#39;attente selon l&#39;ordre premier entré, premier sorti (FIFO) ou par ordre de priorité, selon les files d&#39;attente qu&#39;ils consultent. Lorsqu&#39;un agent retire un job d&#39;une file d&#39;attente, il peut, si nécessaire, build une image pour ce job. L&#39;agent soumet ensuite le job à la ressource cible avec les options de configuration spécifiées dans la configuration de la file d’attente.

<Note>
  Les agents sont très flexibles et peuvent être configurés pour prendre en charge une grande variété de cas d’usage. La configuration requise pour votre agent dépend de votre cas d’usage spécifique. Voir la page dédiée à [Docker](/fr/platform/launch/setup-launch-docker/), [Amazon SageMaker](/fr/platform/launch/setup-launch-sagemaker/), [Kubernetes](/fr/platform/launch/setup-launch-kubernetes/) ou [Vertex AI](/fr/platform/launch/setup-vertex/).
</Note>

<Note>
  W&amp;B recommande de lancer les agents avec la clé API d&#39;un compte de service, plutôt qu&#39;avec la clé API d&#39;un utilisateur spécifique. L&#39;utilisation de la clé API d&#39;un compte de service présente deux avantages :

  1. L&#39;agent ne dépend pas d&#39;un utilisateur en particulier.
  2. Pour Launch, l’auteur associé à un run créé via Launch est considéré comme étant l’utilisateur qui a soumis le job Launch, plutôt que l’utilisateur associé à l’agent.
</Note>

<div id="agent-configuration">
  ### Configuration de l’agent
</div>

Configurez l’agent Launch à l’aide d’un fichier YAML nommé `launch-config.yaml`. Par défaut, W&amp;B recherche le fichier de configuration dans `~/.config/wandb/launch-config.yaml`. Vous pouvez aussi spécifier un autre répertoire lorsque vous activez l’agent Launch.

Le contenu du fichier de configuration de votre agent Launch dépend de l’environnement de l’agent, de la ressource cible de la file d’attente Launch, des exigences du builder Docker, des exigences du registre cloud, etc.

Quel que soit votre cas d’utilisation, certaines options de configuration de base sont disponibles pour l’agent Launch :

* `max_jobs` : nombre maximal de jobs que l’agent peut exécuter en parallèle
* `entity` : l’entité à laquelle appartient la file d’attente
* `queues` : le nom d’une ou de plusieurs files d’attente que l’agent doit surveiller

<Note>
  Vous pouvez utiliser la CLI W&amp;B pour spécifier des options de configuration universelles pour l’agent Launch (au lieu du fichier de configuration YAML) : nombre maximal de jobs, entité W&amp;B et files d’attente Launch. Voir la commande [`wandb launch-agent`](/fr/models/ref/cli/wandb-launch-agent) pour en savoir plus.
</Note>

L’extrait YAML suivant montre comment spécifier les clés de configuration de base de l’agent Launch :

```yaml title="launch-config.yaml"
# Nombre maximum de runs simultanés à effectuer. -1 = aucune limite
max_jobs: -1

entity: <entity-name>

# Liste des files d'attente à interroger.
queues:
  - <queue-name>
```

<div id="configure-a-container-builder">
  ### Configurer un builder de conteneurs
</div>

L’agent Launch peut être configuré pour build des images. Vous devez configurer l’agent pour utiliser un builder de conteneurs si vous prévoyez d’utiliser des jobs Launch créés à partir de dépôts Git ou d’artefacts de code. Voir [Créer un job Launch](/fr/platform/launch/create-launch-job/) pour plus d’informations sur la création d’un job Launch.

W&amp;B Launch prend en charge trois options de builder :

* Docker : Le builder Docker utilise un démon Docker local pour build des images.
* [Kaniko](https://github.com/GoogleContainerTools/kaniko) : Kaniko est un projet Google qui permet de build des images dans des environnements où un démon Docker n’est pas disponible.
* Noop : L’agent ne tentera pas de build les jobs et se contentera de récupérer des images préconstruites.

<Note>
  Utilisez le builder Kaniko si votre agent s’exécute dans un environnement où un démon Docker n’est pas disponible (par exemple, un cluster Kubernetes).

  Voir [Configurer Kubernetes](/fr/platform/launch/setup-launch-kubernetes/) pour plus de détails sur le builder Kaniko.
</Note>

Pour spécifier un builder d’images, incluez la clé builder dans la configuration de votre agent. Par exemple, l’extrait de code suivant montre une partie de la configuration Launch (`launch-config.yaml`) qui indique d’utiliser Docker ou Kaniko :

```yaml title="launch-config.yaml"
builder:
  type: docker | kaniko | noop
```

<div id="configure-a-container-registry">
  ### Configurer un registre de conteneurs
</div>

Dans certains cas, vous pouvez vouloir connecter un agent Launch à un registre cloud. Voici quelques scénarios courants où cela peut être utile :

* Vous souhaitez exécuter un job dans un environnement autre que celui où vous l&#39;avez buildé, par exemple sur une station de travail plus puissante ou sur un cluster.
* Vous souhaitez utiliser l&#39;agent pour builder des images et les exécuter sur Amazon SageMaker ou VertexAI.
* Vous souhaitez que l&#39;agent Launch fournisse les identifiants nécessaires pour extraire des images depuis un dépôt d&#39;images.

Pour en savoir plus sur la manière de configurer l&#39;agent afin qu&#39;il interagisse avec un registre de conteneurs, voir la page [configuration avancée de l&#39;agent](/fr/platform/launch/setup-agent-advanced/).

<div id="activate-the-launch-agent">
  ## Activer l’agent Launch
</div>

Activez l’agent Launch à l’aide de la commande CLI W&amp;B `launch-agent` :

```bash
wandb launch-agent -q <queue-1> -q <queue-2> --max-jobs 5
```

Dans certains cas, vous souhaiterez peut-être qu’un agent Launch interroge des files d’attente depuis l’intérieur d’un cluster Kubernetes. Voir la [page de configuration avancée des files d’attente](/fr/platform/launch/setup-queue-advanced/) pour plus d’informations.
