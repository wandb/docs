Les jobs Launch servent de modèles pour reproduire des runs W&amp;B. Les jobs sont des Artifacts W&amp;B qui capturent le code source, les dépendances et les entrées nécessaires à l’exécution d’une charge de travail.

Créez et exécutez des jobs avec la commande `wandb launch`.

<Note>
  Pour créer un job sans le soumettre à l’exécution, utilisez la commande `wandb job create`. Voir la [documentation de référence de la commande](/fr/models/ref/cli/wandb-job/wandb-job-create) pour plus d’informations.
</Note>

<div id="git-jobs">
  ## Jobs Git
</div>

Vous pouvez créer un job basé sur Git, dans lequel le code et les autres éléments suivis sont clonés à partir d’un commit, d’une branche ou d’un tag spécifiques d’un dépôt Git distant avec W&amp;B Launch. Utilisez l’option `--uri` ou `-u` pour spécifier l’URI contenant le code et, facultativement, l’option `--build-context` pour indiquer un sous-répertoire.

Exécutez un job &quot;hello world&quot; à partir d’un dépôt Git avec la commande suivante :

```bash
wandb launch --uri "https://github.com/wandb/launch-jobs.git" --build-context jobs/hello_world --dockerfile Dockerfile.wandb --project "hello-world" --job-name "hello-world" --entry-point "python job.py"
```

La commande effectue les opérations suivantes :

1. Clone le [dépôt de jobs W&amp;B Launch](https://github.com/wandb/launch-jobs) dans un répertoire temporaire.
2. Crée un job nommé **hello-world-git** dans le projet **hello**. Le job est associé au commit en tête de la branche par défaut du dépôt.
3. Build une image de conteneur à partir du répertoire `jobs/hello_world` et du fichier `Dockerfile.wandb`.
4. Démarre le conteneur et exécute `python job.py`.

Pour build un job à partir d&#39;une branche spécifique ou d&#39;un hash de commit, ajoutez l&#39;argument `-g`, `--git-hash`. Pour obtenir la liste complète des arguments, exécutez `wandb launch --help`.

<div id="remote-url-format">
  ### Format de l’URL distante
</div>

Le dépôt Git distant associé à un job Launch peut utiliser une URL HTTPS ou SSH. Le type d’URL détermine le protocole utilisé pour récupérer le code source du job.

| Type d’URL distante | Format d’URL                                     | Exigences relatives à l’accès et à l’authentification                                |
| ------------------- | ------------------------------------------------ | ------------------------------------------------------------------------------------ |
| https               | `https://github.com/organization/repository.git` | nom d’utilisateur et mot de passe pour vous authentifier auprès du dépôt Git distant |
| ssh                 | `git@github.com:organization/repository.git`     | clé SSH pour vous authentifier auprès du dépôt Git distant                           |

Notez que le format exact de l’URL varie selon le fournisseur d’hébergement. Les jobs créés avec `wandb launch --uri` utiliseront le protocole de transfert spécifié par le `--uri` fourni.

<div id="code-artifact-jobs">
  ## Jobs d’artefacts de code
</div>

Vous pouvez créer des jobs à partir de n’importe quel code source stocké dans un artefact W&amp;B. Utilisez un répertoire local avec l’argument `--uri` ou `-u` pour créer un nouvel artefact de code et un job.

Pour commencer, créez un répertoire vide et ajoutez un script Python nommé `main.py` contenant le code suivant :

```python
import wandb

with wandb.init() as run:
    run.log({"metric": 0.5})
```

Ajoutez un fichier `requirements.txt` avec le contenu suivant :

```txt
wandb>=0.17.1
```

Enregistrez le répertoire en tant qu’artefact de code et lancez un job avec la commande suivante :

```bash
wandb launch --uri . --job-name hello-world-code --project launch-quickstart --entry-point "python main.py"
```

La commande précédente effectue les actions suivantes :

1. Enregistre le répertoire actuel en tant qu’artefact de code nommé `hello-world-code`.
2. Crée un job nommé `hello-world-code` dans le projet `launch-quickstart`.
3. Build une image de conteneur à partir du répertoire actuel et du Dockerfile par défaut de Launch. Ce Dockerfile installera le fichier `requirements.txt` et définira le point d’entrée sur `python main.py`.

<div id="image-jobs">
  ## Jobs d’image
</div>

Vous pouvez également créer des jobs à partir d’images Docker prêtes à l’emploi. C’est utile lorsque vous disposez déjà d’un système de build établi pour votre code ML, ou lorsque vous ne prévoyez pas de modifier le code ou les dépendances du job, mais souhaitez tout de même tester différents hyperparamètres ou différentes tailles d’infrastructure.

L’image est récupérée depuis un registre Docker et exécutée avec le point d’entrée spécifié, ou avec le point d’entrée par défaut si aucun n’est indiqué. Passez un tag d’image complet à l’option `--docker-image` pour créer et exécuter un job à partir d’une image Docker.

Pour exécuter un job simple à partir d’une image prête à l’emploi, utilisez la commande suivante :

```bash
wandb launch --docker-image "wandb/job_hello_world:main" --project "hello-world"           
```

<div id="automatic-job-creation">
  ## Création automatique de job
</div>

W&amp;B crée et suit automatiquement un job pour tout run dont le code source est suivi, même si ce run n’a pas été créé avec Launch. Les runs sont considérés comme ayant un code source suivi si l’une des trois conditions suivantes est remplie :

* Le run a un dépôt Git distant associé et un hash de commit.
* Le run a enregistré un artefact de code. Voir [`Run.log_code`](/fr/models/ref/python/experiments/run#log_code).
* Le run a été exécuté dans un conteneur Docker avec la variable d’environnement `WANDB_DOCKER` définie sur un tag d’image

L’URL du dépôt Git distant est déduite du dépôt Git local si votre job Launch est créé automatiquement par un run W&amp;B.

<div id="launch-job-names">
  ### Noms des jobs Launch
</div>

Par défaut, W&amp;B génère automatiquement un nom de job. Ce nom dépend de la façon dont le job est créé (GitHub, artefact de code ou image Docker). Vous pouvez également définir le nom d&#39;un job Launch à l&#39;aide de variables d&#39;environnement ou du SDK Python de W&amp;B.

Le tableau suivant décrit la convention de nommage des jobs utilisée par défaut selon la source du job :

| Source           | Convention de nommage                   |
| ---------------- | --------------------------------------- |
| GitHub           | `job-<git-remote-url>-<path-to-script>` |
| Artefact de code | `job-<code-artifact-name>`              |
| Image Docker     | `job-<image-name>`                      |

Nommez votre job à l&#39;aide d&#39;une variable d&#39;environnement W&amp;B ou du SDK Python de W&amp;B

<Tabs>
  <Tab title="Variable d'environnement">
    Définissez la variable d&#39;environnement `WANDB_JOB_NAME` avec le nom de job de votre choix. Par exemple :

    ```bash
    WANDB_JOB_NAME=awesome-job-name
    ```
  </Tab>

  <Tab title="W&B Python SDK">
    Définissez le nom de votre job avec `wandb.Settings`. Passez ensuite cet objet lorsque vous initialisez W&amp;B avec `wandb.init`. Par exemple :

    ```python
    settings = wandb.Settings(job_name="my-job-name")
    wandb.init(settings=settings)
    ```
  </Tab>
</Tabs>

<Note>
  Pour les jobs basés sur une image Docker, l&#39;alias de version est automatiquement ajouté au job en tant qu&#39;alias.
</Note>

<div id="containerization">
  ## Conteneurisation
</div>

Les jobs sont exécutés dans un conteneur. Les jobs d’image utilisent une image Docker prébuildée, tandis que les jobs Git et les jobs d’artefact de code nécessitent une étape de build du conteneur.

La conteneurisation des jobs peut être personnalisée à l’aide d’arguments passés à `wandb launch` et de fichiers présents dans le code source du job.

<div id="build-context">
  ### Contexte de build
</div>

Le terme « contexte de build » désigne l’arborescence des fichiers et répertoires envoyés au démon Docker pour build une image de conteneur. Par défaut, Launch utilise la racine du code source du job comme contexte de build. Pour spécifier un sous-répertoire comme contexte de build, utilisez l’argument `--build-context` de `wandb launch` lors de la création et du lancement d’un job.

<Note>
  L’argument `--build-context` est particulièrement utile lorsque vous travaillez avec des jobs Git qui se réfèrent à un monorepo contenant plusieurs projets. En spécifiant un sous-répertoire comme contexte de build, vous pouvez build une image de conteneur pour un projet spécifique au sein du monorepo.

  Voir l’[exemple ci-dessus](#git-jobs) pour une démonstration de l’utilisation de l’argument `--build-context` avec le dépôt officiel des jobs Launch de W&amp;B.
</Note>

<div id="dockerfile">
  ### Dockerfile
</div>

Le Dockerfile est un fichier texte qui contient des instructions pour build une image Docker. Par défaut, Launch utilise un Dockerfile par défaut qui installe le fichier `requirements.txt`. Pour utiliser un Dockerfile personnalisé, indiquez le chemin du fichier avec l&#39;argument `--dockerfile` de `wandb launch`.

Le chemin du Dockerfile est indiqué par rapport au contexte de build. Par exemple, si le contexte de build est `jobs/hello_world` et que le Dockerfile se trouve dans le répertoire `jobs/hello_world`, l&#39;argument `--dockerfile` doit être défini sur `Dockerfile.wandb`. Voir l’[exemple ci-dessus](#git-jobs) pour une démonstration de comment utiliser l&#39;argument `--dockerfile` avec le dépôt officiel des jobs Launch de W&amp;B.

<div id="requirements-file">
  ### Fichier requirements
</div>

Si aucun Dockerfile personnalisé n&#39;est fourni, Launch recherchera dans le contexte de build les dépendances Python à installer. Si un fichier `requirements.txt` est trouvé à la racine du contexte de build, Launch installera les dépendances répertoriées dans ce fichier. Sinon, si un fichier `pyproject.toml` est trouvé, Launch installera les dépendances depuis la section `project.dependencies`.
