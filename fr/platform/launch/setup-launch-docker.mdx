Le guide suivant décrit comment configurer W&amp;B Launch pour utiliser Docker sur une machine locale, à la fois comme environnement de l’agent Launch et comme ressource cible de la file d’attente.

Utiliser Docker pour exécuter des jobs et comme environnement de l’agent Launch sur la même machine locale est particulièrement utile si vos ressources de calcul se trouvent sur une machine qui ne dispose pas d’un système de gestion de cluster (comme Kubernetes).

Vous pouvez également utiliser des files d’attente Docker pour exécuter des charges de travail sur des stations de travail puissantes.

<Note>
  Cette configuration est courante pour les utilisateurs qui exécutent des expériences sur leur machine locale, ou qui disposent d’une machine distante à laquelle ils se connectent en SSH pour soumettre des jobs Launch.
</Note>

Lorsque vous utilisez Docker avec W&amp;B Launch, W&amp;B build d’abord une image, puis build et exécute un conteneur à partir de cette image. L’image est buildée avec la commande Docker `docker run <image-uri>`. La configuration de la file d’attente est interprétée comme des arguments supplémentaires transmis à la commande `docker run`.

<div id="configure-a-docker-queue">
  ## Configurer une file d’attente Docker
</div>

La configuration de la file d’attente Launch (pour une ressource cible Docker) accepte les mêmes options que celles définies dans la commande CLI [`docker run`](/fr/models/ref/cli/wandb-docker-run).

L’agent reçoit les options définies dans la configuration de la file d’attente. Il fusionne ensuite les options reçues avec les surcharges éventuelles de la configuration du job Launch afin de produire une commande `docker run` finale, qui est exécutée sur la ressource cible (dans ce cas, une machine locale).

Deux transformations syntaxiques s’appliquent :

1. Les options répétées sont définies dans la configuration de la file d’attente sous forme de liste.
2. Les options de type indicateur sont définies dans la configuration de la file d’attente comme une valeur booléenne `true`.

Par exemple, la configuration de la file d’attente suivante :

```json
{
  "env": ["MY_ENV_VAR=value", "MY_EXISTING_ENV_VAR"],
  "volume": "/mnt/datasets:/mnt/datasets",
  "rm": true,
  "gpus": "all"
}
```

Donne la commande `docker run` suivante :

```bash
docker run \
  --env MY_ENV_VAR=value \
  --env MY_EXISTING_ENV_VAR \
  --volume "/mnt/datasets:/mnt/datasets" \
  --rm <image-uri> \
  --gpus all
```

Les volumes peuvent être spécifiés soit sous forme de liste de chaînes, soit sous forme de chaîne unique. Utilisez une liste si vous spécifiez plusieurs volumes.

Docker transmet automatiquement, depuis l’environnement de l’agent Launch, les variables d’environnement auxquelles aucune valeur n’est attribuée. Cela signifie que, si l’agent Launch possède une variable d’environnement `MY_EXISTING_ENV_VAR`, cette variable est disponible dans le conteneur. C’est utile si vous souhaitez utiliser d’autres clés de configuration sans les publier dans la configuration de la file d’attente.

L’option `--gpus` de la commande `docker run` vous permet de spécifier les GPU disponibles pour un conteneur Docker. Pour en savoir plus sur l’utilisation de l’option `gpus`, voir la [documentation Docker](https://docs.docker.com/config/containers/resource_constraints/#gpu).

<Note>
  * Installez le [NVIDIA Container Toolkit](https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/install-guide.html#docker) pour utiliser des GPU dans un conteneur Docker.
  * Si vous créez des images à partir d’un job basé sur du code ou sur un artefact, vous pouvez remplacer l’image de base utilisée par l’[agent Launch](#configure-a-launch-agent-on-a-local-machine) afin d’y inclure le NVIDIA Container Toolkit.
    Par exemple, dans votre file d’attente Launch, vous pouvez remplacer l’image de base par `tensorflow/tensorflow:latest-gpu` :

    ```json
    {
      "builder": {
        "accelerator": {
          "base_image": "tensorflow/tensorflow:latest-gpu"
        }
      }
    }
    ```
</Note>

<div id="create-a-queue">
  ## Créer une file d’attente
</div>

Créez une file d’attente utilisant Docker comme ressource de calcul avec la CLI W&amp;B :

1. Accédez à la [page Launch](https://wandb.ai/launch).
2. Cliquez sur le bouton **Créer une file d’attente**.
3. Sélectionnez l’**entité** dans laquelle vous souhaitez créer la file d’attente.
4. Saisissez un nom pour votre file d’attente dans le champ **Nom**.
5. Sélectionnez **Docker** comme **ressource**.
6. Définissez la configuration de votre file d’attente Docker dans le champ **Configuration**.
7. Cliquez sur le bouton **Créer une file d’attente** pour la créer.

<div id="configure-a-launch-agent-on-a-local-machine">
  ## Configurer un agent Launch sur une machine locale
</div>

Configurez l’agent Launch à l’aide d’un fichier de configuration YAML nommé `launch-config.yaml`. Par défaut, W&amp;B recherche ce fichier de configuration dans `~/.config/wandb/launch-config.yaml`. Vous pouvez facultativement spécifier un autre répertoire lorsque vous activez l’agent Launch.

<Note>
  Vous pouvez utiliser la CLI W&amp;B pour spécifier les principales options de configuration de l’agent Launch (au lieu du fichier de configuration YAML) : le nombre maximal de jobs, l’entité W&amp;B et les files d’attente Launch. Voir la commande [`wandb launch-agent`](/fr/models/ref/cli/wandb-launch-agent) pour plus d’informations.
</Note>

<div id="core-agent-config-options">
  ## Options de configuration principales de l’agent
</div>

Les onglets suivants montrent comment spécifier les options de configuration principales de l’agent à l’aide du CLI W&amp;B et d’un fichier de configuration YAML :

<Tabs>
  <Tab title="W&B CLI">
    ```bash
    wandb launch-agent -q <queue-name> --max-jobs <n>
    ```
  </Tab>

  <Tab title="Fichier de configuration">
    ```yaml title="launch-config.yaml"
    max_jobs: <n concurrent jobs>
    queues:
    	- <queue-name>
    ```
  </Tab>
</Tabs>

<div id="docker-image-builders">
  ## Builders d’images Docker
</div>

L’agent Launch sur votre machine peut être configuré pour builder des images Docker. Par défaut, ces images sont stockées dans le dépôt local d’images de votre machine. Pour permettre à votre agent Launch de builder des images Docker, définissez la clé `builder` de la configuration de l’agent Launch sur `docker` :

```yaml title="launch-config.yaml"
builder:
	type: docker
```

Si vous ne voulez pas que l’agent build des images Docker et préférez utiliser des images prébuildées à partir d’un registre, définissez la clé `builder` de la configuration de l’agent Launch sur `noop`

```yaml title="launch-config.yaml"
builder:
  type: noop
```

<div id="container-registries">
  ## Registres de conteneurs
</div>

Launch utilise des registres de conteneurs externes tels que Docker Hub, Google Container Registry, Azure Container Registry et Amazon ECR.
Si vous souhaitez exécuter un job dans un environnement différent de celui dans lequel vous l&#39;avez buildé, configurez votre agent afin qu&#39;il puisse récupérer des images depuis un registre de conteneurs.

Pour en savoir plus sur la façon de connecter l&#39;agent Launch à un registre cloud, voir la page [Configuration avancée de l&#39;agent](./setup-agent-advanced#agent-configuration).
