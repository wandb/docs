---
title: Termes et concepts de Launch
description: "Découvrez les principaux concepts de W&B Launch, notamment les jobs, les files d'attente, les ressources cibles, les agents et les environnements d'agent."
---

Avec W&amp;B Launch, vous placez des [jobs](#launch-job) dans des [files d&#39;attente](#launch-queue) afin de créer des runs. Les jobs sont des scripts Python instrumentés avec W&amp;B. Les files d&#39;attente contiennent une liste de jobs à exécuter sur une [ressource cible](#target-resources). Les [agents](#launch-agent) récupèrent les jobs dans les files d&#39;attente et les exécutent sur les ressources cibles. W&amp;B suit les jobs Launch de manière similaire aux [runs](/fr/models/runs/).

<div id="launch-job">
  ### Job Launch
</div>

Un job Launch est un type spécifique d&#39;[artefact W&amp;B](/fr/models/artifacts/) qui représente une tâche à accomplir. Par exemple, les jobs Launch courants incluent l&#39;entraînement d&#39;un modèle ou le déclenchement d&#39;une évaluation de modèle. Les définitions de job incluent :

* Le code Python et d&#39;autres fichiers de ressources, y compris au moins un point d&#39;entrée exécutable.
* Des informations sur l&#39;entrée (paramètre de configuration) et la sortie (métriques enregistrées).
* Des informations sur l&#39;environnement (par exemple, `requirements.txt`, `Dockerfile` de base).

Il existe trois grands types de définitions de job :

| Types de job                                  | Définition                                                                                                                              | Comment exécuter ce type de job                                                                                                                |
| --------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| Jobs basés sur des artefacts (ou sur le code) | Le code et les autres ressources sont enregistrés en tant qu&#39;artefact W&amp;B.                                                      | Pour exécuter des jobs basés sur des artefacts, l&#39;agent Launch doit être configuré avec un générateur.                                     |
| Jobs basés sur Git                            | Le code et les autres ressources sont clonés à partir d&#39;un commit, d&#39;une branche ou d&#39;un tag spécifiques dans un dépôt Git. | Pour exécuter des jobs basés sur Git, l&#39;agent Launch doit être configuré avec un générateur et des identifiants de dépôt Git.              |
| Jobs basés sur des images                     | Le code et les autres ressources sont intégrés dans une image Docker.                                                                   | Pour exécuter des jobs basés sur des images, l&#39;agent Launch peut nécessiter une configuration avec des identifiants de dépôt d&#39;images. |

<Note>
  Bien que les jobs Launch puissent effectuer des activités non liées à l&#39;entraînement du modèle — par exemple, déployer un modèle sur un serveur d&#39;inférence Triton — tous les jobs doivent appeler `wandb.init` pour se terminer correctement. Cela crée un run à des fins de suivi dans un Workspace W&amp;B.
</Note>

Retrouvez les jobs que vous avez créés dans la W&amp;B App sous l&#39;onglet `Jobs` de l&#39;espace de travail du projet. À partir de là, les jobs peuvent être configurés et envoyés vers une [file d&#39;attente Launch](#launch-queue) afin d&#39;être exécutés sur différentes [ressources cibles](#target-resources).

<div id="launch-queue">
  ### File d’attente Launch
</div>

Les *files d’attente* Launch sont des listes ordonnées de jobs à exécuter sur une ressource cible spécifique. Les files d’attente Launch fonctionnent selon le principe premier entré, premier sorti (FIFO). En pratique, il n’y a pas de limite au nombre de files d’attente que vous pouvez avoir, mais il est recommandé d’utiliser une file d’attente par ressource cible. Les jobs peuvent être placés en file d’attente depuis l’interface de l’application W&amp;B, la CLI W&amp;B ou le SDK Python. Vous pouvez ensuite configurer un ou plusieurs agents Launch pour récupérer des éléments dans la file d’attente et les exécuter sur la ressource cible de celle-ci.

<div id="target-resources">
  ### Ressources cibles
</div>

L&#39;environnement de calcul sur lequel une file d&#39;attente Launch est configurée pour exécuter des jobs s&#39;appelle la *ressource cible*.

W&amp;B Launch prend en charge les ressources cibles suivantes :

* [Docker](/fr/platform/launch/setup-launch-docker/)
* [Kubernetes](/fr/platform/launch/setup-launch-kubernetes/)
* [AWS SageMaker](/fr/platform/launch/setup-launch-sagemaker/)
* [Google Cloud Vertex AI](/fr/platform/launch/setup-vertex/)

Chaque ressource cible accepte un ensemble différent de paramètres de configuration appelés *configurations de ressource*. Les configurations de ressource héritent des valeurs par défaut définies pour chaque file d&#39;attente Launch, mais peuvent être remplacées indépendamment pour chaque job. Consultez la documentation de chaque ressource cible pour plus de détails.

<div id="launch-agent">
  ### Agent Launch
</div>

Les agents Launch sont des programmes légers et persistants qui vérifient régulièrement les files d&#39;attente Launch pour y trouver des jobs à exécuter. Lorsqu&#39;un agent Launch reçoit un job, il build d&#39;abord l&#39;image ou la récupère à partir de la définition du job, puis l&#39;exécute sur la ressource cible.

Un agent peut interroger plusieurs files d&#39;attente. Cependant, il doit être correctement configuré pour prendre en charge l&#39;ensemble des ressources cibles sous-jacentes de chaque file d&#39;attente interrogée.

<div id="launch-agent-environment">
  ### Environnement d’agent Launch
</div>

L’environnement d’agent est l’environnement dans lequel un agent Launch s’exécute et interroge régulièrement les jobs.

<Note>
  L’environnement d’exécution de l’agent est indépendant de la ressource cible d’une file d’attente. En d’autres termes, les agents peuvent être déployés n’importe où, à condition d’être suffisamment configurés pour accéder aux ressources cibles requises.
</Note>
