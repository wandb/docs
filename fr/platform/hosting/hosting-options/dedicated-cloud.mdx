Le Cloud dédié de W&amp;B est une plateforme à locataire unique, entièrement gérée, déployée dans les comptes cloud AWS, Google Cloud ou Azure de W&amp;B. Chaque instance de Cloud dédié dispose de son propre réseau isolé, ainsi que de ressources de calcul et de stockage isolées de celles des autres instances de Cloud dédié de W&amp;B. Vos métadonnées et données spécifiques à W&amp;B sont stockées dans un stockage cloud isolé et traitées à l’aide de services de calcul cloud isolés.

Le Cloud dédié de W&amp;B est disponible dans [de multiples régions mondiales pour chaque fournisseur de cloud](./dedicated-cloud/regions)

<div id="rate-limits">
  ## Limites de débit
</div>

W&amp;B applique des limites de débit par défaut sur Cloud dédié afin de maintenir la stabilité de l’instance. Voir [Limites de débit](/fr/platform/hosting/hosting-options/dedicated-cloud/rate-limits) pour connaître les valeurs par défaut, la façon dont les limites sont appliquées et comment demander des limites plus élevées lorsque vous augmentez l’échelle de l’entraînement.

<div id="compliance">
  ## Conformité
</div>

* **SOC 2** : la plateforme d’hébergement du Cloud dédié de W&amp;B répond aux exigences du [Service and Organization Controls (SOC) 2 Type 2](https://www.aicpa-cima.com/topic/audit-assurance/audit-and-assurance-greater-than-soc-2), publié par l’[Auditing Standards Board of the American Institute of Certified Public Accountants (AICPA)](https://www.aicpa-cima.com/home). Un rapport SOC 2 évalue les contrôles d’une organisation de services en matière de sécurité, de disponibilité, d’intégrité du traitement, de confidentialité et de respect de la vie privée. Le Cloud dédié de W&amp;B fait l’objet d’audits internes et externes périodiques afin de vérifier le maintien de sa conformité. Se référer au [W&amp;B Security Portal](https://security.wandb.ai/) pour demander le rapport SOC 2 et d’autres documents relatifs à la sécurité et à la conformité.
* **HIPAA** : lorsqu’il est correctement configuré, le Cloud dédié de W&amp;B répond aux exigences du [Health Insurance Portability and Accountability Act of 1996 (HIPAA)](https://www.hhs.gov/hipaa/for-professionals/index.html). La conformité à l’HIPAA est une responsabilité partagée entre W&amp;B, le client et tout service tiers impliqué dans le déploiement. Les organisations soumises à l’HIPAA doivent disposer d’un **Business Associate Agreement** enregistré auprès de W&amp;B. Se référer au [W&amp;B Security Portal](https://security.wandb.ai/) pour demander plus d’informations.

<div id="data-security">
  ## Sécurité des données
</div>

Vous pouvez utiliser Bring your own bucket (BYOB) avec le [connecteur de stockage sécurisé](/fr/platform/hosting/data-security/secure-storage-connector) [aux niveaux de l&#39;instance et de l&#39;équipe](/fr/platform/hosting/data-security/secure-storage-connector#configuration-options) pour stocker vos fichiers, tels que des modèles, des jeux de données, etc.

Comme dans le Cloud mutualisé de W&amp;B, vous pouvez configurer un bucket unique pour plusieurs équipes ou utiliser des buckets distincts pour différentes équipes. Si vous ne configurez pas le connecteur de stockage sécurisé pour une équipe, ces données sont stockées dans le bucket défini au niveau de l&#39;instance.

<Frame>
  <img src="/images/hosting/dedicated_cloud_arch.png" alt="Schéma d'architecture du Cloud dédié" />
</Frame>

En plus de BYOB avec le connecteur de stockage sécurisé, vous pouvez utiliser la [liste d&#39;autorisation d&#39;adresses IP](/fr/platform/hosting/data-security/ip-allowlisting) pour restreindre l&#39;accès à votre instance de Cloud dédié aux seuls emplacements réseau approuvés.

Vous pouvez vous connecter de manière privée à votre instance de Cloud dédié à l&#39;aide de la [solution de connectivité sécurisée du fournisseur de cloud](/fr/platform/hosting/data-security/private-connectivity).

Il vous incombe de vous assurer que votre déploiement est conforme aux politiques de votre organisation et aux [Security Technical Implementation Guidelines (STIG)](https://en.wikipedia.org/wiki/Security_Technical_Implementation_Guide), le cas échéant.

<div id="identity-and-access-management-iam">
  ## Gestion des identités et des accès (IAM)
</div>

Utilisez les fonctionnalités de gestion des identités et des accès pour assurer une authentification sécurisée et une autorisation efficace dans votre organisation W&amp;B. Les fonctionnalités suivantes sont disponibles pour l’IAM dans les instances de Cloud dédié :

* Authentifiez-vous avec le [SSO via OpenID Connect (OIDC)](/fr/platform/hosting/iam/sso) ou avec [LDAP](/fr/platform/hosting/iam/ldap).
* [Configurez les rôles utilisateur appropriés](/fr/platform/hosting/iam/access-management/manage-organization#assign-or-update-a-users-role) à l’échelle de l’organisation et au sein d’une équipe.
* Définissez le périmètre d’un projet W&amp;B afin de limiter les personnes autorisées à le consulter, le modifier et y soumettre des runs W&amp;B à l’aide des [projets restreints](/fr/platform/hosting/iam/access-management/restricted-projects).
* Utilisez des JSON Web Tokens avec la [fédération d’identités](/fr/platform/hosting/iam/identity_federation) pour accéder aux API W&amp;B.

<div id="maintenance">
  ## Maintenance
</div>

Comme avec le Cloud mutualisé de W&amp;B, vous n’avez pas à assumer la charge ni les coûts liés au provisionnement et à la maintenance de la plateforme W&amp;B avec le Cloud dédié.

Pour comprendre comment W&amp;B gère les mises à jour sur le Cloud dédié, se référer au [processus de publication des versions du serveur](/fr/platform/hosting/server-upgrade-process).

<div id="data-retention-policy">
  ## Politique de conservation des données
</div>

Par défaut, une instance de Cloud dédié conserve les éléments suivants pendant 7 jours après leur suppression :

* Runs et historique
* Fichiers de run non liés aux artefacts, tels que les médias, les fichiers de configuration et les fichiers journaux
* Artifacts et références d’artefacts

Tant que cette période n’est pas écoulée, ces éléments peuvent être restaurés. Contactez l’[assistance](mailto:support@wandb.ai) ou votre AISE.

Pour répondre à vos exigences de conservation des données, vous pouvez modifier la durée de conservation des données de votre instance de Cloud dédié. Selon votre cas d’usage, sélectionnez l’onglet **Variable d’environnement** ou **Helm** pour plus de détails.

<Tabs>
  <Tab title="Variable d’environnement">
    Pour modifier la politique de conservation des données, définissez la variable d’environnement `GORILLA_DATA_RETENTION_PERIOD` sur un nombre d’heures. Par exemple, pour conserver les données supprimées pendant 14 jours (336 heures) :

    ```bash
    export GORILLA_DATA_RETENTION_PERIOD="336h"
    ```
  </Tab>

  <Tab title="Helm">
    Pour modifier la politique de conservation des données, définissez la valeur Helm `env.dataRetentionPeriod` sur un nombre d’heures. Par exemple, pour conserver les données supprimées pendant 14 jours (336 heures) :

    ```helm
    env: dataRetentionPeriod: "336h"
    ```
  </Tab>
</Tabs>
