title: Configurer le SSO avec OIDC

---

La prise en charge par W&amp;B des fournisseurs d&#39;identité compatibles avec OpenID Connect (OIDC) permet de gérer les identités des utilisateurs et leur appartenance à des groupes via des fournisseurs d&#39;identité externes tels qu&#39;Okta, Keycloak, Auth0, Google et Entra.

<div id="openid-connect-oidc">
  ## OpenID Connect (OIDC)
</div>

W&amp;B prend en charge les flux d&#39;authentification OIDC suivants pour l&#39;intégration avec des fournisseurs d&#39;identité (IdP) externes.

1. Flux implicite avec envoi de formulaire
2. Flux de code d&#39;autorisation avec Proof Key for Code Exchange (PKCE)

Ces flux authentifient les utilisateurs et fournissent à W&amp;B les informations d&#39;identité nécessaires (sous la forme de jetons ID) pour gérer le contrôle d&#39;accès.

Le jeton ID est un JWT qui contient les informations d&#39;identité de l&#39;utilisateur, telles que son nom, son nom d&#39;utilisateur, son adresse e-mail et les groupes auxquels il appartient. W&amp;B utilise ce jeton pour authentifier l&#39;utilisateur et l&#39;associer aux rôles ou groupes appropriés dans le système.

Dans le contexte de W&amp;B, les jetons d&#39;accès autorisent les requêtes vers les API au nom de l&#39;utilisateur, mais comme W&amp;B se concentre principalement sur l&#39;authentification des utilisateurs et l&#39;identité, seul le jeton ID est nécessaire.

Vous pouvez utiliser des variables d&#39;environnement pour [configurer les options IAM](/fr/platform/hosting/iam/advanced_env_vars) pour votre instance [Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud) ou [Autogéré](/fr/platform/hosting/hosting-options/self-managed).

Pour vous aider à configurer des fournisseurs d&#39;identité pour des installations W&amp;B [Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud) ou [Autogéré](/fr/platform/hosting/hosting-options/self-managed), suivez les recommandations ci-dessous pour les différents IdP. Si vous utilisez la version SaaS de W&amp;B, contactez [support@wandb.com](mailto:support@wandb.com) pour obtenir de l&#39;assistance dans la configuration d&#39;un tenant Auth0 pour votre organisation.

<div id="configure-your-idp">
  ## Configurer votre IdP
</div>

Cette section explique comment configurer votre fournisseur d’identité (IdP) pour OIDC. Sélectionnez l&#39;onglet de votre IdP pour plus de détails.

<Tabs>
  <Tab title="Cognito">
    Suivez la procédure ci-dessous pour configurer AWS Cognito pour l&#39;autorisation :

    1. Commencez par vous connecter à votre compte AWS et accédez à l&#39;application [AWS Cognito](https://aws.amazon.com/cognito/).

       <Frame>
         <img src="/images/hosting/setup_aws_cognito.png" alt="Configuration d’AWS Cognito" />
       </Frame>

    2. Fournissez une URL de rappel autorisée pour configurer l&#39;application dans votre IdP :
       * Ajoutez `http(s)://YOUR-W&B-HOST/oidc/callback` comme URL de rappel. Remplacez `YOUR-W&B-HOST` par le chemin d’hôte de votre W&amp;B.

    3. Si votre IdP prend en charge la déconnexion universelle, définissez l’URL de déconnexion sur `http(s)://YOUR-W&B-HOST`. Remplacez `YOUR-W&B-HOST` par le chemin d’hôte de votre W&amp;B.

       Par exemple, si votre application s’exécutait à l’adresse `https://wandb.mycompany.com`, vous remplaceriez `YOUR-W&B-HOST` par `wandb.mycompany.com`.

       L’image ci-dessous montre comment fournir les URL de rappel et de déconnexion autorisées dans AWS Cognito.

       <Frame>
         <img src="/images/hosting/setup_aws_cognito_ui_settings.png" alt="Configuration de l’hôte" />
       </Frame>

       *wandb/local* utilise par défaut l’[`implicit` grant avec le type de réponse `form_post`](https://auth0.com/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post).

       Vous pouvez également configurer *wandb/local* pour effectuer un grant `authorization_code` qui utilise le flux [PKCE Code Exchange](https://www.oauth.com/oauth2-servers/pkce/).

    4. Sélectionnez un ou plusieurs types de grant OAuth pour configurer la manière dont AWS Cognito transmet les jetons à votre application.

    5. W&amp;B requiert des portées OpenID Connect (OIDC) spécifiques. Sélectionnez les éléments suivants dans l’application AWS Cognito :

       * &quot;openid&quot;
       * &quot;profile&quot;
       * &quot;email&quot;

       Par exemple, l’interface utilisateur de votre application AWS Cognito devrait ressembler à l’image suivante :

       <Frame>
         <img src="/images/hosting/setup_aws_required_fields.png" alt="Champs requis" />
       </Frame>

       Sélectionnez la **méthode d’authentification** dans la page des paramètres ou définissez la variable d’environnement OIDC&#95;AUTH&#95;METHOD pour indiquer à *wandb/local* quel grant utiliser.

       Vous devez définir la méthode d’authentification sur `pkce`.

    6. Vous avez besoin d’un ID client et de l’URL de votre émetteur OIDC. Le document de découverte OpenID doit être disponible à l’adresse `$OIDC_ISSUER/.well-known/openid-configuration`

       Par exemple, vous pouvez générer l’URL de votre émetteur en ajoutant votre ID de pool d’utilisateurs à l’URL Cognito IdP depuis l’onglet **App Integration** de la section **User Pools** :

       <Frame>
         <img src="/images/hosting/setup_aws_cognito_issuer_url.png" alt="URL de l’émetteur AWS Cognito" />
       </Frame>

       N’utilisez pas le &quot;domaine Cognito&quot; pour l’URL de l’IDP. Cognito fournit son document de découverte à l’adresse `https://cognito-idp.$REGION.amazonaws.com/$USER_POOL_ID`

    Ensuite, [Configurer le SSO dans W&amp;B](#set-up-sso-in-w%26b).
  </Tab>

  <Tab title="Entra">
    Azure AD (Entra ID) prend en charge deux modes de configuration OIDC pour W&amp;B. Choisissez la configuration qui correspond à vos exigences de sécurité :

<div id="set-up-sso-in-wb">
  ## Configurer le SSO dans W&amp;B
</div>

Pour configurer le SSO, vous devez disposer des privilèges d’administrateur ainsi que des informations suivantes :

* ID client OIDC
* méthode d’authentification OIDC (`implicit` ou `pkce`)
* URL de l’émetteur OIDC
* secret client OIDC (facultatif ; cela dépend de la configuration de votre IdP)

Si votre IdP exige un secret client OIDC, spécifiez-le en définissant la [variable d’environnement](/fr/platform/hosting/env-vars) `GORILLA_OIDC_SECRET`.

* Dans la W&amp;B App, accédez à **System Console** &gt; **Settings** &gt; **Advanced** &gt; **User Spec** et ajoutez `GORILLA_OIDC_SECRET` à la section `extraENV`, comme illustré ci-dessous.
* Dans Helm, configurez `values.global.extraEnv` comme illustré ci-dessous.
  ```yaml
  values:
  global:
      extraEnv:
      GORILLA_OIDC_SECRET="<your_secret>"
  ```

<Note>
  Si vous ne parvenez pas à vous connecter à votre instance après avoir configuré le SSO, vous pouvez redémarrer l’instance avec la variable d’environnement `LOCAL_RESTORE=true` définie. Cela génère un mot de passe temporaire dans les journaux des conteneurs et désactive le SSO. Une fois les problèmes liés au SSO résolus, vous devez supprimer cette variable d’environnement pour réactiver le SSO.
</Note>

<div id="security-assertion-markup-language-saml">
  ## Security Assertion Markup Language (SAML)
</div>

W&amp;B ne prend pas en charge SAML.
