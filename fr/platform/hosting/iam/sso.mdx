title: Configurer le SSO avec OIDC
description: "Configurez le SSO à l’aide d’OpenID Connect avec des fournisseurs d’identité comme Okta, Azure AD et AWS Cognito pour les instances W&B."
---

La prise en charge par W&amp;B des fournisseurs d’identité compatibles avec OpenID Connect (OIDC) vous permet de gérer les identités des utilisateurs et leur appartenance à des groupes via des fournisseurs d’identité externes tels qu’Okta, Keycloak, Auth0, Google et Entra.

<div id="openid-connect-oidc">
  ## OpenID Connect (OIDC)
</div>

W&amp;B prend en charge les flux d’authentification OIDC suivants pour l’intégration avec des fournisseurs d’identité (IdP) externes.

1. Flux implicite avec envoi de formulaire
2. Flux de code d’autorisation avec Proof Key for Code Exchange (PKCE)

Ces flux authentifient les utilisateurs et fournissent à W&amp;B les informations d’identité nécessaires (sous la forme de jetons d’identité) pour gérer le contrôle d’accès.

Le jeton d’identité est un JWT qui contient les informations d’identité de l’utilisateur, telles que son nom, son nom d’utilisateur, son adresse e-mail et les groupes auxquels il appartient. W&amp;B utilise ce jeton pour authentifier l’utilisateur et l’associer aux rôles ou groupes appropriés dans le système.

Dans le contexte de W&amp;B, les jetons d’accès autorisent les requêtes vers les API au nom de l’utilisateur, mais comme W&amp;B se concentre principalement sur l’authentification des utilisateurs et l’identité, seul le jeton d’identité est nécessaire.

Vous pouvez utiliser des variables d’environnement pour [configurer les options IAM](/fr/platform/hosting/iam/advanced_env_vars) pour votre instance [Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud) ou [Autogéré](/fr/platform/hosting/hosting-options/self-managed).

Pour vous aider à configurer des fournisseurs d’identité pour des installations W&amp;B [Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud) ou [Autogéré](/fr/platform/hosting/hosting-options/self-managed), suivez les recommandations ci-dessous pour les différents IdP. Si vous utilisez la version SaaS de W&amp;B, contactez [support@wandb.com](mailto:support@wandb.com) pour obtenir de l’assistance dans la configuration d’un tenant Auth0 pour votre organisation.

<div id="configure-your-idp">
  ## Configurer votre IdP
</div>

Cette section explique comment configurer votre fournisseur d’identité (IdP) pour OIDC. Sélectionnez l’onglet de votre IdP pour plus de détails.

<Tabs>
  <Tab title="Cognito">
    Suivez la procédure ci-dessous pour configurer AWS Cognito pour l’autorisation :

    1. Connectez-vous d’abord à votre compte AWS, puis accédez à l’application [AWS Cognito](https://aws.amazon.com/cognito/).

       <Frame>
         <img src="/images/hosting/setup_aws_cognito.png" alt="Configuration d’AWS Cognito" />
       </Frame>

    2. Indiquez une URL de rappel autorisée pour configurer l’application dans votre IdP :
       * Ajoutez `http(s)://YOUR-W&B-HOST/oidc/callback` comme URL de rappel. Remplacez `YOUR-W&B-HOST` par l’hôte de votre instance W&amp;B.

    3. Si votre IdP prend en charge la déconnexion universelle, définissez l’URL de déconnexion sur `http(s)://YOUR-W&B-HOST`. Remplacez `YOUR-W&B-HOST` par l’hôte de votre instance W&amp;B.

       Par exemple, si votre application s’exécute à l’adresse `https://wandb.mycompany.com`, remplacez `YOUR-W&B-HOST` par `wandb.mycompany.com`.

       L’image ci-dessous montre comment renseigner les URL de rappel autorisées et de déconnexion dans AWS Cognito.

       <Frame>
         <img src="/images/hosting/setup_aws_cognito_ui_settings.png" alt="Configuration de l’hôte" />
       </Frame>

       Par défaut, *wandb/local* utilise le flux d’autorisation [`implicit` avec le type de réponse `form_post`](https://auth0.com/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post).

       Vous pouvez également configurer *wandb/local* pour utiliser une autorisation `authorization_code` avec le flux [PKCE Code Exchange](https://www.oauth.com/oauth2-servers/pkce/).

    4. Sélectionnez un ou plusieurs types de subvention OAuth afin de configurer la manière dont AWS Cognito transmet les jetons à votre application.

    5. W&amp;B requiert des portées OpenID Connect (OIDC) spécifiques. Sélectionnez les éléments suivants dans l’application AWS Cognito :

       * &quot;openid&quot;
       * &quot;profile&quot;
       * &quot;email&quot;

       Par exemple, l’interface de votre application AWS Cognito doit ressembler à l’image suivante :

       <Frame>
         <img src="/images/hosting/setup_aws_required_fields.png" alt="Champs requis" />
       </Frame>

       Sélectionnez la **méthode d’authentification** dans la page des paramètres ou définissez la variable d’environnement OIDC&#95;AUTH&#95;METHOD pour indiquer à *wandb/local* quelle autorisation utiliser.

       Vous devez définir la méthode d’authentification sur `pkce`.

    6. Vous avez besoin d’un ID client et de l’URL de votre émetteur OIDC. Le document de découverte OpenID doit être disponible à l’adresse `$OIDC_ISSUER/.well-known/openid-configuration`

       Par exemple, vous pouvez générer l’URL de votre émetteur en ajoutant votre ID de pool d’utilisateurs à l’URL IdP Cognito depuis l’onglet **App Integration** de la section **User Pools** :

       <Frame>
         <img src="/images/hosting/setup_aws_cognito_issuer_url.png" alt="URL de l’émetteur AWS Cognito" />
       </Frame>

       N’utilisez pas le &quot;domaine Cognito&quot; comme URL d’IdP. Cognito fournit son document de découverte à l’adresse `https://cognito-idp.$REGION.amazonaws.com/$USER_POOL_ID`

    Ensuite, [Configurer le SSO dans W&amp;B](#set-up-sso-in-w%26b).
  </Tab>

  <Tab title="Okta">
    Suivez la procédure ci-dessous pour configurer Okta pour l’autorisation :

    1. Connectez-vous au [portail Okta](https://login.okta.com/).

    2. Dans le volet de gauche, sélectionnez **Applications**, puis de nouveau **Applications**.
       <Frame>
         <img src="/images/hosting/okta_select_applications.png" alt="Menu Applications d’Okta" />

    3. Cliquez sur &quot;Create App integration&quot;.
       <Frame>

    4. Dans l’écran &quot;Create a new app integration&quot;, sélectionnez **OIDC - OpenID Connect** et **Single-Page Application**. Cliquez ensuite sur &quot;Next&quot;.
       <Frame>
         <img src="/images/hosting/okta_create_a_new_app_integration.png" alt="Sélection OIDC Single-Page Application" />
       </Frame>

    5. Dans l’écran &quot;New Single-Page App Integration&quot;, renseignez les valeurs suivantes, puis cliquez sur **Save** :
       * App integration name : par exemple &quot;W&amp;B&quot;
       * Grant type : sélectionnez **Authorization Code** et **Implicit (hybrid)**
       * Sign-in redirect URIs : https://YOUR&#95;W&#95;AND&#95;B&#95;URL/oidc/callback
       * Sign-out redirect URIs : https://YOUR&#95;W&#95;AND&#95;B&#95;URL/logout
       * Assignments : sélectionnez **Skip group assignment for now**
       <Frame>
         <img src="/images/hosting/okta_new_single_page_app_integration.png" alt="Configuration de Single-Page App" />

       L’interface Okta affiche le nom de l’entreprise sous **Organization Contact**.
       <Frame>
         <img src="/images/hosting/okta_identify_oidc_issuer_url.png" alt="Paramètres de l’organisation Okta" />
       </Frame>

    L’URL de l’émetteur OIDC a le format suivant : `https://COMPANY.okta.com`. Remplacez COMPANY par la valeur correspondante. Notez-la.

    Ensuite, [Configurez le SSO dans W&amp;B](#set-up-sso-in-w%26b).

  <Tab title="Entra">
    Azure AD (Entra ID) prend en charge deux modes de configuration OIDC pour W&amp;B. Choisissez la configuration qui correspond à vos exigences de sécurité :

    * [Client public](#public-client) : utilise PKCE sans secret client. Plus simple à configurer, il convient à la plupart des déploiements.
    * [Client confidentiel](#confidential-client) : utilise PKCE avec un secret client pour renforcer la sécurité. Requis si vous devez définir la variable d’environnement `GORILLA_OIDC_SECRET`.

    <Warning>
      Ne mélangez pas les configurations. Si vous sélectionnez &quot;Single-page application&quot; dans Azure AD, ne renseignez pas de secret client. Si vous avez besoin d’un secret client, vous devez sélectionner &quot;Web&quot; comme type de plateforme.
    </Warning>

<div id="set-up-sso-in-wb">
  ## Configurer le SSO dans W&amp;B
</div>

Pour configurer le SSO, vous devez disposer des privilèges d’administrateur et des informations suivantes :

* ID client OIDC
* Méthode d’authentification OIDC (`implicit` ou `pkce`)
* URL de l’émetteur OIDC
* Secret client OIDC (facultatif ; dépend de la façon dont vous avez configuré votre IdP)

Si votre IdP requiert un secret client OIDC, spécifiez-le via la [variable d’environnement](/fr/platform/hosting/env-vars) `GORILLA_OIDC_SECRET`.

* Dans l’application W&amp;B, accédez à **System Console** &gt; **Settings** &gt; **Advanced** &gt; **User Spec** et ajoutez `GORILLA_OIDC_SECRET` à la section `extraENV`, comme indiqué ci-dessous.
* Dans Helm, configurez `values.global.extraEnv` comme indiqué ci-dessous.
  ```yaml
  values:
  global:
      extraEnv:
      GORILLA_OIDC_SECRET="<your_secret>"
  ```

<Note>
  Si vous ne parvenez pas à vous connecter à votre instance après avoir configuré le SSO, vous pouvez redémarrer l’instance avec la variable d’environnement `LOCAL_RESTORE=true` définie. Cela génère un mot de passe temporaire dans les journaux des conteneurs et désactive le SSO. Une fois les problèmes liés au SSO résolus, vous devez supprimer cette variable d’environnement pour réactiver le SSO.
</Note>

<div id="security-assertion-markup-language-saml">
  ## Security Assertion Markup Language (SAML)
</div>

W&amp;B ne prend pas en charge SAML.
