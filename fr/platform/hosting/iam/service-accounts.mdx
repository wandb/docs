Un compte de service désigne un utilisateur non humain ou une machine capable d&#39;exécuter automatiquement des tâches courantes dans des projets au sein d&#39;une équipe ou entre plusieurs équipes. Les comptes de service sont idéaux pour les pipelines CI/CD, les tâches d&#39;entraînement automatisées et les autres flux de travail entre machines.

<div id="overview">
  ## Aperçu
</div>

Les comptes de service offrent un moyen sécurisé d’automatiser les flux de travail W&amp;B sans utiliser des identifiants personnels d’utilisateur ni des identifiants codés en dur. Ils peuvent être créés selon deux portées :

* **Limité à l’organisation** : créés par les administrateurs de l’organisation, avec un accès sur toutes les équipes.
* **À portée d&#39;équipe** : créés par les Administrateurs d’équipe, avec un accès limité à une équipe spécifique

La clé API d’un compte de service permet à l’appelant de lire ou d’écrire dans les projets compris dans la portée du compte de service. Cela permet une gestion centralisée des flux de travail automatisés pour le suivi des expériences dans W&amp;B Models ou la journalisation des traces dans W&amp;B Weave.

Les comptes de service sont particulièrement utiles pour :

* **Pipelines CI/CD** : journaliser automatiquement les runs d’entraînement du modèle depuis GitHub Actions, GitLab CI ou Jenkins
* **Jobs planifiés** : réentraînement nocturne du modèle, evaluation runs périodiques ou flux de travail de validation des données
* **Surveillance en Production** : journaliser les métriques d’inférence et la performance du modèle à partir des systèmes de production
* **Notebooks Jupyter** : notebooks partagés dans des environnements JupyterHub ou Google Colab
* **Jobs Kubernetes** : flux de travail automatisés exécutés dans des clusters K8s
* **Airflow/Prefect/Dagster** : outils d’orchestration de pipelines ML

<Note>
  Les comptes de service sont disponibles sur [Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud), les [instances autogérées](/fr/platform/hosting/hosting-options/self-managed) avec une licence Enterprise, et les comptes Enterprise dans le [Cloud mutualisé](/fr/platform/hosting/hosting-options/multi_tenant_cloud).
</Note>

<div id="organization-scoped-service-accounts">
  ## Comptes de service limités à l’organisation
</div>

Les comptes de service limités à l’organisation disposent d’autorisations en lecture et en écriture dans tous les projets de l’organisation, quelle que soit l’équipe, à l’exception des [projets restreints](/fr/platform/hosting/iam/access-management/restricted-projects/#visibility-scopes). Avant qu’un compte de service limité à l’organisation puisse accéder à un projet restreint, un administrateur de ce projet doit ajouter explicitement le compte de service au projet.

<Note>

Cela permet d’éviter l’échec d’une charge de travail si la variable `WANDB_ENTITY` n’est pas définie dans l’environnement de votre entraînement du modèle ou de votre application d’IA générative. Pour utiliser un compte de service limité à l’organisation pour un projet d’une autre équipe, vous devez définir la variable d’environnement `WANDB_ENTITY` sur l’équipe concernée.
</Note>

Si vous ne configurez pas d’équipe dans votre environnement d’entraînement du modèle ou d’application d’IA générative utilisant un compte de service à portée d&#39;équipe, les runs du modèle ou les traces Weave sont journalisés dans le projet nommé au sein de l’équipe parente du compte de service. Dans ce cas, l’attribution de l’utilisateur à l’aide des variables `WANDB_USERNAME` ou `WANDB_USER_EMAIL` *ne fonctionne pas*, sauf si l’utilisateur référencé fait partie de l’équipe parente du compte de service.

<Warning>
  Un compte de service à portée d&#39;équipe ne peut pas journaliser de runs dans un [projet d’équipe ou à portée restreinte](/fr/platform/hosting/iam/access-management/restricted-projects/#visibility-scopes) d’une équipe autre que son équipe parente, mais il peut journaliser des runs dans un projet à visibilité ouverte au sein d’une autre équipe.
</Warning>

<div id="external-service-accounts">
  ### Comptes de service externes
</div>

En plus des comptes de service intégrés, W&amp;B prend également en charge, avec le SDK et la CLI W&amp;B, les comptes de service externes à portée d’équipe via la [fédération d’identités](./identity_federation#external-service-accounts) avec des fournisseurs d’identité (IdP) capables d’émettre des JSON Web Tokens (JWT).

<div id="best-practices">
  ## Bonnes pratiques
</div>

Suivez ces recommandations pour garantir une utilisation sécurisée et efficace des comptes de service dans votre organisation :

* **Utilisez un gestionnaire de secrets** : stockez les clés API des comptes de service dans un système de gestion des secrets sécurisé (par exemple, AWS Secrets Manager, HashiCorp Vault, Azure Key Vault) plutôt que dans des fichiers de configuration en texte brut.

* **Principe du moindre privilège** : créez des comptes de service à portée d&#39;équipe lorsque c&#39;est possible, plutôt que des comptes limités à l’organisation, afin de limiter l&#39;accès aux seuls projets nécessaires.

* **Comptes de service distincts pour chaque cas d&#39;usage** : créez des comptes de service séparés pour différents flux de travail d&#39;automatisation (par exemple, un pour le CI/CD, un autre pour le réentraînement planifié) afin d&#39;améliorer la traçabilité et de permettre un contrôle d&#39;accès granulaire.

* **Audits réguliers** : passez périodiquement en revue les comptes de service actifs et supprimez ceux qui ne sont plus utilisés. Consultez les journaux d’audit pour surveiller l&#39;activité des comptes de service.

* **Gestion sécurisée des clés API** :
  * N&#39;ajoutez jamais de clés API au contrôle de version
  * Utilisez des variables d&#39;environnement pour transmettre les clés aux applications
  * Renouvelez les clés si elles sont accidentellement exposées

* **Conventions de nommage** : utilisez des noms descriptifs qui indiquent l&#39;objectif du compte de service :
  * Bon : `ci-model-training`, `nightly-eval-pipeline`, `prod-inference-monitor`
  * À éviter : `service-account-1`, `test-sa`, `temp`

* **Attribution de l&#39;utilisateur** : lorsque plusieurs membres de l’équipe utilisent le même flux de travail d&#39;automatisation, définissez `WANDB_USERNAME` ou `WANDB_USER_EMAIL` pour savoir qui a déclenché chaque run :
  ```bash
  export WANDB_API_KEY="<service_account_key>"
  export WANDB_USERNAME="john.doe@company.com"
  ```

* **Configuration de l&#39;environnement** : pour les comptes de service à portée d&#39;équipe, définissez toujours `WANDB_ENTITY` pour garantir que les runs sont journalisés dans la bonne équipe :
  ```bash
  export WANDB_ENTITY="ml-team"
  export WANDB_PROJECT="production-models"
  ```

* **Gestion des erreurs** : mettez en place une gestion appropriée des erreurs et des alertes en cas d&#39;échec de l&#39;authentification afin d&#39;identifier rapidement les problèmes liés aux identifiants du compte de service.

* **Documentation** : tenez à jour une documentation indiquant :
  * Quels comptes de service existent et à quoi ils servent
  * Quels systèmes/flux de travail utilisent chaque compte de service
  * Les coordonnées de l&#39;équipe responsable de chaque compte

<div id="troubleshooting">
  ## Dépannage
</div>

Problèmes courants et solutions :

* **Erreurs « Unauthorized »** : vérifiez que la clé API est correctement définie et que le compte de service a accès au projet cible
* **Les Runs n&#39;apparaissent pas** : vérifiez que `WANDB_ENTITY` est défini avec le bon nom d&#39;équipe
* **L&#39;attribution de l&#39;utilisateur ne fonctionne pas** : assurez-vous que l&#39;utilisateur spécifié dans `WANDB_USERNAME` est membre de l&#39;équipe
* **Accès refusé aux projets restreints** : ajoutez explicitement le compte de service à la liste d&#39;accès du projet restreint
