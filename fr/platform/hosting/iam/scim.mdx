<div id="overview">
  ## Aperçu
</div>

L’API System for Cross-domain Identity Management (SCIM) permet aux administrateurs de l’instance ou de l’organisation de gérer les utilisateurs, les groupes et les rôles personnalisés au sein de leur organisation W&amp;B. Les groupes SCIM correspondent aux Teams de W&amp;B.

L’API SCIM de W&amp;B est compatible avec les principaux fournisseurs d’identité, notamment Okta, ce qui permet le provisionnement et le déprovisionnement automatisés des utilisateurs. Pour configurer le SSO avec Okta et d’autres fournisseurs d’identité, voir la [documentation SSO](/fr/platform/hosting/iam/sso/).

Pour consulter des exemples pratiques en Python montrant comment interagir avec l’API SCIM, rendez-vous sur notre dépôt [`wandb-scim`](https://github.com/wandb/examples/tree/master/wandb-scim).

<div id="supported-features">
  ### Fonctionnalités prises en charge
</div>

* **Filtrage** : l’API prend en charge le filtrage des endpoints `/Users` et `/Groups`
* **Opérations PATCH** : prise en charge de PATCH pour les mises à jour partielles des ressources
* **Prise en charge des ETags** : mises à jour conditionnelles à l’aide d’ETags pour détecter les conflits
* **Authentification par compte de service** : les comptes de service de l’organisation peuvent accéder à l’API

<Note>
  Si vous êtes administrateur de plusieurs organisations Enterprise [Cloud mutualisé](/fr/platform/hosting/hosting-options/multi_tenant_cloud), vous devez configurer l’organisation vers laquelle les requêtes d’API SCIM sont envoyées afin de vous assurer que les requêtes d’API SCIM envoyées à l’aide de votre clé API s’appliquent à la bonne organisation. Cliquez sur votre image de profil, puis sur **Paramètres utilisateur**, puis vérifiez le paramètre **Default API organization**.

  L’option d’hébergement choisie détermine la valeur de l’espace réservé `<host-url>` utilisé dans les exemples de cette page.

  De plus, les exemples utilisent des identifiants utilisateur tels que `abc` et `def`. Dans les requêtes et réponses réelles, les identifiants utilisateur sont des valeurs hachées.
</Note>

<div id="authentication">
  ## Authentification
</div>

Choisissez de vous authentifier à l’aide d’une identité utilisateur ou d’un compte de service, après avoir examiné les principales différences.

<div id="key-differences">
  ### Différences clés
</div>

* Qui doit l’utiliser : les Users conviennent mieux aux actions d’administration interactives et ponctuelles ; les comptes de service conviennent mieux à l’automatisation et aux intégrations (CI/CD, outils de provisionnement).
* Informations d’identification : les Users envoient un nom d’utilisateur et une clé API ; les comptes de service envoient uniquement une clé API (sans nom d’utilisateur).
* Valeur de l’en-tête Authorization : les Users encodent `username:API-KEY` ; les comptes de service encodent `:API-KEY` (avec deux-points initial).
* Portée et autorisations : les deux nécessitent des privilèges d’administration ; les comptes de service sont liés à l’organisation et sans interface utilisateur, ce qui offre des pistes d’audit plus claires pour l’automatisation.
* Où obtenir les informations d’identification : les Users copient leur clé API dans les Paramètres utilisateur ; les clés de compte de service se trouvent dans l’onglet Service account de l’organisation.
* Cloud mutualisé : si vous avez accès à plusieurs organisations du Cloud mutualisé, vous devez définir la Default API organization pour vous assurer que les appels à l’API SCIM sont dirigés vers l’organisation voulue.

<div id="users">
  ### Users
</div>

Utilisez vos identifiants personnels d’administrateur lorsque vous effectuez des tâches d’administration interactives. Construisez l’en-tête HTTP `Authorization` au format `Basic <base64(username:API-KEY)>`.

Par exemple, authentifiez-vous avec `demo:p@55w0rd` :

```bash
Authorization: Basic ZGVtbzpwQDU1dzByZA==
```

<div id="service-accounts">
  ### Comptes de service
</div>

Utilisez un compte de service limité à l’organisation pour l’automatisation ou les intégrations. Formez l’en-tête HTTP `Authorization` comme suit : `Basic <base64(:API-KEY)>` (notez le caractère deux-points initial et le nom d’utilisateur vide). Trouvez les clés API des comptes de service dans le tableau de bord de l’organisation, sous l’onglet **Service account**. Référez-vous à [Comptes de service limités à l’organisation](/fr/platform/hosting/iam/service-accounts/#organization-scoped-service-accounts).

Par exemple, authentifiez-vous avec la clé API `sa-p@55w0rd` :

```bash
Authorization: Basic OnNhLXBANTV3MHJk
```

<div id="user-management">
  ## Gestion des utilisateurs
</div>

La ressource utilisateur SCIM correspond aux utilisateurs de W&amp;B. Utilisez ces points de terminaison pour gérer les utilisateurs de votre organisation.

<div id="get-user">
  ### Obtenir un utilisateur
</div>

Récupère les informations d’un utilisateur spécifique au sein de votre organisation.

<Note>Cette opération ne récupère pas les comptes de service.</Note>

<div id="endpoint">
  #### Point de terminaison
</div>

* **URL**: `<host-url>/scim/Users/{id}`
* **Méthode**: GET

<div id="parameters">
  #### Paramètres
</div>

| Paramètre | Type   | Requis | Description                    |
| --------- | ------ | ------ | ------------------------------ |
| `id`      | string | Oui    | ID unique de l&#39;utilisateur |

#### Exemple

<Tabs>
  <Tab title="Obtenir la requête de l’utilisateur">
    ```bash
    GET /scim/Users/abc
    ```
  </Tab>

  <Tab title="Obtenir la réponse de l’utilisateur">
    ```bash
    (statut 200)
    ```

    ```json
    {
        "active": true,
        "daysActive": 42,
        "displayName": "Dev User 1",
        "emails": {
            "Value": "dev-user1@example.com",
            "Display": "",
            "Type": "",
            "Primary": true
        },
        "id": "abc",
        "lastActiveAt": "2023-10-15T14:32:10Z",
        "meta": {
            "resourceType": "User",
            "created": "2023-10-01T00:00:00Z",
            "lastModified": "2023-10-01T00:00:00Z",
            "location": "Users/abc"
        },
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User"
        ],
        "userName": "dev-user1"
    }
    ```

    La réponse contient des détails sur l’activité de l’utilisateur dans l’organisation :

    * **`daysActive`** : Nombre total de jours pendant lesquels l’utilisateur a été actif dans l’organisation.
    * **`lastActiveAt`** : Horodatage ISO 8601 de l’activité la plus récente de l’utilisateur. Renvoie `null` si l’utilisateur n’a jamais été actif.

    La définition de « actif » diffère selon le type de déploiement :

    * **Cloud dédié / Autogéré** : Un utilisateur est actif s’il se connecte, ouvre n’importe quelle page dans l’application W&amp;B, journalise des runs, utilise le SDK ou interagit de quelque manière que ce soit avec le serveur W&amp;B.
    * **Cloud mutualisé** : Un utilisateur est actif s’il effectue, dans le périmètre de l’organisation, une action traçable après le 8 mai 2025. Voir [Audit logging actions](/fr/platform/hosting/monitoring-usage/audit-logging#actions) pour la liste complète.
  </Tab>
</Tabs>

<div id="list-users">
  ### Lister les utilisateurs
</div>

Récupère la liste de tous les utilisateurs de votre organisation.

<Note>Cette opération ne récupère pas les comptes de service.</Note>

<div id="filter-users">
  #### Filtrer les utilisateurs
</div>

Le point de terminaison `/Users` prend en charge le filtrage des utilisateurs par nom d’utilisateur ou par adresse e-mail :

* `userName eq "value"` - Filtrer par nom d’utilisateur
* `emails.value eq "value"` - Filtrer par adresse e-mail

<div id="example">
  ##### Exemple
</div>

```bash
GET /scim/Users?filter=userName eq "john.doe"
GET /scim/Users?filter=emails.value eq "john@example.com"
```

<div id="create-user">
  ### Créer un utilisateur
</div>

Crée un nouvel utilisateur au sein de votre organisation.

<div id="endpoint">
  #### Point de terminaison
</div>

* **URL**: `<host-url>/scim/Users`
* **Méthode**: POST

<div id="parameters">
  #### Paramètres
</div>

| Paramètre  | Type   | Requis | Description                                                         |
| ---------- | ------ | ------ | ------------------------------------------------------------------- |
| `emails`   | array  | Oui    | Tableau d’objets e-mail. Doit inclure une adresse e-mail principale |
| `userName` | string | Oui    | Nom d’utilisateur du nouvel utilisateur                             |

#### Exemple

<Tabs>
  <Tab title="Requête de création d’utilisateur (Dédié/Autogéré)">
    ```bash
    POST /scim/Users
    ```

    ```json
    {
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User"
        ],
        "emails": [
            {
                "primary": true,
                "value": "dev-user2@example.com"
            }
        ],
        "userName": "dev-user2"
    }
    ```
  </Tab>

  <Tab title="Requête de création d’utilisateur (multilocataire)">
    ```bash
    POST /scim/Users
    ```

    ```json
    {
        "schemas": [
            "urn:ietf:params:scim:schemas:core:2.0:User",
            "urn:ietf:params:scim:schemas:extension:teams:2.0:User"
        ],
        "emails": [
            {
                "primary": true,
                "value": "dev-user2@example.com"
            }
        ],
        "userName": "dev-user2",
        "urn:ietf:params:scim:schemas:extension:teams:2.0:User": {
            "teams": ["my-team"]
        }
    }
    ```
  </Tab>

<div id="delete-user">
  ### Supprimer un utilisateur
</div>

<Warning>
  **Conservez l’accès administrateur**

  Vous devez vous assurer qu’au moins un utilisateur administrateur est présent en permanence dans votre instance ou votre organisation. Sinon, aucun utilisateur ne pourra configurer ou administrer le compte W&amp;B de votre organisation. Si une organisation utilise SCIM ou un autre processus automatisé pour déprovisionner des utilisateurs de W&amp;B, une opération de déprovisionnement pourrait supprimer par inadvertance le dernier administrateur restant de l’instance ou de l’organisation.

  Pour obtenir de l’assistance afin de définir des procédures opérationnelles, ou pour rétablir l’accès administrateur, contactez [l’assistance](mailto:support@wandb.com).
</Warning>

Supprime complètement un utilisateur de votre organisation.

<Note>Cette opération fonctionne uniquement pour les utilisateurs, pas pour les comptes de service. Supprimez un compte de service dans les paramètres de l’équipe W&amp;B.</Note>

<div id="endpoint">
  #### Point de terminaison
</div>

* **URL**: `<host-url>/scim/Users/{id}`
* **Méthode**: DELETE

<div id="parameters">
  #### Paramètres
</div>

| Paramètre | Type   | Requis | Description                            |
| --------- | ------ | ------ | -------------------------------------- |
| `id`      | string | Oui    | ID unique de l’utilisateur à supprimer |

#### Exemple

<Tabs>
  <Tab title="Requête de suppression d’utilisateur">
    ```bash
    DELETE /scim/Users/abc
    ```
  </Tab>

  <Tab title="Réponse de suppression d’utilisateur">
    ```bash
    (Statut 204)
    ```
  </Tab>
</Tabs>

<Note>
  Pour désactiver temporairement l’utilisateur, référez-vous à l’API [Désactiver l’utilisateur](#deactivate-user), qui utilise le point de terminaison `PATCH`.
</Note>

<div id="deactivate-user">
  ### Désactiver un utilisateur
</div>

Désactive un utilisateur dans votre organisation. Le résultat varie selon le type de déploiement :

* **Cloud dédié** / **Autogéré** : définit le champ `active` de l’utilisateur sur `false`. Pour rétablir l’accès d’un utilisateur désactivé à votre organisation, voir [Réactiver un utilisateur](#reactivate-user).
* **Cloud multilocataire** : supprime l’utilisateur de l’organisation. Pour rétablir l’accès de l’utilisateur, ajoutez-le de nouveau à votre organisation. Voir [Créer un utilisateur](#create-user-request-multi-tenant). Dans le Cloud multilocataire, le compte d’un utilisateur n’est pas géré par l’organisation.

<Note>Cette opération fonctionne uniquement pour les utilisateurs, pas pour les comptes de service. La désactivation d’un compte de service n’est pas prise en charge. Gérez les comptes de service de l’équipe dans les paramètres de l’équipe W&amp;B.</Note>

<div id="endpoint">
  #### Point de terminaison
</div>

* **URL**: `<host-url>/scim/Users/{id}`
* **Méthode**: PATCH

<div id="parameters">
  #### Paramètres
</div>

| --------- | ------ | ------ | ----------------------------------------- |
| `id`      | string | Oui    | L’ID unique de l’utilisateur à désactiver |
| `op`      | string | Oui    | `replace`                                 |
| `value`   | objet  | Oui    | Objet contenant `{"actif": false}`       |

<div id="create-team">
  ### Créer une équipe
</div>

* **Point de terminaison** : **`<host-url>/scim/Groups`**
* **Méthode** : POST
* **Description** : Créer une nouvelle ressource d’équipe.
* **Champs pris en charge** :

| Champ           | Type                | Requis                                                                     |
| --------------- | ------------------- | -------------------------------------------------------------------------- |
| `displayName`   | String              | Oui                                                                        |
| `members`       | Tableau multivaleur | Oui (le sous-champ `value` est requis et correspond à un ID d’utilisateur) |
| `storageBucket` | Objet               | Non                                                                        |

Vous pouvez configurer [Bring your own bucket (BYOB)](/fr/platform/hosting/data-security/secure-storage-connector) pour l’équipe lors de sa création en incluant un objet `storageBucket`. S’il est omis, l’équipe utilise le stockage par défaut ou le stockage défini au niveau de l’instance. Provisionnez le bucket (politique, CORS, identifiants) et déterminez le format de l’adresse de stockage selon le fournisseur à l’aide du guide BYOB. L’objet `storageBucket` comporte les sous-champs suivants :

* **Requis** : `name` (nom du bucket), `provider` (l’une des valeurs suivantes : &#39;coreweave`, `aws`, `gcp`, `azure`, `minio`). Sensible à la casse.
* **Facultatif** : `path` (préfixe de chemin dans le bucket), `kmsKeyId` (clé KMS pour le chiffrement, par exemple pour AWS), `awsExternalId` (accès AWS entre comptes), `azureTenantId` (ID du locataire Azure), `azureClientId` (ID client de l’identité managée Azure).

W&amp;B valide que le bucket existe et est accessible avant de créer l’équipe. Si la validation échoue, la requête SCIM échoue et l’équipe n’est pas créée.

Une valeur `provider` non valide renvoie `400 Bad Request` avec une erreur SCIM qui répertorie les valeurs autorisées.

<div id="etag-support">
  ### Prise en charge des ETags
</div>

L’API SCIM prend en charge les ETags pour les mises à jour conditionnelles, afin d’éviter les conflits liés aux modifications simultanées. Les ETags sont renvoyés dans l’en-tête de réponse `ETag` et dans le champ `meta.version`.

<div id="etags">
  #### ETags
</div>

Pour utiliser les ETags :

1. **Obtenir l’ETag actuel** : lorsque vous récupérez une ressource avec une requête GET, notez l’en-tête ETag dans la réponse
2. **Mise à jour conditionnelle** : incluez l’ETag dans l’en-tête `If-Match` lors de la mise à jour

#### Exemple

```
# Obtenir l'utilisateur et noter l'ETag
GET /scim/Users/abc
# La réponse inclut : ETag: W/"xyz123"

# Mettre à jour avec l'ETag
PATCH /scim/Users/abc
If-Match: W/"xyz123"

{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
    "Operations": [
        {
            "op": "replace",
            "path": "organizationRole",
            "value": "admin"
        }
    ]
}
```

Une réponse d’erreur `412 Precondition Failed` indique que la ressource a été modifiée depuis que vous l’avez récupérée.

<div id="error-handling">
  ### Gestion des erreurs
</div>

L&#39;API SCIM renvoie les réponses d&#39;erreur SCIM standard suivantes :

| Code de statut | Description                                                 |
| -------------- | ----------------------------------------------------------- |
| `200`          | Succès                                                      |
| `201`          | Créé                                                        |
| `204`          | Aucun contenu (suppression réussie)                         |
| `400`          | Mauvaise requête - paramètres ou corps de requête invalides |
| `401`          | Non autorisé - échec de l&#39;authentification              |
| `403`          | Interdit - permissions insuffisantes                        |
| `404`          | Introuvable - la ressource n&#39;existe pas                 |
| `409`          | Conflit - la ressource existe déjà                          |
| `412`          | Échec de la précondition - ETag non correspondant           |
| `500`          | Erreur interne du serveur                                   |

<div id="implementation-differences-per-deployment-type">
  ### Différences d’implémentation selon le type de déploiement
</div>

W&amp;B maintient deux implémentations distinctes de l’API SCIM, dont les fonctionnalités diffèrent :

| Fonctionnalité                                  | Cloud dédié | Autogéré |
| ----------------------------------------------- | ----------- | -------- |
| Mise à jour de l’e-mail de l’utilisateur        | -           | ✓        |
| Mise à jour du nom d’affichage de l’utilisateur | -           | ✓        |
| Désactivation de l’utilisateur                  | ✓           | ✓        |
| Réactivation de l’utilisateur                   | -           | ✓        |
| Plusieurs e-mails par utilisateur               | ✓           | -        |

<div id="limitations">
  ## Limites
</div>

* **Nombre maximal de résultats** : 9999 éléments par requête.
* **Environnements monolocataires** : ne prennent en charge qu’une seule adresse e-mail par utilisateur.
* **Suppression d’une équipe** : non prise en charge via SCIM (utilisez l’interface web W&amp;B).
* **Réactivation d’un utilisateur** : non prise en charge dans les environnements de Cloud mutualisé.
* **Limites de licences** : les opérations peuvent échouer si les limites de licences de l’organisation sont atteintes.
