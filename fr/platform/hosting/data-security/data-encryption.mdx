---
title: Chiffrement des données dans le Cloud dédié
description: "Découvrez comment W&B chiffre les données dans le Cloud dédié à l’aide de clés natives du cloud et de la politique de clé de chiffrement gérée par le client."
---

W&amp;B utilise une clé native du cloud gérée par W&amp;B pour chiffrer la base de données gérée par W&amp;B et le stockage d’objets dans chaque [Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud), en s’appuyant sur la fonctionnalité de clé de chiffrement gérée par le client (CMEK) de chaque cloud. Dans ce cas, W&amp;B agit en tant que `client` du fournisseur de cloud, tout en vous fournissant la plateforme W&amp;B en tant que service. L’utilisation d’une clé gérée par W&amp;B signifie que W&amp;B contrôle les clés qu’il utilise pour chiffrer les données dans chaque cloud, renforçant ainsi son engagement à fournir une plateforme hautement sûre et sécurisée à tous ses clients.

W&amp;B utilise une `clé unique` pour chiffrer les données dans chaque instance client, ce qui ajoute une couche supplémentaire d’isolation entre les tenants du Cloud dédié. Cette fonctionnalité est disponible sur AWS, Azure et Google Cloud.

<Note>
  Les instances Cloud dédié sur Google Cloud et Azure que W&amp;B a provisionnées avant août 2024 utilisent la clé gérée par défaut du fournisseur de cloud pour chiffrer la base de données gérée par W&amp;B et le stockage d’objets. Seules les nouvelles instances que W&amp;B crée depuis août 2024 utilisent la clé native du cloud gérée par W&amp;B pour le chiffrement correspondant.

  Les instances Cloud dédié sur AWS utilisent la clé native du cloud gérée par W&amp;B pour le chiffrement depuis avant août 2024.
</Note>

W&amp;B n’autorise généralement pas les clients à fournir leur propre clé native du cloud pour chiffrer la base de données gérée par W&amp;B et le stockage d’objets dans leur instance Cloud dédié, car plusieurs équipes et profils au sein d’une organisation peuvent avoir accès à son infrastructure cloud pour diverses raisons. Certaines de ces équipes ou certains de ces profils peuvent ne pas avoir conscience que W&amp;B est un composant critique de la pile technologique de l’organisation, et pourraient donc supprimer complètement la clé native du cloud ou révoquer l’accès de W&amp;B à celle-ci. Une telle action pourrait corrompre toutes les données de l’instance W&amp;B de l’organisation et la laisser dans un état irrécupérable.

Si votre organisation doit utiliser sa propre clé native du cloud pour chiffrer la base de données gérée par W&amp;B et le stockage d’objets afin d’autoriser l’utilisation du Cloud dédié pour vos flux de travail d’IA, W&amp;B peut étudier cette demande à titre exceptionnel. Si la demande est approuvée, l’utilisation de votre clé native du cloud pour le chiffrement serait conforme au `modèle de responsabilité partagée` du Cloud dédié W&amp;B. Si un utilisateur de votre organisation supprime votre clé ou révoque l’accès de W&amp;B à celle-ci à tout moment pendant que votre instance Cloud dédié est active, W&amp;B ne serait pas responsable des pertes ou corruptions de données qui en résulteraient, ni de la récupération de ces données.
