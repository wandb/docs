<div id="overview">
  ## Aperçu
</div>

L’opérateur Kubernetes W&amp;B est la méthode recommandée pour déployer W&amp;B Server sur Kubernetes (dans le cloud ou sur site). Pour obtenir un aperçu de l’opérateur, comprendre pourquoi W&amp;B l’utilise et comment fonctionne la hiérarchie de configuration, voir [Autogéré](/fr/platform/hosting/hosting-options/self-managed#about-the-wb-kubernetes-operator).

<div id="before-you-begin">
  ## Avant de commencer
</div>

Avant de déployer W&amp;B avec l’opérateur Kubernetes, assurez-vous que votre infrastructure respecte toutes les exigences :

1. **Consultez les exigences relatives à l’infrastructure** : voir la page [Exigences d’infrastructure pour les déploiements autogérés](/fr/platform/hosting/self-managed/requirements/) pour obtenir tous les détails sur :

* Exigences relatives aux versions logicielles (Kubernetes, MySQL, Redis, Helm)
  * Exigences matérielles (architecture CPU, recommandations de dimensionnement)
  * Configuration du cluster Kubernetes
  * Exigences en matière de réseau, de SSL/TLS et de DNS

2. **Obtenez une licence W&amp;B Server** : voir la section [Licence](/fr/platform/hosting/self-managed/requirements#license) sur la page des exigences.
3. **Provisionnez les services externes** : configurez MySQL, Redis et le stockage d’objets avant le déploiement.

Pour plus de contexte, voir la page [architecture de référence](/fr/platform/hosting/self-managed/ref-arch/).

<div id="openshift-kubernetes-clusters">
  ### Clusters Kubernetes OpenShift
</div>

W&amp;B prend en charge le déploiement sur les [clusters Kubernetes OpenShift](https://www.redhat.com/en/technologies/cloud-computing/openshift) dans des environnements cloud, sur site ou isolés du réseau.

</Note>

<div id="run-the-container-as-an-un-privileged-user">
  #### Exécuter le conteneur en tant qu’utilisateur non privilégié
</div>

Par défaut, les conteneurs utilisent un `$UID` de 999. Spécifiez un `$UID` &gt;= 100000 et un `$GID` de 0 si votre orchestrateur exige que le conteneur s’exécute sous un utilisateur non root.

<Note>
  W&amp;B doit démarrer avec le groupe root (`$GID=0`) afin que les autorisations du système de fichiers fonctionnent correctement.
</Note>

Configurez les contextes de sécurité pour chaque composant W&amp;B. Par exemple, pour configurer le composant API :

```yaml
api:
  install: true
  image:
    repository: wandb/megabinary
    tag: 0.74.1  # Remplacez par votre version actuelle
  pod:
    securityContext:
      fsGroup: 10001
      fsGroupChangePolicy: Always
      runAsGroup: 0
      runAsNonRoot: true
      runAsUser: 10001
      seccompProfile:
        type: RuntimeDefault
  container:
    securityContext:
      allowPrivilegeEscalation: false
      capabilities:
        drop:
          - ALL
      privileged: false
      readOnlyRootFilesystem: false
```

Si nécessaire, configurez un contexte de sécurité personnalisé pour d’autres composants, comme `app` ou `console`. Pour plus de détails, voir [Contexte de sécurité personnalisé](#custom-security-context).

<div id="deploy-wb-server-application">
  ## Déployer l’application W&amp;B Server
</div>

<Note>
  **L’opérateur Kubernetes W&amp;B avec Helm est la méthode d’installation recommandée** pour tous les déploiements W&amp;B autogérés, y compris dans le cloud, sur site et dans des environnements isolés du réseau.
</Note>

Choisissez votre méthode de déploiement :

<Tabs>
  <Tab title="interface de ligne de commande Helm">
    W&amp;B fournit un chart Helm pour déployer l’opérateur Kubernetes W&amp;B sur un cluster Kubernetes. Cette approche vous permet de déployer W&amp;B Server avec la CLI Helm ou un outil de livraison continue comme ArgoCD.

    Pour les considérations spécifiques au déploiement, voir [Considérations spécifiques à l’environnement](#environment-specific-considerations) et [Déployer avec Terraform sur un cloud public](#deploy-with-terraform-on-public-cloud). Pour les environnements déconnectés, voir [Déployer sur Kubernetes isolé du réseau](/fr/platform/hosting/self-managed/on-premises-deployments/kubernetes-airgapped/).

    Suivez ces étapes pour installer l’opérateur Kubernetes W&amp;B avec la CLI Helm :

    1. Ajoutez le dépôt Helm W&amp;B. Le chart Helm W&amp;B est disponible dans le dépôt Helm W&amp;B :
       ```shell
       helm repo add wandb https://charts.wandb.ai
       helm repo update
       ```

    2. Installez l’opérateur sur un cluster Kubernetes :
       ```shell
       helm upgrade --install operator wandb/operator -n wandb-cr --create-namespace
       ```

    3. Configurez la ressource personnalisée de l’opérateur W&amp;B pour déclencher l’installation de W&amp;B Server. Créez un fichier nommé `operator.yaml` avec votre configuration de déploiement W&amp;B. Reportez-vous à la [Référence de configuration](#configuration-reference-for-wb-server) pour connaître toutes les options disponibles.

       Voici une configuration minimale :

       ```yaml
       apiVersion: apps.wandb.com/v1
       kind: WeightsAndBiases
       metadata:
         labels:
           app.kubernetes.io/name: weightsandbiases
           app.kubernetes.io/instance: wandb
         name: wandb
         namespace: default
       spec:
         values:
           global:
             host: https://<HOST_URI>
             license: eyJhbGnUzaH...j9ZieKQ2x5GGfw
             bucket:
               <details depend on the provider>
             mysql:
               <redacted>
           ingress:
             annotations:
               <redacted>
       ```

    4. Démarrez l’opérateur avec votre configuration personnalisée afin qu’il puisse installer, configurer et gérer l’application W&amp;B Server :

       ```shell
       kubectl apply -f operator.yaml
       ```

       Attendez que le déploiement soit terminé. Cela prend quelques minutes.

    5. Pour vérifier l’installation à l’aide de l’interface web, créez le premier compte d’utilisateur administrateur, puis suivez les étapes de vérification décrites dans [Vérifiez l’installation](#verify-the-installation).
  </Tab>

  <Tab title="Terraform">
    Déployez W&amp;B avec Terraform pour des déploiements en infrastructure as code. Choisissez entre :

    * **Helm Terraform Module**: Déploie l’opérateur sur une infrastructure Kubernetes existante
    * **Cloud Terraform Modules**: Déploiement complet de l’infrastructure et de l’application pour AWS, Google Cloud et Azure

    Pour les considérations propres au déploiement, voir [Considérations spécifiques à l’environnement](#environment-specific-considerations) et [Déployer avec Terraform sur un cloud public](#deploy-with-terraform-on-public-cloud). Pour les environnements déconnectés, voir [Déployer sur Kubernetes isolé du réseau](/fr/platform/hosting/self-managed/on-premises-deployments/kubernetes-airgapped/).

    #### Helm Terraform Module

    Cette méthode permet de personnaliser les déploiements en fonction d’exigences spécifiques, tout en s’appuyant sur l’approche d’infrastructure as code de Terraform pour garantir cohérence et reproductibilité. Le module Terraform officiel de W&amp;B basé sur Helm se trouve [ici](https://registry.terraform.io/modules/wandb/wandb/helm/latest).

    Utilisez le code suivant comme point de départ. Il inclut toutes les options de configuration nécessaires pour un déploiement de niveau production :

    ```hcl
    module "wandb" {
      source  = "wandb/wandb/helm"

      spec = {
        values = {
          global = {
            host    = "https://<HOST_URI>"
            license = "eyJhbGnUzaH...j9ZieKQ2x5GGfw"

            bucket = {
              <details depend on the provider>
            }

            mysql = {
              <redacted>
            }
          }

          ingress = {
            annotations = {
              "a" = "b"
              "x" = "y"
            }
          }
        }
      }
    }
    ```

    Notez que les options de configuration sont les mêmes que celles décrites dans la [Référence de configuration](#configuration-reference-for-wb-server), mais que la syntaxe doit respecter le HashiCorp Configuration Language (HCL). Le module Terraform crée la définition de ressource personnalisée (CRD) de W&amp;B.

    Pour voir comment W&amp;B utilise ses propres modules Helm Terraform pour déployer des installations de Cloud dédié pour ses clients, suivez ces liens :

    * [AWS](https://github.com/wandb/terraform-aws-wandb/blob/45e1d746f53e78e73e68f911a1f8cad5408e74b6/main.tf#L225)
    * [Azure](https://github.com/wandb/terraform-azurerm-wandb/blob/170e03136b6b6fc758102d59dacda99768854045/main.tf#L155)
    * [Google Cloud](https://github.com/wandb/terraform-google-wandb/blob/49ddc3383df4cefc04337a2ae784f57ce2a2c699/main.tf#L189)

    #### Modules Terraform pour le cloud

    W&amp;B fournit un ensemble de modules Terraform pour AWS, Google Cloud et Azure. Ces modules déploient des infrastructures complètes, notamment des clusters Kubernetes, des équilibreurs de charge, des bases de données MySQL, entre autres, ainsi que l’application W&amp;B Server. L’opérateur Kubernetes W&amp;B est déjà inclus dans ces modules Terraform officiels W&amp;B spécifiques au cloud, dans les versions suivantes :

    | Terraform Registry                                                  | Code source                                      | Version |
    | ------------------------------------------------------------------- | ------------------------------------------------ | ------- |
    | [AWS](https://registry.terraform.io/modules/wandb/wandb/aws/latest) | https://github.com/wandb/terraform-aws-wandb     | v4.0.0+ |
    | [Azure](https://github.com/wandb/terraform-azurerm-wandb)           | https://github.com/wandb/terraform-azurerm-wandb | v2.0.0+ |
    | [Google Cloud](https://github.com/wandb/terraform-google-wandb)     | https://github.com/wandb/terraform-google-wandb  | v2.0.0+ |

    Cette intégration garantit que l’opérateur Kubernetes W&amp;B est prêt à l’emploi pour votre instance avec une configuration minimale, offrant ainsi une voie simplifiée pour déployer et gérer W&amp;B Server dans votre environnement cloud.

    Pour des instructions détaillées sur l’utilisation de ces modules spécifiques au cloud, voir [Déployer avec Terraform sur un cloud public](#deploy-with-terraform-on-public-cloud) ci-dessous.

<div id="environment-specific-considerations">
  ## Considérations spécifiques à l’environnement
</div>

Kubernetes fonctionne de la même manière, qu’il soit déployé sur site ou dans le cloud. Les principales différences tiennent aux conventions de nommage et aux services gérés (par exemple, MySQL vs RDS, ou S3 vs stockage d’objets sur site). Cette section aborde les considérations qui varient selon l’environnement.

<div id="on-premises-and-bare-metal">
  ### Sur site et en bare metal
</div>

Lorsque vous déployez sur un cluster Kubernetes sur site ou en bare metal, prêtez attention aux points suivants.

<div id="load-balancer-configuration">
  #### Configuration de l’équilibreur de charge
</div>

Les clusters Kubernetes sur site nécessitent généralement une configuration manuelle de l’équilibreur de charge. Les options incluent :

* **Équilibreur de charge externe** : Configurer un équilibreur de charge matériel ou logiciel existant (F5, HAProxy, etc.)
* **Contrôleur d’ingress Nginx** : Déployer nginx-ingress-controller avec NodePort ou le réseau de l’hôte
* **MetalLB** : Pour les clusters Kubernetes bare metal, MetalLB fournit des services d’équilibreur de charge

Pour des exemples détaillés de configuration de l’équilibreur de charge, voir la [section réseau de l’architecture de référence](/fr/platform/hosting/self-managed/ref-arch#networking).

<div id="persistent-storage">
  #### Stockage persistant
</div>

Assurez-vous que votre cluster Kubernetes dispose d’une `StorageClass` configurée pour les volumes persistants. Les composants W&amp;B peuvent nécessiter un stockage persistant pour la mise en cache et le stockage temporaire de données.

Options de stockage sur site courantes :

* Classes de stockage basées sur NFS
* Stockage Ceph/Rook
* Volumes persistants locaux
* Solutions de stockage d’entreprise (NetApp, Pure Storage, etc.)

<div id="dns-and-certificate-management">
  #### Gestion du DNS et des certificats
</div>

Pour les déploiements sur site :

* Configurer les enregistrements DNS internes pour qu’ils pointent vers votre nom d’hôte W&amp;B
* Obtenir des certificats SSL/TLS auprès de votre autorité de certification (CA) interne
* Si vous utilisez des certificats auto-signés, configurer l’opérateur pour qu’il approuve votre certificat d’autorité de certification

Voir les [exigences SSL/TLS](/fr/platform/hosting/self-managed/requirements#ssl-tls) pour plus de détails sur la configuration des certificats.

<div id="openshift-deployments">
  #### Déploiements OpenShift
</div>

W&amp;B prend entièrement en charge le déploiement sur les clusters Kubernetes OpenShift. Les déploiements OpenShift nécessitent des configurations supplémentaires du contexte de sécurité en raison des politiques de sécurité plus strictes d&#39;OpenShift.

Pour plus de détails sur la configuration propre à OpenShift, voir [clusters Kubernetes OpenShift](#openshift-kubernetes-clusters) ci-dessus. Pour des exemples complets de déploiement OpenShift dans des environnements isolés du réseau, voir [Déployer sur Kubernetes isolé du réseau](/fr/platform/hosting/self-managed/on-premises-deployments/kubernetes-airgapped#openshift-configuration).

<div id="object-storage-for-on-premises-and-s3-compatible">
  #### Stockage d’objets pour les environnements sur site et compatibles S3
</div>

Après avoir provisionné votre bucket de stockage d’objets (voir [Provisionnement du stockage d’objets](/fr/platform/hosting/data-security/secure-storage-connector)), configurez-le dans votre Ressource personnalisée W&amp;B.

**AWS S3 (sur site)**

Pour AWS S3 sur site (via Outposts ou un stockage compatible) :

```yaml
bucket:
  kmsKey: <kms key arn>  # Clé KMS facultative pour le chiffrement
  name: <bucket name>    # Exemple : wandb
  path: ""               # Laisser comme chaîne vide
  provider: s3
  region: <region>       # Exemple : us-east-1
```

**Stockage compatible avec S3 (MinIO, Ceph, NetApp, etc.)**

Pour les systèmes de stockage compatibles avec S3 :

```yaml
bucket:
  kmsKey: null
  name: <s3 endpoint>    # Exemple : s3.example.com:9000
  path: <bucket name>    # Exemple : wandb
  provider: s3
  region: <region>       # Exemple : us-east-1
```

Pour activer TLS pour un stockage compatible S3, ajoutez `?tls=true` au chemin du bucket :

```yaml
bucket:
  path: "wandb?tls=true"
```

<Warning>
  Le certificat doit être reconnu comme fiable. Les certificats auto-signés nécessitent une configuration supplémentaire. Voir les [exigences SSL/TLS](/fr/platform/hosting/self-managed/requirements#ssl-tls) pour plus de détails.
</Warning>

**Considérations importantes pour le stockage d’objets sur site**

Si vous exploitez votre propre stockage d’objets, tenez compte des points suivants :

1. **Capacité de stockage et performances** : surveillez attentivement la capacité disque. Avec une utilisation moyenne de W&amp;B, il faut compter de quelques dizaines à quelques centaines de gigaoctets. Une utilisation intensive peut entraîner une consommation de stockage de l’ordre du pétaoctet.
2. **Tolérance aux pannes** : utilisez au minimum des baies RAID pour les disques physiques. Pour un stockage compatible S3, utilisez des configurations distribuées ou hautement disponibles.
3. **Disponibilité** : configurez la surveillance pour garantir que le stockage reste disponible.

**Considérations relatives à MinIO**

<Warning>
  MinIO Open Source est en [mode maintenance](https://github.com/minio/minio), sans développement actif. Les binaires précompilés ne sont plus fournis, et seuls les correctifs de sécurité critiques sont examinés au cas par cas. Pour les déploiements de production, W&amp;B recommande d’utiliser des services de stockage d’objets gérés ou [MinIO Enterprise (AIStor)](https://min.io/product/aistor).
</Warning>

Les alternatives Enterprise pour le stockage d’objets sur site incluent :

* [Amazon S3 on Outposts](https://aws.amazon.com/s3/outposts/)
* [NetApp StorageGRID](https://www.netapp.com/data-storage/storagegrid/)
* MinIO Enterprise (AIStor)
* [Dell ObjectScale](https://www.dell.com/en-us/shop/cty/sf/objectscale)

Si vous utilisez un déploiement MinIO existant ou MinIO Enterprise, vous pouvez créer un bucket à l’aide du client MinIO :

```bash
mc config host add local http://$MINIO_HOST:$MINIO_PORT "$MINIO_ACCESS_KEY" "$MINIO_SECRET_KEY" --api s3v4
mc mb --region=us-east-1 local/wandb-files
```

<div id="deploy-with-terraform-on-public-cloud">
  ## Déployer avec Terraform sur un cloud public
</div>

<Note>
  W&amp;B recommande des options de déploiement entièrement gérées, comme les types de déploiement [W&amp;B Cloud mutualisé](/fr/platform/hosting/hosting-options/multi_tenant_cloud) ou [W&amp;B Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud). Les services entièrement gérés de W&amp;B sont simples à utiliser, sécurisés et nécessitent peu, voire aucune, configuration.
</Note>

W&amp;B fournit des modules Terraform pour déployer la plateforme sur des fournisseurs de cloud public. Ces modules automatisent le provisionnement de l’infrastructure et l’installation de W&amp;B Server.

Avant de commencer, W&amp;B recommande de choisir l’un des [backends distants](https://developer.hashicorp.com/terraform/language/backend) disponibles pour Terraform afin d’y stocker le [fichier d’état](https://developer.hashicorp.com/terraform/language/state). Le fichier d’état est la ressource nécessaire pour appliquer des mises à niveau ou apporter des modifications à votre déploiement sans recréer tous les composants.

Sélectionnez votre fournisseur de cloud :

<Tabs>
  <Tab title="AWS">
    W&amp;B recommande d&#39;utiliser le [W&amp;B Server AWS Terraform Module](https://registry.terraform.io/modules/wandb/wandb/aws/latest) pour déployer la plateforme sur AWS.

    Le module Terraform déploie les composants obligatoires suivants :

    * Équilibreur de charge
    * AWS Identity &amp; Access Management (IAM)
    * AWS Key Management System (KMS)
    * Amazon Aurora MySQL
    * Amazon VPC
    * Amazon S3
    * Amazon Route53
    * Amazon Certificate Manager (ACM)
    * Amazon Elastic Load Balancing (ALB)
    * Amazon Secrets Manager

    Les composants facultatifs incluent :

    * ElastiCache pour Redis
    * SQS

    ### Permissions prérequises

    Le compte qui exécute Terraform doit pouvoir créer tous les composants décrits ci-dessus et disposer des autorisations nécessaires pour créer des **IAM Policies** et des **IAM Roles** et attribuer des rôles aux ressources.

    ### Étapes générales

    Les étapes de cette section s&#39;appliquent à toutes les options de déploiement.

    1. Préparez l’environnement de développement.
       * Installez [Terraform](https://developer.hashicorp.com/terraform/tutorials/aws-get-started/install-cli)
       * W&amp;B recommande de créer un dépôt Git pour le versioning.

    2. Créez le fichier `terraform.tfvars`.

       Le contenu du fichier `tvfars` peut être personnalisé en fonction du type d’installation, mais la configuration minimale recommandée ressemble à l’exemple ci-dessous.

       ```bash
       namespace                  = "wandb"
       license                    = "xxxxxxxxxxyyyyyyyyyyyzzzzzzz"
       subdomain                  = "wandb-aws"
       domain_name                = "wandb.ml"
       zone_id                    = "xxxxxxxxxxxxxxxx"
       allowed_inbound_cidr       = ["0.0.0.0/0"]
       allowed_inbound_ipv6_cidr  = ["::/0"]
       eks_cluster_version        = "1.29"
       ```

       Veillez à définir les variables dans votre fichier `tvfars` avant de déployer, car la variable `namespace` est une chaîne de caractères utilisée comme préfixe pour toutes les ressources créées par Terraform.

       La combinaison de `subdomain` et de `domain` forme le FQDN de votre instance W&amp;B. Dans l’exemple ci-dessus, le FQDN W&amp;B sera `wandb-aws.wandb.ml`, ainsi que le `zone_id` DNS dans lequel l’enregistrement FQDN sera créé.

       Les paramètres `allowed_inbound_cidr` et `allowed_inbound_ipv6_cidr` doivent également être définis. Dans le module, il s’agit d’une entrée obligatoire. L’exemple suivant autorise l’accès à l’installation W&amp;B depuis n’importe quelle source.

    3. Créez le fichier `versions.tf`

       Ce fichier contiendra les versions requises de Terraform et du fournisseur Terraform pour déployer W&amp;B sur AWS :

       ```bash
       provider "aws" {
         region = "eu-central-1"

         default_tags {
           tags = {
             GithubRepo = "terraform-aws-wandb"
             GithubOrg  = "wandb"
             Enviroment = "Example"
             Example    = "PublicDnsExternal"
           }
         }
       }
       ```

       Consultez la [documentation officielle de Terraform](https://registry.terraform.io/providers/hashicorp/aws/latest/docs#provider-configuration) pour configurer le fournisseur AWS.

       Facultativement — mais c’est vivement recommandé — ajoutez la [configuration de backend distant](https://developer.hashicorp.com/terraform/language/backend) mentionnée au début de cette documentation.

    4. Créez le fichier `variables.tf`

       Pour chaque option configurée dans le `terraform.tfvars`, Terraform exige une déclaration de variable correspondante.

       ```
       variable "namespace" {
         type        = string
         description = "Préfixe de nom utilisé pour les ressources"
       }

       variable "domain_name" {
         type        = string
         description = "Nom de domaine utilisé pour accéder à l'instance."
       }

       variable "subdomain" {
         type        = string
         default     = null
         description = "Sous-domaine pour accéder à l'interface Weights & Biases."
       }

       variable "license" {
         type = string
       }

       variable "zone_id" {
         type        = string
         description = "Domaine sur lequel créer le sous-domaine Weights & Biases."
       }

       variable "allowed_inbound_cidr" {
        description = "CIDRs autorisés à accéder à wandb-server."
        nullable    = false
        type        = list(string)
       }

       variable "allowed_inbound_ipv6_cidr" {
        description = "CIDRs autorisés à accéder à wandb-server."
        nullable    = false
        type        = list(string)
       }

       variable "eks_cluster_version" {
        description = "Version Kubernetes du cluster EKS"
        nullable    = false
        type        = string
       }
       ```

    ### Déploiement recommandé

<div id="migrate-to-operator-based-helm-chart">
  ### Migrer vers un chart Helm basé sur l’opérateur
</div>

Suivez ces étapes pour migrer vers le chart Helm basé sur l’opérateur :

1. Obtenez la configuration W&amp;B actuelle. Si W&amp;B a été déployé avec une version du chart Helm non basée sur l’opérateur, exportez les valeurs comme suit :
   ```shell
   helm get values wandb
   ```
   Si W&amp;B a été déployé avec des manifestes Kubernetes, exportez les valeurs comme suit :
   ```shell
   kubectl get deployment wandb -o yaml
   ```
   Vous disposez maintenant de toutes les valeurs de configuration nécessaires pour l’étape suivante.

2. Créez un fichier nommé `operator.yaml`. Suivez le format décrit dans la [Référence de configuration](#configuration-reference-for-wb-operator). Utilisez les valeurs de l’étape 1.

3. Faites passer le nombre de pods du déploiement actuel à 0. Cette étape arrête le déploiement actuel.
   ```shell
   kubectl scale --replicas=0 deployment wandb
   ```

4. Mettez à jour le dépôt du chart Helm :
   ```shell
   helm repo update
   ```

5. Installez le nouveau chart Helm :
   ```shell
   helm upgrade --install operator wandb/operator -n wandb-cr --create-namespace
   ```

6. Configurez le nouveau chart Helm et déclenchez le déploiement de l’application W&amp;B. Appliquez la nouvelle configuration.
   ```shell
   kubectl apply -f operator.yaml
   ```
   Le déploiement prend quelques minutes.

7. Vérifiez l’installation. Assurez-vous que tout fonctionne en suivant les étapes de [Vérifiez l’installation](#verify-the-installation).

8. Supprimez l’ancienne installation. Désinstallez l’ancien chart Helm ou supprimez les ressources créées à partir de manifestes.

<accordion title="Configuration LDAP héritée">
  Cette ancienne approche n’est plus recommandée. Cette section est fournie à titre de référence.

  **Sans TLS**

  ```yaml
  global:
    ldap:
      enabled: true
      # Adresse du serveur LDAP, y compris "ldap://" ou "ldaps://"
      host:
      # Base de recherche LDAP à utiliser pour trouver des utilisateurs
      baseDN:
      # Utilisateur LDAP à utiliser pour la liaison (si vous n’utilisez pas de liaison anonyme)
      bindDN:
      # Nom du secret et clé contenant le mot de passe LDAP à utiliser pour la liaison (si vous n’utilisez pas de liaison anonyme)
      bindPW:
      # Attribut LDAP pour l’adresse e-mail et noms d’attributs d’ID de groupe, sous forme de chaînes séparées par des virgules.
      attributes:
      # Liste des groupes LDAP autorisés
      groupAllowList:
      # Activer TLS pour LDAP
      tls: false
  ```

  **Avec TLS**

  La configuration du certificat TLS LDAP nécessite une ConfigMap créée au préalable avec le contenu du certificat.

  Pour créer la ConfigMap, vous pouvez utiliser la commande suivante :

  ```console
  kubectl create configmap ldap-tls-cert --from-file=certificate.crt
  ```

  Utilisez ensuite la ConfigMap dans le YAML, comme dans l’exemple ci-dessous.

  ```yaml
  global:
    ldap:
      enabled: true
      # Adresse du serveur LDAP, y compris "ldap://" ou "ldaps://"
      host:
      # Base de recherche LDAP à utiliser pour trouver des utilisateurs
      baseDN:
      # Utilisateur LDAP à utiliser pour la liaison (si vous n’utilisez pas de liaison anonyme)
      bindDN:
      # Nom du secret et clé contenant le mot de passe LDAP à utiliser pour la liaison (si vous n’utilisez pas de liaison anonyme)
      bindPW:
      # Attribut LDAP pour l’adresse e-mail et noms d’attributs d’ID de groupe, sous forme de chaînes séparées par des virgules.
      attributes:
      # Liste des groupes LDAP autorisés
      groupAllowList:
      # Activer TLS pour LDAP
      tls: true
      # Nom de la ConfigMap et clé contenant le certificat d’autorité de certification du serveur LDAP
      tlsCert:
        configMap:
          name: "ldap-tls-cert"
          key: "certificate.crt"
  ```
</accordion>

<div id="custom-security-context">
  ### Contexte de sécurité personnalisé
</div>

Chaque composant W&amp;B prend en charge des configurations de contexte de sécurité personnalisées de la forme suivante :

```yaml
pod:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1001
    runAsGroup: 0
    fsGroup: 1001
    fsGroupChangePolicy: Always
    seccompProfile:
      type: RuntimeDefault
container:
  securityContext:
    capabilities:
      drop:
        - ALL
    readOnlyRootFilesystem: false
    allowPrivilegeEscalation: false 
```

<Note>
  La seule valeur valide pour `runAsGroup:` est `0`. Toute autre valeur entraîne une erreur.
</Note>

Par exemple, pour configurer le pod de l’application, ajoutez une section `app` à votre configuration :

```yaml
global:
  ...
app:
  pod:
    securityContext:
      runAsNonRoot: true
      runAsUser: 1001
      runAsGroup: 0
      fsGroup: 1001
      fsGroupChangePolicy: Always
      seccompProfile:
        type: RuntimeDefault
  container:
    securityContext:
      capabilities:
        drop:
          - ALL
      readOnlyRootFilesystem: false
      allowPrivilegeEscalation: false 
```

Le même principe s&#39;applique à `console`, `weave`, `weave-trace` et `parquet`.

<div id="what-is-the-purposerole-of-each-individual-pod">
  ### Quel est l’objectif ou le rôle de chaque pod ?
</div>

* **`wandb-app`** : le cœur de W&amp;B, y compris l’API GraphQL et l’application web. Il fournit la majeure partie des fonctionnalités de la plateforme.
* **`wandb-console`** : la console d’administration, accessible via `/console`.
* **`wandb-otel`** : l’agent OpenTelemetry, qui collecte les métriques et les journaux des ressources au niveau de Kubernetes pour les afficher dans la console d’administration.
* **`wandb-prometheus`** : le serveur Prometheus, qui capture les métriques de différents composants pour les afficher dans la console d’administration.
* **`wandb-parquet`** : un microservice backend distinct du pod `wandb-app` qui exporte les données de la base de données vers le stockage d’objets au format Parquet.
* **`wandb-weave`** : un autre microservice backend qui charge les tableaux de requête dans l’interface utilisateur et prend en charge diverses fonctionnalités clés de l’application.
* **`wandb-weave-trace`** : un framework permettant de suivre, d’expérimenter, d’évaluer, de déployer et d’améliorer des applications basées sur des LLM. Ce framework est accessible via le pod `wandb-app`.

<div id="how-to-access-the-wb-operator-console-if-ingress-doesnt-work">
  ### Comment accéder à la console de l’opérateur W&amp;B si Ingress ne fonctionne pas
</div>

Exécutez la commande suivante sur un hôte pouvant accéder au cluster Kubernetes :

```console
kubectl port-forward svc/wandb-console 8082
```

Accédez à la console dans votre navigateur à l’adresse `https://localhost:8082/`.

Voir [accès à la console de gestion de l’opérateur Kubernetes W&amp;B](#access-the-wb-management-console) pour savoir comment obtenir le mot de passe (option 2).

<div id="how-to-view-wb-server-logs">
  ### Comment afficher les journaux de W&amp;B Server
</div>

Le pod de l’application est nommé **wandb-app-xxx**.

```console
kubectl get pods
kubectl logs wandb-XXXXX-XXXXX
```

<div id="how-to-identify-the-kubernetes-ingress-class">
  ### Comment identifier la classe d’ingress Kubernetes
</div>

Vous pouvez connaître la classe d’ingress installée sur votre cluster en exécutant

```console
kubectl get ingressclass
```
