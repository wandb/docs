Cette page fournit un aperçu complet des exigences en matière d’infrastructure et de logiciel pour le déploiement de W&amp;B Autogéré. Examinez ces exigences avant de commencer votre déploiement.

<Note>
  W&amp;B recommande des options de déploiement entièrement gérées, telles que [W&amp;B Cloud mutualisé](/fr/platform/hosting/hosting-options/multi_tenant_cloud) ou [W&amp;B Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud). Les services W&amp;B entièrement gérés sont simples et sécurisés à utiliser, et nécessitent peu, voire aucune, configuration.
</Note>

Pour des conseils d’architecture complets, voir l’[architecture de référence](/fr/platform/hosting/self-managed/ref-arch/).

<div id="kubernetes">
  ## Kubernetes
</div>

Le serveur W&amp;B est déployé sous la forme d’un [opérateur Kubernetes](/fr/platform/hosting/self-managed/operator/) qui gère plusieurs pods. Votre cluster Kubernetes doit respecter ces exigences :

* **Version** : Voir [Exigences relatives aux versions logicielles](#software-version-requirements) ci-dessus
* **Contrôleur d’ingress** : Un contrôleur d’ingress entièrement configuré et fonctionnel (Nginx, Istio, Traefik ou ingress d’un fournisseur de cloud)
* **Volumes persistants** : Capacité à provisionner des volumes persistants
* **Architecture CPU** : architecture 64 bits Intel ou AMD (ARM n’est pas pris en charge)

W&amp;B prend en charge le déploiement sur des [clusters Kubernetes OpenShift](https://www.redhat.com/en/technologies/cloud-computing/openshift) dans des environnements cloud, sur site et isolés du réseau. Pour obtenir des instructions de configuration spécifiques, voir la [section OpenShift](/fr/platform/hosting/self-managed/operator/#openshift-kubernetes-clusters) du guide de l’opérateur.

Pour connaître l’ensemble des exigences Kubernetes, y compris la configuration de l’équilibreur de charge et de l’ingress, voir la [section Kubernetes de l’architecture de référence](/fr/platform/hosting/self-managed/ref-arch/#kubernetes).

<div id="mysql-database">
  ## Base de données MySQL
</div>

<SelfManagedMysqlRequirements />

**W&amp;B recommande vivement d&#39;utiliser des services de base de données gérés** tels qu&#39;AWS RDS Aurora MySQL, Google Cloud SQL for MySQL ou Azure Database for MySQL pour les déploiements en Production. Les services gérés offrent des sauvegardes automatisées, la surveillance, une haute disponibilité et l&#39;application de correctifs, tout en réduisant considérablement la complexité opérationnelle.

<div id="mysql-configuration-parameters">
  ### Paramètres de configuration de MySQL
</div>

Si vous utilisez votre propre instance MySQL, configurez MySQL avec les paramètres suivants :

```
binlog_format = 'ROW'
binlog_row_image = 'MINIMAL'
innodb_flush_log_at_trx_commit = 1
innodb_online_alter_log_max_size = 268435456
max_prepared_stmt_count = 1048576
sort_buffer_size = '67108864'
sync_binlog = 1
```

Ces paramètres ont été validés par W&amp;B pour garantir des performances et une fiabilité optimales.

<div id="dns">
  ### DNS
</div>

Le nom de domaine pleinement qualifié (FQDN) du déploiement W&amp;B doit pointer, via un enregistrement A, vers l’adresse IP de l’ingress/équilibreur de charge.

<div id="load-balancer-and-ingress">
  ### Équilibreur de charge et ingress
</div>

L’opérateur Kubernetes W&amp;B expose les services à l’aide d’un contrôleur d’ingress Kubernetes, qui achemine les requêtes vers les points de terminaison des services en fonction des chemins d’URL. Le contrôleur d’ingress doit être accessible depuis toutes les machines qui exécutent des charges de travail de machine learning ou qui accèdent au service via un navigateur web.

Pour plus de détails sur les options d’équilibreur de charge, les exigences relatives au contrôleur d’ingress et des exemples de configuration, voir la [section de l’architecture de référence consacrée à l’équilibreur de charge](/fr/platform/hosting/self-managed/ref-arch/#load-balancer-and-ingress).
