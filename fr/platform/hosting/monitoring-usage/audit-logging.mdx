Utilisez les journaux d’audit W&amp;B pour suivre l’activité des utilisateurs au sein de votre organisation et répondre à vos exigences de gouvernance d’entreprise. Les journaux d’audit sont disponibles au format JSON. Voir [Schéma du journal d’audit](#audit-log-schema).

La méthode d’accès aux journaux d’audit dépend du type de déploiement de votre plateforme W&amp;B :

| Type de déploiement de la plateforme W&amp;B                            | Mécanisme d’accès aux journaux d’audit                                                                                                                                                                                                                                                                                                                                     |
| ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| [Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud)        | <ul><li>[BYOB au niveau de l’instance](/fr/platform/hosting/data-security/secure-storage-connector) : synchronisés vers le bucket au niveau de l’instance (BYOB) toutes les 10 minutes. Également disponibles via [l’API](#fetch-audit-logs-using-api).</li><li>Stockage par défaut au niveau de l’instance : disponible uniquement via [l’API](#fetch-audit-logs-using-api).</li></ul> |
| [Cloud mutualisé](/fr/platform/hosting/hosting-options/multi_tenant_cloud) | Disponible uniquement pour les plans Enterprise. Disponible uniquement via [l’API](#fetch-audit-logs-using-api).                                                                                                                                                                                                                                                           |
| [Autogéré](/fr/platform/hosting/hosting-options/self-managed)              | Synchronisés vers le bucket au niveau de l’instance toutes les 10 minutes. Également disponibles via [l’API](#fetch-audit-logs-using-api).                                                                                                                                                                                                                                           |

Après avoir récupéré les journaux d’audit, vous pouvez les analyser à l’aide d’outils tels que [Pandas](https://pandas.pydata.org/docs/index.html), [Amazon Redshift](https://aws.amazon.com/redshift/), [Google BigQuery](https://cloud.google.com/bigquery) ou [Microsoft Fabric](https://www.microsoft.com/microsoft-fabric). Certains outils d’analyse des journaux d’audit ne prennent pas en charge le format JSON ; consultez la documentation de votre outil d’analyse pour connaître les recommandations et les exigences relatives à la transformation des journaux d’audit au format JSON avant l’analyse.

Pour plus de détails sur le format des journaux, voir [Schéma du journal d’audit](#audit-log-schema) et [Actions](#actions).

<div id="audit-log-retention">
  ## Rétention des journaux d’audit
</div>

* Si vous devez conserver les journaux d’audit pendant une durée déterminée, W&amp;B recommande de les transférer périodiquement vers un stockage à long terme, soit à l’aide de buckets de stockage, soit via l’API Audit Logging.
* Si vous êtes soumis au [Health Insurance Portability and Accountability Act of 1996 (HIPAA)](https://hhs.gov/hipaa/for-professionals/index.html), les journaux d’audit doivent être conservés pendant au moins 6 ans dans un environnement où ils ne peuvent être ni supprimés ni modifiés par un acteur interne ou externe avant la fin de la période de conservation obligatoire. Pour les instances [Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud) conformes à [BYOB](/fr/platform/hosting/data-security/secure-storage-connector), vous devez configurer des garde-fous pour votre stockage géré, y compris pour tout stockage de rétention à long terme.

<div id="audit-log-schema">

</div>

| Clé                       | Définition                                                                                                                                                                               |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `action`                  | L&#39;[action](#actions) de l&#39;événement.                                                                                                                                             |
| `actor_email`             | L&#39;adresse e-mail de l&#39;utilisateur qui a initié l&#39;action, le cas échéant.                                                                                                     |
| `actor_ip`                | L&#39;adresse IP de l&#39;utilisateur qui a initié l&#39;action.                                                                                                                         |
| `actor_user_id`           | L&#39;ID de l&#39;utilisateur connecté qui a effectué l&#39;action, le cas échéant.                                                                                                      |
| `artifact_asset`          | L&#39;ID de l&#39;artefact associé à l&#39;action, le cas échéant.                                                                                                                       |
| `artifact_digest`         | L&#39;empreinte de l&#39;artefact associée à l&#39;action, le cas échéant.                                                                                                               |
| `artifact_qualified_name` | Le nom complet de l&#39;artefact associé à l&#39;action, le cas échéant.                                                                                                                 |
| `artifact_sequence_asset` | L&#39;ID de la séquence d&#39;artefacts associée à l&#39;action, le cas échéant.                                                                                                         |
| `cli_version`             | La version du SDK Python qui a initié l&#39;action, le cas échéant.                                                                                                                      |
| `entity_asset`            | L&#39;ID de l&#39;entité ou de l&#39;équipe associé à l&#39;action, le cas échéant.                                                                                                      |
| `entity_name`             | Le nom de l&#39;entité ou de l&#39;équipe associé à l&#39;action, le cas échéant.                                                                                                        |
| `project_asset`           | Le projet associé à l&#39;action, le cas échéant.                                                                                                                                        |
| `project_name`            | Le nom du projet associé à l&#39;action, le cas échéant.                                                                                                                                 |
| `report_asset`            | L&#39;ID du rapport associé à l&#39;action, le cas échéant.                                                                                                                              |
| `report_name`             | Le nom du rapport associé à l&#39;action, le cas échéant.                                                                                                                                |
| `response_code`           | Le code de réponse HTTP renvoyé pour l&#39;action, le cas échéant.                                                                                                                       |
| `timestamp`               | La date et l&#39;heure de l&#39;événement au [format RFC3339](https://www.rfc-editor.org/rfc/rfc3339). Par exemple, `2023-01-23T12:34:56Z` représente le 23 janvier 2023 à 12:34:56 UTC. |
| `user_asset`              | La ressource utilisateur affectée par l&#39;action (plutôt que l&#39;utilisateur qui effectue l&#39;action), le cas échéant.                                                             |
| `user_email`              | L&#39;adresse e-mail de l&#39;utilisateur affecté par l&#39;action (plutôt que l&#39;adresse e-mail de l&#39;utilisateur qui effectue l&#39;action), le cas échéant.                     |

<div id="personally-identifiable-information-pii">
  ### Information personnelle identifiable (PII)
</div>

L’information personnelle identifiable (PII), telle que les adresses e-mail et les noms des projets, des équipes et des rapports, est disponible uniquement via l’option du point de terminaison API.

* Pour [Autogéré](/fr/platform/hosting/hosting-options/self-managed) et
  [Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud), un administrateur d’organisation peut [exclure les PII](#exclude-pii) lors de la récupération des journaux d’audit.
* Pour [Cloud mutualisé](/fr/platform/hosting/hosting-options/multi_tenant_cloud), le point de terminaison API renvoie toujours les champs pertinents pour les journaux d’audit, y compris les PII. Ceci n&#39;est pas configurable.

<div id="before-you-begin">
  ## Avant de commencer
</div>

1. Les administrateurs de l’organisation peuvent récupérer les journaux d’audit. Si vous recevez une erreur `403`, assurez-vous que vous, ou votre compte de service, disposez des autorisations nécessaires.
2. **Cloud mutualisé** : si vous êtes membre de plusieurs organisations Cloud mutualisé, vous *devez* configurer l’**organisation API par défaut**, qui détermine vers quelle organisation les appels d’API de journalisation d’audit sont dirigés. Sinon, vous recevrez l’erreur suivante :

   ```text
   user is associated with multiple organizations but no valid org ID found in user info
   ```

   Pour spécifier votre organisation API par défaut :

   1. Cliquez sur votre image de profil, puis sur **Paramètres utilisateur**.
   2. Pour **l’organisation API par défaut**, sélectionnez une organisation.

   Cela ne s’applique pas à un compte de service, qui ne peut être membre que d’une seule organisation Cloud mutualisé.

<div id="fetch-audit-logs">
  ## Récupérer les journaux d’audit
</div>

Pour récupérer les journaux d’audit :

1. Déterminez le point de terminaison API approprié pour votre instance :

   * [Autogéré](/fr/platform/hosting/hosting-options/self-managed) : `<wandb-platform-url>/admin/audit_logs`
   * [Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud) : `<instance-name>.wandb.io/admin/audit_logs`
   * [Cloud mutualisé (Enterprise requis)](/fr/platform/hosting/hosting-options/multi_tenant_cloud) : `https://api.wandb.ai/audit_logs`

   Dans les étapes suivantes, remplacez `<API-endpoint>` par votre point de terminaison API.
2. (Facultatif) Construisez les paramètres de requête à ajouter au point de terminaison. Dans les étapes suivantes, remplacez `<parameters>` par la chaîne obtenue.
   * `anonymize` : si l’URL inclut le paramètre `anonymize=true`, toutes les PII sont supprimées. Sinon, les PII sont incluses. Référez-vous à [Exclure les PII lors de la récupération des journaux d’audit](#exclude-pii-when-fetching-audit-logs). Non pris en charge sur le Cloud mutualisé, où tous les champs sont inclus, y compris les PII.
   * Configurez la plage de dates des journaux à récupérer à l’aide d’une combinaison de `numdays` et `startDate`. Chaque paramètre est facultatif, et ils interagissent entre eux.
     * Si aucun des deux paramètres n’est inclus, seuls les journaux d’aujourd’hui sont récupérés.
     * `numDays` : entier indiquant le nombre de jours à remonter à partir de `startDate` pour récupérer les journaux. S’il est omis ou défini sur `0`, les journaux sont récupérés uniquement pour `startDate`. Sur le **Cloud mutualisé**, vous pouvez récupérer au maximum 7 jours de journaux d’audit, même si vous définissez `numDays` sur une valeur plus élevée.
     * `startDate` : pris en charge uniquement sur le **Cloud mutualisé**. Définit les journaux les plus récents à récupérer, au format `startDate=YYYY-MM-DD`. S’il est omis ou explicitement défini sur la date du jour, les journaux sont récupérés d’aujourd’hui jusqu’à `numDays`.
3. Construisez l’URL complète du point de terminaison au format `<API-endpoint>?<parameters>`.
4. Exécutez une requête HTTP `GET` sur le point de terminaison API complet à l’aide d’un navigateur web ou d’un outil comme [Postman](https://www.postman.com/downloads/), [HTTPie](https://httpie.io/) ou cURL.

La réponse de l’API contient des objets JSON séparés par des sauts de ligne. Les objets comprennent les champs décrits dans le [schéma](#audit-log-schema), comme lorsque les journaux d’audit sont synchronisés vers un bucket au niveau de l’instance. Dans ce cas, les journaux d’audit se trouvent dans le répertoire `/wandb-audit-logs` de votre bucket.

<div id="use-basic-authentication">
  ### Utiliser l’authentification de base
</div>

Pour utiliser l’authentification de base avec votre clé API afin d’accéder à l’API des journaux d’audit, définissez l’en-tête `Authorization` de la requête HTTP sur la valeur `Basic`, suivie d’un espace, puis de la chaîne encodée en base 64 au format `username:API-KEY`. En d’autres termes, remplacez le nom d’utilisateur et la clé API par vos propres valeurs, séparées par le caractère `:`, puis encodez le résultat en base 64. Par exemple, pour vous authentifier en tant que `demo:p@55w0rd`, l’en-tête doit être `Authorization: Basic ZGVtbzpwQDU1dzByZA==`.

### Exclure les PII lors de la récupération des journaux d’audit <a id="exclude-pii" />

Pour [Autogéré](/fr/platform/hosting/hosting-options/self-managed) et [Cloud dédié](/fr/platform/hosting/hosting-options/dedicated-cloud), un administrateur d’organisation W&amp;B ou d’instance peut exclure les PII lors de la récupération des journaux d’audit. Pour [Cloud mutualisé](/fr/platform/hosting/hosting-options/multi_tenant_cloud), le point de terminaison API renvoie toujours les champs pertinents pour les journaux d’audit, y compris les PII. Ceci n&#39;est pas configurable.

Pour exclure les PII, passez le paramètre d’URL `anonymize=true`. Par exemple, si l’URL de votre instance W&amp;B est `https://mycompany.wandb.io` et que vous souhaitez obtenir les journaux d’audit de l’activité des utilisateurs au cours de la dernière semaine tout en excluant les PII, utilisez un point de terminaison API comme :

```text
https://mycompany.wandb.io/admin/audit_logs?anonymize=true&<additional-parameters>.
```

<div id="actions">
  ## Actions
</div>

Ce tableau décrit les actions possibles pouvant être enregistrées par W&amp;B, classées par ordre alphabétique.

| Action                        | Définition                                                                                       |
| ----------------------------- | ------------------------------------------------------------------------------------------------ |
| `artifact:create`             | L’artefact est créé.                                                                             |
| `artifact:delete   `          | L’artefact est supprimé.                                                                         |
| `artifact:read`               | L’artefact est consulté.                                                                         |
| `project:delete`              | Le projet est supprimé.                                                                          |
| `project:read`                | Le projet est consulté.                                                                          |
| `report:read`                 | Le rapport est consulté. <sup><a href="#1">1</a></sup>                                           |
| `run:delete_many`             | Le lot de runs est supprimé.                                                                     |
| `run:delete`                  | Le run est supprimé.                                                                             |
| `run:stop`                    | Le run est arrêté.                                                                               |
| `run:undelete_many`           | Le lot de runs est restauré depuis la corbeille.                                                 |
| `run:update_many`             | Le lot de runs est mis à jour.                                                                   |
| `run:update`                  | Le run est mis à jour.                                                                           |
| `sweep:create_agent`          | L’agent de balayage est créé.                                                                    |
| `team:create_service_account` | Le compte de service est créé pour l’équipe.                                                     |
| `team:create`                 | L’équipe est créée.                                                                              |
| `team:delete`                 | L’équipe est supprimée.                                                                          |
| `team:invite_user`            | L’utilisateur est invité dans l’équipe.                                                          |
| `team:uninvite`               | L’utilisateur ou le compte de service est retiré de l’équipe.                                    |
| `user:create_api_key`         | La clé API de l’utilisateur ou du compte de service est créée. <sup><a href="#1">1</a></sup>     |
| `user:create`                 | L’utilisateur est créé. <sup><a href="#1">1</a></sup>                                            |
| `user:deactivate`             | L’utilisateur est désactivé. <sup><a href="#1">1</a></sup>                                       |
| `user:delete_api_key`         | La clé API de l’utilisateur ou du compte de service est supprimée. <sup><a href="#1">1</a></sup> |
| `user:initiate_login`         | L’utilisateur initie la connexion. <sup><a href="#1">1</a></sup>                                 |
| `user:login`                  | L’utilisateur se connecte. <sup><a href="#1">1</a></sup>                                         |
| `user:logout`                 | L’utilisateur se déconnecte. <sup><a href="#1">1</a></sup>                                       |
| `user:permanently_delete`     | L’utilisateur est supprimé définitivement. <sup><a href="#1">1</a></sup>                         |
| `user:reactivate`             | L’utilisateur est réactivé. <sup><a href="#1">1</a></sup>                                        |
| `user:read`                   | Le profil de l’utilisateur est consulté. <sup><a href="#1">1</a></sup>                           |
| `user:update`                 | L’utilisateur est mis à jour. <sup><a href="#1">1</a></sup>                                      |

<a id="1">1</a>: Sur [Cloud mutualisé](/fr/platform/hosting/hosting-options/multi_tenant_cloud), les journaux d’audit ne sont pas collectés pour :

* Les projets Open ou publics.
* L’action `report:read`.
* Les actions `User` qui ne sont pas liées à une organisation spécifique.
