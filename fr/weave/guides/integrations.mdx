Weave propose deux façons d’intégrer votre stack IA :

* **Tracer les agents** : pour les applications agentiques multi-tours créées avec des SDK d’agents (comme l’OpenAI Agents SDK ou Google ADK) ou exécutées dans des harnesses d’agents (comme Claude Code, Codex ou Pi.dev). Ces intégrations capturent les sessions, les tours de conversation, les appels LLM et les appels d’outil, et les affichent dans la vue **Agents** de Weave.
* **Tracer les applications LLM** : pour le code qui appelle des fournisseurs de LLM (OpenAI, Anthropic, Bedrock et autres) ou utilise des frameworks d’orchestration (LangChain, LlamaIndex, DSPy et autres). Ces intégrations capturent les appels LLM individuels et les étapes du pipeline sous forme de **Calls** dans la vue **Traces**.

Si vous ne savez pas quelle approche choisir, commencez par [Tracer vos agents](/fr/weave/guides/tracking/trace-agents) pour les nouveaux projets d’agents multi-tours, ou [Tracer les applications LLM](/fr/weave/guides/tracking/tracing) pour les flux de travail existants basés sur des appels LLM.

<div id="trace-llm-applications">
  ## Tracer les applications LLM
</div>

Si votre application appelle directement l’API d’un fournisseur de LLM ou utilise un framework d’orchestration, Weave peut automatiquement intercepter les traces (grâce à l’autopatching) pour de nombreuses bibliothèques et de nombreux frameworks populaires. En important le SDK Weave dans votre code et en l’initialisant avec `weave.init`, chaque requête est enregistrée comme un **Appel** Weave avec les entrées, les sorties, la latence, l’utilisation des jetons et le coût.

Pour les bibliothèques que Weave ne prend pas en charge via l’autopatching, vous pouvez appliquer manuellement [Weave Ops](/fr/weave/guides/tracking/create-call#2-tracking-of-custom-functions) à votre code pour capturer les traces.
