# 아티팩트 변경으로 CI/CD 이벤트 트리거하기

아티팩트가 변경될 때 자동화를 생성하세요. 아티팩트 버전 관리를 위한 다운스트림 작업을 자동화하고 싶을 때 아티팩트 자동화를 사용하세요. 자동화를 생성하려면, 원하는 [액션](#액션-유형)을 기반으로 한 [이벤트 유형](#이벤트-유형)을 정의하세요.

아티팩트 변경으로 트리거되는 자동화의 일반적인 사용 사례에는 다음이 포함됩니다:

* 평가/보류 데이터세트의 새 버전이 업로드되면, 모델 레지스트리의 최고 학습 모델을 사용하여 추론을 수행하고 성능 정보가 포함된 리포트를 생성하는 [런칭 작업 트리거](#런칭-자동화-생성).
* 학습 데이터세트의 새 버전이 "프로덕션"으로 표시되면, 현재 최고 성능 모델의 설정에서 [재학습 런칭](#런칭-자동화-생성) 작업을 트리거합니다.

:::info
아티팩트 자동화는 프로젝트 범위 내에서만 적용됩니다. 이는 프로젝트 내의 이벤트만 아티팩트 자동화를 트리거한다는 의미입니다.

이는 W&B 모델 레지스트리에서 생성된 자동화와 대조됩니다. 모델 레지스트리에서 생성된 자동화는 모델 레지스트리의 범위 내에 있으며, [모델 레지스트리](../model_registry/intro.md)에 연결된 모델 버전에 수행된 이벤트가 트리거될 때 활성화됩니다. 모델 버전에 대한 자동화를 생성하는 방법에 대한 정보는 [모델 레지스트리 장](../model_registry/intro.md)의 [모델 CI/CD를 위한 자동화](../model_registry/automation.md) 페이지를 참고하세요.
:::

## 이벤트 유형
*이벤트*는 W&B 생태계에서 발생하는 변경입니다. 프로젝트의 아티팩트 컬렉션에 대해 두 가지 다른 이벤트 유형을 정의할 수 있습니다: **컬렉션에서 아티팩트의 새 버전이 생성됨** 및 **아티팩트 별칭이 추가됨**.

:::tip
**컬렉션에서 아티팩트의 새 버전이 생성됨** 이벤트 유형은 아티팩트의 각 버전에 반복적인 작업을 적용할 때 사용하세요. 예를 들어, 새 데이터세트 아티팩트 버전이 생성될 때마다 자동으로 학습 작업을 시작하는 자동화를 생성할 수 있습니다.

**아티팩트 별칭이 추가됨** 이벤트 유형은 특정 별칭이 아티팩트 버전에 적용될 때 활성화되는 자동화를 생성할 때 사용하세요. 예를 들어, 누군가가 "test-set-quality-check" 별칭을 아티팩트에 추가할 때 작동하도록 데이터세트에 대한 다운스트림 처리를 트리거하는 액션을 트리거하는 자동화를 생성할 수 있습니다.
:::

## 액션 유형
액션은 어떤 트리거의 결과로 발생하는 반응적인 변형(내부 또는 외부)입니다. 프로젝트의 아티팩트 컬렉션에서 이벤트에 대응하여 생성할 수 있는 액션 유형은 두 가지입니다: 웹훅과 [W&B 런칭 작업](../launch/intro.md).

* 웹훅: HTTP 요청으로 W&B에서 외부 웹 서버와 통신합니다.
* W&B 런칭 작업: [작업](../launch/create-launch-job.md)은 재사용 가능하고 구성 가능한 실행 템플릿으로, 새로운 [실행](../runs/intro.md)을 빠르게 런칭할 수 있게 해줍니다. 이는 데스크톱의 로컬 또는 Kubernetes on EKS, Amazon SageMaker 등의 외부 컴퓨팅 리소스에서 사용할 수 있습니다.


다음 섹션에서는 웹훅과 W&B 런칭으로 자동화를 생성하는 방법을 설명합니다.

## 웹훅 자동화 생성
W&B App UI를 사용하여 액션을 기반으로 웹훅을 자동화하세요. 이를 위해 먼저 웹훅을 설정한 다음, 웹훅 자동화를 구성합니다.

### 인증 또는 권한 부여를 위한 비밀 추가
비밀은 팀 수준 변수로, 자격증명, API 키, 비밀번호, 토큰 등과 같은 개인 문자열을 숨길 수 있게 해줍니다. W&B는 평문 내용을 보호하고자 하는 모든 문자열을 저장하기 위해 비밀을 사용하도록 권장합니다.

웹훅에서 비밀을 사용하려면 먼저 그 비밀을 팀의 비밀 관리자에 추가해야 합니다.

:::info
* W&B 관리자만 비밀을 생성, 편집 또는 삭제할 수 있습니다.
* 외부 서버로 HTTP POST 요청을 보낼 때 비밀을 사용하지 않는 경우 이 섹션을 건너뛸 수 있습니다.
* 비밀은 Azure, GCP, AWS 배포에서 [W&B 서버](../hosting/intro.md)를 사용할 때도 사용할 수 있습니다. 다른 배포 유형을 사용하는 경우 W&B에서 비밀을 사용하는 방법에 대해 W&B 계정 팀과 상의하세요.
:::

W&B가 웹훅 자동화를 사용할 때 생성하도록 제안하는 비밀 유형은 두 가지입니다:

* **액세스 토큰**: 보안 웹훅 요청을 돕기 위해 발신자를 인증합니다.
* **비밀**: 페이로드에서 전송된 데이터의 진위성과 무결성을 보장합니다.

웹훅을 생성하려면 아래 지침을 따르세요:

1. W&B App UI로 이동합니다.
2. **팀 설정**을 클릭합니다.
3. 페이지를 아래로 스크롤하여 **팀 비밀** 섹션을 찾습니다.
4. **새 비밀** 버튼을 클릭합니다.
5. 모달이 나타납니다. **비밀 이름** 필드에 비밀의 이름을 입력합니다.
6. **비밀** 필드에 비밀을 추가합니다.
7. (선택 사항) 웹훅이 추가적인 비밀 키 또는 토큰을 인증하는 데 필요한 경우(예: 액세스 토큰과 같은) 다른 비밀을 생성하려면 5단계와 6단계를 반복합니다.

웹훅 자동화를 구성할 때 사용할 비밀을 지정하세요. 자세한 내용은 [웹훅 구성](#웹훅-구성) 섹션을 참조하세요.

:::tip
비밀을 생성하면 W&B 워크플로에서 `$`를 사용하여 해당 비밀에 액세스할 수 있습니다.
:::

### 웹훅 구성
웹훅을 사용하기 전에 먼저 W&B App UI에서 해당 웹훅을 구성해야 합니다.

:::info
* W&B 관리자만 W&B 팀을 위해 웹훅을 구성할 수 있습니다.
* 웹훅이 추가적인 비밀 키 또는 토큰을 인증하는 데 필요한 경우 이미 [하나 이상의 비밀을 생성했는지 확인하세요](#인증-또는-권한-부여를-위한-비밀-추가).
:::

1. W&B App UI로 이동합니다.
2. **팀 설정**을 클릭합니다.
4. 페이지를 아래로 스크롤하여 **웹훅** 섹션을 찾습니다.
5. **새 웹훅** 버튼을 클릭합니다.
6. **이름** 필드에 웹훅의 이름을 입력합니다.
7. **URL** 필드에 웹훅의 엔드포인트 URL을 제공합니다.
8. (선택 사항) **비밀** 드롭다운 메뉴에서 웹훅 페이로드를 인증하는 데 사용하려는 비밀을 선택합니다.
9. (선택 사항) **액세스 토큰** 드롭다운 메뉴에서 발신자를 인증하는 데 사용하려는 액세스 토큰을 선택합니다.
9. (선택 사항) **액세스 토큰** 드롭다운 메뉴에서 웹훅을 인증하는 데 필요한 추가적인 비밀 키 또는 토큰을 선택합니다(예: 액세스 토큰).

:::note
POST 요청에서 비밀과 액세스 토큰이 지정된 위치를 보려면 [웹훅 문제 해결](#웹훅-문제-해결) 섹션을 참조하세요.
:::

### 웹훅 추가
웹훅을 구성했고 (선택적으로) 비밀이 있는 경우 프로젝트 워크스페이스로 이동하세요. 왼쪽 사이드바에서 **자동화** 탭을 클릭합니다.

1. **이벤트 유형** 드롭다운에서 [이벤트 유형](#이벤트-유형)을 선택합니다.
![](/images/artifacts/artifact_webhook_select_event.png)
2. **컬렉션에서 아티팩트의 새 버전이 생성됨** 이벤트를 선택한 경우, 자동화가 응답해야 하는 **아티팩트 컬렉션** 이름을 **아티팩트 컬렉션** 드롭다운에서 제공합니다.
![](/images/artifacts/webhook_new_version_artifact.png)
3. **액션 유형** 드롭다운에서 **웹훅**을 선택합니다.
4. **다음 단계** 버튼을 클릭합니다.
5. **웹훅** 드롭다운에서 웹훅을 선택합니다.
![](/images/artifacts/artifacts_webhooks_select_from_dropdown.png)
6. (선택 사항) JSON 표현식 편집기에 페이로드를 제공합니다. 일반적인 사용 사례 예제는 [예제 페이로드](#예제-페이로드) 섹션을 참조하세요.
7. **다음 단계**를 클릭합니다.
8. **자동화 이름** 필드에 웹훅 자동화의 이름을 제공합니다.
![](/images/artifacts/artifacts_webhook_name_automation.png)
9. (선택 사항) 웹훅에 대한 설명을 제공합니다.
10. **자동화 생성** 버튼을 클릭합니다.

### 예제 페이로드

다음 탭은 일반적인 사용 사례를 기반으로 한 예제 페이로드를 보여줍니다. 예제에서는 페이로드 매개변수에서 조건 개체를 참조하기 위해 다음 키를 사용합니다:
* `${event_type}` 액션을 트리거한 이벤트 유형을 나타냅니다.
* `${event_author}` 액션을 트리거한 사용자를 나타냅니다.
* `${artifact_version}` 액션을 트리거한 특정 아티팩트 버전을 나타냅니다. 아티팩트 인스턴스로 전달됩니다.
* `${artifact_version_string}` 액션을 트리거한 특정 아티팩트 버전을 나타냅니다. 문자열로 전달됩니다.
* `${artifact_collection_name}` 아티팩트 버전이 연결된 아티팩트 컬렉션의 이름을 나타냅니다.
* `${project_name}` 액션을 트리거한 변형을 소유한 프로젝트의 이름을 나타냅니다.
* `${entity_name}` 액션을 트리거한 변형을 소유한 엔티티의 이름을 나타냅니다.

:::info
GHA 워크플로를 트리거하는 데 필요한 권한 집합을 액세스 토큰이 가지고 있는지 확인하세요. 자세한 내용은 [GitHub 문서](https://docs.github.com/en/rest/repos/repos?#create-a-repository-dispatch-event)를 참조하세요.
:::

  W&B에서 GitHub 액션을 트리거하기 위해 저장소 디스패치를 보내세요. 예를 들어, `on` 키에 대한 저장소 디스패치를 트리거로 수락하는 워크플로가 있다고 가정해 보세요:

  ```yaml
  on:
    repository_dispatch:
      types: BUILD_AND_DEPLOY
  ```

  저장소에 대한 페이로드는 다음과 같을 수 있습니다:

  ```json
  {
    "event_type": "BUILD_AND_DEPLOY",
    "client_payload": 
    {
      "event_author": "${event_author}",
      "artifact_version": "${artifact_version}",
      "artifact_version_string": "${artifact_version_string}",
      "artifact_collection_name": "${artifact_collection_name}",
      "project_name": "${project_name}",
      "entity_name": "${entity_name}"
      }
  }

  ```

:::note
웹훅 페이로드의 `event_type` 키는 GitHub 워크플로 YAML 파일의 `types` 필드와 일치해야 합니다.
:::

  자동화가 구성된 이벤트 또는 모델 버전에 따라 렌더링된 템플릿 문자열의 내용과 위치는 다릅니다. `${event_type}`은 "LINK_ARTIFACT" 또는 "ADD_ARTIFACT_ALIAS"로 렌더링됩니다. 아래 예제 매핑을 참조하세요:

  ```json
  ${event_type} --> "LINK_ARTIFACT" 또는 "ADD_ARTIFACT_ALIAS"
  ${event_author} --> "<wandb-user>"
  ${artifact_version
